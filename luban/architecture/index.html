整体架构

OpenLuban 由 4 个应用服务 组成，职责清晰、可独立部署。理解这条调用链，是理解整个生态的入口。

关键边界：访客和运营走的是两条不同的前端，但背后共享同一个 BFF + 业务后端。这样设计，是因为访客要的是 SSR 秒开，运营要的是设计器的交互自由——两者诉求冲突，强行塞进一个前端只会两头不讨好。

请求与数据流

两条流并行存在：运行态（访客打开活动页，走 web SSR）和设计态（运营在设计器保存页面，走 engine）。共享同一套 Schema，但渲染路径完全分离。

设计态保存会写 MySQL 并失效 Redis，所以运营一保存、访客下次打开就是新页面——这正是低代码平台「所见即所发」的核心。缓存命中走绿线（秒开），回源走紫线（首次或失效后）。

Schema 规范

Schema 是整个引擎的「单一数据源」。设计器产出它，渲染端消费它，后端存它。结构分四段：version / metadata / tree（页面树）/ handlers（事件）。

{
  "version": "1.0",
  "tree": {
    "material": "LubanPage",
    "version": "1.0.0",
    "props": { "title": "示例页面", "layout": "vertical" },
    "children": [
      {
        "material": "LubanButton",
        "version": "1.0.0",
        "props": { "type": "primary", "label": "点击我" },
        "events": { "click": "handleClick" }
      }
    ]
  },
  "handlers": { "handleClick": { "type": "navigate", "url": "/detail" } }
}

每个节点三要素：material（物料名 + 版本）、props（属性）、可选 events（事件，指向 handlers）。渲染前引擎做 6 道校验——结构、物料存在性、版本兼容、props 合规、循环引用、事件处理器存在——任一不过都不中断渲染，而是显示错误占位并上报。这是刻意设计：一个物料坏了不能拖垮整页。

渲染管线

不管设计态还是运行态，渲染管线是同一条 10 步流程：

接收 schema → 结构校验 → 收集物料 → 等待加载（缺失则占位）
→ props 校验 → 循环引用检测 → 递归渲染（每节点 ErrorBoundary）
→ 绑定事件 → 完成

关键在「每个物料挂载点用 ErrorBoundary 包裹」——单个物料 throw 被捕获，显示占位，整棵树继续渲染。性能上，物料按需 dynamic import、渲染结果 memo 化、大列表虚拟滚动、多次 schema 更新合并为一次渲染。

设计态 vs 运行态

同一条管线，两种「渲染上下文」：

设计态 DesignRenderer运行态 RuntimeRenderer
宿主engine 设计器 /designerweb Nuxt SSR
额外能力拖拽 / 选中 / 属性编辑 / 撤销重做无，只渲染
Schema 来源运营实时编辑，内存态后端持久化的稳定版本
性能优先级交互响应（编辑要跟手）首屏速度（SSR 秒开）

差异不在渲染本身，而在渲染之外附加了什么。设计态套一层交互壳（选中框、拖拽响应、属性面板联动），运行态把这层壳剥掉只留渲染。同一份物料代码，两种用法。

多端渲染一致

schema 一致 → 各端渲染一致 是目标，但各端运行时差异会引入分叉。OpenLuban 的约束：物料样式只用标准 CSS 子集，避开 clip-path、backdrop-filter 这类各端支持度不一的属性；关键样式在 web / flutter 多端做交叉验证。这是把「一次编写多端运行」从口号落到工程纪律——靠限制表达力换一致性。

引擎刻意不做业务逻辑、不做持久化、不做权限、不直连后端 API——这些都在 BFF / 后端。引擎只管「把 Schema 渲染成页面」这一件事，边界越窄，越不容易腐烂。
