AI 助手与模块全景

OpenLuban 的 AI 助手能让运营用自然语言描述需求，直接生成可编辑的低代码页面——生成的是结构化 Schema，不是终态图片，运营可继续在设计器里微调。

工作原理

这是一条带人工确认（HITL）的 agent 流水线，不是「一键生成」黑盒：

  • 运营在 engine 的右侧抽屉（AiAssistantPanel.vue）用自然语言描述页面需求。
  • 前端经 SSE 流式 / WebSocket 把请求送到 Python 侧 FastAPI（/ai/chat、/ai/agent）。
  • LangGraph 状态图编排：调模型（LiteLLM 统一封装，首选 DeepSeek，可切 GLM / 通义）→ 从 Qdrant 检索可用物料 → 产出待确认的页面 Schema。
  • 关键在 checkpoint + HITL：生成的 Schema 先预览，运营确认 / 拒绝 / 要求改，而不是直接覆盖画布。拒绝会回到生成节点重来。
  • 确认后才注入设计器画布，运营继续拖拽微调。

这套设计的核心判断：AI 生成的是可编辑 Schema 而非终态图。所以 AI 不是替代运营，而是把「从零搭」变成「改一份草稿」——降低起手成本，又保留人的最终控制权。HITL 是刻意加的摩擦：低代码页面是要上线收线索的，不能让模型一键定稿。

模块全景

下表与 Hub 子项目全景口径一致，补「职责边界」列说明它和谁交互：

子项目职责边界（与谁交互）状态
engine设计器 + 运营后台→ bff（数据）活跃
bff聚合 / 鉴权 / 转码← engine/web，→ backend活跃
backend业务后端（唯一权威）← bff，→ MySQL/Redis活跃
web访客公开站点 SSR← 访客，→ bff活跃
ui组件库 + 物料→ engine 设计器、web 运行态活跃
ai图片/自然语言→页面助手← engine 抽屉，→ LiteLLM/Qdrant活跃
docs架构设计文档—活跃
clientflutter 多端客户端← backend API早期
sprint-mcp敏捷开发 MCP server← 开发工具链辅助

口径修正：早期文档把 luban-backend-go 列为「同业务另一套实现」，实际 meta 仓已废弃 Go 后端，Spring Boot backend 是唯一活跃后端，本仓不再纳入 Go 版。

技术栈矩阵

子项目技术栈
engineTypeScript · Vue 3 · Vite
bffTypeScript · Next.js
backendJava · Spring Boot · Flyway
webTypeScript · Nuxt（SSR）
uiVue 3 · Storybook（vue3-vite）· pnpm workspace
aiPython 3.12 · FastAPI · LangGraph · LiteLLM · Qdrant · PostgreSQL
clientFlutter

这套「前端 TS / 后端 Java / AI Python」的多语言分工是刻意的：每层用最趁手的工具，靠 Schema 做跨语言契约。前端不会因为后端是 Java 就别扭，AI 也不会因为要迁就前端就用 JS 写 agent。
