我用 AI 编码助手做全栈项目两年,主力是 Claude Code,但同一个项目里 Cursor、OpenCode、Codex、Trae 都会切着用。每个 IDE 都要配 rules、commands、skills,第一年里我反复在做同一件事:改一份 rules,去五个目录同步五遍。

这篇讲的就是怎么把这块最底层地基——配置怎么不重复——解决掉(和命令体系是配套的:那篇讲「有哪些命令」,这篇讲「这些命令怎么在多个 IDE 之间不重复地维护」)。

为什么不能只守着一个 IDE?没有一个 AI 助手在所有任务上都最优。我的体感:Claude Code 做长链路规划(写方案、拆任务)最强,Cursor 改 bug 时定位相关文件最顺手,Codex 适合挂后台跑批量重构,Trae 在某些中文语境下理解更准。任务不同,用的工具就不同。

这种重复不只是烦。它的真正代价是配置会漂移:Cursor 里的命令修了 bug,Trae 里还是旧的;Codex 加了新 skill,Reasonix 不知道。漂移积累到一定程度,AI 的行为就不可预测了——同一个 /kd-review 命令,在 Cursor 里收敛,在 Codex 里死循环。更隐蔽的是,漂移往往是静默的:你以为五个 IDE 行为一致,其实早就不一致了,直到某天在某个 IDE 里踩了坑才发现。

我把这套重复彻底干掉,沉淀成一套 SOP:一个 .agents/ 目录作单一事实源,五个 IDE 用不同策略同步,锁文件保证可复现。这篇按标准 SOP 文档的格式写,你照着能在自己的项目里复现。

一、这套 SOP 要解决什么

先说清楚它只解决什么问题,避免你套用到不合适的场景。

痛点:一个项目同时用多个 AI 编码助手时,rules(项目规则)、commands(自定义命令)、skills(技能)这三类配置需要在每个 IDE 的约定目录里都存在。手动复制会导致:

  • 配置漂移:改了 A 处忘了改 B 处,行为不一致
  • 版本失控:没有锁文件,换台机器装出来的 skill 集合不一样
  • 心智负担:每次新增命令,要记得「同步到哪些地方」

这套 SOP 的目标是:改一处,全 IDE 生效,且可复现安装。

适用前提:你的项目已经在用两个以上的 AI 编码助手。如果只用一个 IDE(比如只用 Claude Code),这套方案是过度设计,直接把配置写在它的约定目录即可。

我这套跑了五个 IDE,情况是这样:

  • Claude Code(主力):读 .agents/ + CLAUDE.md
  • Cursor:读 .cursor/commands、.cursor/rules、.cursor/hooks
  • Codex:读 .codex/skills、.codex/config.toml
  • Reasonix:读 .reasonix/skills
  • Trae:读 .trae/commands、.trae/rules、.trae/skills
  • OpenCode:读 opencode.json + .opencode/

五个 IDE 的配置目录各不相同,但内容高度重叠——rules 项目规则是一份,commands 命令定义是一份,skills 是同一批。重叠的部分,就是单一事实源的切入点。

二、核心原则(三条,记住就够)

在讲具体怎么做之前,先把三条不可动摇的原则摆出来。所有后续操作都围绕这三条展开。

原则一:.agents/ 是唯一事实源,其它 IDE 目录都是它的投影。

所有 rules、commands、skills 的真实内容只存在于 .agents/ 下。其它 IDE 的目录要么是 symlink、要么是指针、要么是受控副本——总之不允许在 .cursor/commands/ 里直接编辑命令内容。一旦你打破了这条,漂移就回来了。

原则二:同步策略匹配 IDE 的加载机制,不强求统一。

这是最容易踩的坑。很多人一听单一事实源,就「全用 symlink」。但不是所有 IDE 都认 symlink:有的 IDE 解析路径会跟随 symlink,有的会复制副本进缓存,有的干脆不读软链。每种 IDE 用它能接受的方式同步,而不是一刀切。后面会展开三种策略。

原则三:第三方 skill 用锁文件锁定,不进版本库的内容必须可复现。

skills 分两类:自研的(在 .agents/skills/ 里,进 git)和第三方的(从 GitHub 拉取的,如 ECC 生态)。第三方的不进版本库,但要保证「换台机器 clone 下来,跑一条命令能装出完全相同的 skill 集合」。靠的是 skills-lock.json 锁文件——记录每个 skill 的来源 URL 和哈希。

这三条原则是地基。下面的操作步骤,每一步都对应其中一条。

三、目录结构:单一事实源长什么样

事实源放在 .agents/ 下,分四个子目录,职责清晰:

.agents/
├── rules/        # 项目规则(AI 必须遵守的硬约束)
├── commands/     # 自定义命令(如 /kd-review、/prd)
├── skills/       # 自研技能 + 第三方技能(软链)
└── ecc/          # 第三方技能生态(ECC 等)的源码

每个目录的角色:

  • rules/ 是 AI 的「宪法」。每条规则是一个 .md,内容是「在什么情况下必须怎么做」。比如 flyway-new-migration.md 规定迁移文件用秒级时间戳命名。
  • commands/ 是 AI 的「工作流脚本」。每条命令是一个 .md,定义一个完整工作流(如代码审查的 21 个步骤)。这块和我之前写的 AI 命令体系是一回事。
  • skills/ 是 AI 的「能力包」。每个 skill 一个目录,内含 SKILL.md 描述触发条件和执行步骤。这部分数量最多(我这里有 60+ 个),也是同步最容易乱的地方。
  • ecc/ 是第三方技能生态源码。比如 everything-claude-code 这种社区项目,clone 进来后用 symlink 引到 skills/。

关键约束:这四个目录里的内容,只在这里编辑。所有 IDE 投影都指向这里,而不是反过来。

四、三种同步策略:匹配 IDE 的加载机制

这是整套 SOP 的技术核心。不同 IDE 对配置目录的读取方式不同,我用三种策略分别应对。

策略一:symlink(软链接)

适用 IDE:Cursor、Trae、Reasonix(部分)。这些 IDE 的加载逻辑是「读目录下的文件」,会跟随 symlink 解析到真实内容。

做法很直接:

# .cursor/commands/kd-review.md 指向 .agents/commands/kd-review.md
cd .cursor/commands
ln -s ../../.agents/commands/kd-review.md kd-review.md

改了 .agents/commands/kd-review.md,Cursor 下次读命令时自动看到新内容。零同步成本。

判断你的 IDE 适不适用 symlink:在它的配置目录建一个 symlink,启动 IDE 看命令能不能识别。能识别就用这招,最省事。

策略二:指针文件(pointer file)

适用 IDE:Codex。它的 skill 加载机制比较特殊——不直接读目录内容,而是读一个「指针文件」,指针文件里写真实 skill 的路径。

Codex 的 .codex/skills/ 下每个文件长这样:

# .codex/skills/accessibility 文件内容(62 字节)
/Users/your-project/.agents/skills/accessibility

就是一行路径。Codex 读到这行,去真实路径加载 skill。

这种策略的好处是对 IDE 透明——IDE 不用知道 symlink 是什么,只要它会读文件内容并按内容找路径就行。坏处是路径是绝对路径,换机器或换克隆目录就失效。

解决办法:用一个初始化脚本,克隆后自动重生成所有指针文件。脚本逻辑是「扫 .agents/skills/ 下所有目录,在 .codex/skills/ 生成同名指针文件,内容是对应绝对路径」。这个脚本进 git,新人 clone 后跑一次就行。

脚本本身不复杂,核心就一个循环:遍历 .agents/skills/ 下的目录名,拼出绝对路径,写入 .codex/skills/<同名文件>。但它解决了一个真问题——没有它,新人 clone 后第一次打开 Codex 会发现所有 skill 都不可用,排查半天才发现是「指针文件没生成」。初始化脚本是 SOP 的第一道入口,跑一次,后面就顺了。

策略三:受控副本(controlled copy)

适用 IDE:完全不认 symlink 也不认指针的 IDE,或者某些命令必须以副本形式存在(比如 Trae 对部分命令有缓存)。

受控副本的意思是:允许副本存在,但副本的生成和更新必须脚本化,禁止手动编辑副本。

# 同步脚本:symlink 不通的就 copy
sync_trae_specific() {
  # 这些命令 Trae 需要副本(有自定义缓存逻辑)
  cp .agents/commands/10-bs.md .trae/commands/10-bs.md
  cp .agents/commands/compact.md .trae/commands/compact.md
}

受控副本最大的风险是人忘了跑脚本,副本过期。所以这套有两个兜底:

  • 同步脚本挂在 pre-commit hook 上,提交前自动跑一遍(参考契约门禁的做法)
  • README 里写明「.trae/commands/ 下的文件是自动生成的,不要手动改」,文件头加 # AUTO-GENERATED, edit in .agents/

三种策略不是互斥的,同一个项目里可能混用。我的项目就是:Cursor 全 symlink,Trae symlink + 少量副本,Codex 全指针,Reasonix symlink,OpenCode 走配置文件。策略跟着 IDE 走,不跟着强迫症走。

有人会问:为什么不强求所有 IDE 都用同一种策略,比如「全部 symlink」?因为你改变不了 IDE 的加载机制,只能适应它。强行让不认 symlink 的 IDE 用 symlink,结果就是命令时灵时不灵,比漂移还难排查。务实的选择是:接受每个 IDE 的机制差异,用对应的策略喂给它,让结果一致(五个 IDE 都读到同一份内容),而不是让手段一致。

判断某个 IDE 该用哪种策略,有个简单的方法:在它的配置目录建一个 symlink,重启 IDE 看命令能不能识别。能识别就用 symlink(最省事);不能,试指针文件(建个内容是路径的文件,看 IDE 会不会按内容去找);都不行,只能受控副本。这个试探过程每个 IDE 只要做一次,确定后就写进 SOP,之后不用再想。

五、第三方 skill 的锁文件:可复现安装

自研 skill 进 git 就完事了,麻烦的是第三方 skill——它们从 GitHub 拉取,不进版本库。如果不锁,换台机器 clone 项目,skill 集合就是空的或残缺的。

skills-lock.json 解决这个问题。它的结构:

{
  "skills": [
    {
      "name": "brainstorming",
      "origin": "github:superpowers-org/superpowers",
      "ref": "v5.1.0",
      "path": ".agents/skills/brainstorming",
      "sha256": "a1b2c3..."
    },
    {
      "name": "kd-review",
      "origin": "local",
      "path": ".agents/skills/kd-review"
    }
  ]
}

每个 skill 记录四样东西:来源(local 还是 github)、版本/引用、本地路径、内容哈希。

安装时跑一条命令:

make skills-install   # 读 skills-lock.json,逐个拉取并校验哈希

这条命令做三件事:

  • origin: local 的跳过(已经在 git 里)
  • origin: github 的按 ref 拉取,落地到指定 path
  • 拉完算 sha256,和锁文件不符就报错(防止上游被篡改或版本漂移)

为什么要哈希校验:GitHub 的 tag 理论上不可变,但 fork、force-push、账号被盗都可能改变内容。锁了哈希,即使上游变了,你装出来的也是当初锁定的那个版本。这是可复现性的最后一道闸。

这套机制让 skill 管理变得和 package-lock.json 管依赖一样——确定性安装,确定性复现。

六、新增一个命令/Skill 的标准操作流程

日常最频繁的操作是「加一个新命令」或「加一个新 skill」。这套 SOP 下,标准流程是五步,缺一步就会出问题。先看全貌,再展开每一步。

第一步:在 .agents/ 下创建源文件。命令建在 .agents/commands/<name>.md,skill 建在 .agents/skills/<name>/SKILL.md。这是唯一的编辑位置。

SKILL.md 的骨架遵循 superpowers 生态的约定:开头是 YAML front matter(name + description),正文写「何时触发」「核心步骤」「示例」。一个最小骨架长这样:

---
name: my-new-skill
description: 触发条件描述,一句话说清什么时候用它
---

## 何时用本 skill
- 场景一
- 场景二

## 核心步骤
1. 第一步做什么
2. 第二步做什么

第二步:为每个 IDE 建立投影。按前面说的策略:能 symlink 的 symlink,只能用指针的生成指针,必须副本的 copy。不要直接在 .cursor/commands/ 里建文件。

这一步最容易出岔子的是「漏建某个 IDE 的投影」。我以前就干过:加了命令,symlink 到了 Cursor,忘了 Trae 也要一份,结果在 Trae 里用命令提示「命令不存在」,排查半天。所以后来我写了个投影清单表,每加一个源文件,对照着给五个 IDE 都建投影,打勾确认。

第三步:如果是第三方 skill,更新 skills-lock.json。新增一条记录,填 origin、ref、path,跑一次安装脚本让它自动算 sha256 填进去。

这里有个细节:origin 字段决定后续的更新行为。origin: github:org/repo 的 skill,更新时从 GitHub 拉最新 ref;origin: local 的,直接用 git 里那份。别填错 origin——把自研 skill 标成 github 来源,下次「安装」时会把 git 里的内容覆盖掉,白改。

第四步:跑验证脚本。这个项目里有个 verify_cursor_hooks.py 之类的脚本,检查所有 IDE 配置目录的投影是否和源一致。新人最常犯的错就是加了源文件忘了建投影,验证脚本能当场抓住。

验证脚本的核心逻辑很简单:扫 .agents/ 下所有 .md 和 SKILL.md,对每个源文件,检查它的每个 IDE 投影是否存在;对受控副本类,算源和副本的 sha256 比对。不一致就 exit 1,pre-commit hook 拦住提交。这和契约门禁是同一套思路——把「应该做但容易忘的事」变成机器自动检查。

第五步:提交。commit message 写明「新增 X 命令,已同步至五个 IDE」。源文件、投影(指针/副本)、锁文件一起进 commit,保证这次新增是原子的。

为什么要原子提交:如果源文件和投影分两次 commit,中间有人 pull,会拿到「有源没投影」的中间状态,他的 IDE 里命令就不可用。一次 commit 进齐,任何人 checkout 到这个 commit 都是完整可用的。

这五步看着繁琐,但因为它是确定性的,完全可以做成一个命令自动化。我的做法是一个 new-skill 脚本,传 skill 名字,它自动建目录、建 SKILL.md 骨架、生成所有 IDE 投影、跑验证。纪律性的东西,最好让脚本兜底,不要靠人记。

这也是为什么我把这套流程画成图挂在文档里——不是为了好看,是为了让每一个加命令的人,对着图走一遍,确认五步都做了。脚本处理常规情况,图处理「脚本没覆盖的异常情况」时的人工兜底。

七、常见问题与陷阱

实际跑这套 SOP,有几个坑一定会踩到,提前讲清楚。

陷阱一:symlink 在 Windows 上不生效。

Windows 默认要管理员权限才能建 symlink,这对很多开发者是阻塞。解决办法:要么开启 Windows 开发者模式(允许普通用户建 symlink),要么在 Windows 上降级用受控副本策略——让同步脚本检测 OS,Windows 走 copy,Unix 走 symlink。我的同步脚本第一行就是 if [[ "$OSTYPE" == "msys"* ]]; then CP=cp; else CP="ln -s"; fi。

陷阱二:IDE 缓存导致 symlink 更新不生效。

有些 IDE(Cursor 某些版本)启动时会把命令读进内存缓存,之后即使 symlink 指向的内容变了,IDE 里还是旧的。这不是 SOP 的问题,是 IDE 的行为。应对:改完命令后重启 IDE,或者用 IDE 提供的「reload」命令。在文档里写明这条,免得排查半天发现是缓存。

陷阱三:相对路径在指针文件里失效。

Codex 的指针文件如果写相对路径(../../.agents/skills/x),Codex 解析时可能以工作目录为基准,而不是指针文件所在目录,导致找不到。指针文件用绝对路径,配一个初始化脚本生成——脚本以项目根为基准算绝对路径,换机器重跑一次。这是指针策略必须接受的代价。

陷阱四:第三方 skill 上游删库。

锁文件锁的是某个 commit 的哈希,但如果上游仓库整个删了,重新 clone 时就拉不到。解决办法:关键 skill fork 到自己可控的账号下,锁文件里 origin 指向自己的 fork。我所有依赖的第三方 skill 都 fork 了一份,origin 写的是 github:my-org/xxx。

陷阱五:把 IDE 专属配置也强行统一。

不是所有配置都该单一事实源化。IDE 专属的东西留在 IDE 目录:比如 .cursor/hooks/ 里 Cursor 特定的 hook 脚本、.codex/config.toml 里 Codex 的 TMPDIR 设置——这些是 IDE 独有的,没有共享价值,不要硬塞进 .agents/。单一事实源只管跨 IDE 共享的内容,这是原则一的隐含边界。

我早期犯过这个错:把 .cursor/hooks/git-pr-shell-guard.js(一个禁止 push master 的 Cursor 专用 hook)也塞进 .agents/,想着「统一管理」。结果这个 hook 用了 Cursor 特有的 API,在别的 IDE 里根本跑不起来,反而成了噪音。共享的前提是内容真的跨 IDE 通用,否则只会制造混乱。

陷阱六:锁文件的 sha256 和实际内容对不上。

这种情况发生在:你手动改了第三方 skill 的本地内容(比如给它打了个 patch),但没更新 skills-lock.json 里的 sha256。下次有人 clone 项目跑 skills-install,脚本算出的哈希和锁文件不符,直接报错卡住。

正解是:改了本地内容,立刻重算 sha256 更新锁文件。或者更彻底——不要 patch 第三方 skill,要改就 fork 一份变成自己的(origin 指向 fork)。锁文件的哲学是「内容必须和锁完全一致」,任何偏离都要么更新锁,要么改 origin,不能两头不沾。

陷阱七:新人不知道要跑初始化脚本。

指针文件用绝对路径、第三方 skill 要拉取——这些都依赖一个初始化脚本。但新人 clone 项目后,常常直接开始用 IDE,根本不知道要跑脚本。结果 IDE 里命令残缺、skill 不可用,排查半天发现是「没初始化」。

应对:把初始化命令写进 README 最显眼的位置(我放在 README 第一段,标题就是「克隆后必做的一件事」),并且和项目本身的 make install / pnpm install 绑在一起跑。新人装依赖时初始化脚本顺带就跑了,不给他「忘记初始化」的机会。

八、验证标准:怎么知道这套 SOP 跑对了

SOP 不能只定义操作,必须定义怎么验证操作对了。这套方案的验证有三层。

第一层:结构验证。跑一个脚本,扫 .agents/ 下所有 rules/commands/skills,检查每个 IDE 目录的投影是否存在。缺投影就报错。这层抓「漏建投影」。

第二层:哈希验证。对受控副本类(Codex 指针、Trae 副本),算源文件和投影的哈希,不一致就报错。这层抓「副本过期」。

第三层:行为验证。在一个 IDE 里改一条命令,启动另一个 IDE 跑这条命令,确认改的内容生效。这层抓「IDE 缓存导致假同步」。这层最难自动化,定期手动抽查。

三层验证挂在 pre-commit hook 上(前两层)和 CI 里(全三层)。参考事故即规范的思路——任何一次「副本过期导致 IDE 行为不一致」的事故,都要转化成验证脚本的一条新检查。

九、从存量迁移:不是推倒重来

如果你的项目已经有一堆散落在各 IDE 目录的配置,别想着一天之内全部迁到 .agents/。存量迁移要渐进,否则迁移过程本身就制造一轮大漂移。

迁移分三步走。第一步,先梳理现状:列出每个 IDE 目录下现有的 rules、commands、skills,标出哪些内容是重复的、哪些是某个 IDE 独有的。这一步只看不动。

第二步,选一个 IDE 试点。通常是 Cursor(它对 symlink 最友好)。先把 Cursor 的命令迁成指向 .agents/ 的 symlink,跑通验证,确认行为没变。一个 IDE 跑通,再迁下一个,不要五个一起迁。

第三步,逐个 IDE 推进,每迁一个就更新对应的同步策略和验证脚本。最后把锁文件建起来,跑一次完整的 skills-install 确认可复现。

这套渐进迁移的关键是:每个阶段都保证「行为不变」。迁移是为了减少重复劳动,不是为了改变 AI 的行为。如果迁移后某个命令的行为变了,说明迁移引入了 bug,要回退排查。我用这套方法把 40 多个散落命令、60 多个 skill 迁进 .agents/,前后花了一周,但迁完后维护成本断崖式下降。

结语

这套 SOP 的本质,是用「单一事实源 + 分策略投影 + 锁文件」这三件事,把 AI 编码助手的配置从「每个 IDE 各配一遍的手工劳动」变成「改一处、全 IDE 生效、可复现的工程」。

它不是什么高深的技术——symlink、指针文件、锁文件,都是几十年前的老办法。但把它们组合起来对齐到一个明确的目标(配置不漂移、安装可复现),就能解决 AI 协同研发里最隐蔽也最耗人的一类重复劳动。

这套方案适合多大规模的团队?我的经验是:只要你在用两个以上的 AI 编码助手,就值得搭。一个人用也值得——省的是你自己的重复劳动;三五人的团队更值得,因为配置漂移在人多了之后会指数级放大,你不搭 SSOT,光是对齐「为什么你的 Cursor 里命令跟我的不一样」就要开好几次会。

更大的团队(十人以上)用这套,还要额外加一层:配置变更的 review 流程。改 .agents/ 里的规则,最好走 PR 而不是直推,让至少一个人 review——因为规则一改,所有人的 IDE 行为都跟着变,影响面太大,值得多一道把关。

这套方案我跑了半年,新增命令的成本从「改五处、praying 别漏」降到「改一处、跑脚本、完事」。省下来的不只是时间,更是对 AI 行为的可预期——无论在哪个 IDE 里,/kd-review 跑出来的都是同一套逻辑。

落到执行上,三句话:内容只在 .agents/ 里改、投影按 IDE 机制选策略、锁文件保证可复现。这三条背后有脚本和验证兜底,你日常记这三条就够了。重复劳动交给机器,精力留给真正要想的问题。

如果你正被「换 IDE 就重配一遍」折磨,今天就把 .agents/ 建起来——先统一命令和规则这两类,跑顺了再迁 skills。一次性的搭建成本,换长期零重复维护,这笔账怎么算都划算。配置这件事,做得对的话是一次痛苦、长期受益;做得不对则是每次都痛苦、永远在返工。
