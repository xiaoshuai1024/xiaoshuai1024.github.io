【AI 研发实战】为什么端到端测试在 AI 编码时代被严重低估

这篇是我「AI 开发」阶段质量体系里的一块,和之前几篇是一套:

  • 那篇 三层测试纪律讲的是「单元测试、集成测试、端到端测试三层都得在、谁也别偷懒」;
  • 覆盖率诚实讲的是「别拿覆盖率当 KPI」;
  • 编译能过 ≠ 代码正确讲的是「门禁得跑到 verify」。

这篇专门把端到端测试拎出来当主角。因为我的判断是:它一直是金字塔尖上最不受待见的那层——慢、贵、脆,人人往后排。但在 AI 大量写代码的今天,它恰恰是杠杆最大、最该加码的那一层。

我的结论先放这:单元测试告诉你「零件没坏」,端到端测试告诉你「整车能不能开」。AI 让零件造得太快、装得太猛,这道整车下线检测被严重低估了。

先说为什么 AI 时代的测试压力是暴涨的

两件事同时发生了,叠在一起才可怕。

第一件,单位时间产出的代码量暴涨。以前一个工程师一天能写多少行有效逻辑,大家心里有数;现在主力工具是 Claude Code,辅以 Cursor、OpenCode、Codex,Agent 一个下午能产出的量,赶上过去一周。产能上去了,可审查的时间没涨,人 review 的带宽是死的。

第二件,团队在转向全栈。以前前后端分得很清,前端出 bug 多半在前端那层,后端出 bug 多半在后端那层,各管一段。现在全栈化了,一个人/一个小团队从页面一直写到数据库,中间跨了好几个域。没有哪一个人,对「从前端点击到落库」这条完整链路有完整心智。

这两件事单拎一个出来,都还能应付。叠加在一起,问题就变了味道:bug 不再平均地分布在「某个函数写错了」这种颗粒度上,而是整体上移,大量集中在「拼起来才出问题」的地方——服务和服务之间、域和域之间、接口承诺和实际行为之间。

而这个「拼起来」的层面,恰恰是单元测试够不着的那一层。

单元测试为什么兜不住

先说清楚,我不反对单元测试。它永远需要——快、便宜、能反复跑,抓纯逻辑和分支边界是它最擅长的事。我反对的是「有了单元测试就觉得稳了」这个错觉。

单元测试兜不住的,是几样东西:

  • 环境差异。单元测试里你 mock 掉了下游服务和数据库,跑得飞快。但真实环境里那个下游服务返回的字段比你 mock 的多了一个,序列化行为不一样,单元测试看不见。
  • 实现被替换。单元测试验的是「我假设它这样,我这样调」。可假设的那个实现,和线上真实的那个实现,常常不是同一个东西。mock 出来的接口永远会按时返回,真实的不会。
  • 系统交互。一个请求要穿过网关、A 服务、消息队列、B 服务,最后写库。单元测试把这段切成一段段各自测,每段都对,拼起来挂了。挂的那个点,落在段与段的缝里,单元测试一个都摸不到。
  • 真实调用链。单元测试不碰真实的网络、真实的中间件、真实的并发。而很多 bug 恰好就活在「真实调用链」里——超时、重试、幂等、串号。

说白了,单元测试验证的是「这个函数对不对」,可线上炸的从来不是「某个函数」,而是「这条链路」。

我自己就吃过这个亏。一个接口,单元测试写得满满当当,全绿。联调那天一跑,挂了——原因不在接口内部逻辑,而在它依赖的那个服务,线上版本和单元测试里 mock 的版本,字段语义对不上。全绿给了我一个「稳了」的幻觉,而这个幻觉差点带到生产。

这件事之后我就想明白了一个朴素的道理:单元测试是零件质检,端到端测试是整车下线。零件全合格,不等于整车能开出门。

端到端测试为什么是唯一能「跨」的

我把端到端测试的价值浓缩成一个字,就是「跨」。它能跨单元测试、集成测试怎么都够不到的边界:

  • 跨系统:一个操作从 C 端入口出发,穿过网关、A 服务、消息队列、B 服务,最后落到数据库。单元测试只能测其中一段,端到端测试测的是这条整链。哪一段断了、哪一段字段对不上,一次跑穿就暴露。
  • 跨域名:订单域、支付域、分润域,各自有各自的领域模型。一个业务流程要横穿好几个域。域和域交界处的字段映射、状态同步,是 bug 的高发地,而这片地只有端到端测试能踩到。
  • 跨业务:一个完整的业务闭环——下单、支付、分润、对账——要跑通才算「这个功能真的能用」。单元测试测不到这种「业务闭环」,它测的是零件;端到端测试测的是业务跑没跑通。
  • 落到数据库验证最终态:这条我最想强调。端到端测试不是「接口返回 200 就算过」,它跑完之后直接去查库,断言数据真的写对了——状态机迁移对不对、金额对不对、tenant_id 隔离对不对。接口返回和真实落库之间,隔着一个「数据到底对不对」的天堑,端到端测试是少数能跨过去的。

这四条,每一条都是单元测试、集成测试的天花板,端到端测试才够得到。

看这张图最该看的一点不是「端到端测试最强」,而是AI 时代的 bug 分布整体上移了。以前 bug 散在金字塔各层,单元测试能逮住相当一部分;现在大量 bug 挤在上面那两层(集成、系统),单元测试摸都摸不到。这就是为什么我说端到端测试的权重该被拉高——不是因为它变强了,是因为敌人变高了。

我们这个全栈项目里,端到端测试到底怎么落地的

讲了半天道理,说点干的。我在一个全栈项目里(多租户 SaaS,前端 + 运营后台 + Java 后端),端到端测试是这么几条具体的规矩。

规矩一:全链路跑穿,而且断言落库。

我们的端到端测试不是「点一下接口看 200」。它会从一个真实入口点出发,把整条链路跑穿,跑完之后直接连数据库,查最终态断言。

举个真事。有次 AI 做了个「性能优化」,把扣库存从同步改成了异步——接口秒回 200,体验确实快了。单元测试全绿(因为它把下游和库存服务全 mock 掉了,异步逻辑在 mock 下看不出问题)。本地联调也顺,因为低并发下异步基本都来得及。

是端到端测试的落库验证把这事逮住的:跑完一笔下单,直接查库存表,断言「扣减已生效」。一跑,红——库存压根没扣,异步任务在那个改动里被静默吞了。接口 200 和库存真扣,隔着一条命。

如果当时只断言接口返回,这事能一路溜进主干,等大促当天库存对不上才爆。接口对 ≠ 数据对,这条就是端到端测试落库验证的价值。

// 落库端到端测试：接口 200 不算数，跑完整条链路后直接查库，断言最终态
OrderResult res = entry.placeOrder(tenantA, sku, qty);     // 从真实入口点下单
assertThat(res.getCode()).isEqualTo(200);                  // 这行单元测试也能过，没价值

// 真正兜命的是这两行 —— 断言库存真扣了
StockRow row = stockRepo.lockByTenantAndSku(tenantA, sku);
assertThat(row.getAvailable()).isEqualTo(origin - qty);    // 异步吞任务时，这行直接红

规矩二:跨业务闭环,断言各域数据一致。

我们还把「一个完整业务闭环」写成端到端测试:一笔订单,从下单到支付到分润,跨三个域一路跑。

为什么要跨到这种粒度?因为单域的单元测试永远全绿。有次 AI 重构了订单域的内部数据结构,单看订单域,单元测试全过,逻辑自洽。可它改了一个跨域字段的含义,订单写出去的金额,到了分润域被理解成了另一个口径——分润算错了。

单域单元测试一个都测不到这事,因为每个域关起门来都「对」。是那条跨业务的端到端测试,断言「分润金额 = 订单金额 × 配置比例」,一跑就红了。

这种 bug 的特点就是:每个零件都合格,拼起来整体错了。单元测试的颗粒度天生切不到「拼起来」,只有跨业务的端到端测试能切到。

// 跨业务端到端测试：一笔订单跑完 下单 → 支付 → 分润 三个域，断言各域数据一致
Order order = flow.placeAndPay(tenantA, amount);             // 跨订单域 + 支付域
ProfitShare share = profitDomain.getShare(order.getId());    // 分润域

// 跨域一致性 —— 分润金额 = 订单金额 × 配置比例
assertThat(share.getAmount())
    .isEqualByComparingTo(order.getAmount().multiply(rate(order)));
// 单域单元测试全绿也逮不到这行：订单域改了字段语义，分润域按错口径算

规矩三:多租户越权,从入口点验到库。

多租户项目里,越权是命门。A 租户的数据,B 租户绝对不能看到。我们有一条端到端用例专门干这个:用 A 租户账号造一份数据,再切到 B 租户账号去查,断言查不到。

注意,这条不是在某个接口层面测,而是从真实入口点进,验到数据库层。一个 JOIN 漏了 tenant_id,单元测试看不出(mock 掉了数据源),编译不管(语法没问题),人工评审容易漏(那条 SQL 单看没毛病)。但它就是会在端到端测试里红——B 租户居然查到了 A 的数据。

// 多租户越权端到端测试：从真实入口点进，A 造数，换 B 登录态查，断言查不到 —— 验到库层
String dataId = entry.createAs(tenantA, payload);           // A 租户登录态创建
List<Item> hit = entry.queryAs(tenantB);                    // 换 B 租户登录态查询

// 越权断言：B 的结果里绝不能出现 A 的数据
assertThat(hit).noneMatch(it -> it.getId().equals(dataId));
// JOIN 漏 tenant_id 时这行直接红 —— 单元测试（mock 掉数据源）和编译都看不见

后来这种「JOIN 漏 tenant_id」被进一步固化成 SQL 审查清单,这是 事故即规范那篇的事。但把它第一时间逮住的,是端到端测试。

规矩四:fail-fast,第一个红就停。

端到端测试贵在「跑」,一个套件几十分钟不稀奇。所以我们的规矩是 maxFailures: 1——第一个用例红,立刻停,后面全不跑了。

理由很直接:端到端测试本来就是端到端的,前面一断,后面基本白跑(登录挂了,后面所有用例都废)。停在第一个,省下来的时间拿去看「到底哪断了」,而不是等半小时看一屏红。这条对 Agent 尤其重要,因为 Agent 不会自己判断「这个失败要不要继续」,得用配置钉死。(这条纪律在三层测试那篇细聊过。)

这四条叠起来,端到端测试在我们项目里就不是「锦上添花」,而是「兜命的那道闸」。

端到端测试这么好,为什么不全堆

讲了一堆端到端测试的好,得把另一面也讲清楚,否则就是另一种忽悠。

它的毛病摆出来很扎眼:慢、贵、脆。一个端到端用例的运行成本,可能是单元测试的几百倍。它还脆——依赖一变、数据一脏、环境一抖,它就 flaky,红绿不定,久而久之大家就开始不信任它,甚至想跳过。

所以「全堆端到端测试」是错的,「越多越好」也是错的。我的做法是只堆关键链路:

  • 不追求端到端覆盖率,追求关键业务链路 100% 覆盖。每个分支都写端到端测试是浪费(那是单元测试的活),但「下单、支付、分润」这条主线,一个都不能少。
  • 端到端测试只测主线通不通 + 关键边界,不测每个 if 分支。分支逻辑交给单元测试,它只回答一个问题:「这条业务,真实跑起来通不通、数据对不对」。
  • 用 testcontainers 起真实依赖,让环境可复现。它脆,很大程度是环境不可控。我们用 testcontainers 真起数据库和中间件,每次跑都是干净环境,flaky 大幅下降。
  • 测试数据立契约:租户软删除(别真删)、固定账号复用、谁造的数据谁清(这条也是三层测试那篇的纪律)。数据一脏,必 flaky。

一句话:端到端测试贵,所以更要花在刀刃上——关键链路全覆盖,边缘分支零投入。 这和 覆盖率那篇的态度是一脉相承的:别追那个百分比,追「关键路径有没有被网住」。

收个口:不是推翻金字塔,是重心上移

写到这,得防一个误读——「那单元测试是不是可以少写了」。

不是。三层一个都不能少,这个立场我在 三层测试纪律那篇说得很硬。单元测试仍要快、多、抓逻辑分支;集成测试仍要兜单服务和真实中间件;端到端测试兜跨系统的业务链路。三层各管一段,谁也替不了谁。

我在说的不是「拆金字塔」,是金字塔的重心该上移了。

以前,bug 均匀散在各层,投资可以重点放在底层(单元测试又快又便宜,性价比高)。现在 AI 把 bug 的分布整体往上推了——大量 bug 挤在「集成、系统、数据」这层,而这层只有端到端测试够得到。如果投资比例还停在「单元测试占大头、端到端测试象征性写几条」,等于把主力防在最底层,让最危险的那层门户大开。

所以我的态度很明确:金字塔形状没变,但 AI 时代,该往端到端测试那一层加码。 该加的人、该加的时间、该加的环境投入,都往上挪。

这层投入最不讨好——它慢、它的功劳常被当成「本来就该对」、它红了还会被嫌「挡了进度」。但它是整车能不能开出门的最后一道闸。AI 造零件造得越快,这道闸越不能省。

一句话总结

单元测试验零件,端到端测试验整车。AI 让零件造得又快又多,bug 整体上移到了「拼起来才出事」的那层——而那层,只有端到端测试跨得过去。

所以别再让端到端测试当金字塔顶上那个「可写可不写」的装饰了。它被低估了太久。

你们团队现在端到端测试的投入比例是多少?是「关键链路全覆盖」,还是「象征性写几条交差」?评论区聊聊,我想知道这事是不是只有我们较真。
