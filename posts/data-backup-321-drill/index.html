我见过太多团队对「备份」的理解停留在「mysqldump 定时跑一下」。这种理解会害死人。备份和恢复是两件事,而真正决定数据安全的,是恢复——更准确地说,是你能不能在灾难发生时,确定性地把数据救回来。

(这篇和多租户数据隔离、事故即规范是一组:那两篇讲「怎么不让数据出问题」,这篇讲「万一出了,怎么救得回来」。)

我主导的多租户 SaaS 项目有一次差点翻车:一个清理 OSS 孤儿图片的定时任务,因为差集逻辑写反,批量物理删了一批用户图片。那次之后我定了一条规矩:任何数据资产,都必须有一套可执行的恢复 SOP,而且每月真刀真枪演练一次。这套 SOP 我跑了快一年,这篇按标准运维文档的格式把它拆开。

一、这套 SOP 的目的与边界

目的:保证 MySQL 数据库、OSS 对象存储、秘钥配置这三类核心数据资产,在任何单一故障(磁盘损坏、误删、云厂商区域故障、勒索软件)下,RPO ≤ 5 分钟,RTO ≤ 1 小时。

边界——什么不纳入这套 SOP:

  • Redis:缓存和会话,可从 MySQL 重建,自身有 RDB+AOF 持久化兜底
  • Elasticsearch:业务索引可从 MySQL reindex 重建;AI 向量库重建成本高但属非事务数据

把边界划清楚很重要。什么都想备份等于什么都没备份好——资源有限,精力有限,必须聚焦在「丢了真的会死」的数据上。MySQL(交易数据)、OSS(用户内容)、秘钥(服务能不能启动)这三类是命根子,其它都能从它们重建。

适用场景:单实例或主从架构的中型 SaaS(日均 API 百万级、OSS 日增 GB 级)。如果你已经是多活架构,这套方案是降级的;如果你只是个人项目,这套是过度设计。

二、核心原则:3-2-1 与「演练即门禁」

整套 SOP 立在两条原则上,缺一条就垮。

原则一:3-2-1 备份原则。

这是备份领域的经典铁律,含义是:

  • 3 份副本(生产数据 + 2 份备份)
  • 2 种不同介质(比如本地磁盘 + 对象存储)
  • 1 份异地(不同可用区,最好不同云厂商)

3-2-1 的精神是消除单点:任何一种介质、任何一个机房、任何一家云厂商出问题,都至少还有一份能用的副本。我的项目里,MySQL dump 同时存「异地从库本地磁盘 + 七牛 OSS + 阿里云 OSS」三份,OSS 对象在「七牛主 bucket + 七牛备用 bucket + 阿里云冷备」三处。这就是 3-2-1 的落地。

原则二:演练失败阻断发布。

这是比备份本身更重要的一条。没演练过的备份等于没备份——你不知道 dump 文件是不是损坏的、不知道恢复要多久、不知道从库能不能正常提升为主。我的项目里,每月一次恢复演练,演练失败就冻结当期发布,直到问题解决。这条门禁逼着运维不敢懈怠,也逼着备份脚本必须真能跑通。

这两条原则贯穿后面所有操作。

三、MySQL 的三层备份体系

MySQL 是最核心的数据资产,单靠一种备份方式都不够。我用三层叠加。

第一层:binlog 异地持续复制。

这是 RPO 的主力。主库开启 binlog(binlog_format=ROW、sync_binlog=1、expire_logs_days=7),异地部署一台只读从库,通过 binlog 持续复制。延迟监控阈值 5 分钟,超了飞书告警。

这一层保证:主库挂了,从库最多落后 5 分钟的数据。它是热备,恢复时直接把从库提升为主即可,几秒钟的事。

关键配置不能省:

[mysqld]
server-id = 1
log-bin = mysql-bin
binlog_format = ROW
binlog_row_image = FULL
sync_binlog = 1                      # 每事务刷盘,保证不丢
innodb_flush_log_at_trx_commit = 1
expire_logs_days = 7

从库部署要求两条硬规矩:独立服务器/独立云账号(避免同云同账号单点)、只读不承接业务流量(只给备份和恢复用)。

「独立云账号」这一条常被忽略,但它至关重要。如果你的主库和从库在同一个云账号下,那么账号被封、欠费、被入侵,主从一起完蛋——单点并没消除。我的做法是主库在 A 云、从库在 B 云(不同厂商),用 binlog 跨云复制。跨云复制延迟会比同区域高一点(通常几百毫秒到几秒),但换来的是真正单点消除,值得。

另外,从库的硬件规格不能低于主库。很多人图省钱,从库用个低配机器,平时复制跟得上,一旦主库跑个大事务(比如批量更新十万行),从库回放速度跟不上,延迟越积越大,最后崩了。从库规格 ≥ 主库,这是保底。

第二层:每日全量 mysqldump。

binlog 复制解决「最近 5 分钟」,但万一 binlog 文件本身损坏、或者从库数据也被污染了(比如误删 SQL 同步过去了),就需要一个干净的逻辑快照兜底。

每日凌晨 03:00 从从库(不是主库,避免压主库)做 mysqldump:

mysqldump \
  --host=<异地从库IP> --user=backup \
  --single-transaction --routines --triggers --events \
  --master-data=2 \
  dayuancheng \
  | gzip > /backup/mysql/dayuancheng-$(date +%Y%m%d).sql.gz

sha256sum /backup/mysql/dayuancheng-$(date +%Y%m%d).sql.gz >> checksums.txt
rclone copy /backup/mysql/dayuancheng-$(date +%Y%m%d).sql.gz \
  oss-backup:backup/mysql/

三个细节决定这层有没有用:

  • --single-transaction 保证 InnoDB 一致性快照,不锁表
  • --master-data=2 记录 binlog 位点,供 PITR(时间点恢复)接力
  • sha256sum 校验 + rclone 跨云上传,本地留一份、跨云 OSS 留一份

第三层:备份清单 manifest + 日报。

这层不是数据备份,是备份的备份的可观测性。每天 06:00 一个脚本汇总当日所有备份的状态(MySQL dump 是否成功、大小、哈希;OSS 同步对象数;从库复制延迟),生成 JSON 清单,推送到飞书。

为什么单独拎一层?因为备份静默失败是最可怕的——脚本挂了没人知道,等真要恢复时才发现最近一个月的 dump 都是空的。日报逼着你每天看一眼「备份健康度」,异常当场暴露。

日报格式别搞复杂。我的是一张表:日期、任务名、状态、产出大小、哈希校验、从库延迟秒数,失败行标红附日志路径。运维扫一眼就知道今天有没有问题,不用点进去翻。日报推到飞书群,所有人可见——可见性本身就是监督,运维知道有人在看,就不会让备份脚本默默挂掉。

三层叠加,MySQL 的 RPO 锁在 5 分钟内,且任何单层失效都有下层兜底。

四、保留策略:3-2-1 的副本数怎么算

备份不能无限保留(存储成本),也不能只留最近一份(误删可能几天后发现)。我的保留策略按周期分级:

版本副本数存储位置保留周期
日备3从库本地 + 跨云 OSS(2 份不同区域)7 天
周备(每周日)3同上4 周
月备(每月 1 日)3同上12 个月

这套设计解决一个真实场景:误删的发现延迟。我见过最极端的案例是,某条数据被误删两周后才被发现——因为业务逻辑是月度结算时才读到那条记录。如果只保留 7 天日备,两周前的数据早被覆盖了。周备和月备就是为这种「延迟发现」兜底。

有人会问:保留周期越长,存储成本越高,值吗?我的算账方式是——数据丢失的代价,永远高于存储成本。一个月备存 12 个月,增加的 OSS 存储费可能就几十块;但丢失一个月的业务数据,光是客户索赔和信任损失就远超这个数。备份存储是保险费,不是成本项,省不得。

另外,保留策略要分级清楚:日备快速恢复(最近的误删),周备应对中延迟发现,月备应对极端的长延迟发现或审计需要。别图省事只留日备——「误删两周后发现」这种事,概率不高但一旦发生就是灾难。多存几个周期的成本,买的是对「低概率高损失」事件的保险。

副本数固定是 3,对应 3-2-1 的「3 份副本」。三份分别在不同位置(本地磁盘、七牛 OSS、阿里云 OSS),满足「2 种介质、1 份异地」。

有人会问:云厂商不是提供自动快照吗,为什么还要自己 dump?快照是云厂商层面的磁盘级备份,有用,但不能替代 mysqldump。原因有二:一是快照是整个磁盘的,恢复要整个实例回滚,没法只恢复某张表;二是快照锁死在一家云厂商,不满足「1 份异地」。mysqldump 产出的是独立 SQL 文件,能跨云、能单表恢复、能 grep 出特定数据。快照做兜底,dump 做主力和跨云搬运,两者互补不替代。

五、OSS 跨云冷备:别让对象存储成为单点

很多团队对 OSS 有个错觉:云厂商承诺 99.999999999% 持久性,所以不用备份。这个错觉会被两件事击碎——云厂商区域级故障和误删。

我的项目里那次「孤儿图片清理任务反向差集」事故,就是在云厂商的高持久性承诺下,因为代码 bug 把正确图片当孤儿删了。云厂商管「硬件故障导致的数据丢失」,不管「你自己代码删错」。

OSS 的容灾策略三件套:

版本控制 + 软删除:主 bucket 开启版本控制,业务代码禁止直接调 delete API,改为「软删除标记 + 30 天 TTL」。清理任务分两段:先扫描软标记移到回收 bucket,回收 bucket 再 TTL 14 天后才真删。这样误删有 44 天的挽救窗口。

七牛跨区域镜像:用七牛官方存储空间镜像或 qshell qupload2 增量同步到备用 bucket,备用与主 bucket不同区域(主华东、备华南)。每小时同步一次。

跨云 rclone 冷备——3-2-1 的关键一环:

rclone sync qiniu:zhaokang666 alioss:backup-oss/zhaokang666-$(date +%Y%m%d) \
  --checksum --transfers 8 \
  --log-file=/var/log/rclone-oss-backup.log

跨云(七牛 → 阿里云)是精髓。只跨区域不跨云,云厂商整体出问题(账号被封、区域级故障、甚至跑路)时照样完蛋。跨到另一家云厂商,才算真正消除单点。

每日中午做一次全量对账:两边 object count 偏差 > 1% 就告警。这个阈值很关键——rclone 增量同步偶尔会漏,对账是「跨云一致性」的验证闸。

那次孤儿图片事故之后,我把 OSS 的防误删机制做了彻底加固。核心教训是:云厂商的高持久性承诺,管的是「硬件坏了我赔」,不管「你代码删错我赔」。所以我定了条规矩——业务代码里任何一处调 OSS delete API,都必须先经过 code review,并且必须走「软删除标记 + TTL 回收」两段式,禁止直接物理删除。

清理任务的逻辑也改了。原来是「扫描孤儿 → 直接删」,改成「扫描孤儿 → 移到回收 bucket → 回收 bucket TTL 14 天后才真删」。这样即使差集算错了,也只是把图片移进了回收站,44 天内随时能捞回来。给误删一个冷静期,比事后从备份恢复便宜得多。

六、秘钥加密冗余:被忽视的致命资产

最后这块最容易被忽略,但出事就是服务直接起不来。.env.prod 里有数据库密码、Redis 密码、微信 AppSecret、七牛 AK/SK、微信支付证书、租户主密钥——这些东西丢了或损坏,备份再多 MySQL 也白搭,因为服务连数据库都连不上。

秘钥备份用 age 加密(比 GPG 更简单现代),流程:

# 1. age 公钥加密
age -r <recipient.pub> .env.prod > .env.prod.age

# 2. 上传跨云 OSS
rclone copy .env.prod.age oss-backup:config-backup/env.prod-$(date +%Y%m%d%H%M).age

# 3. 记录变更日志
echo '{"when":"...","who":"...","what":"..."}' >> config-changelog.json

三个细节:

  • RPO 是 0:秘钥变更(改密码、续证书)立即触发备份,不等定时任务。因为秘钥一旦变,旧备份就失效,服务用新秘钥起不来时,你需要的是最新的那份。
  • 公私钥分离:加密公钥在服务器上,解密私钥由运维离线保管(U 盘、密码管理器),不存服务器。这样即使服务器被入侵,攻击者拿到加密后的 .age 文件也解不开。
  • 变更日志:每次备份记 Who/When/What,恢复时能追溯「这个版本是谁、为什么改的」。

秘钥恢复流程相对简单:从跨云 OSS 拉最新 .age → age --decrypt -i private.key → 部署 → 校验 MD5 与变更日志一致。但要演练——秘钥恢复演练失败,意味着加密体系断了,照样阻断发布。

为什么秘钥要单独拎出来讲?因为它有个其它备份没有的特点:秘钥是「启动钥匙」,丢了它,即使 MySQL 备份完好,服务也起不来。数据库密码在 .env.prod 里,如果这个文件丢了或损坏,你 dump 灌进去也连不上。所以秘钥的备份优先级,实际和 MySQL 同级,甚至更紧急——MySQL 恢复要小时级,秘钥恢复要分钟级(15 分钟 RTO)。

还有一个容易忘的点:秘钥之间的关联性。数据库密码改了,.env.prod 要更新;微信支付证书续期了,.env.prod 里的证书路径要更新。这些变更必须一起备份,不能只备了新密码没备新证书。我的做法是任何秘钥变更都触发一次完整 .env.prod 备份,而不是只备改了的那一项——保证备份的是一个一致的整体,不是拼凑的碎片。

七、月度恢复演练:这套 SOP 的灵魂

前面六节都是「怎么备份」,这一节是**「怎么证明能恢复」**。这是整套 SOP 的灵魂,也是绝大多数团队缺的一环。

演练每月一次,固定五个步骤,在完全隔离的环境做(新建空 MySQL 实例,不影响生产和从库):

第一步:隔离环境建空实例。新开一台服务器或容器,装空 MySQL,绝不复用任何现网实例。隔离是铁律——演练绝不能污染生产数据。

第二步:全量 dump 恢复。从跨云 OSS 拉最新 dump,mysql < dump.sql.gz 灌进去。这一步验证「dump 文件没损坏、能恢复」。

第三步:binlog PITR 接力。从从库取 binlog 文件 + dump 的 --master-data=2 位点,用 mysqlbinlog 做时间点恢复(PITR)到最新。这一步验证「dump + binlog 能接力,恢复到故障前最后一刻」。

第四步:关键表对账。对核心表(users、orders、circle_post 等)做行数对账,验证恢复出的数据和现网一致或仅在合理延迟范围内:

SELECT COUNT(*) FROM users;
SELECT COUNT(*) FROM orders WHERE created_at >= CURDATE() - INTERVAL 1 DAY;

第五步:输出演练报告 + 门禁判定。报告归档,记录每步耗时、数据一致性结果。演练失败 → 阻断当期发布,直到根因解决并重跑演练通过。

这五步里,第三步 PITR 接力是最容易出问题的——dump 的 binlog 位点和从库实际 binlog 文件对不上、位点漂移、mysqlbinlog 参数用错,都可能让演练卡在这步。正因为容易出问题,才必须月月练。练熟了,真灾难来时才不会手忙脚乱。

演练报告还有一个隐藏价值:它量化了真实 RTO。报告里记录「从拉 dump 到对账通过」总耗时,这个数会告诉你「1 小时 RTO」的承诺到底靠不靠谱。如果实际要 90 分钟,要么优化流程,要么诚实修订 SLA。

八、异常处理:备份失败怎么办

SOP 必须覆盖异常路径,否则出事就抓瞎。几个最常见的故障及应对:

mysqldump 失败:脚本 set -euo pipefail,失败即非零退出,crontab 的 >> logfile 2>&1 捕获日志,飞书 webhook 推送告警。应对:不自动重试(dump 失败往往是数据库本身有问题,重试会掩盖根因),人工介入看日志,通常是权限、连接数、磁盘空间三类问题。

如何快速定位是哪一类?看报错关键字:Access denied 是权限(检查 backup 账号的 GRANT);Too many connections 是连接数(查是不是有长事务占着连接);No space left on device 是磁盘(看 /backup 挂载点余量)。这三类覆盖了 dump 失败的 80% 场景,日志里一搜关键字就知道方向。

从库复制延迟超 5 分钟:飞书告警。应对:先看从库负载(IO、CPU),再看是否有大事务(大表 DDL、批量更新)阻塞复制。持续超阈值说明从库规格不够,要升级或拆分。

判断是不是大事务阻塞,有个技巧:在从库执行 SHOW SLAVE STATUS,看 Seconds_Behind_Master 和 Slave_SQL_Running_State。如果 SQL 线程在 System lock 或 Sending data 状态卡很久,基本就是有个大事务在回放。用 mysqlbinlog 解析当前回放位点的 binlog,能看到具体是哪条 SQL。

OSS 跨云对账偏差 > 1%:告警。应对:先跑一次 rclone check 看是哪些对象缺失。常见原因是同步任务中途失败(网络抖动),重跑通常能补齐。如果持续偏差,要查是不是有对象被业务侧删除但软删除逻辑没生效。

区分「网络抖动漏传」和「真丢数据」:连续两天偏差都在涨,且重跑补不齐,那是真丢(多半软删除逻辑漏了);只是某天抖了一下、重跑就齐,那是网络。持续偏差是危险信号,一次性偏差通常是良性。

演练失败(第三步 PITR 接不上):这是最严重的,直接阻断发布。应对:暂停所有数据库相关变更,排查 binlog 连续性(是否被 purge、位点是否对得上 dump)。修好后必须重跑完整演练通过,才解除发布冻结。

PITR 接不上的根因通常是两类:一是 dump 时的 binlog 位点和从库实际的 binlog 文件对不上(比如 dump 完到演练之间,binlog 被 expire_logs_days=7 清掉了);二是 mysqlbinlog 命令的 --start-position 参数算错。前者靠「演练用新鲜的 dump」避免(别用一周前的 dump 演练),后者靠脚本固化位点计算、不手敲参数。

秘钥备份失败:告警。应对:如果是加密失败(公钥问题),立即修;如果是上传失败,重试。关键:秘钥备份失败期间,任何秘钥变更都要冻结,直到备份恢复——否则改了秘钥却没备份,一旦服务起不来就是绝境。

每一条异常处理都要写进运维手册,不能只靠「出事再想」。SOP 的意义就是把异常路径也前置化。

九、验证标准与告警闭环

整套 SOP 的有效性,靠三层验证持续确认。

第一层:每日备份日报。6:00 的 manifest 脚本汇总当日所有备份任务的状态,飞书推送。这是日常健康检查,异常当天就知道。

第二层:每小时 OSS 对账。跨云对象数偏差监控,1% 阈值告警。这是跨云一致性的持续验证。

第三层:每月恢复演练。这是最硬的验证——前面两层只证明「备份存在且健康」,演练才证明「真的能恢复」。

三层之外还有一条贯穿性的:告警必须有闭环。任何告警触发后,必须在飞书里确认收到、记录处理结果、归档到当月演练报告。告警没人看等于没告警——参考事故即规范的思路,任何一次「告警被忽略导致事故」都要反推告警机制哪里要改。

最后是验收清单(每月演练后逐条核对):MySQL 主库 binlog 已开且格式为 ROW、异地从库延迟 < 5min、每日 mysqldump 产出且校验通过、dump 满足 3-2-1 保留、OSS 版本控制与跨云同步生效、秘钥已 age 加密跨云冗余、飞书告警链路打通、本月演练已通过。任何一条不满足,发布门禁就不放行。

结语

这套 SOP 的内核只有两条:3-2-1 消除单点,月度演练证明可恢复。剩下的 binlog 参数、mysqldump 选项、age 加密、跨云 rclone,都是给这两条服务的。

很多人对备份有侥幸心理,觉得「云厂商承诺 11 个 9 的持久性,我不用操心」。这个侥幸会被两类事件击碎:代码 bug 导致的误删(云厂商不管),和云厂商自身的区域级故障(历史上发生过不止一次)。数据资产的优势不是云厂商的承诺,而是你自己搭的、演练过的恢复能力。

备份这件事,标准要高于现在的需要。3-2-1 听起来重,拆开看每一步都不复杂;月度演练听着费事,跑顺了每次就一小时。这点投入,换的是「无论发生什么,数据都能在 5 分钟内、1 小时恢复」。

最后一件事:今天就检查一下你的备份。mysqldump 在跑吗?dump 文件能恢复吗?你上一次真正演练是什么时候?如果任何一个答案让你犹豫,你需要这套 SOP。别等出事才发现备份是摆设——那时候损失的每一份数据,都是今天偷懒没做的演练的利息。备份的价值,不在「我做了备份」,在「我能证明备份能恢复」。前者是仪式,后者才是保险。
