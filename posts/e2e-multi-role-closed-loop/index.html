你的 E2E 只是在点 UI:真实业务多角色闭环测试 SOP

我审过很多项目的 E2E 测试,十有八九长一个样:一个用户登录,点几个按钮,断言页面出现某个文本,通过。这种测试我称之为「单用户点 UI」——它能抓住「页面崩了」「按钮不见了」这类前端问题,但抓不住最危险的一类 bug:跨角色、跨端、跨时间的业务闭环出错。

举个真实的例子。社交模块里,A 给 B 点赞,B 应该收到一条通知。单用户 E2E 永远测不出这个——因为它只有一个用户,A 点赞后没人去查「B 的通知列表有没有多一条」。这种 bug 会静默潜伏,直到用户投诉才暴露。(这篇和测试纪律、覆盖率诚实是一套:那两篇讲「测试要不要写够」,这篇讲「写出来的测试到底有没有抓住真 bug」。)

我定了一套写 E2E 的标准做法,核心是让多个真实角色互操作,把断言落在数据上,跨端用握手衔接。这套 SOP 在我的多租户 SaaS 项目里跑了大半年,抓出过十几条单用户测试根本碰不到的闭环 bug。这篇按标准 SOP 文档的格式写明白。

一、这套 SOP 要解决什么问题

先划清边界——它只解决一类特定问题。

痛点:传统 E2E 测试以「单用户点 UI」为主,抓不住以下三类 bug:

  • 跨角色业务闭环出错(A 操作 → B 应感知,但 B 没感知)
  • 静默后端失败(前端显示成功,后端数据没落库或落错)
  • 跨端数据传递断裂(小程序下单 → 运营后台查不到订单)

这三类 bug 的共性是:UI 层看着全绿,业务事实是错的。单用户、纯 UI 断言的 E2E 对它们完全失明。

这套 SOP 的目标:用「多真实角色 + 数据落库断言 + 跨端握手」组合拳,让 E2E 能抓住业务闭环级的错误,而不只是页面渲染错误。

适用范围:有明确业务角色划分、且角色间存在状态流转的系统。典型如社交(用户互相关注/互动)、电商(买家下单→卖家发货→平台结算)、SaaS(租户管理员→普通成员→平台运营)。

不适用:纯展示型网站、无状态工具类应用。这类没有业务闭环可言,单用户点 UI 就够了,套这套是过度设计。

二、核心原则:三条铁律

整套 SOP 由三条原则支撑,每条都对应一类传统 E2E 的盲区。

原则一:多真实角色互操作,而非单用户独角戏。

这是和传统 E2E 最大的区别。涉及 A→B 业务流转的测试,必须用两个独立的登录态(两套 token)模拟两个真实用户。A 发起动作,B 感知结果,两边都用真实身份走真实 API。不要用一个用户假装两个角色——那等于自欺欺人,测不出角色间状态隔离和通知投递的问题。

原则二:断言落在持久化数据,而非 UI 表现。

UI 断言(expect(page).toHaveText('成功'))只能证明「前端收到了 200」,证明不了「后端真的把数据写对了」。关键断言必须落在数据库或 API 返回的持久化数据上:通知有没有真的入库、订单状态对不对、聚合数算得准不准。这一条专治「静默后端失败」——前端绿了但后端没落库的情况。

原则三:跨端用握手标记衔接,而非各测各的。

涉及多端的业务(小程序下单 + 运营后台管理),两端不能各写各的 E2E 就完事。必须有一个跨端编排层,用握手标记(如 HANDSHAKE_ORDER_123)把一端产生的数据传递给另一端验证。否则「小程序下的单,运营后台能不能查到」这种跨端一致性就没人保证。

三条原则缺一不可。少了第一条,测不出角色流转;少了第二条,放过静默失败;少了第三条,跨端断裂无人兜底。

三、角色与数据准备:mock-login 模拟真实用户

多角色互操作的前提,是能低成本地拿到多个真实用户的登录态。生产环境不能用真实用户密码登录,我用的是 mock-login 机制(仅 dev/test profile 开放)。

/api/dev/mock-login 接口接收 userId 和 nickname,返回该用户的 token 和完整 user 对象。测试里这样准备两个角色:

const u1 = await api('/api/dev/mock-login', 'POST',
  { userId: 'u_aa186ee1ded84021', nickname: '张三', avatarUrl: 'x' });
const u2 = await api('/api/dev/mock-login', 'POST',
  { userId: 'u_589e86a6dd254e41', nickname: '李四', avatarUrl: 'x' });
const t1 = u1.data?.token, t2 = u2.data?.token;  // 两个独立 token
const id1 = u1.data?.user?.id, id2 = u2.data?.user?.id;

关键点:

  • userId 必须是数据库里真实存在的用户 ID,不是随便编的。测试前用 Flyway seed 或造数脚本确保这两个用户存在。
  • 两个 token 完全独立,分别代表两个会话,任何状态隔离问题都会暴露。
  • 租户 ID 必须与项目配置一致。我踩过的坑:测试里硬编码 X-Tenant-Id: "1",但项目默认租户是 t_eb78751bae84,后端返回 “Tenant not found”,所有登录失败。租户 ID 永远从项目配置取,不要硬编码。

数据准备这一步看似简单,却是整套 SOP 的地基。角色造不好,后面的闭环全是假的。

还有个要点:两个用户最好处于同一个租户下。多租户系统里,跨租户的操作通常是被隔离的——A 租户的用户给 B 租户的内容点赞,应该被拒绝。如果你要测「跨租户隔离」,那是另一类测试(应该断言操作被拒绝);但如果要测正常的通知闭环,两个用户必须在同一租户内。这个细节搞错,会出现「明明操作了却收不到通知」的假 bug,排查很久才发现是租户不对。

造数脚本要可重复运行。每次跑 E2E 前,可能需要重置这两个用户的状态(清掉旧通知、清掉测试产生的内容)。我习惯给每个测试用户配一个「清洗脚本」,setup 阶段先跑清洗,保证每次测试的起始状态干净。否则上一次测试残留的通知,会污染这次的通知列表断言。

还有个多角色特有的数据问题:两个用户之间的关联数据要一起准备。比如要测「B 评论 A 的帖子后 A 收到通知」,那 A 名下得先有一篇帖子存在,B 才有东西可评论。如果只造了两个用户没造帖子,评论接口会报「帖子不存在」,整个闭环跑不起来。造数要顺着业务链路走:先造接收方的内容,再造发起方的动作,顺序反了就缺数据。这个顺序最好固化在 setup 脚本里,不要靠人记。

四、业务闭环编排:让角色真的互动起来

角色准备好了,接下来编排真实的业务闭环。以社交互动通知为例,完整的闭环是:用户2(动作发起方)对用户1的内容做互动 → 用户1(感知方)查通知列表 → 断言通知真的入库。

闭环里的每个 case,都是「发起方用 t2 调业务 API → 感知方用 t1 查结果」:

// CASE 1: 李四(用户2)点赞张三(用户1)的内容 → 张三应收到通知
const r1 = await api('/api/tenant-core/notifications/send', 'POST',
  { templateCode: 'INTERACT_LIKE',
    userId: id1,                          // 通知接收方 = 用户1
    variables: { userName: '李四' },
    jumpUrl: '/pages/circle/post-detail?postId=1' }, t2);  // 用用户2的 token

// CASE 2: 李四 评论张三 → 通知
// CASE 3: 李四 关注张三 → 通知
// CASE 4: 李四 发帖@张三 → 通知

注意几个设计要点:

  • 每个 case 覆盖一种通知类型(点赞 INTERACT_LIKE、评论 INTERACT_COMMENT、关注 INTERACT_FOLLOW、@提及 POST_MENTION)。不要只测一种就声称「通知功能 OK」——不同 templateCode 走不同的渲染和聚合逻辑,漏一种就漏一类 bug。
  • 发起方和接收方明确区分。userId: id1(接收方)配 t2(发起方 token),这种「错位」正是多角色测试的精髓——它验证了「A 的操作能正确投递给 B」。
  • 聚合场景单独测。连续发 3 条点赞,验证通知是否聚合(「等 3 人点赞了你」而非 3 条独立通知)。聚合是通知系统最容易出 bug 的地方:

聚合为什么容易出 bug?因为它涉及时序和窗口。通知系统通常用「时间窗口 + 相同类型」判断是否合并:窗口内同类型通知聚成一条,窗口外的独立成条。这个逻辑的边界条件极多——窗口多长?跨窗口的怎么算?聚合后点击跳转去哪条内容?单用户点 UI 永远测不到这些,只有连续触发多条才能撞上。

我测聚合时有个固定套路:连续发 3 条同类通知,中间间隔 200ms,然后查列表。断言两件事:一是列表里应该出现 aggregateCount >= 3 的聚合条目;二是这个聚合条目的标题应该是「等 3 人点赞了你」这种聚合文案,而不是 3 条独立通知。任何一个不对,聚合逻辑就有问题。这个测试抓出过一次真实的聚合窗口配置错误——窗口被配成了 0,导致所有通知都不聚合,用户被通知列表刷屏。

// CASE 5: 多次点赞 → 测试聚合
for (let i = 0; i < 3; i++) {
  await api('/api/tenant-core/notifications/send', 'POST',
    { templateCode: 'INTERACT_LIKE', userId: id1,
      variables: { userName: ['王五', '赵六', '钱七'][i] },
      jumpUrl: '/pages/circle/post-detail?postId=1' }, t2);
  await new Promise(r => setTimeout(r, 200));   // 留时间让聚合生效
}

五、断言落库:抓住静默后端失败

这是整套 SOP 最关键的一节。前面所有「发送通知」的 case,断言都不能只看「发送接口返回 SUCCESS」——那只证明接口没报错,证明不了通知真的进了库、真的能被用户查到。

真正断言,落在「接收方查自己的通知列表」上:

// 用户1 用自己的 token 查通知列表
const list = await api('/api/tenant-core/notifications/in-app?page=1&size=20', 'GET', null, t1);
const items = list.data?.data?.list || [];
const total = list.data?.data?.total || items.length;

// 断言一:列表非空(证明通知真的入库了)
c('消息列表非空', total > 0, `${total} 条`);

// 断言二:包含互动通知(证明点赞/评论/关注都投递成功)
c('包含互动通知', items.some(n => n.category === 'INTERACTION'));

// 断言三:包含@提醒(证明 POST_MENTION 通路正常)
c('包含@提醒', items.some(n => n.title?.includes('@') || n.category === 'INTERACTION'));

这三个断言的意义:

  • 断言一抓「通知根本没入库」的静默失败。如果发送接口返回 SUCCESS 但实际没写库,这里 total 会是 0。
  • 断言二和断言三抓「部分类型通知丢失」。比如点赞通知正常但 @提醒丢失,说明 POST_MENTION 这条链路断了——这种部分失败最隐蔽。
  • 分类统计逐条打印,方便排查哪类通知出了问题:
for (const n of items) {
  console.log(`id=${n.id} [${n.category}] ${n.title?.slice(0,30)} | ${n.isRead?'已读':'未读'} | agg=${n.aggregateCount||1}`);
}

核心原则:断言必须穿透到持久化数据层。UI 断言能放过「接口返回 200 但数据没落对」的 bug,数据层断言不会。这是我反复强调的一点——E2E 的价值不在于证明页面能点,而在于证明业务事实成立。

什么叫「静默后端失败」?举一个我真实遇到的例子。评论功能上线后,前端表现完全正常——用户发评论,页面立刻显示新评论,看着没问题。但实际上后端的 MQ 消费者在某次部署后挂了,评论根本没写进数据库,前端显示的是乐观更新的假象。等用户刷新页面,评论全没了。

这种 bug 用「点 UI 看 Toast 提示」的 E2E 永远抓不到——因为接口确实返回了 200(乐观更新不需要等后端)。但如果 E2E 在「发评论」后,直接查数据库 SELECT * FROM comment WHERE post_id = X ORDER BY created_at DESC LIMIT 1,断言这条评论真的在库里,就能当场抓住。

判断哪些断言该落库有个原则:凡是涉及状态变更的业务(发帖、下单、支付、评论),关键断言都要落库。纯查询类(列表展示、搜索)可以只断言 UI,因为它们不改数据。「这个操作到底改没改数据」是分界线——改数据的,必须验数据。

六、跨端握手:多端业务怎么串联

前面的例子是单端(都走 HTTP API)的多角色闭环。但很多业务是跨端的:用户在小程序下单,运营在后台管理,平台在服务端结算。这种跨端业务,各端各测各的会漏掉「端与端之间的数据断裂」。

我的做法是引入一个跨端编排层 + 握手标记。以交易闭环为例:

编排层把流程切成阶段,每阶段在一个端跑,阶段间用握手标记传递数据:

// 阶段 1:小程序端(automator)下单,产出订单号
const orderResult = await runMpStage('place-order');
const orderId = orderResult.orderId;

// 握手:订单号写入跨端桥接文件
await bridge.set('HANDSHAKE_ORDER_' + orderId, { stage: 'placed', orderId });

// 阶段 2:运营后台端(Playwright)读取握手,验证订单可见
await bridge.wait('HANDSHAKE_ORDER_' + orderId, 'placed');
const opsResult = await runOpsStage('verify-order', { orderId });
// 断言:运营后台确实能查到这个订单
assert(opsResult.found === true, '运营后台应能查到小程序下的单');

// 阶段 3:服务端验证结算状态
await bridge.wait('HANDSHAKE_ORDER_' + orderId, 'ops-verified');
const settleResult = await runBackendCheck(orderId);
assert(settleResult.status === 'SETTLED', '订单应进入结算');

握手标记的设计要点:

  • 标记是业务标识(订单号、退款 ID),不是技术标识。这样即使两端实现不同,业务语义是对齐的。
  • 桥接层有等待语义(bridge.wait),解决两端时序问题——运营后台等小程序把订单写进桥接,而不是假设下单一定先完成。
  • 每阶段独立可重跑。如果阶段 2 失败,可以从阶段 2 重跑,不用从头下单。这对调试效率很重要。

跨端握手的核心价值:它把「跨端一致性」从没人管的灰色地带,变成了一个有明确断言的测试用例。「小程序下的单运营后台查不到」这种 bug,只有在跨端编排里才会被抓到。

七、diff 驱动的增量选测:全量太慢怎么办

业务闭环 E2E 比单用户 E2E 重得多(要造多角色、跑多端、等聚合)。全量跑一次可能十几分钟,不适合每次提交都跑。这就引出一个问题:怎么知道这次改动该跑哪些 E2E?

答案是用 Flow Registry 做 diff 驱动的增量选测。核心是建立一个「文件路径 → 业务域 → E2E 用例」的映射注册表:

# e2e-flow-registry.yaml(节选)
domains:
  notification:
    impactedPaths:
      - "backend/src/main/java/**/notification/**"
      - "backend/src/main/java/**/NotificationController.java"
      - "frontend/src/pages/circle/**"
    e2eTests:
      - "test_e2e_real_business.mjs"      # 多角色通知闭环
      - "test_h5_notif.mjs"               # H5 通知投递
  trade:
    impactedPaths:
      - "backend/src/main/java/**/order/**"
      - "frontend/src/pages/order/**"
      - "operation-backend/src/views/trade/**"
    e2eTests:
      - "run-p0-trade.mjs"                # 交易 P0 闭环
      - "trade-p0-cross-e2e.mjs"          # 跨端交易

提交时,脚本扫 git diff,看改了哪些文件,匹配到对应业务域,再查该域有哪些 E2E 用例,只跑受影响的那些。这样一次小改动不会触发全量 E2E,只跑相关的几条。

三级门禁配合:

  • commit 时:只跑 diff 命中的 E2E(快,几秒到几十秒)
  • PR 时:跑该业务域全量 E2E(中等,几分钟)
  • release 时:跑所有 E2E(慢,但发版频率低)

这套机制解决了一个经典矛盾:E2E 要全才有保障,但全量跑太慢。diff 驱动选测让你日常只跑必要的,在关键节点(发版)才付全量的成本。保障没降,时间省了。

Flow Registry 的设计有个关键点:pathMapping 要覆盖「这个业务域涉及的所有层」。比如交易域,既要包含后端的 order/**,也要包含前端的 pages/order/**,还要包含运营后台的 views/trade/**。因为一个订单功能的改动,可能从任何一层引入 bug——后端改了订单状态机、前端改了下单页交互、运营改了订单管理页,任何一个都可能破坏交易闭环。mapping 漏了一层,那一层的改动就不会触发交易 E2E,等于这个门禁有了缺口。

维护 Flow Registry 的投入是一次性的:花半天把所有业务域和它们涉及的路径梳理清楚,写进 yaml。之后新增功能时,顺手在对应域的 impactedPaths 里补上新文件路径。这个维护成本很低,但换来的「精准选测」收益极高——我项目里 30 多个 E2E 用例,日常提交只跑命中的 2-3 条,从十几分钟降到几十秒,而保障没降。

还有个反直觉的点:注册表宁可多映射,不可漏映射。多映射(改了无关文件也跑相关 E2E)只是多花点时间;漏映射(改了相关文件却没跑 E2E)是放过 bug。成本不对称,所以阈值要偏向保守——拿不准某个路径属不属于某业务域时,就归进去。

八、异常处理:闭环测试失败怎么排查

多角色闭环 E2E 失败时,排查比单用户复杂——因为涉及多个角色、多个端、多个时间点。SOP 必须定义排查顺序,否则会乱撞。

排查铁律:Console → Network → 后端日志,按顺序。

第一步看 Console(浏览器或小程序开发者工具的 JS 报错)。第二步看 Network(失败的 HTTP 请求、状态码、响应体里的 requestId)。第三步才看后端日志。禁止跳过前两步直接改测试或改业务代码——那是糊弄过关。

多角色特有的排查点:

  • 通知没收到:先确认发起方(t2)的请求是否真的成功(看 status 字段,不是 200 而是 data.status === 'SUCCESS')。发起方失败了,接收方当然收不到。再看接收方(t1)查列表的 token 对不对——用错 token 查别人的列表,永远查不到自己的通知。
  • 聚合数不对:通常是时序问题。连续发通知太快,聚合任务还没跑,查出来是分散的。解决是 await sleep(200) 给聚合留时间,或在断言前主动触发一次聚合。
  • 跨端断裂:先看握手标记有没有写进桥接层。小程序端下单成功但没写握手,运营端会一直 wait 超时。这类问题多半是阶段间的「写握手」逻辑漏了。

环境类失败要分流处理。后端没起、数据库连不上、微信 CLI 没装——这些是环境问题,不是代码问题。禁止用改测试代码的方式绕过环境问题(比如 skip 掉依赖 8080 的用例)。环境没就绪时,正解是先把环境弄好,而不是让测试假装通过。这条和测试纪律是一脉相承的——禁止假绿。

分流「代码 bug」和「环境问题」有个判断标准:同样的测试在别人机器上能过,在你这不过,大概率是环境;所有人都不过,那是代码 bug。所以排查时先问自己「这个失败是稳定的还是偶发的」。稳定的、所有人都复现的,直接按代码 bug 排(走 Console→Network→后端日志);偶发的、换个环境就好的,先查环境(服务起了没、依赖装了没、端口占没占)。

跨端测试还有一个特有的坑:两端时序竞争。小程序端下单还没完成,运营端就去查订单,自然查不到。这不是 bug,是编排问题。解决办法就是前面说的 bridge.wait——运营端必须等到小程序端写出握手标记才开始查。任何跨端测试,都要显式处理两端时序,不能假设 A 一定先于 B 完成。

九、验证标准:怎么判断闭环 E2E 写对了

一套 E2E 是不是真的「闭环」,有几条硬标准可以自查。

标准一:是否用了至少两个独立登录态。如果整条链路只有一个 token,那是单用户测试,不是闭环。扫一遍测试代码,确认发起方和感知方用的是不同的 token。

标准二:关键断言是否落在数据层。数一下断言,如果有超过一半是 expect(page).toHaveText 这种 UI 断言,而数据库/API 层断言很少,说明还停留在「点 UI」阶段。健康的比例是数据层断言占 60% 以上。

标准三:是否覆盖了反向验证。不只测「A 操作后 B 收到了」,还要测「A 没操作时 B 没收到」——验证不会串通知。反向验证是防止「通知系统乱投递」的闸。

标准四:跨端链路是否有握手。凡涉及多端的,必须有显式的握手衔接,不能是「小程序测完、运营后台测完,默认它俩能对上」。

标准五:是否纳入 Flow Registry。闭环 E2E 必须注册到 diff 映射表,否则它不会在相关改动时被自动触发,等于白写。

这五条标准也是 code review E2E 时的检查清单。任何一条不满足,要求作者补齐再合入。

结语

这套 SOP 的内核是一句话:E2E 要证明业务事实成立,而不只是页面能点。多角色、数据断言、跨端握手、diff 选测,分别堵的是「角色流转盲区」「静默后端失败」「跨端断裂」「全量太慢」这四个窟窿。

很多人对 E2E 的理解停留在「模拟用户点页面」。这种理解能抓到「按钮样式错了」「页面崩了」这类前端 bug,但抓不住最贵的业务 bug——用户点了点赞对方收不到通知、小程序下了单运营后台查不到。前者用户一眼能看到会反馈,后者静默潜伏,等发现时往往已经影响一批人。

同样写一条 E2E,测业务闭环的捕捉价值是测 UI 的十倍。闭环 bug 更隐蔽、影响面更大、排查成本更高。写闭环 E2E 前期成本确实高(要造多角色、编排流程、设计握手),但它抓住的每一条 bug,单用户测试都碰不到。

落地不用一步到位。先把断言落库这一步做到位——成本最低、见效最快,再逐步加上多角色和跨端流程。下次写 E2E 前,先问自己:这条用例,能不能抓住「后端没落库但前端假装成功」的 bug?抓不住,就不算真正的端到端。E2E 的「端到端」,指的是业务事实的端到端——从用户操作到数据落地,中间任何一段断了我都得知道,这才对得起「端到端」这三个字的分量。只测 UI 的「端到端」,其实只是「前端到前端」,后端那段最贵的、最容易静默失败的业务逻辑,根本没被覆盖到。把断言落进数据库,才算是真把后端那段补上了。
