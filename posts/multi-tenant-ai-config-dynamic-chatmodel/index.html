多租户 AI 助手里最难的一块,不是 RAG、不是 Agent,是让每个租户用自己的大模型。这篇专门讲怎么用"动态 ChatModel 工厂"把这件事做扎实——从表设计、加密脱敏,到工厂的缓存与失效,再到为什么不用 Spring AI 的 starter 自动装配。

一、问题:为什么不能写死一个全局 ChatModel

随便找一个 Spring AI 的入门 demo,大概率是这样:在 application.yml 里配一个 spring.ai.openai.api-key,启动时由 starter 自动装配出一个全局 ChatModel bean,业务里 @Autowired 直接用。单租户、单 Key、单模型,这样没问题。

但我们的平台是多租户的。同一个 AI 助手,服务几十个租户,每个租户:

  • 用自己的供应商(A 用 DeepSeek、B 用通义千问、C 用智谱 GLM);
  • 用自己的 API Key(各自的账单、各自的额度、各自的限流);
  • 用自己的模型参数(温度、max_tokens、具体模型型号都不一样);
  • 在运营后台随时改这些配置,改完要立刻生效。

一个全局 ChatModel bean 在第一点上就垮了——它的 apiKey 在启动时定死,运行时拿不到租户上下文。这就是多租户 AI 和单租户 AI 最本质的区别:模型实例必须按租户动态创建、缓存、失效。

方案能否多租户配置生效复杂度
全局单 ChatModel bean(starter 装配)不能(只有一个 Key)改配置要重启低
每次请求都 new 一个模型能实时极高(每次建连,性能不可接受)
动态 ChatModel 工厂 + 缓存 + 失效(本文)能配置变更 evict 后实时生效中

结论很清楚:需要在两者之间取一个平衡点——按租户懒加载,建好后缓存复用,配置变了再失效重建。这就是下面整个设计的起点,也是多租户 AI 和单租户 AI 最根本的区别所在。

二、整体设计:配置表 + 服务层 + 工厂三层

先把全貌画出来。一次"租户发起对话 → 取到属于他的 ChatModel"的完整链路:

这条链路里有三个关键动作:① 工厂用 computeIfAbsent 保证同租户只建一次;② 新建时走"查配置 → 解密 Key → 供应商兜底 → 构建"四步;③ 后台改了配置,调 evict 清掉对应租户的缓存,下次请求自然重建。

整套设计分成三层,类关系如下:

这里有个关键的职责拆分:TenantAiConfigService 负责"配置怎么存、怎么脱敏、怎么解密";TenantChatModelFactory 只负责"拿到配置后怎么建模型、怎么缓存"。前者是数据安全层,后者是运行时层,互不耦合。下面逐层展开。

三、配置存哪:tenant_ai_config 表

租户的模型配置存一张表,字段就是工厂构建模型需要的全部参数,再加治理字段:

CREATE TABLE tenant_ai_config (
  tenant_id            VARCHAR PRIMARY KEY,           -- 一租户一行
  provider             VARCHAR(32) DEFAULT 'deepseek', -- deepseek|glm|qwen
  base_url             VARCHAR,                        -- 留空则用供应商默认
  model                VARCHAR(64) DEFAULT 'deepseek-v4-flash',
  api_key_encrypted    VARCHAR(512),                   -- AES 加密后落库
  temperature          DECIMAL(3,2) DEFAULT 0.7,
  max_tokens           INT DEFAULT 4096,
  enabled              TINYINT DEFAULT 0,              -- 0=未启用 1=启用
  rate_limit_rpm       INT DEFAULT 30,                 -- 每分钟限流
  monthly_token_limit  BIGINT DEFAULT 10000000         -- 月度 token 预算
);

这张表的设计哲学是:一租户一行,字段即参数。provider 决定走哪家供应商,base_url/model/api_key_encrypted/temperature/max_tokens 是构建 OpenAiChatModel 的直接入参,enabled/rate_limit_rpm/monthly_token_limit 是治理字段(限流、预算用)。主键直接是 tenant_id,查询就是按主键查,O(1)。

这张表还有一个设计决策值得说:为什么不用 JSON 字段存配置,而用独立列。有些方案把整个配置(供应商、模型、Key、参数)塞进一个 JSON 列,灵活性高(加字段不用改表)。但我们选了独立列,理由是:独立列可以单独索引(比如按 enabled 筛选启用的租户),JSON 列做不到;独立列的变更更容易审计(改了 temperature 就是一条列级变更,JSON 列改了看不出改了哪个字段);独立列的类型检查更严格(temperature 是 DECIMAL,传错了类型直接报错,JSON 列什么都收)。灵活性换严谨,在配置这种「结构稳定、字段不多」的场景下,独立列比 JSON 更合适。JSON 列适合「结构经常变、字段不固定」的场景,配置表不属于这类。

四、配置服务:加密、脱敏、双视图

TenantAiConfigService 是数据安全的核心。它对同一份配置提供两个视图——这是整套安全设计的关键。

4.1 给前端:脱敏返回

运营在后台看自己的配置时,API Key 永远不能是明文。getConfig 返回时过一道 SensitiveMasker:

public TenantAiConfigVO getConfig(String tenantId) {
    TenantAiConfig raw = configMapper.selectByTenantId(tenantId);
    if (raw == null) return null;
    TenantAiConfigVO vo = TenantAiConfigVO.from(raw);
    vo.setApiKey(SensitiveMasker.mask(raw.getApiKeyEncrypted())); // wx12****cd
    return vo;
}

前端拿到的 Key 永远是 sk-a1b2****x9y8 这种,看不到全貌。

4.2 给工厂:不脱敏,另走解密

工厂构建模型时需要真正明文 Key,所以单独提供一个不脱敏的 getRawConfig 和一个专门解密 Key 的 getDecryptedApiKey:

// 给工厂用,不脱敏
public TenantAiConfig getRawConfig(String tenantId) {
    return configMapper.selectByTenantId(tenantId);
}

// 解密 API Key,只在工厂内部调用
public String getDecryptedApiKey(String tenantId) {
    TenantAiConfig raw = configMapper.selectByTenantId(tenantId);
    return aesEncryptor.decrypt(raw.getApiKeyEncrypted());
}

把"读配置"和"解密 Key"拆成两个方法,而不是在 getRawConfig 里顺手把 Key 解密了,是有意为之——解密是个高危操作,要让它单独成方法、单独被审计。任何调用 getDecryptedApiKey 的地方都心知肚明自己在碰明文 Key。

这个拆分还有个连带好处:降低明文 Key 的传播范围。如果 getRawConfig 返回的配置对象里就包含明文 Key,那所有调 getRawConfig 的地方(可能不止工厂,还有配置展示、审计等)都接触到了明文 Key,攻击面变大。拆开后,只有调 getDecryptedApiKey 的工厂才碰到明文,其他地方用 getRawConfig 只拿到加密后的 Key,攻击面缩到了最小。安全设计里「缩小敏感数据的传播范围」是基本原则——接触明文的人(代码路径)越少,泄漏的风险越低。

4.3 写入:加密落库 + 脱敏占位符跳过

写入有两条规则:① 落库前必加密;② 更新时如果前端把脱敏占位符原样传回,跳过覆盖,避免把 **** 当真值存进去。

public void saveConfig(TenantAiConfigDTO dto) {
    TenantAiConfig existing = configMapper.selectByTenantId(dto.getTenantId());
    String key = dto.getApiKey();
    if (existing != null && SensitiveMasker.isMaskedPlaceholder(key)) {
        // 前端没改 Key,把 **** 原样传回了,保留旧值
        key = existing.getApiKeyEncrypted();
    } else {
        key = aesEncryptor.encrypt(key); // 新 Key,加密落库
    }
    dto.setApiKey(key);
    configMapper.upsert(dto);
}

这个 isMaskedPlaceholder 判断很关键。前端的表单逻辑是:加载配置时显示 wx12****cd,用户如果没动这个框,提交时原样把 wx12****cd 传回来。服务端必须识别出"这是个脱敏占位符,不是真 Key",否则就会把 **** 加密后存进库,这个租户的模型当场报废。

对应的 Controller 很薄,GET 脱敏返回,PUT 加密写入:

@GetMapping("/{tenantId}")
public TenantAiConfigVO get(@PathVariable String tenantId) {
    return tenantAiConfigService.getConfig(tenantId); // 脱敏
}

@PutMapping("/{tenantId}")
public void update(@PathVariable String tenantId, @RequestBody TenantAiConfigDTO dto) {
    dto.setTenantId(tenantId);
    tenantAiConfigService.saveConfig(dto); // 加密写入
}
接口入口Key 处理
GET(给前端)getConfigSensitiveMasker 脱敏,返回 wx12****cd
PUT(给前端)saveConfig识别占位符跳过 / 否则 AesEncryptor 加密落库
getRawConfig(给工厂)内部不脱敏,但不解密 Key
getDecryptedApiKey(给工厂)内部AesEncryptor 解密出明文

五、动态 ChatModel 工厂(核心)

这是整套架构最值钱的部分。先看全貌,再拆细节。

5.1 多供应商:三家走一套 OpenAI 协议

工厂能构建不同供应商的模型,前提是"统一协议"。我对接的 DeepSeek、智谱 GLM、通义千问三家都兼容 OpenAI 协议,所以没有为任何一家写 SDK,只用 Spring AI 的 OpenAiChatModel,靠 baseUrl + apiKey + model 三参数区分。各家默认值写在一个 map 里:

// SpringAiConfig.java —— 三家供应商的默认值
private static final Map<String, ProviderDefaults> DEFAULTS = Map.of(
    "deepseek", new ProviderDefaults("https://api.deepseek.com",                           "deepseek-v4-flash"),
    "glm",      new ProviderDefaults("https://open.bigmodel.cn/api/paas",                  "glm-4-flash"),
    "qwen",     new ProviderDefaults("https://dashscope.aliyuncs.com/compatible-mode/v1",  "qwen-plus")
);

加一家新供应商,就是在这个 map 里加一行,工厂代码一行不动。

这个设计的长期价值在于:供应商可热插拔。今天接了 DeepSeek,明天要接百度文心一言,不用改工厂代码、不用改业务代码,只要在 DEFAULTS map 里加一行 "wenxin", new ProviderDefaults(...) ,租户在后台选 provider = "wenxin" 就能用了。整个接入过程不需要发版(如果 DEFAULTS 做成配置中心可配的话),甚至不需要重启。这种「加供应商 = 加配置」的扩展性,是统一协议带来的最大红利——如果每家写了独立 SDK,加一家就是一次开发+测试+发版的完整流程,周期以周计;统一协议下,加一家是分钟级的配置变更。

5.2 工厂:ConcurrentHashMap + 懒加载 + evict

工厂内部用一个 ConcurrentHashMap<String, ChatModel> 按 tenantId 缓存模型实例。三个核心方法:

public class TenantChatModelFactory {
    private final ConcurrentHashMap<String, ChatModel> cache = new ConcurrentHashMap<>();

    public ChatModel getOrCreate(String tenantId) {
        return cache.computeIfAbsent(tenantId, this::createForTenant);
    }

    public void evict(String tenantId) {
        cache.remove(tenantId);
    }

    private ChatModel createForTenant(String tenantId) { /* 见 5.3 */ }
}

两个设计点要讲清楚:

  • 懒加载(computeIfAbsent):第一次用到某租户才建模型,建完缓存复用。computeIfAbsent 是原子操作,同一租户并发请求只会建一次,不会重复建。
  • evict 失效:租户在后台改了模型或 Key,调一次 evict(tenantId) 清掉这个租户的缓存,下次请求自然走 createForTenant 重建。配置变更实时生效,不用重启。

5.3 createForTenant:查配置 → 兜底 → 解密 → 构建

createForTenant 是真正"造模型"的地方,四步走:

private ChatModel createForTenant(String tenantId) {
    // 1. 查租户配置(不脱敏的原始配置)
    TenantAiConfig config = tenantAiConfigService.getRawConfig(tenantId);

    // 2. 供应商默认值兜底:租户没配的字段用供应商默认值补上
    String provider = tenantAiConfigService.getProvider(tenantId); // 默认 deepseek
    ProviderDefaults defaults = DEFAULTS.getOrDefault(provider, DEFAULTS.get("deepseek"));
    String baseUrl = StringUtils.defaultIfBlank(config.getBaseUrl(), defaults.baseUrl());
    String model   = StringUtils.defaultIfBlank(config.getModel(),   defaults.model());

    // 3. 解密 API Key(明文,只在这里出现)
    String apiKey = tenantAiConfigService.getDecryptedApiKey(tenantId);

    // 4. 构建 OpenAiChatModel
    OpenAiApi api = OpenAiApi.builder().apiKey(apiKey).baseUrl(baseUrl).build();
    OpenAiChatOptions options = OpenAiChatOptions.builder()
        .model(model)
        .temperature(config.getTemperature().doubleValue())
        .maxTokens(config.getMaxTokens())
        .build();
    return OpenAiChatModel.builder().openAiApi(api).defaultOptions(options).build();
}

注意第 2 步的兜底逻辑:provider 用 getProvider(默认 deepseek),baseUrl/model 用 StringUtils.defaultIfBlank——租户表里这两个字段留空,就取供应商默认值。这意味着租户最简配置只要填一个 api_key_encrypted,其他全走默认,新租户接入门槛极低。

租户隔离落到模型实例层——不是靠 prompt 提醒模型"你是哪个租户",是每个租户有自己独立的、带自己 Key 和参数的 ChatModel 实例。这是硬隔离:模型实例里物理上绑死了这个租户的 Key,不存在串租户的可能。

这种「实例级隔离」比「prompt 级隔离」可靠得多。prompt 级隔离是「同一个模型实例,在 prompt 里告诉它当前是哪个租户」——模型可能忘、可能理解错、可能被 prompt 注入绕过,隔离是「软」的。实例级隔离是「不同的租户用不同的模型实例」,实例的 Key 物理绑定,模型连「别的租户存在」都不知道,隔离是「硬」的。多租户 AI 系统的隔离必须是硬的——数据隔离靠 metadata filter(硬),模型隔离靠实例分离(硬),两层都硬,才敢说「不会串租户」。

六、关键决策:为什么不用 Spring AI 的 starter 自动装配

这里有个反直觉的取舍:我没有用 Spring AI 的 starter 自动装配(spring-ai-starter-model-openai),而是手动 OpenAiChatModel.builder()。

原因是 starter 的前提是 application.yml 里配一个全局 api-key:

# starter 自动装配期望的配置 —— 全局 key
spring:
  ai:
    openai:
      api-key: ${OPENAI_API_KEY}   # 一个值,启动时定死

而我们的 Key 是按租户动态来的,全局 Key 这个前提就不成立。手动构建换来多租户的灵活度——什么时候建、用哪个 Key、用哪个模型,全在工厂里按租户说了算。

方案优点缺点是否支持多租户
starter 自动装配零配置,开箱即用只能一个全局 Key不支持
手动 OpenAiChatModel.builder()(本文)每个租户独立 Key/模型/参数多写一层工厂支持

平台级倒是有一份 AssistantProperties(@ConfigurationProperties("app.assistant")),定义平台级的默认 provider/model/apiKey/baseUrl/timeout/temperature/maxTokens——但它的角色是平台兜底和未接入租户的默认通道,不是多租户的运行时入口。租户一旦在自己的 tenant_ai_config 里配了,就以租户配置为准。

@ConfigurationProperties("app.assistant")
public class AssistantProperties {
    private String provider = "deepseek";
    private String model    = "deepseek-v4-flash";
    private String apiKey;
    private String baseUrl;
    private int timeout     = 60;
    private double temperature = 0.7;
    private int maxTokens   = 4096;
}

平台开箱能用,租户想定制再覆盖,这就是"租户配置优先,平台默认兜底“的分层。

七、一次配置变更的完整时序

文字讲完,看一次"运营在后台改了租户的模型,配置如何实时生效"的时序:

关键点:saveConfig 写完库后主动调一次 evict,把工厂里这个租户的旧实例清掉。下一次请求来,getOrCreate 发现缓存空了,走 createForTenant 用新配置重建。整个过程不重启、不重新部署,配置秒级生效。

这个「写库 + evict」的顺序有个要注意的点:必须先写库再 evict,不能反过来。如果先 evict 再写库,evict 之后、写库之前的窗口期,如果有请求进来,getOrCreate 会用旧配置(库里还没改)重建一个旧实例,这个旧实例又被缓存了,然后写库完成——结果是「库是新的,缓存是旧的(刚重建的旧实例)」,evict 白做了。先写库再 evict,保证 evict 时库已经是新的,重建时读到的就是新配置。顺序错了,配置变更就「有时生效有时不生效」,这种间歇性 bug 排查起来极痛苦。

evict 还有个边界:evict 之后正在进行的请求怎么办。某个租户的请求正在用旧实例生成回答(流式输出进行中),这时候 evict 了,正在进行的请求不受影响(它已经拿到了旧实例的引用,evict 只是清缓存不影响已持有的引用),但下一个请求会用新实例。这个行为是合理的——正在进行的请求让它完成(用户体验上不打断),新请求走新配置。不需要为 evict 做请求中断,那样反而引入复杂性。

八、几条沉淀下来的判断

做完这一层,我对"多租户 AI 配置"的判断:

  1. 多租户 AI 的本质难点是"模型实例动态化”,不是 prompt 工程。一个全局 bean 解决不了,必须在运行时按租户建模型。
  2. 动态 ChatModel 工厂三件套缺一不可:ConcurrentHashMap 缓存保证性能、computeIfAbsent 懒加载保证按需、evict 失效保证配置实时生效。
  3. 租户隔离落到模型实例层才是硬隔离——每个租户独立的 ChatModel,Key 物理绑定,不存在串租户。
  4. 别为每家模型写 SDK,国内外主流模型大多兼容 OpenAI 协议,统一走它,新增供应商就是加一行默认值。
  5. 配置安全三件套:落库 AesEncryptor 加密、出库 SensitiveMasker 脱敏、更新识别 **** 占位符跳过——把"明文 Key 只在解密那一刻出现"做到位。
  6. 不用 starter 自动装配,手动 OpenAiChatModel.builder() 换多租户的灵活度,这笔交易划算。
  7. 租户配置优先、平台默认兜底,新租户填一个 Key 就能跑,零配置启动。

把"模型实例动态化"这层做扎实,多租户 AI 才能真正按租户独立运行——以上层的 RAG、Agent、流式所有能力为前提,没有这层地基,上面的能力都是空中楼阁,搭不起来也跑不稳。

深度展开:动态 ChatModel 工厂的进阶设计

动态 ChatModel 工厂解决的问题是:不同租户可能用不同的模型 / 不同的供应商,代码不能写死。基础设计(配置表 + 缓存 + evict)前面讲完了,这里展开几个进阶设计。

第一个进阶是 fallback 机制。主模型挂了(供应商服务不可用、API 超时、Key 失效),如果不处理,这个租户的 AI 助手就完全不可用。我们的做法是在配置表里预留 fallback_provider 和 fallback_model 字段——主模型调用失败时,工厂自动用 fallback 配置建一个备用模型实例,让对话继续。fallback 不追求质量最优(备用模型可能不如主模型好),追求的是「可用性不断」——用户感知到回答质量下降了,但至少还能用,比白屏强。fallback 的触发条件也要控制:不是每次超时都切(可能是偶发网络抖动),而是连续失败 N 次才切,避免频繁切换导致体验抖动,用户体验也更平稳。

第二个进阶是模型实例的健康检查。缓存的 ChatModel 实例可能「建的时候是好的,后来坏了」——比如供应商换了 API 版本、Key 被租户在供应商后台撤销了。缓存的实例还在,但调用会失败。我们的做法是给缓存实例加「上次成功调用时间」标记,如果一个实例超过 24 小时没成功调用过,下次 getOrCreate 时主动 evict 重建——宁可重建一次,也不用一个可能坏了的旧实例。这个「定期淘汰」比「一直缓存」更可靠,代价是偶尔多建一次模型,成本极低,完全可接受。

第三个进阶是配置变更的广播。多实例部署时,租户在后台改了配置,只有处理这个请求的实例 evict 了缓存,其他实例的缓存还是旧的。解法是配置变更时发一个消息(走 Redis pub/sub 或消息队列),所有实例订阅这个消息,收到后各自 evict。不广播,多实例间就会出现「配置改了但有的实例还用旧配置」的不一致。这个不一致在单实例时不存在,多实例时必须处理——这也是为什么「单实例跑通」和「多实例生产」之间有明显的 gap,缓存失效在多实例下要靠广播补齐,否则就会出现配置不一致的问题。

深度展开:配置安全的纵深防御

API Key 是多租户 AI 系统里最敏感的数据——拿到一个租户的 Key,就能冒充这个租户调用模型,消耗它的额度。所以 Key 的安全不能只靠「加密存库」一道防线,要做纵深防御。

第一道防线是加密存储。Key 落库前 AES 加密,数据库被拖库了也拿不到明文。这道防线的要点是加密密钥本身的管理——加密密钥不能存在代码里(代码泄漏了密钥就泄漏了),要存在环境变量或密钥管理服务(如 Vault)里。我们的做法是加密密钥走环境变量注入,不进代码仓库、不进配置文件,从源头切断「代码泄漏导致密钥泄漏」的链路。

第二道防线是脱敏返回。前端展示 Key 时脱敏成 wx12****cd,即使前端被 XSS 攻击,攻击者也拿不到完整 Key。这道防线防的是「展示环节的泄漏」——不是所有泄漏都是拖库,前端页面被截屏、被 XSS、被浏览器插件读取,都可能导致展示中的 Key 泄漏。脱敏让这些场景下泄漏的只是掩码,不是真 Key。

第三道防线是明文 Key 的最小暴露。Key 的明文只在 getDecryptedApiKey 调用的那一刻存在于内存里,调用完就被 GC 回收,不缓存明文、不记日志、不传给不相关的层。这道防线防的是「内存中的泄漏」——如果明文 Key 被缓存(比如存在某个 Map 里),内存 dump 或 heap 分析就能拿到。最小暴露的原则是「明文 Key 的生命周期越短越好,只在绝对必要时存在」。

第四道防线是访问审计。每次调用 getDecryptedApiKey 都记一条审计日志(谁、什么时候、为哪个租户解密了 Key)。正常情况下只有工厂会调这个方法,如果审计日志里出现了非工厂的调用,就是异常——要么是代码 bug(不该调的地方调了),要么是安全事件(有人在非授权路径解密 Key)。审计不是防泄漏,是泄漏后能追溯——「Key 是什么时候、被谁解密的」,这是事后追责的依据。

这四道防线叠起来,Key 的安全才有纵深:加密防拖库,脱敏防展示泄漏,最小暴露防内存泄漏,审计防事后无据。任何一道被突破,其他几道还在兜底。单靠「加密存库」一道防线,一旦加密密钥泄漏,所有 Key 全部裸奔——这也是为什么安全要做纵深防御,不能只靠一道,一道破了就全盘皆输。

深度展开:多租户 AI 的成本治理

动态 ChatModel 工厂解决了「每个租户用自己的模型」,但「用」是有成本的——每次调用模型都消耗 token,token 就是钱。多租户场景下,成本治理和模型隔离一样重要。

成本治理的第一个维度是月度 token 预算(monthly_token_limit)。每个租户每月能消耗多少 token,超了就停。这防的是「一个租户的异常使用(比如有用户疯狂发超长问题)吃掉平台总成本」——没有月度预算,一个租户的异常消耗可能让平台的 API 账单翻倍。月度预算是「总量控制」,防异常、防超支,是成本的第一道闸。

成本治理的第二个维度是速率限制(rate_limit_rpm)。每分钟最多调多少次,防的是「瞬时洪峰」——某个租户突然发起大量并发请求,把供应商的并发限制打满,影响其他租户。速率限制是「流量控制」,防洪峰、防互相影响,是成本的第二道闸。

这两个维度要配合:月度预算管「总量」,速率限制管「节奏」。只有月度预算没有速率限制,一个租户可以在月初一分钟内把月度预算烧光;只有速率限制没有月度预算,一个租户可以持续低速消耗,月底照样超支。两者配合,既防「一瞬烧光」也防「持续超支」,两个维度缺一不可,少任何一个都有漏洞。

成本治理还有个容易被忽略的点:成本可观测。每个租户消耗了多少 token、花了多少钱,这些数据要能查、能看。我们的做法是每次模型调用都记录 token 消耗(从 API 响应里取 usage 字段),按租户聚合,在后台展示「租户 token 消耗趋势」。这个可观测性不只用于计费,还用于「发现异常」——某个租户的 token 消耗突然飙升,可能是被滥用,也可能是 AI 助手的 prompt 变长了(比如 RAG 召回了太多片段),两种都需要人工介入排查,不能放任不管。成本可观测是成本治理的「眼睛」,没有它,预算和限流都是盲控,出了问题也不知道问题出在哪。
