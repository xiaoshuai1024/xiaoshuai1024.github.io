一片荒芜时先做哪个:初创团队工程化基建的四层优先级

我从前公司带过来一套完整的工程化体系——监控告警、自动化测试、CI 流水线、Code Review 流程,什么都有。跳到初创团队的第一天,打开电脑,什么都没有。

没有监控系统,线上出没出事全靠用户投诉。没有自动化测试,改一行代码心里没底。没有 CI,部署靠手动 mvn package 打个 jar 包再 scp 到服务器。没有敏捷流程,需求口头传达,排期拍脑袋。

更难受的是,团队里没人觉得这些是问题。业务在跑,功能在加,「先把功能做出来」是唯一的优先级。

这种「一片荒芜」的局面,是很多从大厂跳到初创的人都会遇到的情况。难的不是「工程化有哪些东西」——这个大厂都教过你了。难的是:这么多东西都要做,先做哪个?后做哪个?什么能缓一缓?

我自己花了半年,一点一点把这套基建搭起来。现在回头看,如果让我重来一次,我会用一套完全不同的顺序。

先说结论:优先级不按重要性排,按投入产出比 + 阻塞解除排

很多人排工程化优先级,默认按「重要性」排:测试最重要(质量是生命)、监控也重要、CI 也重要、敏捷也重要……最后发现什么都重要,什么都想做,半年过去什么都没落地。

我的做法是换一个维度:不看这东西「重不重要」,看它「能多快见效 + 不做它会卡住什么」。

这套顺序的逻辑是:先做投入产出比最高的,再做能放大前面效果的。

  • 监控第一:投入最小(一周搭完),见效最快(立刻能看到线上有没有事)。不做它,你后面做的所有东西都是瞎的——你不知道测试有没有跑、CI 有没有生效、部署有没有出问题。
  • 测试第二:投入中等(一个月搭框架),但它解除的阻塞最大——没有测试,你不敢重构、不敢升级、不敢让新人碰核心代码。
  • 流水线第三:投入较大(两周配 CI/CD),但它把前两步放大——监控的数据靠流水线部署的探针采集,测试的价值靠流水线自动执行才不会变成摆设。
  • 敏捷第四:投入最持续(一直在磨合),但它放大团队协作效率——前面三步是技术基建,敏捷是让团队用起来这些基建的流程保障。

下面四节,我按这个顺序,逐个讲每一层我做了什么、为什么这么排、踩过什么坑。

第一层:监控——不是因为重要,是因为一周就能看见效果

荒原里最怕什么

初创项目最可怕的不是「有 bug」,是「不知道有 bug」。

我刚去的时候,有一次用户反馈「页面打不开」。运维去服务器看了一眼,说「CPU 正常啊」。开发去看日志,日志只有 console.log,什么有用信息都没有。最后排查了三个小时,发现是某个接口超时——但没有任何监控告诉我「这个接口超时了」。

这种「靠用户投诉发现问题」的模式,在大厂是不可想象的。大厂的监控体系会主动告诉你「P99 延迟涨了 50ms」,甚至在你之前就告警了。初创团队没有这个,所以问题永远是滞后发现的——等用户受不了来投诉,问题已经发酵了几小时甚至几天。

先搭什么,后搭什么

监控也分层,不能一上来就搞全链路追踪。我的顺序是:

  1. 基础设施监控先上(2 天):服务器 CPU、内存、磁盘、网络。用 Prometheus + Node Exporter,或者更轻量的方案(初创团队可以直接用云厂商的监控)。这一步解决「服务器还活着吗」的问题。
  2. 应用存活 + 核心接口监控(3 天):健康检查接口、核心业务接口的响应时间和错误率。这一步解决「服务能用吗、好不好用」的问题。
  3. RequestId 链路追踪(1 周):给每个请求打一个 ID,串起网关、服务、数据库的日志,一条 RequestId 贯穿整条后端调用链。这一步解决「出了问题怎么查」的问题。

详细怎么搭,我写过一篇可观测性:RequestId 与监控告警,这里不重复。

我要强调的是为什么这个顺序:从粗到细。先确保「能发现有问题」(基础设施监控),再确保「能定位到哪个服务」(应用监控),最后确保「能查到具体哪次请求」(链路追踪)。反过来做的话,你会在还没搞清楚「服务器活没活」的时候,就去折腾分布式追踪——那是本末倒置。

告警降噪:比加监控更重要的事

监控搭起来之后,最容易犯的错是告警太多。

我刚加完监控那一周,告警群一天响几百次。CPU 超过 80% 告警、某个接口慢了告警、磁盘用了 70% 告警……一周之后,所有人都把告警群设成免打扰。然后某天凌晨服务挂了,没有一个人看到告警。

告警不分级,等于没有告警。 后来我按「需不需要人立刻介入」分了三档:

  • P0(电话 + 短信):核心接口 5xx 错误率超阈值、服务健康检查失败。必须立刻处理。
  • P1(飞书 @值班人):P99 延迟飙升、非核心接口异常。工作时间处理。
  • P2(只进看板,不告警):CPU 波动、磁盘缓慢增长。每天扫一眼就行。

这套分级背后的判断是:告警的价值不在于「多全面」,在于「响的时候真的有事」。几百条告警没人看,不如十条告警条条都处理。

监控这块后来我又往「自愈」方向做了——不只是告警,还能自动重启、自动扩容、AI 自动介入排查。那是更后期的进化,我写在了智能监控自愈里。但所有这些的前提,都是先把最基础的「能看到问题」搭起来。

第二层:自动化测试——解除「不敢动代码」的阻塞

为什么测试排第二,不排第一

很多人觉得「测试是质量的生命,应该第一做」。但在没有监控的时候做测试,你连「测试有没有跑、跑的结果有没有人看」都不知道。

测试的价值要发挥出来,有两个前提:第一,你得能看到测试的结果(这需要监控);第二,测试得自动跑(这需要流水线)。两个前提都没有的时候,你写一堆测试,最后就是躺在仓库里没人运行,变成摆设。

但这不代表测试可以往后排。监控搭完之后,测试是我做的第二件事,因为它解除的是一个更严重的阻塞:没有测试,整个团队都不敢动代码。

荒原里的代码有多难动

我刚去的时候,有一个核心模块经常出 bug,我想重构。团队里一个老员工跟我说:「这个模块别动,上次有人改了一行,线上炸了。」

问题在于,没有人知道这个模块到底怎么工作的。没有文档、没有测试、注释几乎没有。你要改它,只能靠人肉调试,改完心里完全没底。

这就是没有测试的代价:代码变成了「只读」的——可以加功能(在边上加新代码),但不敢改老代码。时间一长,技术债越堆越多,每个模块都变成「别碰的雷区」。

先搭测试骨架,再追覆盖率

测试也不是一上来就追覆盖率。我的顺序是:

  1. 冒烟测试先上:覆盖最核心的「用户能不能注册、能不能登录、能不能下单」这几条主链路。这几个测试是保命的——任何一次改动,先跑这几个,挂了就别上线。一周就能搭完。
  2. API 契约测试:核心接口的输入输出约定,保证接口改动不会悄悄破坏调用方。这部分我用了 ArchUnit 做架构守护,详见我那篇。
  3. E2E 端到端测试:模拟真实用户操作,跑完整业务流程。这部分我做了多角色闭环 E2E。
  4. 单元测试补覆盖率:最后才是逐个模块补单测,把覆盖率拉起来。

这个顺序的逻辑和监控一样:先保命,再保质量,最后追指标。冒烟测试保命(主链路不挂),契约测试保质量(接口不被破坏),E2E 保体验(用户流程通),单测追指标(覆盖率)。

反过来做的人,会一上来就追求 80% 覆盖率,花两个月写一堆单测,结果主链路一个冒烟测试都没有——某天部署完主链路挂了,80% 覆盖率的单测一个都没报错,因为单测压根没覆盖到那条链路。

覆盖率的诚实问题

说到覆盖率,这是初创团队最容易自欺欺人的地方。

我见过有团队把覆盖率写到 95%,发朋友圈炫耀。但仔细看,95% 里一半是 if (x != null) 这种分支的覆盖,真正的业务逻辑压根没测。墙上的 95% 很好看,配置里的有效覆盖率可能只有 25%。

这件事我专门写过一篇覆盖率门禁的「说」与「做」,核心判断是:覆盖率数字本身没有意义,有意义的是「覆盖率背后的测试质量」。初创团队与其追求好看的数字,不如先把那几条核心链路测扎实。

测试纪律:不能跳过、不能占位

测试搭起来之后,要立规矩,否则会慢慢腐烂。我定的两条纪律:

  • 测试不能跳过:CI 里任何人不能加 @Ignore / .skip。要跳过,必须在 PR 里说明原因,且必须当天修复。跳过一次没人管,跳过十次测试套件就废了。
  • 测试不能占位:不允许 expect(true).toBe(true) 这种空测试。宁可不写,不要写占位的——占位测试会骗后面的覆盖率统计。

详细的测试纪律,见UT/IT/E2E 三层测试纪律。

第三层:CI/CD 流水线——放大前两步的效果

为什么流水线排第三

前两步搭完,你有了监控和测试。但这时候有个问题:测试要人手动跑,监控的探针要人手动部署。

我刚加完测试那阵子,每次发版前我都得手动 mvn test,跑十分钟,然后盯着控制台看有没有红的。有时候忙起来忘了跑,或者跑到一半觉得「应该没事」就停了,直接发版。结果就是,测试写了,但没有真正拦住 bug。

人是最不可靠的执行环节。 只要靠人去跑,就一定会漏。所以流水线是必须的——它把「跑测试」这件事从人的自觉,变成机器的铁律。

流水线排第三,是因为它放大前两步:监控的数据靠流水线部署的版本采集,测试的价值靠流水线自动执行兑现。没有流水线,前两步是「写了但没真正用起来」。

云效流水线怎么搭的

我们用的是阿里云效(云效就是阿里云的 DevOps 平台,流水线 + 制品仓库 + 项目协作一体)。选云效的原因很实际:初创团队用阿里云,云效是自带的,不用额外搭 Jenkins,省运维成本。

云效流水线我配了四个阶段:

阶段一:代码提交触发

代码 push 到主分支(或提 MR),自动触发流水线。这一步解决的是「什么时候跑」——不用人记着,提交了就跑。

阶段二:构建 + 测试 + 静态扫描

这一步是流水线的核心,跑三件事:

  • 编译构建(mvn package / gradle build)
  • 全量测试(单测 + 契约测试)
  • 静态代码扫描(SonarQube 或类似工具,查代码异味、安全漏洞)

这一步对应我前面说的「契约闸」——在代码合并前,用机器校验 API 契约、SQL 规范、权限注解。详见提交前的四道契约闸。

阶段三:制品打包

构建产物打成 Docker 镜像(或 jar 包),推到云效制品仓库。这一步的关键是版本号管理——每个制品都带 Git commit hash,保证部署的版本是可追溯的。出问题时,能立刻知道线上跑的是哪个 commit。

阶段四:三环境部署

制品依次部署到 dev → staging → prod。每个环境部署前都跑一轮冒烟测试,挂了就熔断,不往下一个环境走。

这套流程背后的逻辑是:把「能不能上线」的判断,从「我觉得没问题」变成「机器验证没问题」。人觉得没问题,经常是「我没看仔细」;机器验证没问题,至少契约和测试这两道是实的。

没有专职运维怎么办

初创团队还有一个现实问题:没有专职运维。流水线谁配?部署谁操作?

我的做法是:让每个开发都自己配自己负责模块的流水线。一开始我写了一份模板流水线,所有人 fork 改改就能用。这样有两个好处:一是开发最懂自己的模块,配出来的流水线最贴合;二是流水线变成了「大家的」,不是「运维的」,不会出现「运维不在流水线没人动」。

代价是,我得花时间 review 每个人的流水线配置,防止有人把测试步骤删了(真的会有,为了「流水线跑快点」)。后来我把核心步骤(测试、扫描、冒烟)设成流水线模板的必选项,不让删——云效支持模板锁定,这个功能救了好几次。

第四层:敏捷协作——让团队用起来前三步

为什么敏捷排最后

前三步都是技术基建。但基建搭好了,团队不按这个流程走,等于白搭。

举个真实的例子:我搭完 CI 之后,很开心地通知大家「以后提 PR 自动跑测试」。结果第一个礼拜,有个开发直接 push 到主分支,绕过了 PR 流程,CI 压根没触发。我问他为什么不走 PR,他说「PR 要等人 review,太慢了,我直接 push」。

技术基建是骨架,流程是让骨架动起来的肌肉。 没有流程约束,人会本能地走捷径——直接 push 比提 PR 快,手写代码比写测试快,拍脑袋排期比开需求评审快。

敏捷协作排最后,是因为它放大前三步的效果:监控要靠「有人盯告警」的值班制度,测试要靠「不通过不让合并」的流程,流水线要靠「必须走 PR」的规矩。

云效项目管理怎么用的

敏捷这块我们也用的云效——云效除了流水线,还有项目协作(需求池、迭代、看板)。

我们没有照搬 Scrum 或者看板里的某一套,而是按团队实际情况裁剪了一套:

  • 需求池 + 迭代:云效里建需求池,每两周一个迭代,迭代开始时把需求拉进迭代。不做复杂的 Story Point 估算,直接按「小/中/大」标,对应 0.5 天 / 1-2 天 / 3-5 天。
  • 每日站会:每天早上 15 分钟,过一下昨天做了什么、今天做什么、有没有阻塞。不搞仪式感,就是同步进度。
  • PR 必须有人 review 才能合并:云效流水线可以设「PR 合并前置条件」,我们设了「至少一个 reviewer 批准 + CI 通过」。这一条把前面三步的技术基建,变成了流程上的硬约束。

最难的不是搭流程,是让大家守流程

敏捷流程搭起来不难,云效点点就配好了。难的是让大家守。

初创团队最大的阻力是「赶进度」。需求方天天催,开发觉得「走流程太慢了」,各种理由要绕过流程:这个需求很急,直接改线上吧;这个 bug 很小,不用写测试了;这次发布很稳,不用走完整流水线了。

我的处理方式是硬性约束 + 弹性放行:

  • 硬性约束:主分支保护、PR 必须过 CI、生产环境只能从制品仓库部署。这几条是铁律,写在云效配置里,谁也绕不过。技术手段兜底,不靠人的自觉。
  • 弹性放行:对于真正紧急的 hotfix,允许走简化流程(只跑冒烟测试,不跑全量),但事后必须补全测试,而且要在迭代回顾里复盘「为什么需要 hotfix」。

这套思路的核心是:用技术手段保证底线,用流程手段处理例外。底线不能破(主分支永远受保护),例外可以商量(紧急 fix 可以简化,但要补)。纯靠流程,人会绕;纯靠技术,又太僵化。

大厂那套直接搬过来会怎样:三处必须裁剪

前面讲了四层怎么搭。但还有一个问题没讲清楚:为什么不能直接把大厂那套搬过来?

我试过。刚来的前两个月,我把前公司的 Jenkins 配置、测试框架、监控大盘原样复制过来。结果两个礼拜就放弃了——太重了,团队根本用不起来。

大厂的工程化体系,是建立在「大厂的约束条件」上的:几十上百人的团队、明确的分工(有专职 SRE、测试工程师、DevOps)、充足的机器资源。初创团队这些条件一个都没有。直接搬,等于让一辆重型卡车在没有公路的山路上跑。

有三处必须裁剪:

裁剪一:监控——砍掉分布式追踪,保留基础监控

大厂的监控体系是全链路的:一次请求从网关到几十个微服务,每一步都有 trace。这套东西搭起来,光配置就要一个团队。

初创团队没必要。我砍掉了分布式追踪(团队规模小,服务不多,问题用 RequestId 查日志就能定位),保留了基础设施监控 + 应用监控 + 告警分级。够用,而且一周就能搭完。

等团队规模到了、微服务拆得多了,再补追踪也不迟。基建是跟着团队规模演进的,不是一步到位的。

裁剪二:测试——砍掉「先全量覆盖再开发」,改成「核心链路先行」

大厂的测试覆盖率门禁是 80% 起,有些团队 90%。新人进来,写一个功能得配两倍代码量的测试。

初创团队这么干,产品得饿死。我把覆盖率门禁设成了 25%——不是「满足于 25%」,是「先把那 25% 用在刀刃上」。核心链路 100% 覆盖,非核心的慢慢补。详细的取舍见我那篇覆盖率诚实。

裁剪三:流水线——砍掉「七阶段发布」,保留三环境

大厂的发布流程可能有七八个阶段:构建 → 单测 → 集成测试 → 性能测试 → 安全扫描 → 预发 → 灰度 → 全量。每个阶段一个团队负责。

初创团队搞这么多阶段,一个迭代都不够走流程。我砍到了三环境(dev/staging/prod),每个环境跑一轮冒烟测试。核心判断是:阶段越多,边际收益越低。从「没有流水线」到「三环境」,收益是巨大的(从人肉部署到自动部署);从「三环境」到「七阶段」,收益很小,但成本翻好几倍。

做完这四层之后,最大的变化是什么

四层基建搭完,大概花了半年。搭完之后最大的变化,不是「bug 变少了」或者「效率变高了」这种结果——这些当然有,但不是最关键的。

最关键的变化是:团队的心态变了。

搭完之前,每个人写代码都是提心吊胆的——怕改出 bug、怕部署出事、怕线上挂了不知道。搭完之后,大家敢重构了(有测试兜底)、敢频繁发版了(有流水线保证)、敢放心睡觉了(有监控告警)。这种「敢动」的信心,比任何具体的效率提升都值钱。

我后来把这套基建又往前推了一步——用 AI 加持,让监控能自愈、让测试能自动修复、让流水线能自治。那是进入 AI 开发阶段之后的事了,详见AI 时代的智能监控自愈和把"用 AI"做成工程体系。但所有这些进化的根,都是最初那半年在一片荒芜里搭起来的四层基建。

如果你现在也在一个工程化基建缺失的团队,我的建议很简单:别想着一步到位,也别照搬大厂。先从投入产出比最高的监控开始,一层一层往上搭,每一层都让它先跑起来、先见效,再考虑下一层。 工程化不是一场「把所有东西都建好」的战役,是一个「每次都比昨天好一点」的长期过程。最难的不是搭东西,是搭完之后让它真正跑起来、被团队用起来——而这件事,只有从「投入产出比」出发的顺序,才走得通。
