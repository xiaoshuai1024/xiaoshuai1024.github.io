【AI 研发实战】编译能过 ≠ 代码正确:一次"幽灵删除"事故复盘

这条事故属于我「AI 开发」阶段的质量门禁系列:当 AI 开始大量写代码,「编译绿」这件事的含金量被彻底重估。复盘之后,我们团队立了一条铁律挂在评审回路最显眼的位置——

mvn compile 通过 ≠ 代码正确。

过去那些年,我们把"编译绿了"默认等同于"代码对了"。这个默认在人工时代勉强成立,在 AI 大量写代码的今天,代价会被成倍放大。

一句话讲完整件事:编译器只认语法契约,不校验行为;AI 删一段并发守卫时,编译照样绿、它还底气十足说"已完成"。所以门禁不能停在"能编译",得升级到"得过 verify";任何"我搞定了"必须自带证据(Proof-of-Work),焊进门禁,不靠自觉。

一次看起来无害的"搜索优化"

事情起因是一次很普通的提交,说明写的是"搜索优化"。diff 拉出来看也没问题:搜了什么、优化了什么,逻辑闭环,命名干净,像是 AI 认真做的一次重构。没人会为这种提交专门去翻一遍并发模块——毕竟提交说明里一个字都没提并发。

我们当时的评审回路已经跑起来了,流程走得很顺:

  • Agent 评审 → 自动修复 → 收敛;
  • 然后本地 mvn compile 一把过,绿灯;
  • 提交,合进去。

直到后来有人回头去看那段并发相关的核心代码,才发现不对劲——整整 270 行,没了。不是改坏,是静默删除。那是一处并发安全的守卫逻辑,平时不出声,但在高并发路径上是兜命的。

那 270 行到底是什么

既然删掉的是"守卫",就得说清楚守卫长什么样,不然这事听着玄学。

并发安全守卫在我接触的 Java 服务里,通常落地成这么几类东西:入口的限流(令牌桶 / 漏桶,挡住突发流量)、关键资源的分布式锁(同一把 key 不能两个节点同时改)、写操作的幂等键(同一笔请求重复提交只生效一次)、还有租户 / 用户级隔离(查询不能串号)。270 行代码,基本就是这几类的组合:加锁顺序、超时降级、重试上限、隔离上下文传递。

这类代码有个共同点:它不参与"功能能不能跑通",只参与"高并发时会不会出事"。你在本地单线程点一遍接口,它一行都不会被执行;压测、大促、灰度放量那一刻,它才是兜命的那道闸。

我特地拆一下"删了也不立刻炸"的原因,这点是整个事故最难解释、也最容易被低估的地方:

  • 触发是概率性的。分布式锁只在两个请求同时打到不同节点时才显形;幂等键只在重复提交时才暴露;限流只在 QPS 过阈值时才生效。本地手测一遍,触发概率约等于零。
  • 失败是延迟的。数据写花、串号、超卖,往往要等到对账、或者用户投诉才被发现,事故现场已经凉了,日志可能还查不全。
  • 影响是积累的。删一段守卫不会让 P99 变差,只会让"某一次高峰"直接塌方。日常监控全是绿的,出事那一刻就是大事故。

这三个特征叠加,就解释了为什么这类 bug 几乎不可能在"刚提交"时被抓到——它的潜伏期可能是几天、几周,甚至到大促当天才爆。等爆的时候,事故链早就被后续提交层层覆盖,溯源极其痛苦。

这就是为什么删掉它,编译器不报错、单测不红、本地联调也照跑、线上能稳跑好几天。它和功能的耦合是运行时的、概率性的,不是编译期的、确定性的。一句话,它是行为依赖,不是语法依赖——语法层面,删了它引用全部干净,等于没存在过。

为什么编译会"绿"

删掉 270 行并发守卫之后,mvn compile 照样通过,一点脾气没有。

要理解这点,得把编译器的职责边界画清楚。它真正管的就这几样:类型匹配(参数是 String 你别传 int)、方法签名(调用的方法在不在、可见性对不对)、符号引用(类、字段、import 指向的东西真的存在)、语法结构(括号、分号、泛型擦除)。这些是编译期就能确定的契约。

但它一行都不管的,是运行期才暴露的行为:并发是否安全、业务语义是否正确、数据会不会写花、锁会不会死锁、限流挡不挡得住。这些东西编译器看不见——它只能看见代码"写没写对",看不见代码"跑起来对不对"。

把边界摆成一张表会更直观:

维度编译器管的(语法契约)编译器不管的(运行行为)
类型 / 签名✅ 参数类型、方法可见性、泛型—
符号引用✅ 类、字段、import 真实存在—
语法结构✅ 括号、分号、语法糖展开—
并发安全—❌ 锁顺序、竞态、可见性
业务语义—❌ 重复提交是否幂等、边界条件
数据一致性—❌ 是否串号、是否脏写
性能边界—❌ 限流挡不挡得住、超时降级

右边那一整列,恰好就是并发守卫的全部职责。所以删守卫在左边那一列里没有任何痕迹可循——没有悬空调用、没有未使用变量警告、没有类型不匹配,编译器从它的视角看,这次删除干净利落,毫无副作用。它没有"装作没看见",它是真的没看见,因为这件事根本不在它的检查清单上。

那个绿灯的本质,是编译器在它职责范围内诚实地"没意见",而我们把"没意见"误读成了"没问题"。这两种"没"隔着一整条命,但在 CI 的绿灯面前,它们看起来一模一样。

这层误读在人工时代没怎么炸,是因为人会掂量:删一段带"锁"字、“限流"字样的代码,本能就会多看两眼。AI 没这个本能。它只优化"当下能编译、当下能跑通”,长期正确性不在它的目标函数里。于是这条边界在 AI 时代被直接放大:删守卫,在它认知内是一次合理优化,天经地义。

换句话说,并发守卫这种代码,天生就是"删了也能编译、删了也不会立刻炸"的代码。它不是语法依赖,而是行为依赖——编译器看不见它,AI 也看不见它,但生产环境看得见。

根因:当时的门禁只跑到"编译"那一层

事后复盘,我们没去怪 AI——AI 删代码,在它认知范围内是合理的优化动作。真正的问题是反馈回路太浅。

那一版的评审回路,提交前置门禁只跑到 compile:能编译过,就放行。测试没跑、契约没扫、E2E 没动。

Maven 的生命周期是一串串联的 phase,从前往后大致是 validate → compile → test → package → integration-test → verify → install → deploy。关键在于后面的 phase 依赖前面的、但前面的不会反向触发后面的:你跑 compile,它只干"源码编成 class"这一件事,test 往后的全部不碰。这就是为什么"我本地 mvn compile 绿了"这件事,在质量层面约等于啥也没验证。

verify 不一样。它在 integration-test 之后,设计目的就是"在装包 / 部署前做最后一道完整校验"——单测、集成测试、契约脚本、自定义的 verify mojo,该跑的都跑完,任意一个失败整个构建就红。我们当时等于只跑了生命周期的第一格,就拿它当质量结论用了。

换句话说,我们用"语法正确"这一把尺子,去量了一段"行为正确"才算数的代码,自然量不出来。

这就是事故的真正根因:不是 AI 删错了,是门禁没拦住。

顺带说一句:为什么 AI 删守卫时不会有半点心虚,这事的根也在这里。它的优化目标是"任务收敛 + 编译通过 + 反馈正向",这三个信号在删守卫的瞬间全部满足——任务收到"删除无用代码"的指令、删完编译绿、用户(或 Agent 评审)反馈"看着挺干净"。在它的目标函数里,这是一次教科书级的成功重构。它不会去想"这段代码在高并发时干嘛",因为那不是它被评估的维度。这也是为什么指望"AI 自己报告风险"基本是奢望——它报告的,永远是它目标函数范围内的风险。

修复:把"能跑"换成"得验"

改法很直接——把提交前置门禁从"能编译"升级成"得过 verify"。在 Maven 语境里,compile 只编译,verify 才会跑完整的验证阶段。我们在这个阶段里塞进了三层东西,各管各的盲区:

一是分层测试(UT / IT / E2E)。 并发守卫这种代码,得靠真正会触发并发路径的测试网住它——你删一行,就有测试红给你看。UT 网住纯逻辑(锁的获取顺序、幂等键生成),靠 ConcurrentHashMap + CountDownLatch 起多线程去抢同一把锁,断言"只成功一次";IT 网住带中间件的路径(用 testcontainers 真起一个 Redis 跑分布式锁,验证租约、续约、释放);E2E 兜真实链路。三层的分工就是:越往下越快、越精准;越往上越慢、越接近真相。这次事故里,如果当时有"50 个并发线程抢同一个订单号、断言只扣一次库存"的 IT,删守卫的那一刻就红。

二是契约扫描。 跨层的、编译器管不到的约定(比如某个路由注解、某段权限 key、某个限流策略注解),用脚本把它们当事实源 diff 一遍。谁动了契约却没改文档 / 没改调用方,脚本直接红。这一层防的是另一种"删了也能编译"——契约漂移。比如把 @RateLimit(qps=100) 改成 @RateLimit(qps=10000),编译照样过,但限流基本等于关了,契约扫描对比"事实源"就能揪出来。

三是 E2E 兜底。 哪怕单测漏了,真实路径还能再拦一道。这一层慢、贵,但它是"删了行为类代码后,真实链路会不会塌"的最后一道闸。

这三层一起跑,任何一个红,就不让提交。核心结论是:从那以后,这类"删了也能编译、但行为崩了"的改动,在门禁这一层就被摁住了。

落地的钩子很简单,就是给 git 的 pre-commit / pre-push 挂一个强制 verify,外加在 CI 里再跑一遍防止绕过。pom.xml 那边把契约扫描绑到 verify phase 上:

<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-verifier-plugin</artifactId>
  <executions>
    <execution>
      <phase>verify</phase>
      <goals><goal>verify</goal></goals>
    </execution>
  </executions>
</plugin>

关键是别让这层门禁"可选项"——一旦它变成"赶时间可以跳过",这道闸就废了。所以 CI 上 verify 失败直接 block 合并,本地 pre-push 也拦一道,谁要绕过得在群里吼一声,留痕。

三层各自的拦截目标,我单独拎出来做个速查,方便你对着自己团队对一遍:

层拦什么典型场景
UT纯逻辑 / 锁顺序 / 幂等删了 checkAndLock,并发抢锁测试当场红
IT带中间件的真实路径Redis 锁租约被改,testcontainers 跑挂
E2E真实链路兜底单测都漏了,真实下单流程超卖被拦
契约扫描跨层契约漂移限流注解 QPS 被调大,事实源 diff 报红

沉淀成铁律:完成前先自证

这次事故最后固化成一条规矩,我到现在都当成底线:

任何"我搞定了"的声明,必须自带证据。 你说改完了,先把 verify 跑过、把测试门禁亮出来,这叫 Proof-of-Work。口头承诺、自信、甚至 diff 看起来很整洁,都不算数。

具体怎么算"自带证据",我们落地成了一张提交前自检清单:

  • 门禁绿:mvn verify 本地全绿,把日志 / 截图贴到 PR 描述,而不是只写"已自测";
  • 覆盖未降:这次改动涉及的包,行覆盖率不低于改动前(用 jacoco diff 卡);
  • 契约未漂移:契约扫描脚本 0 红,改了注解就得有对应的调用方 / 文档变更;
  • 行为测试命中:改动的代码路径,至少有一条 UT 或 IT 实际跑到那几行(光有覆盖率不够,得是真正触发行为的用例)。

这四条单独看都朴素,叠在一起就是"完成"的硬定义——少一条,就只能叫"我觉得改完了",不算完成。

这道"自证"在 AI 协同里尤其重要——因为 AI 不会心虚。它删 270 行并发守卫时,一样底气十足地告诉你"已完成优化"。它不是在骗你,它真的认为自己优化了:目标函数里只有"编译通过 + 任务收敛",没有"行为长期正确"。删掉一段"看起来没人调"的守卫,对它来说就是一次干净的精简,天经地义。你如果问它"你确定没问题吗",它大概率还会再确认一遍"没问题,已验证"——因为它验证的标准,就是它自己的目标函数。

信它,就重蹈了我们的覆辙。

所以我把自证焊进了门禁:不靠人去问、不靠自觉,跑不过就是不让进。门禁不红,谁都不许说自己搞定了。AI 也好、人也罢,在门禁面前只有"通过 / 没通过"两种状态,没有"我尽力了"。

这条教训,对纯人工评审一样成立

最后补一句:别以为这是 AI 特有的坑。

纯人工评审里,“编译过了 + 我看了一遍 diff + 看起来没毛病"也是每天都在发生的放行逻辑。一个删掉并发守卫的 PR,人眼一样可能扫过去觉得"这段没人调,删了合理”——尤其在 diff 一两百行、评审疲劳、commit message 写得还行的前提下。

我复盘过当时那段 diff 为什么人也没看出来:那段守卫代码命名很朴素(checkAndLock、dedup),没有显眼的"WARNING / 重要 / DO NOT REMOVE"注释;调用处分散在三四个地方,每处只调一行,看着像"普通工具方法";commit message 只写了"搜索优化",完全没提并发。三件事叠加,人眼扫一遍就会得出"删了合理"的结论。差别只是 AI 改得更快、范围更大,所以塌方也更突然。

这也是为什么我对纯人工评审越来越没信心——人眼会被 diff 的整洁度骗,会被 commit message 骗,会被"看起来没毛病"骗。门禁比人诚实:它不看 message,只看跑没跑过。

这条经验我在主力工具 Claude Code,以及 Cursor、OpenCode 上都验证过——编译能过从来都不等于代码正确,这件事和谁来写代码无关。

只是在 AI 时代,你得用更硬的门禁把这个等式彻底拆开。否则早晚会有一段被静默删掉的守卫,在某次高并发时教你做人。

你团队现在是卡在 compile 还是 verify?有没有删掉也能编译、但行为当场崩的改动溜进去过?评论区聊聊,我想知道这条坑是不是只有我们踩过。
