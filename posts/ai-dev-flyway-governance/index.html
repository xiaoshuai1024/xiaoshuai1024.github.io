500+ 功能模块 + 400 个迁移:多租户 SaaS 的 Flyway 治理实战

上一篇讲分润,这篇讲它的地基——数据库迁移。

研发账本里提过这个项目有 500+ 功能模块(633 张业务表)。但表不是一次建好的,它们分散在近 400 个迁移脚本里,4 个人并行写,跨 2 个月。没有一套治理,这种规模必崩:版本号撞车、生产漏执行、多人改同一张表冲突、历史表脏数据。

这篇拆我们怎么用 Flyway 把它管住,全是真金白银的教训,包括一次生产 gap 事故的止血。

为什么选 Flyway 不选 Liquibase。两个都成熟,选 Flyway 是因为它的迁移用纯 SQL 写,而 Liquibase 用 XML/YAML/JSON 描述变更。对这个项目,纯 SQL 有两个优势:一是 AI 写迁移更顺——AI 对 SQL 语法的掌握远好过 Liquibase 的 DSL,生成质量更高;二是迁移文件可直接复制到数据库客户端手动执行,排查止血时不用再翻译。Liquibase 的抽象层在「跨数据库」场景有价值,但我们只用 MySQL,这层抽象是纯负担。工具选型要看场景,不是越强大越好。

还有一个背景:为什么迁移治理在这个项目这么吃重。因为这个项目是「4 人 + AI、2 个月冲百万行」的节奏,schema 变更密度极高——平均每天十几个迁移,4 个人加多个 AI worktree 并行写。这种密度下,如果没有自动化校验,撞号、漏执行、冲突是必然的,不是偶然。治理不是锦上添花,是能不能跑起来的前提。

一、规模先摆出来

db/migration/ 目录下,迁移脚本的真实分布:

维度数据
迁移脚本总数近 400 个(squash 之后)
baseline 文件1 个,4546 行,集中了 207 张表的完整 schema
其余增量迁移约 190 个,含 86 个建表、98 个改表、36 个 seed
涉及 tenant_id 的迁移93 个(多租户是核心隔离维度)

4 个人 + AI,2 个月产出这个量。这种密度光靠「大家注意点」是不可能的,必须靠机制。

对比一下行业里常见的规模。一个中型 SaaS 项目,迁移脚本通常在几十到一百个量级,表两三百张。我们 500+ 功能模块、近 400 个迁移,密度是普通项目的 3-4 倍,而且是在 2 个月里产出的(普通项目是按年累积)。这种密度下,任何依赖「人记得」的规范都会失效——人记不住 400 个版本号,也盯不住每次合并。所以这套治理的设计原则就是:凡是能机器校验的,绝不依赖人。版本号唯一性靠构建期脚本、gap 漏执行靠启动校验、tenant_id 缺失靠 review checklist 兜底,人只做机器判断不了的决策。

二、第一道坎:squash——把历史压成一个 baseline

项目跑到第 6 周,迁移脚本累积到了 300 多个,最早那批表早就稳定了。问题来了:新环境初始化要跑 300+ 个脚本,又慢又容易在某个老脚本上挂;多租户字段 tenant_id 在历史脚本里反复加,逻辑冗余。

具体慢到什么程度。新环境(或 CI 里起一个干净库)跑一遍完整迁移要十几分钟,其中大半时间花在那些早就稳定的早期脚本上——它们的表已经建好、数据已经稳定,但每次新环境都要把建表、改列、加索引的完整历史重跑一遍。更要命的是,某个早期脚本如果依赖了当时的环境状态(比如先建 A 表再往 A 表插 seed),顺序一乱或中间某步失败,后面全卡住。

我们做了一次 squash(合并历史迁移),把这 300 多个脚本合并进一个 V1__init_multi_tenant_tables.sql baseline。

为什么是第 6 周做,不是更早或更晚。更早做,迁移还没稳定,合并完没几天又得重新 squash;更晚做,新环境的初始化成本已经累积到影响开发节奏了。第 6 周是个甜区:核心 schema(207 张表)基本定型,后续主要是增量字段和索引调整,squash 一次能管很久。这个时机判断比 squash 技术本身更重要——squash 早了浪费,晚了已经造成痛苦。

baseline 文件头有一行注释,标明它的来历:

-- Squashed V1: full schema for dayuancheng (generated 2026-05-28 from prod)
-- Contains all 207 tables. Previous V2~V3308 migrations consolidated.

合并后的 baseline 实测 4546 行、207 张 CREATE TABLE IF NOT EXISTS。几个关键设计。

全部用 CREATE TABLE IF NOT EXISTS。squash 后还要兼容已经在跑的老环境,IF NOT EXISTS 保证幂等。这个选择有个权衡:squash 后的老环境,它自己历史表里已经有这 207 张表,如果 baseline 用纯 CREATE TABLE,Flyway 一执行就会报「表已存在」直接挂。IF NOT EXISTS 让 baseline 对「从零建」和「老环境补登记」两种情况都成立。

保留 AUTO_INCREMENT=N 的真实水位。比如 audit_event AUTO_INCREMENT=778、asset_account AUTO_INCREMENT=332——线上自增已经到这个数,squash 进去时把这个水位一起带上,避免主键冲突。如果不带水位,新环境初始化后主键从 1 开始,而业务逻辑里可能已经引用了高 ID(比如截图、日志),虽然概率低但一旦撞上就是数据错乱。带水位是「宁可主键有空洞,也不能撞」。

dev 库和共享库的 flyway_schema_history 同步重置,都只剩 V1 这一条记录。这一步必须和 squash 同步做,否则 Flyway 看到历史表里有 V1~V3308,又看到 baseline 是 V1,会判定版本冲突拒绝启动。重置历史表等于告诉 Flyway「从现在起,这个库的起点就是 V1」。

squash 之后新写的迁移,版本号继续往上递增(我们后来切到时间戳,下一节讲)。所以 squash 不是「把历史全抹了」,而是「把已经稳定的部分压成一个起点,新变更继续增量」。这个心智模型很重要:baseline 是地基,增量迁移是持续生长的楼层,两者不能混。谁要是 squash 之后还在增量迁移里去改 baseline 的表定义,就又把两层混在一起了——正确做法是改表永远走新的增量迁移,baseline 只在下次 squash 时才更新。

这一步是 AI 协同帮我做的。生成 schema dump、重置历史表、批量删旧文件,这种高风险但机械的活,交给 AI 执行、我来审。但「能不能 squash、什么时候 squash」这个判断,是人拍的。

三、第二道坎:版本号——从号段制到时间戳

squash 解决了历史,接下来的问题是:4 个人加多个 AI worktree,并行写新迁移,版本号怎么不撞。这套我们演进了三代,踩完坑才稳。

G1:接龙,最早翻车。 最早就是 V1, V2, V3... 接龙。4 个人各开分支,都觉得自己是下一个,一合并就撞,第一周就翻车。

G2:号段制,按人分号段。 第二版按人分号段:甲 1000-1999、乙 2000-2999、丙 3000-3999,中间留 gap buffer。还写了个 resolve_flyway_segment.py 脚本,根据 git 邮箱自动算下一个版本号:

# 简化逻辑:扫本地文件 + 远端 dev 分支 + 预留桩,返回 maxUsed + gap
next_version = max_used_in_segment(author) + 50  # gap=50

号段制解决了撞号,但带来新问题:跨号段合并时 gap 用完就尴尬;依赖人记得跑脚本预约号段,而 AI 不会主动预约;多个 AI worktree 并行时,号段归属判断会乱。

号段制的真实分布现在还能在目录里看到:V1000、V2014、V2015 这种就是号段制时期的产物(号段 1000-1999 用了 V1000,号段 2000-2999 用到了 V2015 左右)。后面切换到时间戳后,V202400001 这种 10 位时间戳成了主流。两套版本号在目录里并存,这不影响 Flyway 执行——它只按版本号数值排序,不关心格式。但这种「历史痕迹」留着,本身就是这套治理演进的活化石。

G3:时间戳,当前主用,彻底解决。 最后定在秒级时间戳:V20260614141405__desc.sql。用 touch V$(date +%Y%m%d%H%M%S)__desc.sql 创建。4 个人加 N 个 AI worktree,谁也不会撞——除非两个人在同一秒创建,这个概率低到忽略,且构建期有兜底校验。

时间戳还有个隐性好处:版本号本身就带了时间信息,排查问题时一眼能看出这个迁移是哪天加的。号段制的 V1284 你得查 CHANGELOG 才知道是啥时候,时间戳的 V202406171400 直接就知道是 2024 年 6 月 17 日下午两点。迁移多了之后,这个时间可读性很值钱。

这一演进的教训是:治理方案别一上来追求完美,先跑起来,撞了再改。号段制那套脚本现在还留着(万一回退),但实际已经不用了。

四、第三道坎:三道防线,保证版本号真不撞

光有规范不够,必须有自动校验。我们立了三道防线:构建期硬校验、时间戳天然唯一、启动期 gap 检测加自动 repair。

防线 1 是关键,构建期硬校验。 pom.xml 里把一个 Python 脚本绑到 Maven 的 validate 阶段:

<execution>
  <id>verify-flyway-unique-versions</id>
  <phase>validate</phase>
  <executable>${kd.flyway.version.check.python}</executable>
  <arguments>
    <argument>${project.basedir}/scripts/verify_flyway_migration_versions.py</argument>
  </argument>
</execution>

脚本就做一件事:扫所有 V*.sql,两个文件解析到同一版本 token 就报错退出。这意味着任何人(包括 AI)想提交一个重复版本号,本地构建都过不了。

脚本逻辑很简单但管用:遍历 db/migration/ 目录,对每个 V*.sql 文件用 filename[1:].split("__")[0] 提取版本 token(比如 V202401001__xxx.sql 提取出 202401001),按版本号分组,任何一组里文件数 > 1 就报错。报错信息会把撞车的两个文件名都打出来,方便定位是谁撞了谁。脚本同时兼容 Python 2.7 和 3.x,因为团队里有人本地 python 还指向 2.7,mvn validate 调脚本时不能因为版本挂掉。

这个脚本还顺手跳过 R__ 开头的可重复执行迁移(Repeatable migration),只校验 V__ 版本迁移。因为可重复迁移没有版本号、靠 checksum 触发,不在撞号风险范围内。

这里有个踩坑细节:pom 里属性故意用 kd.flyway.* 前缀,不用 flyway.*。因为 Flyway Maven 插件会把 flyway. 开头的属性当成自己的配置项,报 Unknown property。小坑,但卡过人。

防线 2 是时间戳天然唯一。 这是最省心的防线——只要版本号是秒级时间戳,两个人在同一秒创建迁移的概率低到忽略。但防线 2 不能独立存在,它依赖防线 1 兜底:万一真撞了(比如两个 AI worktree 在同一秒各创建一个),防线 1 的构建期校验会拦住。两道防线叠加,撞号这事在生产里一次都没发生过。

防线 3 是生产兜底。 启动时做 gap 校验,失败可配置成不阻断启动 + 自动 repair。这个是踩了下面的坑才加的。

五、生产事故:一次 gap 漏执行,和它的止血

这块是真实事故,我完整讲出来。

生产环境的 Flyway 配置里 out-of-order: false(顺序执行),最大已执行版本是 3200。问题来了:有 8 个版本号 ≤ 3200 的迁移,因为发布顺序原因没被执行——按 out-of-order=false 的规则,它们被跳过了,永远不会执行。

这 8 个漏掉的版本号是实打实的:1026(修菜单路由名)、1027(圈会议申请加 tenant_id)、1284(达人佣金率字段)、1383(修 tenant_id 排序规则)、1433(联卡退款状态注释)、1483(招募流程修复)、3069(联卡状态加 completed)、3070(创建搜索词典)。横跨好几个模块,有的还涉及 tenant_id 这种多租户隔离字段——漏了就是隔离有洞。

上线后发现几张表字段不对,排查到是这 8 个迁移漏了。止血是一套完整的 scripts/prod/ 脚本:

脚本作用
manual_flyway_gap_fix.sql幂等执行那 8 个迁移的 DDL/DML
record_flyway_gap_history.sql手工 INSERT 8 条进 flyway_schema_history,维持 installed_rank 连续
record_flyway_gap_checksums.sqlrepair 失败时,从 dev 库同步 checksum
run_prod_gap_fix.sh一键编排

手工插历史表那段最讲究。checksum 先置 NULL,installed_by='manual-prod-gap-fix',用窗口函数 @base_rank := @base_rank + 1 维持 installed_rank 连续,WHERE NOT EXISTS 保证幂等。

为什么要这么小心 installed_rank。Flyway 用 installed_rank 给历史记录排序,如果手动插的 8 条 rank 不连续或有空洞,下次启动 Flyway 校验会报「rank 不连续」拒绝启动。窗口函数从 MAX(installed_rank)+1 开始递增,保证新插的 8 条紧跟在已有记录后面,rank 连续无空洞。WHERE NOT EXISTS 则是防重——万一这个止血脚本跑了两次,第二次不会重复插。

checksum 置 NULL 是个临时态。手动插的记录没有真实执行过,checksum 是空的,下次 Flyway 启动会发现「这个版本的 checksum 不匹配」报错。所以止血脚本的下一步是 flyway:repair,让 Flyway 用当前文件重新算 checksum 回填。repair 失败时还有兜底:record_flyway_gap_checksums.sql 从 dev 库同步已知的正确 checksum 过来。三层兜底,因为这 8 条记录一旦没修干净,生产就起不来。

事故后的改动两条。生产 out-of-order 改成 true(commit message 写得明明白白:fix: prod Flyway out-of-order true 避免低版本迁移被跳过),允许补执行漏掉的迁移。加启动期 gap 校验,把「漏执行」从上线后发现,提前到启动时拦截。

复盘这个事故,根因不只是配置。out-of-order: false 在单线程顺序发布的场景是对的——它保证迁移按版本号严格递增执行。但我们的发布是并行的:多个 feature 分支各自带着迁移合并到 dev,dev 再发到生产,发布顺序和版本号顺序经常对不上。一个 V1026 的迁移可能在 V3200 之后才合并进来,顺序执行模式下它就被永久跳过。根因是「并行开发 + 顺序执行」的模型错配,改 out-of-order 只是纠正这个错配。

教训是:数据库变更没有「以后再说」,任何一次发布都得保证迁移全部生效,否则就是隐患。gap 校验加进启动流程后,这种隐患在下次启动就被拦住,不用等上线后查数据才发现。把「事后发现」前移到「启动拦截」,是这套治理里性价比最高的一次改动,没有之一。

六、多租户迁移的三个真实坑

多租户场景下,Flyway 治理还有几个专属的坑,都是踩过的。

坑 1:历史表 tenant_id 缺失,得回填再收紧。 有的老表建的时候没加 tenant_id,后来要补。不能直接 ALTER MODIFY ... NOT NULL——会因为有 NULL 值失败。正确姿势是两步:

-- 第一步:回填历史 NULL
UPDATE tenant_circle_meeting m JOIN tenant_circle c ON c.id = m.circle_id
SET m.tenant_id = c.tenant_id WHERE m.tenant_id IS NULL;
-- 第二步:收紧为 NOT NULL(故意让它失败来暴露脏数据)
ALTER TABLE tenant_circle_meeting MODIFY COLUMN tenant_id VARCHAR(64) NOT NULL;

迁移文件注释直接写明风险:存在跨租户数据混杂风险。收紧 NOT NULL 这一步故意不兜底,让它失败,把脏数据逼出来。

坑 2:幂等 ADD COLUMN。 并发补字段时,可能两个迁移都想加同一列。用 information_schema + PREPARE stmt 做幂等:列存在就 SELECT 1 跳过,不存在才 ADD。

这个坑在 4 人并行时高频出现。甲在自己的 worktree 里给 merchant 表加了 logo_url 列,乙在另一个 worktree 里也给 merchant 加了 logo_url——两人各自看自己分支没问题,合并后两个迁移都跑,第二个就报「列已存在」失败。幂等 ADD COLUMN 让两个迁移都能跑过,第二个发现列已经在就直接跳过。代价是迁移脚本变长(要先查 information_schema),但比起合并冲突,这点长度值得。

坑 3:tenant_id 类型不统一。 unified_activity 表的 tenant_id 一会是 int 一会是 varchar,产生了一连串修复迁移(V3987、V20260610004、V20260610005)。这种类型反复横跳,是早期没定规范的债,后面用类型校验兜住。

类型不统一为什么是大事。多租户系统里 tenant_id 是隔离命脉,它出现在几乎所有表的索引里。有的表是 int、有的是 varchar(64),跨表 JOIN 时类型不匹配会触发隐式转换,直接让索引失效——本该走索引的查询变成全表扫描,在数据量上来后直接拖垮库,这种性能问题是慢性的、不容易第一时间发现。所以后来定了一条硬规范:tenant_id 全站统一 varchar(64),新表强制遵守,老表靠修复迁移慢慢收敛。

坑 4(补充):多租户索引必须带 tenant_id 前缀。 这不是 Flyway 的问题,是迁移治理延伸出来的规范。多租户表的所有业务索引,必须以 tenant_id 打头,比如 uk_order(tenant_id, order_no) 而不是 uk_order(order_no)。因为所有查询都会带 tenant_id 过滤,索引不带 tenant_id 前缀等于全租户扫描,在数据量上来后直接拖垮库。这条规范写进了迁移 review 的 checklist:任何一张多租户表的索引,reviewer 第一眼就看有没有 tenant_id 前缀。

七、AI 协同给迁移治理带来的变化

最后说点 AI 的部分。迁移脚本是最适合 AI 写的——DDL 规则明确、有大量样板、错了构建期就报。但有几个原则我定死了。

AI 起草,人审 tenant_id 和唯一键。多租户隔离字段漏一个就是事故,这块绝不放给 AI 自由发挥。具体怎么审:每张新建的表,reviewer 第一件事是看有没有 tenant_id 列、有没有把它放进唯一键和业务索引里。AI 起草时会漏,因为它不理解「这张表属于哪个租户域」,只照着表名猜。漏了 tenant_id 的多租户表,上线就是数据串租户,这是最严重的事故类型。

AI 写迁移,必须配套写回滚。db/rollback/ 目录放回滚脚本。这条纪律是squash之前立下来的——因为迁移累积到 300 多个时,万一某个迁移有问题需要回退,没有回滚脚本就只能手动改库,风险极高。AI 写迁移时我会要求它同时输出回滚 SQL:加了列回滚就是 DROP、建了表回滚就是 DROP TABLE、改了类型回滚就是改回去。回滚脚本不一定执行,但它必须存在,是「万一」的保险。

版本号由时间戳自动生成,不让 AI 自己挑数字——它会瞎接龙。AI 生成迁移时如果让它自己起版本号,它要么从已有文件接龙(撞号),要么随便编一个(格式不对)。所以版本号生成这一步固定用 touch 命令,AI 只负责写 SQL 内容,不碰版本号。

禁止 MySQL 8+ 独有语法。这个是踩坑定的——有个迁移在生产 MySQL 5.7 上直接失败。AI 不知道生产 MySQL 版本,得靠规范约束。具体禁的是:CHECK 约束(5.7 不强制)、JSON_TABLE 函数、窗口函数在 DDL 里的用法(虽然 5.7 支持部分窗口函数但行为和 8.0 有差异)。这些在 dev 库(MySQL 8.0)跑得好好的,上生产(5.7)就炸。规范写死「按 5.7 兼容」,从源头堵住。

这套治理跑下来,近 400 个迁移、4 人并行,没有因为迁移冲突影响过一次集成。生产那次 gap 是历史遗留(out-of-order 配置问题),不是冲突。治理的价值不是为了好看,是为了让 4 个人能像 1 个人一样改同一份 schema。

七半、迁移治理的几条经验沉淀

跑完这套,有几条经验值得单拎出来,是踩完坑反复验证过的。

迁移脚本必须幂等。 这个原则贯穿所有迁移:建表用 CREATE TABLE IF NOT EXISTS、加列用 information_schema 判断、改数据用 WHERE NOT EXISTS 防重。为什么这么执着幂等。因为迁移脚本会被多次执行——squash 时重跑、止血时补跑、新环境初始化时跑。任何一次非幂等,都可能在某个时机炸。幂等写法看着啰嗦,但它是迁移能被反复执行的保险。

迁移和代码强绑定发布。 一份代码改动如果依赖一个新字段,那这个迁移必须和代码同一个发布包上线,不能代码先上、迁移后补,也不能迁移先上、代码后跟。我们用「迁移在启动时自动执行」来保证这一点——Spring Boot 启动时 Flyway 先跑完所有 pending 迁移,代码才能起来。这把「代码和 schema 不匹配」的风险压到了启动那一刻,而不是散落在运行期。

生产迁移绝不删列。 列可以加,可以改类型(向后兼容地改),但绝不直接 DROP COLUMN。因为线上可能有旧版本代码还在读这个列,删了直接报错。要删列,流程是:先发版本让代码不再读写这列 → 观察一个发布周期 → 再发迁移删列。两步走,中间留缓冲。这条是红线,因为删列是最容易造成线上事故的迁移操作。

checksum 不匹配要 repair,不要手动改历史表。 偶尔会遇到「文件内容变了导致 checksum 不匹配」(比如有人手滑改了已执行的迁移文件),正确做法是 flyway:repair 重新算 checksum,不是手动 UPDATE 历史表。手动改历史表容易改出 rank 空洞或 type 错误,越改越乱。repair 是 Flyway 官方的「对账」操作,它知道该改什么。

这些经验看着琐碎,但每一条背后都是一个真实事故或险情。迁移治理没有高深的架构,全是这种「不省事」的纪律累积出来的。把这些纪律固化进脚本和 checklist,让 4 个人和多个 AI 都能照着走,才撑住了这个规模的 schema 变更不翻车。
