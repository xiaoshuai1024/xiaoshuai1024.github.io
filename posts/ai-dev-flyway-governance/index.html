【AI 研发实战】850 张表 + 近 400 个迁移:多租户 SaaS 的 Flyway 治理实战

上一篇讲分润,这篇讲它的地基——数据库迁移。

研发账本里提过这个项目有 850 张表。但表不是一次建好的,它们分散在近 400 个迁移脚本里,4 个人并行写,跨 2 个月。没有一套治理,这种规模必崩:版本号撞车、生产漏执行、多人改同一张表冲突、历史表脏数据。

这篇拆我们怎么用 Flyway 把它管住,全是真金白银的教训,包括一次生产 gap 事故的止血。

一、规模先摆出来

db/migration/ 目录下,迁移脚本的真实分布:

维度数据
迁移脚本总数近 400 个(squash 之后)
baseline 文件1 个,4546 行,集中了 207 张表的完整 schema
其余增量迁移约 190 个,含 86 个建表、98 个改表、36 个 seed
涉及 tenant_id 的迁移93 个(多租户是核心隔离维度)

4 个人 + AI,2 个月产出这个量。这种密度光靠「大家注意点」是不可能的,必须靠机制。

二、第一道坎:squash——把历史压成一个 baseline

项目跑到第 6 周,迁移脚本累积到了 300 多个,最早那批表早就稳定了。问题来了:新环境初始化要跑 300+ 个脚本,又慢又容易在某个老脚本上挂;多租户字段 tenant_id 在历史脚本里反复加,逻辑冗余。

我们做了一次 squash(合并历史迁移),把这 300 多个脚本合并进一个 V1__init_multi_tenant_tables.sql baseline。

baseline 文件头有一行注释,标明它的来历:

-- Squashed V1: full schema for multi-tenant (generated 2026-05-28 from prod)
-- 把 V2~V3308 的 306 个 SQL 迁移 + 10 个 Java 迁移合并

合并后的 baseline 实测 4546 行、207 张 CREATE TABLE IF NOT EXISTS。几个关键设计。

全部用 CREATE TABLE IF NOT EXISTS。squash 后还要兼容已经在跑的老环境,IF NOT EXISTS 保证幂等。

保留 AUTO_INCREMENT=N 的真实水位。比如 audit_event AUTO_INCREMENT=778——线上自增已经到 778,squash 进去时把这个水位一起带上,避免主键冲突。

dev 库和共享库的 flyway_schema_history 同步重置,都只剩 V1 这一条记录。

这一步是 AI 协同帮我做的。生成 schema dump、重置历史表、批量删旧文件,这种高风险但机械的活,交给 AI 执行、我来审。但「能不能 squash、什么时候 squash」这个判断,是人拍的。

三、第二道坎:版本号——从号段制到时间戳

squash 解决了历史,接下来的问题是:4 个人加多个 AI worktree,并行写新迁移,版本号怎么不撞。这套我们演进了三代,踩完坑才稳。

G1:接龙,最早翻车。 最早就是 V1, V2, V3... 接龙。4 个人各开分支,都觉得自己是下一个,一合并就撞,第一周就翻车。

G2:号段制,按人分号段。 第二版按人分号段:甲 1000-1999、乙 2000-2999、丙 3000-3999,中间留 gap buffer。还写了个 resolve_flyway_segment.py 脚本,根据 git 邮箱自动算下一个版本号:

# 简化逻辑:扫本地文件 + 远端 dev 分支 + 预留桩,返回 maxUsed + gap
next_version = max_used_in_segment(author) + 50  # gap=50

号段制解决了撞号,但带来新问题:跨号段合并时 gap 用完就尴尬;依赖人记得跑脚本预约号段,而 AI 不会主动预约;多个 AI worktree 并行时,号段归属判断会乱。

G3:时间戳,当前主用,彻底解决。 最后定在秒级时间戳:V20260614141405__desc.sql。用 touch V$(date +%Y%m%d%H%M%S)__desc.sql 创建。4 个人加 N 个 AI worktree,谁也不会撞——除非两个人在同一秒创建,这个概率低到忽略,且构建期有兜底校验。

这一演进的教训是:治理方案别一上来追求完美,先跑起来,撞了再改。号段制那套脚本现在还留着(万一回退),但实际已经不用了。

四、第三道坎:三道防线,保证版本号真不撞

光有规范不够,必须有自动校验。我们立了三道防线:构建期硬校验、时间戳天然唯一、启动期 gap 检测加自动 repair。

防线 1 是关键,构建期硬校验。 pom.xml 里把一个 Python 脚本绑到 Maven 的 validate 阶段:

<execution>
  <id>verify-flyway-unique-versions</id>
  <phase>validate</phase>
  <executable>${kd.flyway.version.check.python}</executable>
  <arguments>
    <argument>${project.basedir}/scripts/verify_flyway_migration_versions.py</argument>
  </argument>
</execution>

脚本就做一件事:扫所有 V*.sql,两个文件解析到同一版本 token 就报错退出。这意味着任何人(包括 AI)想提交一个重复版本号,本地构建都过不了。

这里有个踩坑细节:pom 里属性故意用 kd.flyway.* 前缀,不用 flyway.*。因为 Flyway Maven 插件会把 flyway. 开头的属性当成自己的配置项,报 Unknown property。小坑,但卡过人。

防线 3 是生产兜底。 启动时做 gap 校验,失败可配置成不阻断启动 + 自动 repair。这个是踩了下面的坑才加的。

五、生产事故:一次 gap 漏执行,和它的止血

这块是真实事故,我完整讲出来。

生产环境的 Flyway 配置里 out-of-order: false(顺序执行),最大已执行版本是 3200。问题来了:有 8 个版本号 ≤ 3200 的迁移,因为发布顺序原因没被执行——按 out-of-order=false 的规则,它们被跳过了,永远不会执行。

上线后发现几张表字段不对,排查到是这 8 个迁移漏了。止血是一套完整的 scripts/prod/ 脚本:

脚本作用
manual_flyway_gap_fix.sql幂等执行那 8 个迁移的 DDL/DML
record_flyway_gap_history.sql手工 INSERT 8 条进 flyway_schema_history,维持 installed_rank 连续
record_flyway_gap_checksums.sqlrepair 失败时,从 dev 库同步 checksum
run_prod_gap_fix.sh一键编排

手工插历史表那段最讲究。checksum 先置 NULL,installed_by='manual-prod-gap-fix',用窗口函数 @base_rank := @base_rank + 1 维持 installed_rank 连续,WHERE NOT EXISTS 保证幂等。

事故后的改动两条。生产 out-of-order 改成 true(commit message 写得明明白白:fix: prod Flyway out-of-order true 避免低版本迁移被跳过),允许补执行漏掉的迁移。加启动期 gap 校验,把「漏执行」从上线后发现,提前到启动时拦截。

教训是:数据库变更没有「以后再说」,任何一次发布都得保证迁移全部生效,否则就是定时炸弹。

六、多租户迁移的三个真实坑

多租户场景下,Flyway 治理还有几个专属的坑,都是踩过的。

坑 1:历史表 tenant_id 缺失,得回填再收紧。 有的老表建的时候没加 tenant_id,后来要补。不能直接 ALTER MODIFY ... NOT NULL——会因为有 NULL 值失败。正确姿势是两步:

-- 第一步:回填历史 NULL
UPDATE tenant_circle_meeting m JOIN tenant_circle c ON c.id = m.circle_id
SET m.tenant_id = c.tenant_id WHERE m.tenant_id IS NULL;
-- 第二步:收紧为 NOT NULL(故意让它失败来暴露脏数据)
ALTER TABLE tenant_circle_meeting MODIFY COLUMN tenant_id VARCHAR(64) NOT NULL;

迁移文件注释直接写明风险:存在跨租户数据混杂风险。收紧 NOT NULL 这一步故意不兜底,让它失败,把脏数据逼出来。

坑 2:幂等 ADD COLUMN。 并发补字段时,可能两个迁移都想加同一列。用 information_schema + PREPARE stmt 做幂等:列存在就 SELECT 1 跳过,不存在才 ADD。

坑 3:tenant_id 类型不统一。 unified_activity 表的 tenant_id 一会是 int 一会是 varchar,产生了一连串修复迁移(V3987、V20260610004、V20260610005)。这种类型反复横跳,是早期没定规范的债,后面用类型校验兜住。

七、AI 协同给迁移治理带来的变化

最后说点 AI 的部分。迁移脚本是最适合 AI 写的——DDL 规则明确、有大量样板、错了构建期就报。但有几个原则我定死了。

AI 起草,人审 tenant_id 和唯一键。多租户隔离字段漏一个就是事故,这块绝不放给 AI 自由发挥。

AI 写迁移,必须配套写回滚。db/rollback/ 目录放回滚脚本。

版本号由时间戳自动生成,不让 AI 自己挑数字——它会瞎接龙。

禁止 MySQL 8+ 独有语法。这个是踩坑定的——有个迁移在生产 MySQL 5.7 上直接失败。AI 不知道生产 MySQL 版本,得靠规范约束。

这套治理跑下来,近 400 个迁移、4 人并行,没有因为迁移冲突影响过一次集成。生产那次 gap 是历史遗留(out-of-order 配置问题),不是冲突。治理的价值不是为了好看,是为了让 4 个人能像 1 个人一样改同一份 schema。
