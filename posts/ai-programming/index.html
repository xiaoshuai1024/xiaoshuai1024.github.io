AI 编程的门槛和重心:从提速 40% 到不再自己写样板

老实讲，这篇我不想再写那种「AI 真好用，大家都来拥抱」的稿子。过去一年我天天在用 Claude Code、Cursor、OpenCode、Copilot，最大的感受其实不是"快了多少"，而是这件事：写代码的门槛和重心，正在悄悄挪地方。

下面就把这个判断摊开讲——结合我的真实项目经历、踩过的坑、以及后来形成的几条应对规矩。这一篇是调子，后面整个 AI 研发实战系列（认知转变、岗位转型、工程化治理）都会从这儿接出去。

工具盘点：不是谁都能打，先认清楚各自的定位

先把市面上几款主力工具的边界说清楚，免得你拿着锤子到处找钉子。按真实使用频率排序，我的日常主力排序是 Claude Code > Cursor > OpenCode > Codex / Copilot，理由是各自的场景适配不同。

  • Claude Code（主力，使用频率最高）：命令行里的 agent，能自己跑工具、读文件、改文件、跑测试、自主迭代验证。适合「帮我把这事整体做完，做完自己验证一遍」的场景——批量重构、跑通端到端流程、从需求到可运行代码一气呵成。它的长处是上下文长、能自主迭代、敢端到端接活，短处是你得给它明确的边界，不然它容易自己跑偏。我现在但凡能端到端丢的活，首选它。
  • Cursor：把模型直接焊进编辑器——你在哪一行问，它就读哪一行的上下文。适合中型改动：重构一段逻辑、写一个完整接口、跨文件改命名。代价是它对项目结构和规范比较挑，项目乱它会跟着乱。
  • OpenCode：开源命令行场景，或者需要本地可控时切过去。定位接近 Claude Code，胜在可控性。
  • Codex / GitHub Copilot：更轻，定位是"行内补全 + IDE 联想"。写一个函数写到一半它帮你补完下一行，或者注释里写句话它生成实现。适合写样板、补常规逻辑，不适合让它从零搭一个模块。

国内还有 CodeGeeX、通义千问代码版这些，网络环境下更顺，定位和 Copilot 接近，按需选。

一句话：没有"最好"的工具，只有"这件事更该丢给谁"。 我日常是 Claude Code 干端到端的活、Cursor 做编辑器内的中型改动、Copilot 补行内补全，按场景切。

工具切换有个隐性成本要提:每个工具的上下文不共享。你在 Cursor 里喂的上下文,切到 Claude Code 得重新喂一遍;Claude Code 里跑通的逻辑,回到 Cursor 编辑时它不知道你刚才干了什么。所以工具不是越多越好,选两三个主力、用熟用透,比装一堆每个都半生不熟强。我固定 Claude Code + Cursor 两个主力,Copilot 做辅助,这个组合磨合够了,切换的摩擦最小。工具多了,光在工具间切来切去就耗掉不少精力。

起点：OpenLuban 项目，从试试看到离不开

我真正开始重度用 AI 编程，是从个人开源项目 OpenLuban（我离职后把大厂鲁班的架构经验重新开源实现的一套低代码平台，全新代码，与原公司系统无关）开始的。一个人扛大部分开发，技术栈横跨好几个：

  • 前端 Vue3 + Nuxt3
  • 后端 SpringBoot
  • 偶尔 Node.js 写个 BFF 层
  • 有时还要补点 golang 脚本处理数据

多种语言来回切，最容易卡壳的不是"会不会"，而是不常用的语法记混了，得翻文档、查博客，一卡就是十几分钟。这种卡顿单看一次不大，但一天攒下来特别消耗状态。

抱着试试看的心态下了 Cursor。本来只想拿来补补语法、调个小 bug，结果用了不到半天，我就开了 Cursor Pro。

转折点很具体：我把需求说清楚，比如「用 SpringBoot 写一个站点列表查询接口，带分页和条件筛选，关联用户表」，它直接给出可运行的代码，连异常处理、参数校验都写好了。这件事让我意识到，AI 接管的不是某个语言点，而是那一整层"查语法、套样板、对模板"的执行活。 从那以后，这类活我基本不自己写了。

效果：提速是真的，但更值钱的是别的

先说能拿数字说话的。以前写一个完整后端接口——实体类、Mapper、Service、Controller，再到调通——至少半小时；用 Cursor 之后，10 分钟搞定，结构还规整，格式不用手调。前端组件也一样，「写一个 Vue3 表单组件，输入框 + 下拉框 + 校验 + 提交按钮禁用逻辑」，它生成完我调一下类名和样式就能复用。

提速最猛的场景，是多语言切换。刚写完 Java，转头要写 Node.js 的 Egg.js 接口，语法细节容易记串；让 Cursor 先给我一个基础模板，再按需求改，省掉了翻官方文档那段最拖节奏的时间。

粗算下来，这个"单兵作战"的全栈项目，整体进度至少提速 40%。

但「提速 40%」这个数字,我不想让人产生一个误解——以为用了 AI 就能少花 40% 的时间。实际的情况是:时间没少花,花的地方变了。以前 8 小时里,5 小时敲代码、2 小时调 bug、1 小时想设计;现在 8 小时变成 1 小时带 AI 生成代码、3 小时 review 和校验、2 小时调 bug、2 小时想设计。总时间差不多,但「想」和「验」的比重大幅上升。这其实是更累的——敲代码是体力活,可以摸鱼;review 和想设计是脑力活,摸不了鱼。所以很多人用了 AI 一阵子会觉得「怎么比以前还累」,这不是错觉,是重心转移的体感代价。

还有一个关于提速的诚实交代:40% 是「稳定之后的数字」,刚开始用 AI 那阵子,反而更慢。因为你得花时间学工具、调提示词、踩幻觉坑、建 Claude.md。这个「磨合期」可能持续两三周,期间效率不升反降。很多人在这个阶段就放弃了,觉得「AI 还不如我自己写」。但熬过磨合期,把规矩建起来,提速才会显现。AI 编程的提速不是即时的,是「先慢后快」的曲线——前期投入,后期复利。

但老实讲，用久了之后我觉得 40% 这个数字反而不是重点。真正变化的是：我不再把"写出来"当成主要工作了。 以前一天里大半时间花在敲样板、查语法、调格式上；现在这些被 AI 兜走了，我的时间更多花在「这个接口到底该不该这么设计」「这个边界条件有没有漏」上。重心从"怎么写"挪到了"做什么、怎么才对"。

这一点，是后面整个系列我想反复讲的事。

这个重心转移还有一个不太直观的副作用:「会写代码」这件事的价值在贬值,「会判断代码对不对」的价值在升值。以前一个工程师强不强,很大程度看他敲代码快不快、熟不熟——这部分被 AI 削平了,快和熟不再是稀缺能力。但「这段代码该不该这么写」「这个边界漏没漏」「这个设计在半年后会不会坑」,这类判断能力,AI 帮不上忙,反而因为 AI 生成代码速度快了,这类判断的频率更高了——以前一天写 5 个接口,要判断 5 次;现在一天 AI 生成 20 个接口,你要判断 20 次。代码量涨了,判断的密度也涨了,但判断能力没自动涨,这就是很多团队「用了 AI 反而出 bug 更多」的根因——生成快了,把关没跟上。

所以用 AI 编程,真正要补的不是「会用工具」(那很快就能上手),是「会判断」。判断能力靠什么练?靠真实地踩坑、真实地 review 别人的代码、真实地为线上事故负责。这些 AI 替不了,也不该让它替。

两个躲不开的问题：幻觉和代码质量

AI 不是白送的好处。用深了你会发现两个坑特别明显，绕不过去。

第一个是幻觉。 AI 会编造根本不存在的 API、方法，甚至引用一个不存在的依赖。我让 Cursor 写过一次 Redis 缓存工具类，它生成了一个看起来很合理、实际并不存在的 Redis 方法，复制过去直接报错，排查了半天才发现是它编的。这类坑最阴的地方在于：代码看起来太像真的了，你顺着它的思路去查，反而越查越偏。

第二个是代码质量参差。 有时候它能跑，但不优雅——重复代码多、没做优化、甚至埋着潜在性能问题。能跑不等于对，这一点一定要分清。

代码质量的坑还有一种更隐蔽的表现：「能跑但跑得慢」。AI 生成的代码，经常在「功能正确性」上没问题，但在性能上是灾难——比如在循环里查数据库、N+1 查询、该用批量接口的地方逐条调用。这些代码单测能过(因为测了功能没测性能)、code review 也容易漏(因为逻辑看着对),要等数据量上来或者上线高峰期才暴露。我对 AI 生成的数据库相关代码特别警惕,一定手动过一遍查询逻辑——「这个循环里有没有查库」「这个关联有没有 N+1」,这两条成了我 review 时的肌肉记忆。

为什么 AI 会这样？说穿了原因不复杂：模型优化的是"当下能跑、看起来对"，它没有你的业务上下文。 它不知道你这个项目的边界在哪、哪些字段不能为空、哪些操作有副作用。所以它给的是一个"平均意义上最像答案"的东西，而不是"你这个项目里最对"的东西。

理解了这一点，就能理解为什么 AI 写的代码特别「中规中矩」。它给的是统计意义上的最可能解——大量代码里出现最多的写法。但「最常见」不等于「最适合你的场景」。你的项目可能有特殊的性能要求、特殊的业务约束、特殊的团队规范,这些 AI 都不知道。所以 AI 的输出永远是个「起点」,不是「终点」——起点质量不错,但终点还得你把它拉过去。把它当终点直接用,迟早会为那些「中规中矩但不对」的细节买单。

我后来固定下来的四条规矩

坑踩多了就形成了规矩。下面这四条我现在基本每次都走一遍。

1. 写一份 Claude.md，把项目规范喂给它。 把技术栈、目录结构、常用工具类、接口契约、命名约定都整理进一个文件，让 AI 先"读"一遍再生成。这一步直接从源头压幻觉——它不再是凭空猜，而是在你的规范约束下猜。我现在的项目都带这么一份，更新规范时顺手维护，是性价比最高的一招。

2. 生成的代码一定要二次校验，重点逻辑手动过。 复制粘贴就用是大忌。我会过一遍语法错误和逻辑漏洞，尤其是数据库操作和权限校验这两类——这两类错了代价大、还不好测出来。这步看似慢，实则最省时间，因为它帮你挡掉的是那种"上线后半夜被叫起来"的坑。

3. 提示要精准，别给模糊需求。 不要说「写一个登录接口」，要说「用 SpringBoot 写登录接口，接收用户名密码，密码用 BCrypt 校验，返回 JWT 令牌含角色信息，处理用户名不存在、密码错误两类异常」。提示越精准，输出质量越高，返工越少。这条其实是把"想清楚需求"这件事前移了——你把边界想清楚，AI 才有素材做对。

4. 攒自己的提示词模板。 分页接口、前端表单组件、CRUD 模板这些高频场景，我把对应的提示词存下来，下次直接改几个字段复用。既省时间，又能保证稳定输出，不用每次重新想怎么描述。

这四条串起来，其实就是一条链：先把规范喂给它(规矩一)→ 把需求说精准(规矩三)→ 生成后校验关键逻辑(规矩二)→ 高频场景沉淀成模板复用(规矩四)。每一条都在「把人的判断，挪到 AI 接管不到的位置」——规范和需求是生成前的判断，校验是生成后的判断，中间的执行交给 AI。

这张图里最该注意的是箭头的方向——四条规矩不是平级的清单，是有先后逻辑的。先喂规范（没有规范，AI 全靠猜），再精准提需求（没有边界，AI 给的是通用答案），生成后校验（不校验，幻觉直接进代码），最后沉淀模板（不沉淀，下次又从零开始）。跳过任何一条，后面的效果都会打折。很多人用了 AI 觉得「不好用」，回头一看，往往是四条里漏了一两条——比如只精准提示但不校验，或者只校验但不喂规范，效果自然上不去。

几条用下来的判断

把上面这些用熟之后，再回头看我日常怎么用 AI，其实就收敛成几条朴素的判断。

会用的前提是会把上下文喂全。 写后端时先把实体类贴给它再要 Service/Controller，它能把字段对上；写前端时先贴组件依赖和样式规范，生成的代码才贴合项目。上下文给得越足，输出越靠谱。这一步偷懒，后面全是返工。

喂上下文有个度要把握:不是越多越好,是越相关越好。把整个项目代码库都贴给 AI,它的注意力会被无关信息稀释,反而抓不住重点。我的做法是「精准喂」——写订单接口,就贴订单实体、订单 Mapper、相关的 DTO 和异常类,不贴用户、商品那些不相关的。AI 的上下文窗口是有限的,塞满了无关内容,真正该参考的反而被挤出去了。精准喂比全量喂效果好,这个经验是用出来的,不是猜的。

还有一个关于上下文的实操经验:用文件而不是粘贴。Claude Code 能直接读项目文件,你告诉它「先读 src/main/java/…/Order.java 和 OrderService.java」,比把代码粘贴到对话里好——文件方式它能读到最新的、完整的,粘贴容易漏掉或截断。能用文件就别粘贴,这是 Claude Code 这类 agent 相比纯聊天模型的最大优势。

调试报错这件事，AI 比翻博客快。

调试报错这件事，AI 比翻博客快。 遇到奇怪的报错，把报错信息和相关代码一起贴给它，让它分析原因甚至给修改方案。之前我有个 Vue3 响应式数据异常，自己排查了半个多小时没头绪，丢给 Cursor，它一下指出来是我把 ref 和 reactive 用混了，改完就好。这种时候它的价值不是替你思考，而是在你卡住的地方换一个视角。

重构旧代码丢给它，但判断留给自己。 接手别人的烂代码、或自己以前写的烂代码，让它帮你提取重复逻辑、简化命名、补注释，速度比自己手撸快得多。但"哪些改动是安全的、哪些动了会出事"，这个判断还得你自己下。

重构用 AI 还有个特别适合的场景:大面积但机械的重构。比如「把全项目的 var 换成 let」「给所有 Service 方法补上 @Transactional 注解」「统一异常处理风格」。这类改动量大但逻辑简单,人工做又烦又容易漏,丢给 AI 按规则批量改,它做得又快又全。但「把回调改成 async/await」这种涉及控制流变化的重构,就不能完全信 AI——它可能改对了语法,但改错了执行顺序(异步变同步、并发变串行),这种语义变化 AI 不一定能察觉。机械重构放心交,语义重构要人盯。

调试报错这件事，AI 比翻博客快。 遇到奇怪的报错，把报错信息和相关代码一起贴给它，让它分析原因甚至给修改方案。之前我有个 Vue3 响应式数据异常，自己排查了半个多小时没头绪，丢给 Cursor，它一下指出来是我把 ref 和 reactive 用混了，改完就好。这种时候它的价值不是替你思考，而是在你卡住的地方换一个视角。

最后一条也是最重要的一条：核心逻辑别外包给 AI。 它能帮你写重复代码、补全语法，但核心业务逻辑和架构设计，必须你自己把控。原因不是 AI 不够强，而是它没有你的业务上下文，做出来的"对"是通用意义上的对，不是你项目里的对。 把判断外包出去，短期省事，后期维护时全是要还的债。

这条规矩还有个更深的含义:核心逻辑不只是「写得对」,更是「想得对」。架构怎么分层、边界划在哪、哪些操作要事务保证、哪些数据要隔离——这些决策,是在你脑子里想清楚的,不是 AI 生成的。AI 可以帮你把想清楚的东西快速落地成代码,但「想清楚」这一步,它替代不了。如果你连「想清楚」都丢给 AI,那 AI 给的就是它「猜」的方案,你既没法判断它对不对(因为你没想过),也没法在它错的时候纠偏(因为你不知道对的该是什么)。核心逻辑外包,本质是放弃了「想」的责任,这才是最危险的。

怎么判断「这段算不算核心逻辑」?我有个简单的标准:这段代码如果错了,会不会造成数据丢失、越权、资金损失。会,就是核心逻辑,自己写或至少自己逐行过;不会(比如样式、格式化、纯展示),放心交给 AI。按风险分级,把注意力压在出错代价最高的地方。

结尾：不是替代，是重心转移

绕回开头的那个判断。

AI 编程改变的不是"程序员写代码快不快"，而是写代码这件事的门槛和重心在挪。语法、样板、模板代码这一层执行活，正在被 AI 接管；人该把精力挪到「想清楚要做什么、怎么做才是对的、边界在哪」上。这个转移对每个开发者都一样，区别只在于你认不认这件事、以及有没有把方法跟上。

幻觉、代码质量这些坑是真的，但它们不是"AI 不行"的证据，而是"用 AI 需要新规矩"的证据——Claude.md、二次校验、精准提示、提示词模板，就是这套新规矩的雏形。

这也是为什么这篇是序章。认知转变、岗位怎么转型、工程化怎么治理、团队怎么协作，这些才是 AI 研发实战真正要展开讲的东西。 后面我会一篇篇接着写，把这套"用 AI 但不被 AI 坑"的方法论铺开。

这里要给后面系列立一个基调:我写的不是「AI 有多强」的赞美诗,是「AI 在真实项目里会出什么问题、怎么兜」的实战记录。赞美 AI 没有意义,它已经够多人夸了;讲清楚它的边界、它的坑、用它的规矩,才是对同行真正有用的东西。后面每一篇,都会围绕「这个环节 AI 帮了什么、又添了什么乱、我们怎么治」展开——不回避问题,也不否定价值,如实记录,这是这个系列的写法。
