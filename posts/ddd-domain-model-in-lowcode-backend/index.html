低代码平台的后端，如果只按「表结构 + Controller + Service」平铺，业务规则很快会散得到处都是——同一个规则在三个地方写了三遍还各不相同。这篇以 luban-backend 的实现为蓝本，讲 DDD 的领域模型思维怎么把这个烂摊子收起来：限界上下文怎么划、聚合根怎么封装不变量、领域异常怎么统一、ArchUnit 怎么把分层锁成测试。

项目归属说明：本文 DDD 建模经验源自大厂时期主导的内部系统"鲁班"，代码实现以个人开源项目 OpenLuban（架构同源、代码全新重写）为准。

一、为什么低代码后端需要领域模型

低代码后端的核心职责可以概括为四块:

  • 内容模型:站点(Site)与页面(Page + PageSchema)的创建、发布、按路径访问。
  • 身份与权限:用户(User)与认证(登录、/auth/me)、基于角色的访问控制(RequireUser / RequireAdmin)。
  • 系统配置:系统设置(SystemSettings)的读写与缓存(如 Redis)。
  • 对外能力:按站点 slug + 路径对外暴露已发布页面,无需鉴权(供官网等消费)。

若仅按「表结构 + Controller + Service」平铺,很容易出现三类问题:

  • 业务规则散落在各处,修改一处忘记另一处,时间一长就出现"同一个规则在三个地方写了三遍、还各不相同"的失控。
  • 错误语义不统一,前端/BFF 难以稳定处理——同一个"资源不存在",有的接口返回 500,有的返回 404,有的返回 200 带个 error 字段。
  • 扩展新资源或新上下文时边界模糊,影响可维护性——新功能该放哪个包、能不能依赖那个 Service,全靠拍脑袋。

引入 DDD 的领域模型思维(不必强求完整 DDD 框架),可以在保持实现轻量的前提下,获得清晰的边界、一致的业务语义和可演进的 API 契约。下面从战略与战术两方面结合 luban-backend 说明。

“不必强求完整 DDD 框架"这句要展开——很多人对 DDD 的误解是"要么全做要么不做”,觉得得引入四层架构、CQRS、事件溯源全套才算 DDD。其实 DDD 的核心价值在"领域模型为中心"这个思维,不在用了多少模式。luban-backend 用了聚合根、领域事件、ArchUnit 守分层,但没引入事件溯源(用的是 Outbox + 当前状态)、没搞完整的 CQRS(只是读写分表)。挑对你场景有用的模式用,不为模式而模式,这才是务实的 DDD 落地。低代码后端这种 CRUD 为主、偶有复杂状态流转的场景,聚合根 + 事件 + 架构守护这三件套就够用了,硬上事件溯源反而是过度设计。

二、战略设计:限界上下文与能力边界

从业务能力出发,可将低代码后端划分为以下限界上下文(Bounded Context),与当前模块/包结构对应关系如下。

限界上下文职责主要实体与概念对外暴露
站点与页面(Content)站点与页面的生命周期、同站点下 path 唯一、schema 存储Site(聚合根)、Page(实体,属 Site)管理端 CRUD;公开接口按 slug+path 查已发布页
用户与认证(Identity & Auth)用户管理、登录校验、当前用户解析User(聚合根)、角色/状态登录、/auth/me;Header 注入 X-User-ID / X-User-Role
系统设置(Settings)全局配置的读写与缓存SystemSettings(单例式聚合)GET/PUT /settings,仅 Admin
公开访问(Public)按站点 slug + path 返回已发布页面,只读、无鉴权复用 Site、Page,只读、过滤 status=publishedGET /public/sites/:slug/pages?path=…

架构要点有三条:

  • Content 与 Public 共享同一套 Site/Page 持久化模型,但 Public 不写、只按「已发布」读,语义上属于「对外只读能力」,与管理端写操作解耦。
  • Identity & Auth 通过 BFF 注入的 Header 与后端达成契约,后端不签发 JWT,只校验并消费身份,避免与 BFF 的职责重叠。
  • Settings 使用 Redis 缓存(如 settings:global),读多写少,与 MySQL 的持久化形成「写库 + 缓存」的简单 CQRS 式读写路径。

这些边界在代码中体现为:SiteService / PageService、AuthService / UserService、SettingsService、PublicPageService 等应用服务的分工,以及 Controller 按资源与鉴权要求路由到对应服务。

这套上下文划分有个工程上的硬约束:上下文边界靠 ArchUnit 守,不是靠人自觉。ModuleIsolationRulesTest 强制"跨业务模块不得访问对方的 repository/mapper 包"——这条规则的意义不止是整洁,它在为未来拆微服务预留边界。今天 monolith 里守好上下文边界,明天要按上下文拆服务,边界是现成的、清晰的;今天边界糊在一起,明天拆服务就是地狱。限界上下文不只是设计概念,它是架构演进的预备工作。

领域异常:domain-friendly 的设计

BusinessException 有个设计细节值得单独讲:它用原始的 int statusCode,不是 Spring 的 HttpStatus。这是有意为之——为了 domain 包零框架依赖。如果异常类依赖 HttpStatus,那 domain 包就间接依赖了 Spring,ArchUnit 的 domain_should_not_depend_on_spring_or_jakarta 规则就破了。

历史教训写在 ArchUnit 测试的注释里:“历史上 BusinessException 曾带 HttpStatus,已重构为 int statusCode”。这个重构不是为了好看,是为了让 domain 包能被任何上下文(包括非 Spring 的)安全引用。领域层的纯净,是从一个个这样的小取舍里攒出来的——异常类不带框架注解、聚合根不持有基础设施、值对象不依赖外部服务,每一处都在守住"domain 可独立、可测试、可迁移"这条线。

三、战术设计:实体、聚合与不变性

3.1 实体与聚合根:贫血实体 + 充血聚合根两层

代码里其实是两套模型并存:shared/entity/* 是贫血 POJO(纯数据载体,与表一一对应),shared/domain/*Aggregate 是充血聚合根(封装不变量、状态机、领域事件)。实体被聚合根封装,聚合根才是领域逻辑的归属地。

  • Site:id、name、slug、baseUrl、status、seoJson、analyticsJson、createdAt、updatedAt;slug 全局唯一。
  • Page:id、siteId、name、path、status、schemaJson、seoJson、publishedAt、publishedBy、createdAt、updatedAt;同一 site 下 path 唯一。
  • User:id、username、name、role、status、password(不暴露给 API)、createdAt、updatedAt;username 唯一。

聚合根不是简单的"实体加几个方法",它承担的是领域规则的封装。以 PageAggregate 为例,它定义了页面的状态机:draft → published → archived,允许 draft → archived 直跳,published ↔ archived 可来回,published → published 幂等 no-op。状态转换的合法性校验在聚合根内部(isValidPublishTransition),非法转换直接抛 BusinessException.invalidStateTransition()。这个状态机不是写在 Service 里、写在文档里,是写在聚合根里——谁拿到这个聚合根,都绕不过这套规则。

PageAggregate.publish(publishedBy) 这个方法做的事远不止改个 status:它校验状态转换合法性、设置 publishedAt/publishedBy 审计字段、生成 PagePublishedEvent 领域事件。Service 调它,只是触发;规则怎么走,聚合根说了算。这就是"充血"的含义——领域知识沉淀在对象里,而不是散落在 Service 的 if-else 里。

User 聚合根有个设计原则值得讲:聚合根零框架依赖。User 聚合根里不持有 PasswordEncoder(那是基础设施),BCrypt 编码由 Application Service 完成后传入,聚合根只校验"传进来的密码是已哈希的非空串"。这样 domain 包可以完全不依赖 Spring,被任何上下文安全引用。领域层纯净,是 DDD 落地质量的试金石——domain 包要是满地 @Service/@Autowired,那不是 DDD,是披着 DDD 皮的事务脚本。

3.2 业务规则与不变性

不变性(invariant)是 DDD 里聚合根存在的理由——它保证"无论怎么操作,数据始终满足某些约束"。在应用服务层和聚合根里显式落实领域规则,例如:

  • Page 创建/更新:先校验 siteRepository.existsById(siteId),再执行 path 唯一性约束;冲突时抛出 PAGE_PATH_CONFLICT(409)。
  • Site 创建/更新:slug 唯一,冲突时 SLUG_CONFLICT(409)。
  • User 创建/更新:username 唯一,冲突时 USERNAME_CONFLICT(409);密码经 BCrypt 编码后落库。
  • 登录:仅当用户存在、status=active 且密码匹配时返回 user + claims;否则 INVALID_CREDENTIALS 或 USER_DISABLED。

这些规则集中在各自的聚合根和 Service 中,配合 Mapper 的持久化与数据库唯一约束,形成「领域不变性」的多重守卫。

这里有个"防御纵深"的思路值得讲:不变性不是靠一层守的,是靠多层。以 Page 的 path 唯一为例,第一层是聚合根(创建时校验),第二层是 Service(捕获 DB 异常翻译),第三层是数据库的 uk_site_path 唯一索引。三层任何一层都能拦住冲突,为什么叠三层?因为单层都不可靠:聚合根校验有并发漏洞(两个请求同时通过),Service 校验同样;只有数据库的唯一约束是绝对的(并发请求里必有一个会撞约束)。但光靠数据库约束也不行——它的报错是 DataIntegrityViolationException,直接抛给用户看就是 500,得在 Service 层翻译成业务可读的 PAGE_PATH_CONFLICT。聚合根校验 + Service 翻译 + DB 兜底,三层各司其职,缺了哪层体验都差。

User 聚合根的密码处理也体现"分层职责":聚合根只校验"传入的是已哈希非空串",不持有 PasswordEncoder;BCrypt 编码在 Application Service 里做。加密是基础设施关注点,不该污染领域层。这个拆分让聚合根保持纯净,也方便换加密算法(换 BCrypt 为 argon2,只改 Service,聚合根不动)。

3.3 领域异常与统一 API 契约

将「业务失败」建模为领域异常(如 BusinessException),并映射到统一的 HTTP 状态码与错误体,是领域模型对外呈现的重要一环。

luban-backend 的做法分三层:

  • 异常类型:BusinessException,携带 httpStatus、code、message、details。
  • 工厂方法:siteNotFound()、pageNotFound()、userNotFound()、pagePathConflict()、usernameConflict()、slugConflict()、invalidCredentials()、userDisabled()、unauthenticated()、permissionDenied()、invalidArgument(message)、invalidStateTransition(from,to)、leadInvalidTransition(from,to)、quotaExceeded(msg)、shortLinkInactive() 等,对应文档中的错误码表,共 50+ 个工厂方法覆盖所有上下文。
  • 全局处理:@RestControllerAdvice 将 BusinessException 转为 { "code", "message", "details" } 的 JSON 响应,与 BFF/前端约定一致。

这样,领域层的「资源不存在」「冲突」「未认证」「无权限」等语义,都能以稳定、可编程的方式传递到调用方,便于前端/BFF 做统一错误处理与用户提示。

这套异常体系有个连带收益:它是活的错误字典。50 多个工厂方法,每个对应一类业务错误,想看后端有哪些业务失败场景,翻一遍 BusinessException 的工厂方法就清楚了,不用满代码库搜 throw new。新增一个业务错误,先在 BusinessException 加工厂方法、登记错误码,再在代码里抛——这个顺序保证错误码集中管理、不散落成魔法字符串。集中管理错误码,和集中管理 API 契约是同一个思路:单一信源,谁改谁先改字典。

四、应用服务与分层

分层采用「Controller → Application Service → Repository(接口) → RepositoryImpl(封装 Mapper)/ Projection / Port → Mapper」的结构。这里的关键是 Repository 是领域层定义的接口,实现在基础设施层——依赖倒置,领域层不直接碰 Mapper。

  • Controller:负责 HTTP 入参解析、鉴权结果消费(如从 UserContext 取 userId/role)、调用应用服务并返回 DTO。
  • Application Service(*Service):编排用例、加载聚合根、调用聚合根方法、持久化、发布事件。不写领域规则(规则在聚合根),只做编排。
  • Repository(接口):领域层定义的抽象,如 PageRepository.findById(id, siteId): Optional<PageAggregate>。领域层只面向接口编程,不关心是 MySQL 还是别的什么。
  • RepositoryImpl / Projection / Mapper:基础设施层,封装 MyBatis Mapper,实现 Repository 接口。

以 PageService.create 为例,看编排怎么写:

@Transactional(rollbackFor = Exception.class)
public PageResponse create(String siteId, ...) {
    if (!siteRepository.existsById(siteId)) throw BusinessException.siteNotFound();  // 先校验站点存在
    PageAggregate agg = PageAggregate.newPage(...);   // 聚合根工厂,规则在工厂里
    try {
        pageRepository.save(agg);
    } catch (DataIntegrityViolationException e) {
        if (isDuplicate(e)) throw BusinessException.pagePathConflict();  // DB 唯一约束兜底
        throw e;
    }
}

这里有个细节:siteRepository.existsById(siteId) 用的是轻量 SELECT 1,不是把整个 Site 加载出来。存在性校验要用最便宜的查询,把整个聚合根加载出来只为判存在,是浪费。path 唯一性靠 DB 的 uk_site_path 唯一约束兜底,Service 层抓 DataIntegrityViolationException 翻译成 pagePathConflict()——应用层校验可能漏(并发),DB 唯一约束绝不会漏,两者结合才是完整的。

草稿与发布分离:published_pages 投影

PublicPageService(对外读)读的不是 pages 表,是 published_pages 发布快照表。发布时,PageService.publish() 在同一事务里做两件事:调聚合根的 publish() 改状态,再调 PublishedPageProjection.upsertOnPublish() 把发布内容写到快照表。

为什么要事务内双写?因为 published_pages 是 pages 聚合的读侧投影(C 端公开访问的物化视图),不是另一个聚合。它的写入必须与 pages 在同一事务内完成,不能用 AFTER_COMMIT 事件异步写——否则会出现"发布成功了但对外还看不到"的窗口期。投影协作者独立于 PageRepositoryImpl,是因为它的写入时机和聚合根强绑定,但又不该塞进 Repository(那是持久化,不是投影)。

这种草稿/发布分离在对外站点场景特别重要:运营在 pages 表改草稿,线上展示的是 published_pages 快照,互不干扰。CQRS 的读模型分离,在这里落到了两张物理表上。

五、与 BFF 的契约与鉴权

低代码后端与 BFF 的协作遵循「契约先行」:

  • Base path:/backend,健康检查可为 /ping 或 /backend/ping。
  • 鉴权:受保护接口依赖 Header X-User-ID(必填)、X-User-Role;缺失则 401(UNAUTHENTICATED);需 Admin 的接口再校验 role,否则 403(PERMISSION_DENIED)。
  • 错误体:统一 { "code", "message", "details" },与 API 文档 一致。
  • 请求/响应:JSON 字段 camelCase,时间 ISO 8601;登录响应为 user + claims,不返回 password。

后端不关心 JWT 的签发与解析,只信任 BFF 注入的 Header,从而将「认证」留在 BFF、「授权」留在后端的领域规则(RequireUser / RequireAdmin)中,职责清晰。

鉴权这块还有个按 method 区分的细节:/backend/sites 的 GET 普通用户能读,但 POST/PUT/DELETE 需 Admin;/backend/settings 任何方法都需 Admin。不是简单的"需要登录 vs 需要 admin"二分,是按资源 + 操作粒度控制。判断 admin 时只认 role == 'admin',严禁用 username == 'admin' 猜——这条注释写得明明白白,因为靠用户名猜权限是安全事故的温床。

用 ArchUnit 把 DDD 规则锁成测试

DDD 落地最大的敌人不是"不会设计",是"设计完守不住"——三个月后有人为了快,在 Controller 里直接调 Mapper、在 domain 包里塞个 @Service,分层就破了。光靠 code review 盯,迟早漏。

我们的解法是用 ArchUnit 架构测试,把 DDD 规则写成可执行的断言。8 个测试类,核心的几条:

  • domain_should_not_have_framework_annotations:domain 包禁 @Service/@Component/@Repository 注解
  • domain_should_not_depend_on_spring_or_jakarta:domain 禁依赖 Spring/Jakarta 任何类型(注释:“历史上 BusinessException 曾带 HttpStatus,已重构为 int statusCode”)
  • aggregates_should_be_final:*Aggregate 类必须 final(防继承破坏不变量)
  • publicside_should_not_depend_on_operatorside:C 端禁依赖运营端(安全边界)

这些规则一破,测试直接红,CI 过不去。DDD 的分层不是靠文档和约定,是靠测试强制执行——这才是"工程化落地"和"PPT 上的 DDD"的区别。ArchUnit 把架构判断变成了代码,谁破坏边界,构建就教谁做人。

三侧分包:CQRS 的物理隔离

源码顶层按 publicside / operatorside / shared 三侧分包,这不是为了好看,是 CQRS 的物理隔离:

  • operatorside(运营写端):需要鉴权的管理请求,/sites、/pages、/leads、/users 等。19 个 Controller。
  • publicside(C 端只读端):公开免鉴权的请求,/public/**、留资提交。读 published_pages,不写。
  • shared(共享):纯领域层 + 基础设施,被两侧共同依赖。聚合根、实体、Mapper、异常都在这。

关键的边界规则是 publicside 禁止依赖 operatorside。这是安全边界——C 端(对外、免鉴权、最易被攻击)绝不能引用运营端的任何代码,否则攻击面会顺着依赖链蔓延。这条规则不光靠人记,靠 PublicsideIsolationTest 守着,ArchUnit 一旦发现 publicside 依赖了 operatorside,测试就红。安全边界要用测试守,不能靠自觉。

六、小结与演进建议

以 luban-backend 为例,DDD 在低代码后端中的应用可以概括为三层:

  1. 战略上:按「站点与页面」「用户与认证」「系统设置」「公开访问」划分限界上下文,明确各上下文的职责与对外能力。更进一步,用 publicside / operatorside / shared 三侧分包做 CQRS 的物理隔离——C 端只读、运营端写、共享领域,边界落到包结构上。
  2. 战术上:用充血聚合根(Site、Page、User 等 11 个 *Aggregate)封装状态机和不变量;用应用服务编排用例、加载聚合根、持久化;用领域异常 + 统一错误体表达业务失败。聚合根是领域知识的归属地,不是贫血的数据袋。
  3. 工程上:用 ArchUnit 架构测试把分层规则锁死(domain 零框架依赖、聚合根必须 final、publicside 禁依赖 operatorside);用事务性 Outbox 保证领域事件不丢;用 published_pages 投影做草稿/发布分离。这些都是"设计完怎么守住"的工程手段,不是 DDD 理论本身。

这套落地的核心心得:DDD 不是理论考试,是工程实践。会画限界上下文图、会说聚合根实体值对象这些概念,不算落了 DDD;真正落了,是 domain 包零框架依赖、是聚合根封装了状态机、是 ArchUnit 守住了分层、是事件不丢。判断一个项目的 DDD 是真落地还是 PPT,看 domain 包干不干净就行——干干净净没有 Spring 注解、聚合根是 final、事件有完整投递保障,那才是真做了;domain 包里塞着 @Service、Controller 直接调 Mapper、领域逻辑散在 if-else 里,那就是披着 DDD 皮的事务脚本。

后续若业务扩展,可考虑三个方向:

  • 在聚合内引入更丰富的值对象(如 Path、Slug、PageStatus),进一步收口校验逻辑。
  • 对「页面发布」等关键操作发出领域事件,供审计、搜索或缓存失效使用。
  • 在保持与 Go 版 API 兼容的前提下,将公开读路径(Public)与管理端写路径在存储或缓存策略上进一步分离,以支撑更大流量。

不过这几个方向,有些我们已经落地了——领域事件和读路径分离都已经做进去了。领域事件这块,我们用的是事务性 Outbox 模式,不是简单的 Spring 事件。这部分单独展开。

领域事件:事务性 Outbox,不是"未来再说"

前面把领域事件列在"未来方向"里,其实它已经完整落地了,而且用的是生产级的事务性 Outbox方案,不是简单的 ApplicationEventPublisher.publishEvent 就完事。

聚合根产生事件,Service 在事务提交后发布,监听器消费。这套说起来轻巧,但有个分布式系统的经典难题:事务提交和事件投递,怎么保证都成功? 如果先提交事务再发事件,事务成功了事件发失败,事件丢了;如果先发事件再提交事务,事件发了事务回滚,下游收到一个不该有的事件。两阶段提交太重,Spring 的事件机制又不跨事务。

我们的解法是 DomainEventPublisher 双写 + Outbox 补偿。publishAll 在同一事务内做两件事:① 调 applicationEventPublisher.publishEvent(触发 AFTER_COMMIT handler,即时 best-effort);② 调 outboxMapper.insert(事务内落盘到 domain_outbox 表,保证事件不丢)。事件表和业务表在同一个事务里,要么一起成功要么一起回滚,不会出现"业务成了事件没了"。

光落盘不够,还得投递。OutboxRelayScheduler 是个定时任务,每 5 秒扫一次 domain_outbox 里 processed_at IS NULL 的记录(每批 50 条),反序列化成事件再投递。投递成功标记已处理;失败则 attempts+1,指数退避(2 的 attempts 次方秒,封顶 1 小时);超过最大重试 5 次进死信(标记已处理,人工介入)。幂等由消费端保证——relay 可能重投同一事件,消费端必须能处理重复。

这套设计的取舍很明确:用一张 outbox 表 + 一个补偿调度器,换来"事件不丢"的可靠性。代价是多了一张表、多了一个调度任务、消费端要做幂等。对低代码平台这种"发布页面要触发缓存失效、SEO 刷新、短链更新"的场景,这点代价完全值得——丢一个发布事件,线上就会出现"页面发了但旧内容还在"的故障。

草稿即领域模型:schema 的另一种读法

低代码后端最容易被忽略的是领域建模——大家忙着做"配置驱动",忘了配置背后的领域逻辑。

我们在低代码后端里落 DDD 的做法:schema 就是领域模型的序列化。设计器拖拽出的"订单表单",是"订单"这个聚合根的领域模型——它的字段、校验规则、状态流转,都是领域知识。

具体落地:

  • 聚合根 = schema 的根节点(订单 schema 是聚合根,订单明细 schema 是聚合内的实体)
  • 不变量 = schema 校验规则(订单总金额 = 明细金额之和,这是不变量,设计器强制校验)
  • 领域服务 = 后端处理器(跨聚合的操作,比如"下单 + 扣库存",走领域服务)

这套读法的好处是,低代码的 schema 不再是"一堆配置",而是有领域语义的结构化模型。改 schema 等于改领域模型,所以 schema 的变更要走和领域模型一样的严谨度——版本化、可回滚、可审计。我们后端的 lowcode schema 接口有 draft/publish/rollback 和版本历史,就是因为把 schema 当领域模型对待。
