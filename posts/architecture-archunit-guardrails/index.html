Controller 直连数据库怎么治:用 ArchUnit 把架构钉成测试

做过几年架构的人都遇到过这个场面:某天 code review,发现一个 Controller 直接 @Autowired 了 JdbcTemplate 在查数据库,绕过了 Service。问起来,答「这次比较急,先这么写」。

架构图贴在墙上,分层写得清清楚楚。可代码是活的,它会随每一次「图方便」悄悄偏移。等你发现时,Controller 已经散落在各处直连数据库、两个业务模块缠成了死结、循环依赖改都改不动。

靠人盯,盯不住的。 这篇就讲我怎么把架构边界写成测试,让机器来盯。

一、架构腐化的三种典型

不是有人故意搞破坏,都是「这一次图方便」累积出来的。我见得最多的三种:

跨层调用最常见。Controller 本来只该做路由和参数校验,但有人嫌 Service 套一层麻烦,直接把 JdbcTemplate 拉进来查库。一次两次没事,等这个模式扩散开,分层就成了摆设。

模块缠绕更要命。我们项目有十几个业务模块——tenant、trade、circle、marketing…本来该各管各的。但今天 tenant 模块需要算一笔佣金,顺手 import 了 trade 的 Repository;明天 trade 又用了 tenant 的工具类。半年后想拆微服务,发现拆不动,因为没人记得清谁依赖谁。

循环依赖是缠绕的极端形态。A 依赖 B,B 又反过来依赖 A,编译能过(运行时可能爆),但删一个就崩一串。这种债最难还。

循环依赖还有个特别恶心的变体:间接循环。A 不直接依赖 B,而是 A 依赖 C,C 依赖 B,B 依赖 A。编译能过,但依赖图上 A→C→B→A 形成了环。这种循环肉眼很难发现(你要同时看三个文件才能拼出这个环),但 dependency-cruiser 的 no-circular 规则一扫就出来。这也是为什么架构守护要靠工具不能靠人——人能发现「A 直接依赖 B」的显式循环,但「A→C→B→A」的间接循环,人眼根本拼不出来,工具能。

这三种腐化还有一个共同特点:它们都是「温水煮青蛙」。单看每一次违规,都很小——「就这一次图方便,下次不这样了」。但「下次」永远不会来,下一次还是图方便。等你发现架构烂了,已经积了几百处违规,还债的工作量大到没人敢动。架构守护的价值,就是在「第一次图方便」时就拦住,不让温水开始煮。防微杜渐说起来老套,但在架构腐化这件事上,它是唯一有效的策略。

二、先解决一个现实问题:存量怎么办

引入架构测试,第一个拦路虎是存量。

我们项目引入 ArchUnit 的时候,代码已经写了两个月了。一跑,Controller 直连数据层的违规 467 处,业务层用 SQL 注解的 84 处。要是让测试直接 fail,CI 当场全红,谁也推不动。

所以我们用了 ArchUnit 的 FreezingArchRule——冻结存量、只拦新增。

机制很简单:把规则包一层 Frozen.frozen(rule),首次运行时把所有存量违规记进一个基线文件(入库),之后只对新增违规报错。这样老代码不挡路,但谁再写新的越界代码,CI 立刻拦下来。

public final class Frozen {
    public static ArchRule frozen(ArchRule rule) {
        return FreezingArchRule.freeze(rule);
    }
}

我们那次首跑冻结了 551 处存量违规,基线文件入库。从那以后,存量违规只能减不能增——还一处就删一处基线,代码和基线一起收敛。这是老项目引入架构守护唯一可行的姿势:不追求一步到位,先把增量锁死。

「还一处删一处」这个流程有个实操经验:把还存量当成独立任务,不要夹在功能 PR 里。功能 PR 的重点是「把功能做出来」,夹带还存量会让 PR 的 diff 变大,review 时功能改动和存量修复混在一起,看不清哪行是功能、哪行是还债。我们的做法是:存量修复单独开 PR,标题标明「[架构债] 还 XXX 违规」,review 时只看「基线删的条目和代码改动对不对」,简单明了。把还债和功能分开,既不让还债拖慢功能,也不让功能掩盖还债。

还有一个关于 freeze 的认知要纠正:freeze 不是「永久豁免」,是「暂时记着,迟早要还」。有些团队 freeze 之后就把存量忘了,基线文件越来越大,违规永远不还。freeze 的正确用法是「冻结 + 定期还债计划」——比如每个迭代分配 10% 的精力还存量,基线文件逐月缩小,最终目标是「基线归零,所有违规清完」。虽然这个目标可能永远达不到(新规则又会冻结新存量),但「基线在缩小」这个趋势要保证,否则 freeze 就变成了「合法的技术债」。

三、后端 ArchUnit:五类规则守住边界

我们项目的后端,架构规则分五类,共 14 条。我挑关键的讲。

第一类,分层依赖(ARCH-LAYER)。 最核心的两条:

// Controller 不得直接访问 Repository / Mapper / JdbcTemplate
@ArchTest
static final ArchRule ARCH_LAYER_01Controller不访问数据层 =
    Frozen.frozen(noClasses()
        .that().haveSimpleNameEndingWith("Controller")
        .should().accessClassesThat(
            simpleNameEndingWith("Repository")
                .or(simpleNameEndingWith("Mapper"))
                .or(type(JdbcTemplate.class)))
        .as("Controller 不得直接访问 Repository / Mapper / JdbcTemplate")
        .because("数据访问须经 Service 委托,Controller 仅负责路由与参数校验"));

// @Transactional 只能在 Service 层
@ArchTest
static final ArchRule ARCH_LAYER_02Transactional仅Service =
    Frozen.frozen(noClasses()
        .that().haveSimpleNameNotEndingWith("Service")
        .should().beAnnotatedWith(Transactional.class)
        .as("@Transactional 只能在 Service 层使用")
        .because("事务边界统一在 Service 层管理"));

注意每条规则都带 .as() 和 .because()。规则本身就是文档——违反时测试报告直接打印中文说明和理由,新人一看就懂为什么挂了,不用去翻规范文档。

第二类,模块隔离(ARCH-ISO)。 这类是为微服务拆分预留边界的:

// tenant 模块不得依赖 ops 模块(C 端与运营端解耦)
@ArchTest
static final ArchRule ARCH_ISO_01tenant不依赖ops =
    Frozen.frozen(noClasses()
        .that().resideInAPackage("com.example.tenant..")
        .and().resideOutsideOfPackage("com.example.tenant..controller.ops..")
        .should().dependOnClassesThat().resideInAPackage("com.example.ops..")
        .as("tenant 不得依赖 ops(controller.ops 运营 Controller 白名单豁免)"));

还有一条更狠的:13 个业务模块之间,不得访问对方的 repository/mapper 包。用自定义 ArchCondition 实现——遍历每个类的依赖,一旦发现跨模块调到别人的数据访问内部包,就报违规。这条规则等于在代码层画了 13 个圈,圈里的数据只能圈里的人碰。

第三类,数据访问纪律(ARCH-DATA)。 这是 Q4 阶段加的,为微服务拆分铺路:

// 业务层(非 Mapper/Repository)不得调用 JdbcTemplate
@ArchTest
static final ArchRule ARCH_DATA_01业务层不调JdbcTemplate =
    Frozen.frozen(noClasses()
        .that(BUSINESS_LAYER)
        .should().accessClassesThat(type(JdbcTemplate.class))
        .as("业务层不得调用 JdbcTemplate")
        .because("JdbcTemplate 只能在数据访问层(为微服务拆分预留边界)"));

这条第一次跑,命中 467 个文件。但用 freeze 冻结后,新写的业务类再想直连 JdbcTemplate,直接被拦。

第四类,命名规范(ARCH-NAMING)。 这类看着琐碎,但很有用——命名即契约:

  • @RestController 的类必须以 Controller 结尾;
  • 以 Controller 结尾的类必须有 @RestController 注解。

这两条是对称校验,双向堵:防止有人起名叫 XxxController 却忘了加注解,也防止有人加了注解却忘了改名字。

第五类,安全门禁(ARCH-SAFE)。 两条,都来自真实事故:

// 定时任务所在类不得调用 TenantContext.getTenantId
@ArchTest
static final ArchRule ARCH_SAFE_02定时任务不调TenantContext =
    Frozen.frozen(noClasses()
        .that(DECLARES_SCHEDULED_METHOD)
        .should().callMethod(TenantContext.class, "getTenantId")
        .as("@Scheduled 所在类不得调用 TenantContext.getTenantId")
        .because("定时任务无 HTTP 上下文,getTenantId 必为 null"));

这条是因为踩过坑:定时任务里调 TenantContext.getTenantId(),但定时任务没有 HTTP 上下文,拿到的是 null,导致一批数据写错了租户。踩过的坑固化成规则,这就是「事故即规范」的架构层落地。

安全门禁这类规则有个特点:它不是「架构规则」,是「业务安全规则」。分层依赖、模块隔离是纯架构的,和业务无关;但「定时任务不调 TenantContext」是多租户业务的安全约束,离开了多租户场景就没意义。把业务安全约束也写进 ArchUnit,是很多人没想到的用法——ArchUnit 不只能守架构边界,还能守业务边界。只要这个约束能表达成「某类代码不能调用某个方法」,就能写成 ArchUnit 规则。这个认知打开了架构守护的应用面:不只是「Controller 不碰数据库」这种架构规则,「定时任务不碰租户上下文」「支付回调不碰订单状态机」这种业务安全规则,也能机器化守护。

四、前端:dependency-cruiser 守依赖方向

后端有 ArchUnit,前端对应的是 dependency-cruiser。它干的事一样——把依赖方向写成规则,让机器查。

前端的腐化方式和后端不太一样,最典型的是底层反向依赖业务层。比如 src/api/(封装接口请求的底层)不知什么时候 import 了 src/pages/(业务页面),底层的就不再「底层」了,改一处动全身。

我们项目前端用 dependency-cruiser 守了 7 条,核心逻辑是给每个目录定一个层级,只许上层依赖下层,不许反过来:

对应的规则很直白:

// api 是底层,禁止反向依赖业务层
{
  name: 'api-is-leaf',
  severity: 'error',
  comment: 'src/api/ 是底层,禁止反向依赖业务层',
  from: { path: '^src/api/' },
  to: { path: '^src/(pages|components|stores|composables|domain)/' }
},
// 禁止循环依赖
{
  name: 'no-circular',
  severity: 'error',
  from: {},
  to: { circular: true }
}

api、utils、constants 是叶子节点——它们不能反过来依赖业务层。composables 不能依赖页面。这些规则配上 no-circular(全局禁循环依赖),基本把前端的分层钉死了。

还有一条很妙的——E2E 双线隔离:

// e2e/h5 不能 import e2e/miniprogram,反之亦然
{
  name: 'e2e-h5-not-miniprogram',
  severity: 'error',
  comment: 'e2e/h5 不能 import e2e/miniprogram',
  from: { path: '^e2e/h5/' },
  to: { path: '^e2e/miniprogram/' }
}

我们项目是同一套代码出 h5 和小程序两端,E2E 测试也分两套。这条规则防止 h5 的测试脚本偷偷引用了小程序的,导致测试串味。这种细分场景的规则,是真刀真枪用出来的,不是抄模板。

跑法上,前端用了 severity 分级模拟后端的 freeze:error 级拦新增(循环依赖、双线隔离),warn 级容存量(api-is-leaf)。配套命令 npm run arch:test 守护、arch:report 还能生成 dot 架构图可视化。

五、eslint 自定义规则:补 dependency-cruiser 管不到的

dependency-cruiser 管的是文件间依赖,但有些约定它管不了,得靠自定义 eslint 规则。我们项目前端写了 5 条 app/ 命名空间的规则:

  • app/no-raw-uni-request:禁止直调 uni.request,只能走统一的 http.ts。白名单就两个文件。这是防接口调用散落各处、绕过统一鉴权和错误处理。
  • app/require-data-e2e:交互元素(button、带 @click 的)必须带 data-e2e 属性——给 E2E 测试留选择器。
  • app/no-chinese-class:禁止 CSS class 含中文。踩过坑,uni-app 的 WXSS 编译会直接失败。
  • app/no-emoji:全仓禁 emoji,UI 和标识符里都不能有。
  • app/max-vue-file-lines:.vue 文件不超过 500 行,超了就 warning。

这几条看着杂,但每条背后都是踩过的坑。架构守护不全是高大上的分层,也包括这些土但管用的小钉子。

自定义 eslint 规则还有个值得说的经验:规则要能自解释。app/no-raw-uni-request 这条规则触发时,eslint 报错信息要写清楚「禁止直调 uni.request,请使用 http.ts 统一封装,原因:绕过统一鉴权和错误处理」。开发看到报错,不用去查文档就知道「为什么禁、该用什么替代」。如果报错信息只写「禁止直调 uni.request」,开发不知道该用什么替代,要么绕过规则(加 eslint-disable 注释),要么来问人,两种都增加沟通成本。规则的报错信息就是最好的文档——它在最需要的时候(违规的那一刻)出现,比任何 wiki 都及时。

六、为什么 AI 时代更需要架构守护

最后说个时机问题:为什么我现在特别强调架构守护?

因为 AI 写代码,不认架构边界。

AI 优化的目标是「让这个功能跑通」,它不在乎 Controller 该不该碰数据库、不在乎 tenant 该不该依赖 ops。给它自由,它会选最短路径,而最短路径往往就是越界。

我们项目 4 个人 + AI,日均 14 次提交。如果靠人 review,根本看不过来 AI 生成的东西越没越界。有了架构测试,AI 写的越界代码提交时 CI 就挂,根本进不来。 这是 AI 协同时代,架构守护从「锦上添花」变成「必需品」的原因。

AI 不认架构边界,还有个更深层的原因:AI 的训练语料里,「最佳实践」和「最短路径」是混在一起的。它见过大量「Controller 直接查库」的代码(因为确实有人这么写),也见过「Controller 走 Service」的代码。当它生成代码时,两种模式它都可能选,取决于哪个「看起来更省事」。没有架构守护,AI 选最短路径(直连数据库)的概率不低;有了架构守护,选了最短路径就被拦,它只能退回来选正确的路径。架构守护在这里的作用,是「把 AI 的选择空间从「包含错误路径」收窄到「只包含正确路径」」,让它即使想偷懒也偷不了。

七、收尾:约定要能被机器检查,才叫约定

回头看这套东西,核心就一句话:写进文档的约定会随时间失效,写进测试的约定不会。

架构图会过时,规范文档没人看,周会强调没人记。但 ArchUnit 测试和 dependency-cruiser 规则,每次 mvn verify、每次 npm run arch:test 都会跑。约定一旦变成机器能检查的规则,它就不会被人忘掉。

而且这套东西和团队大小无关——4 个人需要,40 个人更需要。区别只在于:团队越小,越没人力 review,越得靠机器盯。这套架构守护,与其说是技术决策,不如说是「让小团队能跑起来」的组织决策。

如果你也在为一个不断膨胀的项目头疼,我的建议是:别急着画新架构图,先把你最在意的边界写成一条架构测试。先拦住新增,再慢慢还存量,这条路走得最稳。

这条建议有个更朴素的说法:架构守护的起点,是一条规则,不是一张图。很多人想做架构治理,第一反应是画架构图、写架构文档、开架构评审会——这些都有用,但都不是最有效的。最有效的是写一条 ArchUnit 规则,让它每次构建都跑。一条规则能拦住的违规,比十次架构评审会都多——因为规则是每次构建都跑的,比人盯可靠得多,评审会一周才一次根本不够。从一条规则开始,尝到「机器替你盯」的甜头,再慢慢加第二条、第三条,架构守护就自然铺开了,团队也自然接受了。

深度展开:ArchUnit 的规则分层与演进

ArchUnit 的规则不是"越多越好",是要分层、抓重点。我们把规则分成三层,从粗到细,每一层解决不同的问题。

第一层是分层依赖规则,这是「必须有」的底线。Controller 只能依赖 Service,Service 只能依赖 Repository,Repository 不能反向依赖 Controller。这是架构的地基,违反就是灾难——Controller 直连数据库意味着「路由层和数据层耦合」,改数据层要动路由层,改路由层怕碰数据层,整个分层形同虚设。这层规则不需要多,守住「依赖方向单向」就够了,但必须 100% 执行,零容忍。这层规则是架构守护的「1」,没有它后面的都是「0」。

第二层是模块隔离规则,这是「应该有」的进阶。订单模块不能直接 import 用户模块的内部类,必须走公开接口。这层防的是"模块边界被侵蚀"——模块间的依赖如果不约束,会从「通过接口调用」慢慢退化成「直接 import 内部实现」,模块的封装性被打破,微服务拆分时发现拆不动。这层规则的数量取决于模块数量,模块越多规则越多,但每条规则的逻辑都一样:模块间只许通过公开 API 交互,不许碰内部实现。

第三层是命名规范规则,这是「锦上添花」的补充。Service 必须以 Service 结尾,Controller 必须在 controller 包下。这层防的是"新人不知道规矩乱放"——命名不一致时,ArchUnit 的规则匹配会失效(规则按命名找 Controller,你起名叫 XxxHandler 就匹配不到,分层规则就管不住它)。命名规则保证了「命名即契约」——叫 Controller 的就是路由层,叫 Service 的就是业务层,规则可以按命名精确匹配。这层看着琐碎,但它是前两层规则能生效的前提:命名不规范,按命名的规则就匹配不到,分层和隔离就形同虚设。

这三层不是一开始全上的,是逐步演进出来的。第一层(分层依赖)从项目第一天就该有,它是地基;第二层(模块隔离)等模块多到「跨模块依赖开始失控」时再加;第三层(命名规范)等团队大到「新人不知道命名约定」时再补。一上来全上三层,规则太多,团队抵触,推不动;只上第一层不上后两层,架构守护的覆盖面不够,该漏的还是漏。按需演进,每加一层都有明确的触发条件(「模块依赖失控了」加第二层,「命名混乱了」加第三层),这样推进阻力最小。

深度展开:架构守护的维护成本

架构守护不是写完规则就一劳永逸,它有维护成本,而且容易被忽视。

最大的维护成本是基线文件的管理。FreezingArchRule 冻结的存量违规,记在一个基线文件里,这个文件本身需要维护。存量修了一处,要从基线里删一条;基线文件格式错了,测试会报莫名其妙的错;基线文件 merge 冲突(两个人同时修了存量,基线都改了),要手动解决。我们早期没重视基线管理,基线文件越积越乱,最后没人敢动它——动错了测试全红。后来立了规矩:修存量时,PR 里必须同时改基线文件,review 时检查「基线删的条目和代码修的位置对得上」,基线文件和代码同步管理,不能脱节。

第二个维护成本是规则本身的演进。业务变了,规则要跟着变——比如某天把一个模块拆成了微服务,原来「模块不得依赖另一个模块」的规则就没意义了(已经物理隔离了),要删掉;新加了一个模块,要补「新模块不得被老模块直接 import」的规则。规则不是写完就不动的,它要跟着架构演进。我们的做法是:每次架构调整(拆模块、合并模块、加新层)时,同步审查 ArchUnit 规则,该加的加、该删的删、该改的改。架构调整不带规则审查,规则就会和实际架构脱节——规则管着旧架构,新架构的违规管不住。

第三个维护成本是规则误报的处理。有些规则会误报——比如某个特殊的工具类,技术上违反了「Controller 不依赖 Repository」(它 import 了一个 Repository 类型),但语义上是合理的(它只是用了 Repository 的某个常量,没真的查库)。误报如果不处理,要么团队开始忽略规则(「这个报错不用管,误报」),要么给误报加白名单(白名单越来越长,规则千疮百孔)。我们的处理方式是:误报要说明理由、记录在案、定期复审。不是加个白名单就完事,而是在规则注释里写清楚「这个文件为什么豁免、什么时候可以去掉豁免」,让误报有据可查,不是悄悄放行。

深度展开:架构守护与团队文化的配合

架构守护是技术手段,但它能不能落地,最终取决于团队文化。

最关键的文化是:架构测试失败时,默认「代码错了」而不是「规则错了」。很多团队引入架构守护后,测试一红,开发的第一反应是「这条规则太严了,我改一下规则放行」。如果每次都改规则放行,规则就越来越松,最后形同虚设。正确的心态是:测试红了,先假设「我的代码违反了架构约定」,去找正确的写法;只有确认「规则本身不合理」(比如规则过时了、规则有 bug),才改规则。这个「默认信规则」的文化不立起来,架构守护就是摆设。

还有一个文化要点:架构规则要团队共识,不是架构师一个人定的。架构师一个人定的规则,团队不理解为什么这么定,执行时就会抵触——「这条规则太麻烦了,为什么要这么分」。我们的做法是:每条 ArchUnit 规则都带 .because() 说明理由,规则新增时在团队评审时过一遍,让大家理解「为什么这条规则是必要的」。有了共识,执行时大家是「守自己的规矩」,不是「被架构师管着」,抵触情绪小得多。

最后,架构守护要容忍灰度。引入初期,存量违规一堆,不可能一次全修;规则执行初期,难免有边界 case 误报。团队对架构守护要有耐心:存量慢慢还、规则慢慢调、误报慢慢清。追求一步到位,规则太严团队崩;追求松松垮垮,规则太宽没效果。灰度推进,先拦住新增、再还存量,是唯一能长期跑下去的节奏。
