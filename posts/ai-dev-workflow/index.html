【AI 研发实战】我们的 AI 研发工作流:从原型到上线的全链路

这是我 AI 开发阶段的主干实践。前端的工程化纪律、全栈的链路意识,在这一步被重新组织成一条可复用的 AI 研发流水线,后面讲评审、测试、契约的文章,都拿它当地图。

先说我的判断:大多数人用 AI 写代码,还停留在「光标后面补两行」,出 bug 了再问一句「这怎么报错了」。这跟当年用 IDE 自动补全区别不大,只是更聪明了点。

真正让玩法变了的,是把整条研发链路摊开,让 AI 在每一步都进场,把它当成流水线上的工位,而不是高级补全。

全链路总览

整条链路长这样:

  • 确认需求 → HTML 原型 → 计划 → 多轮并行头脑风暴 → TDD 执行(红/绿/重构)→ 自治多 Agent 评审 → 提交/MR。

每一步都不是「问 AI 一句话」那么简单。下面逐个拆开。

第一步:需求之后,先出 HTML 原型

确认需求之后,先出 HTML 原型,而不是直接开写。

原因很直接:对着空气写需求,人和 AI 都容易跑偏。先把界面用 HTML 画出来,需求和实现的歧义在原型阶段就消掉了大半。

第二步:计划阶段,确立唯一事实源

计划阶段是整个工作流里我最强调的一环。

做法是做两轮计划,产出一份任务图 JSON,并约定它是唯一事实源。

这条规矩的本质是解决并行的一致性:每个 agent 各自理解任务、各干各的,很快撞车——你改的文件我也要改,我设计的接口跟你对接不上。任务图摆在中间,谁做什么、依赖谁,一目了然。

没有这个事实源,并行越多越乱。

第三步:多轮并行头脑风暴

然后是多轮并行头脑风暴。

我们有一条近乎硬约束的原则:能开并行 subagent 就尽量多开,团队甚至把「尽可能多地开并行 subagent 执行」写进了规矩里。

两种用法的本质区别在于:

  • 串行问 AI,是把它当搜索引擎。
  • 并行扇出去,是把它当一支团队——同一个问题让多个 agent 同时从不同角度去想,再汇总。

效率和质量都不是串行能比的。

第四步:TDD 执行,红 / 绿 / 重构

执行阶段走 TDD:红、绿、重构。

我的做法是让 AI 先写测试,再看着测试补实现,而不是直接写实现。理由是:

  • AI 写测试比写实现在行——实现容易「幻觉」。
  • 测试要贴合真实行为,反而能反过来约束实现。

质量比「直接写实现」高一个台阶。

第五步:自治多 Agent 评审回路

执行完不是直接提交,而是过一道自治多 Agent 评审回路。这是整个系列的灵魂,细节我单独写了一篇,这里只说它在工作流里的位置。

它是提交前的最后一道闸。评审完、收敛了,还要再过一道硬测试门禁才允许走 MR。

工作流的尾巴不是「人觉得写完了」,而是「流程判定它能进仓库了」。

第六步:MCP 集成,接通项目孤岛

再补一块容易被忽略的拼图:MCP 集成。

我们主力按使用频率是 Claude Code、Cursor、OpenCode,同时通过 MCP 把代码托管、IM、记忆、设计稿这些孤岛接进来。

接通之后,AI 不再是只能看你粘贴代码的「瞎子」,而是能自己看设计稿、翻仓库、查记忆里记下的项目约定。输出从「看着像」变成了「真的贴合这个项目」。

本质:把「人盯」变成「流程兜底」

链路走顺之后,我的判断是:工作流的本质,是把「人盯」变成「流程兜底」。

以前人在每一步盯着:需求对不对、代码乱不乱、提交前跑没跑测试。问题是人会累、会忘、会偷懒。现在把判断挪进流程:

  • 任务图兜一致性。
  • 并行兜效率。
  • TDD 兜实现质量。
  • 评审回路兜最终门禁。

链路是不是太重了?

有人会问:这套链路是不是太重了?

我的回答是:链路是骨架。具体到一个 bugfix,你当然可以跳过原型、跳过头脑风暴。但骨架要在。

有流水线而选择不用,和压根没有流水线,是两回事。

一条铁律:compile 能过 ≠ 代码正确

这套工作流也不是没教训。我们最疼的一次,是某次「搜索优化」提交:编译照过,却静默删掉了 270 行并发安全代码——因为当时的评审回路只盯编译、没盯测试。

这条事故催生了铁律:compile 能过不等于代码正确,只有 verify 算数。

小结

AI 研发不是换个补全工具,是重新搭一条流水线。三条原则贯穿始终:

  • 能并行就尽量并行。
  • 能固化就别靠记忆。
  • 能让流程兜底就别指望人盯。
