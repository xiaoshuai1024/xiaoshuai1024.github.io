RAG 在网上被讲得玄之又玄,什么"企业知识大脑"“智能问答中枢”。说穿了就是一句话:在问大模型之前,先从你的私有文档里检索出相关片段,塞进 prompt,让它"看着资料"回答。 它的价值不在魔法,在让大模型基于你的私有、可更新、可追溯的知识回答,而不是只靠训练时的旧记忆。

上一篇《构建一个多租户 AI 助手》讲了整体架构,这篇把其中最被神话的一块——RAG 知识库单独拆开,从 0 到 1 讲它到底怎么落地。

这篇不讲概念,讲工程:多租户怎么隔离、切片怎么配、向量库怎么选、入库失败的文档怎么重试、改了切片策略怎么重建。代码全部来自生产系统。

一、要解决的核心问题

运营后台的 AI 助手要能基于每个租户的私有运营文档回答问题——A 租户问"我们店铺的商品上下架流程",助手得去查 A 租户的运营手册,不能拿 B 租户的文档乱答。

需求拆成四个硬约束:

  1. 多租户隔离:A 绝对检索不到 B 的文档,这是底线,不是优化项。
  2. 文档可管理:运营能在后台上传、编辑、发布、删除文档,不是只能命令行灌库。
  3. 检索有质量闸:不相关的片段不能塞进 prompt,否则反而拉低回答质量。
  4. 可演进:切片策略、embedding 模型变了,得能批量重建,不能推倒重来。

RAG 的是四步:切片 → 向量化 → 存储 → 检索。但工程上远不止这四步,下面逐层展开。

二、整体设计

几个关键设计决策,后面逐个展开:

  1. 双存储分工:MySQL 存可编辑的业务数据(标题/正文/标签/状态),ES 存向量做检索,两边各司其职。
  2. 多租户靠 metadata filter:所有租户共用一个 ai_knowledge 索引,检索时按 tenant_id 过滤。
  3. 质量闸在检索层:similarityThreshold(0.7) 卡掉不相关结果。
  4. 生命周期可重建:DRAFT/PUBLISHED 状态机 + es_indexed 标记 + 批量 rebuild。

三、存储:Elasticsearch 向量库

向量库选型是第一个决策点。我选了 Spring AI 的 Elasticsearch store(spring-ai-elasticsearch-store),理由很朴素:团队已经在用 ES 做日志和全文检索,基础设施现成。PG 向量、Milvus、Qdrant 都不错,但引入一个新中间件的成本(运维、监控、备份)远大于复用 ES。

配置只有几行:

spring:
  ai:
    vectorstore:
      elasticsearch:
        index-name: ai_knowledge
        initialize-schema: true

initialize-schema: true 让 Spring AI 启动时自动建好带 dense_vector 字段的索引 mapping,业务不用手写 ES 的 schema。embedding 由 starter 处理——你只管 add(文档),它算向量、写索引全包了。

向量库方案优点缺点
复用 ES(我用)基础设施现成,可与全文检索混用超大规模(亿级)性能不及专用库
Milvus / Qdrant 专用库极致检索性能、原生 ANN 索引多一套中间件,运维成本高
pgvector事务和向量同库,简单大规模下性能瓶颈明显

判断:不要为了"用专用向量库"而引入新中间件。 知识库文档量在百万级以下,ES 的 HNSW 完全够用,复用现有基建比追新更稳。

这个选型还有个维度值得说:团队能不能维护。Milvus、Qdrant 这类专用向量库,性能确实强,但它们有自己的运维体系——集群部署、索引重建、版本升级、监控指标,都和 ES 不一样。引入一个新中间件,意味着团队要多学一套运维知识,出问题时多一个排查方向。如果团队已经在用 ES 做日志和全文检索,ES 的运维大家已经熟了,向量检索只是 ES 的一个新用法,学习成本几乎为零。这个「运维成本」在选型时经常被忽略——大家只比性能和功能,忘了「上线后谁来维护」。一个性能强但没人会维护的向量库,出一次故障就能把 RAG 服务拖死。

还有个选型误区:别一上来就担心「亿级向量」。很多团队文档量才几千篇,就开始纠结「ES 扛不扛得住亿级向量」。这是典型的过度设计——你的量级根本到不了需要专用库的规模,提前为「万一到了亿级」选了专用库,承担了它全部的运维成本,但「万一」可能永远不来。量级到了再换,Spring AI 的 VectorStore 抽象层就是为这一天准备的——换库只改注入,上面业务代码不动。先用最省事的(复用 ES),不够了再换,比一开始就选重的更务实。

四、切片:TokenTextSplitter 怎么配

文档不能整篇塞进去,要切片。我用 Spring AI 的 TokenTextSplitter:

private static final TokenTextSplitter SPLITTER =
    new TokenTextSplitter(500, 50, 50, 100, true);
// 参数:chunkSize=500, minChunkSizeChars=50, minChunkLengthToEmbed=50,
//       maxNumChunks=100, keepSeparator=true

每个参数都有讲究:

  • chunkSize 500 token:针对运营文档(段落式说明文)调出来的。太大,一个块混了多个话题,检索精度下降;太小,上下文不足,模型答得片面。
  • overlap 50:minChunkSizeChars 控制相邻块最小重叠——一句话被从中间切断时,重叠区能保住上下文,这是不能省的。
  • keepSeparator true:保留段落分隔符,切片尽量贴着自然段落走,而不是纯按 token 硬切。

切片大小是质量与成本的权衡表:

chunkSize检索精度上下文完整度embedding 成本
太小(100)高(命中精确)差(只言片语)低
500(我用)中高好(完整段落)中
太大(2000)低(话题混杂)好高

500 不是拍脑袋,是拿一批真实运营文档跑过 A/B——太小模型答得片面,太大检索会把无关段落也带进来。

切片还有一个容易被忽略的细节:中文切片不能照搬英文的 token 数。英文一个 token 约等于 0.75 个单词,500 token 约一段话;中文一个汉字约 1.5-2 个 token(取决于 tokenizer),500 token 实际只有 250-330 个汉字,可能不够一个完整段落。我们早期照搬了英文教程的 500 token,中文文档切得太碎,后来改成按中文字符数(约 400 字)来调,才切得合理。中文 RAG 的参数,要在中文语料上重新调,别信英文教程的默认值。

切片策略还有「按结构切」的进阶玩法。纯按 token 切是「无脑切」,不管文档结构;更好的做法是先按文档的标题/段落结构切(识别 Markdown 的 ##、HTML 的 <h2>),在结构边界处切,保证每个 chunk 是一个完整的语义单元。我们目前用 TokenTextSplitter + keepSeparator 做了初步的结构感知,更精细的「按标题层级切」是演进方向——但这个复杂度不低,建议等基础 RAG 跑稳了再做,一上来就追求完美切片,容易陷在调参里出不来。

五、多租户隔离:metadata filter,不是分索引

这是多租户 RAG 最关键的问题:怎么保证 A 租户检索不到 B 租户的文档?

最容易想到的方案是每个租户一个独立索引(ai_knowledge_tenant_a、ai_knowledge_tenant_b)。我没有这么做——租户多了索引会爆炸,ES 的 mapping、分片、别名管理成本指数上升。我选的是所有租户共用一个 ai_knowledge 索引,靠 metadata 过滤隔离。

入库时,给每个 chunk 注入元数据:

Document doc = new Document(content, Map.of(
    "tenant_id", tenantId,   // 租户隔离的主键
    "doc_id",    docId,      // 关联回 MySQL 文档
    "title",     title,      // 检索时可展示来源
    "tags",      tags,        // 业务标签,可做二级过滤
    "source",    source       // 来源(手动录入/导入)
));
List<Document> chunks = SPLITTER.apply(List.of(doc));
vectorStore.add(chunks);  // Spring AI 自动算 embedding 写 ES

tenant_id 是隔离的核心——它不是"建议字段",是每次检索的强制过滤条件。检索时用 Spring AI 的 filterExpression 限定本租户:

List<Document> hits = vectorStore.similaritySearch(SearchRequest.builder()
    .query(query)
    .topK(5)                                              // 最多取 5 块
    .similarityThreshold(0.7)                             // 相似度 < 0.7 不返回
    .filterExpression("tenant_id == '" + tenantId + "'")  // 多租户硬隔离
    .build());

三道闸叠在一起:

  • filterExpression("tenant_id == ...") 是租户隔离闸,ES 在向量检索时就按 metadata 过滤,A 物理上查不到 B 的向量。
  • similarityThreshold(0.7) 是质量闸,相似度低的不返回,避免"检索到了但不相关"的噪声污染 prompt。
  • topK(5) 是数量闸,prompt 容量有限,塞太多反而稀释信号。
隔离方案隔离强度管理/存储成本适用
一租户一独立索引物理隔离高(索引爆炸)强合规、超大规模
共用索引 + metadata filter(我用)逻辑隔离(够强)低绝大多数 SaaS 场景
分索引 + 别名路由中中折中,但复杂度高

判断:除非有强合规要求(金融/医疗的物理隔离),否则一律用 metadata filter。 一个索引管所有租户,过滤在 ES 检索时做,零额外存储成本,管理也只对一张索引。

metadata filter 还有个安全上的考量:过滤必须下沉到检索层,不能在应用层做。我们的 filterExpression 是传给 ES 的,ES 在向量检索时就按 metadata 过滤——A 租户的请求,ES 物理上只返回 A 的向量,B 的向量根本不进候选集。如果改成「ES 返回 topK 结果,应用层再按 tenant_id 过滤」,问题就大了:ES 返回的 topK 可能全是 B 的向量,应用层过滤完一个都不剩,检索失败;更危险的是,如果应用层过滤有 bug(比如忘了过滤),B 的向量直接进 prompt,跨租户泄漏。过滤下沉到检索层,从物理上保证「不该返回的根本不返回」,比应用层过滤安全一个量级。这和多租户数据隔离里「SQL 带 tenant_id 条件」是同一个思路——约束下沉到最接近数据的那层。

六、管理生命周期:KbManagementController

RAG 不只是"检索",还有一整套管理——这是很多教程跳过、但生产必备的部分。文档不是灌进去就完了,要能编辑、发布、重建、删除。

业务数据存 MySQL:

CREATE TABLE ai_knowledge_document (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  tenant_id    VARCHAR(64)  NOT NULL,
  title        VARCHAR(255) NOT NULL,
  content      TEXT         NOT NULL,
  tags         VARCHAR(512),
  source       VARCHAR(128),
  status       VARCHAR(16)  DEFAULT 'DRAFT',  -- DRAFT / PUBLISHED
  es_indexed   TINYINT      DEFAULT 0,         -- 0=未入库 1=已入 ES
  created_at   DATETIME     DEFAULT CURRENT_TIMESTAMP,
  updated_at   DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_tenant_status (tenant_id, status)
);

status 和 es_indexed 两个标记撑起整个状态机。KbManagementController 暴露的管理能力:

接口作用副作用
GET /documents列出本租户文档(按 tenant 过滤)无
POST /documents新建文档落 MySQL,status=DRAFT
POST /documents/{id}/publishDRAFT→PUBLISHED触发 ingest 入 ES
POST /rebuild-index批量重建某租户索引删旧向量 + 全量重灌
DELETE /documents/{id}删除文档删 MySQL + 删 ES 向量

发布是最核心的链路——它把状态流转和入库绑在一起:

@PostMapping("/documents/{id}/publish")
public void publish(@PathVariable Long id) {
    KbDocument doc = kbDocumentMapper.selectById(id);
    // 仅 DRAFT 可发布,幂等防重复入库
    if ("DRAFT".equals(doc.getStatus())) {
        ingestService.ingest(doc);              // 切片 + embedding + 写 ES
        kbDocumentMapper.updateStatus(id, "PUBLISHED");
    }
}

这里有个幂等设计:只有 DRAFT 状态才触发 ingest,PUBLISHED 的文档重复点发布不会重复入库。es_indexed 标记则用于重建索引时做断点续传——批量重灌出错时,能精准定位哪些文档没入成功。

删除是双向清理——MySQL 记录和 ES 向量都要删,否则会出现"列表里没了,但检索还能搜到"的灵异现象:

@Transactional
public void delete(Long id) {
    KbDocument doc = kbDocumentMapper.selectById(id);
    // 删 ES:按 tenant_id + doc_id 精准定位该文档所有 chunk
    vectorStore.delete(filterExpression(
        "tenant_id == '" + doc.getTenantId() + "' && doc_id == '" + id + "'"));
    kbDocumentMapper.deleteById(id);  // 删 MySQL
}

双向删除有个事务上的讲究:MySQL 删了但 ES 删失败怎么办?@Transactional 保证 MySQL 的删除可回滚,但 ES 的删除不在事务内(ES 不支持分布式事务),删失败 MySQL 已经回滚了但 ES 的向量还在——变成「列表里有但检索不到」(因为列表查 MySQL,检索查 ES)。反过来,ES 删成功但 MySQL 删失败,变成「列表里有但检索不到」。两种不一致都很麻烦。我们的解法是先删 ES 再删 MySQL:ES 删成功后,即使 MySQL 删失败,文档还在列表里,用户重新点删除即可(幂等);反过来 MySQL 先删,ES 留了「幽灵向量」,用户看不到也删不掉,更难处理。顺序选「先删难恢复的那边」,把不一致导向「可重试」的方向。

七、类设计:三层各司其职

管理、入库、检索三件事的类关系:

KbManagementController 管业务编排,IngestService 管 RAG 管线(切片 + 入库),VectorStore(Spring AI)管向量存储。换向量库(比如哪天换 Milvus)只动 VectorStore 这层的注入,上面两层不动——这是用 Spring AI 抽象层换来的解耦红利。

八、降级:@ConditionalOnBean(VectorStore.class)

生产环境不是每套都部署了 ES。开发机、测试环境可能没起 ES,直接装配 VectorStore 会启动报错。我用了一个干净的降级:

@Service
@ConditionalOnBean(VectorStore.class)  // 有 ES 才装配
public class IngestService { ... }

@ConditionalOnBean(VectorStore.class) 的含义:Spring 容器里有 VectorStore bean(即 ES starter 成功装配)时,IngestService 才被创建。没有 ES 的环境,知识库相关能力整体跳过装配,AI 助手仍能对话——只是没有 RAG 增强,退化为纯 LLM 问答。这种"核心能力可选"的设计,让系统在不同环境(本地/测试/生产)能优雅降级,而不是一刀切启动失败。

降级设计还有个细节要处理:降级后的用户体验要明确。没有 RAG 时,AI 助手回答会差一截(没有租户私有知识),如果用户不知道「现在是降级模式」,会觉得「这个助手怎么变笨了」。我们的做法是:降级模式下,助手回答时带一句「当前为基础模式,未接入知识库」,让用户知道差异。这个提示看着小,但能避免用户把「降级模式的回答」当成「正常水平的回答」,产生「这个 RAG 没用」的误判。降级不是偷偷降,要透明地降。

降级还有一个延伸场景:ES 临时不可用。ES 集群重启、网络抖动时,VectorStore bean 还在(启动时装配成功),但检索调用会超时。这种「运行时不可用」@ConditionalOnBean 管不了(它在启动时判断)。我们的做法是给检索调用加超时 + try-catch,超时就降级为纯 LLM 问答,不 block 整个对话。ES 恢复后自动恢复 RAG。这种「运行时降级」比「启动时降级」更常用,因为 ES 临时故障比「根本没装 ES」更常见。

九、踩坑与演进

做完这一套,几条沉淀下来的教训:

  1. overlap 不能省。最早我把 overlap 设成 0,检索经常命中半句话,模型答得前言不搭后语。加 50 token 重叠后明显好转——一句话被切断的概率从肉眼可见降到几乎无感。
  2. 删除必须双向。只删 MySQL 不删 ES 是经典 bug,会出现"列表没了但还能搜到"。vectorStore.delete(filter) 按 tenant_id && doc_id 删,不能靠 chunk 的自增 id——重灌后 id 会变。
  3. 入库要标记 es_indexed。ES 写入可能部分失败(集群压力、超时)。有这个标记,重建索引时能只补未成功的,不用每次全量重灌。
  4. similarityThreshold 要按业务调。0.7 不是万能值。运营文档(强相关场景)我用 0.7;如果是开放性问答(弱相关场景),0.5 更合适。阈值卡太死会漏召回,卡太松会引噪声。

还有一个踩坑值得提:文档更新后要重新入库,但不能靠「删了重加」。早期我们更新文档的做法是先 delete 旧向量再 add 新向量,两步之间如果有检索请求进来,会命中「空窗期」——文档刚刚删了还没来得及加,检索返回空,用户以为文档没了。后来改成「先 add 新向量(带新版本号),再 delete 旧版本号的向量」——更新期间新旧并存,检索总能命中,空窗期消失。这种「先加后删」的顺序,和前面删除的「先删 ES 再删 MySQL」方向相反,因为场景不同:更新要保证「随时可检索」,删除要保证「不留幽灵」。顺序的选择,取决于你要保证哪个不变量。

演进方向上,下一步想做的两件事:混合检索(向量 + BM25 关键词,ES 原生支持 RRF 融合,召回率能再提一截),以及 chunk 重排(用交叉编码器对 topK 结果二次精排,把最相关的顶上去)。

十、几条判断

  • RAG 不神秘,就是"切片 → 向量化 → 存储 → 检索"四步,别被概念吓住,工程上每一步都有成熟方案。
  • 向量库优先复用现有基建(ES/PG),别为追新去引入专用库,运维成本不划算。
  • 切片的 overlap 不能省,否则上下文被切断。
  • 多租户隔离用 metadata filter,一个索引管所有租户。
  • similarityThreshold 是质量闸,别把不相关的也塞进 prompt。
  • 知识库是"管理 + 检索"两件事,状态机 + es_indexed 标记撑起可重建的生命周期,缺了管理这半,检索做得再好也上不了生产。

RAG 的工程化,难点不在调通一次检索,在把它做成多租户、可管理、可演进的知识体系。检索只是冰山一角,水面下是入库、幂等、重建、降级、双向清理这些"脏活累活"——而这些决定它能不能上生产,能不能在真实的多租户业务里长期稳定地跑下去。

深度展开:RAG 的检索质量调优

RAG 的难点不在"搭起来",在"答得准"。检索质量取决于四个参数的协同,这四个参数不是独立调的,是互相影响的——调了一个,另外几个的最优值也会变。

chunk size × overlap:chunk 太大,一个 chunk 混多主题,检索不准;太小,上下文丢失。中文我们用 400 字 + overlap 100 字,这个组合是反复调出来的。overlap 的意义在「跨块引用」——某个关键信息恰好横跨两个 chunk 的边界,没有 overlap,这个信息在两个 chunk 里都是残缺的,检索到哪个都答不准。有了 overlap,至少有一个 chunk 包含完整信息。overlap 不是越大越好(太大等于 chunk 变大,浪费存储和 embedding 成本),100 字(约 chunk 的四分之一)是个经验平衡点。

top-k × 相似度阈值:k=5 召回 5 个片段,但不是每个都相关。阈值 0.75 过滤掉低于这个的,实际相关的可能只有 2-3 个。k 是"召回数量",阈值是"质量门槛",两者配合才出效果。只调 k 不调阈值,k 太大召回一堆噪声;只调阈值不调 k,阈值卡太死可能一个都召不回。两者的关系是:阈值决定「召回的质量下限」,k 决定「召回的数量上限」,先调阈值保证质量,再调 k 保证够用。

embedding 模型:这个参数最容易被忽略,但影响最大。不同 embedding 模型对中文的理解差异很大——用多语言模型 vs 中文专用模型,检索准确率能差 20%。我们早期用一个通用 embedding 模型,中文运营文档的检索质量一直上不去,换成中文优化过的模型后,同样的切片和阈值,准确率直接跳了一个台阶。embedding 模型是 RAG 的「地基」,地基不好,上面调参再久都是白费。

调参的方法:拿一批已知答案的问题做基准集,改一个参数就跑一遍基准,看准确率变化。这不是玄学,是实验科学。我们调了两周,准确率从 60% 提到 85%。

基准集怎么建有个经验:问题要覆盖三种类型——「单文档可答」(答案在一个 chunk 里)、「跨文档综合」(答案要综合多个 chunk)、「无答案」(知识库里根本没有,期望模型说「不知道」)。第三类特别重要,它测的是「不瞎编」的能力——RAG 最怕的是知识库里没有答案,模型硬编一个,基准集不覆盖「无答案」类,这个致命缺陷就测不出来。我们的基准集 30% 是「无答案」类,专门测模型的拒答能力。

深度展开:RAG 的常见失败模式

调通 RAG 之后,线上用起来会发现几种典型的「答得不好」的模式,每种都有对应的解法。

失败模式一:检索到了但答错了。 chunk 召回正确,但模型把 chunk 里的信息理解错了。这种通常是 chunk 上下文不足——chunk 里有个「它」指代前文,但前文被切到上一个 chunk 了,模型不知道「它」指谁。解法是加 overlap(我们就是这么发现 overlap 重要的),或者在切片时做「语义切片」而不是纯按 token 切——尽量在段落边界切,不在句子中间切。

失败模式二:没检索到但瞎编了。 知识库里没相关内容,但模型基于自己的训练知识编了个答案。这种最危险,因为模型编得「很有道理」,用户分不清是知识库说的还是模型编的。解法是调高 similarityThreshold(卡掉低相似度的「假召回」),并在 prompt 里明确要求「如果检索结果不相关,就说不知道,不要自己编」。这条 prompt 约束看着简单,但能挡住大部分瞎编——模型不是不会说「不知道」,是默认不说,你要明确告诉它。

失败模式三:检索到了太多,模型选了错的那个。 topK 太大,召回了 10 个 chunk,其中 2 个相关 8 个无关,模型被无关信息干扰,答偏了。这种解法是减小 topK + 调高阈值,只留最相关的 2-3 个。prompt 里信息越多不代表越好——无关信息是噪声,噪声多了信号就被淹没。「少而准」比「多而全」对 RAG 更有效,这个认知和直觉相反(直觉是「多给点信息总能答得更好」),但实践反复证明,精简的 context 准确率更高。
