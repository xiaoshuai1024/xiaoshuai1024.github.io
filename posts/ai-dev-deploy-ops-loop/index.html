代码写完、评审过了、测试绿了,然后呢?要把它安全地送到线上,并在跑起来之后兜住它。这是 AI 研发工作流的最后一公里。

跟同行聊起自动化运维,大家第一反应都是 K8s、Argo CD、GitOps、蓝绿发布。我的回答每次都让对方愣一下:我们是单机部署,连 Docker 都没用,但每天凌晨有一个 AI Loop 在自己跑测试、自己修 bug、自己提 MR。

听起来很割裂——一边土到「scp 上传 jar」,一边又潮到「AI 自治」。但这是我想讲的重点:小团队落地自动化运维,真正能跑通的不是「全自主闭环」那个性感概念,而是「命令化骨架 + 局部自治 Loop」的混合形态。

骨架兜住「确定要做、且不可逆」的事;Loop 试探「可以交给机器、且可逆」的事。这条线划在哪,是整篇文章的灵魂。

先有骨架:命令化的部署流水线

整个体系的地基是一条 /deploy-prod-be-ops 命令。注意是命令,不是 CI pipeline。

为什么不做成流水线?三个原因。一是我们单机部署、人随时在场,部署是「按下回车」的事,没必要为它养一套 GitLab Runner。二是部署不可逆——jar 推上去、服务一重启,旧版本就没了,这种动作必须留一双人的眼睛盯着。三是命令化比流水线灵活得多:它能在本地跑、能传参、能临时改,而流水线一旦定下来改起来反而麻烦。

讲到这里,一定会有人问:那为什么不上 K8s、不上 Docker? 这不是「不想」,是「算过账不划算」。K8s 带来的是弹性伸缩、滚动更新、自愈——这些价值在一个「4 人团队、单机部署、流量可预期」的多租户 SaaS 上,基本用不到。而它的成本是实打实的:要养一套集群、要写 Helm chart、要处理证书和 Ingress、出问题排查链路还更长。技术选型的判据不是「先进不先进」,是「这个阶段的问题,K8s 真的能解决吗」。我们这个阶段的问题是「怎么安全地、可重复地把 jar 送上线」,scp + systemd 已经足够,再多就是过度工程。等哪天流量上来、要扩容了,K8s 自然会进画面——但不是现在。

这条命令长什么样?六个步骤,每一步都是被真实坑逼出来的:

# Step 1-2:构建后端 jar 和运营后台 dist
source .env.prod && cd backend && mvn -q clean package -Dmaven.test.skip=true
source .env.prod && cd operation-backend && make build-prod

# Step 3-4:scp 上传 jar + tar 打包 dist 上传
sshpass -p "$PROD_SSH_PASSWORD" scp ... backend-0.0.1-SNAPSHOT.jar root@$PROD_SSH_HOST:...
cd operation-backend && tar czf /tmp/op-dist.tar.gz -C dist .
sshpass ... ssh root@$HOST "cd /www/wwwroot/operation-backend && rm -rf assets index.html && tar xzf /tmp/op-dist.tar.gz"

# Step 5-6:重启后端 + 验证日志无 ERROR
source .env.prod && bash scripts/deploy/restart-backend-prod.sh

这里有两个被坑出来的设计,值得展开讲。

第一个坑:nginx root 部署陷阱。 运营后台的 nginx root 直接是 /www/wwwroot/operation-backend/,不是 /dist/ 子目录。最自然的写法是 scp -r dist,把整个 dist 目录传上去——结果 nginx 读的是根目录的旧文件,部署不生效,你以为上线了其实线上还是老版本。所以必须先 rm -rf assets index.html 再 tar xzf 解压到根目录。这种坑不会报错、不会失败,只会让你半夜发版后盯着屏幕疑惑「怎么没变化」。

第二个坑:pkill 杀进程时自杀。 重启后端要先杀掉旧进程。最直觉的写法是 pkill -f backend-0.0.1-SNAPSHOT.jar,但这条命令通过 ssh 远程执行时,远程 shell 的 -c 参数里就包含 backend 这个字符串,pkill 会匹配到 ssh 自己的进程,把自己杀掉。更隐蔽的是,pgrep -fc 计数会因此虚高,导致健康检查误判。

修法是用正则字面量绕开匹配:pkill -9 -f '[b]ackend-0.0.1-SNAPSHOT.jar'。[b]ackend 这个正则匹配的还是 backend 这个字符串,但 pgrep/pkill 看到的字面量是 [b]ackend,不含 backend,就不会匹配到自己。这是 Linux 运维里一个老掉牙的技巧,但没踩过的人想破头也想不出来。

重启之后还有一道硬验证:systemctl 服务 active + 健康检查返回 200 + 只有 1 个 backend 进程 + 进程是 systemd 起的(不是手动 nohup)。这四条缺一不可——只看健康检查的话,可能是旧进程占着端口、新进程绑端口失败,健康检查命中的是旧进程,误以为成功了。

部署流水线的本质,是把**「发版」这件高频、机械、容易手抖**的事,固化成一串可重复的命令。它不是自动化运维的全部,但它是全部的前提。

部署完不是结束:生产验证的三层冒烟

部署完按理说该松口气了。但 编译能过 ≠ 代码正确 那篇的事故告诉我们:编译绿、构建成功、服务起来——这三件事都不代表线上真的能用。所以部署完必跑一道 /prod-testing,这是发版的第二只手。

它分三层,覆盖范围由窄到宽:

  • Layer 1:API 冒烟(22 个核心接口)。 首页、搜索、Banner、结算批次、提现、对账、圈子、套卡……这些是「服务起来但接口挂了」能第一时间抓到的。quick 模式只跑这层,大约 3 分钟。
  • Layer 2:C 端 H5 页面。 首页、商家结算、佣金明细、达人提现。生产环境没有 H5(C 端是小程序),所以要在本地起 H5 指向生产 API 跑。
  • Layer 3:运营后台页面。 结算、对账、圈子管理、套卡管理。full 模式跑完这三层,大约 12 分钟。

为什么要分三层、还分 quick 和 full?因为冒烟的颗粒度得和「能不能发现问题」对齐。API 挂了,Layer 1 三分钟就能确认;但页面渲染崩了、或者运营后台某个表单提交报错,Layer 1 测不到,必须跑到 Layer 3。发版赶时间的时候,至少跑 quick;大改动之后,full 不能省。

但要诚实说一句:冒烟有盲区,而且不小。它测的是「主路径能走通」,测不了「边界条件对不对」。比如佣金结算的某个阶梯规则改了,冒烟只会验证「能算出一笔佣金」,不会验证「阶梯临界点的佣金算得对不对」——后者得靠 三层测试纪律 里的单元测试和集成测试,在部署前就守住。冒烟是「兜底」,不是「全证」。把它当全证,会漏掉那些「能跑但算错」的 bug,这种 bug 比「直接报错」的更危险。

这里有个反直觉的设计:验证脚本默认不自动清理测试数据。很多人会觉得「自动化不就是跑完自己收拾干净吗」——但在生产库里,「自动清理」是高危动作。测试脚本创建的临时数据(测试用户、API 日志)随 token 过期会自己失效,需要手动清理的只有 sys_user 表里的测试用户。让脚本去 DELETE FROM sys_user,等于把删生产数据的权力交给一个自动化流程,这风险不值得冒。

冒烟这一关告诉我们:「部署成功」和「线上可用」是两件事。前者是动作的完成,后者是结果的对账。中间隔着的那道验证,谁也不能省。

线上出问题怎么办:不破坏生产的生产排障

发版完成、冒烟过了,线上还是会出问题——慢查询、偶发 500、某个租户数据异常。这时候最痛的不是「问题难找」,而是「想调试却不敢动」。

痛点很具体:要复现一个生产 bug,你得连生产库;但本地服务一连生产库,就处在「一不小心就写脏数据」的危险边缘。一次手滑的 UPDATE、一个忘记加 WHERE 的 DELETE,就是事故。

我们的做法是 /prod-debug 命令,用一套严格的纪律把生产调试变得安全:

# 后端以 prod profile 启动,连生产 MySQL/Redis
cd backend && make prod
# 前端/运营后台的 API 指向生产域名
API_BASE_ORIGIN="https://v1-api.dayuancheng.cn"

技术上是切 profile + 切域名,但真正起作用的是三条铁律,而不是切换本身:

  • 不修改任何业务代码。 排障归排障,代码一行不改;要改,走正常的开发流程。
  • 不创建数据库迁移。 Flyway 在生产是 out-of-order: false,本地起一个迁移会和线上版本冲突,排查反而被自己制造的冲突带偏。
  • 不触发 CD / 不重启线上服务。 本地调试,线上服务纹丝不动。

这三条把「调试」这个动作死死锁在「只读」范围里。健康检查 120 秒内返回 200 才算就绪,期间盯着启动日志诊断常见问题:MySQL/Redis 连接失败、Flyway 冲突、端口被占用。

排障这一关的本质,是承认「人会手抖」,然后用纪律和工具把人手抖的破坏面收窄到零。它不解决 bug 本身,它解决的是「调试这个动作本身可能制造新事故」这个元问题。

真正对标参考文章的:夜间 AI Loop

前三部分都是「命令化骨架」——人按下回车,机器按步骤执行。现在讲整个体系里唯一一个自主循环:nightly-test.sh。这是对标 Loop Engineering 的核心。

它是一个自管理的 daemon(不是系统 cron),每天凌晨 00:00 自动触发,跑完五个阶段:

Phase 0: 保存工作区状态(分支 + 未提交改动 stash 保护)
Phase 1: 同步 dev 分支(fetch + checkout dev + reset --hard origin/dev)
Phase 2: 8 步全量测试(后端单测 ∥ 前端单测 → 后端 IT → 运营单测 → 起后端 → 运营 E2E → H5 E2E → 小程序编译)
Phase 3: 失败 → claude -p 自动修复 → 切 fix/nightly-xxx 分支 → 跑完整测试套件验证 → 提 MR(不合并)
Phase 4: 恢复工作区(切回原分支 + pop stash)

每个阶段都有飞书通知,无论是 PASS 还是 FAIL。这一关我重点讲三个设计判断,因为它们决定了这个 Loop 能不能真正自治。

判断一:为什么是 daemon,不是系统 cron

最自然的实现是写个 crontab,0 0 * * * bash nightly-test.sh。我们没用。原因有两个。

一是跨平台。这套脚本要在 macOS、Linux、Windows(Git Bash)上都能跑,cron 是 Linux 专有的,launchd 是 macOS 专有的,统一不了。脚本里用 uname -s 检测平台,用自管理的 nohup bash $0 _daemon_loop & 起一个常驻进程,自己算下次触发时间自己 sleep,一套代码三个平台通吃。

二是不污染系统。系统级的 cron/launchd 一旦注册,即使项目删了、机器换人,这条定时任务还在偷偷跑,而且没人记得它是干嘛的。daemon 是项目级、会话级的:PID 文件在项目目录里,机器重装或项目删除,它自然就没了。这种「生命周期和项目绑定」的定时,比系统级 cron 干净得多。

代价是要自己处理进程管理(PID 文件、kill -0 检查存活、文件锁防重入)。但这些都是几十行 shell 能搞定的事,换来的是「定时任务随项目走、随人走」的清晰归属。

判断二:Agent 只改代码,MR 不自动合并

夜间测试失败后,Loop 会调 claude -p 让 Agent 修 bug。但有一条死规矩:Agent 只能改代码、提 MR,绝对不能合并。

具体流程是:Agent 从 dev 切一个 fix/nightly-<run_id> 分支,分析失败日志里能确定的代码问题(编译错误、NPE、断言失败),修复它,然后只 commit + push,不创建 MR。MR 的创建是脚本在外部验证通过后才做的。

为什么要分两步、且把合并权留在人手里?因为 Agent 修的 bug 对不对,不能由 Agent 自己说了算。所以修复完,脚本会切到 fix 分支再跑一遍完整测试套件——单测、集成测、E2E、编译,全过才算修复成功;只要有一项不过,fix 分支直接删掉,Agent 这次就算没修成。

这套「修复 + 独立验证 + 人工合并」的三段式,是给 Agent 自治画的边界:它可以干活,但干得对不对,由独立于它的客观验证来判断;干完的成果能不能进仓库,由人来拍板。这跟 无 CI 多 Agent 评审回路 里「评审 Agent 只读、修复 Agent 只写、读写分离」是同一个设计哲学。

还有一个细节:Agent 跑的时候带了明确约束——「无法确定根因的(环境问题、E2E flaky、数据问题),输出说明,不要强行修改代码」「不要修改测试断言」。这两条挡住了 AI 协同里最常见的两种作弊:改不动代码就去改测试让它过,和不知道为啥挂了就瞎改一通碰运气。

判断三:测完必须 stash 恢复工作区

夜间测试是凌晨跑,这时候开发者的机器上可能正开着 feature 分支、有一堆未提交的改动。如果 Loop 直接 git checkout dev,这些改动就丢了。

所以脚本在切 dev 之前,会把每个仓库(主仓 + frontend/backend/operation-backend 三个子模块)的未提交改动 stash 起来,记下当前分支名;测试全部跑完,再切回原分支、pop stash 恢复。开发者第二天打开电脑,工作区和昨晚一模一样,根本不知道凌晨有个 Loop 来过。

这个设计看着不起眼,但它是 Loop 能「在开发者机器上常驻」的前提。一个会破坏开发者现场的自动化流程,没人愿意让它跑。你让它跑,它就得有「来无影去无踪」的自觉。

真实数据:这个 Loop 跑了一年

光讲设计容易显得「这是 demo」,补几个真实数字,说明它扛得住真实改动量:

  • 全量测试一轮跑下来,8 个步骤串行 + 部分并行,稳定耗时 40–60 分钟。后端单测和前端单测是并行起的,但 E2E 必须等后端起来才能跑,这部分省不掉。
  • Agent 修复的命中率大约六成:能明确判断的代码问题(编译错误、NPE、断言写错)它能修;环境问题、E2E flaky、数据依赖问题,它修不动,会老老实实输出说明。
  • 最痛的一种是**「修了又冒、冒了又修」**:某个 E2E 因为测试数据污染反复挂,Agent 改了三次都没过,最后熔断叫人。这种时候 Agent 自治的价值是「把能修的先修了、把不能修的标记清楚」,不是「包治百病」。

这些数字说明一件事:自治 Loop 不是「替代人」,是「把人从机械劳动里腾出来,只处理真正需要判断的部分」。六成命中率意味着,六成的夜间失败第二天早上已经有 MR 等着 review,人只需要看剩下的四成。

Loop 的另一半:定时评审与备份

夜间测试 Loop 之外,还有两个轻量 Loop 在跑,共同点是「定时扫描 + 阈值告警」,但管的不是代码,是另两件事。

凌晨 3 点的定时评审。 scheduled-review.sh 每天 03:00 对当前 feature 分支相对 dev 的改动做一次 headless review,重点扫数据安全、破坏性操作、多租户隔离、凭证安全这几类高危问题。扫完数 🔴 阻断问题的个数,超阈值就飞书告警。

注意它只告警,不改代码——这跟夜间测试 Loop 不同。原因前面说过:评审是主观判断,没有客观验证手段,所以它只能「报警」,不能「自治」。它的价值是把「白天写完没来得及仔细看的改动」在夜里再扫一遍,把高危问题在合并前抓出来。

每日数据库备份。 mysql-daily-dump.sh 每天凌晨 3 点跑,从只读从库取数(不压主库),mysqldump --single-transaction 拿一致性快照,gzip 压缩后算 SHA256 校验,rclone 上传到跨云 OSS。保留策略是 3-2-1:日备 7 份 + 周备 4 份 + 月备 12 份。任何一步失败,飞书告警。

这里有两个细节特别能体现「真刀真枪用出来的设计」:一是只从只读从库取,脚本启动时会 SELECT @@read_only 确认是 1,如果连到主库了会 WARN——这是防「备份脚本配置错了连到主库,然后某个误操作写脏了主库」的兜底。二是dump 文件过小直接判失败,因为「空库或连接失败」也会生成一个几字节的文件,不校验大小就会把这种假成功当真备份。

观点:哪些能交给 AI 自治、哪些不能

前面讲了一堆「怎么做」。这部分讲「为什么这么做」——这是我跑了一年多之后,对「AI 时代的自动化运维边界」的判断。

参考文章讲的「全自主闭环」很性感:agent 扫日志 → 诊断 → 修复 → 验证 → 部署预发,全程无人。我们没有做到那一步,而且是刻意没做到。原因不在技术,在三个判断标准。

标准一:可逆 vs 不可逆。

部署、删数据、改生产配置——这些是不可逆的(或者逆转成本极高)。jar 推上去旧版本没了,数据删了找不回来。这种动作必须留一双人的眼睛,哪怕只是「按下回车前看一眼」。

测试失败、代码 bug、评审告警——这些是可逆的。修错了再来一遍,合并前还能撤。可逆的事,才适合交给机器自治。我们的夜间 Loop 只动代码、不合并、不部署,就是这个边界。

标准二:有没有客观验证手段。

「测试过没过」是客观的——跑一遍测试套件,绿就是绿,红就是红。所以夜间测试失败可以交给 Agent 修,修完用同一套测试验证。

「这段代码该不该这么写」是主观的——没有客观对错,只有 trade-off。所以定时评审只能告警,不能自动改。让 Agent 自动改「它觉得更好的代码」,改完用什么验证?没有客观尺子,就只能靠人。

有没有客观验证,是能不能自治的标志。客观的事交给机器,主观的事交给人,这是分工的底层逻辑。

标准三:失败成本。

备份失败了、告警没发出来、Loop 卡死了——这些事的失败成本,决定了要不要留人工兜底。我们每天备份上传完不通知人(成功是常态),但失败一定飞书告警,因为「备份静默失败」是灾难片的标准开头。

Loop 也有熔断:夜间测试如果连续跑不下去、Agent 反复修不好,不会无限烧 token,会停下来叫人。机器兜底归兜底,但不能让它在死循环里空转——这跟评审回路里的「10 轮熔断」是同一种克制。

把三条标准叠在一起,我们的边界就清晰了:可逆 + 有客观验证 + 失败成本低 → 全自治(夜间测试修复);不可逆 / 无客观验证 / 失败成本高 → 人手 + 命令化(部署、排障、备份确认)。

参考文章的「全自主闭环」对应的是可逆 + 客观验证 + 低失败成本的极致形态。我们离它有距离,不是做不到,是很多环节不满足这三条。强行自治不满足条件的事,不是先进,是冒险。

反模式:这些做法不算自动化运维

讲完正确做法,有必要说说哪些不算自动化运维。这些反模式在团队内外都见过,看着在自动化,实际既不安全也不省心。

反模式一:把部署脚本塞进 CI 就以为「自动化了」。 很多人觉得「我有 GitLab CI 跑部署脚本」就是自动化运维。但如果没有部署后的验证(冒烟、健康检查),这只是「把不可逆动作自动化了」,出问题更难发现。没有验证的部署自动化,比手动部署更危险——因为它给你「已经自动化了、应该没问题」的虚假安全感。

反模式二:让 Agent 直接改生产代码或配置。 这是最诱人也最危险的反模式。让 AI 直接连生产库改数据、直接改线上配置、直接重启服务——快是真快,但不可逆 + 不可审计。一旦改错,你既不知道改了什么(没有 MR、没有 review),也撤不回来。AI 自治的边界,永远是「可逆 + 有验证」,越界就是玩火。

反模式三:Loop 失败不告警。 自动化流程最怕的不是失败,是静默失败。Loop 跑挂了、定时任务停了、备份没传上去——如果没有告警,你会一直以为它在正常跑,直到某天需要恢复才发现「原来这一个月都没备份」。任何自治 Loop,失败都必须有声响。我们的每个 Loop 都挂了飞书 webhook,PASS 和 FAIL 都通知,就是为了让「它还在跑」这件事是可见的。

反模式四:测试数据自动清理。 前面讲 /prod-testing 时提过。让自动化流程在生产库里 DELETE 数据,等于把删库的权力交给脚本。一个 WHERE 条件写错、一个时区算错,就是生产事故。测试数据的清理,宁可手动,不可自动——除非你有一个独立的、和生产库物理隔离的测试库。

识别这四个反模式,比记住正确做法更重要——它们的共同点是**「形式上自动化了,实质上的安全边界没守住」**。看起来都省了事,实际把风险藏进了你看不见的地方。运维自动化的成败,不在于你搭了多少流水线,而在于关键的安全边界——发布门禁、验证闭环、事故响应——有没有被自动化绕过去。
