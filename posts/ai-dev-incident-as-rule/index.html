出事 → 群里炸锅 → 复盘会 → 写一份 wiki → 链接甩群里 → 三个月后没人记得 → 同样的坑再踩一次。我见过太多事故复盘,流程都长这一个样。

后来我想明白一件事:一份规范的价值,取决于它能不能被机器执行。多数团队的事故复盘产出物是「一份文档」,我的回答是「一条机器可执行的规则」。文档会过时、没人看,规则会一直生效——在 AI 协同研发的高频交付节奏下,这个差距被放大:交付越快,踩坑越快,如果每个坑只变成文档,挡不住下一次同样的坑。

我用 5 个真实事故案例,讲清楚「事故 → 规则」的转化路径:不同类型的事故固化成什么载体(hook/脚本/Agent规则)、怎么判断该用哪种、以及怎么让这套机制自我生长。

事故复盘最常见的死法

我见过太多事故复盘,流程都长一个样:

  1. 出事 → 群里炸锅 → 复盘会。
  2. 写一份 wiki → 链接甩群里。
  3. 三个月后没人记得,同样的坑再踩一次。

我自己也写过这种 wiki。后来我想明白一件事:一份规范的价值,取决于它能不能被机器执行。

  • 挂在 wiki 上的教训,是"希望下次有人记得"。
  • 固化成代码的教训,是"下次机器直接拦你"。

这两者的 ROI 差了一个数量级。

这个差距为什么是一个数量级。因为 wiki 依赖三个不确定环节:有人写(写的人得有时间)、有人看(看的人得有动力)、有人记(记的人得不出错)。三个环节任何一个断了,规范就失效。而代码规则只依赖一个环节:写一次(之后就自动跑)。一个是三连依赖,一个是一次投入,可靠性差一个数量级。而且 wiki 的「失效是静默的」——没人看也不会报错,你以为规范在生效其实没有;代码规则的「失效是显性的」——hook 没了、脚本坏了会立刻报错或构建红,你能马上发现。静默失效 vs 显性失效,这是 wiki 规范和代码规则的另一个本质差距。

所以这两年我们团队立了个规矩——事故即规范。每起线上事故或险情,复盘完不是写文档,是落成一条可执行规则。下面是我这几年攒下来的真实转化路径和几个例子。

规则的三种载体

先说方法论。我们把事故教训固化成三种东西,各有各的适用场景:

  1. git hook(运行时拦截):针对"绝对不能发生"的动作,比如推错分支、跑危险命令。在动作发生的那一刻直接拦下,不留任何侥幸。
  2. 脚本(提交前门禁):针对"跨层契约对不齐"这类编译器管不到的错误,放在 pre-commit,提交前就扫描。
  3. agent 规则(评审时检查):针对需要"理解语义"才能判断的问题,写进 Agent 的规则文件,让评审回路在每次提交时自动查一遍。

判断标准很简单:这条规则能不能不靠人、自己跑起来。能,就固化;不能,就老老实实写文档,但别指望它生效。

这三种载体的本质区别是「拦截时机」。git hook 在「动作发生的瞬间」拦截,是最早的关口;pre-commit 脚本在「提交前」拦截,次早;Agent 评审规则在「评审时」拦截,最晚。关口越早,问题暴露越早,修复成本越低——一个危险 push 在 hook 拦下,零成本;同一个问题如果放到评审才发现,代码已经写完、可能还跑了测试,返工成本高得多。所以载体的选择原则是:能早拦的别晚拦。能用 hook 的不用脚本,能用脚本的不留给 Agent。

还有个维度是「判断复杂度」。git hook 只能做简单判断(分支名、文件名),pre-commit 脚本能做中等判断(正则扫 SQL),Agent 规则能做语义判断(理解逻辑对不对)。判断越复杂,需要的「智能」越高,载体越靠后。硬要给复杂判断写 hook 是做不到的(hook 不会读语义);硬要给简单判断写 Agent 规则是浪费(Agent 评审是要花 token 的)。载体和能力要匹配,这个匹配本身就是工程判断。

三种载体的决策表

不是所有事故教训都该用同一种载体固化。我们的判断标准:

事故类型适用载体触发时机例子
绝对禁止的动作(推错分支 / 危险命令)git hook动作发生瞬间推 dev 分支 → 拦
跨层契约对不齐(URL/SQL/权限/迁移)pre-commit 脚本提交前JOIN 漏 tenant_id → 扫
语义层错误(逻辑不对 / 边界漏)Agent 评审规则评审回路自动查缺幂等校验 → 标红
结构违反(分层乱 / 循环依赖)架构测试(ArchUnit)构建时Controller 直连 DB → 红
流程疏漏(忘了跑测试 / 忘了回滚)CI 流水线强制合并前verify 没过 → block

从硬到软排序:hook > 脚本 > 架构测试 > Agent 规则 > CI。越靠前越硬(不可绕过),越靠后越灵活(可被覆盖但留痕)。事故的严重度决定载体——安全级事故用 hook,优化级问题用 Agent 规则。

这个排序有个实操含义:同一个教训,尽量往硬的载体固化。能用 hook 就别只用脚本(脚本可能被 --no-verify 绕过),能用脚本就别只留给 Agent(Agent 规则理论上可被 Agent 忽略)。越硬的载体越可靠,因为它不依赖任何人的自觉。但这不意味着所有规则都该用 hook——hook 太硬会卡死正常流程,只有「绝对不能发生」的事才值得用 hook。载体选择是在「可靠性」和「灵活性」之间权衡,硬载体可靠但不灵活,软载体灵活但可被绕过。

载体选择的反面教材

不是所有规则都该固化。有些"规则"固化了反而是负担:

  • ❌ 把"命名风格用驼峰"固化成 hook → 命名风格是 lint 的事,不该用 hook(粒度错了)。
  • ❌ 把"这个接口要小心"固化成 Agent 规则 → 太模糊,Agent 不知道怎么判"小心"。
  • ❌ 把一次性数据迁移的注意事项固化成长期规则 → 迁移完就废了,规则却永久占着上下文。

判断标准:这条规则是否会再次适用?会,固化;不会,写注释就够了。规则越多,Agent 的上下文越重,反而降低判断质量。

规则过载是个真实的风险。我见过一些团队「规则成瘾」,什么鸡毛蒜皮都固化成规则,结果 Agent 启动时加载几百条规则,上下文被撑爆,反而判断不准了——规则太多等于没规则,Agent 抓不住重点。所以固化规则要克制:只固化「会重复犯且后果严重」的,一次性的事、后果轻微的事,不固化。规则库要精不要多,每一条都该是「不固化就会再踩且踩了会疼」的。定期还要清理过时规则——业务变了、技术栈换了,旧规则可能已经不适用,留着只会污染上下文。规则库和代码库一样,需要定期「垃圾回收」。

案例 1:误推 dev → dev,加分支校验函数

有一次,有人(我就不说是不是我了)在本地切到 dev 分支拉了点东西,顺手把一个改动 commit 进 dev 再 push 上去——直接污染了主干分支。

好在是中午,被发现得早,回滚没费太大劲。

但如果发生在半夜、或者污染没被发现跑了一晚上,回滚就复杂了——dev 上的提交可能已经被别人 pull 走、基于它开了新分支,回滚一个节点会连锁影响。所以「早发现」是运气,「别让它发生」才是治理。复盘的重点不是「怎么回滚快」,是「怎么让推 dev 这件事不可能成功」。

复盘的时候大家都说"以后注意别在 dev 上 commit"。这句话的保质期大概两周。于是我们在 git hook 里加了个分支校验函数:本地分支非 feature/、bugfix/、hotfix/ 开头的,一律拒绝 commit、拒绝 push。

dev、main 这些分支想推都推不动,hook 直接报错退出。从那以后这类事故再没发生过。规则一次写好,永久生效,比任何群公告都管用。

具体的 hook 代码长这样:

# .git/hooks/pre-commit(分支校验函数)
#!/bin/bash
branch=$(git rev-parse --abbrev-ref HEAD)

# 只允许在 feature / bugfix / hotfix 分支提交
if ! [[ "$branch" =~ ^(feature|bugfix|hotfix)/ ]]; then
  echo "❌ 禁止在 $branch 分支提交。"
  echo "   请切到 feature/xxx 或 bugfix/xxx 分支再提交。"
  echo "   这条规则来自一次误推 dev 分支的事故。"
  exit 1
fi

# 同样的逻辑挂在 pre-push,双重兜底
# .git/hooks/pre-push
while read local_ref local_oid remote_ref remote_oid; do
  if [[ "$remote_ref" =~ refs/heads/(dev|main|master)$ ]]; then
    echo "❌ 禁止直接推到 $remote_ref。走 PR 流程。"
    exit 1
  fi
done

两个细节:第一,错误信息里写明"这条规则来自哪次事故"——下次有人被拦,看到这条信息就明白为什么有这条规则,不会想"这破规则烦死了"然后去删。第二,pre-commit 和 pre-push 双重兜底,因为 hook 也可能被 --no-verify 绕过,两道关比一道稳。

案例 2:并发守卫被"幽灵删除",加测试门禁

这个我在另一篇里详细写过。简单说:一次"搜索优化"提交,静默删掉了 270 行并发安全代码,而 mvn compile 照样通过——因为编译只校验语法契约,不校验行为。

根因不是写代码的人手抖,是当时的评审回路只跑编译、不跑测试,删了行为也没人发现。

排查这个过程值得讲。发现问题时,症状是「并发场景下偶尔出脏数据」。第一反应是怀疑并发控制没写好,去读代码发现并发守卫那段代码根本不在了——不是注释掉了,是整段被删了,而且 git 历史里那次提交的 message 写的是「搜索优化,清理冗余」。AI(或人)在优化搜索逻辑时,把并发的 synchronized 块和相关的守卫逻辑当成了「搜索的冗余代码」删掉,因为它看起来「和搜索无关」。

这事最坑的地方是:mvn compile 全绿。因为删除是合法的语法操作——删掉一段有效代码,剩下的代码依然能编译。编译器只检查语法,不检查「这段并发控制该不该删」。所以编译过 ≠ 代码对,这个认知就是这么被逼出来的。

修复方式不是"以后评审多看两眼",而是把分层测试 + 契约扫描 + E2E 直接写进了提交前置门禁。没有 verify 通过,根本进不了提交环节。

教训被翻译成了一条门禁规则,不是一份"请大家重视并发安全"的文档。这次事故还催生了另一个规则:任何「删除超过 50 行」的改动,评审 Agent 要重点标注「确认这些代码不是并发/安全相关」。因为大段删除是最容易误删关键代码的操作,给它加专门的审查,从源头防「幽灵删除」。

案例 3:多租户越权,固化成 SQL 审查清单

这条最值得说。多租户 SaaS 项目,租户之间是行级隔离,所有业务表都带 tenant_id,每个 JOIN 都得带上它做过滤。

有一次,一段新写的 SQL 在某个 JOIN 上漏了 tenant_id——结果是跨租户越权,A 租户能查到 B 租户的数据。这是安全级的事故,不是性能 bug。

这类问题编译不报、单测也可能蒙混过去,得有人盯着 SQL 看。靠人盯迟早漏。我们的做法是把它固化进两处:

  • 一是 Agent 评审规则,每次提交时自动扫一遍 SQL 里有没有 JOIN 漏租户字段。
  • 二是写进规范文件的启动检查清单,任何 Agent 接触多租户相关代码都会加载它。

从"一次差点出大事"到"每次提交自动检查",中间没有靠任何人的记忆。

这次事故的教训值得展开。多租户越权这类问题,它的可怕之处在于「编译不报、测试可能蒙混、运行不立即暴露」。A 租户查到 B 租户的数据,不会报错(数据是合法的),不会 crash(查询正常返回),只有人去对账时才发现「这个数据怎么串了」。这意味着它能在生产环境潜伏很久,直到造成实际损失(比如 A 租户看到 B 租户的订单)才被发现。所以这类问题的防线必须前置——不能等运行时发现,要在写代码那一刻就拦住。Agent 评审规则扫 SQL 就是这个前置防线:每次提交自动扫 JOIN 有没有带 tenant_id,不带就标红,根本不进主干。

还有一个延伸教训:多租户隔离不能只靠 SQL 层的 tenant_id。纵深防御要多层——SQL 层(带 tenant_id)是第一层,Service 层(按租户过滤)是第二层,数据层(RLS 行级安全,如果数据库支持)是第三层。任何一层失守,其他层还能兜。单靠一层(不管哪层)都是脆弱的,因为那一层的规则一旦被绕过(比如有人写了个不带 tenant_id 的直查),整个隔离就崩了。多租户隔离这种安全级要求,必须冗余防御。

案例 4:契约脚本被误删,恢复并升级成四层

还有一次更黑色幽默:我们之前写的四道契约闸(URL↔路由、SQL↔DDL、权限↔种子、迁移版本唯一)——这套脚本本身,在某个大重构里被人整块删掉了。

门禁没了,跨层错误又开始漏,直到下一次提交才被发现。

这事最讽刺的地方:我们写了一套「防代码出错」的规则,结果这套规则自己被当「无用代码」删了。防线自己失守,比被防护对象出错更隐蔽——因为删规则不会报错(规则不是必需的运行时代码,删了编译照样过),只有当下一个本该被拦的跨层错误漏进来,才会发现「咦,怎么没拦住」。所以这次发现得晚,跨层错误已经在主干待了几天。

复盘后我们做的事:恢复脚本,然后把它和架构测试(ArchUnit,强制 Controller→Service→Repository 分层)绑在一起,作为构建的一部分。

关键设计在这里:脚本可以被删,但删了构建会红——因为分层测试本来就该在那儿。两套机制互相兜底,任何一边被误删都能被另一边接住。

这个案例揭示了一个更深的问题:防线本身也是代码,也会被误删。你写了一堆 hook、脚本、规则来防业务代码出错,但谁来防这些防线本身出错。答案是多冗余——同一类问题用多个机制兜,任何一个被删了,还有别的能接住。契约闸被删了,架构测试(ArchUnit)还在;ArchUnit 被删了,Agent 评审规则还在。单点防线不可靠,多重防线才可靠。这和「多租户隔离要纵深防御」是同一个道理:安全不靠单点,靠冗余。

还有个实操建议:防线代码要加保护性注释。每个 hook、脚本、规则文件,头部都写明「这条规则来自 XX 事故,删了会导致 XX 后果」。这样下次有人重构想删它时,会看到注释知道它的来历和作用,不会轻易下手。我们的契约闸脚本头部就写着「删除将导致跨层错误无拦截,参见 incident-2026-XX」,删的人一看就知道不能删。

自我演进:当有人说"别再犯"

这套机制最让我满意的一点是它能自我生长。

当用户(或我自己)对 Agent 说"记住这个 / 以后别再犯这个错"的时候,这条话会被自动落盘成一条新规则,加进按文件类型触发、或 Agent 启动时加载的规则文件里。

也就是说,事故不用专门走"复盘 → 转规则"的流程,在日常对话里就沉淀下来了。规则越攒越多,而且每条都是真实血泪换来的,不是拍脑袋写的。

这套自我生长机制有个关键:落盘要及时且准确。及时,是指「别再犯」这句话说完,当场就落盘,不能「等有空再补」——等有空就是永远不补。准确,是指落盘的规则要能精确描述「什么时候触发、拦什么」。模糊的规则(比如「以后小心点」)落盘了也没用,Agent 不知道怎么执行;精确的规则(比如「涉及金额的 SQL 必须带 _fen 后缀」)才能被机器执行。所以自我生长的闭环是:人发现错误 → 说「别再犯」→ 系统追问「具体是什么规则」→ 人精确描述 → 落盘。这个追问环节很重要,它把模糊的教训逼成精确的规则。

还有一个隐性收益:规则库本身就是项目的「事故档案」。每条规则背后都是一次真实事故或险情,读规则库就知道这个项目踩过哪些坑、防线在哪里。新人(或新 Agent)读规则库,等于快速继承了团队的所有事故经验,不用重新踩一遍。这是规则化的复利——它不只防未来,还沉淀了过去。

一句话收尾

我越来越觉得,工程规范这件事,衡量标准不该是"文档写得多全",而是**“机器能自动执行的占多少”**。

  • 挂在 wiki 上的规范会过时、会没人看、会和实际代码脱节。
  • 跑在 hook、脚本、Agent 里的规范,只要代码还在动,它就一直生效。

事故复盘的终点,不该是一篇文档,该是一条规则。

这个判断在 AI 时代尤其成立。以前「写规则」是人的活,成本高、速度慢,一个团队一年固化不了几条规则。现在 AI 能快速把教训翻译成 hook、脚本、规则代码,固化速度跟上了事故发生的速度。以前是「事故比规则快」(踩坑的速度超过固化规则的速度),现在有了 AI 协同,「规则能跟上事故」了。这是 AI 对工程治理的一个隐性贡献——它不只是写业务代码快,也让「从事故到规则」的转化变快,让团队的安全网织得更密。

还有一个反直觉的点:规则越多,团队越快。听起来矛盾,但确实如此。因为规则替代了「人盯人」的成本——没有规则时,每个改动都要老人审、老人盯,老人成了瓶颈;有了规则,大部分错误被机器拦下,老人只在「规则拦不住的复杂判断」时介入。规则把老人的精力从「审常规」解放出来,用在「定规则和处理例外」上。所以规则多的团队反而快——因为它的质量保障不依赖人盯,依赖系统拦,系统的速度比人快得多。

最后给个可操作的指标:团队的「规则覆盖率」。统计一下,团队踩过的坑里,有多少已经固化成了机器可执行的规则(分子),还有多少只是写进了文档或留在人脑里(分母)。这个比值越高,团队越「自治」——意味着大部分已知坑都被机器拦住,不需要人反复提醒。这个比值低,说明团队还在靠「人盯人」,效率低且不可靠。提升这个比值的方式很简单:每起事故,必须产出一个固化规则,不固化不算复盘完。把这条写进复盘流程的硬要求,规则覆盖率自然就上去了,团队也就从「人治」逐步走向「系统自治」。这个转变看起来慢,但一旦规则库攒到一定密度,团队的质量保障就基本不依赖人盯了,这才是工程治理成熟的标志,也是团队能规模化扩张而不失控的根基所在。
