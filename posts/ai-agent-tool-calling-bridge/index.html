AI 助手要"能干活"而不只是"能聊天",靠的就是 Tool Calling(工具调用)。这篇讲我怎么把工具定义和 Spring AI 解耦:业务同学写起来零框架负担,换框架一行不动,写操作有确认门禁,每次调用全审计。

一、要解决的核心问题

我们的多租户 AI 助手要服务运营同学。运营同学说的不是"帮我查订单"这种结构化指令,而是自然语言:“今天的订单怎么样"“把这几个待审的代下单批一下"“这几个订单统一备注成已沟通”。

对模型来说,这些话背后是不同的业务动作。难的不是对话,是怎么把自然语言安全地落到真实业务方法上。这中间有几个工程问题必须先回答:

  1. 谁决定调哪个工具? 不能写 if 用户问订单就查订单 的硬编码分支。自然语言的组合空间是无限的,if-else 永远写不完。
  2. 工具怎么定义? 业务同学写一个工具,要不要被 AI 框架的 API 绑架?今天用 Spring AI,明天换 LangChain4j,要不要把所有工具重写一遍?
  3. 多租户怎么穿透? A 租户的运营说话,工具只能查 A 租户的订单,这个隔离边界怎么从对话层一路传到业务 service?
  4. 写操作怎么兜底? 模型要是手一抖调错写操作(比如批量改错订单),怎么拦?Agent 直接改生产数据,谁来背锅?

这篇就是回答这四个问题。核心思路一句话:调度交给模型,工具实现交给业务,中间架一层自研桥接,把两边彻底解耦。

这四个问题里,第四个(写操作兜底)是最容易被新手忽略、却最先在真线上出事的。很多人做 Agent 工具调用,demo 阶段全用只读工具,觉得一切都很美好;一旦上写操作,模型一个理解偏差,批量改错一批订单,善后成本极高。所以这套设计从一开始就把「写操作和只读操作分开,写操作必须过确认门禁」当硬约束,而不是事后补丁。这个意识有没有,决定了你的 Agent 是 demo 还是生产系统。

二、整体设计

先看全貌。一次"运营问一句话、模型决定调工具、框架自动执行、结果回喂模型续生成"的完整链路:

几个关键设计决策,后面逐个展开:

  1. 不自研 Agent 框架,只桥接工具。工具调用循环交给 Spring AI,业务只实现工具本身。
  2. 工具定义和 Spring AI 解耦。自研 @AiTool + ToolHandler 一层,业务代码看不到任何框架类型。
  3. 租户上下文用 toolContext 透传,多租户隔离从对话层穿透到工具层。
  4. 写操作走确认门禁。Redis token + 参数哈希 + 消费即删,防重放防误触。

三、自研工具层:@AiTool + ToolHandler

最关键的决定是:工具定义不依赖 Spring AI 的 API。原因很简单:工具是业务资产,框架会换。如果每个工具都写死 @Tool、依赖 Spring AI 的类型,换框架时所有工具都要改。我要的是:业务同学写工具就是写一个普通方法,框架换了他一行都不用动。

3.1 四个核心概念

自研一层薄薄的东西,就四个核心概念:

  • @AiTool:声明式注解,标注工具的 name / description / requiresConfirmation / module。
  • ToolHandler:函数式接口,业务实现它,收到参数 + 上下文,返回结果。
  • ToolContext:record,携带 tenantId / userId / conversationId,工具靠它做租户隔离。
  • ToolResult:record,统一的成功/失败返回结构,带 ok / data / errorMessage。

这层的类关系:

这张图是这层的全部设计:业务工具(OrderQueryTool)只依赖自研的 ToolHandler 接口、@AiTool 注解、ToolContext 和 ToolResult,全是自研类型,没有一个 Spring AI 的类。ToolRegistry 在启动时收集所有 ToolHandler bean,桥接层 AiToolBridgeConfig 负责把它们适配成 Spring AI 的 FunctionToolCallback。换 AI 框架时,只改桥接层,业务工具一行不动。

3.2 业务同学怎么写工具

看真实代码就懂了。一个只读查询工具:

@AiTool(name = "query_orders", description = "查询订单列表,支持按状态/时间筛选",
        requiresConfirmation = false, module = "trade")
public class OrderQueryTool implements ToolHandler {

    @Override
    public ToolResult execute(Map<String, Object> args, ToolContext ctx) {
        // ctx.tenantId() 做租户隔离——这点是硬约束,下面单独讲
        OrderQueryDTO query = OrderQueryDTO.builder()
            .tenantId(ctx.tenantId())
            .status((String) args.get("status"))
            .startDate((String) args.get("startDate"))
            .build();
        List<OrderVO> orders = orderService.listOrders(query);
        return ToolResult.success(Map.of("orders", orders, "total", orders.size()));
    }
}

再看一个写操作工具,这是 requiresConfirmation = true 的关键区别:

@AiTool(name = "batch_tag_orders", description = "给多个订单批量打标签",
        requiresConfirmation = true, module = "trade")
public class BatchTagOrdersTool implements ToolHandler {

    @Override
    public ToolResult execute(Map<String, Object> args, ToolContext ctx) {
        // requiresConfirmation=true 时,这里只在确认门禁放行后才会被执行
        List<Long> orderIds = ((List<?>) args.get("orderIds")).stream()
            .map(o -> Long.valueOf(o.toString())).toList();
        String tag = (String) args.get("tag");
        int affected = orderService.batchTag(ctx.tenantId(), orderIds, tag);
        return ToolResult.success(Map.of("affected", affected));
    }
}

业务同学的工具类,完全看不到 Spring AI 的任何类型。ctx.tenantId() 直接拿来用,不需要从 ThreadLocal 取,也不需要从请求头解析。租户上下文是框架透传进来的(下一节讲怎么传的)。这就是解耦的目标。

3.3 ToolResult:统一的成功/失败语义

ToolResult 用 record + 静态工厂,强制业务用 success() / error() 而不是随便 new:

public record ToolResult(boolean ok, Map<String, Object> data, String errorMessage) {
    public static ToolResult success(Map<String, Object> data) {
        return new ToolResult(true, data, null);
    }
    public static ToolResult error(String message) {
        return new ToolResult(false, null, message);
    }
}

这个统一返回结构是审计和确认门禁的基础:桥接层和门禁不关心业务工具内部返回什么,只看 ok 是不是 true、data 是什么、有没有 errorMessage。模型拿到失败的 ToolResult 也会据此调整后续生成(比如告诉用户"查不到这个订单”)。

3.4 ToolRegistry:启动时扫描注册

工具 bean 不需要手动注册。ToolRegistry 在 Spring 启动时自动扫描所有 ToolHandler bean,按 @AiTool.name() 注册进一个 Map<String, ToolHandler>,同时生成对应的 function schema 供桥接层使用:

@Component
public class ToolRegistry {
    private final Map<String, ToolHandler> handlers = new ConcurrentHashMap<>();
    private final List<ToolDefinition> definitions = new CopyOnWriteArrayList<>();

    @PostConstruct
    void scan() {
        applicationContext.getBeansOfType(ToolHandler.class)
            .values()
            .forEach(handler -> {
                AiTool meta = AnnotationUtils.findAnnotation(
                    AopProxyUtils.ultimateTargetClass(handler), AiTool.class);
                handlers.put(meta.name(), handler);
                definitions.add(buildDefinition(meta));
            });
    }

    public List<ToolDefinition> getAllToolDefinitions() { return definitions; }
    public ToolHandler get(String name) { return handlers.get(name); }
}

业务同学加一个工具,就是写一个带 @AiTool 注解的 ToolHandler 实现类,@Component 标上,Spring 启动时自动被发现、自动注册、自动暴露给模型。零配置接入——这是这套设计的体验底线。

零配置接入背后还有个容易被忽视的工程细节:工具的参数 schema 是怎么生成的。业务同学写工具时,参数是从 Map<String, Object> 里取的,并没有一个强类型的 DTO 声明每个字段。那模型怎么知道这个工具接受哪些参数?答案是 @AiTool 注解上还可以配一个参数描述,或者由桥接层根据方法签名反射生成。这里有个权衡:用 Map 换来了灵活性(加字段不用改 schema 生成逻辑),代价是失去了编译期的类型检查——参数名写错了,编译不报错,要等运行时模型调不通才发现。我们的折中是给关键字段加运行时校验(取不到就返回 ToolResult.error),既保留了灵活性,又不至于让拼写错误溜到线上。

这套「零配置 + 自动注册」还有个长期收益:工具数量的增长不会增加接入成本。从 16 个工具涨到 50 个,业务同学该写的还是那一个 @AiTool 类,注册逻辑一行都不用动。如果接入成本随工具数线性增长(每加一个要改配置文件、改注册表),工具一多就成了维护负担,团队就会抵触加新工具,Agent 的能力也就被「接入麻烦」卡死了天花板。把接入成本压成常数,Agent 的能力才能随业务自然生长。

四、桥接层:适配到 FunctionToolCallback

自研层定义工具,但执行还得靠 Spring AI。一个 AiToolBridgeConfig 把两边接起来:

@Configuration
public class AiToolBridgeConfig {

    @Bean
    public List<FunctionToolCallback> toolCallbacks(ToolRegistry registry) {
        return registry.getAllToolDefinitions().stream()
            .map(def -> {
                ToolHandler handler = registry.get(def.name());
                return FunctionToolCallback.builder(def.name(), (args, springCtx) -> {
                        // 关键:从 Spring AI 的 ToolCallingContext 取租户上下文,
                        // 构造自研的 ToolContext,把隔离边界一路传到业务工具
                        ToolContext toolCtx = new ToolContext(
                            (String) springCtx.getContext().get("tenantId"),
                            (String) springCtx.getContext().get("userId"),
                            (String) springCtx.getContext().get("conversationId"));
                        return handler.execute(args, toolCtx);
                    })
                    .description(def.description())
                    .inputType(Map.class)   // Spring AI 反射生成 JSON Schema
                    .build();
            })
            .toList();
    }
}

两个关键设计点:

第一,用的是 Spring AI 的 FunctionToolCallback(不是 @Tool 注解)。@Tool 注解是声明式的,适合方法级别的简单注册;我们要的是程序化、可批量、可注入上下文的注册,FunctionToolCallback.builder() 更合适。

第二,.inputType(Map.class)——让 Spring AI 反射生成参数 schema。工具参数不固定结构(查订单有查订单的参数、改商品有改商品的参数),用 Map 接,LLM 传什么 JSON 都能接到 Map<String, Object> 里,业务工具自己解析。这换来了参数的完全灵活性——加字段不改 schema 生成逻辑。

桥接层的责任边界

桥接层只做三件事,不做第四件:

  • 收集:从 ToolRegistry 拿所有工具定义。
  • 适配:把自研的 ToolHandler 包成 Spring AI 的 FunctionToolCallback。
  • 透传:把 ToolCallingContext 里的租户上下文提取出来,构造自研 ToolContext 传给业务工具。
  • 不做:调度逻辑(调哪个工具、循环多少轮)——这是 Spring AI 的 internalToolExecutionEnabled 干的。

这个边界很重要。桥接层薄到几乎无逻辑,所以它极稳定,业务工具和 Spring AI 各自演进都不影响它。

「薄」这个特性,是桥接层最值钱的地方,也最容易被破坏。一旦有人图方便,在桥接层里塞了业务逻辑(比如「query_orders 这个工具特殊处理一下,绕过租户校验」),桥接层就开始变厚、变脆,业务和框架的耦合就从这里漏进来。我们给桥接层立了条规矩:它只准出现「收集、适配、透传」三种代码,出现任何业务 if-else 都是坏味道。code review 时盯死这条,桥接层才能一直保持「换框架只改这一个文件」的可移植性。这条纪律不立,半年后桥接层就会长成谁也不敢动的泥潭。

桥接层选 FunctionToolCallback 而不是 @Tool 还有个隐性理由:程序化注册便于做灰度。16 个工具不是一次性全上的,有的是先内测、有的是分租户开放。程序化注册可以在构建 FunctionToolCallback 列表时按条件过滤(比如某租户只暴露 8 个工具),这种动态控制是注解驱动做不到的。注解是编译期就定死的,程序化是运行时可裁剪的,后者对多租户场景友好得多。

五、让模型自动多轮调度:internalToolExecutionEnabled

工具注册完,聊天时开启 Spring AI 的自动调度:

OpenAiChatOptions options = OpenAiChatOptions.builder()
    .toolCallbacks(toolCallbacks)
    .internalToolExecutionEnabled(true)   // 关键:框架自动跑工具调用循环
    .toolContext(Map.of(
        "tenantId",        tenantId,
        "userId",          userId,
        "conversationId",  conversationId))
    .build();

internalToolExecutionEnabled(true) 是这套设计的发动机。它让 Spring AI 自动跑工具调用循环:模型说要调 query_orders → 框架调对应 FunctionToolCallback → 桥接层构造 ToolContext 调业务工具 → 拿到 ToolResult → 框架把结果喂回模型 → 模型基于结果继续生成。业务代码完全不写这个循环。

toolContext 把租户上下文透传进每次工具调用。模型决定调哪个工具时,框架会把这个 Map 塞进 ToolCallingContext,桥接层从里面取出来构造自研 ToolContext。多租户隔离穿透到工具层——业务工具里 ctx.tenantId() 直接用,不需要从任何隐式上下文(比如 ThreadLocal)取,显式、可测、可审计。

手写工具循环 vs internalToolExecutionEnabled

这里有个真实的取舍。要不要自己写工具调用循环?

方案业务代码量循环正确性调试性
手写循环(自己解析 tool_calls、执行、回喂、续生成)高(每轮 if/while,容易写错流式)自己负责(多轮流式极易踩坑)自己打日志,自由但要自己保证完整
internalToolExecutionEnabled(true)(我用)零(框架全包)框架保证(已处理多轮/流式/异常)靠审计日志(ai_assistant_tool_log)补可观测性

我选后者。手写工具循环的正确性成本很高——多轮调用叠加流式输出,边界条件极其容易写错。框架已经处理好这些,业务没必要重复造。代价是失去对循环的细粒度控制,但通过审计日志能补齐可观测性(下一节讲),这个取舍是值的。

六、16 个工具:只读与写操作的分级

我注册了 16 个工具,按副作用分两类:

类别工具是否需确认说明
只读查询query_orders / get_order_detail / list_products / get_shop_info / search_users / query_reviews / query_stats / list_wechat_templates否无副作用,模型直接调
写操作create_product / update_product / update_shop / update_order_remark / batch_remark_orders / batch_tag_orders / approve_proxy_order / publish_miniapp是有副作用,走确认门禁

写操作配 requiresConfirmation = true。这个标志在 @AiTool 注解上声明,桥接层和确认门禁据此判断要不要拦截。Agent 直接改生产数据,这一层兜底是必须的——模型要是把"备注这几个订单"理解成"备注所有订单",批量执行下去就是生产事故。

@Tool 注解 vs 自研桥接

为什么不用 Spring AI 自带的 @Tool 注解?这是个高频问题。

维度Spring AI @Tool自研 @AiTool + 桥接
业务工具是否依赖框架是(方法签名、类型都绑死 Spring AI)否(业务只写自研接口)
换框架成本高(所有工具重写)低(只改桥接层)
上下文透传框架自带 ToolContext自研 ToolContext 经桥接层透传
程序化注册弱(注解驱动)强(FunctionToolCallback.builder)
适合场景工具少、不打算换框架工具是业务资产、要长期演进

工具是业务资产,不该依赖框架 API。16 个工具涉及交易、商品、店铺、用户多个核心域,这是公司积累下来的业务能力,不是 AI 框架的一次性配置。自研一层的成本是写一个 AiToolBridgeConfig(几十行),换来的是业务工具和 AI 框架彻底解耦——这笔账划得来。

七、可观测性:全量工具调用审计

Agent 接入生产,可追溯比能力更重要。模型调了什么工具、传了什么参数、返回什么、耗时多少——这些必须能查。我落了一张审计表 ai_assistant_tool_log:

CREATE TABLE ai_assistant_tool_log (
  id              BIGINT PRIMARY KEY AUTO_INCREMENT,
  tenant_id       VARCHAR,
  conversation_id VARCHAR,
  tool_name       VARCHAR,      -- query_orders / batch_tag_orders ...
  args_json       TEXT,          -- 入参 JSON 快照
  result_json     TEXT,          -- 返回结果 JSON 快照
  success         TINYINT,       -- 成功/失败
  duration_ms     INT,           -- 耗时
  created_at      DATETIME,
  INDEX idx_tenant_conv (tenant_id, conversation_id),
  INDEX idx_tool (tool_name)
);

桥接层在每次工具调用前后切面式落日志(不在业务工具内部落,保持业务工具纯净):

// AiToolBridgeConfig 内部,简化版
long start = System.currentTimeMillis();
ToolResult result;
try {
    result = handler.execute(args, toolCtx);
} catch (Exception e) {
    result = ToolResult.error(e.getMessage());
}
long duration = System.currentTimeMillis() - start;

toolLogService.record(ToolLogRecord.builder()
    .tenantId(toolCtx.tenantId())
    .conversationId(toolCtx.conversationId())
    .toolName(def.name())
    .argsJson(JsonUtils.toJson(args))
    .resultJson(result.ok() ? JsonUtils.toJson(result.data()) : result.errorMessage())
    .success(result.ok())
    .durationMs(duration)
    .build());

return result;

这张表干三件事:故障排查(某次 Agent 回答异常,查 log 看工具调了什么)、效果分析(哪个工具调用最频繁、失败率最高、最慢)、合规审计(写操作改了什么,全可追溯)。出问题时按 conversation_id 一查,整条调用链清清楚楚。

审计日志还有个用法值得展开:反哺工具设计的优化。跑一段时间后,effect 分析能告诉你哪些工具的 description 写得不好——某个工具调用失败率高,往往不是工具实现的问题,是 description 让模型在「该不该调它」上判断错了。我们有一个工具 search_users,早期 description 写成「搜索用户」,模型遇到「查一下昨天注册的人」这种话,犹豫半天不知道该调它还是调 query_orders。后来 description 改成「按昵称、手机号、注册时间搜索用户,返回用户列表」,调用准确率立刻上来。这种优化没有审计数据是发现不了的——你得看到「模型在该调的时候没调」,才能反推是描述不够明确。审计日志不只是出事后查的,更是平时优化工具的依据。

还有个审计的细节:日志要在桥接层落,不在业务工具里落。这个选择是有意为之的。业务工具如果自己负责落日志,一来污染了业务代码的纯净性(违背「业务零框架依赖」的设计目标),二来每个工具都要重复写一遍日志逻辑,迟早有人漏写。放在桥接层统一落,既保证全量覆盖,又让业务工具保持纯粹。这个「横切逻辑集中在桥接层」的取舍,和日志、监控、链路追踪这类横切关注点的处理原则是一致的——它们不该散落在业务代码里。

落日志还有个性能取舍:审计是同步落库还是异步。同步落库,每次工具调用都要等一次数据库写入,对响应延迟有影响(尤其工具调用密集时);异步落库(丢进队列或线程池),延迟不受影响,但牺牲了「日志一定先于响应落盘」的强保证。Agent 场景下工具调用不算特别高频,我们选了同步,换取「出事时日志一定在」的确定性。如果以后工具调用频次涨上去,这个选择要重新评估。

八、写操作门禁:ConfirmationGate

写操作的危险性前面讲过。我设计了 ConfirmationGate——一个基于 Redis token 的确认门禁机制。核心设计:

  • Redis token:模型要执行写操作时,先生成一个一次性 token 返回给前端,而不是直接执行。
  • 参数 SHA-256 哈希:token 绑定本次调用的参数哈希,前端回传时带上原参数,门禁校验哈希一致才放行——防参数被中间篡改。
  • 5 分钟 TTL:token 5 分钟内有效,过期自动失效。
  • 消费即删:token 用一次就删,防重放——同一个 token 不能用来重复执行同一个写操作。

这是个设计机制,讲清思路就够:把"模型想执行写操作"和"写操作真正执行"之间,插一道需要运营人工确认的闸。模型生成的 token + 参数哈希进 Redis,前端弹确认框给运营看(显示工具名、参数摘要、影响范围),运营点确认,前端拿 token + 参数回调门禁,门禁校验通过才真正调业务工具。误触、重放、参数篡改在这道闸都会被拦下。

确认门禁还有个产品设计上的讲究:给运营看的确认信息必须说人话。早期我们的确认框直接展示原始 JSON 参数,运营看到 {"orderIds":[123,456,789],"tag":"已沟通"} 一脸茫然,只能盲点确认——等于门禁形同虚设。后来改成「将为 3 个订单(订单号 123、456、789)批量打上标签『已沟通』,此操作不可撤销」这样的自然语言摘要,运营才真的在看、在判断。门禁的价值不在多一道点击,在于它把模型的意图翻译成运营能理解的语义,让人有能力做这个判断。确认框的信息不设计好,门禁就是走过场。

门禁的另一个设计点是「撤销」语义。有些写操作天然可逆(打个标签,再取消就行),有些不可逆(发布小程序、审批通过)。我们在 @AiTool 上预留了一个语义标记,对不可逆操作,确认框会额外强调「此操作不可撤销」并要求二次确认。把可逆性也纳入门禁的考量,是因为不同写操作的风险等级差很大,一刀切地要求确认或一刀切地放行都不合理,门禁要能感知风险等级做分级处理。

九、几条沉淀下来的判断

做完这一套,我对"Agent 工具层设计"的判断:

  1. Agent 的是"模型调度工具",调度交给模型,业务只实现工具——这是 Agent 区别于普通聊天的本质。
  2. 别自研 Agent 框架,工具调用循环交给 Spring AI 的 internalToolExecutionEnabled,把精力放在工具定义的解耦上才是值得投入的设计。
  3. 工具是业务资产,不该依赖框架 API。自研 @AiTool + ToolHandler 一层,业务代码看不到任何 Spring AI 类型,换框架只改桥接层。
  4. 租户上下文用 toolContext 显式透传,让多租户隔离从对话层穿透到每个工具,不靠隐式 ThreadLocal。
  5. 写操作要确认 + 全量审计。Agent 接生产,可追溯、可拦、可确认,比能力更重要。
  6. 桥接层要薄。它只做收集、适配、透传,不做调度——这样它极稳定,业务和框架各自演进都不影响它。

把工具层做薄、做解耦,Agent 才能从"demo 能跑"变成"业务敢用"。这篇是 Agent 工具层的深度拆解,和 多租户 AI 助手总纲 里的第六章是对应关系。

深度展开:工具桥接的循环检测

Agent 调用工具,工具又触发 Agent 调用工具,一不小心就死循环:A 调 B,B 调 A,token 烧光,用户对着转圈的加载框干瞪眼。这不是理论风险,是真实踩过的坑——某个工具的 description 写得诱导模型「为了确认结果再调一次查询」,模型就真的来回调,几轮下来 token 烧掉一大半。

我们用三道防线兜住它。第一道是调用栈记录:每次工具调用,记录「是谁触发了这次调用」,如果调用栈里出现环(A→B→A),立刻中断。这道防线治的是工具之间的互相触发。第二道是深度限制:工具调用嵌套不超过 5 层,超过就停。有些循环不是严格的环,而是越调越深的螺旋,栈检测抓不住,深度限制兜底。第三道是token 预算:整轮对话设 token 上限,烧到 80% 警告、100% 强制停止。这是最后的安全网,前两道都漏了,token 预算至少保证不会无限烧钱。

这三道防线之外,还有个源头治理容易被忽略:工具的 description 写法。description 如果暗示「调用本工具后应该再调用 XX 工具验证」,就是在主动诱导循环。我们的规矩是 description 只描述「这个工具做什么、什么时候该用」,绝不描述「用完之后该干嘛」——后续调什么由模型根据返回结果判断,不由工具描述引导。源头不治理,光靠三道防线拦截,循环会频繁触发,用户体验是断断续续的「调到一半被打断」,一样糟糕。
