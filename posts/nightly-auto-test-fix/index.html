你睡的时候,机器没睡。

我给项目搭了一套夜间全量测试 + 自动修复的 daemon。每天凌晨 00:00,它自己拉 dev 最新代码,跑 8 步全量测试——后端单测、集成测、运营单测、运营 E2E、前端 H5 E2E、小程序编译验证。挂了,Claude Agent 自动诊断日志、切 fix 分支改代码、用完整测试套件回归一遍,过了才提 MR(不合并),飞书推个「已修复 + MR 链接」叫醒你。

第二天你打开飞书,要么是「全过」,要么是「已修复待 review」,要么是「修不动了快来」。没有「早上来发现昨晚合的代码炸了」这个选项。

这篇拆开讲这套系统怎么设计的、为什么这么设计。

整体流程:五个阶段

每次触发(定时或手动 --once)跑五个阶段,核心逻辑是:测试归测试,开发者的工作区不能动。

Phase 0·预检:先把现场记下来

平台检测、Node 初始化、抢运行锁(防两个 daemon 同时跑)。关键是 save_state——把每个仓库当前的分支名记到文件里。这一步是后面无损还原的伏笔。

Phase 1·工作区准备:切到 dev,但你的改动不丢

开发者可能正卡在某个 feature 分支,有未提交的改动。脚本切 dev 之前,先把这些改动 stash 保护起来(git stash push -u -m "nightly-test: <repo>"),再 checkout dev 拉最新。你的活儿一条都不会丢。

这里特意用 stash 而不是 auto-commit,代码注释写得很直白:用 stash 而非 commit,避免污染任何分支的历史。auto-commit 会往你当前的分支塞一个 wip 提交,即便后面能回退,也会在 reflog 里留痕、搞乱提交树;stash 是 Git 原生的暂存机制,用完 git stash pop 恢复,干干净净。stash 前还会自动确保 .nightly-test/ 在 .gitignore 里——否则日志目录会被 stash 走,导致后续写不了状态文件。这些边角细节,踩过才会写进去。

Phase 2·全量测试:8 步,fail-fast

8 步按依赖串起来,能并行的并行(后端单测 + 前端单测一起跑)。任何一步挂,后面不跑,直接进 Phase 3。不浪费时间在已经没意义的步骤上。

Phase 3·失败处理:三种情况,三种走法

这是整套系统最有意思的地方。失败不是一刀切,分代码问题和环境问题:

  • 环境问题(后端启不来、DB 不可达):跳过修复,直接飞书告警叫人。Agent 修不了环境,硬上只会越改越乱。
  • 代码问题(编译错、断言失败、NPE):Agent 修 → 脚本验证 → 提 MR,三步走。
  • Agent 修不动(根因定不下来):输出分析报告,不强行改,飞书通知。

代码修复这一路,有一个我特别坚持的设计:Agent 改完,不直接提 MR,先把 fix 分支拉回来跑一遍完整 8 步测试。

为什么?防回归。Agent 只看了失败那一步的日志,它的修复可能把别的步骤搞挂。只重跑失败步骤会漏,必须全量回归。验证通过才提 MR;验证不过,fix 分支直接删,飞书告诉你「修复未通过验证」,宁可让人来也别合进坑。

具体看 Agent 的修复流程:run_agent_fix 先从 dev 切出 fix/nightly-${run_id} 分支,然后调 claude -p "$prompt" 让 Agent 干活。给 Agent 的 prompt 里有几条硬约束,这几条约束本身就是踩坑沉淀的:

  • 只修复能明确判断的代码问题(编译错、单测断言失败、NPE 等)
  • 无法确定根因的(环境问题、E2E flaky、数据问题):输出说明,不要强行改代码
  • 不要修改测试断言(除非断言确实写错了)
  • 不要创建 MR(由外部验证步骤负责)

最后一条尤其关键——把"建 MR"这个动作从 Agent 手里拿走。Agent 的任务是改代码,建 MR 是流程动作,两件事的判断标准不一样。Agent 自己建 MR 容易跳过验证就合,拿走之后必须先过完整回归才由脚本建 MR。这是"信任但要验证"在流程上的体现。

注意 Agent 跑修复时给的 --allowedTools 是 Read,Write,Edit,Bash,Grep,Glob,含 Write/Edit;而会话级那套 review 只给 Read,Bash,Grep,Glob。review 是只读审查,fix 是要动手改,工具权限不一样。这个区分本身就是一道安全闸——review 的 Agent 没法误改代码,因为它根本没有写权限。

Phase 4·恢复:always,含 trap 兜底

不管测试过没过、修没修,最后一定 restore_state——把每个仓库切回 save_state 记录的原始分支,fix 分支也清理掉,后端停掉,锁释放。脚本被 kill(SIGTERM)时 trap 也会兜底执行。第二天你打开电脑,工作区和你昨晚离开时一模一样。

两个我觉得最值的设计

save_state / restore_state 工作区保护。很多人搭夜间 CI,测试是跑通了,但开发者第二天发现自己分支被切走了、改动被吞了。这套用「先记后切 + stash 保护 + 必然恢复」三道保险,把开发者工作区当成不可侵犯的领地。测试是测试,你的活是你的活,井水不犯河水。

这里还有个抢锁的细节值得讲。daemon 要防两个实例同时跑,我们没用 flock(跨平台麻烦),用的是 mkdir 原子操作当锁——mkdir 在同一目录下要么成功要么失败,天生原子,再往锁目录里写个 PID 文件。更妙的是它带了僵尸锁自愈:每次抢锁时检查持有者 PID 还活着没(kill -0),如果持有者进程已经死了但锁没清,就自动清掉重抢。报错文案也给人留了退路:另一个测试实例正在运行。如果确定没有,手动删除: rm -rf .nightly-test/run.lock。

修复走完整回归而非单步重试。这是防 AI 修复的最大坑——AI 改一处容易,但它不会主动检查「这个改动会不会搞挂别人」。强制全量 8 步回归,等于把 AI 的修复也当成可疑代码审查一遍。AI 修的代码,也得过 AI 自己设的关。

修复完怎么建 MR:一个踩出来的兜底

修复验证通过后要建 MR,这里有个真实踩出来的坑。脚本本来想调 create-mr.sh 建合并请求,但那个脚本是 MCP-first 设计——优先走云效 MCP,只在传 --allow-fallback 时才退化成脚本建 MR;而且它只认 --source/--target/--title 这种命名参数。夜间脚本调用时传的是位置参数,对不上,主路径直接失败。

实际兜底靠的是 Git 的 push option:git push -o merge_request.create -o merge_request.target=dev。这个 fallback 比调 MCP 脚本还稳,不依赖外部服务。MR 标题模板是 fix: 夜间自动修复 — ${failed_step} (${run_id})。踩过这个坑才明白:自动化的最后一步(建 MR)最依赖外部系统,也最容易断;留一个不依赖任何外部服务的 Git 原生兜底,才能让整条链路真正 7×24 跑得起来。

8 步测试的真实长相

前面一直说"8 步全量测试",具体是哪 8 步、各自跑什么命令,这里展开。这 8 步按依赖串起来,能并行的并行(后端单测和前端单测一起跑):

步骤label目录命令
1backend-unitbackendmvn -q test
2backend-integrationbackendmvn -q verify -DskipITs=false
3frontend-unitfrontendpnpm test(jest)
4ops-unitoperation-backendpnpm test(vitest)
5backend-startupbackendmvn spring-boot:run(拉起后端给 E2E 用)
6ops-e2eoperation-backendpnpm run test:e2e(playwright)
7frontend-e2e-h5frontendpnpm run test:e2e:h5
8build-mp-weixinfrontendpnpm run build:mp-weixin

每步有 30 分钟超时(TEST_STEP_TIMEOUT 默认 1800 秒)。超时这块跨平台兜底很讲究:有 timeout 命令用 timeout,macOS 上没有就用 gtimeout,实在没有就用 perl -e "alarm $sec; exec @ARGV" 兜底。跨平台 shell 脚本的一多半代码量,都在这种"有没有那个命令"的兜底上。

跑完每步会落盘几个文件:step-<runid>-<label>.log(完整日志)、step-exit-<runid>-<label>.txt(退出码)。日志文件大小本身就能说明问题——前端单测日志动不动 3.6 MB,后端单测 358 KB,差别就是测试体量的差别。

自动修复的三种失败场景

落到实际,修复会撞上三种情况,脚本都做了分流:

场景行为
Agent 改完,全量回归通过创 MR 到 dev(不合并) + 飞书「已修复 + MR 链接」
Agent 改完,全量回归没过删 fix 分支 + 飞书「修复未通过验证,需人工」
Agent 定不下根因输出分析,不动代码 + 飞书通知

最后一种我特意留的——Agent 不确定就别说「已修复」。强改只会越改越糟,不如把分析交出来让人来。这跟 PUA 那个 skill里的「穷尽方法论再说搞不定」是同一个思路:坦诚比硬撑值钱。

这种"修不动"的情况,真实的飞书文案是这样的:Agent 分析后未找到可修复的代码问题。失败步骤: ${failed_step}。需人工排查。——明确告诉你是哪一步挂了、Agent 试过了但没辙,而不是含糊地说"已修复"骗你合进去。我宁可早上看到"修不动了快来",也不想看到"已修复"结果 review 时发现改了个寂寞。

一次真实运行的快照

光说机制可能还是抽象,贴一次真实跑出来的结果。2026-07-05 10:31 跑了一次(--no-fix 模式),.nightly-test/ 下留了完整的痕迹:

  • last-run-id.txt → 20260705-103155
  • result-20260705-103155.json → status: "fail", failed_step: "backend-unit"
  • step-exit-...-backend-unit.txt → 1(后端单测挂了,退出码 1)
  • step-exit-...-frontend-unit.txt → 1(前端单测也挂了——因为这两个是并行跑的,都跑了)
  • git-state-20260705-103150.txt → 当时开发者卡在 feature/wechat-article-sync 分支

这是个绝佳的演示案例:fail-fast 生效了——后端单测挂了之后,集成测(依赖后端)直接没跑,省了时间;并行跑的前端单测不受影响继续跑完;工作区状态被忠实记录,restore_state 会把分支切回 feature/wechat-article-sync。整个失败的现场被完整保留下来,第二天开发者一打开,看到的就是"哪一步挂了、退出码多少、我当时在哪个分支",不用回忆也不用重新复现。

为什么不用系统 crontab + 普通 CI

我另一套 会话级定时 review是挂在 claude 会话里的,窗口关了就停。这套不一样——它是服务器上的 daemon,独立于任何会话,7×24 跑。

跟普通 CI 的区别在「自动修复」。Jenkins/GitLab CI 跑完测试给你红灯,修不修你自己看着办。这套是红灯亮了,Agent 自动上场,修完还防回归地验一遍。CI 报问题,这套报「问题 + 已修复 + MR 待 review」。多出来的,是修复这个动作。

横向对比一下主流方案:

维度Jenkins/GitLab CI会话级 cron-review这套夜间 daemon
运行位置CI 服务器claude 会话内业务服务器
触发push/PR/手动会话内每天定时7×24 定时
测试失败后报红灯报告 + 告警自动修复 + 防回归 + MR
工作区保护CI 隔离环境,无此问题切分支有风险save_state/restore_state
会话关闭不影响停止不影响

这套填的空白很明确:会话级方案关了就停,普通 CI 不修。daemon 是「7×24 跑 + 失败自动修」的组合,前两个都做不到。

这套 daemon 的实现也值得说一句——它不依赖系统的 crontab 或 launchd,是自管理的。daemon_start() 用 nohup bash "$0" _daemon_loop & 起一个后台进程,_daemon_loop() 里 while true 循环,每轮算下一次触发的时间,然后分段 sleep 300(5 分钟)醒一次。为什么要 5 分钟醒一次而不是直接 sleep 到明天?因为这样能响应 stop 命令——stop 的实现就是把 PID 文件删掉,循环每 5 分钟检查一次 PID 文件还在不在,不在就退出。如果一觉 sleep 10 小时,stop 就得等到第二天才生效。这种"分段 sleep + 检查信号"是 daemon 实现的常见手法,比依赖系统调度器更可控。定时配置持久化在 .nightly-test/schedule.conf(写入 hour/min/no_fix),重启机器后 daemon 还能按上次的配置继续跑。

怎么装

# 1. 服务器上(macOS/Linux 都行,Windows 用 Git Bash)
cd /path/to/your-fullstack-repo
# 脚本已在 scripts/cron/nightly-test.sh

# 2. 配飞书通知(可选,不配也能跑)
export FEISHU_WEBHOOK_NIGHTLY_TEST="https://open.feishu.cn/open-apis/bot/v2/hook/YOUR_TOKEN"

# 3. 服务器无桌面 Chrome,加这个
export NIGHTLY_SERVER_MODE=1

# 4. 启动 daemon(默认每天 00:00)
bash scripts/cron/nightly-test.sh start

# 自定时间 + 关闭自动修复
bash scripts/cron/nightly-test.sh start --time 02:00 --no-fix

常用命令:

命令作用
start [--time HH:MM] [--no-fix]启动 daemon,默认 00:00
stop停止
status看运行状态和上次结果
history [N]最近 N 次历史(默认 7)
--once [--no-fix]立即跑一次,不循环

第二天你会收到什么

飞书通知按结果分三种,一眼就知道该干啥:

全过(最常见,什么都不用做):

[PASS] 全量测试通过
所有步骤通过。平台: macos
RUN_ID: 20260705-080000

已修复待 review(挂了但 Agent 修好了,你早上 review MR):

[FAIL] 测试失败 — Agent 已修复并验证通过
失败步骤: frontend-unit
修复分支: fix/nightly-20260705-080000
MR: https://codeup.aliyun.com/xxx/repo/-/merge_requests/123

修不动(Agent 也搞不定,需要人工):

[FAIL] 测试失败 — Agent 修复未通过验证
失败步骤: frontend-unit
修复分支: fix/nightly-20260705-080000(已清理)
需人工排查。

最关键的差别:第二种是这套系统最大的价值——你早上不用从看日志开始,直接看 MR review。第一种你忽略,第三种你才需要动手。

常见坑

部署过程中踩过几个,提前说,免得照抄的人也踩一遍:

claude CLI 找不到。daemon 启动时 PATH 极简,nvm 没加载。脚本里有显式 source nvm.sh + 指定 Node 22,但你如果换机器要确认 ~/.nvm/ 路径对得上。

后端启动超时。脚本 Step 5 拉起后端,如果 MySQL/Redis 不可达,这步会卡。确认测试库 192.168.100.67:3306 和 :3379 网络可达——这一步环境问题,Agent 修不了,会直接飞书告警叫人。

E2E 在服务器上跑不起来。服务器无桌面 Chrome,Playwright 找不到浏览器。必须 export NIGHTLY_SERVER_MODE=1 用 Playwright 自带的 Chromium,或者跑一次 pnpm run install:e2e。这个坑坑在它只在服务器上冒出来,本地开发完全遇不到——本地有 Chrome,Playwright 直接用;一上服务器,没有桌面环境,它就找不到浏览器了。NIGHTLY_SERVER_MODE=1 就是告诉脚本"这是无桌面的服务器,别找系统 Chrome,用 Playwright 内置的"。

分支没切回来。脚本被 kill -9 强杀时 trap 不执行,工作区会停在 dev 分支。用 kill(SIGTERM)别用 kill -9;真出问题了,看 .nightly-test/git-state-*.txt 手动切回去。

kill -9 这个坑值得多说一句。我们的 trap 注册了三个信号:trap cleanup_on_exit EXIT INT TERM。EXIT/INT(Ctrl-C)/TERM 都能触发清理(停后端、恢复分支、恢复 stash、释放锁)。但 SIGKILL(也就是 kill -9)是操作系统级别的强制终止,任何 trap 都拦不住——这是 OS 的设计,不是脚本的 bug。所以 daemon 的 stop 命令是先发 TERM,等一会儿没退才 kill -9 兜底,顺序就是为了给 trap 留执行机会。日常用的时候记住:停 daemon 用 stop 或 kill(发 TERM),永远别上来就 kill -9。

这套系统不修什么

说了这么多它能修的,得讲清楚它不修什么——这部分往往比"能修什么"更重要,因为它决定了系统不会帮倒忙。

环境问题不修。MySQL 不可达、Redis 连不上、后端启不起来,这些 Agent 一概不动,直接飞书告警叫人。原因很简单:Agent 改的是代码,环境问题是基础设施,Agent 没有权限也不该去动数据库连接配置。硬上只会越改越乱——Agent 可能会为了让测试过,把连接超时调大、把断言注释掉,这等于掩盖问题。

E2E flaky 不修。有些 E2E 本身就不稳定(网络抖动、第三方服务慢),Agent 分不清"我的代码挂了"和"E2E 本身 flaky",硬修会把测试改成永远过的样子。这种一律让人来判断。

根因定不下来的不修。Agent 分析完日志,如果它自己都说"可能是 A,也可能是 B,没法确定",那就输出分析报告,不动代码。强改一个不确定的根因,改错方向的概率比改对高。

这三条"不修"的纪律,本质是给 Agent 划了能力边界:它只修能明确归因的代码问题,其余一律交人。把边界划清,Agent 才不会在它不该出手的地方乱搞。

一个判断

测试自动化这件事,大部分人停在「自动跑测试、自动报结果」。再进一步是「测试失败自动通知」。但真正效率瓶颈不在测试本身,在测试失败之后的修复——人早上来,看日志、定位、改、提 MR,这一套下来一上午没了。

把这一步也自动化,关键是别盲目信任 AI 的修复。AI 改的代码必须用完整套件回归一遍,过了才让它进 MR,过不了宁可退回人工。这套机制运行下来,大部分夜里挂的代码问题,第二天早上已经有 MR 等 review 了,我只需要看一眼合不合。

这里有个效率上的体感:以前没这套系统,早上来发现 dev 挂了,从看日志、定位、改、提 MR,一上午就没了,而且这一上午全在救火,原本计划的工作全推迟。有了这套之后,早上来要么是"全过"(直接干自己的活),要么是"有 MR 等 review"(花 20 分钟 review 合掉),要么是"修不动"(才需要动手)。三种情况里,只有第三种会占用大块时间,前两种几乎不影响当天的节奏。把救火的时间还给开发,这才是这套系统最实在的收益。

信任,但验证。 这是整套系统唯一的设计原则。

这条原则不是我发明的,是工程界的常识。但这套系统把它落到了具体的流程里:信任 Agent 能修(给它 Write/Edit 权限、让它改代码),但验证它的修复(完整 8 步回归、过了才提 MR)。信任让它有机会发挥速度优势,验证堵住它犯错的口子。两者缺一不可——只信任不验证,就是放任 vibe coding;只验证不信任,那不如自己改,要 Agent 干嘛。

深度展开:夜间巡检的工程化设计

夜间巡检不是"晚上跑个 cron",是一套完整的工程化系统。核心要解决三个问题:什么时候跑、挂了怎么办、修不好怎么办。

什么时候跑:凌晨 2-4 点,业务低峰期。我们用过 0 点跑,结果和数据库备份任务撞了,资源争抢导致两边都慢。后来错开到 2 点,跑完正好 4 点,运营 9 点上班时报告已经在飞书群里了。

错峰这个细节是用代价换来的。0 点跑听着合理(新一天开始),但 0 点往往是数据库备份、日志轮转、各种定时任务的"早高峰",全挤在一起。全量测试要起后端、连数据库、跑 E2E,资源占用不小,和备份任务抢资源,两边都慢、还可能互相影响(测试把库搞脏,备份就把脏数据备份了)。错开到 2 点,避开了备份窗口,跑完 4 点,留足缓冲。定时任务的"什么时候跑"和"跑什么"一样重要,撞车比不跑还糟。

挂了怎么办:巡检任务自己也有可能挂(进程被 kill、OOM、机器重启)。所以巡检本身要带心跳——每 10 分钟往一个"巡检心跳文件"写时间戳,另一个监控任务扫这个文件,超过 30 分钟没更新就告警。守护进程的进程,也是工程化的一部分。

“守护进程的进程"听着绕,但这是分布式系统里很现实的问题。你搭了一个夜间巡检,谁来保证巡检自己活着?机器重启了、OOM 了、被误 kill 了,巡检挂了你都不知道,第二天以为"昨晚没报告=全过”,其实是巡检根本没跑。所以得有个外部的、更简单的监控盯着巡检的心跳——它比巡检本身更不容易挂(因为它啥也不干,就定时写个时间戳)。这是"谁来监督监督者"的工程化回答:用更简单的系统监督更复杂的系统。

修不好怎么办:AI 自动修复只跑一次,失败就停下,把失败用例 + 错误日志 + AI 的修复尝试打包发飞书,叫人早上处理。绝不让 AI 在夜里无限重试,那是烧 token 的无底洞。

“绝不让 AI 无限重试"这条值得强调。AI 修复失败后,最容易犯的错就是"再让它试一次”,然后陷入死循环——每次重试都烧 token、改出来的代码越来越偏、最后 dev 分支被改得面目全非。我们定的规矩是:Agent 修一次,验证不过就删分支叫人。把"重试"这个动作从 Agent 手里拿走,让它变成人的决策。AI 的优势是"快",不是"锲而不舍";凌晨让它无限重试,是在用它最弱的特性(判断力)去赌它最强的特性(速度),方向错了。

这套设计的自动化不是为了取代人,是为了让人在早上 9 点看到一份已经梳理好的报告,而不是从零开始排查。

为什么坚持"修复走完整回归"

这套系统里我最坚持的一条,是 Agent 修完必须跑完整 8 步回归,而不是只重跑失败那一步。这背后是一次吃亏的经历。

早先版本图省事,Agent 修完只重跑失败步骤——后端单测挂了,改完只重跑后端单测,过了就提 MR。结果有一次:Agent 修好了后端单测,但它的改动牵连了运营 E2E(改了一个共用的工具类方法签名),只重跑后端单测根本发现不了。MR 合进 dev,第二天运营 E2E 全红。

根因就是:Agent 的视野是局部的。它只看了失败步骤的日志,它的修复针对的是那一个点,但代码改动的影响面是全局的。强制跑完整 8 步,等于强行把 Agent 的视野从"一个点"拉到"整个系统",让它(以及它带进 MR 的改动)接受全套检验。这比让 Agent 自己声明"我检查过没有副作用"可靠得多——AI 不会主动检查副作用,但跑全套测试会替它检查。
