上一篇把总账立住了。这篇挑它最硬的一块——微信分润——拆给你看。

分润为什么难。一笔钱要拆成几份、分几次、走不同的通道、还得多租户隔离,搞不好就是资金事故。这是整个项目里我盯得最紧、门禁立得最硬的部分。

一、先看这笔钱怎么流

多租户电商里,一笔订单的钱不是直接进商家口袋。微信支付支持「分账」——付款时先冻结到特约商户,确认收货后再按比例拆给平台、达人、商家。

有人会问,为什么这么绕,用户付钱直接到账不好吗。因为这是多角色分润的电商,一笔订单涉及平台抽佣、达人带货佣金、商家货款、配送费,各方的钱到账时点不一样(平台要即时确认收入、达人要审核后才结算、商家要等完结分账解冻)。如果用户付款直接进商家口袋,平台和达人的钱就得商家手动转,这在多租户 SaaS 里根本跑不通。微信分账这套机制就是为了多角色资金拆分设计的,绕是绕,但它把「钱怎么拆」变成了平台可控的状态流转。

钱大致这样流动:用户支付时标记 profit_sharing=true,资金冻结到店铺特约商户池;确认收货后触发分账,平台佣金即时分润,达人佣金延迟分账(留在达人冻结池),商家货款暂时冻结;达人提现审核通过后,才真正把达人那一份分账到零钱;最后完结分账,解冻商家货款可提。

关键决策是达人佣金为什么延迟。确认收货那一刻,达人可能还没签约分账接收方、可能退款、可能违规。所以钱先标 UN_SPLIT 留在冻结池,等提现审核通过才真正分账。这就是后面提现状态机的来由。

补充一点,这套系统里「提现」其实有三种人,出口不一样。

达人(TALENT)走微信分账:提现审核通过后,系统把这个人关联的 cps_order_attribution 里所有 UN_SPLIT 的佣金逐单调微信分账 API,分到他的 PERSONAL_OPENID。这是最绕的一种,因为它不是「转账」而是「分账」,要回挂到原始微信支付单。

配送点团长(DELIVERY_POINT_LEADER)和商家(MERCHANT)走出款渠道:不挂原始支付单,直接调微信商家转账到零钱或银行卡代付。团长和平台的资金关系相对干净,商家则是分账完结解冻货款后才有钱可提。

三种人看起来都是「提现」,但余额怎么算、钱从哪出、走哪条通道,完全不同。后面会展开。

再补一个容易忽略的环节:确认收货触发分账,不是「一次性把钱拆完」。它是一个编排过程(processTalentSplitForWithdrawal):先把这个达人关联的所有 UN_SPLIT 佣金行捞出来,逐单标 SPLITTING、逐单调微信分账 API、成功的标 SPLIT、最后还要尝试完结该订单的商家货款解冻。

这个逐单处理是有讲究的。一个达人一次提现可能关联几十笔订单的佣金,如果打包成一个请求,中间任意一笔失败整批回滚,前面的也白调了。逐单调的好处是:失败的那笔回退到 UN_SPLIT(代码里 catch 块专门做了回滚),成功的笔保留 SPLIT,下次重试只跑失败那几笔,不重复打款。代价是慢,但资金系统宁可慢也不能错。

还有个前置动作:分账前要先「确保达人签约了分账接收方」(ensureTalentReceiverAdded)。微信分账要求收款方是已绑定的接收方,没签约就分账会被拒。所以每一单分账前先检查、没签约先发起签约,再调分账。这一步漏了,提现就会整批失败,日志里全是「接收方不存在」。

二、分账比例:基点(bp)而不是百分比

钱的事,小数点算错就是事故。我们全程用基点(万分之一),绝不用浮点百分比。

TenantSplitConfigService 里分账比例是这样定义的:

public record SplitConfig(
        String tenantId,
        @Min(0) @Max(5000) int platformCommissionRateBp,   // 平台抽成
        @Min(0) @Max(5000) int talentCommissionRateBp,      // 达人佣金
        @Min(0) @Max(5000) int otherSplitRateBp,            // 其他分账
        @Min(0) @Max(5000) int totalSplitLimitBp,           // 总分账上限
        boolean enabled, ...) {
    public int totalRateBp() {
        return platformCommissionRateBp + talentCommissionRateBp + otherSplitRateBp;
    }
}

几个硬约束。

比例上限 5000bp(50%),单方不能超过。

总比例守门 30%:totalRateBp() > 3000bp 时自动降级为即时分账,不走达人延迟分账,避免分账方拿太多、商家货款不够解冻。

配送费单独算:次日达有固定金额(nextDayDeliveryFeeFen),配送中心和自提点按 bp 分,合计不超 10000bp。

默认值我定的是平台抽 800bp(8%)、达人 500bp(5%)、其他 300bp(3%),合计 1600bp,上限 3000bp(30%)。新租户进来沿用这套默认,再按业务调。这个默认不是拍脑袋——达人佣金太低没人愿意带量,平台抽太高商家没动力入驻,1600bp 是我们运营反复试出来的平衡点。改这个数要连带看分账完结后商家实际到手多少,不能只盯平台收入。

为什么要单独搞一个 deliveryTotalRateBp() 守配送费 100%。因为配送费的分润规则跟订单佣金不是一套:配送中心和自提点的钱是「配送费里切一刀」,逻辑上必须分完(10000bp),否则就有一笔配送费没人拿,挂在系统里成悬账。订单佣金的守门是 30%,配送费的守门是 100%,两个守门不能混。

为什么用基点。整数运算,不存在 0.1+0.2≠0.3 的浮点坑。金额单位也是「分」(_fen 后缀),不是「元」。这两条是资金系统的铁律。

三、提现状态机:六态 + 重试三件套

达人提现是分润的出口,也是最容出事的地方(双花、重复打款、打款失败)。我们用了一个六态状态机:

public enum WithdrawalStatus {
    PENDING,     // 待审核
    APPROVED,    // 审核通过,待打款
    REJECTED,    // 审核拒绝
    PROCESSING,  // 打款中
    SUCCESS,     // 打款成功
    FAILED;      // 打款失败
}

流转逻辑:申请进入 PENDING,审核通过条件更新到 APPROVED,出款时乐观锁推进到 PROCESSING,成功到 SUCCESS、失败重试回退 APPROVED(attempt<3),重试耗尽到 FAILED,人工可把 FAILED 重置回 APPROVED。

三处防双花、防重打的设计,是真金白银的教训。

申请时余额校验用 FOR UPDATE 悲观锁。这里有个细节容易漏:三种提现人锁的行不一样。达人锁 cps_order_attribution(佣金归属表)里该达人的一行,团长锁 next_day_delivery_settlement(配送结算表)的一行,商家锁 merchant_frozen_ledger(商家冻结账本)的 APPROVED 行。锁的对象要跟余额来源表对上,锁错表等于没锁。

余额的计算口径也跟着不一样。达人余额 = 已结算佣金 - 已发起提现(含 SUCCESS),这里故意把 SUCCESS 也算进已提现,因为已打款成功的金额不能再被提一次。团长余额 = 已结算配送费 - 已提现,商家余额 = 冻结账本里 APPROVED 合计 - 已提现。三个口径写在三个方法里(getWithdrawableBalance / getLeaderWithdrawableBalance / getMerchantWithdrawableBalance),没有用一个通用方法套,因为它们查的根本是不同的表。

审核通过用条件更新:UPDATE ... WHERE id=? AND status='PENDING',affected==0 就抛并发冲突,防止两个人同时点通过。这里还有个 V2 硬化规则:审核备注必填,拒绝原因也必填。资金操作的每一次状态变更都要留痕,空备注的审核是事故温床。

出款时乐观锁:APPROVED → PROCESSING 同时 attempt_count+1。失败重试有次数上限(默认 3 次),耗尽才 FAILED。重试不是立刻跑,而是回退到 APPROVED 等定时任务 autoRetryApprovedWithdrawals 兜底,每条提现独立事务,一条失败不拖累其他。

还有一个反长事务的坑。V4 重构时,把出款的 HTTP 调用挪到了事务外面。原来打款请求包在事务里,微信接口一慢,数据库连接就被占住,直接连接池打满。改成 TransactionTemplate 精确控事务边界:DB 的条件更新在事务内,微信 HTTP 调用在事务外。这套模式后来成了项目里所有「调外部接口 + 改状态」场景的标准写法。

银行卡出款有个兜底场景要提。银行代付没对接真实 API、或者运营线下手动打款时,系统提供 confirmManualPayout:运营填备注(比如银行流水号),把 APPROVED/PROCESSING/FAILED 直接标成 SUCCESS。这条路径存在是因为真实世界里不是所有钱都能走线上,留人工兜底比强求全自动更稳。

定时任务也是这套状态机的兜底层。autoRetryApprovedWithdrawals 定时扫所有 APPROVED 状态的提现,逐条重新跑出款——每条用 TransactionTemplate 包独立事务,一条失败不影响下一条。为什么要这个任务。因为出款失败的回退是「退回 APPROVED 等重试」,如果没人捞,这些提现会永远卡在 APPROVED。定时任务就是那个「没人管的时候我来兜」的角色。

手续费顺带说一句。UniversalWithdrawalService 里算手续费用 BigDecimal 配 withdrawFeeRate,默认 0(不收),要收也是百分比折半进位(HALF_UP)。实发金额 netAmountFen = amountFen - feeFen,出款走实发金额,不走申请金额。资金系统里「申请多少」和「实发多少」必须分两个字段,不能图省事。

四、出款渠道路由:不是 if-else,是责任链

提现打款有两条路:微信转账到零钱、银行卡代付。怎么选。我们抽象了一个 PayoutChannelRouter。

路由优先级是:租户偏好 → 服务商模式走微信 → 直连微信 → 银行卡 → dev/test 兜底用 MOCK。生产环境没有可用渠道,直接抛异常,绝不静默吞掉。

这里有个分叉点要先讲清。processPayout 里判断 withdrawerType,达人不走 PayoutChannelRouter,直接调 TalentCommissionWithdrawalService 走微信分账(因为要回挂原始支付单);团长和商家才走 PayoutChannelRouter 选渠道。把分账和转账分两条代码路径,是因为它们的资金来源和 API 完全不同,硬塞进一个路由会到处 if-else,不如在入口分流。

设计上有几个细节值得说。

渠道抽象成接口(AbstractPayoutChannel),每个渠道实现 channelType / isAvailable / pay / query。将来要接银联代发,不动路由逻辑。

大额要加密实名。金额 ≥ 2000 元(200_000 分),微信要求传 RSA-OAEP 加密的 user_name,凭证从 tenant_wechat_profile 取(本身 AES-GCM 加密存储)。这里有个金额下限的坑:微信商家转账到零钱最低 0.3 元(30 分),低于这个金额直接被微信拒,代码里 request.amountFen() < 30 先抛异常,免得打到微信那头再报错回来,日志还不好定位。

服务商模式和直连模式走不同路径。服务商模式请求要带 sub_mch_id(指定出资的二级商户),走 /partner/transfer-bills 端点;直连模式不带 sub_mch_id,走普通 transfer-bills。一个租户到底走哪种,看 tenant_wechat_profile 里的 pay_mode 字段。平台自己出款(达人提现)特殊:用服务商凭证但走直连端点、不带 sub_mch_id,因为资金从服务商账户出,不挂子商户。

每次出款落审计。payout_channel_log 记录完整请求、响应、状态,还要标 WECHAT_DIRECT 还是 WECHAT_SERVICE_PROVIDER,方便对账时按模式分流。资金系统,审计是底线。

银行卡出款(BankPayoutChannel)走另一条路。绑定账户里的银行卡号是 AES-GCM 加密存的,出款时要先解密出明文卡号,连同开户人、开户行一起传给银行代付接口。这块代码结构和微信渠道一样实现 AbstractPayoutChannel,但解密逻辑多一层——卡号泄漏是合规事故,所以解密只在出款那一刻发生,全程不落明文日志。

MOCK 渠道(MockPayoutChannel)只在 dev/test 环境通过 Spring profile 注册,生产环境压根没有这个 Bean。PayoutChannelRouter 在生产环境找不到可用渠道时直接抛异常,绝不用 MOCK 兜底——资金系统宁可报错也不能假装成功。

补充一句,我们用的是微信「商家转账到零钱」(/v3/fund-app/mch-transfer/transfer-bills)这套新接口,不是老的企业付款接口。服务商模式和直连模式走不同路径。

五、一次诚实的降级:冲正引擎,从启用到禁用

这块单独讲,因为它最能说明真实工程是怎么回事——它跟着业务走,不跟着理想走。

我们最早设计了一套完整的冲正引擎(ClawbackEngine):达人佣金结算后,如果用户退款,按退款比例把达人已拿的佣金冲回来,记入商户成本。设计得很完整。

幂等键 (tenantId, refundNo, originRecordId, originRecordType) 唯一。冲回比例 = 退款金额 × 10000 / 订单实付,BigDecimal HALF_UP。累计上限校验:本次冲回 + 已冲回 ≤ 原始佣金。状态机 PENDING → SUCCESS/FAILED,失败重试上限 3 次。MQ 异步(commission.clawback.queue)加同步双通道。

几个防御性写法是资金系统踩出来的。退款金额先 clamp 到订单实付以内:refundAmountFen > orderPaidAmountFen 时强制拉回等于实付金额,因为真实数据里确实出现过退款金额大于订单实付的脏数据(部分退款叠加优惠券回退算错),不挡掉会算出超过 10000bp 的冲回率。订单实付 ≤ 0 直接 skip,避免除零。

冲回金额用 BigDecimal 算两步:先算冲回率 refundAmount × 10000 / orderPaidAmount(整数 bp),再算单条冲回 originAmount × rateBp / 10000。两步都用 HALF_UP,不是图省事用整数除法截断——资金系统截断就是少算钱,少算的金额会慢慢累积成对账差异。

商户成本记录是容错的。冲回流水写成功后,调 merchantCostService.recordCost 把冲回金额记进商户成本。这个调用失败了怎么办。代码里是 try-catch 吞掉异常,只打 error 日志,不影响冲回流水本身。因为冲回流水是资金凭证,必须先保住;商户成本是对账用的汇总,挂了由对账流程发现修复。两条数据的可靠性优先级不同,不能让汇总失败回滚掉凭证。

重试也有讲究。retryFailed 只捞 FAILED 且 retryCount < 3 的记录,失败的逐条重跑。指数退避(10s / 30s / 90s)预留给 MQ 层调度,代码里直接重试——因为 MQ 消费本身已经有重试间隔,代码层再叠退避会拉长到不可接受。资金重试要的是「尽快补上,但别无限重」,3 次封顶,失败转人工。

但上线后,业务规则改了。判断是:用户退款和达人引流是两件事,达人佣金一旦结算就不该因退款撤销,否则达人体验崩坏,且对账复杂度爆炸。

于是退款回调入口被降级为 no-op,代码里留了一句明确的注释:

// 业务规则(2026-06-11 起):用户退款与达人佣金无关,
// 达人佣金一旦结算绝不因退款撤销。
// 方法保留入口签名以维持调用链稳定,方法体降级为 no-op。

我没有删掉 ClawbackEngine。它是完整实现、经过测试的,只是当前不被调用。留着的原因是业务规则可能再变,留着意味着架构能力还在,只是开关关了,比每次重写强。

补充一点,真正还在做「分润回退」的是平台佣金侧:退款时调微信 profitsharing/return-orders 把平台已分走的钱退回去。这跟达人佣金冲正是两件事,别混。

五半、超期佣金:达人不要的钱怎么处理

冲正之外还有个边界问题:达人佣金结算后一直不提现,挂在那儿怎么办。

微信分账有账期限制,达人佣金在 cps_order_attribution 里标 UN_SPLIT 留太久,会占用店铺特约商户的冻结额度,影响后续分账。所以系统有 handleExpiredCommissions:查所有 UN_SPLIT 且超过宽限期(talent_grace_days,默认配在 cps_tenant_config)的佣金,按租户配置的 talent_expired_action 处理。

两种动作。AUTO_SPLIT(默认)自动发起分账给达人,标 EXPIRED_AUTO;MERGE_MERCHANT 把这笔佣金并回商户成本,标 EXPIRED_MERGED,等于达人不要了归商家。租户自己选,平台不替租户拍板。

这个机制看似不起眼,但它解决的是「悬账」——资金系统里最危险的状态就是钱不知道该归谁。每一笔佣金要么进了达人零钱,要么明确并入商户成本,不允许长期挂 UN_SPLIT 充当缓冲。悬账留着,早晚变成对账时的无头案。

六、资金系统的三条铁律

这套分润做下来,我给自己定了三条规矩,也写进了仓库的 CLAUDE.md,让 AI 和新人都遵守。

金额用分,比例用基点,全程整数运算。浮点数在资金系统里没有容身之处。这条不光约束金额和比例,连手续费这种「衍生金额」也用 BigDecimal 配 HALF_UP 算,绝不让 double 参与任何资金计算。新人最常犯的错就是用 double 算手续费再转回 long,看似差几分钱,跑一个月对账差异能到几千块。

每一个状态变更都要落审计。split_record、payout_channel_log、commission_clawback_record——资金系统的「过去」必须可追溯。审计不光记「改成什么」,还要记「谁改的、什么时候、带什么备注」,审核操作尤其如此。audit_remark 必填不是洁癖,是出事复盘时唯一能还原决策的线索。

并发靠数据库约束,不靠代码记忆。条件更新、唯一键、悲观锁,能用数据库挡的就别在应用层挡。代码里的 if (status == PENDING) 检查防不住两个请求同时进来,只有 WHERE status='PENDING' 的条件更新让数据库当裁判,才能保证只有一个请求改成功。

六半、对账:分润的最后一道闸

前面讲的都是「钱怎么动」,但资金系统能不能信,最后看的是对账。

这套分润的对账逻辑挂在几个账本上。merchant_frozen_ledger(商家冻结账本)记录每一笔货款的冻结、解冻、提现流水;platform_income_ledger(平台收入账本)记录平台佣金收入;pickup_point_ledger 记配送点分润。每个账本都能按租户、按时间窗汇总出「应该有多少钱」。

对账的口径是双向核对。一是内部对账:账本汇总金额要等于各状态流水的累加,比如商家可提现余额必须等于「冻结账本 APPROVED 合计 - 已提现合计」,对不上就是有流水漏记或状态机跳步。二是外部对账:平台佣金侧的累计金额,要跟微信商户后台的分账记录对得上,差一笔都要查。

这套对账不是写完就放着,是定时跑的。任何差异告警进群,资金差异的告警优先级最高,半夜也得有人看。对账机制的存在,反过来逼着前面的每一步都老老实实落审计——因为对账会揪出任何「状态变了但没记流水」的偷懒。账本和流水对不上,这套分润就不能算上线。

七、为什么这块能这么快做出来

回头说点题外话。这块分润代码量大(后端 140+ 文件、90+ 迁移表)、业务复杂,但我们没花几个月。

原因不是 AI 帮我写完了分润逻辑。资金的核心设计——状态机、基点、并发控制——是我自己定的,AI 写不了。AI 帮的是周边:Controller/Service 样板、Mapper、迁移 DDL、测试用例、文档。我把判断和架构做对,AI 把执行铺开。

具体说说我做什么、AI 做什么。我来定:六态状态机的状态枚举和流转规则、三种防双花的锁策略(哪个表加 FOR UPDATE、哪个用条件更新、哪个用乐观锁)、分账比例的基点约束边界、出款渠道的抽象接口签名、冲正的幂等键设计。这些一旦定错就是资金事故,不能交给 AI 自由发挥。

AI 来做:把 WithdrawalStatus 枚举铺成完整的 Service 方法体、按我给的表结构生成 Flyway DDL、按状态机写单元测试用例(每个状态转换一个 case)、写 Controller 的参数校验和异常处理、把 JdbcTemplate 的 SQL 按规范补齐 COALESCE 和 LIMIT。这些是确定性的活,AI 干得比我快还少错。

资金这种高风险模块,越是老兵 + AI 越安全,因为老兵知道红线在哪,AI 负责把红线之内的活快速干完。怕的是新手 + AI:新手不知道哪条线不能碰,AI 又特别擅长把「看起来合理」的代码快速生成出来,两下一凑,浮点金额、漏锁、状态机跳步这些问题就埋进去了。

代码评审在这里就是最后的安全网。我审 AI 生成的资金代码,重点盯三样:有没有用浮点、状态变更有没有落审计、并发路径有没有数据库约束兜底。这三样过不了,代码写得再优雅也打回。

分润这块先讲到这。下一篇拆数据库迁移治理——500+ 功能模块、近 400 个迁移脚本,我们是怎么在不翻车的前提下管住的。
