上一篇把总账立住了。这篇挑它最硬的一块——微信分润——拆给你看。

分润为什么难。一笔钱要拆成几份、分几次、走不同的通道、还得多租户隔离,搞不好就是资金事故。这是整个项目里我盯得最紧、门禁立得最硬的部分。

一、先看这笔钱怎么流

多租户电商里,一笔订单的钱不是直接进商家口袋。微信支付支持「分账」——付款时先冻结到特约商户,确认收货后再按比例拆给平台、达人、商家。

钱大致这样流动:用户支付时标记 profit_sharing=true,资金冻结到店铺特约商户池;确认收货后触发分账,平台佣金即时分润,达人佣金延迟分账(留在达人冻结池),商家货款暂时冻结;达人提现审核通过后,才真正把达人那一份分账到零钱;最后完结分账,解冻商家货款可提。

关键决策是达人佣金为什么延迟。确认收货那一刻,达人可能还没签约分账接收方、可能退款、可能违规。所以钱先标 UN_SPLIT 留在冻结池,等提现审核通过才真正分账。这就是后面提现状态机的来由。

二、分账比例:基点(bp)而不是百分比

钱的事,小数点算错就是事故。我们全程用基点(万分之一),绝不用浮点百分比。

TenantSplitConfigService 里分账比例是这样定义的:

public record SplitConfig(
        String tenantId,
        @Min(0) @Max(5000) int platformCommissionRateBp,   // 平台抽成
        @Min(0) @Max(5000) int talentCommissionRateBp,      // 达人佣金
        @Min(0) @Max(5000) int otherSplitRateBp,            // 其他分账
        @Min(0) @Max(5000) int totalSplitLimitBp,           // 总分账上限
        boolean enabled, ...) {
    public int totalRateBp() {
        return platformCommissionRateBp + talentCommissionRateBp + otherSplitRateBp;
    }
}

几个硬约束。

比例上限 5000bp(50%),单方不能超过。

总比例守门 30%:totalRateBp() > 3000bp 时自动降级为即时分账,不走达人延迟分账,避免分账方拿太多、商家货款不够解冻。

配送费单独算:次日达有固定金额,配送中心和自提点按 bp 分,合计不超 10000bp。

为什么用基点。整数运算,不存在 0.1+0.2≠0.3 的浮点坑。金额单位也是「分」(_fen 后缀),不是「元」。这两条是资金系统的铁律。

三、提现状态机:六态 + 重试三件套

达人提现是分润的出口,也是最容出事的地方(双花、重复打款、打款失败)。我们用了一个六态状态机:

public enum WithdrawalStatus {
    PENDING,     // 待审核
    APPROVED,    // 审核通过,待打款
    REJECTED,    // 审核拒绝
    PROCESSING,  // 打款中
    SUCCESS,     // 打款成功
    FAILED;      // 打款失败
}

流转逻辑:申请进入 PENDING,审核通过条件更新到 APPROVED,出款时乐观锁推进到 PROCESSING,成功到 SUCCESS、失败重试回退 APPROVED(attempt<3),重试耗尽到 FAILED,人工可把 FAILED 重置回 APPROVED。

三处防双花、防重打的设计,是真金白银的教训。

申请时余额校验用 FOR UPDATE 悲观锁。SUCCESS 状态也算「已提现」,防止有人提现成功后又发起一笔。

审核通过用条件更新:UPDATE ... WHERE id=? AND status='PENDING',affected==0 就抛并发冲突,防止两个人同时点通过。

出款时乐观锁:APPROVED → PROCESSING 同时 attempt_count+1。失败重试有次数上限(默认 3 次),耗尽才 FAILED。

还有一个反长事务的坑。V4 重构时,把出款的 HTTP 调用挪到了事务外面。原来打款请求包在事务里,微信接口一慢,数据库连接就被占住,直接连接池打满。

四、出款渠道路由:不是 if-else,是责任链

提现打款有两条路:微信转账到零钱、银行卡代付。怎么选。我们抽象了一个 PayoutChannelRouter。

路由优先级是:租户偏好 → 服务商模式走微信 → 直连微信 → 银行卡 → dev/test 兜底用 MOCK。生产环境没有可用渠道,直接抛异常,绝不静默吞掉。

设计上有几个细节值得说。

渠道抽象成接口(AbstractPayoutChannel),每个渠道实现 channelType / isAvailable / pay / query。将来要接银联代发,不动路由逻辑。

大额要加密实名。金额 ≥ 2000 元(200_000 分),微信要求传 RSA-OAEP 加密的 user_name,凭证从 tenant_wechat_profile 取(本身 AES-GCM 加密存储)。

每次出款落审计。payout_channel_log 记录完整请求、响应、状态。资金系统,审计是底线。

补充一句,我们用的是微信「商家转账到零钱」(/v3/fund-app/mch-transfer/transfer-bills)这套新接口,不是老的企业付款接口。服务商模式和直连模式走不同路径。

五、一次诚实的降级:冲正引擎,从启用到禁用

这块单独讲,因为它最能说明真实工程是怎么回事——它跟着业务走,不跟着理想走。

我们最早设计了一套完整的冲正引擎(ClawbackEngine):达人佣金结算后,如果用户退款,按退款比例把达人已拿的佣金冲回来,记入商户成本。设计得很完整。

幂等键 (tenantId, refundNo, originRecordId, originRecordType) 唯一。冲回比例 = 退款金额 × 10000 / 订单实付,BigDecimal HALF_UP。累计上限校验:本次冲回 + 已冲回 ≤ 原始佣金。状态机 PENDING → SUCCESS/FAILED,失败重试上限 3 次。MQ 异步(commission.clawback.queue)加同步双通道。

但上线后,业务规则改了。判断是:用户退款和达人引流是两件事,达人佣金一旦结算就不该因退款撤销,否则达人体验崩坏,且对账复杂度爆炸。

于是退款回调入口被降级为 no-op,代码里留了一句明确的注释:

// 业务规则(2026-06-11 起):用户退款与达人佣金无关,
// 达人佣金一旦结算绝不因退款撤销。
// 方法保留入口签名以维持调用链稳定,方法体降级为 no-op。

我没有删掉 ClawbackEngine。它是完整实现、经过测试的,只是当前不被调用。留着的原因是业务规则可能再变,留着意味着架构能力还在,只是开关关了,比每次重写强。

补充一点,真正还在做「分润回退」的是平台佣金侧:退款时调微信 profitsharing/return-orders 把平台已分走的钱退回去。这跟达人佣金冲正是两件事,别混。

六、资金系统的三条铁律

这套分润做下来,我给自己定了三条规矩,也写进了仓库的 CLAUDE.md,让 AI 和新人都遵守。

金额用分,比例用基点,全程整数运算。浮点数在资金系统里没有容身之处。

每一个状态变更都要落审计。split_record、payout_channel_log、commission_clawback_record——资金系统的「过去」必须可追溯。

并发靠数据库约束,不靠代码记忆。条件更新、唯一键、悲观锁,能用数据库挡的就别在应用层挡。

七、为什么这块能这么快做出来

回头说点题外话。这块分润代码量大(后端 140+ 文件、90+ 迁移表)、业务复杂,但我们没花几个月。

原因不是 AI 帮我写完了分润逻辑。资金的核心设计——状态机、基点、并发控制——是我自己定的,AI 写不了。AI 帮的是周边:Controller/Service 样板、Mapper、迁移 DDL、测试用例、文档。我把判断和架构做对,AI 把执行铺开。

资金这种高风险模块,越是老兵 + AI 越安全,因为老兵知道红线在哪,AI 负责把红线之内的活快速干完。

分润这块先讲到这。下一篇拆数据库迁移治理——850 张表、近 400 个迁移脚本,我们是怎么在不翻车的前提下管住的。
