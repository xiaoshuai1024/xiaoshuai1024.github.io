用户反馈"下单失败",登上去一查:前端 console 干净,network 显示 500,后端日志同时间段上千条 ERROR。哪一条是这一单的?不知道。堆再多工具也救不了这种痛——可观测性的关键不是工具多,是它们能不能串成一条能排查的链路。

线上出问题,最怕的不是错,是"不知道哪儿错"

做过稳定性的人大概都有过这种时刻:用户反馈"系统卡了"“报错了”,你登上去一看,前端没报错日志,后端日志一大堆,翻半天翻不到那一条。

最后只能去问"什么时候、什么操作、哪个页面",靠用户回忆拼现场。

这种痛感,堆再多工具也救不了。我对这件事的判断很明确:可观测性不是工具数量的问题,是这些东西能不能串成一条链路。

工具再多,各管各的、对不上号,排查时还是两眼一抹黑。你上三个 APM、两个日志平台,它们之间没有共同的 requestId 把同一次请求串起来,排查时还是得靠"猜"——猜这次请求在 A 工具里是哪条、在 B 平台里又对应哪条。工具数量解决不了对齐问题,串成链路才能。这篇文章,复盘我们团队从"堆工具"到"串链路"走了哪几步。

说个真实场景,你就能体会"串不起来"有多痛。某次用户反馈"下单失败",我们登上去一查:前端 console 干净、network 显示 500、后端日志里同时间段有上千条 ERROR。到底是哪一条对应这个用户?不知道。只能让用户复述操作时间,翻着时间戳一条条对,半小时才定位。后来铺了 requestId,同样的场景:用户报障时把错误提示里的 requestId 发过来,后端日志一搜,这一单从进入拦截器、走到下单 Service、在哪一步抛了 CONCURRENT_CONFLICT,完整轨迹一条线拉出来,3 分钟搞定。这就是"串成链路"和"堆工具"的效率差。

第一根线:全链路 RequestId

最便宜、收益最高的一件事,就是每个请求注入一个唯一的 RequestId,然后让它贯穿这个请求经过的所有日志。

听起来简单,但它解决的是排查里最要命的问题:对齐。

前端报了一个错,后端到底走没走到、走到哪一步挂了——以前靠猜,现在按 RequestId 一搜,前因后果全在一条时间线上。

我们团队的规矩是:排查线上问题第一件事,就是拿 RequestId 去后端日志里捞这条请求的完整轨迹。没有这个 id,前后端日志就是两座孤岛——前端日志记的是"页面 A 报了错",后端日志记的是"接口 B 抛了异常",两边都对,但你没法证明"页面 A 的错就是接口 B 的异常引起的"。中间可能隔着网关、缓存、异步任务,任何一个环节都可能是断点。RequestId 把这条链路缝起来,从用户点的那一刻到后端落库,一条 id 串到底。

RequestId 的生成也有讲究:前端生成优先,后端兜底。正常情况下前端 axios 拦截器给每个请求生成一个 id 放进 header,后端 Filter 取出来用——这样同一个 id 贯穿前后端。但如果请求没经过前端(比如第三方直接调后端、或 MQ 消费),header 里没 id,后端 Filter 兜底生成一个。两段逻辑都在那个 resolveRequestId 方法里,优先用外面传进来的,没有才自己造。这个顺序保证:同一次请求,前后端用同一个 id,绝不串号。

RequestId 怎么注入:Filter 比 Interceptor 更靠前

具体实现上,RequestId 要在前端、网关、后端三段都贯通,缺一段就断链。后端这块我们用的是 Servlet Filter,不是 HandlerInterceptor——这点是有意选的:

// 后端 Filter:继承 OncePerRequestFilter,从请求头取 id,没有就生成,塞进 MDC
public class RequestTraceFilter extends OncePerRequestFilter {
    public static final String REQUEST_ID_HEADER = "X-Request-Id";
    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse resp, FilterChain chain) {
        String rid = resolveRequestId(req.getHeader(REQUEST_ID_HEADER));
        request.setAttribute("requestId", rid);
        response.setHeader(REQUEST_ID_HEADER, rid);   // 回写给前端
        RequestTraceContext.setRequestId(rid);
        MDC.put("requestId", rid);
        try {
            chain.doFilter(request, response);
        } finally {
            log.info("request end method={} path={} status={} durationMs={}",
                    request.getMethod(), request.getRequestURI(), response.getStatus(),
                    System.currentTimeMillis() - start);   // 顺手把请求耗时也打进日志
            MDC.remove("requestId");          // 关键:线程池复用,MDC 不清会串号
            RequestTraceContext.clear();        // ThreadLocal 也要清
        }
    }
    private String resolveRequestId(String header) {
        if (header == null || header.isBlank()) {
            return UUID.randomUUID().toString().replace("-", "");   // 去掉横线
        }
        return header.trim();   // 外部传进来的 id 顺手去空格
    }
}

ts=%d{yyyy-MM-dd HH:mm:ss.SSS} level=%level requestId=%X{requestId:-} tenant=%X{tenantId:-} thread=%thread msg=%msg%n
// 前端 axios 拦截器:每个请求带上 id,出错时把 id 显示给用户
axios.interceptors.request.use(cfg => {
  cfg.headers['X-Request-Id'] = crypto.randomUUID().replace(/-/g, '')
  return cfg
})
message.error(`系统异常,请联系客服,编号:${rid}`)

为什么用 Filter 而不是 Interceptor?Filter 比拦截器更靠前。拦截器只在请求能找到对应的 Controller 时才介入,而 Filter 早在 Servlet 链的最外层。这意味着 404、鉴权失败返回 401 这类根本走不到 Controller 的请求,也能拿到 requestId。排查线上问题最怕的就是这种"边界请求"——拦截器拦不到,日志里一片空白。Filter 兜在最外面,所有请求无死角。我们注册时还显式 setOrder(1),排在租户 Filter、鉴权 Filter 之前,确保后续每一个 Filter/拦截器都能读到 requestId。

这里有个最容易漏的细节:MDC.remove 必须在 finally 里调。Tomcat 线程是复用的,上一次请求的 MDC 不清,下一次请求会带上前一次的 requestId——日志串号,排查时会把你引向完全错误的轨迹。线程池复用 + ThreadLocal = 必须清理,这是 Java 后端的基础纪律。顺带把 RequestTraceContext(ThreadLocal)一起清,不光 MDC 要清。

跨进程透传:走 MQ 消息头

跨服务调用怎么把 requestId 带过去?我们项目里跨进程走的是 RabbitMQ,不是 HTTP 的 Feign,所以透传做在 MQ 消息头上,用了个 MqTraceHelper:

// 生产者:发消息前把当前线程的 requestId 注入消息头
message.getMessageProperties().setHeader("x-request-id", RequestTraceContext.getRequestId());

// 消费者:消费时从消息头恢复到当前线程
Object traceId = message.getMessageProperties().getHeader("x-request-id");
if (traceId instanceof String rid && !rid.isBlank()) {
    MDC.put("requestId", rid);
    RequestTraceContext.setRequestId(rid);
}

这里有个真实的设计细节:MQ 消息没带 requestId 时,消费者会兜底生成一个 mq- 前缀的 id。这个前缀不是随便加的——从日志一眼就能区分"这条来自 MQ 消费"还是"来自 HTTP 请求"。排查跨进程问题时,这个前缀能帮你快速定位链路是在哪一段断的。

异步任务(@Async / 线程池)更麻烦,MDC 不会自动传到子线程,得用 TaskDecorator 包一层。我们写了 MdcTaskDecorator,不止复制 requestId,还同时传播 tenantId(多租户场景这个不能丢);而且考虑了线程池嵌套复用——子线程先备份自己原有的上下文,执行完恢复,避免池里上一个任务遗留的 requestId 串过来。所有 @Async 线程池统一 setTaskDecorator(new MdcTaskDecorator()),异步日志照样能按 requestId 捞。这一步漏了,异步链路的日志就查不到了——常见的"定时任务出问题但日志对不上"就是这个原因。

第二根线:统一错误信封

RequestId 解决了"找日志",但还有个更烦的问题:错误格式不统一。

以前我们的接口,每个同学自己拼错误返回结构:有的返回 code、有的返回 status、有的返回 errno。前端解析的时候痛苦得要命,日志里的异常堆栈更是五花八门。

后来我们统一成 AppException + ApiErrorCode:

  • 业务里抛统一的异常类,错误码集中管理;
  • 前端拿到的错误结构是固定的,告别"每个接口自己拼错误结构"。

这件事看着小,价值在于前后端对错误的理解一致了。前端不用猜返回结构长啥样,后端也不用记每个接口的错误码塞在哪儿。前端写错误处理时,只认那套 ApiErrorCode 枚举——ORDER_NOT_FOUND 就引导回订单列表,OPS_PERM_DENIED 就跳登录,CONCURRENT_CONFLICT 就提示重试。新增一个错误码,前端这份表里加一行,所有接口的同类错误统一处理。错误码集中管理还有个隐性收益:它是活文档。哪个模块有哪些错误、错误对应什么状态码,翻一遍 ApiErrorCode 枚举全知道,不用翻散落在各处的 throw new XxxException。

错误信封统一了,RequestId 带上错误码,一条日志就把"谁错了、错在哪、什么码"讲清楚。

统一错误信封的格式

我们的错误返回结构,强制收敛成 code / message / requestId 三个字段,用 LinkedHashMap 保证字段顺序:

// 后端:所有业务异常都走 AppException,错误码集中在枚举里
public final class AppException extends RuntimeException {
    private final ApiErrorCode errorCode;     // 枚举名就是错误码,如 ORDER_NOT_FOUND
    private final String messageOverride;     // 可选,覆盖默认文案
}

// 错误码集中管理:域前缀 + 语义,共 90+ 个码
public enum ApiErrorCode {
    ORDER_NOT_FOUND("订单不存在", HttpStatus.NOT_FOUND),
    ORDER_STATUS_INVALID("订单状态不允许当前操作", HttpStatus.BAD_REQUEST),
    POINTS_INSUFFICIENT_BALANCE("积分余额不足"),
    OPS_AUTH_SESSION_INVALID("未登录或会话已失效", HttpStatus.UNAUTHORIZED),
    OPS_PERM_DENIED("权限不足", HttpStatus.FORBIDDEN),
    CONCURRENT_CONFLICT("并发操作冲突,请稍后重试", HttpStatus.CONFLICT),
    SEARCH_UNAVAILABLE("搜索暂不可用,请稍后再试", HttpStatus.SERVICE_UNAVAILABLE),
    // ...
}

// 统一错误返回结构(三字段,没有 timestamp)
public static Map<String, Object> of(ApiErrorCode errorCode, String message, String requestId) {
    LinkedHashMap<String, Object> m = new LinkedHashMap<>(4);
    m.put("code", errorCode.getCode());
    m.put("message", message != null ? message : errorCode.getDefaultMessage());
    m.put("requestId", requestId == null ? "" : requestId);
    return m;
}
// 前端:拿到的错误结构固定,不用每个接口猜
interface ErrorEnvelope {
  code: string        // 'ORDER_NOT_FOUND' / 'OPS_PERM_DENIED'
  message: string     // 给用户看的中文提示
  requestId: string   // 排查用
}
// 统一拦截:根据 code 决定提示,根据 requestId 上报

这套结构的关键是 code 字段:前端拿到 code 就知道是哪类错,可以精准提示(订单不存在 → 引导回列表;权限不足 → 引导登录)。绝对不允许后端直接把 e.getMessage() 扔回去——那是给开发看的堆栈信息,不是给用户看的。

错误码命名规范也要统一:用 域前缀 + 语义(ORDER_NOT_FOUND、OPS_PERM_DENIED),不要 ORDER_001 这种纯数字——数字记不住,域前缀 + 语义一眼能认出是哪个业务模块的错。我们这套错误码现在有 90 多个,前端可以按 code 写一份完整的错误处理表,哪个 code 对应什么提示、什么引导动作,清清楚楚。

还有一个容易被忽略的环节:全局异常处理器覆盖面要全。我们那个 GlobalExceptionHandler 写了 14 个 @ExceptionHandler,从 AppException 到 MethodArgumentNotValidException、DuplicateKeyException(唯一键冲突要返回 409 而不是 500)、甚至 BadSqlGrammarException,所有分支都统一走 of(...) 出 code/message/requestId。更狠的是,连 Filter 层失败的请求(401/403/404 根本走不到 Controller)也单独写了个 CustomErrorController,把 Spring Boot 转发到 /error 的请求再转成统一信封——保证整个错误体系无一漏网。

错误信封统一了,RequestId 带上错误码,一条日志就把"谁错了、错在哪、什么码"讲清楚。还有个细节:成功响应也注入 requestId。我们写了个 ResponseBodyAdvice,在所有 Map<String,Object> 成功响应体前自动加 requestId 字段。这样前端不管成功还是失败,统一取 body.requestId 就行,不用区分"成功响应没 requestId,错误响应才有"。让 requestId 成为所有响应的标准字段,前端代码因此能统一处理。

盯什么:Prometheus + Grafana

日志解决的是"事后查",监控解决的是"事中盯"。我们用 Prometheus 采集、Grafana 看板,重点盯几类指标:

  • API 延迟:别光看平均值,得看 P95/P99,平均值的谎言我吃过亏;
  • QPS:知道系统在扛多少流量,心里有底;
  • 成功率:这个比延迟更重要,成功率掉下来就是真出事了;
  • Token 消耗:接了大模型的能力,Token 是真金白银,得盯;
  • JVM 指标:GC、内存、线程,Java 服务的命门。

平均值这个坑值得多说一句。我们吃过亏:某段时间接口平均响应 50ms,看着挺好,可用户反馈明显变慢。深挖才发现,P99 已经飙到 2 秒——只是被大量快速的请求把平均数拉低了。平均值会掩盖长尾,1 个 2 秒的慢请求混在 99 个 50ms 的快请求里,平均下来才 70ms,看着一切正常。所以盯延迟必须盯分位数(P95、P99),盯的是"最慢的那批请求有多慢",而不是"平均有多快"。Prometheus 的 histogram_quantile(0.95, ...) 就是干这个的。

成功率比延迟更优先,这是我踩坑后的排序。延迟高,用户体感差但功能还能用;成功率掉下来,说明请求直接失败了,这是硬伤。所以告警先盯成功率,延迟其次。盯成功率还有个好处:它是端到端的——从用户点击到拿到响应,任何一环断了都会拉低成功率,不像延迟那样容易被某一层缓存美化。延迟指标可以靠缓存把读路径做得很漂亮,把脏活全藏到写路径里;成功率藏不住,它如实反映用户到底成没成。

我的经验是,看板别贪多。堆一堆大屏,平时没人看,出事了也找不到那条线。盯住核心的几个,把阈值和告警配好,比 50 个面板强。

我们的真实看板就克制地配了 6 个面板:JVM 内存(jvm_memory_used_bytes)、GC 次数(rate(jvm_gc_pause_seconds_count[1m]))、HTTP 请求速率(rate(http_server_requests_seconds_count[1m}),即 QPS)、HTTP 请求 95% 延迟(histogram_quantile(0.95, rate(http_server_requests_seconds_bucket[1m]))——这就是"别看平均值,看 P95"的落地)、数据库连接池(hikaricp_connections_active / _idle / _pending)、系统资源(system_cpu_usage + heap 占比)。6 个面板把 JVM、HTTP、连接池、CPU 全覆盖,够了。

除了这些基础指标,我们还自埋了一批业务指标。这块很多人忽视——监控不光盯系统,得盯业务。比如支付成功率掉了,系统指标全绿,但业务已经出事了。我们用 Micrometer 埋了:支付/退款/提现的成功失败计数(payment.success.total、payment.fail.total)、结算在途笔数(settlement.pending.count,Gauge)、结算在途时长分布(settlement.duration.seconds,直接出 P95/P99)、金额累计(payment.amount.fen,单位用分避免浮点)。连定时任务的成功失败/耗时都通过 AOP 切 @Scheduled 进了监控(job.duration、job.success、job.failure)——定时任务不是黑盒,它跑挂了你得知道。

ELK:不只是查日志

ELK(Elasticsearch + Logstash + Kibana)在我们这,除了做日志检索,还承担一件事:MySQL 到 ES 的同步,用来做业务搜索和审计。

为什么要做这层同步?MySQL 擅长事务和强一致,但不擅长全文检索和模糊搜索;ES 反过来。商品搜索、订单多条件筛选这类场景,直接查 MySQL 会把数据库拖垮,同步到 ES 用它的倒排索引,既快又不影响主库。所以我们的做法是:MySQL 作为业务真相源,变更后经 Logstash pipeline 增量同步到 ES,搜索请求走 ES,写操作还是回 MySQL。搜索和分析能力外包给 ES,业务一致性留在 MySQL。

这里有个真实的坑要提:dev 环境 ES 是 7.17.24,prod 是 8.7.0。精确到小版本,两份文档都写得明明白白(deploy/docker-compose.yml 用的是 elasticsearch:7.17.24,生产是宝塔的 dk_elk 栈 8.7.0)。

两个大版本之间 API 有变化,本地调试能跑的查询上了生产可能就报错。而且坑是双重的:dev 的 ES 无密码,prod 的 ES 8 开了 xpack 安全(要 elastic 账号)。吃过几次亏之后,我们定了一条规矩:跟 ES 打交道的代码,dev 和 prod 都得过一遍。Logstash pipeline 也分两套(business-sync.noauth.conf 给 dev,business-sync.conf 带账号给 prod),同步脚本 sync-elk-pipeline-reindex.sh dev|prod 自动切。版本差异 + 认证差异叠在一起,只在一边测等于没测。

最后一环:飞书告警

监控盯得再好,人也不能 24 小时盯着看板。所以最后一步是让告警找到人:Grafana 的阈值或者异常事件,直接打到飞书机器人,群里 @ 相关同学。

告警的关键不是多,是准。我们的告警是后端服务自己算出来的,每 5 分钟评估一次,盯的是真实业务阈值:支付成功率低于 95% 告警、退款失败率高于 10% 告警、提现成功率低于 90% 告警、结算在途超过 2 小时告警。这些阈值不是拍脑袋定的,是按业务可承受的底线反推的。

早期我们什么异常都告警,群里一天刷几百条,大家都设成免打扰,真出事反而没人看。后来把噪音压下去,只留真正需要人介入的,告警才重新"值钱"起来。慢 SQL 这块还做了去重:同一份慢查询 30 分钟内不重复告警,持久化到 slow_query_log 表运营后台可查。这样群里不会被同一条慢查询刷屏,但记录一个没丢,事后还能回查。

团队的排查三步法

讲了这么多工具,最后落地的还是一套排查顺序,我们团队固化成了规矩:

  1. 先看 Console 报错——前端层面有没有问题;
  2. 再看 Network 失败请求——接口是不是挂了、状态码是啥;
  3. 最后对齐后端日志——拿 RequestId 去后端日志里捞完整轨迹。

这个顺序不是随便排的,背后的逻辑是从离用户最近的地方往远查。用户感知到的问题,最先体现在前端:页面白屏、按钮点不动、弹了个错误框。这些在 Console 和 Network 里直接能看到,不用查后端。如果前端没问题,再看 Network 里的接口——状态码 4xx 是前端参数错或没权限,5xx 才需要往后端日志钻。这样一层层往里走,避免一上来就扎进后端几百 MB 的日志里翻。

这三步看着朴素,核心好处是不会一上来就瞎翻后端日志。

多少次"系统慢"最后发现是前端的一个报错引起的,压根不是后端的事。先从离用户最近的地方查起,能省掉大量无效翻日志的时间。

这套三步法我们固化成了团队的排查肌肉记忆:线上反馈一进来,值班同学的第一反应不是 ssh 上后端,是先让用户或客服把 requestId 或截图发过来。有了 requestId,Console、Network、后端日志这三步按顺序走,基本 10 分钟内能定位。没有 requestId,这三步就走不动——这正是前面花那么多力气铺 RequestId 链路的意义:它把"排查"从靠运气变成靠流程。

排查还有一个反直觉的经验:先怀疑前端,再怀疑后端。不是因为前端容易出问题,而是因为前端问题定位快、验证成本低得多。改个前端、刷新页面就能验;后端问题要查日志、改代码、重新部署,验证成本高得多。从验证成本低的查起,即使最后发现是后端的事,前面排除前端也花不了几分钟——但反过来,一上来钻后端,如果是前端的问题,你就白费一上午。

写在最后

可观测性这个东西,平时感觉不到它的存在——系统跑得好好的,谁会去夸监控做得好。但一到出事,它就是救命的。

四根线各司其职:

  • RequestId:让你能"找到那条线";
  • 统一错误信封:让你能"读懂那个错";
  • 监控告警:让你能"第一时间知道出事了";
  • ELK:让你能"翻历史、做审计"。

它们合在一起,才是一条真正能排查的链路。

单独上任何一个,都只是个工具。串起来,才是可观测性。

我见过不少团队卡在"工具都上了,可观测性还是不行"。根子就在于工具之间没有那条共同的线——APM 一套日志、ELK 一套日志、业务监控又一套,排查一次问题要在三四个系统之间跳,靠时间戳和人脑对齐。RequestId 这条线的作用,就是把三四个系统缝成一个:同一个 id,在哪都能搜到。工具是孤岛,RequestId 是桥。 桥修好了,可观测性才从"堆工具"变成"能排查的链路"。

可观测性这件事,我有个体感想强调:它的价值是「平时不显、战时救命」,但「平时不显」正是它最难做好的原因。因为不出事的时候,没人会夸监控做得好,预算评审时可观测性基建最容易被砍(“这个看板一年没人看,砍了吧”)。可一旦出事,所有人第一反应都是"为什么没告警"“为什么找不到日志”。所以可观测性基建的投入,得靠团队负责人顶住"平时看不见价值"的压力持续做——RequestId 链路、统一错误信封、监控告警分级,每一根线都是这样攒下来的。

还有一个工程细节值得拎出来:日志要分级落文件。我们的 prod 日志拆了三个 appender:全量日志(app.log,10MB × 14 天)、只 ERROR 级的 error.log(10MB × 30 天)、控制台给容器/systemd 抓。单独切一个 error 文件,是因为出事时几百 MB 的全量日志里淘 ERROR 太慢,而 error.log 直接定位。配合 requestId,在 error.log 里按 id 一搜,出错的请求轨迹全在那。日志分文件不是为了好看,是为了出事时能快。

告警分级表:不是所有异常都该叫醒人

告警最大的敌人不是"漏报",是"噪音"。早期我们什么异常都进飞书群,一天几百条,结果大家都设免打扰,真出事反而没人看。后来按严重度分级,告警才重新值钱起来:

等级触发条件通知方式响应时限
P0(致命)核心服务 5xx 率 > 5%、数据库挂了电话 + 飞书 @全员5 分钟内响应
P1(严重)P99 > 3s 持续 5 分钟、成功率 < 95%飞书 @值班同学15 分钟内
P2(警告)缓存命中率掉到 80% 以下、连接池 > 80%飞书群(不 @)工作时间内处理
P3(提示)单个接口偶发超时、慢查询仅记日志,不打扰下次迭代排查

分级标准:这条告警需不需要"现在、立刻、人介入"?需要就 P0/P1,不需要就降级。好告警的标准是 actionable——收到就能立刻动手处理的,才是好告警;收到还得想半天"这要不要管"的,都是噪音。

一个反例:有人把"CPU 使用率 > 70%“配成 P1 告警。但 CPU 高不一定是问题(可能是正常高峰),收到告警的人不知道该干啥——扩容?重启?查代码?这种告警就是噪音。正确的做法是配"CPU > 90% 持续 10 分钟”,并且配套处置预案(扩容 / 限流 / 排查慢查询)。告警不带处置预案,等于把"我该怎么办"这个问题甩给值班的人,他多半也是干瞪眼。

告警降噪我们走的是从"全量告警"到"分级 + 去重"的演进。最开始图省事,所有异常都进飞书群,结果一天几百条,半个月后所有人都设了免打扰,告警通道等于废了。后来做了两件事才救回来:一是按 P0-P3 分级,只有 P0/P1 才 @ 人,P2 只进群不 @,P3 只记日志;二是慢 SQL、同类异常做时间窗口去重(同一份 30 分钟内不重复告警)。降噪之后,群里一天只剩几条告警,每条都值得点开看,通道重新生效了。 告警通道和红旗一样,用滥了就没人当回事。
