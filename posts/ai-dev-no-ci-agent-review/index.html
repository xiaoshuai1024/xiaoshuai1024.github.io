一条 CI 都没有:自治多 Agent 评审回路怎么兜住代码质量

跟同行聊起质量保障,大家第一反应都是「你们 CI 怎么配的?跑多少分钟?」。我的回答每次都让对方愣一下:我们的仓库,一条 CI 流水线都没有。

没有 GitHub Actions,没有 GitLab CI,没有 Jenkins。.yml 里一个 job 都找不到。

那质量靠谁?靠一套自治多 Agent 评审回路,我已经稳定跑了一年多。这篇把它拆开:它兜住了什么,又诚实地讲一句,它替代不了什么。

为什么一开始就没配 CI,不是嫌麻烦。这个项目是 4 个人 + AI、两个月冲百万行代码的节奏,代码改动密度极高,一天几十次提交是常态。传统 CI 跑一轮完整测试要十几分钟,等它绿了黄花菜都凉了——AI 协同研发的节奏根本等不起串行流水线。

其次,CI 抓的问题有限。流水线擅长抓「编译过不过、测试红不红、lint 报不报」,但抓不了「这段代码的设计对不对」「跨模块的契约对不对」「这坨改动有没有引入语义错误」。而 AI 写代码最容易犯的就是后者——编译能过、测试能绿,但逻辑是错的(270 行静默蒸发就是典型)。我需要一道能读语义的关,CI 给不了。

这两个动机合在一起,就催生了这套 Agent 评审回路:用并行的、能读代码的 Agent 替代 CI 的静态检查和人工评审,用双计数器收敛让它自治,用硬测试门禁兜住动态行为。它不是「省掉 CI」,是「换一种更适合 AI 协同节奏的质量保障形态」。

回路全貌:从「人盯」到「流程兜底」

先说整条回路长什么样,后面再展开关键设计:

scope 分桶 → 并行只读评审 → 流式自动修复 → 双计数器收敛 → 硬测试门禁 → 放行提交。

整个评审回路是我们那条 AI 研发工作流 的最后一道关。前面 TDD 已经把红/绿/重构走完了,到这里要决定「这坨改动能不能进仓库」。

关键的判断是:既然能开并行 subagent,就尽量多开。这是团队写进硬约束的一句话。所以这套回路从设计第一天起,就是奔着「并行扇出」去的,不是排队串行。

scope 分桶:并行扇出的第一步

很多人把并行评审理解成「开 N 个 Agent 同时看一坨代码」,这是错的。N 个 Agent 看同一份 diff,要么互相重复(都说同一个问题),要么互相打架(一个说要加锁一个说不用)。并行的前提是scope 不重叠。

我们的做法是按模块分桶。一次改动落到仓库,先按目录和模块边界切成几个互不重叠的 scope:后端 Controller/Service 一桶、Mapper 和 SQL 一桶、前端组件一桶、迁移脚本一桶。每个桶派一个评审 Agent,它只看自己桶里的文件,别的桶不碰。

分桶有个硬规则:桶与桶之间不能有交叉文件。一个文件只能属于一个桶,否则两个 Agent 评审同一份代码,收敛计数会重复算。遇到横跨多模块的改动(比如一个接口从 Controller 改到 Mapper),按改动的主落点归桶,评审 Agent 在结论里标注「本次改动涉及 XX 模块」,留给全量复审那轮去查跨模块问题。

分桶粒度也是个权衡。桶太细(每个文件一桶),Agent 数量爆炸、调度开销大;桶太粗(整个后端一桶),又退化成串行了。我们实测下来,单桶 30-50 个文件是甜区,一次大改动切 6-8 个桶,并行跑完大概 3-5 分钟。超过 50 个文件的桶就再按子模块拆。

分桶还有个隐藏好处:它天然带「问题归因」。某个桶反复冒问题,说明那个模块改动质量低,要么是改动者没吃透模块,要么是模块本身耦合太重。跑完回路看各桶的问题分布,能反向定位团队/ AI 在哪个模块上更容易出错,下次重点盯。这套回路除了兜质量,还顺手产出了改动的「健康度画像」。

关键设计一:评审 Agent 只读,而且 worktree 隔离

这一条是被事故逼出来的硬规矩。

早期做法是让评审 Agent 直接在工作区里看代码、顺便「帮我改两行」。代价是:评审 Agent 自作主张地把一段它「觉得冗余」的代码删了,而那段是并发守卫。这件事我专门写过复盘 《编译能过 ≠ 代码正确》:270 行静默蒸发,mvn compile 照样绿。

从那以后铁了心:评审 Agent 一律只读。

  • 按模块自动分桶,每个 Agent 各自一个 worktree 隔离;
  • 绝不改代码,一个字都不许动;
  • 它只输出「这里有问题、严重度多少、建议怎么改」。

改的活儿,交给另一类 Agent。这个「读写分离」看着啰嗦,但它是整套回路能自治的前提:评审的嘴和修复的手,得是两个独立的角色,谁也不能既当裁判又当运动员。

评审 Agent 的输出:结构化,不是自由发挥

评审 Agent 的输出不能是「这段代码感觉有点问题」这种自由文本,否则修复 Agent 没法消费。我们定了固定格式,每条问题必须带四个字段:文件路径 + 行号、问题描述、严重度(blocker / major / minor)、修复建议。少了任何一个字段,这条结论直接丢弃,不进收敛计数。

为什么卡这么死。因为评审结论是要被另一个 Agent 程序化消费的,自由文本等于没说。早期版本允许 Agent 自由描述,结果修复 Agent 经常理解错——评审说「这里并发不安全」,修复 Agent 理解成「加个 synchronized」,实际要的是乐观锁。结构化输出逼着评审 Agent 把话说死,把「建议怎么改」明确写出来,修复 Agent 才有据可依。

严重度分级也有讲究。blocker(阻塞级)是「不修不能提交」,比如资金计算用浮点、状态机漏锁;major(一般)是「应该修但能先过」,比如命名不规范、缺注释;minor(建议级)是「锦上添花」,通常不进收敛计数,只是记一笔。只让 blocker 和 major 进双计数器,minor 太多会污染收敛判断。

修复 Agent 的边界:哪些能直接改,哪些必须问人

修复 Agent 不是什么都能动。我们划了三条线:

阻塞级(blocker)且修复方案明确的——比如把 double 改成 long、补一个 @Transactional——修复 Agent 直接改,不用问人。这类修复风险低、收益高,卡着等人反而拖慢回路。

一般级(major)或修复方案有多种选择的——比如「这个方法太长要不要拆」「这个异常该抛还是该吞」——修复 Agent 只给出候选方案,把决策权交回人。因为这种选择往往涉及设计意图,AI 猜错代价大。

任何涉及业务语义的改动——比如改一个状态枚举的值、动一个金额计算公式——一律不许修复 Agent 自己改,必须人工介入。资金和权限这两块,「读」可以放给 Agent,「写」必须人来。

这条边界是事故逼出来的。早期修复 Agent 自作主张把一个 FAILED 状态的重试逻辑「优化」掉了,理由是「看起来重复」。结果提现失败后再也不重试,资金卡在中间态。从那以后,涉及状态机和金额的代码,修复 Agent 连碰都不许碰。

关键设计二:流式修复 + 双计数器收敛

评审结果怎么变成修复?我们是流式的:一个评审 Agent 的结果回来,立刻派一个修复 Agent 去处理,不用等全部评审跑完。阻塞级的问题甚至不用问人,直接改。

难点的核心是:怎么知道「评审完了」?这也是这套回路最值钱的设计。

单计数器为什么不够

最早只挂了一个计数器:「问题数为 0 就算收敛」。很快翻车——某一轮正好评审的是一个空 diff,问题数是 0,回路就高高兴兴地宣布「完成」了。其实大量问题只是还没被看到,不是不存在。

双计数器收敛规则

改成双计数器之后:

  • 🔴 严重问题:要连续 2 轮清零;
  • 🟡 一般问题:要连续 3 轮清零。

两个计数器同时达标,才算真收敛。

为什么严重要 2 轮、一般要 3 轮?这是定出来的工程参数,不是拍脑袋:

  • 严重的改得勤、修完马上能验证,2 轮够确认;
  • 一般的容易在修严重的时被连带搅出来,得多看一轮。

双计数器收敛伪代码

# 评审回路收敛判定(简化骨架)
def should_converge(rounds_history):
    """根据历史评审结果,判断是否收敛"""
    severe_clear_streak = 0    # 严重问题连续清零轮数
    minor_clear_streak = 0     # 一般问题连续清零轮数
    
    for round_result in rounds_history[-5:]:  # 看最近 5 轮
        if round_result.severe_count == 0:
            severe_clear_streak += 1
        else:
            severe_clear_streak = 0  # 一旦有问题,清零重计
        if round_result.minor_count == 0:
            minor_clear_streak += 1
        else:
            minor_clear_streak = 0
    
    # 双计数器同时达标才算收敛
    return severe_clear_streak >= 2 and minor_clear_streak >= 3

# 熔断判定
def should_circuit_break(rounds_history):
    """连续 10 轮问题数不下降,熔断"""
    if len(rounds_history) < 10:
        return False
    last_10 = rounds_history[-10:]
    total_problems = [r.severe_count + r.minor_count for r in last_10]
    # 问题数没有下降趋势(非严格递减)
    return not all(total_problems[i] > total_problems[i+1] for i in range(len(total_problems)-1))

关键细节:severe_clear_streak 一旦被打断(某轮又冒出严重问题),两个计数器都清零——因为一般问题也可能因为这次严重修复被搅动。两个计数器是耦合的,不能各算各的。

双计数器实战:一次差点翻车的收敛

讲个真实场景。某次改动跑了 4 轮:第 1 轮评出 3 个严重、5 个一般;第 2 轮严重清零(连续 1 轮)、一般还剩 2 个;第 3 轮严重又冒出 1 个(修复第 2 轮问题时引入的新问题)、一般清零。如果用单计数器看一般问题,第 3 轮它清零了,可能误判「差不多了」。但双计数器里,严重的连续清零被打断,两个计数器同时归零重来,回路继续跑。第 4、5 轮才真正双双达标。

这就是双计数器的价值:它捕捉到「修一个问题时引入了另一个问题」这种连鎖反应。单计数器只看一个维度,看不到修复动作对另一个维度的副作用。严重和一般是耦合的——修严重的常常搅出一般的,修一般的偶尔也碰出严重的,两个计数器互相校验,才不会在半截宣布胜利。

熔断也在这场景里发挥作用。假设这个连鎖一直不停,严重问题修了冒、冒了修,跑到第 10 轮还没收敛趋势,熔断触发把人叫回来。机器兜底有边界,不能让它在「修了又冒」的死循环里空烧 token。

关键设计三:每 5 轮全量复审 + 10 轮熔断

光有双计数器还不够,还有个隐蔽的坑:增量评审会近视。

如果每轮都只评审「这轮新改的」,Agent 就永远只看眼前那一小块,全局反而漏了。所以我们加了每 5 轮强制全量复审——不管这几轮改没改,到第 5 轮就把整个变更范围重新扫一遍。这一条抓出过好几个「局部都对、拼起来炸」的问题。

另一个极端是无限循环:某个问题修了又冒、冒了又修,回路在那空转。于是加了熔断:

  • 连续 10 轮问题数不下降,直接中止;
  • 把人叫回来介入。

机器兜底归兜底,但不能让它在死循环里烧 token。

全量复审和熔断是一对互补的设计。全量复审治「近视」——防止增量评审漏掉全局问题;熔断治「远视」——防止回路在某个永远修不完的问题上空耗。两个机制一个保「看得全」,一个保「停得下」,合起来才让这套自治回路既不漏问题、也不无限烧。

全量复审那轮还顺手干一件事:重新分桶。因为前几轮的修复可能改动跨了模块边界,原来按模块切的桶已经不准了。第 5 轮全量复审时,按当前最新的文件状态重新分桶再评审,等于「刷新视野」。这一步看着多余,但能抓到「修复 A 模块时顺手改了 B 模块的文件,而 B 模块的评审 Agent 根本没看到」这种隐性跨模块改动。

熔断触发后人做什么。看回路最后几轮的评审日志,找「为什么修不完」的根因。常见三种:修复方案本身有问题(修了引入新问题)、问题依赖外部信息(AI 猜不到设计意图)、改动太大根本不该一个回路搞定。前两种调整方案继续跑,第三种拆成多个小改动分批走。熔断不是失败,是回路在说「这个我搞不定,该你了」。

真实规模:它没你想象的轻量

有人会觉得「这玩意儿就是个花架子,真到大改动量肯定跑不动」。几个真实数字:

  • 某一轮,同时跑着 21 个评审 Agent + 13 个修复 Agent;
  • 某次一次性评审 320 个文件,按模块分给 8 个 Agent 并行处理。

所以它不是「跑个小 demo」的玩具,是真拿来兜一个多租户 SaaS 的全栈改动量的。重要的是这套回路扛得住「几百文件量级」的真实评审。

诚实地讲成本。一次完整回路(5-8 轮收敛)跑下来,token 消耗大概是单次人工评审的 5-10 倍——因为要开多个 Agent、要全量复审、要跑够轮数。换成钱,一次大改动的评审成本大概几十块到一百多块。听着不少,但跟「一个线上事故的代价」比,跟「一个高级工程师半天 review 的时间成本」比,这笔账划得来。

回路跑得多快也值得说。并行扇出后,一轮评审(含修复)大概 2-4 分钟,5 轮收敛 15-20 分钟。这个速度是人工评审做不到的——人看 320 个文件至少要大半天,而且看到后面注意力一定下降。机器不会累,但它会重复和机械,所以并行 + 全量复审的组合,本质是用机器的「不知疲倦」换人的「判断力」,各取所长。

什么场景这套回路不划算。小改动(改一两个文件、修个 typo)跑完整回路是杀鸡用牛刀,token 花了收益却很低。所以实际使用时会看改动量:小改动直接人工过,只有中大型改动(涉及多模块、超 20 文件)才启动完整回路。这套门禁是给「有翻车风险」的改动准备的,不是给所有改动无差别套上。

收敛之后,还有一道硬门禁

注意:回路收敛 ≠ 可以提交。

收敛只是说明「评审这一关过了」。真正放行前,还得过一道硬测试门禁:

  • 分层测试(UT / IT / E2E)全得过;
  • 契约扫描全得过。

为什么?因为评审 Agent 再聪明,也只是「读代码读出来的判断」,它替代不了「真把代码跑一遍」的行为校验。这个教训,是 270 行那段事故逼出来的——当时的评审回路只跑编译、没跑测试,编译绿了就以为没事。

测试怎么兜、纪律怎么立,见 《三层测试纪律》;提交前那几道契约闸,见 《四道契约闸》。这套门禁和评审回路是配合关系,不是替代关系。

评审和测试为什么必须分开

有人会问,评审 Agent 既然能读代码,为什么不连测试也一起评了。因为评审是「静态判断」,测试是「动态验证」,两个维度抓的问题不一样。

评审 Agent 读代码,能发现「这个锁加错位置了」「这个金额用了浮点」这类静态可见的问题。但它发现不了「这段代码跑起来在并发场景下会死锁」「这个 SQL 在百万行数据下会慢成什么样」——这些只有真跑一遍才知道。评审是看图纸找设计错误,测试是试车找运行错误,缺一不可。

那 270 行事故为什么发生在评审回路下。因为当时的回路配置是「编译过就放行」,没挂测试门禁。评审 Agent 看到的代码编译没问题,但被删掉的 270 行里有测试覆盖的代码,测试一跑就红。从那以后硬门禁里测试是必过项,不是可选项。评审收敛只是第一关,测试全绿才是第二关,两关都过才放行提交。

这套「评审 + 测试」的双门禁,比传统 CI 只跑测试要厚一层。CI 是「测试过就放行」,我们是「人(评审 Agent)觉得没问题 + 机器(测试)跑出来没问题」才算过。多出来的那一层评审,抓的是测试覆盖不到的设计缺陷和跨层契约问题。

诚实地说:它替代了 CI 的什么、替代不了什么

最后讲点反共识的——这套 Agent 回路,并没有完全替代 CI。

它替代了 CI 里「静态检查、代码评审、跨层契约对齐」这一大块,而且做得比死板的流水线灵活得多:它能读语义、能分模块、能自动修。

但它替代不了 CI 的两件事:

  • 稳定的、与提交解耦的、人人信得过的「事实源」;
  • 强制不可绕过性。

Agent 回路是「软」的——worktree 会清、prompt 会漂、收敛规则会被人改;CI 是「硬」的——推上去就跑、谁也绕不过。

所以现阶段这套 Agent 回路对一个「AI 重度参与、人随时在场」的小团队,是够用的——快、灵活、能并行、能自治。等哪天团队大了、要给「不信任的人/Agent」一个不可绕过的兜底,我还是会把 CI 加回来的。不是二选一,是不同阶段的不同兜底。

没有 CI 的这段日子,质量没掉;反而因为不再迷信「流水线绿了 = 没事」,注意力真正放回了「代码到底对不对」这件事上。

这套回路最怕什么

诚实地说,这套自治回路有三个软肋,得知道边界在哪。

第一,prompt 会漂。评审 Agent 的判断标准写在 prompt 里,prompt 一改,评审口径就变了,而历史评审结论不会跟着重算。这意味着同一坨代码,上个月评 2 个问题,这个月可能评 5 个或 0 个——不是代码变了,是 Agent 的「眼睛」变了。对策是 prompt 进版本管理,改动要走评审,但这层约束比 CI 配置文件软得多。

第二,模型能力会变。底层模型升级(比如换个更强的模型),评审风格、严格度、甚至对某些模式的认知都会变。我们遇到过模型升级后,原本能收敛的回路突然多冒出大量「风格问题」,得重新调收敛阈值。模型是回路的地基,地基换了上层得跟着调。

第三,它依赖人随时在场。这套回路是「人 + 机器」的协作,不是全自动。熔断了要人来介入,边界判断要人来拍板,业务语义的改动要人来审。如果团队里没有懂代码的人盯着,这套回路会退化成「机器自说自话」,那比没有还危险。

知道了软肋,就知道这套回路适用在哪:AI 重度参与、有懂代码的人在场、追求迭代速度的小团队。超出这个场景——比如外包团队、开源社区、不可信的提交者——还是老老实实配 CI,用「不可绕过」换「不够智能」,这是更稳的兜底。适用边界的清醒,比吹嘘「替代了 CI」诚实得多,也更不容易在错的场景翻车。

一句话总结

没有 CI 不是偷懒,是用一套更贴合 AI 协同节奏的机制替代了它。这套机制的代价是软约束(prompt 漂、模型变、依赖人),收益是快、并行、能读语义。它兜住了 CI 兜不住的设计和契约问题,但兜不住 CI 兜得住的「不可绕过性」。知道兜住了什么、兜不住什么,比争论「要不要 CI」重要得多,也更经得起时间检验和团队规模变化的考验。
