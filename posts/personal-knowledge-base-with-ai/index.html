收藏夹吃灰怎么治:用 RAG 给自己搭一个能提问的私有大脑

我攒了十年的收藏夹、笔记、PDF,真到要用的时候一个都翻不出来。面试前临时背,学新东西时重新查,重复劳动。

资料够多,问题在它们是死的——只能人去翻,不能被提问、不能被检索、不能主动复述。

《RAG 知识库从 0 到 1》讲的是给业务系统搭知识库。这篇反过来,讲给自己搭一个学习大脑。

一、个人学习的三个真实痛点

先对号入座,看看是不是都中过:

  • 收藏即遗忘。看到好文章存下来,“以后再看”,然后就没有以后了。收藏夹两千条,真记得的不到十条。
  • 笔记即坟墓。记完的笔记再也打开。不是不想复习,是检索成本太高——记不清写在哪、关键词是什么。
  • 学了等于没学。一个知识点当时懂了,过两个月忘干净。因为没有一个"定期把知识推回脑子里"的机制。

资料如果能被提问、被检索、能主动复述,学习闭环就成立了。这是个人 RAG 知识库要解决的事。

这三个痛点背后有一个共同的根因:知识的存储方式和大脑的使用方式不匹配。大脑不是靠「翻目录」找知识的,是靠「联想」——看到一个场景,联想到相关的知识。但收藏夹和笔记软件都是「目录式」的,要找知识得先记得「它存在哪」,这本身就需要记忆,而记忆正是最容易失效的。RAG 知识库把存储从「目录式」变成了「联想式」——你不用记得知识存在哪,你只需要用自然语言描述你的问题,RAG 自动把相关的片段「联想」出来。这个匹配方式,和大脑的工作方式一致,所以它比翻笔记有效得多。

还有一个被忽视的痛点:知识之间是孤岛。前端笔记里记了「虚拟 DOM 的 diff 算法」,架构笔记里记了「微前端的子应用隔离」,这两条知识其实是相关的(微前端隔离就用到了 diff 原理),但它们存在不同的笔记里,你从来不会把它们关联起来。RAG 知识库可以跨主题检索——你问「微前端隔离和虚拟 DOM 有什么关系」,它从两个库里各捞片段,帮你在孤岛之间搭桥。这种「跨域连接」是人脑最累的活,但恰好是 RAG 最擅长的。

二、核心思路:把资料变成"可对话的私有大脑"

RAG 的原理在业务版讲过:问大模型之前,先从私有资料里检索相关片段,塞进 prompt,让它看着资料回答。

个人版的架构一模一样,只是「租户」换成了「学习主题」:

  • 前端资料库:笔记、收藏、PDF、博客剪藏
  • 切片与向量化:把长文切成块,算成向量存起来
  • 按主题隔离:前端一个库、Java 一个库、架构一个库,互不串味
  • 检索 + 对话:问问题时,先从对应主题库捞片段,再让 AI 看着回答

关键差异在用法。业务知识库只做问答,个人学习库要支持更多场景,靠的是改 prompt,不是改架构。

数据流向:从原始资料到回答

整套数据流一次说清,免得你照着搭时漏环节:

  1. 资料入库:把 PDF / Markdown / 网页剪藏扔进 incoming/ 目录。
  2. 切片:按段落 + 重叠窗口(overlap 200 字)切,避免关键上下文被切断。这一步的参数直接影响检索质量——切太大召回不准,切太小上下文丢失。
  3. 向量化:每块文本过 embedding 模型,生成向量(用 OpenAI text-embedding-3-small 或本地 bge-large-zh)。
  4. 入库:向量 + 原文一起存进 ES / pgvector,带上 topic 字段做 metadata filter。
  5. 检索:用户提问 → 同样向量化 → 在指定 topic 库里做 top-k(一般 k=5)相似度检索。
  6. 组装 prompt:把 5 个片段拼进 system prompt,加上"看着资料回答,没覆盖就说不知道"的约束。
  7. 生成:大模型出回答,标注出处片段编号。

《RAG 知识库从 0 到 1》有完整的 Spring AI 落地代码。真正难的不是写出来,是调参——chunk size、overlap、top-k、相似度阈值,这四个参数决定了回答质量,得拿自己的资料反复调。我踩过最大的坑是 chunk size 默认 1000 字太大,导致一个 chunk 里混了两个主题,检索时召回不准。改成 400 字 + overlap 100 字后,精度立刻提上来。

三、怎么用:一个库,三种问法

同一个知识库,换一句 system prompt,就能切换学习场景。三种模式的切换画出来,「架构不变、只换 prompt」的设计就清楚了:

这张图最该注意的是中间那行标注「换 prompt」——三种模式共用同一个知识库、同一套检索逻辑、同一个 embedding 模型,区别只在 system prompt 不同。这意味着加一种新模式几乎零成本:写一段新的 prompt 设定,就能从「查资料」变成「当老师讲给我听」或者「帮我出一份练习题」。这种「架构不变、prompt 扩展」的设计,让个人知识库的用法可以随学习需求自由生长,不用改代码。

模式一:查资料(“这个知识点是什么”)

最基础。问"什么是 MVVM 的双向绑定",AI 从笔记里捞相关片段,结合自己的理解回答,并标注出处。比搜索引擎强在:它看的是你自己的笔记,术语和深度是你熟悉的。

这个模式有个搜索引擎比不了的优势:回答的深度匹配你的水平。搜索引擎搜「MVVM 双向绑定」,出来的文章有浅有深,你可能翻到一篇给初学者看的科普文,信息密度太低;也可能翻到一篇源码级分析,太深看不懂。RAG 知识库搜的是你自己记的笔记,笔记的深度就是你的深度——你记到什么程度,回答就在什么程度,不深不浅,正好是「你跳一跳能够到」的水平。这个「深度匹配」对学习特别重要:太浅没收获,太深看不懂,匹配自己的水平才能高效吸收。

你是学习助手。根据下面检索到的笔记片段回答问题,
术语和深度匹配笔记原貌;笔记里没有的部分明确说"笔记未覆盖",
不要编造。每个论点标注出处片段编号。

这个 prompt 里「笔记里没有的部分明确说笔记未覆盖」是一条关键约束。它防止 AI 在笔记覆盖不全时用自己的知识「补全」——补全的部分看起来和笔记里的一样,你分不清哪句是你记的、哪句是 AI 编的,学习时就会把 AI 的「补充」当成自己的笔记内容,产生虚假的掌握感。强制 AI 标明「笔记未覆盖」,你一眼就知道「这部分我没学过」,该补资料就补资料,而不是被 AI 的「全知」误导。

模式二:模拟面试(“考考我”)

这是我用得最多的模式。把 system prompt 换成面试官设定,AI 从知识库抽知识点提问,你作答,它对照资料打分、补充。主动回忆比被动重读有效十倍。

模拟面试模式为什么比「重读笔记」有效?背后是学习科学里「主动回忆」和「间隔重复」两条原理。重读笔记是「被动输入」,信息从笔记进眼睛进脑子,看似熟悉,但这种熟悉是「识别感」不是「回忆能力」——你看到答案觉得「对对对我知道」,但真被问到时想不起来。模拟面试是「主动回忆」,你必须从脑子里把答案「拽出来」,这个拽的过程才是真正巩固记忆的。RAG 知识库的模拟面试模式,等于给你配了一个不知疲倦的考官,随时考你、随时点评,而且考题都来自你自己的笔记——精准命中你需要复习的内容。

你是资深面试官。从提供的资料里挑一个知识点向我提问,
等我作答后再点评:先指出错漏,再对照资料给标准答案,最后追问一个。
一次只问一题,我答完才进下一题。

这个 prompt 有个细节值得说:「一次只问一题,我答完才进下一题」。这条约束防止 AI 一次抛五个问题把你淹没——面试的核心是「一问一答、即时反馈」,不是「一次性把题全给你」。AI 默认倾向「把信息给全」,不加这条约束,它会一次列五个问题,你答了第一个,它已经把五个的答案都给了,后面的你就不用想了——学习效果归零。「一次一题」的约束,逼你逐个主动回忆,每个知识点都被「拽」了一遍。

模式三:知识连接(“这个和那个有什么关系”)

问"Apollo 配置中心和 Nacos 在多租户场景下的取舍",AI 从多个主题库检索,把分散的笔记串起来对比。这是人脑最累的活,交给 AI 正合适,也最能把零散的知识串成体系。

知识连接模式为什么特别有价值?因为知识的价值不在「记住单个知识点」,在「知道知识点之间的关系」。单个知识点(Apollo 支持多环境、Nacos 支持配置推送)谁都能查到,但「在多租户 + 高并发场景下,Apollo 的环境隔离和 Nacos 的命名空间哪个更适合」,这种跨知识点的对比和取舍,才是真正体现理解深度的。而正是这种对比,人脑最不擅长——你要同时记住两个方案的细节,在脑子里做对比,认知负荷极高。RAG 知识库可以同时从两个主题库检索,把两边的细节都摆出来,让 AI 帮你做对比,你只需要在 AI 的对比基础上做最终判断。认知负荷从「全靠脑子」降到「看对比做判断」,效率提升不止一个量级,也让人愿意做这种以前懒得做的对比。

四、按主题分库:个人版的"多租户"

业务系统按公司租户隔离,个人库按学习主题隔离,机制一样:metadata filter。

我的分法按成长线分,和这个博客的脉络一致:

  • 前端库:Vue/React、工程化、性能优化
  • 后端库:Java、Spring、数据库、中间件
  • 架构库:微服务、DDD、高并发设计
  • AI 开发库:RAG、Agent、Prompt 工程

检索时指定主题库,避免"问前端问题时 AI 翻出后端笔记乱答"。这是个人库最容易忽略但最该做的——不分库,检索质量直接垮。

分库决策表:不是越多越好

分库粒度有讲究。分太粗,检索时多主题混在一起,精度差;分太细,管理成本高,跨主题查询又麻烦。我的标准:

情况建议分库数理由
你是单一领域深耕(纯前端)1 个库资料少,分库意义不大
跨 2-3 个领域(前端 + 后端)2-3 个库边界清晰,检索精度明显提升
全栈成长(前端→后端→架构→AI)4-5 个库按成长线分,匹配我的分法
资料超过 1 万条按主题 + 时间二级分单库向量数过大,检索会慢

一个常见误判:有人一上来就按"每本书一个库"分,结果 50 个库,管理崩溃。库的粒度应该是"知识领域",不是"资料来源"。一本书的前 5 章和后 5 章可能跨多个领域,强行按书分反而割裂上下文。

五、什么不能塞进去(重要)

个人知识库最容易踩的坑,不是技术,是内容边界。有些东西看着像资料,塞进去是灾难。

绝对不能入库的:

  • 公司内部代码、真实业务数据、客户信息。泄密红线,哪怕本地部署也不行——一旦 prompt 里有敏感片段,谁都没法保证它不外泄。
  • 薪资、合同、医疗、证件信息。这些不是"知识",是"档案",不该混进学习库。
  • 公司未公开的方案、架构、人员安排。涉及保密义务,写进笔记也不行。

可以入库但要脱敏的:

  • 工作中踩的坑、做过的设计——抽象掉真实业务,只留技术决策(就像这个博客所有文章做的)。
  • 内部工具的用法——改写成通用描述,不暴露工具名和接口。

原则就一条:这个知识库将来万一公开了,我慌不慌? 慌的就不入库。宁可少存,不可错存。

这条原则有个实操上的延伸:入库前做一次「脱敏审查」。不是每条资料都能一眼判断「能不能入库」——有些资料半公开半敏感(比如公司内部的技术方案,技术思路可以分享但业务数据不能)。我们的做法是:入库前把每条资料过一遍,把敏感部分(真实业务数据、客户名、内部系统名)抽象成通用描述,只留技术决策和思路。这个过程有点费时间,但它是保证知识库「可公开」的必要步骤——脱敏一次,后面用起来就放心了,不用担心「这个知识库万一被别人看到怎么办」。

还有一个关于内容质量的判断:不是所有资料都值得入库。有些收藏的文章,内容浅、重复、或者过时了,入库只会增加噪音、拉低检索精度。入库前做个简单筛选:这篇文章三个月后我还会参考吗?会,入库;不会,扔掉。知识库的质量不取决于「存了多少」,取决于「存的有多精」。100 条精选资料比 1000 条垃圾收藏的检索质量高得多——因为垃圾资料会污染检索结果,你问一个问题,召回的第一条是垃圾文章,正确的回答反而被排到后面。

六、把它做成 MCP,而不是又一个网页

最关键的一步:别把知识库做成一个要主动打开的网页。

人是有惰性的。知识库做得再好,只要它"在另一个标签页",你就会懒得切过去。我的做法是把它做成一个 MCP 工具——接入日常的 AI 助手(Claude Code),不需要专门打开任何东西。

写代码遇到不懂的概念,直接问 Claude Code,它会自动调知识库工具去笔记里捞,把当年记的理解带回来。学习不再是"专门抽时间学",而是"在干活时自然地复习"。

这种「干活时自然复习」的效果,比专门抽时间学强很多,原因在于场景匹配。你问 Claude Code「这个接口的事务该怎么设计」,它从知识库捞出了你半年前记的「事务边界要在 Service 层」的笔记——这个知识在「你正在写接口」的场景下被激活,你的大脑会把它和当前的代码关联起来,形成「场景→知识」的强记忆。而专门抽时间读笔记时,没有具体场景,知识是「脱context」的,记了也容易忘。RAG 知识库 + MCP 的组合,让每次干活都变成一次带场景的复习,学习效率远高于专门学习。

一切皆可脚本化那篇讲过,工具的标准形态正在从"脚本"演进到"MCP"。学习知识库是这条线上最值得做的一个。

七、一个判断

很多人觉得"学习要靠自律"“笔记要定期整理”。这些话对,但没什么实际用处——人就是不自律、就是不整理,这是人性,改变不了,不如顺着它,用系统来补。

与其对抗人性,不如把"整理"和"复习"交给系统:资料一次入库,AI 负责检索、出题、对比、串联。你只负责问和答——这两件事,才是学习里真正需要人做的,其余的杂活交给系统就好。

调参踩坑:RAG 质量翻车的 5 个常见原因

知识库搭起来不难,但"答得对"很难。我调了几个月才让回答质量稳定,踩的坑总结给你:

坑一:chunk size 不合适。 默认 1000 字太大,一个 chunk 混多个主题,检索时召回的片段跟问题对不上。我的经验:中文资料用 300-500 字 + overlap 100 字最稳。代码类资料按函数边界切,不要硬按字数。

坑二:top-k 设太大。 有人觉得"多召回几条信息更全",把 k 设成 20。结果 prompt 塞太满,模型反而抓不住重点,回答变啰嗦。k=5 是甜点,不够再调到 8,不要超过 10。

坑三:相似度阈值设太低。 为了"召回率高"把阈值设成 0.5,结果召回一堆不相关的。我的阈值是 0.75,低于这个直接回"笔记未覆盖"。宁可少答,不可错答——错答比不答伤害大。

坑四:embedding 模型跟资料语言不匹配。 用英文 embedding 模型压中文资料,精度掉一截。中文资料用 bge-large-zh 或 m3e-base,英文资料用 text-embedding-3-small,不要混用。

坑五:更新资料没重新向量化。 改了笔记原文,忘记重新跑向量化,知识库还在用旧片段回答。我的做法:资料入库走 git,commit 触发自动重新切片 + 向量化,保证向量永远是最新原文的。

这五个坑,任意一个都能让知识库"看起来能用,实际答得烂"。调参时对照排查,基本能覆盖 80% 的质量问题。

这五个坑里,坑一(chunk size)和坑五(更新没重新向量化)是最容易被忽略的。chunk size 是「初始配置」,搭的时候调一次就忘了,后面答得烂不会想到是 chunk 的问题;重新向量化是「流程问题」,不是参数问题,靠习惯不靠调参。这两个坑的特点是「不是技术难,是容易忘」——所以我们的解法也不是「记住」,而是「机制化」:chunk size 写进配置文件的注释里(标注「中文资料建议 400 字」),更新向量化走 git commit 自动触发(不用人记得跑)。把容易忘的事交给机制,比靠自觉可靠。

还有一个调参的经验值得分享:调参要有耐心,不要一次改四个参数。chunk size、top-k、阈值、embedding 模型,这四个参数互相影响——改了一个,另外三个的最优值可能也变了。如果你一次改四个,答得好了不知道是哪个参数的功劳,答得差了也不知道是哪个参数的锅。正确的做法是一次只改一个,跑一遍基准,记录效果,确认这个参数的最优值后,再改下一个。这个方法慢,但每一步都有明确的因果关系,调出来的参数组合是可信的。一次改四个看似快,实际调完不知道对不对,反而更慢。

深度展开:个人知识库的长期维护

知识库搭好、调参完成,不等于一劳永逸。它像花园,不维护就长草。

最大的维护工作是定期清理过时内容。技术领域知识更新快,半年前记的「Vue3 的最佳实践」,半年后可能已经过时(Vue 升级了、社区方案变了)。过时内容留在库里,检索时会干扰新内容的召回——你问「Vue3 状态管理怎么选」,召回的第一条可能是半年前的「用 Vuex」,而最新的「用 Pinia」被排在后面。我们的做法是每季度做一次「知识库体检」:挑几个高频问题,看召回的片段是否过时,过时的标注删除或更新。不清理过时内容,知识库会慢慢变成「历史博物馆」,答的都是旧答案。

第二个维护是补充高质量新内容。知识库不是只出不进,要持续喂新资料。但喂的时候要筛选——不是所有新文章都值得入库(参考前面的「不是所有资料都值得入库」)。我们的习惯是:看到好文章先放 incoming/ 目录,周末统一过一遍,值得入库的脱敏后入库,不值得的扔掉。这个「入库前筛选」的节奏,保证了知识库的「信噪比」不下降。

第三个维护是定期复盘检索效果。用了一段时间后,你会发现有些问题知识库答得好、有些答得差。答得差的,要么是资料不够(那个主题的笔记太少),要么是参数不适配(那个主题的资料适合不同的 chunk size)。复盘后,该补资料的补资料,该调参数的调参数。知识库不是「搭完就不管」的工具,是「用了才知道哪里要改」的工具——用得越多,越知道哪里需要补,越用越准。
