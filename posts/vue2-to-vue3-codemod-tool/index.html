几百个文件、几十种破坏性变更,手改得改两周,还没人能 review 得过来——而且改的顺序会互相干扰,两个人动同一个文件,合并冲突能扯一下午。我做百万行 Vue2→Vue3 升级时,第一周就放弃了手改,写了个 babel codemod。这篇讲它怎么做到"敢在真实项目上跑"。

自动化升级的底气,从来不是"改得多快",是"改得多稳"。 一个敢往生产代码上跑的 codemod,是事务化 + 保守 + 安全网,缺一条都得崩。下面展开。

为什么不能手改

Vue2 到 Vue3 的破坏性变更远不止 new Vue → createApp。我盘点过,光"必动"的就一长串:

  • 入口:new Vue().$mount() → createApp(App).mount()
  • 全局 API:Vue.use / Vue.mixin / Vue.directive / Vue.filter 都挂在 new Vue 实例上,Vue3 要改成挂在 app 实例上(app.use(...)、app.directive(...))
  • Vue.prototype.xxx → app.config.globalProperties.xxx
  • filters 整个被移除,模板里 {{ price | currency }} 全得改成方法或计算属性
  • v-model 的默认 prop 从 value/input 变成 modelValue/update:modelValue,组件库里几百个自定义 v-model 全得跟着改
  • $listeners 合并进了 $attrs,有 inheritAttrs: false 的组件行为会变
  • Vue.config.productionTip、Vue.config.ignoredElements 这些被删,留着运行时报错
  • <teleport>、<suspense> 是新东西,这次没动业务逻辑,只在脚手架里留好位置

一个中型项目几百个文件,上面这些散在各处。手改又慢又容易漏,而且——更关键的是——没人能 review 几百个人手改的 diff。改完根本不知道对没对。

手改还有个隐性成本很少有人算:改的顺序会互相干扰。A 先改了 new Vue→createApp,B 再去改 import 时,文件已经不是原来的样子了,如果两个人改了同一个文件的不同部分,合并冲突能扯一下午。几百个文件、几十种破坏性变更,如果全靠手改,光是协调「谁先改哪个文件」就够耗掉好几天。codemod 把这些改动收口成一次原子操作,顺序问题直接消失——这也是为什么大型迁移几乎必然要走自动化,不只是效率问题,是协作成本的问题。

但我也不信任"一键替换"。正则会把不该改的也改了:createApp 出现在注释里、v-model 出现在字符串里,正则认不出来。所以核心选择是:用 babel 做 AST 级转换,只动真正该动的,其余原样保留。

核心转换:new Vue → createApp

工具的一条核心规则,是把 Vue2 的入口:

new Vue({ render: h => h(App) }).$mount('#app')

转成 Vue3 的:

createApp(App).mount('#app')

它不是字符串替换,而是解析成 AST,识别 new Vue({...}).$mount(...) 这个模式:提取 render 函数里的 App 标识符,再重建为 createApp(App).mount(...)。

这种"看语法结构,不看字面文本"的能力,是 AST 比正则强的地方。比如 entry 文件里那个 App 到底是变量、是导入、还是从别处来的——AST 能讲清楚,正则讲不清楚。再比如 new Vue 这个字面串,出现在注释里(// 这里以前是 new Vue,现在不用了)、出现在字符串里(错误信息文案),正则会统统命中,AST 只认真正 NewExpression。

关键是保守。如果检测到文件里有多个 new Vue().$mount(),它直接跳过,只在报告里记一笔——因为不确定该转哪个。宁可不动,也不能改错。多入口的项目很常见(微前端、多页应用、SSR 渲染器),这时候瞎转一个,启动直接白屏。

类似的保守判断还有几个:看到 render: h => h(App) 里不是干净的标识符引用(比如直接传 JSX、传嵌套函数),也跳过——这种手工改更稳;看到 $mount 的参数不是字符串字面量(比如 $mount(someEl)),也跳过,因为 Vue3 的 mount 只收字符串选择器。

这几个保守判断串起来,就是一棵「该不该转」的决策树。把它画出来,什么时候跳过、什么时候转,一目了然:

这棵决策树的设计哲学是:默认不动,只有满足全部条件才转。和很多人想的相反,codemod 的默认行为不是「能转就转」,而是「能不转就不转」。原因很实在——转错的代价比漏转大得多。漏转的文件会进 note,人 review 时补上就行,成本可控;转错的文件会静默地改坏代码,如果不仔细看 diff,问题就溜进去了。所以保守不是胆小,是错误成本不对等时的理性选择:错误成本高的操作,默认就该是「不动」。

转换时顺手清掉 Vue.config.productionTip = false。这里有个细节:它不是"转",是"删"。Vue3 根本没这个 API 了,留着这行会在运行时抛 Cannot set property productionTip of undefined。codemod 识别到这行赋值就摘掉,不用人去记。清理时机也有讲究——只在确定这个文件是入口(已经命中过 new Vue 转换)的前提下才删,免得把测试文件、文档示例里那行 demo 也一起删了。

import 也跟着修:vue 的默认导入改成命名导入 { createApp };如果代码里还有别处用 Vue. 命名空间的(比如 Vue.nextTick、Vue.set),就保留一个 import * as Vue from 'vue',不能一刀切删了。Vue.set / Vue.delete 在 Vue3 下其实也没了(响应式不需要),但这属于运行时行为,codemod 这次只改入口、不动业务,留个 note 在报告里让人手动处理。

哪些破坏性变更这次故意没动

老实讲,破坏性变更里只有一部分适合自动化。剩下的硬要 codemod 处理,出错率比手改还高,得不偿失。这次划的边界是:

  • filters 移除:{{ price | currency }} 这种管道语法 Vue3 整个删了,要改成方法或计算属性。但 filter 在模板里的用法千奇百怪(链式、带参数、和 v-bind 混用),AST 改完很可能漏掉某个分支。codemod 只扫描统计了"用了 filter 的文件数",写进报告让人手改。
  • v-model 默认 prop 变化:组件库里的自定义 v-model(Vue2 用 value/ input,Vue3 用 modelValue/update:modelValue)是重灾区。但一个组件是不是"自定义了 v-model",要追溯 model: { prop, event } 选项和 props 定义,还要看父组件怎么用——这是跨文件分析,codemod 这次不碰,只在报告里列出所有声明了 model 选项的组件,人去逐个改。
  • $listeners 合并到 $attrs:这个影响的是组件透传逻辑,跑测试才能发现,codemod 静态分析不出来。
  • 全局 API 挂载:Vue.use(VueRouter) → app.use(VueRouter),这个理论上能自动转,但前提是能正确找到 app 这个实例变量。在入口文件里能转,在插件注册文件里就难说了。所以这一条也只做入口文件的转换,其余的写进 note。

划这条边界的原则是:codemod 只做"局部、确定、可验证"的改动,跨文件、依赖运行时语义的一律交还给人。宁可少改,不可改错。codemod 帮你把 70% 的机械活儿干掉,剩下 30% 的判断活儿,报告里列得清清楚楚,人 review 完照着改。

这条边界划在哪,是整套设计里最需要经验判断的地方。划得太保守,codemod 能干的活儿太少,等于没省多少人工;划得太激进,误改率上升,反而要花更多时间修 codemod 闯的祸。我的经验是:先保守,跑一遍看 skip 列表有多长。skip 列表特别长(说明很多本可以自动的活儿漏了),就往激进调一档;skip 列表里出现大量「转了之后测试挂」的,就往保守退一档。这个边界是拿真实跑出来的数据调出来的,不是拍脑袋定的。第一版 codemod 我保守到只敢转入口和 import,跑了两轮发现 filters 这种其实也能扫统计,才慢慢把边界往外扩。

还有一类改动我故意没列进边界讨论,因为它们压根不该用 codemod 处理:架构层的调整。比如 Vue2 的 mixins 在 Vue3 里推荐改成 composables,这不是语法转换能解决的——它涉及怎么拆分 mixin 的逻辑、怎么组织 composables 的依赖,属于设计决策。这种改动用 codemod 硬转,出来的代码能跑但结构一塌糊涂,还不如留着让人重构。codemod 的能力边界,不止在「AST 能不能识别」,还在「这个改动是不是纯机械的」——涉及设计判断的,再聪明的 codemod 也替代不了人。

事务化:先计划,再预检,最后执行

这是整个工具最值得讲的设计。改动不是边算边写,而是分三步:

  1. 计划(Plan):把所有要做的改动算出来,塞进一个 ChangeQueue。注意,这一步不碰磁盘,只往队列里加 change。队列里大概有这几类:

    • 依赖升级:package.json 里把 vue 从 2.x 提到 3.x,加 @vitejs/plugin-vue、vue-router@4、vuex@4 或 pinia,把 @vue/cli-*、vue-template-compiler 这些 vue-cli 时代的包移出去
    • 脚本迁移:package.json 的 scripts 里 vue-cli-service serve/build → vite / vite build,顺带把环境变量前缀从 VUE_APP_ 改成 VITE_
    • vite 脚手架:新建 vite.config.ts,挪 index.html 到根目录、加 <script type="module" src="/src/main.ts">;原来 public/index.html 里手写的 <%= BASE_URL %> 模板变量也换成 vite 的绝对路径写法
    • AST codemod:上面说的入口转换、productionTip 清理、import 修正
    • 类型生成:补几个 Vue3 下 TS 用的 shims-vue.d.ts、env.d.ts,免得 .vue 文件 IDE 报红

    队列的价值是收口。所有改动都过这一条窄路,每条 change 都是结构化数据,而不是散落在各处的 fs.writeFile。这点看着工程化,实际救命:跑挂的时候你能精确说出"队列里第 7 条 change 出了问题",而不是"某个文件改坏了"。

  2. 预检(Preflight):执行前检查队列里有没有冲突——比如两个步骤都要改 package.json 但不是同类操作(一个 updateJson 一个 writeFile),或者要写的文件已经在磁盘上但内容跟预期对不上。这种情况下直接报错中止,一个字都不写。

    预检还顺手做一件事:扫一遍有没有多入口(多个 new Vue().$mount()),前面说过的"不确定就跳过",决策就在这里做,而不是在执行阶段边跑边发现。

  3. 执行(Apply):预检过了才真正落盘;任何一步失败立即停下,提示用 restore 还原。

change 只有三种:mkdir、writeFile、updateJson,用 TS 的 exhaustiveness 收口——将来加新类型不改全就编译报错。这个看着小,实际很有用:codemod 跑久了总会想加新操作类型(比如 deleteFile、rename),有了 exhaustiveness,改 switch 的时候 TS 会把所有没覆盖的分支标红,不会漏一个分支导致静默跳过。

这种"先计划、可预检、可回滚"的事务化设计,是 codemod 敢跑在真实项目上的底气。数据库的事务讲了几十年,放到代码迁移上道理一样。

事务化还有个容易被低估的好处:它是可审计的。ChangeQueue 里每一条 change 都是结构化数据,执行完之后,「这次迁移到底动了什么」是可枚举、可回放的。这在出问题时极其有用——线上某个功能突然挂了,怀疑是迁移引入的,你不用去翻几百个文件的 diff,直接对照 ChangeQueue 看这次迁移有没有碰那个功能相关的文件。碰了就重点查那条 change,没碰就能快速排除迁移这个因素。没有事务化的 codemod,「这次迁移改了什么」是一笔糊涂账,排查时只能逐文件回溯,效率天差地别。

事务化对「迁移能不能复跑」也有价值。第一次跑完发现问题,改了 codemod 规则想再跑一次,得先回到迁移前的干净状态。有了 backup 还原,这个流程是顺畅的:restore → 改规则 → 重跑。没有事务化,第一次跑改了一堆文件,第二次再跑时哪些是「第一次改的」、哪些是「我手改的」已经分不清,重跑就会在半改的状态上再改一遍,越改越乱。事务化让「跑→验证→不对就还原→改规则→重跑」这个循环变得干净,而大规模迁移几乎必然要跑好几轮才稳。

安全网:备份 + 试跑 + 报告

光有事务化还不够,真实项目还得能"后悔":

  • 备份:执行前先打个 zip。这里有个细节——备份感知 .gitignore,不把 node_modules、dist 一起打进去,否则一个 zip 几百兆,还原也慢。
  • 试跑(--dry-run):把整个计划跑一遍但不写盘,输出长这样:
    [plan] 12 changes queued
  [updateJson] package.json  (vue: 2.7.14 -> 3.4.x)
  [writeFile]  vite.config.ts (new)
  [writeFile]  index.html      (moved to root, script type=module)
  [codemod]    src/main.ts    (new Vue -> createApp)
  [codemod]    src/main.ts    (remove Vue.config.productionTip)
  [codemod]    src/main.ts    (import: default -> named { createApp })
  [skip]       src/pages/login/entry.ts  (multiple new Vue() found, needs manual review)
  [skip]       src/components/Price.vue   (filters detected, manual)
  [note]       src/store/index.js         (Vue.set / Vue.delete used, runtime-only)
  ...
[dry-run] 0 files written. Run without --dry-run to apply.
    哪些文件会被动、依赖怎么变、哪些被跳过、哪些只是 note,一眼看全。正式跑之前我一定先 dry-run 一遍,看 skip 列表有没有出乎意料的。skip 列表里如果出现了你预期"应该被改"的文件,那就停下来看为什么——要么是 codemod 漏识别了某种模式,要么是这个文件本来就该手改,两种情况都得确认。
  • 还原(restore):出问题从备份 zip 一键还原。这里有个反直觉的设计——还原前还会再打一个"还原前备份"。为什么?因为有时候你 restore 完发现"哦原来 codemod 那版是对的,我手动改才改崩了",这时候得能再回到 codemod 后的状态。多次后悔都得兜得住。
  • 迁移报告:执行完生成 migration-report.md,把依赖变化(vue 2.7 → 3.4、新增/移除的包)、构建说明(npm run dev 现在是 vite 了)、codemod 改了哪些文件、有哪些需要人工确认的 note(比如被跳过的多入口文件)全列出来。

跑完不是"成功了",是"这是报告,请你 review + 跑测试再上线"。codemod 永远不替你做最后的决定。

这三件套里,迁移报告最容易被忽视,但它其实是 codemod 和人之间最重要的交接物。报告不是给自己看的,是给团队看的——尤其是不直接参与迁移的同事,他们需要知道「这次升级动了什么、哪些地方可能出问题、上线后要重点测什么」。报告写得清楚,团队对这次迁移才有信心;报告含糊,大家心里没底,迁移就算技术上对了,推进时阻力也大。我见过迁移技术上很成功,但因为没出报告、团队不知道发生了什么,上线后一个小故障就被归结到「是不是升级搞坏的」,扯了三天才发现跟迁移无关。报告的沟通价值,不亚于它的技术价值。

报告的写法有个经验:把「需要人工处理的」单独拎出来放最前面。codemod 跑完,最该让人立刻关注的是那 30% 被 skip 或标 note 的文件——这些是要人动手的,埋在报告中间容易被忽略。放在最前面加粗标注「以下 N 个文件需人工确认」,大家一眼就知道还有哪些活儿没干完。codemod 帮你干的活儿(自动转的部分)反而可以往后放,因为那部分已经做完了,不用人再操心。报告的排序要按「人的注意力稀缺性」来排,不是按「改动占比」来排。

用 recast 保留原格式

AST 转换有个坑:重新生成代码会丢失原来的格式(空格、注释、引号风格)。一个文件原本是单引号,生成完变双引号;原本有段作者留的注释,生成完没了——这种 diff 一出来,review 的人直接抓狂,根本分不清"哪行是 codemod 改的、哪行是格式器动的"。

这个工具用 recast 做非破坏式打印——只重新生成被改的节点,其余原样保留。换行、注释、引号风格、甚至那个怪异的缩进,都原样留着。这样 diff 干净,review 时只看到"这一行从 new Vue({...}).$mount('#app') 变成 createApp(App).mount('#app')",其他一概不动。

对比一下 babel generator 默认的行为:它会用自己的一套格式规则重新输出整个文件,结果就是一个几十行的入口文件 diff 能炸出几百行变更,一半都是引号和缩进差异。review 这种 diff 等于在做"找不同",根本没人有耐心看。

review 友好这件事,在团队协作里比"转换正确"还重要。没人 review 的 codemod 等于没跑过。

recast 也有它的局限值得说清楚。它对「被改节点」能精准保留格式,但对「被插入的新节点」就管不了那么细——新生成的代码片段用的是 recast 自己的默认格式,可能和文件原有风格不完全一致。比如原有文件用 2 空格缩进,recast 插入的新代码如果没显式配置,可能是 4 空格。解决办法是给 recast 传入项目的格式配置(从 .editorconfig 或 .prettierrc 读),让新代码也符合团队规范。这个细节不处理,新插入的代码会像一块补丁缝在衣服上,review 时一眼就看出「这段是机器加的」。recast 解决的是「不破坏原有代码」,「新代码符合规范」还得另外配。

还有一个真实踩过的坑:recast 对某些特殊语法的保留会失败。比如装饰器、可选链在旧版 recast 里支持不全,转出来要么格式乱、要么直接报错。遇到这种文件,我们的做法是把它加进 skip 列表,让 recast 完全不碰,人工改。识别这类「recast 搞不定」的文件,本身就是 codemod 该有的预检——与其转坏了再救,不如一开始就识别出来跳过。

这套设计的边界在哪

也不吹。这套设计有几个前提,离开这些前提它就不好使:

  • 项目得能跑测试。codemod 替你做的是机械改动,正确性最终还是靠测试兜底。没测试的项目,再稳的 codemod 我也不敢跑。
  • Vue2 版本别太老。建议先升到 2.7(已经有 Composition API 兼容层),再从 2.7 跳 3.x,中间摩擦小很多。从 2.5 / 2.6 直接跳 3.x,破坏面太大,光依赖冲突就够喝一壶。
  • 入口文件能识别。工具依赖能找到 new Vue().$mount() 这种入口模式,如果项目用了奇形怪状的挂载方式(比如手写 mount 函数、SSR 的 createRenderer),codemod 认不出来,得人工引导。

满足这三条,codemod 能帮你把一个几百文件的升级,从"两周人天"压到"跑一遍 + 半天 review"。不满足,那就老实手改,别图快。

这三条前提里,「项目得能跑测试」是最硬的。它在不在,codemod 的可信度天差地别。有测试,codemod 跑完之后,跑一遍测试就能筛掉绝大多数转换错误,人只要 review 测试没覆盖到的边角;没测试,codemod 的每一个改动都得人逐行确认,几百个文件的 diff,review 到天荒地老也看不完,而且人审静态 diff 的可靠性远低于跑测试。所以真正想用 codemod 做大规模迁移,前置条件不是先把 codemod 写好,是先把测试覆盖率补上来。测试是 codemod 的安全网,没有这张网,codemod 再稳也不敢往生产代码上跑。我推进迁移前,第一件事永远是看测试覆盖率,不够就先补测试,这一步省了,后面全是债。

还有一个隐性前提容易漏:升级窗口要挑。尽量避开业务高峰期,选一个需求相对空闲的迭代来做。迁移期间,代码库会有一个「codemod 跑完但人还在 review」的中间状态,这段时间如果有别的功能同时往主干预代码,合并冲突会极其痛苦。挑一个干净的窗口,让迁移这件事独占主干几天,review 完、测试过了再恢复正常的并行开发。窗口没挑好,迁移和日常需求缠在一起,两头都受影响。

几条判断

  • AST 转换 > 正则替换,正则会误伤,AST 只动该动的。
  • codemod 要事务化:计划/预检/执行/回滚,缺一不可。计划收口、预检拦冲突、执行可中断、回滚可还原。
  • 保守优先:不确定就不动,记 note 给人。多入口跳过、productionTip 只删不转,都是这条原则的体现。
  • 保留原格式(recast),否则 diff 一团糟,没人敢 review。
  • 安全网要厚:备份 + dry-run + 报告三件套,缺一个都不敢往生产代码上跑。
  • 划清边界:能自动化的(入口、import、配置)放手让 codemod 干;跨文件的、依赖运行时语义的(filters、v-model prop、$listeners)交还给人。

把这些原则拆开看,其实就一张表:

改动类型处理方式为什么
new Vue().$mount() → createApp()自动转局部、模式固定、AST 能精确识别
Vue.config.productionTip = false自动删(仅入口文件)Vue3 没这个 API,留着报错
import 修正自动转跟入口转换是同一波
package.json 依赖/脚本自动改结构化数据,updateJson 一把过
vite 脚手架 / index.html自动生成模板化,确定性高
filters、自定义 v-model只扫描,写 note跨文件 + 运行时语义,codemod 改不如人改
$listeners 合并不碰静态分析不出来,靠测试

自动化升级的本质,不是"改得多快",是"改得多稳"。事务化让每一步可追溯,保守让错误不发生或被拦下,安全网让后悔成本可控,recast 让结果可 review——这四件事凑齐,codemod 才配得上"敢跑真实项目"这几个字。
