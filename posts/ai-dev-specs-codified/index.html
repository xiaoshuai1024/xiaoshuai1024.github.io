【AI 研发实战】把项目规范写进代码底层:让新人和 Agent 都"开机即用"

这一篇属于我「AI 研发实战」系列里讲工程规范固化的部分。当我们把 AI 协同做成可复用的研发体系后,规范能不能自动执行,直接决定了人和 Agent 能不能开机即用。

我见过太多份「项目规范」,它们最后都死在同一个地方:某个 wiki 的角落里,落满灰尘,最后一次更新是半年前,内容早就和代码对不上了。

新人入职第一周没人看,三个月后也没人看,直到踩了坑,有人恍然大悟"哦这事儿其实文档里写过"——可那又怎样,文档不会自己跳出来拦你。

我的判断是:不能自己执行的规范,就不是规范,是建议。建议这种东西,在人忙的时候、在 AI 自由发挥的时候,一律等于没有。

所以我们的做法是——把项目规范写进代码底层,让它机器可读、运行时可执行。这一篇就讲我们怎么做的,以及它怎么让新人和新 Agent 都做到"开机即用"。

新人最痛的不是技术,是"这事儿在我们这该怎么干"

先说我们到底在解决什么问题。

新人入职一个项目,真正卡住他的,通常不是某个框架不会用、某个语法没见过——那些查文档就行。

卡住他的是一堆项目专属的隐性规矩:

  • 这个字段为什么要这么命名
  • 那个 SQL 为什么必须带 tenant_id
  • 提交前得先跑哪个脚本
  • 危险操作有哪些不能碰

这些东西没写在任何教程里,只在老同事的脑子里。

更关键的是,这种隐性知识的传递成本极高。每来一个新人,老人就得口述一遍;老人忙了,新人就只能靠踩坑自学。

AGI 时代又多了个新麻烦——每开一个新的 agent,它也是一个"零背景新人",而且它不会主动来问你。

一旦你把"AI 当随时入职的新人"来 onboard,立刻就能看清:这事儿光靠文档不够,得靠机器能自动加载和执行的东西。

传统规范为什么不行

传统的规范有三个硬伤,改不掉的那种。

第一,没人看。 wiki 的链接藏在 IM 群置顶里,大家点一次就忘。

第二,会过时。 代码三个月一迭代,文档没人同步。文档说"提交前跑 A 脚本",实际早就换成 B 了,照着做反而出错。

第三,不可执行。 就算文档全对,它也是一段给人看的文字,不会在你犯错那一刻拦下你。它只能事后当呈堂证供:“你看,文档里写了哦。“可钱已经亏了。

这三个问题的根因是同一个:规范和人之间,隔着一层需要人主动去读、主动去遵守的"自觉”。而自觉是最不可靠的工程依赖。

我们的做派:规则、技能、hook、清单

于是我们把规范拆成了四类东西,全部塞进项目仓库里,跟着代码一起版本管理。

一是规则文件

顶层有一份总规则,每个 agent 启动都加载,里面带一套启动检查清单——它开机第一件事就是照着清单把自己捋一遍。

下面还有一批按文件类型自动触发的规则:

  • 标了 alwaysApply 的永远生效
  • 标了 globs 的只在碰到对应文件时才激活

改到某个 SQL 文件,相关的多租户规则自动浮上来,不用人提醒。

二是「技能」库

我们攒了 90 多个按需加载的技能,每个都是一段"遇到这种活儿该怎么干"的可执行知识。

agent 不用一次性把所有规范塞进上下文,用到哪个加载哪个。

三是运行时 hook

这是最硬的一层。一些危险操作——比如某些会清历史的 git 命令——直接在 hook 里拦下,你哪怕手抖敲错了也不让过。

规范在这里不再是"建议”,是物理上的不允许。

四是启动检查清单

新人也好、新 agent 也好,开机先过一遍清单,确认自己处在"知道规矩"的状态,再开始干活。

当有人说"记住这个",它怎么自动落盘

这套体系最值得讲的一个特性,是它能自我生长。

开发过程中,经常会出现这种对话:“这个以后别再犯了”、“这个规矩记住”。传统做法是,人嘴上说说,或者补一条 wiki——然后照旧没人看。

我们的做法是,当用户说"记住这个 / 别再犯"的时候,这条经验会被自动落盘成一条新规则,进到规则库里,从此对人和对所有 agent 都生效。

下次再有人(或 agent)想犯同样的错,直接被拦。

这其实把规范从"一次性写作"变成了持续沉积:每踩一个坑,沉淀一条规则;规则越攒越厚,项目越来越聪明。

这一点和我们另一篇事故即规范的思路是一脉相承的。

效果:磨合期被压扁了

这套东西真正跑起来之后,最直观的变化是:新人和新 agent 都不用再问人了,开机就能按规矩干活。

以前一个新人上手,得老同事带两周,口口相传一堆"我们这儿的规矩"。现在新人拉下仓库,规则、技能、hook 跟着代码一起到手。

他打开编辑器的第一分钟,规范就已经在工作了——该提示的提示,该拦截的拦截。磨合期被大幅压缩。

对 agent 更是如此。我们这条AI 研发工作流里会开大量并行 subagent,每开一个都是"零背景新人"。

正因为规范是机器可执行的,这些 agent 一落地就知道边界在哪,不用我们一个个去叮嘱。我们的主力是 Claude Code,其次是 Cursor、OpenCode——它们全部走同一套规则加载机制。

关键洞察:这份规范同时服务人和 AI

也是这几年最大的体会:

这份规范不是为了 AI 才写的,它是为了"任何刚加入项目的人或 agent"写的。

AI 不过是把"新成员加入项目"这件事的频率,从一年几次拉升到一天几十次,逼着你不得不把规范做成可执行的。

换句话说,AI 没有改变规范的形态,它只是把规范的价值放大了:

  • 你越是把规范固化进代码底层,就越能驾驭高频的、并行的新成员涌入
  • 你越是依赖 wiki 和口口相传,就越被这件事拖住手脚

所以回到开头那句话:能自己执行的才是规范。把规矩写进代码底层,让新人和 agent 都"开机即用",这是我们这五年在工程规范上做过的、ROI 最高的一件事。
