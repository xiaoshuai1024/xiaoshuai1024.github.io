我见过太多份「项目规范」,它们最后都死在同一个地方:某个 wiki 的角落里,落满灰尘,最后一次更新是半年前,内容早就和代码对不上了。

新人入职第一周没人看,三个月后也没人看,直到踩了坑,有人恍然大悟"哦这事儿其实文档里写过"——可那又怎样,文档不会自己跳出来拦你。

不能自己执行的规范,就不是规范,是建议。这篇回答一个具体问题:项目规范怎么写,才不会变成一堆没人看的废话——答案一句话,把它从「给人看的文档」改成「机器能执行的代码」。展开讲,就是我们在仓库里固化的四层规范:规则文件、技能库、运行时 hook、启动清单。

我见过太多份「项目规范」,它们最后都死在同一个地方:某个 wiki 的角落里,落满灰尘,最后一次更新是半年前,内容早就和代码对不上了。

不能自己执行的规范,就不是规范,是建议。建议这种东西,在人忙的时候、在 AI 自由发挥的时候,一律等于没有。

这个判断不是我一开始就有的,是被逼出来的。项目跑起来后,我开了大量并行 Agent,每个都是「零背景新人」。最初我用传统思路写规范文档,结果 Agent 全程不看,该怎么犯还是怎么犯。直到我把规范写成 hook、写成规则文件,Agent 才真的「守规矩」了。这个转折让我彻底想明白:规范必须可执行,否则在 AI 时代连人这关都过不了,更别说 Agent。

所以我们的做法是——把项目规范写进代码底层,让它机器可读、运行时可执行。这一篇就讲我们怎么做的,以及它怎么让新人和新 Agent 都做到"开机即用"。

新人最痛的不是技术,是"这事儿在我们这该怎么干"

先说我们到底在解决什么问题。

新人入职一个项目,真正卡住他的,通常不是某个框架不会用、某个语法没见过——那些查文档就行。

卡住他的是一堆项目专属的隐性规矩:

  • 这个字段为什么要这么命名
  • 那个 SQL 为什么必须带 tenant_id
  • 提交前得先跑哪个脚本
  • 危险操作有哪些不能碰

这些东西没写在任何教程里,只在老同事的脑子里。

更关键的是,这种隐性知识的传递成本极高。每来一个新人,老人就得口述一遍;老人忙了,新人就只能靠踩坑自学。

而且踩坑自学有个恶果:学到的东西是「碎片化」的。新人踩了 A 坑学到 A 规矩,踩了 B 坑学到 B 规矩,但他不知道还有 C、D、E 规矩存在——直到他也踩了 C、D、E 坑。这种「靠事故驱动学习」的方式,代价是真实的生产事故。规范固化后,新人不用踩坑就能知道所有规矩,因为规矩在代码里、在 hook 里、在技能里,系统性地摆在面前。

AGI 时代又多了个新麻烦——每开一个新的 agent,它也是一个"零背景新人",而且它不会主动来问你。

Agent 比新人更难管的一点是:新人踩坑会有痛感(被骂、被回滚、要背锅),他会长记性;Agent 没有痛感,这次踩了坑,下次新开的 Agent 还是会踩同一个坑,因为它是无状态的、不复用经验的。所以面向 Agent,「踩坑 → 长记性」这条路根本走不通,必须把记性固化成机器可读的规则,让每个新 Agent 启动就继承。

一旦你把"AI 当随时入职的新人"来 onboard,立刻就能看清:这事儿光靠文档不够,得靠机器能自动加载和执行的东西。

传统规范为什么不行

传统的规范有三个硬伤,改不掉的那种。

第一,没人看。 wiki 的链接藏在 IM 群置顶里,大家点一次就忘。

第二,会过时。 代码三个月一迭代,文档没人同步。文档说"提交前跑 A 脚本",实际早就换成 B 了,照着做反而出错。

第三,不可执行。 就算文档全对,它也是一段给人看的文字,不会在你犯错那一刻拦下你。它只能事后当呈堂证供:“你看,文档里写了哦。“可钱已经亏了。

不可执行是最致命的。前两个问题(没人看、会过时)还有补救空间——多提醒几次、定期更新,多少能缓解。但不可执行是结构性的:文字就是文字,它没有「手」去拦你。一个新人写了不带 tenant_id 的 SQL,文档不会跳出来报错;一个 Agent 用了浮点算金额,文档不会编译失败。规范要真正生效,它得有「拦」的能力,而文字天生没有这个能力,只有代码和配置才有。

这三个问题的根因是同一个:规范和人之间,隔着一层需要人主动去读、主动去遵守的"自觉”。而自觉是最不可靠的工程依赖。

传统规范还有一个问题没说透:它是给「人」设计的,不是给「Agent」设计的。人至少会主动翻文档、会问同事,Agent 不会——Agent 启动后直接干活,它不会去 IM 群里问「这事儿我们这怎么干」。所以一旦引入 Agent,传统规范的失效会被放大:人还有 50% 的概率看文档,Agent 是 0%。这就是为什么 AI 时代规范必须做成机器可执行的——不是锦上添花,是从 0 到 1 的刚需。

也有人会说,那 wiki 加个机器人提醒不就行了。试过,没用。IM 机器人提醒的是「有这份文档」,不是「现在该遵守这条规矩」。文档和「当下要做的操作」之间永远隔着一层「人得去读、读了得记、记了得照做」的链路,这层链路只要有一环掉,规范就失效。机器可执行的规范砍掉了这整条链路——不需要你读、不需要你记,犯错那一刻它直接拦。

我们的做派:规则、技能、hook、清单

于是我们把规范拆成了四类东西,全部塞进项目仓库里,跟着代码一起版本管理。

为什么塞进仓库而不是单独的规范平台。因为规范和代码必须同生命周期——代码改了规范要跟着改,规范改了要立刻对所有人(和 Agent)生效。放进仓库,规范就跟代码一起进版本管理、一起 review、一起上线,不存在「代码更新了规范还在旧版」的脱节。这是最关键的设计决策:规范不是独立于代码的「外部文档」,是代码仓库的一等公民。

四层规范从软到硬,越往下越不可绕过:

这四层的「硬度」是递增的。规则文件最软,它是「提示」——Agent 看到了会遵守,但理论上可以忽略;技能库次软,它是「按需参考」——Agent 不调用就不生效;hook 最硬,它是「物理拦截」——你就算想绕也绕不过,因为提交动作本身就被拦住了。这种从软到硬的分层,对应了规范的不同强度需求:大部分规范靠提示就够(规则),具体方法靠按需查阅(技能),不可挽回的操作必须硬拦(hook)。

一是规则文件

顶层有一份总规则,每个 agent 启动都加载,里面带一套启动检查清单——它开机第一件事就是照着清单把自己捋一遍。

下面还有一批按文件类型自动触发的规则:

  • 标了 alwaysApply 的永远生效
  • 标了 globs 的只在碰到对应文件时才激活

改到某个 SQL 文件,相关的多租户规则自动浮上来,不用人提醒。

具体的规则文件长这样:

# .cursor/rules/multi-tenant-sql.mdc(按文件类型触发)
---
description: 多租户 SQL 必须带 tenant_id 过滤
globs: "**/*Mapper.xml,**/*.sql"
alwaysApply: false
---

# 多租户 SQL 规则
任何涉及业务表的查询,JOIN 和 WHERE 必须带 `tenant_id` 条件。
漏掉 = 跨租户越权,安全级事故。

## 检查清单
- [ ] 所有 JOIN 是否都带了 tenant_id?
- [ ] 子查询是否带了 tenant_id?
- [ ] DELETE / UPDATE 是否带了 tenant_id(防止误删全表)?

## 反例
```sql
-- ❌ 漏 tenant_id,跨租户越权
SELECT * FROM orders o JOIN order_item oi ON o.id = oi.order_id;

-- ✅ 正确:两个表都带 tenant_id
SELECT * FROM orders o JOIN order_item oi ON o.id = oi.order_id
  AND o.tenant_id = oi.tenant_id
WHERE o.tenant_id = #{tenantId};

这份规则**只在 Agent 触碰 Mapper.xml 或 .sql 文件时才加载**——不占常驻上下文,但碰到了就一定生效。

按文件类型触发的好处是「精准」。如果所有规则都常驻加载,上下文会被占满,Agent 反而记不住重点。globs 触发让规则「在该出现的时候出现」——写 SQL 时多租户规则浮现,写 Java 时分层规则浮现,各管各的。这套机制让 90 多条规则不会把 Agent 撑爆,因为同一时刻活跃的只有跟当前文件相关的几条。

除了多租户 SQL,我们还固化了几条高频规则。资金计算规则(`globs: "**/*Service.java"`)标了「金额用分、比例用基点,禁止浮点」;Flyway 迁移规则(`globs: "**/db/migration/**"`)标了「版本号用时间戳、必须带 tenant_id、禁止 MySQL 8 语法」;前端组件规则(`globs: "**/*.vue"`)标了「组件必须有 props 类型声明、禁止用 any」。每条都是踩过坑立的,不是拍脑袋写的。

### 二是「技能」库

我们攒了 90 多个按需加载的技能,每个都是一段"遇到这种活儿该怎么干"的可执行知识。

agent 不用一次性把所有规范塞进上下文,**用到哪个加载哪个**。

技能和规则的区别要说清楚。规则是「不能做什么」(禁止浮点、必须带 tenant_id),偏约束;技能是「该怎么做某类事」(怎么建多租户表、怎么写一个 CRUD 模块、怎么做分润状态机),偏方法。规则拦错误,技能教方法,两者配合:技能告诉 Agent 正确的姿势,规则拦住它犯错的姿势。

90 多个技能不是一开始就有的,是项目跑下来一个个沉淀的。最早只有几个核心技能(建表、CRUD 生成、测试生成),后来每解决一类重复问题就固化一个技能——多租户表怎么建、分润状态机怎么设计、迁移脚本怎么写、前端表单怎么统一校验。技能库是项目经验的沉淀池,越用越厚,新 Agent 能调用的「项目智慧」就越多。

技能文件示例:

```markdown
# .agents/skills/multi-tenant-table-setup/SKILL.md
---
name: multi-tenant-table-setup
description: 新建多租户业务表时的检查清单和 DDL 模板
---

# 新建多租户表检查清单
新建业务表时,必须:
1. 带 `tenant_id` 字段(bigint, not null)
2. 在 tenant_id + 业务键上建联合唯一索引
3. Flyway 迁移脚本带 seed 数据
4. Mapper 的所有方法默认按 tenant_id 过滤

## DDL 模板
```sql
CREATE TABLE IF NOT EXISTS {table_name} (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    tenant_id BIGINT NOT NULL COMMENT '租户ID',
    -- 业务字段
    created_at DATETIME NOT NULL,
    deleted_at DATETIME COMMENT '软删时间',
    INDEX idx_tenant_{biz} (tenant_id, {biz_key})
);

### 三是运行时 hook

这是最硬的一层。一些危险操作——比如某些会清历史的 git 命令——直接在 hook 里拦下,你哪怕手抖敲错了也不让过。

```bash
# .git/hooks/pre-commit(拦截危险命令)
#!/bin/bash
# 拦截危险的 git 操作
diff=$(git diff --cached --name-only)

# 1. 防止提交 .env 类敏感文件
if echo "$diff" | grep -qE '\.(env|pem|key)$'; then
  echo "❌ 禁止提交敏感文件"
  exit 1
fi

# 2. 防止提交超大文件(>1MB)
for f in $diff; do
  if [ -f "$f" ]; then
    size=$(stat -f%z "$f" 2>/dev/null || stat -c%s "$f" 2>/dev/null)
    if [ "$size" -gt 1048576 ]; then
      echo "❌ $f 超过 1MB,禁止提交(走 Git LFS)"
      exit 1
    fi
  fi
done

规范在这里不再是"建议”,是物理上的不允许。

hook 这层有几个设计原则。第一,hook 只拦「不可挽回」的操作——提交敏感文件、提交超大文件、提交带调试日志的代码。可挽回的操作(比如代码风格)不放进 hook,因为 hook 失败会阻塞提交流程,放太多会让开发节奏卡死。第二,hook 要快,pre-commit 如果跑超过 5 秒,开发就会想办法绕过它(比如 --no-verify)。所以我们只放文件名检查和大小检查这种瞬时操作,重的检查(测试、lint)放到构建期或 CI。第三,hook 的错误信息要明确——告诉用户「为什么拦、怎么改」,而不是一个光秃秃的 exit 1。

除了 pre-commit,还有 pre-push hook 拦更危险的操作:比如直接 push 到 main/master 分支(强制要求走 PR)、删除远程分支前确认。这些是「一旦做了影响所有人」的操作,必须在 push 那一刻拦住。

四是启动检查清单

新人也好、新 agent 也好,开机先过一遍清单,确认自己处在"知道规矩"的状态,再开始干活。

启动清单对 Agent 尤其重要。Agent 是无状态的,每次启动都是「新来的」,如果不强制过一遍清单,它可能用通用知识干活(比如默认金额用 double),而不是用项目规范(金额用 long)。启动清单强制它在干活前先读 CLAUDE.md / AGENTS.md、加载规则文件、确认知道 tenant_id 规范。这一步把「不知道规矩」从默认状态变成了不可能状态。

当有人说"记住这个",它怎么自动落盘

这套体系最值得讲的一个特性,是它能自我生长。

开发过程中,经常会出现这种对话:“这个以后别再犯了”、“这个规矩记住”。传统做法是,人嘴上说说,或者补一条 wiki——然后照旧没人看。

我们的做法是,当用户说"记住这个 / 别再犯"的时候,这条经验会被自动落盘成一条新规则,进到规则库里,从此对人和对所有 agent 都生效。

下次再有人(或 agent)想犯同样的错,直接被拦。

落盘的流程值得展开。当一句「记住这个」被触发后,系统会先判断这条经验的类型——是约束(规则)还是方法(技能)。约束类(「以后金额别用 double」)落成 .cursor/rules/ 下的规则文件,带 globs 指定触发场景;方法类(「建多租户表要带这几个字段」)落成 .agents/skills/ 下的技能文件,带 description 方便按需检索。

落盘后还要做一件事:给新规则起一个能自解释的文件名。文件名要让人(和 Agent)一眼看出这条规则管什么,比如 money-no-float.mdc、tenant-id-required.mdc。文件名本身就是文档,比内容里的描述更先被看到。这一步看似小事,但规则多了之后(我们 90 多条),文件名的可读性决定了能不能快速找到该改哪条。

这其实把规范从"一次性写作"变成了持续沉积:每踩一个坑,沉淀一条规则;规则越攒越厚,项目越来越聪明。

这套「踩坑 → 沉淀」的循环有个关键:沉淀要及时。踩了坑当场就落盘,不能「等有空再补」——等有空通常就是永远不补。我们定的规矩是:事故复盘会的产出物之一,就是一条新规则或新技能,当场写、当场进仓库。复盘不开完不算完,规则没落盘的复盘等于没复盘。

这一点和我们另一篇事故即规范的思路是一脉相承的。

效果:磨合期被压扁了

这套东西真正跑起来之后,最直观的变化是:新人和新 agent 都不用再问人了,开机就能按规矩干活。

以前一个新人上手,得老同事带两周,口口相传一堆"我们这儿的规矩"。现在新人拉下仓库,规则、技能、hook 跟着代码一起到手。

他打开编辑器的第一分钟,规范就已经在工作了——该提示的提示,该拦截的拦截。磨合期被大幅压缩。

压缩到什么程度。拿我们的数据说话:新人(或新 Agent)从拉下仓库到产出第一个能合并的 PR,过去大概要两周(边问边学规矩),现在平均一到两天。省掉的不是「学技术」的时间(技术该学还得学),是「学这儿的规矩」的时间——这部分被规则和技能包圆了。

更重要的是,产出的质量从「磨合期通常很差」变成了「磨合期也不差」。因为规范是机器执行的,新人没有「忘了规矩」的机会——他想犯的错,hook 拦住了;他不知道的写法,技能教了;他漏掉的检查,规则提示了。规范不是靠他「记住」,是靠系统「兜住」。

对 agent 更是如此。我们这条AI 研发工作流里会开大量并行 subagent,每开一个都是"零背景新人"。

正因为规范是机器可执行的,这些 agent 一落地就知道边界在哪,不用我们一个个去叮嘱。我们的主力是 Claude Code,其次是 Cursor、OpenCode——它们全部走同一套规则加载机制。

对 Agent 来说,这套规范的收益更夸张。一个没有规范固化的项目,每开一个新 Agent 都要人手动叮嘱「别用浮点」「记得带 tenant_id」——开 20 个 Agent 叮嘱 20 次,人会被烦死,而且一定会漏。规范固化后,这些叮嘱变成 Agent 启动时自动加载的规则,20 个 Agent 各自读各自的规矩,人完全不用管。这就是为什么我们能同时开几十个并行 Agent 而不乱——规范是它们共同的「入职培训」,而且是自动完成的。

还有个隐性收益:规范固化让 Agent 的产出一致性大幅提高。没有规范时,20 个 Agent 写出的代码 20 种风格;规范固化后,20 个 Agent 都基于同一套技能和规则,产出的代码像一个人写的。代码一致性这种「软质量」,靠人盯是盯不出来的,靠规范的模板化天然达成。

关键洞察:这份规范同时服务人和 AI

也是这几年最大的体会:

这份规范不是为了 AI 才写的,它是为了"任何刚加入项目的人或 agent"写的。

这句话值得再强调一遍。很多团队把「给 AI 写规范」当成一件额外的事,觉得「我先把手下的人管好,AI 的规范以后再说」。这个想法在 AI 渗透率低的时候还成立,但现在 Agent 一天能开几十个,每个都是「新人」,不给它们规范就等于让几十个不带脑子的实习生同时改你的代码库。给 Agent 写规范不是额外负担,是把「频繁 onboard 新成员」这件事从成本变成自动化的必然选择。

而且,给 Agent 写的规范,对人也一样有用。新人入职,加载同样的规则文件、同样的技能库——他享受到的「入职培训」和 Agent 一模一样,都是机器可执行的。一份规范同时服务人和 AI,边际成本为零。这是规范固化最大的杠杆:投入一次,人 AI 双吃。

AI 不过是把"新成员加入项目"这件事的频率,从一年几次拉升到一天几十次,逼着你不得不把规范做成可执行的。

这个频率变化是质变。一年入职几次,靠人带还能勉强应付;一天涌入几十个新 Agent,靠人带根本不可能——人会被 onboarding 这件事彻底淹没,啥活都干不了。规范固化是唯一能让这种高频涌入不失控的办法,它是 AI 时代项目治理的刚需,不是可选项。

换句话说,AI 没有改变规范的形态,它只是把规范的价值放大了:

  • 你越是把规范固化进代码底层,就越能驾驭高频的、并行的新成员涌入
  • 你越是依赖 wiki 和口口相传,就越被这件事拖住手脚

再往深一层想,这套做法改变的是「规范的所有权」。传统规范的所有权在「写文档的人」手里——文档写得再好,执行靠每个人自觉,等于所有人都是规范的「执行者」,但没人能保证执行。固化进代码后,规范的所有权回到了「系统」手里——系统是唯一的执行者,人是规则的制定者。制定和执行分离,规范就从「靠自觉」变成了「靠系统」,这是质变。

这套做法也不是没有成本。维护 90 多条规则和技能要花精力,规则之间偶尔会冲突(比如一条说「字段用驼峰」,另一条说「数据库字段用下划线」),冲突了要梳理。但这个维护成本远低于「不维护规范、靠事故驱动学习」的代价。规范固化是前期投入、长期收益的事,ROI 会随着项目和团队规模增长越来越高,人越多、Agent 越多,这套机制的摊薄效应越明显。

所以回到开头那句话:能自己执行的才是规范。把规矩写进代码底层,让新人和 agent 都"开机即用",这是我们这几年在工程规范上做过的、ROI 最高的一件事,没有之一。它的价值会随着团队和 Agent 规模一起复利增长,越早投入越早受益。
