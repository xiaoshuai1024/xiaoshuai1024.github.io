荒芜之地长出 CI/CD：没有流水线，三条命令跑通了 CI、CD 和 DevOps

跟同行聊 CI/CD，对方都默认你至少有个 GitLab Runner 或者 GitHub Actions。我接手的一个全栈项目，那段日子什么都没有：没有流水线，没有自动化测试，没有提交规范，连 PR 流程都是口头说一声「我推上去了你看下」。部署是本地 mvn package 出 jar，再 scp 拷到服务器，运营后台的 dist 目录手动传。

按教科书，这种状态就该立刻上 CI 平台。但我做了个反直觉的决定：先不碰平台，用三条 AI 命令把 CI、CD、DevOps 三阶段全跑通了。这三条命令是 /pr-all、/deploy（加 /prod-testing）、/prod-debug——挂在 Claude Code 上，每次发版一条龙走完。

更反直觉的是后面的事。一年后，我们补了一套基于阿里云云效的正规 CI/CD 系统。按常理，正规系统上线，土法命令行该退役了。但我一条都没退，因为这一年跑下来我发现：在「需要判断」的环节上，命令行比系统化流水线更强。云效流水线只会报红等人，命令行能让 AI 自己修完再合并；流水线进不去生产现场，命令行能连生产库调试；流水线没有「停下来问人」这个状态，命令行有。

这篇文章讲的就是这件事：怎么从一无所有搭起三阶段，以及为什么补了正规系统之后，土法命令行反而成了不可替代的那层。先说结论——它不是 CI 系统的替代品，是 CI 系统够不着的那层「判断力」的载体，是流水线进不去、又必须有人盯的那些环节的兜底。

一、荒芜之地：没有 CI 的研发长什么样

先把现状摆清楚，不美化。

那时候改完代码，流程是这样的：自己机器上 mvn clean package 编译，过就推到远端；没单测、没集成测，过不过全靠手点页面，点过就算过；评审是口头一句「帮我看看那个接口」，没人真看就合了；部署是 scp jar 包到生产服务器，运营后台的 dist 目录手动传；回滚靠记忆——出问题翻聊天记录找「上次能跑的那个 jar 在哪」，没有制品库，没有版本号。

举个真实的场景。有一次改了佣金结算的接口，编译过了、手点了几下没问题就推了。第二天运营来报障，说某个商家的佣金算出来是负数。排查发现是新代码里一个边界条件没处理，但这个 bug 在「手点几下」的阶段根本触发不到——它得某个商家当月刚好有一笔退款，金额恰好等于某笔佣金，才会撞上那个边界。这种 bug，没有系统化的测试覆盖，靠人手点是抓不到的。

更要命的是部署的不透明。jar 包 scp 上去，systemctl restart，进程起来了就觉得「上线成功」。但有一次上线后，新代码其实根本没生效——旧进程没杀干净，还占着端口，新进程绑端口失败静默退出，健康检查命中的是旧进程。我们盯着日志以为一切正常，直到用户开始报障才发现线上跑的还是上一版。

这套流程最伤的不是慢，是不透明。谁改了什么、推上去的是哪个版本、现在线上到底能不能用，全是黑盒。半夜发完版，第二天用户报障，你才知道某个接口昨晚就挂了。

有人会说，这阶段直接上 GitLab CI 不就行了？算过账不划算。我们当时是 3 个人、单机部署、流量可预期，养一套 Runner、写 pipeline YAML、处理证书和并发缓存，投入产出比很差。更要命的是，CI 平台解决的是「自动跑测试」，但我们当时连测试都没有——上了 CI 也不知道让它跑什么。更需要的是一套轻量、能立刻拦住「黑盒」的机制，而不是一套重型平台。

这三条命令就是在这个判断下长出来的：/pr-all 管 CI（评审 + 合并纪律），/deploy 加 /prod-testing 管 CD（部署 + 部署后验证），/prod-debug 管 DevOps（生产排障）。每一条都是被真实痛点逼出来的，不是先有理论再找场景。

这里有个设计上的取舍要说清楚：这三条命令不是「CI 系统的替代品」，是「在还没有 CI 系统的阶段，把 CI/CD/DevOps 三件最关键的事先焊住」。等后面真的上了 CI 平台，它们非但不冲突，还互补——这一点文章第五节会展开。

二、CI 阶段：/pr-all 把评审和合并焊死

CI 这两个字母，很多人理解成「自动跑测试」。但 CI 真正的价值是「持续集成」——让代码能安全地、可重复地汇入主干。测试只是其中一环，评审和合并纪律才是大头。

/pr-all 这条命令，干的就是这件事，分四个阶段：

阶段①，同步 dev 解冲突。 提交之前先 pull-all dev，把 origin/dev 合进当前分支，把冲突消化在本地，避免 MR 在远端打回来。冲突处理有硬规矩：默认保留双方逻辑，逐文件展示冲突内容等用户确认，禁止直接 --ours 或 --theirs 丢弃另一方。

这条规矩防的是合并赶时间、看到冲突随手 --ours 保自己的代码——另一边同改的逻辑被静默吞掉，bug 留到上线才暴露。命令强制每个冲突都得人看一眼、点确认，把这种「随手」挡住。

子模块指针冲突也要单独处理——不能直接选一边，要先确认对应子模块的 MR 合到 dev 了没，再 git add <submodule> 指向 dev 最新提交。指针指错，本地和线上会跑不同版本，这种不一致排查极痛。

阶段②，建 MR 到 dev。 依次处理 frontend → backend → operation-backend → 主仓，每个仓库检测有没有改动、没改动就跳过，有改动就 commit + push + 调 Codeup API 建 MR。

这里有个被坑出来的细节：单仓失败不中断。某个子模块建 MR 失败（远端已有同源 MR、Codeup 返回 400），会打警告继续处理后面的仓库，不会因为 set -e 提前退出把 backend 漏掉——多仓库操作最怕「主仓建了 MR，backend 却被跳过」这种半同步状态。

阶段①②是机械活，云效 CI 也能干。真正让这套东西超越「脚本」的，是阶段③④。

阶段③，@code-reviewer 审查。 这一步派一个只读的 subagent，edit: deny, bash: deny，它只能看不能改。聚焦本次 MR 相对 dev 的 diff，输出一份结构化问题清单，按 🔴必须修 / 🟡建议修 / 🔵提示 三级分级，每条给出文件行号、问题描述、修复建议。

注意它不走 /kd-review 那套 21 bucket 全量审查——那是发布前的重型收敛，太重了。/pr-all 里的审查是轻量的，聚焦本次改动，关注安全、性能、可维护性，几分钟出结果。

为什么审查 agent 必须只读？这是给 AI 自治画的红线。审查的活是「挑刺」，挑刺的人手里不能有改代码的权力。如果审查 agent 同时有改代码的权限，它为了让自己「审查通过」，会顺手把发现的问题粉饰过去——把 🔴 降成 🔵，甚至直接把自己的问题改没了。读写分离，让审查和修复是两双不同的眼睛。这条和 无 CI 多 Agent 评审回路 里「评审 Agent 只读、修复 Agent 只写」是同一个设计哲学。

这个读写分离是 AI 协同能不能信任的前提。传统评审里，评审者和提交者是两个人，天然分离；AI 时代审查和修复可能是同一个 AI，不人为切断就成了「自己审自己改」——和没人审没区别。强制用两个角色（只读 subagent 审、主会话改），等于在 AI 内部造了一道制衡。有没有这道制衡，决定了 AI 协同是「能用」还是「敢用」。

阶段④，自动修复 + 收敛后才合并。 这才是整套 CI 的命门，也是云效流水线绝对做不到的部分。

@code-reviewer 报完问题，主会话（注意，不是那个只读 subagent）接手修。修复的优先级是写死的：

  • 🔴 必须修 → 立即修，没有商量
  • 🟡 建议修 → 原则上也修；只有改动过大、成本明显高于本次需求时，向用户说明后跳过
  • 🔵 提示 → 记录，可不改

每修一个仓库，跑对应的构建测试确认没回归——backend 跑 mvn verify，frontend 跑 pnpm test && build:mp-weixin，operation 跑 pnpm test && build。修完 git push 更新 MR。这里有个硬规矩：修完不会立刻合并——要 🔴 全清零、🟡 都处理掉（修了或经人同意跳过）、构建测试都过，才执行 merge-all-to-dev。

这一步还有个收敛循环。修完一遍，可能又引入新问题，于是回阶段③重审，直到连续收敛没有新 🔴。没收敛完，禁止合并。这个循环的价值在于：它不指望一次改对，但保证最终合进 dev 的那个版本，是审查过的、没有 🔴 的、测试过的。

把这套和云效 CI 对比一下，差别就清楚了。云效流水线 push 上去跑 mvn verify，红了就挂，挂了发个通知等人来处理。人来了，打开日志看哪挂了，手动改代码，手动重新 push，再等流水线跑一轮。整个「发现问题→修→验证」的循环，串行依赖人，而且每一轮的延迟以小时计——因为人不可能盯着流水线等。

/pr-all 把这个循环塞进了命令里：发现问题的是只读 agent，修问题的是主会话，验证用的是同一套构建测试，合并是收敛条件触发自动执行的。人只在「🔴 涉及大范围重构」「修复成本明显高于本次需求」或者「修了测试还是挂」这种命令明确点名的高危场景里才介入。云效的 CI 是「发现问题然后等人」，/pr-all 的 CI 是「发现问题然后自己修完再合并」——这就是「超越系统化」的第一个实锤。

补充一个数字感受。云效 CI 挂一次，从通知到人介入到修完重推，平均一到两个小时，因为人得切上下文、看日志、定位、改。/pr-all 的收敛循环里，一个 🔴 从发现到修完验证，是几分钟——因为不需要等人切上下文。一天积累十几次小改动，这个时间差就是几个小时的人力。

举个 🔴 修复的真实例子。有次改动里漏了一个空指针检查——某个返回值在特定路径下是 null，代码没判空直接调方法。@code-reviewer 在阶段③报了 🔴：标注文件行号、说明「此处 getXXX() 在租户未配置时返回 null，未判空直接调用」、给了修复建议「加 null 检查，为空时走默认分支」。主会话接到这条，改代码、跑 mvn verify 验证、push 更新 MR，整个循环不到三分钟。如果走云效 CI，这个 🔴 要等到 mvn verify 真的抛 NPE 才暴露（还得是测试覆盖了那条路径），然后发通知、等人、人改、重推、再等流水线——最快也得半小时。

差别在哪？@code-reviewer 是静态分析式的审查，它读代码就能发现「这里没判空」，不用等测试跑出来。云效 CI 只能等运行时炸了才知道。前者是预防，后者是事后。AI 协同的价值，就是把一部分「事后发现」提前成了「事前预防」。

这条命令的完整规则（冲突处理、收敛判定、何时停下问人）在 30 条 AI 命令锁死研发纪律 里有更细的展开。这篇只讲它怎么对应 CI 阶段。

三、CD 阶段：/deploy + /prod-testing 部署即验证

CD 是「持续交付」——把通过 CI 的代码，安全地、可重复地送到线上。这一阶段的两条命令，一条管「送」，一条管「送完确认能用」。

/deploy-prod-be-ops 六步，每一步都是被真实坑逼出来的。先看整体流程，再逐个拆坑：

Step 1-2 构建后端 jar 和运营后台 dist。 后端 mvn clean package，注意用的是 -Dmaven.test.skip=true 而不是 -DskipTests。这两个参数看着差不多，行为完全不同：-DskipTests 跳过测试执行但仍然编译测试代码；-Dmaven.test.skip=true 连测试编译都跳过。

为什么要跳过测试编译？因为 dev 合并进来后，测试代码里调用的某个 API 签名可能已经被改了——比如某个 service 方法加了参数，主代码改了但对应的测试还没跟上。这种情况下 mvn package 跑到测试编译阶段就挂，部署根本进行不下去。生产构建要的是「主代码能编出 jar」，测试留到 /pr-all 那条带验证的链路里去跑。这不是偷懒，是职责分离：部署只管「把能跑的 jar 送上去」，测试在 CI 阶段就该拦住。

Step 3-4 上传。 jar 直接 scp 上去，运营后台的 dist 先 tar 打包再传。这里有个最坑的细节：运营后台的 nginx root 是 /www/wwwroot/operation-backend/，不是 /dist/ 子目录。

最自然的写法是 scp -r dist 把整个 dist 目录传上去。结果呢？nginx 读的是根目录的旧文件，部署不生效，你以为上线了其实线上还是老版本。这种坑不报错、不失败，没有任何提示，只会让你半夜发版后盯着屏幕疑惑「为什么改的样式没生效」。

我们踩过一次，排查了大半个小时。最后发现 nginx root 配的是 /www/wwwroot/operation-backend/，而 scp -r dist 把文件传到了 /www/wwwroot/operation-backend/dist/——nginx 根本不读那个子目录。修法是先 rm -rf assets index.html（清掉根目录旧文件）再 tar xzf 解压到根目录。Mac 用户还要额外清一波 ._* 文件，那是 macOS tar 产生的 Apple 元数据，不清会让线上多出一堆垃圾文件，偶尔还会干扰 nginx 的 mime 类型判断。

Step 5 重启后端。 这一步最隐蔽的坑是 pkill 自杀。重启要先杀掉旧进程，最直觉的写法是 pkill -f backend-0.0.1-SNAPSHOT.jar。

但这条命令通过 ssh 远程执行时，远程 shell 的 -c 参数里就包含 backend 字符串，pkill 会匹配到 ssh 自己的进程把自己杀掉，连接断了、重启步骤全没跑。更隐蔽的是 pgrep -fc 计数会因此虚高，导致健康检查误判。

修法是老掉牙的 Linux 技巧：pkill -9 -f '[b]ackend-0.0.1-SNAPSHOT.jar'。[b]ackend 正则匹配的还是 backend，但 pkill 看到的字面量是 [b]ackend，不含 backend 子串，就不会匹配到自己。没踩过的人想破头也想不出为什么加个方括号就能好。

Step 6 验证日志无 ERROR。 重启之后还有一道硬验证，四条缺一不可：systemctl 服务 active + 健康检查返回 200 + 只有 1 个 backend 进程 + 进程是 systemd 起的（不是手动 nohup）。

为什么要查四条而不是只看健康检查？只看健康检查会漏掉「新旧并存」的幽灵状态：旧进程占着端口、新进程绑端口失败静默退出，健康检查命中的是旧进程，表面「服务 UP」其实线上还是上一版。查进程数和启动方式，就是为揪出这种情况。

部署到这就算「动作完成」。但 编译能过 ≠ 代码正确 那篇的事故告诉我们：编译绿、构建成功、服务起来——这三件事都不代表线上真能用。所以部署完必跑 /prod-testing，这是发版的第二只手。

它分三层，覆盖范围由窄到宽：

  • Layer 1 API 冒烟：22 个核心接口，首页、搜索、结算、提现、对账。接口挂了这层 3 分钟就能抓到，quick 模式只跑这层。
  • Layer 2 C 端 H5 页面：首页、商家结算、佣金明细、达人提现。生产没有 H5（C 端是小程序），本地起 H5 指向生产 API 跑。
  • Layer 3 运营后台页面：结算、对账、圈子、套卡管理。full 模式跑完三层约 12 分钟。

为什么分三层、还要分 quick 和 full？因为冒烟的颗粒度得和「能不能发现问题」对齐。API 挂了 Layer 1 三分钟就能确认；但页面渲染崩了、某个表单报错，Layer 1 测不到，必须跑到 Layer 3。发版赶时间至少跑 quick 拦住最致命的「服务挂了」；大改动之后 full 不能省。

盲区要诚实承认：冒烟测的是「主路径能走通」，测不了「边界条件对不对」——阶梯临界点的佣金算得对不对、并发场景下数据会不会错乱，得靠单元测试和集成测试在部署前守住。冒烟是兜底，不是全证。把它当全证，会漏掉「能跑但算错」的 bug，这种 bug 比「直接报错」的更危险。

这里有个反直觉的设计：验证脚本默认不自动清理测试数据。很多人觉得「自动化跑完该自己收拾干净」，但在生产库里，自动清理是高危动作。

测试脚本创建的临时数据，大部分随 token 过期会自己失效——测试用户登录拿的 token，几天后过期了，关联的临时记录就成了孤儿数据，不影响业务。但有一类数据必须手动清：sys_user 表里注册的测试账号。如果让脚本去 DELETE FROM sys_user WHERE username LIKE 'test_%' 自动清，等于把删生产数据的权力交给一个自动化流程。

这个风险不是理论上的。WHERE 条件写错一个字符、时区算错导致删多了、测试账号命名规则变了但脚本没跟上——任何一个失误，都是删库级的事故。生产库里「写」这个动作的破坏面远大于收益，宁可事后手动清那一两个测试账号，也不把 DELETE 权力交给脚本。这是用「麻烦」换「安全」的典型取舍。

冒烟这一关告诉我们一句话：「部署成功」和「线上可用」是两件事。前者是动作完成，后者是结果对账。中间那道验证谁也不能省——云效流水线要么不跑这道验证，要么得另接一套，而命令行是把它们焊死在一起的。

六步部署的坑（nginx root、pkill 自杀、systemd 四重验证）在 单机部署 + 夜间 AI Loop 里有完整的踩坑复盘。这篇只讲它怎么对应 CD 阶段。

四、DevOps 阶段：/prod-debug 连生产排查

发版完成、冒烟过了，线上还是会出问题——慢查询、偶发 500、某个租户数据异常。这时候最痛的不是「问题难找」，是「想调试却不敢动」。

痛点很具体：要复现一个生产 bug，你得连生产库；但本地服务一连生产库，就处在「一不小心就写脏数据」的危险边缘。一次手滑的 UPDATE、一个忘记加 WHERE 的 DELETE，就是事故。更麻烦的是，本地起的 Spring Boot 连生产库，Flyway 可能自动跑迁移——如果本地有一个生产还没上的迁移文件，它一启动就给生产库加了一张表，排查没开始，现场先被自己破坏了。

/prod-debug 这条命令，用一套严格的纪律把生产调试变得安全：

# 后端以 prod profile 启动，连生产 MySQL/Redis
cd backend && make prod
# 前端/运营后台 API 指向生产域名
API_BASE_ORIGIN="https://api.your-prod-domain.com"

技术上是切 profile + 切域名，但真正起作用的是三条铁律，不是切换本身：

  • 不修改任何业务代码。 排障归排障，代码一行不改；要改走正常的开发流程（从 /pr-all 重走一遍 CI）。这条防的是「调试时顺手改一行业务代码，想着等下补回正式流程」，结果调试一深入就把这茬忘了，未经验证的改动混进了生产调试。
  • 不创建数据库迁移。 Flyway 在生产是 out-of-order: false，本地起一个迁移会和线上版本冲突，排查反而被自己制造的冲突带偏。要排查 schema 相关问题，只读不写。
  • 不触发 CD / 不重启线上服务。 本地调试，线上服务纹丝不动。重启线上服务要走 /deploy 那条有验证的链路，不能在生产排障时顺手重启。

这三条把「调试」死死锁在只读范围里。健康检查 120 秒内返回 200 才算就绪，期间盯着启动日志诊断常见问题：MySQL/Redis 连接失败、Flyway 冲突、端口被占用、微信支付配置缺失。每个常见问题命令里都写了对应的排查方向，AI 拿到日志能直接定位，不用人去翻历史踩坑。

这一步是云效流水线绝对做不到的。流水线是单向的——代码从仓库流向生产，它不能反向连生产服务器去排查一个已经发生的问题。流水线挂了你只能看它给的日志，没法深入到「这个接口在生产数据下到底返回什么」「这个慢查询在生产库的执行计划长什么样」。

/prod-debug 能做到，因为它跑在开发者的本地机器上，用 prod profile 连真实生产库。开发者拿到的是一个和生产一模一样的运行现场：同样的数据、同样的 Redis 缓存、同样的配置。生产偶发的那个 bug，在本地能稳定复现，能打断点、能加日志、能一步一步跟。这种「深入现场」的能力，是任何流水线都给不了的——流水线只能告诉你「挂了」，不能让你「走进去看」。

还有一点是流水线给不了的：高危操作停下问人。命令里明确写了，遇到 🔴/🟡 问题涉及大范围重构、架构改动，或者修复成本明显高于本次需求本身，停下来汇报，不擅自扩大改动。

举个具体场景。某次排查发现，一个生产 bug 的根因是某个多租户隔离的实现有问题——这是个架构级的改动，不是改两行能解决的。命令不会自己决定「那我重构一下这块吧」，它会停下来汇报：「根因是 X，但要修需要动 Y 模块的多租户逻辑，改动范围超出本次排查，请你确认」。这就是「停下来问人」的价值。

云效流水线没有「询问」这个中间态——要么配置成自动通过，要么配置成失败阻断，没有「这件事我不确定，你来定」这个状态。流水线要么无脑执行（危险），要么无脑拒绝（卡住），它无法表达「需要人判断」。

这个「询问」状态，是 AI 协同里最值钱的能力。它意味着 AI 不是盲目执行，也不是盲目拒绝，而是知道什么自己不该拍板。这一点在系统化流水线上是缺位的——流水线的每一步都是配置写死的，它没有「不确定」这个概念。

五、后来上了云效 CI/CD，为什么命令行没退

故事到这里本该结束：从一无所有到三阶段打通，命令行工作流立了大功。但后面发生了一件让很多人意外的事——我们补了一套正规的 CI/CD 系统。

.yunxiao/ 目录下有一份云效 Flow 流水线配置，干的事是：push 到 master → mvn clean verify（含 ArchUnit 架构规则 + JaCoCo 覆盖率门禁 + 单测）→ jar 上传到制品仓库（项目代号） → 主机部署到生产 ECS → 健康检查。流水线还带「部署历史」，保留 30 天制品，能一键回滚。

按常理，上了正规 CI/CD，土法命令行该退役了。但我一条都没退。原因不在情怀，在那张能力对照表里：

看这张表会发现一个清晰的分界线：

前三行，命令行独占优势。 这些都是「需要判断」的环节。举几个具体场景：CI 跑完发现一个并发安全隐患——这种问题不是「测试挂了」那么明确，要不要修、修到什么程度、是不是该顺手重构一下这块，都得权衡。云效流水线只会报红，等开发者自己看日志、自己判断、自己改。命令行能让 AI 先审、先改、先验证，人只看收敛后的结果。连生产排查更典型——生产偶发的 bug 只能在生产现场复现，流水线是单向的进不去生产，命令行能切 prod profile 直接连。高危操作问人也一样，重构一个老模块这种事，AI 知道「这不该我拍板」，停下来汇报，流水线没有这个中间态。

后三行，云效占优。 这些都是「需要确定性」的环节。push 到 master 自动触发构建，不依赖开发者机器在线、不依赖人记得按回车——开发者下班走了，半夜有人 hotfix 推了一版，云效照样跑验证。制品仓库留每个版本的 jar，带 SHA256 和构建时间，回滚能精确到某个版本；命令行的 jar 只在本地，传完就没了，回滚靠翻聊天记录。覆盖率门禁和 ArchUnit 架构规则是云效原生跑的，mvn verify 里就带，不用额外配；命令行得自己记得跑。

这个分界线回答了「为什么两套并存」：它们管的是不同性质的事。云效管「机械重复、必须每次一样」的事，命令行管「需要判断、每次可能不一样」的事。硬要二选一，要么失去确定性（退役云效，全靠人按命令，人不在场就断档），要么失去判断力（退役命令行，流水线挂了就等人手动救火，AI 的判断力用不上）。

保留两套的代价是多维护一套命令，但收益远大于此。云效兜住了「半夜没人 push 也不会忘跑测试」「每次都有制品能回滚」「覆盖率不达标就拦下来」这种底线，命令行兜住了「出了问题能立刻深入现场」「审查发现问题能自己修完再合并」「遇到重构该停下来就停下来」这种上限。一个守下限，一个撑上限，谁也替不了谁。

实际工作中这两套是配合的：日常 push 走云效兜底线（门禁、制品、无人值守）；要发版或者排查问题，切到命令行撑上限（带验证的部署、连生产调试、自动修复的评审）。开发者根据场景选工具，不是非此即彼。

举个完整的发版流程看看两套怎么配合。开发者写完一个 feature，先跑 /pr-all——这条命令走阶段①②③④，让 AI 审查、修复、收敛后合并到 dev。合并完，代码进了 dev。接下来 push 到 master 触发云效 CI：跑 mvn verify 过门禁（覆盖率、ArchUnit）、构建 jar、上传制品仓库。这一步是云效兜底——它确认合并进 master 的代码能编译、能过门禁、有制品留档。

到这一步，CI 完成了。要真正上线，跑 /deploy-prod-be-ops——构建、scp、重启、四重验证，把 jar 送到生产。部署完必跑 /prod-testing 三层冒烟，确认线上真能用。如果上线后发现某个接口在生产数据下表现异常，切 /prod-debug 连生产库排查。

整条链路里，云效管的是「确定性的那两步」（门禁、制品），命令行管的是「需要判断的那几步」（自动修复合并、带验证的部署、连生产调试）。两套衔接得很顺，没有任何重复——因为它们管的环节本来就不重叠。

顺带说一句这套做法的适用边界。它最适合的是「中小团队、单机或少量节点部署、流量可预期」的项目——比如 3 到 10 人的 SaaS、内部系统、创业期产品。这种规模下，养一套 K8s + Argo CD 的成本远超收益，而「人随时在场」这个前提又成立，命令行 + AI 的判断力能充分发挥。

如果团队上了规模（几十人、微服务几十个节点）、流量需要弹性伸缩、或者有严格的合规审计要求（金融、医疗），那 CI/CD 平台 + 容器编排是必须的，命令行只适合作为开发者的个人提效工具，不能扛起整套基建。这不是技术优劣，是规模决定形态。

特别说一句诚实的话：「命令行超越系统化」不是全方位超越。在「确定性、可审计、无人值守」这些维度上，云效明显更强——这也是为什么我们补了云效而不是只用命令行。超越只发生在「需要判断」的维度上。标题敢这么写，底气全在那张对照表的前三行，不是空喊口号。

六、系统化是下限，能判断才是上限

把这套做法抽出来，其实是在回答一个问题：AI 时代的研发基建，哪些该交给系统，哪些必须留在人和 AI 手里？

系统化流水线解决的是「机械重复」——每次构建都一样、每次门禁都一样、每次部署都一样。这类事交给机器最合适，人也最不可靠（人总会忘、会累、会手抖）。云效 CI/CD 把这类事守得死死的，这是它的价值，也是它的下限。

但有一类事流水线解决不了，就是「需要判断」的——这段代码该不该这么写、这个修复对不对、要不要为了这个需求重构那块老代码。这类事没有客观标准答案，靠的是经验和权衡。系统化流水线遇到这类事只有一个反应：报红，等人。它不知道怎么判断，也不被允许判断。

/pr-all、/deploy、/prod-debug 这三条命令的价值，不在它们能跑（脚本能跑），在它们把「判断」这件事工程化了：审查用只读 agent 挑问题、修复用主会话改、收敛用客观测试验证、高危用「停下来问人」兜底。AI 承担了大部分判断的活，但关键的拍板权——要不要合并、要不要重构、要不要扩大改动——留给了人。

这套机制是 AI 协同独有的，传统 CI/CD 系统天然没有。传统系统里「判断」要么由人全包（人工 review、人工回滚），要么由配置全包（写死门禁规则）。前者慢、依赖人盯；后者死板、应对不了规则之外的情境。AI 时代多了一个选项：让 AI 做大部分判断，人只在 AI 不确定时介入。这就是「超越系统化」的真正含义——不是技术更先进，是分工更合理。

这个分工里有三层，缺一不可：

  • 机器管确定性（云效 CI/CD）：构建、门禁、制品、无人值守，每次都一样。
  • AI 管大部分判断（三条命令）：审查挑问题、修复验证、收敛合并、连生产排查，AI 在规则内自治。
  • 人管关键拍板：要不要合并、要不要重构、要不要扩大改动——这些 AI 会主动停下来问。

三层叠起来，才是完整的研发基建。只靠机器，应对不了需要判断的环节；只靠 AI，丢了确定性和可审计；全靠人，效率塌方。AI 时代的价值，是把判断这件事从「全靠人」变成「AI 扛大头、人只拍板」。

回头看我那三条命令，它们不是土法炼钢的遗迹，是看清了「哪些该交给系统、哪些必须留在人 + AI 手里」之后的设计。后来补云效不是因为命令行不够好，是因为有些确定性的事机器比人可靠；保留命令行不是因为情怀，是因为有些判断的事机器替不了。

写到这里，我想给读者一个判断的起点。如果你也在一个还没有完整 CI/CD 的项目里挣扎，别急着上重型平台——先问自己两个问题：你们最大的痛点是「黑盒」（谁改了什么、线上能不能用都不知道），还是「人盯」（流程靠人记忆、靠人自觉）？如果是前者，先搭一套带验证的命令化流程，把黑盒焊住；如果是后者，先上 CI 平台，把人盯的事交给机器。两个痛点往往都有，那就两套都要，而且不冲突。

荒芜之地上长出来的不是一套 CI/CD 工具，是一种分工的判断力。工具会换、平台会变、AI 模型会迭代，但这种「机器管确定性、AI 管判断、人管拍板」的分工意识，是 AI 时代研发基建真正该沉淀的东西——它比任何具体工具都活得久。
