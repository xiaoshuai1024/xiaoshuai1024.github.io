上一篇讲数据库地基,这篇讲建在地基上的统计中台。

这个项目有十几个数据大屏:平台总览、交易分析、商家 KPI、连锁统计、运营统计、服务商看板……每个大屏几十个指标,全部要多租户隔离。听起来像该上 ClickHouse、上 Flink、上数仓那一套。

我们没有。用的是 MySQL 汇总表 + Redis 实时计数 + ES 搜索,三层数据源拼装。这篇讲清楚为什么这么选,以及落地时的几个关键判断。

先说清楚统计中台在这个项目里解决什么问题。多租户 SaaS 平台,每个租户(商家)要看自己的经营数据(GMV、订单、用户),平台运营要看全局数据(全平台 GMV、活跃租户、增长趋势),服务商要看自己服务的租户数据。这三类视角,加上时间维度(实时、今日、本周、本月),组合出几十个统计场景。如果每个场景都实时去查原始业务表,生产数据库会扛不住(一个大屏查询可能 GROUP BY 几十万行)。统计中台的存在,就是把这些查询压力从生产表转移到预聚合的汇总表,让「看数据」不打扰「做交易」。

统计中台的设计核心是「分层 + 隔离」。分层,是指数据按新鲜度分三层(实时/小时/T+1),各层用不同技术(Redis/汇总表/定时任务);隔离,是指多租户数据严格按 tenant_id 隔离,统计查询绝不扫生产表。这两个设计原则,贯穿整个统计中台的实现。理解了这两点,后面的技术选型和落地细节就好懂了。

一、为什么没上 ClickHouse

很多人一听「统计大屏」,第一反应是上 OLAP。我们也评估过,最后没上,理由很实在。

数据量没到那个量级。这是多租户 SaaS,不是全民级应用,日订单几十万量级,MySQL 加索引加汇总表完全扛得住。上 ClickHouse 是杀鸡用牛刀,运维成本反而上去了。

团队只有 4 个人。多引一个重型组件,就多一份运维负担。SaaS 创业期,运维复杂度是隐形成本。

实时性要求没那么极致。大屏数字延迟 15 分钟完全可以接受,它本来就是决策辅助,不是交易撮合。

不追求技术先进,追求「刚好够用 + 可控」。这个判断贯穿整个统计中台的设计。

这里要展开讲一下「不上 ClickHouse」的完整算账。ClickHouse 的优势是 OLAP 查询快(亿级数据秒级聚合),代价是运维复杂(分布式集群、数据均衡、版本升级)和学习曲线(它有自己一套 SQL 方言)。对一个日订单几十万的 SaaS,MySQL 加汇总表查询响应在百毫秒级,完全满足大屏需求;上 ClickHouse 查询能快到十毫秒,但用户感知不到这 90 毫秒的差距(大屏刷新本来就有延迟)。而运维成本是实打实的——多一个分布式组件,就多一套监控、备份、故障排查的负担,4 个人的团队根本扛不住。所以「不上 ClickHouse」不是技术落后,是 ROI 算账后的理性选择。

还有一个隐性成本:数据同步。ClickHouse 要从 MySQL 同步数据过去(用 Canal 或 Flink CDC),这套同步链路本身就是一个故障源——同步延迟、同步丢失、schema 变更不同步,每个都可能让大屏数据不准。而 MySQL 汇总表直接在同一个库里,没有跨系统同步问题,数据一致性天然有保障。少一个同步链路,就少一类「数据对不上」的事故。所以不上 ClickHouse 也少了一类故障面。

什么时候该上 ClickHouse。当数据量到亿级、MySQL 汇总表查询慢到秒级以上、或者要做复杂的实时多维分析(不能预聚合)时,ClickHouse 的优势才显现。我们没到这个量级,所以不上;等业务量到那个级别,再迁移也不迟——汇总表的 schema 是可以迁移到 ClickHouse 的,前期用 MySQL 不算走弯路。

二、三层数据新鲜度:这是整个设计的核心

大屏的数据来自三个时间尺度,必须分清楚。

T+1 层是主力。 每天凌晨 00:05 跑 StatisticsCalculationJob,把昨天的 trade_order/trade_refund/tenant_user/tenant_shop/tenant_circle_post 五张表,按租户聚合 upsert 进 stat_daily_summary。

@Scheduled(cron = "0 5 0 * * ?")          // 每日 00:05
public void calculateDailyStats() {
    List<String> tenantIds = jdbc.queryForList(
        "SELECT DISTINCT tenant_id FROM trade_order WHERE DATE(created_at) = ?", ...);
    for (String tid : tenantIds) {
        calculateForTenant(yesterday, tid);
        cacheService.invalidateByTenant(tid);   // 失效该租户缓存
    }
    dao.upsertDailySummaryAggregated(yesterday);  // 最后算平台汇总行
}

为什么最后还要算一遍平台汇总。因为大屏有个「全平台」视角。我们在汇总表里用 tenant_id='0' 表示全平台——它既作 SQL 默认值,又作业务语义。每天最后一步把所有真实租户行 SUM 出一行 tenant_id='0'。

小时层补分时数据。 每小时 :05 跑一次,写入 stat_hourly_trade,支撑大屏的 24 小时热力图。

实时层靠 Redis。 「今日 GMV」这种数字,等明天凌晨才有就不对了。订单支付时直接 redis.opsForValue().increment(gmvKey, amountFen) 实时累加,DAU 用 Set 的 SADD userId 去重。TTL 24 小时,跨天自然过期。

一条铁律,我写进了迁移文件的注释:统计查询全部走汇总表,禁止扫原始业务表。大屏哪怕实时性不够,也不能去 GROUP BY 原始订单表——那是把生产数据库往死里打。

这条铁律为什么这么重要。因为原始业务表(如 trade_order)是生产表,线上交易在不停地写。如果大屏去 SELECT COUNT(*) FROM trade_order WHERE ... GROUP BY ...,这个查询要扫全表(或大范围索引),会锁住大量行、占用数据库 CPU,直接影响线上交易的下单、支付。大屏是辅助决策的,它的查询把生产库拖慢了,就是「辅助功能拖垮核心业务」,本末倒置。所以汇总表的存在,本质是「把统计压力从生产表转移到预聚合表」,让统计查询不打扰生产。

汇总表怎么保证数据准。靠定时任务「重算 + 对账」。每天凌晨的重算会把昨天的数据从头 GROUP BY 一遍(这时昨天数据基本不变了,扫原始表安全),写入汇总表;对账任务定期比对汇总表累加值和原始表的真实值,差异超过阈值就告警人工介入。这套「重算保证准、对账发现错」的机制,让汇总表在「不扫生产表」的前提下,依然保持可信。

三、多租户隔离的三种模式

多租户统计,隔离是最容易翻车的。我们有三种模式,对应不同场景。

模式 1:汇总表内置 tenant_id。 最常用。stat_daily_summary.tenant_id,查询走 WHERE tenant_id = ?。平台汇总行用特殊值 '0'(代码常量 ALL_TENANTS = "0")。前端下拉框「全平台」对应的 value 就是 "0"。

模式 2:实时分析强制校验。 实时分析看板(OpsAnalyticsService)每个方法首行 requireTenantId(tenantId),SQL 里 WHERE tenant_id = ?,多表 JOIN 时连 oi.tenant_id = o.tenant_id 都带上,防止 JOIN 时租户串了。

模式 3:搜索的「跨租户扩展」。 这块最特殊。搜索不是硬隔离,而是「可搜索的租户集合」:

List<String> tenantIds = sharingQueryHelper
    .getAccessibleSourceTenants(tenantId, "SEARCH");  // 自身 + 平台共享 + 显式共享

展开成 tenant_id 列表后,用 ES 的 terms 查询做多值过滤。跨租户的搜索结果会带回 tenantId 字段,供前端区分来源。这套设计支持「平台级共享内容 + 租户私有内容」混合检索,是 SaaS 平台型的刚需。

搜索的「跨租户扩展」和统计的「硬隔离」看起来矛盾,其实是两个不同需求。统计大屏的数据隔离是安全要求——A 租户绝不能看到 B 租户的 GMV,这是商业机密;搜索的跨租户是业务需求——用户搜索「苹果」,应该能搜到平台共享的苹果商品,不应该只限于本租户。所以隔离策略要看场景:涉及商业数据的(交易、资金)硬隔离,涉及内容消费的(商品、帖子)可扩展。一刀切地全硬隔离,搜索体验会很差(用户搜不到平台内容);一刀切地全可扩展,统计就泄密了。按场景分模式,是多租户统计的设计要点。

四、ES 搜索:4 模块并行 + 双管线同步

统计之外,ES 还承载统一搜索。SearchService.unifiedSearch() 有几个设计值得说。

4 模块并行查 ES,用 CompletableFuture.supplyAsync(..., searchExecutor),不是串行四次。分词用 ik_smart,标题权重 title^3。

为什么并行这么重要。串行查 4 个模块,每个 50ms,总共 200ms;并行查,4 个同时跑,总共还是 50ms。对搜索这种「用户等着看结果」的场景,200ms 和 50ms 的体验差距明显(用户会觉得 200ms 卡)。而且并行用的是独立线程池(searchExecutor),不占业务线程,即使 ES 慢了也不会阻塞其他请求。这个设计把「搜索快」和「搜索慢不连累别人」两个目标都兼顾了。

标题权重 title^3 是搜索相关性的调优。ES 默认各字段等权,但用户搜索时,标题匹配比内容匹配更重要(搜「苹果」,标题里有「苹果」的商品应该排在描述里提到「苹果」的前面)。title^3 让标题匹配的得分乘以 3,排序更符合直觉。这种相关性调优没有标准答案,要按业务场景试——我们试了 ^2 觉得区分度不够,^5 觉得标题匹配垄断排序,最后 ^3 是手感最好的。

分页上限保护:if (p * ps >= 10000) 直接报错。这是 ES max_result_window 限制,深分页必须用 scroll,不能硬翻。

优雅降级。ES 启动时不可达只告警不阻断,运行时 try-catch 返回 degraded=true 空结果。搜索挂了不能拖垮整个应用。

空结果推荐。4 模块全空时,回退查该租户 6 个商品做 recommendations,避免用户看到空白页。

数据同步是双管线。Logstash 全量增量(每 30 秒),用 tracking_column => update_time 游标增量拉,运维侧配置。RabbitMQ 实时增量,应用侧发消息,EsSyncConsumer 消费 upsert/delete,带 traceId 透传。两条管线互为补充,Logstash 兜底、MQ 保实时。

双管线为什么必要。单用 Logstash,30 秒延迟对新发布的商品不友好(发布后 30 秒才搜得到);单用 MQ,如果消息丢了(消费失败、队列积压),ES 数据就和 MySQL 不一致,且不容易发现。两条管线叠加:MQ 保实时(秒级),Logstash 兜底(30 秒内一定会再同步一次,补上 MQ 丢的)。这种「实时管线 + 兜底管线」的双保险,是数据同步高可用的常见模式。

双管线有个要注意的点:幂等。同一条数据可能被 MQ 和 Logstash 各同步一次,ES 的 upsert 必须幂等(以 MySQL 的数据为准,重复 upsert 结果一样),否则会出现数据错乱。我们的 ES 文档 _id 用 MySQL 主键,upsert 天然幂等(同 ID 覆盖,结果是最后一次的值)。这个设计让双管线可以安全并存,不用担心重复同步。

五、可观测性:到 Prometheus 为止

资金和交易系统,监控是底线。我们接了 Micrometer + Prometheus:

// 注册了一组业务 Counter/Gauge
Counter orderPay = Counter.builder("order.pay.total").register(meterRegistry);
Counter paymentAmount = Counter.builder("payment.amount.fen").register(meterRegistry);
Gauge pendingSettlement = Gauge.builder("settlement.pending.count", atomicLong, AtomicLong::doubleValue).register(...);

暴露在 /actuator/prometheus,覆盖订单/支付/退款/提现/结算/分润全链路的 Counter、Timer、Gauge。

但要诚实说,我们没有 Grafana dashboard,可观测性止步于 Prometheus 端点暴露。原因不是不重视,是阶段性取舍。创业期 4 个人,Prometheus 指标已经能通过 curl /actuator/prometheus 加告警规则覆盖关键链路。Grafana 看板是锦上添花,等业务跑稳、有人力再做。

我之前写过可观测性,那套 RequestId + 统一错误信封 + 监控告警的思路是一脉相承的:先把「能排查」做扎实,再追求「好看」。

同样,PlatformDashboardController 里的 DAU/MAU/PV/UV 字段,目前是 TODO 占位(返回 0),注释写着「from Sensors Analytics」——神策接入没落地。我没把这写进大屏当卖点,因为它是占位,不是实现。诚实比好看重要。

这个「诚实」值得展开。很多项目的大屏会把还没接的指标也画上去(显示 0 或假数据),对外宣称「我们有大屏」。短期看着唬人,长期是债——假数据会误导决策(老板看着 DAU=0 以为产品没人用),且接真实数据时要改前端,返工成本高。我们的做法是占位字段明确标注 TODO,既不假装实现了,也不让它误导决策。等神策真接上了,把 TODO 换成真实调用就行,前端不用改。诚实不只是态度,也是降低未来返工成本的手段。

可观测性的「阶段性取舍」还有一层考量。创业期最重要的是「能排查」,不是「好看」。Prometheus 端点暴露了指标,curl 能查、告警规则能配,关键链路的问题就能被发现。Grafana 看板是「让指标更好看」,但它不增加「能发现的问题」的数量——该告警的还是告警,只是展示形式从 curl 输出变成了图表。所以在人力紧张时,先把「能排查」做扎实(指标 + 告警),「好看」(Grafana)等有余力再做。这个优先级判断,适用于所有创业期的技术投入。

六、前端:按需选实现深度

前端大屏有个反常识的细节:15 个大屏页面,图表实现深度差别极大,我们是故意的。

大屏实现为什么
ScreenView(主大屏)ECharts + D3 + Three.js + Tween.js 混合门面,要炫:粒子背景、数字滚动、24h 热力图、深色科技风
TradeAnalyticsDashboardECharts 折线+柱状业务分析,够用就行,双 Y 轴看 GMV+订单
PlatformDashboardView裸 SVG(<polyline> 手画)早期原型,直接画线最快,没必要引图表库
OperationStatsView纯 KpiCard 卡片,无图表数字看板,卡片比图表更清楚

不同页面按「够用就好」选实现。主大屏 ScreenView 投入重兵,因为它是给老板和客户看的门面;运营统计页就是个数字看板,纯卡片反而比花哨图表清楚。

这种「实现深度不统一」在工程上有个好处:维护成本可控。如果 15 个大屏都按主大屏的标准做(ECharts+D3+Three.js 混合),维护工作量会爆炸——每个图表都要调样式、处理动画、适配数据。按需选深度后,只有主大屏是「重实现」(需要精心维护),其他都是「轻实现」(ECharts 标准配置或纯卡片),维护成本低。这个取舍让 4 个人能维护住 15 个大屏,而不是被大屏的维护工作淹没。

还有个判断:大屏的实现深度要跟「数据的价值」匹配。主大屏展示的是平台核心指标(GMV、订单、用户增长),这些数字要「好看」地呈现(动效、视觉冲击),因为它们是给决策者建立信心的;运营统计页展示的是日常运营数据(昨日订单数、待处理退款),这些数字要「清楚」地呈现(一眼看到具体值),不需要动效。数据价值决定展示方式,不是反过来「为了炫技而炫技」。

ECharts 还是按需引入的,统一入口 utils/echarts.ts:

import { init, use } from "echarts/core";
import { BarChart, LineChart, PieChart } from "echarts/charts";
use([BarChart, LineChart, PieChart, /* 只注册用到的组件 */]);
// 注释:仅注册项目中实际使用的图表类型,大幅减小打包体积

不引整个 echarts,只按需注册用到的图表和组件。这是 AI 写代码时最容易犯的错——import * as echarts 一把梭,打包体积爆炸。靠规范拦住。

七、降级哲学:大屏不能崩

最后讲一个贯穿全局的设计:大屏挂了,不能影响业务系统。

后端:ES 不可达只告警;实时分析超时有兜底;汇总表查不到返回缓存旧值。

前端:每个 API .catch 吞错返回空对象或空数组。大屏宁可显示旧值或空,也不能白屏。PlatformDashboardView 保留旧值不归零。

数据层:Redis 实时计数挂了,T+1 汇总表还有;汇总表任务挂了,还有手动触发入口。

这套降级是「统计是辅助功能」这个定位决定的。统计大屏出问题,业务该跑还跑;但反过来,统计查询把生产库打挂,就是事故。所以宁可统计降级,也不让统计拖垮交易。

降级设计的本质是「区分核心和非核心」。大屏是辅助决策,挂了老板看不到数字,但用户照样能下单;交易是核心,挂了用户下不了单,直接损失收入。所以大屏要为交易让路——大屏慢了、降级了、显示旧值,都能接受;但大屏不能反过来拖慢交易。这个「非核心不拖累核心」的原则,在架构设计里要始终贯彻。很多团队的统计系统翻车,就是因为没把统计当「非核心」,让它和交易争抢资源,最后一起挂。

具体怎么实现「不拖累」。三层:查询层(统计走汇总表不走生产表)、资源层(统计用独立数据库连接池,不跟交易抢连接)、熔断层(统计查询超时直接返回旧值,不堆积)。三层叠加,统计的故障被隔离在统计范围内,不会扩散到交易。这种「故障隔离」的设计,比「统计系统做到高可用」更务实——统计允许降级,所以不用追求它的高可用,只要保证它「挂了不连累别人」就行。

统计中台这块讲完。回头看整个设计,没有什么前沿技术,全是「在 4 个人、2 个月、SaaS 创业期」这个约束下做的最务实选择。MySQL 汇总表解决准实时统计,Redis 解决实时计数,ES 解决搜索,三层各管一摊,互不干扰。这套架构的精髓不在于用了什么先进技术,在于「克制」——该上的不上(ClickHouse),该省的省(Grafana),该标的标(延迟提示),把每一分技术投入都花在刀刃上。这种克制力,比堆技术栈更考验架构师的判断,也是小团队能撑住大数据屏的关键所在,更是这套架构跑下来没翻大跟头的根本原因。记住这一点,比记住任何一个具体的技术选型都重要,它适用于任何规模受限的团队在面对技术选型时做判断,也是这套架构能被复用到其他项目的可迁移经验,更是创业期技术决策的一条通用原则:在约束下做最务实的选择,永远比追求理想方案更靠谱也更可持续,这一点怎么强调都不为过。

深度展开:统计中台的三层数据新鲜度

数据大屏最容易被业务方challenge的是"这个数为什么和订单管理对不上"。根因是统计中台有三层数据,各自的"新鲜度"不同,不了解的人会误以为该一致。

第一层:实时计数(Redis)。 用户下单瞬间 Redis INCR,延迟毫秒级。但 Redis 可能因为各种原因(缓存击穿、重启)和数据漂移,所以它是"近似实时"。

第二层:T+1 汇总(MySQL 汇总表)。 凌晨跑定时任务,从订单表 GROUP BY 出昨天的 GMV、订单数,写入 stat_daily_summary。这是"准实时"——今天的数据要等明天凌晨才汇总。

第三层:对账修正。 实时和 T+1 之间的差异(退款冲减、跨天订单、缓存未失效),通过对账任务发现并修正。这是"修正层"。

业务方看大屏(实时)和看订单管理(原始数据),两者本来就有时间差。统计中台的诚实做法:大屏上明确标"截至 X 时 X 分,实时数据可能有 X 分钟延迟",而不是假装数据是实时的。

这个「标注延迟」是统计系统可信度的关键。如果不标注,业务方会默认大屏是实时的,一旦对不上就怀疑系统出 bug,引发无谓的排查;标注了,业务方就知道「这个数有延迟,对不上是正常的」,不会误判。所以诚实标注不只是态度,是减少沟通成本的工程手段——它消除了「为什么大屏数字和订单管理对不上」这类反复出现的疑问。

三层新鲜度的设计还有个深层考量:它让「实时性」和「准确性」可以分别追求。实时层(Redis)追求快但可能不准(缓存漂移);T+1 层(汇总表)追求准但不实时(隔天才更新);对账层追求「发现并修正前两层的差异」。三层各司其职,不要求一个数据源既快又准——那是不现实的。接受「快和准不能兼得」,用分层让它们各得其所,是统计系统的务实设计。
