一个低代码平台宣称能扛 10W QPS，往往扛不住的不是渲染——渲染结果早被 CDN 挡掉了。真正先崩的，是 schema 的读写：设计器每 5 秒批量保存、运行时每次渲染都要读 schema，读写瓶颈才是高并发的命门。

本文基于鲁班低代码平台各仓库的既定关系与调用链，推演各系统的部署关系、高并发演进与高可用保障，便于在流量与可用性要求提升时按图演进。

项目归属说明：本文的架构经验与高并发数据（峰值 10 万+ QPS、多级缓存、P0 清零等）来自我大厂时期主导的内部低代码系统"鲁班"；而文章展开的六库架构与代码实现细节，以我个人开源项目 OpenLuban（架构同源、代码全新重写，GitHub）为准。原公司系统源码不公开。

一、系统关系与调用链（现状锚点）

鲁班平台由六个仓库组成，边界固定、契约清晰：

系统职责技术栈下游依赖
luban-ui组件库 + 设计器 + Render 运行时Vue3 / Nx无（被集成）
luban管理后台Vue3 / Vite / Pinia / Element Plusluban-bff /api/*
luban-website对外站点（SSR）Nuxt 3luban-bff /api/public/*
luban-bff接入层：认证、聚合、协议适配Next.js（Node）luban-backend /backend/*
luban-backend主后端：站点/页面/用户/设置Spring Boot / MyBatis / Redis / MySQLMySQL、Redis
luban-backend-go与主后端同契约的 Go 实现Go / GinMySQL、Redis

调用关系可分三层看：

  • 管理后台与对外站点只访问 BFF，不直连后端。
  • BFF 通过 BACKEND_BASE_URL 调用主后端（可在 Java / Go 间切换）。
  • 主后端写 MySQL、用 Redis 做系统设置等缓存；公开页按 slug+path 读已发布页面（只读、无鉴权）。

三、部署图（多环境推演）

在「生产高可用、可水平扩展」的前提下，各系统部署关系可推演如下。图中区分管理端流量与对外站点流量，便于后续按路径做差异化优化（如对外读多写少、可强缓存）。

部署要点：

  • luban：静态资源可走 CDN 或对象存储，仅 HTML 入口走 LB。
  • luban-bff：无状态，多实例 + LB 水平扩展；可单独为「公开接口」做实例组或路由策略（读多可加大缓存）。
  • luban-website：Nuxt SSR 多实例，前面 CDN 缓存静态与可选「按 slug+path 的页面结果」。

  • MySQL：主从读写分离可期；写主读从，后端或通过数据源路由实现。

四、高并发优化

4.1 读写分离与缓存

  • MySQL：主库写，从库读（列表、单条查询、公开页查询）。BFF/后端通过数据源路由或 JDBC 配置实现读写分离。
  • Redis：
    • 系统设置：已实现的 settings:global 读缓存，写时更新 DB + 刷新 Redis。
    • 可选：公开页按 (slug, path) 缓存：public:page:{slug}:{path}，TTL 按发布变更频率设定（如 60s～300s），发布/下架时主动失效。
  • BFF 层：对「公开页面」接口可做短时内存/Redis 缓存，减少对后端的重复请求。

读写分离有个常见误区:以为「加个从库就高并发了」。实际上从库只是把读压力分散,不解决「单条查询慢」的问题。如果一条公开页查询要 join 五张表、扫十万行,加再多从库也慢。所以读写分离的前提是「单条查询本身要快」——索引建对、查询不 full scan、热点数据进缓存。从库是水平扩展,缓存和索引是单点优化,两者配合才有意义。

缓存策略的 TTL 设定也有讲究。系统设置这种「几乎不变」的数据,TTL 可以设长(几小时),变更时主动刷新。公开页这种「偶尔变」的数据,TTL 设短(60-300 秒),兼顾新鲜度和命中率。千万不能所有缓存用同一个 TTL——不变的数据设短 TTL 是浪费(频繁回源),频繁变的数据设长 TTL是过期(用户看到旧内容)。按数据的变更频率分级设 TTL,是缓存工程的基本功。

4.2 连接与线程池

  • luban-backend：数据库连接池（如 HikariCP）大小 ≈ 实例数 × 每实例连接数，总连接数不超过 MySQL max_connections；线程池（Tomcat/Netty）与 CPU 核数、IO 比例匹配，避免过多阻塞线程。
  • luban-bff：Node 异步 IO，注意对后端调用的连接复用与超时设置；可配置 BACKEND_BASE_URL 指向后端 LB，避免单点。

连接池大小是最常被调错的参数之一。很多团队默认值是 10-20,在低并发没问题,高并发时会成为瓶颈——请求排队等连接,响应时间飙升。但也不是越大越好:连接池开到 100,4 个实例就是 400 个连接,接近 MySQL 默认 max_connections(151)的上限,再开就「too many connections」了。合理的算法是:总连接数 = 实例数 × 每实例池大小,要留出 30% 余量给运维工具和其他服务。比如 MySQL max_connections 设 500,后端 4 实例,每实例池大小设 80(4×80=320,留 180 余量),这是个稳妥的起点。

线程池同理。Tomcat 默认 200 线程,在 IO 密集型场景(大部分时间在等 DB/Redis 返回)够用;但如果业务有大量 CPU 计算(比如 schema 序列化/反序列化),线程数要往 CPU 核数靠,避免过多线程上下文切换。判断标准是看线程的「阻塞比」——如果线程大部分时间在阻塞等待(等 IO),多开线程有用;如果在跑 CPU,多开线程反而变慢。

4.3 静态与页面级加速

  • luban / luban-website 静态资源：构建产物上传 CDN，HTML 中引用 CDN 域名；首屏与交互资源分离，必要资源预加载。
  • luban-website 已发布页：SSR 结果可按路由（如 /:siteSlug/:path*）在 CDN 或 BFF 侧做短 TTL 缓存，降低后端与 SSR 压力。

静态资源上 CDN 是性价比最高的优化,没有之一。一次配置,长期受益——静态资源(JS/CSS/图片)走 CDN 后,既不占源站带宽,又让用户从最近节点加载(延迟低)。唯一要做好的是缓存失效:发布新版本时,资源文件名带 hash(如 app.a3f5b2.js),hash 变了就是新文件,CDN 自然回源;hash 没变说明内容没变,CDN 缓存继续用。这套「文件名 hash + 长缓存」的方案,让静态资源可以设一年 TTL(缓存命中率接近 100%),发版时靠文件名变化做天然失效。

4.4 限流与熔断

  • 入口：在 LB/网关或 BFF 对按 IP/UserID/接口的 QPS 限流，防止雪崩。
  • BFF → Backend：对后端调用做熔断（如连续失败 N 次或错误率超阈值则熔断），并返回友好降级（如「稍后重试」），避免拖垮后端。

限流和熔断是高并发的「安全气囊」,平时不用,出事时救命。限流防「外部流量突增」(比如被刷、活动爆量),熔断防「内部故障扩散」(比如 DB 慢了,所有请求堆积超时,最终全盘雪崩)。两者方向不同:限流是「拒绝多余请求保护自己」,熔断是「停止调用故障方保护自己」。一个低代码平台,既要防外部爆量(限流),也要防内部连锁故障(熔断),缺一个都可能在流量高峰翻车。

熔断的降级策略要想清楚。公开页熔断了,可以返回 CDN 里的旧缓存(用户无感);管理后台熔断了,要明确提示「服务繁忙稍后重试」(不能静默失败让用户以为是自己操作问题)。降级策略要在平时想好,出事时才来得及用——等出事了再想,根本来不及。

4.5 异步与削峰（可选演进）

  • 非实时操作（如「批量发布」「导出」）可投递到消息队列，由 Worker 消费，避免长耗时请求占满连接与线程。
  • 写操作（创建/更新站点、页面）保持同步；若未来有「页面渲染结果预生成」等重计算，可异步写缓存。

五、高可用保障

5.1 无状态与水平扩展

  • luban-bff、luban-backend、luban-website：均设计为无状态，不依赖本地存储会话；通过 LB 多实例部署，单实例故障由 LB 摘除即可。
  • luban：纯前端 SPA，静态托管即可，可用性依赖 CDN/托管平台。

5.2 数据层高可用

  • MySQL：至少一主一从，主故障时提升从为主（手动或通过编排/工具）；有条件可多从库做读扩展。
  • Redis：主从复制 + 哨兵，或集群模式；避免单点，设置合理 maxmemory 与淘汰策略。

数据层高可用是整个系统可用性的命门。应用层挂一个实例,LB 摘掉,用户几乎无感;数据层挂了,所有实例一起瘫痪。所以数据层的可用性投入要优先于应用层——先保证 MySQL 和 Redis 不成为单点,再谈应用层的多实例。

MySQL 主从切换有个容易忽略的坑:主从延迟。从库是异步复制的,主库写入后,从库有几毫秒到几秒的延迟。如果主库故障时强切从库,这几秒的延迟数据会丢(还没复制过去的)。所以切换决策要权衡:切,可能丢一点数据;不切,系统不可用。对低代码平台这种「丢几秒编辑可接受」的场景,切是合理选择;对资金系统,就要用同步复制(半同步)保证不丢,代价是写性能下降。没有完美方案,只有「按业务容忍度选型」。

Redis 的淘汰策略要根据业务选。系统设置缓存用 allkeys-lru(最近最少使用淘汰)合适;公开页缓存用 volatile-ttl(快过期的先淘汰)合适。选错淘汰策略会导致「热数据被淘汰、冷数据留着」,缓存命中率暴跌。这个细节不踩坑不知道,踩了就刻骨铭心。

5.3 健康检查与自愈

  • 各服务提供 /ping 或 /health，LB 或 K8s 探针定期探测；失败则摘除实例或重启容器。
  • 数据库与 Redis 的连接失败应有重试与退避，并记录告警，便于运维介入。

健康检查要分两级。/ping 是轻量检查(进程活着就返回 200),用于 LB 快速摘除死实例;/health 是深度检查(连 DB、连 Redis 都通才算健康),用于判断「服务虽然活着但能不能干活」。只用 /ping 会漏掉「进程活着但 DB 连不上」的半死状态;只用 /health 会因为依赖检查太重而误判(比如 DB 抽风一下,健康检查失败,实例被摘,但其实应用还能服务)。两级配合:LB 用 /ping 做快速摘除,运维用 /health 做深度诊断,各司其职。

重试与退避是防止「故障放大」的关键。DB 连接失败,如果立即重试,所有请求一起重试,可能把刚恢复的 DB 又打挂(惊群效应)。正确做法是指数退避:第一次失败等 100ms,第二次 200ms,第三次 400ms,逐步拉开重试间隔,给故障方恢复时间。这个细节在高并发场景能避免「小故障滚成大雪崩」。

5.4 契约一致与多运行时

luban-backend 与 luban-backend-go 保持 API 契约一致（见 luban-backend/docs/API.md）。

BFF 仅通过 BACKEND_BASE_URL 切换，即可在某运行时故障时切到另一实现，或按地域/流量拆分部署，提升整体可用性。

六、系统 Roadmap 与蓝图

在「当前六库分工不变、契约稳定」的前提下，可按阶段落地高并发与高可用能力，并预留扩展点。

阶段目标关键动作
P0 当前功能闭环、契约统一管理后台 + 对外站点 + BFF + 主后端（Java/Go 二选一）打通；API 文档与错误码统一。
P1 稳基高可用、可观测MySQL 主从、Redis 主从/哨兵；BFF/Backend 多实例 + 健康检查；集中日志与基础监控（QPS、延迟、错误率）。
P2 扛量高并发、读扩展后端读走从库；公开页 (slug, path) Redis 缓存；BFF/公开接口限流与熔断；静态资源上 CDN。
P3 优化体验与成本luban-website SSR 结果 CDN 或 BFF 缓存；可选消息队列做异步任务；按需引入 Go 版分流或多地域部署。
P4 演进平台化与开放多租户/多站点隔离与配额；开放 API 与 webhook；审计与操作日志；可选 schema 版本与回滚。

Roadmap 的价值不在于「计划多完美」,在于「让团队知道现在在哪一步、下一步该干什么」。低代码平台这种长周期项目,没有 Roadmap 容易陷入「东一榔头西一棒子」——今天加个缓存、明天改个部署,没有主线。有了分阶段的目标(P0 打通、P1 稳基、P2 扛量),每个阶段的验收标准清晰,团队的努力方向一致,演进才有序。而且 Roadmap 要留弹性——业务变化了,阶段优先级可以调,但「分阶段、有验收」这个框架不能丢。

七、小结

这套架构跑下来,我最大的体会是:高并发架构的本质是「把流量挡在正确的层」。读流量挡在缓存层(CDN/Redis),不让它打到 DB;写流量挡在缓冲层(批量队列),不让它压垮 DB;突发流量挡在限流层(LB/BFF),不让它雪崩。每一层都消化掉自己该消化的压力,只把扛不住的往上传,这样每一层的负载都在可控范围内,整个系统才能稳得住,也才扛得住业务量增长带来的持续压力。

六库的分工在这个架构里是合理的:前端三个库(luban-ui/luban/luban-website)管「渲染和交互」,BFF 管「接入和协议」,后端两个库(Java/Go)管「业务和数据」。这个分工让每一层都能独立扩容——前端走 CDN、BFF 多实例、后端多实例、数据层主从,互不牵扯。如果架构是「大单体」,扩容就得整体放大,成本和复杂度都高得多。

多运行时(Java + Go 契约一致)是这个架构的一个特色保险。Java 是主力(生态成熟、开发快),Go 是备胎(性能好、资源省)。正常跑 Java,流量上来或某地域需要低延迟时切 Go。两套实现保持 API 契约一致,BFF 用一个环境变量就能切换。这套设计的代价是「两套代码要同步维护」,收益是「多了一份兜底」——对可用性要求高的场景值得。

最后说一句演进节奏。这套架构不是一次性建成的,是按 Roadmap 分阶段落的——P0 先打通功能,P1 做高可用基线,P2 才上扛量优化。很多团队一上来就想做「高并发架构」,结果功能还没闭环就在折腾缓存和分库,本末倒置。高并发是为业务量服务的,业务量没到,提前做高并发是过度设计;业务量到了再做,是顺势而为。按业务节奏匹配架构演进,比追求「一步到位的完美架构」务实得多,也更容易把每一层做扎实而不是全是半成品,这才是架构演进能持续推进的关键所在,也是低代码平台这类长周期项目能健康生长的根基。

深度展开:低代码平台的高并发瓶颈

低代码平台的高并发,瓶颈不在"渲染"(渲染可以 CDN 缓存),而在schema 的读写。

这个判断是整个架构推演的起点。低代码平台的本质是「schema 驱动」——页面长什么样、组件怎么排、数据怎么绑,全存在 schema 里。运行时渲染页面,本质是读 schema + 执行 schema。所以 schema 的读和写,就是这个平台的命门。把读写的瓶颈想清楚,整个高并发架构就有了主心骨;反之,如果不盯 schema 读写去优化别的地方(比如优化渲染引擎本身),投入产出比会很低——因为瓶颈根本不在那儿。

写瓶颈:schema 保存的乐观锁 + 批量队列

写瓶颈:设计器频繁保存 schema,如果每次都写 DB,DB 扛不住。我们的做法:schema 写入走"乐观锁 + 队列",前端先把改动攒着,每 5 秒批量提交一次。

展开讲这套写入机制。设计器里,用户拖一个组件、改一个属性,都会触发 schema 变更。如果每次变更都立即落库,一个用户编辑 10 分钟可能产生几百次保存请求,DB 的写压力会随着在线编辑人数线性增长——100 个编辑者就能把单库写吞吐打满。

所以做了两层缓冲。第一层是前端节流:改动不立即提交,攒在内存里,每 5 秒(可配)批量提交一次。这一层把「几百次保存」压成「几十次」,写请求量直接降一个数量级。第二层是后端乐观锁:每次提交带 version 字段,后端用 WHERE id=? AND version=? 做条件更新,version 不匹配说明中间被别人改过,返回冲突让前端合并。乐观锁比悲观锁(行锁)轻量得多,不会因为一个编辑者长时间持有锁而阻塞其他人。

这套机制有个权衡:5 秒的批量窗口意味着「最多丢失 5 秒的编辑」。对于低代码设计器这个场景是可以接受的——它不是资金系统,丢 5 秒编辑让用户重做,比 DB 被打满全平台不可用好得多。窗口大小是个可调参数,对协作要求高的场景可以缩短到 1-2 秒,但要注意写请求量会回升。

读瓶颈:多级缓存把 DB 藏起来

读瓶颈:运行时每次渲染都要读 schema,如果每次都查 DB,DB 扛不住。我们的做法:schema 读走多级缓存(CDN → Redis → DB),命中率 99%+,DB 几乎不被打到。

读这边的设计更激进,因为低代码平台是典型的「读多写少」——schema 一旦发布,可能被几万次渲染读取,但写入(编辑)只有几次。这种读写比(可能 1000:1 甚至更高),天然适合多级缓存。

三级缓存各有分工。CDN 缓存「已发布页面的渲染结果」,命中后连服务器都不用请求,直接边缘返回,延迟最低、对后端零压力。Redis 缓存「schema 本身」,CDN 没命中时(比如个性化页面、登录态)走 Redis,命中率高且响应快(毫秒级)。DB 是兜底,只有 CDN 和 Redis 都没命中(首次访问、缓存失效、冷启动)才查 DB。

命中率 99%+ 意味着什么。一万次渲染请求里,只有不到一百次打到 DB。如果 QPS 峰值 10 万,落到 DB 的查询只有 1000 左右——这个量级单库轻松扛住。这就是「读多写少」架构的核心收益:用缓存把 DB 藏起来,DB 的容量规划只针对那 1% 的穿透流量,而不是全量。如果把缓存去掉,DB 要扛 10 万 QPS,那就是另一个量级的成本和复杂度了。

缓存不是没成本。最大的成本是「一致性」:schema 更新了,缓存怎么失效。我们的做法是发布时主动失效——设计器点「发布」,后端先写 DB,再删对应的 Redis key,再purge CDN 对应路径。这套失效链路必须可靠,否则会出现「改了内容但页面没更新」的事故。失效失败要有重试和告警,不能「写了 DB 失效缓存」这一步静默失败。

读多写少是低代码的命格

低代码平台的「读多写少」不是偶然,是这类系统的结构性特征。理解了这一点,就知道为什么缓存投资回报最高。

编辑(写)只有内容创作者在做,而消费(读)是所有终端用户在做。一个企业用低代码搭了 100 个应用,每个应用每天被几千人用——编辑者可能就几个人,读者是几万人。这个 1:10000 的读写比,决定了任何对「写」的优化(比如提升 DB 写吞吐)收益都有限(因为写本来就不多),而对「读」的优化(缓存)收益巨大(因为读是绝对大头)。

所以低代码平台的高并发架构,优先级排序很明确:第一先把读缓存做扎实(收益最大),第二做写的批量缓冲(防 DB 写打满),第三才考虑渲染优化和 SSR 加速(锦上添花)。这个顺序搞反(比如先去优化渲染引擎性能),会发现 QPS 上不去——因为瓶颈在 schema 读,不在渲染。先解决命门,再解决枝节,这是架构优化的基本纪律。
