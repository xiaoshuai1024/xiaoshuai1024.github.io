我在 front-end → 全栈 → AI 开发这条路上走得很早,也很清楚一件事:技术人的成长从来不是线性的,而是被"市场需求 × 自我认知 × 学习方法"三个变量反复拉扯。多数人卡在第二个和第三个——不知道自己在市场上值多少钱,也不知道该怎么把短板补到及格线。

这篇文章讲的就是我怎么用 Claude Code 把一个私有的"学习系统"搭成一个可运行、可量化、可复盘的知识库。不是讲考试技巧,而是讲「在 AI 时代,一个技术人该怎么组织自己的学习」。整套东西都在本地跑,所有数据都是真实可验证的,所有流程都是一条命令触发的。

先看结论:古法学习的核心问题是没反馈回路;AI 驱动学习的核心优势是把反馈回路压到分钟级。

一、古法学习的三个致命缺陷

我过去十年的学习方式和大多数工程师一样:刷掘金 / 看官方文档 / 跟着教程写个 demo / 放到收藏夹里吃灰。这种"查资料式学习"有三个结构性缺陷,不是靠努力能弥补的。

先说一个我自己的账本。我在学 RAG 的时候,把 LlamaIndex 官方文档、几篇高赞博客、一个 GitHub 上的 demo 项目都过了一遍,笔记做了不少,本地还 clone 了 demo。那两周我觉得自己"掌握 RAG 了"。后来有次技术选型会,同事问我"RAG 里 chunk 大小和 overlap 在生产环境怎么定,你们踩过什么坑",我支吾了半天,只说出"按经验来"。回去翻收藏夹,文章还在,但具体数字全忘了,那个 demo 项目也早就没跑过。

这不是个例,这是古法学习的结构性失败。

第一个:学什么靠拍脑袋。 看到 RAG 火了就去学 RAG,看到 LangGraph 出新版就去翻文档,完全是被社区热点牵着走。结果半年下来,收藏夹里堆了几百篇文章,简历上写不出任何一项"市场需要、我精通"的技能。学到最后,自己觉得都会,一被追问就全露馅。

更要命的是,这种学法容易陷入"低难度重复"。人天然喜欢学自己已经会的东西——因为那种"我又看懂一篇"的成就感很爽。结果就是:精通的更精通,薄弱的永远薄弱。没有外部信号告诉你"你该学什么",你永远在自己的舒适区打转。

第二个:学了=会了是幻觉。 看文档时觉得自己懂了,写 demo 时照着抄一遍跑通了,就给自己打个勾。这种"理解"是经不起追问的——别人问"为什么这么设计"“生产环境出过什么问题"“和另一方案相比差在哪”,多半答不上来。看十遍文档不如被追问一次,但古法学习里根本没有"被追问"这个环节。

我自己有次被戳穿的经历:学 Vite 的时候,我照着官方文档配了一遍 plugin,跑通了,心里想"懂了”。后来有次技术讨论,对方问我"Vite 的 pre-bundle 阶段用的什么工具,为什么 dev 比 webpack 快这么多",我只答出"用了 esbuild",但说不清楚 esbuild 做了什么、为什么不直接裸跑 ESM、生产构建为什么又要切回 Rollup。回去看文档才发现,那些内容都在第 3 段,但我当时"看懂了"就跳过去了。

第三个:没有数据沉淀,无法复盘。 三个月前学了什么、当时哪道题答不上来、薄弱点是什么,全靠脑子记。记不住的就当没学过,下次再刷到同样的文章又"学一遍"。没有学习记录、没有错题归档、没有能力趋势图,每一次学习都是一次性的,无法累积复利。

三个问题叠加的结果就是:你花了大量时间"学习",但简历上的技能清单一年到头没怎么变,真要展示的时候拿得出手的东西也没怎么变。这不是个人能力问题,是方法论问题。

二、我把"学习"拆成了一个工程化闭环

想明白这三点之后,我换了一个思路:把"学习"当成一个工程项目来做,有需求来源、有质量门禁、有数据报表、有迭代节奏。整个系统跑在一个私有仓库里(我叫它 learn_for_interview),核心组件是四个命令,对应四个工程动作:

命令工程动作输出产物
/scan-jobs市场技能趋势扫描学习清单(自动追加新技能)
/find-jobs岗位技能需求调研技能需求清单(方向 / 核心技能 / 出现频次)
/skill-test能力测试 + 严格评分评分表 + 雷达图 + 趋势 CSV
/study系统学习 + 测验 + 综合考试学习日志 + 错题本 + 评分快照

这四个命令串起来就是一个闭环:市场扫描发现新需求 → 能力测试定位差距 → 系统学习补缺口 → 测试验证掌握度 → 回到市场扫描看下一轮。每一轮迭代,数据都沉淀在本地 Markdown / JSON 里,可以打开看趋势,可以导出给 Excel,可以随时复盘。

关键设计决策有三个,任何一个缺失,这套系统就退化成花哨的笔记软件。

决策 1:个人信息和学习数据全部本地化

所有"我掌握什么 / 我学了什么 / 我哪里薄弱"的数据,都存在仓库里的几个文件里:

  • docs/能力评估数据.json —— 评分数据的唯一真相源,每次测试追加一条快照
  • docs/个人能力评分表.md —— 由脚本从 JSON 自动生成,禁止手改
  • docs/每日学习日志.md —— 每次学习的详细记录
  • docs/错题本.md —— 所有 score ≤ 40 的技能对应的错题,按技术类型分类
  • docs/技术学习清单.md —— 待学 / 学习中 / 已完成的技术项
  • docs/学习记录.md —— 已学内容的总览,接续上次薄弱点

为什么不放 Notion、不放到云笔记?两个原因:

第一,AI 工具读写本地文件最可靠。 Claude Code 直接 Read / Edit Markdown 和 JSON,零延迟,零幻觉(相对云端 API 调用)。放在 Notion 里,每一次读写都要过一次 API,格式丢失、字段错位是常态。

第二,数据可以版本化、可审计、可导出。 用 Git 管理,每一次评分、每一次错题归档都有 commit 历史。想分析"过去三个月我前端能力的进步曲线"?跑一个脚本从 JSON 里抽数据画个图就行,不需要依赖任何第三方服务。

决策 2:能力评估必须通过测试,禁止自评

这是整套系统里最关键的设计。

我见过太多工程师(包括我自己以前)的"技能清单"是这么写的:精通 Vue / 熟悉 React / 了解 Node.js。这种自评毫无意义——“精通"和"熟悉"的标准是什么?是看过文档算熟悉,还是写过生产项目算熟悉,还是能讲清底层原理算熟悉?

我的做法是:六大能力方向、上百项技能,每项都有 2-3 道由浅入深的测试题,必须作答,必须评分,评分有明确标准:

分数含义判定条件
0待评估没测过
20不了解测试时完全答不上来
40了解概念能说出方向,但关键点有硬伤
60能使用独立完成常规任务,关键点需查证
80熟练掌握生产级,独立排查疑难
100精通能讲清原理,能教别人

并且有从严评分纪律:自评应略低于真实水平,留一档安全边际;有疑虑取低档;答出"根本错误 / 概念混淆 / 把 A 和 B 对调"这种硬伤,强制 ≤ 40,不打人情分。

为什么要这么严?因为自我感觉良好是学习最大的敌人。评 60 分意味着"我这块还差得远”,评 80 分意味着"我可以上生产"。如果自评虚高,学习清单就永远排不到真正薄弱的地方,时间就浪费在已经会的技能上反复刷存在感。

每次测试完,所有 score ≤ 40 的技能对应的错题强制归档到错题本,字段是固定的:题目 / 错误答案 / 正确答案 / 为什么错 / 掌握标志。这套错题本后来成了我复习时最高效的材料——每一题都是我自己踩过的坑,比任何外部教程都贴身。

决策 3:学习过程强制四步走,不允许"看懂就算"

/study 一个命令进去,走完四步才算学完:

📖 第 1 步:学习材料输出
  概述 + 核心概念 + 代码示例 + 常见陷阱 + 技术对比 + 考核要点
      ↓
🧪 第 2 步:知识检测(5-8 题)
  基础理解 2-3 题 + 应用能力 2-3 题 + 深度思考 1-2 题
  逐题作答 → 评分 → 讲解 → 指出薄弱点
      ↓
🎯 第 3 步:综合考试(3-5 轮)
  概念辨析 → 场景应用 → 原理追问 → 设计题 → 反向提问
  逐轮点评(亮点 + 改进 + 追问)
      ↓
📝 第 4 步:自动归档
  学习日志 + 能力评分快照 + 错题本 + 学习清单状态

第 2 步和第 3 步是核心——不经过"被提问"的学习,都算不上学完。AI 在这里扮演的不是"知识搬运工",而是考官 + 教练 + 错题整理员。它问我答,它追问我解释,它挑错我修正。整个过程有输出、有反馈、有纠正,这才是真正把知识从"看过"变成"会用"。

为什么四步必须全走完,不能跳?我试过跳过第 3 步(综合考试),结果发现:即使第 2 步测验全对,一旦进入"概念辨析 + 连环追问"的考试模式,还是会暴露盲点。因为测验考的是"知识点",考试考的是"知识点的串联和表达"。你能答对一道关于 MCP 的选择题,不代表你能用 3 分钟完整讲清 MCP 解决了什么问题、和 REST 的本质区别在哪、生产环境接 MCP 要注意什么。“会做"和"会讲"之间,隔着几十场综合考试的距离。

另外第 4 步的自动归档不能偷懒。每次 /study 完成后,5 个文件必须全部更新:评分快照、学习日志、学习清单状态、生词本、学习记录。少更新一个,下次启动时数据就不连贯。这套"数据连贯性"是靠严格流程保证的,不是靠 AI 自觉——AGENTS.md 里我把每一步的更新动作都写成 checklist,AI 工具走完一步必须执行对应的归档动作,跳过就报错。

三、和"查资料学习"正面硬刚,差在哪

光说自己这套好没用,得摆出和古法的具体差异。我拿两个真实场景对比。

场景 A:学 MCP(Model Context Protocol)

古法路径:

  1. 掘金搜"MCP 协议”,看 3-5 篇文章
  2. 看官方 spec,似懂非懂
  3. 跟着示例跑通一个 demo,觉得会了
  4. 一周后忘了细节,下次用到再搜

AI 驱动路径:

  1. /scan-jobs AI 扫出来"青岛 AI 应用开发岗里 MCP 出现频率在涨",自动追加到学习清单,备注"市场需求高"
  2. /skill-test MCP协议,几道题下来得分 40,发现我对 MCP 的服务端配置、桥接方案和协议细节都停留在"知道有这个东西"的层面,强制进入学习清单的高优先级
  3. /study MCP协议,AI 输出完整材料,然后 8 道题考我,我答错 3 道,它逐题指出错在哪:比如把"服务端声明能力、客户端调用"的边界讲反了,它纠正说"定义权在服务端,Tool 是双向约定,但配置和暴露都在服务端"
  4. 综合考试 3 轮追问,从"什么是 MCP" 追到 “如果让你给团队设计 MCP 接入层,你会怎么分层”
  5. 3 道错题自动归档到错题本 AI 分类,带正确答案和掌握标志
  6. 能力评分 JSON 追加一条快照,node scripts/gen-skill-report.mjs 跑一下,HTML 雷达图和趋势 CSV 自动刷新

两种路径花的时间差不多(1-2 小时),但知识留存率和可复盘性完全不在一个量级。

场景 B:三个月后复盘"我前端能力涨了多少"

古法路径:

  • 翻收藏夹?几百篇文章看不出进度
  • 凭印象?“应该涨了吧”
  • 没有任何数据支撑

AI 驱动路径:

  • 打开 docs/能力评估报告.html,雷达图一目了然
  • 打开 docs/能力评估趋势.csv,导入 Excel 画折线图,每个技能点的分数变化清清楚楚
  • 比如 TypeScript 从最初的 20 分(测试后发现 interface/type 区分不清、泛型工具类型不熟)到系统学习后的明显回升,“JavaScript”、“框架原理"等方向也都有可见的涨幅,这些数字背后都是真实错题和真实作答,不是自我安慰

场景 C:半年没碰的知识点要复习

古法路径:

  • 重新搜文章,从第一页开始看
  • 看到熟悉的段落"这个我会”,跳过;看到不熟悉的"这个也要看",结果全部重看一遍
  • 花了 3 小时,其实只复习了 30% 真正需要补的内容
  • 看完还是不知道"我到底哪里没掌握"

AI 驱动路径:

  • 打开错题本,直接定位到 “JavaScript 分类” 下 2 条待复习的题
  • 打开 学习记录.md,上次学 JS 时的薄弱点写着"TDZ、let 循环绑定、宏微任务分类",直接针对这三点重新测一遍
  • 20 分钟搞定,错题本状态从 🔴 改成 🟢,评分快照追加一条
  • 下次 /study 启动时,生词本里那三个术语已经 🟢 了,不会再被提醒

两种路径花的时间差一个数量级,但更重要的是:AI 路径下的复习是"精准外科手术",古法路径下的复习是"全身 CT 从头扫到尾"。前者每次都有增量,后者每次都在做无用功。

四、这套系统的几条硬约束

工程化系统和"花哨的笔记"之间,差的就是约束。下面这几条是我跑了一个多月沉淀下来的,每一条都对应一次真实踩过的坑。

约束 1:信息真实性原则(最高优先级)

所有信息必须真实、可验证。禁止推测、编造任何经历、技能或岗位信息。所有数据有来源(招聘网站、官方 JD、真实数据)。无法确认的标注"待验证"并给出获取渠道。学习材料中的代码示例必须真实可运行,API 与官方文档一致。

这条原则是我写在 AGENTS.md 最顶部的。为什么这么严格?因为学习材料里掺假 = 一辈子记错。AI 工具最大的风险就是"看起来很对但其实是编的"。我宁可让它说"这个 API 我不确定,建议查官方文档",也不要它编一个假 API 让我背下来。

同样,爬下来的招聘数据、技能需求清单,链接必须真实可访问,技能出现频次必须来自 JD 原文。“待验证"这类标记是常态,不是瑕疵。

约束 2:评分数据单向流动

所有评分只能通过"测试 → 追加 JSON 快照"的方式落地,禁止手改评分表 Markdown。脚本 gen-skill-report.mjs 是唯一的渲染通道。这样能保证:数据源只有一个(JSON),其他产物(Markdown 表格 / HTML 报告 / CSV 趋势)都是它的视图。想改数据,只能改 JSON,不会出现在 Markdown 里改了但 JSON 没改、下次跑脚本又被覆盖掉的尴尬。

约束 3:错题强制归档

这个约束我加了两次才落实。最早错题散落在每日学习日志里,过半个月就找不到。后来我加了一条硬规则:每次 /skill-test 完成后,所有 score ≤ 40 的技能涉及的错题必须归档到 docs/错题本.md,否则就是流程漏洞。配套的自检清单四条,每次跑完过一遍。

这条约束的价值在复习时体现得最明显——打开错题本,每一题都是我答错的,比看任何外部教程都高效。

约束 4:学习前必须检查"学习记录”

每次 /study 启动,第一件事是 Read docs/学习记录.md,避免重复学已经掌握的、接续上次薄弱点。这条看似简单,但如果不强制,AI 工具很容易"每次都从零开始讲一遍",浪费时间。

约束 5:术语当场解释 + 生词本联动

每次学习材料里出现的新术语,AI 当场简短解释,并追加到 docs/生词本.md(状态默认 🔴 未了解)。下次 /study 开始前,先查生词本里还有哪些 🔴 的,主动问"要先讲一下吗?"。这条保证术语不会因为"当时好像懂了"就混过去。

五、几个真实数字

跑了一个多月,这套系统的数据是这样的:

  • 六大能力方向、上百项技能的评分体系,覆盖前端 / 后端 / AI 工程化 / 全栈架构 / Python / 管理
  • 80 多条错题归档在错题本里,按 JavaScript / TypeScript / 网络 / 框架 / 后端 / AI 分类
  • 十几次系统学习,每次 5-8 道题 + 3-5 轮综合考试
  • 每日学习日志记录了每一次学习的主题、得分、薄弱点
  • 能力评估数据 JSON 里有多条带日期的快照,可以回放任意时间点的评分状态

举几个具体例子:我第一轮前端测评的时候,原本以为 TypeScript 自己能用、怎么也有 60 分。走 /skill-test TypeScript 流程,5 道题下来真实得分 20——interface 和 type 的区别说不清,泛型工具类型 Pick / Omit / Partial 一个都写不出来,infer 关键字完全没概念。这 40 分的落差就是"自我感觉"和"真实水平"的落差。

再比如后端并发那块,我"自以为"熟悉线程池和 CompletableFuture,走 /skill-test 测试时被追问"线程池调度顺序"“AQS 的底层结构"“锁升级的完整链路”,结果线程池调度顺序搞反(实际是核心→队列→非核心→拒绝,不是核心→非核心→队列)、AQS 被我说成了 Mark Word(其实是 volatile state + CLH 双向队列)、CompletableFuture 不会写代码。评分直接被打到 30。这种"以为自己会但其实不会"的点,靠看文档是发现不了的。

这些数字背后是完全可追溯的学习轨迹。哪一天学了什么、哪道题答错了、错在哪、后来补上了没有,全部有据可查。这种"学习的可观测性”,是古法学习完全做不到的。

六、主题上升:招聘网站是信号源,但系统本身服务于学习

我把这套系统取名叫 learn_for_interview,仓库名也沿用了下来。但如果你看它的实际使用,面试只是其中一个触发场景,核心用途是个人技能的工程化管理。

为什么技能来源是招聘网站?因为招聘 JD 是目前最真实、最及时、最不带偏见的技能信号源。社区博客有泡沫,官方文档有滞后,只有招聘市场在实时反映"行业真正需要什么"。我把这个信号源引入学习系统,不是为了"找工作",而是为了让学习清单不脱离实际。

市场扫描告诉你"行业在往哪走",这个信号无论你是求职、在职、还是单纯想跟上技术演进都需要。能力评分告诉你"我在哪",这个认知无论你是晋升答辩、转岗评估、还是单纯想量化自己的进步都需要。错题归档告诉你"我哪里薄弱",这个数据无论你是做技术分享还是带团队都需要。趋势图告诉你"我在进步吗",这个反馈无论你是做年度复盘还是自我激励都需要。

换句话说,这套系统的核心不是"面试",而是个人技能的工程化管理。招聘网站只是提供了一个"能力对标对象"(岗位 JD 里的技能要求)、一个"检验时刻"(定期测试)、一个"失败成本"(技能落后会被市场淘汰)。把这些要素抽掉,换成"晋升答辩"、“转岗评估”、“季度复盘”,这套系统一样跑得通。

我的建议是:哪怕你暂时不看机会,也值得花一个周末把"能力评分 + 错题本 + 学习日志"这套东西搭起来。等你真的需要它的时候(无论是换工作、晋升、还是单纯想知道自己这一年学到了什么),你已经有几个月甚至一年的数据在手了。

反过来,如果平时不积累,等接到猎头电话才开始抱佛脚,那时候你连"自己会什么"都说不清,更别提"哪块还差"了。

七、和"查资料学习"的结构性差异

把上面这些摊开来看,AI 驱动学习和查资料学习的差异不在"哪个更高效"这种表面层,而是结构上的不同。

查资料学习是一个开环系统:输入(资料)→ 处理(看)→ 输出(感觉懂了)→ 结束。没有反馈,没有测量,没有归档。

AI 驱动学习是一个闭环系统:输入(市场需求 + 能力差距)→ 处理(学习 + 测试 + 综合考试)→ 输出(评分 + 错题 + 日志)→ 反馈(下一轮学习的优先级)。每一步都有数据,每一步都能回溯。

这背后的思维转变是:学习不是"消费内容",而是"生产数据"。每学一个东西,要能回答三个问题:**我学之前是什么水平?学之后是什么水平?我拿什么证据说我会了?**答不上来,就是没学会。

这种转变听起来反直觉——学习怎么会是"生产数据"呢?但仔细想想,任何没有数据沉淀的学习,本质上都是在"消费内容":消费了作者的时间、消费了你的注意力、消费了你的一两个小时,最后什么也没留下。生产数据的意思是,你每一次学习都留下可追溯的证据:评分变化、错题归档、日志记录。这些数据不是负担,是复利的本金。下一次学习、下一次复习、下一次能力评估,你都是从这些数据上起步,而不是从零基础起步。

另一个转变:AI 的角色变了。过去我们用 AI,是把它当搜索引擎(“MCP 是什么”)、当翻译器(“这段代码什么意思”)。这只是把 Google 换成了 ChatGPT,本质还是"查资料"。在我这套系统里,AI 是考官 + 教练 + 数据录入员——它出题考我,它追问我,它把我答错的题整理归档,它把我的评分刷新到趋势图上。它让我无处可藏,也让我每一次学习都有沉淀。

八、给想搭一套的人几条建议

如果你也想搭类似的东西,我走过一遍之后有几条实操建议:

第一,从"评分表"开始,不要从"学习清单"开始。 先搞清楚自己现在什么水平,再决定学什么。很多人一上来就列"今年要学 20 个技术",结果根本不知道哪些是已经会的、哪些是真正缺的。120 多项技能听起来多,其实按方向分组后,每个方向十几到二十几项,一两个周末就能测完第一轮。

第二,评分必须严格,不要给自己放水。 这条我怎么强调都不过分。60 分和 80 分的差别不是 20 分,是"还需要学"和"可以上生产"的差别。放水只会让学习清单排错优先级。

第三,错题一定要归档,而且格式要固定。 题目 / 错误答案 / 正确答案 / 为什么错 / 掌握标志,五个字段缺一不可。“为什么错"这一条最重要——它逼着你反思是"不知道"还是"知道但想错"还是"混淆了”,这才是真正能改掉的根因。

第四,不要一开始就想做大而全。 我这套系统也是跑了一个多月才稳定下来的,一开始只是几个 Markdown 文件 + 一两个命令。先让"评分 + 错题 + 日志"跑起来,再慢慢加市场扫描、趋势图、综合考试。一步到位是不现实的。

第五,坚持用 AI 工具本身的约束能力,而不是靠自觉。 把规则写在 AGENTS.md 里,写在命令的 prompt 里,让 AI 每次启动时自己读、自己遵守。人都会偷懒,AI 不会——只要你把规则写得足够明确,它会严格执行。这一点是 AI 时代学习和过去最大的不同:我们终于有了一个比我们自己更守规矩的学习伙伴。

第六,接受一开始的数据会"很难看"。 我第一次跑 /skill-test 前端 的时候,24 项技能里,80 分以上的一项都没有,40 分以下的有 19 项,最高分才 60。那种感觉很糟——你做了十年前端,结果被一张表打回原形。但正是这种"难看",才是这套系统最大的价值。它逼着你正视"我会的那些其实都是皮毛",逼着你把时间花在真正薄弱的地方,而不是继续在自己熟悉的领域刷存在感。

第七,把"市场信号"和"自我评估"两个输入分开处理。 市场信号(招聘网站的 JD、社区热点)告诉你"该学什么",自我评估(测试分数、错题)告诉你"现在在哪"。这两条线不能混。混在一起就容易产生"市场火的我就一定要精通"的焦虑,或者"我擅长的就一定有价值"的自欺。分开了,你才能理性判断:市场火但我不感兴趣,可以只了解到概念;市场不火但我擅长,可以作为差异化优势。学习策略的成熟,是学会同时听两个声音,但不被任何一个声音牵着走。

回过头看,古法学习的核心问题不是"不够努力",而是"没有反馈回路"。我们以为自己在学,其实只是在消费内容;我们以为自己会了,其实只是看懂了;我们以为自己在进步,其实只是在原地打转。

AI 时代的学习,不是用 AI 替我们学,而是用 AI 把"学"这件事变成一个可测量、可反馈、可迭代的工程系统。市场数据告诉你该学什么,测试数据告诉你现在什么水平,错题数据告诉你哪里薄弱,趋势数据告诉你进步了多少。所有数据都在本地,所有反馈都在分钟级,所有迭代都有据可查。

这不是方法论上的小修小补,是学习这件事本身被工程化重构了。你愿不愿意花一个周末,把自己的"学习"也搭成这样一个系统?我的建议是:值得。

不需要一步到位,也不需要完全照搬我的设计。从最简单的"评分表 + 错题本 + 学习日志"三个文件开始,跑起来,让数据沉淀下来。一个月后你会发现自己比想象中薄弱很多,但也会发现进步比想象中快很多。这两件事同时发生,才是学习真正的样子。
