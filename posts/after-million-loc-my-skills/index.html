2026 年 6 月,线上数据丢了一批。

一个定时清理任务,逻辑是「扫整个对象存储,凡是数据库里没记录的,就当孤儿删掉」。听着合理,对吧?存储里有、库里没有的,可不就是垃圾。直到某天一个写入链路抖了一下——文件先落库、对象后上传,中间这几秒,定时任务扫过去了,「库里有、存储还没到」的对象被当成孤儿删了个干净。

这是我维护的一个多租户社交电商全栈项目。前端 uni-app(Vue 3)小程序、后端 Spring Boot 3 + MyBatis-Plus + MySQL + Redis + RabbitMQ、运营后台 Vue 3 + Element Plus。代码量早就过百万行,而其中绝大部分是 AI 写的。

事故复盘的时候我没怪模型。我怪的是:我给了 AI 一把没有保险的枪。它执行得又快又准,快到我没来得及看它要删什么。

百万行代码之后,我真正想明白一件事:瓶颈不在模型够不够聪明,在我怎么把工程纪律变成 AI 逃不掉的流程。我选的方式,是在项目里沉淀一套 Skills。

这篇就讲这套东西:我实际在用哪些 skill、每个在项目里管什么、以及它们是怎么装进来的。

一、先说清楚:Skill 不是教 AI 新知识

很多人对 Skill 有个误解,觉得它是「给模型补充训练数据里没有的知识」。不是。

模型的知识已经够多了。它知道 Spring Boot 怎么写、知道 Flyway 怎么迁移、知道 RBAC 怎么设计。它缺的不是知识,是约束——是你的项目里那套「必须先写测试」「DELETE 必须带 tenant_id」「声称完成前必须贴证据」的纪律。

Skill 干的就是这个:把「你应该怎么干活」写成一份可复用的指令,在合适的时机自动加载进会话。它不增加模型的能力上限,它抬高的是模型在你项目里的下限。

这是为什么我不去追「更强的模型」,而去攒「更密的 Skill 网」。模型每代都在变强,但你项目里那个「AI 一高兴就 DELETE FROM 不带 where」的风险,换什么模型都在。能焊死这个的只有流程,不是参数量。

二、这套 Skills 是怎么装进来的

先讲安装,因为这是最容易被问到的。我不是手写了几十个 skill——那不现实,也违背「复用」的初衷。项目里的 skill 来自一套类似 npm 的 skill 包管理器。

核心是项目根的 skills-lock.json,长得跟 package-lock.json 一个味道:

{
  "version": 1,
  "skills": {
    "test-driven-development": {
      "source": "obra/superpowers",
      "sourceType": "github",
      "skillPath": "skills/test-driven-development/SKILL.md",
      "computedHash": "126f1ebf6ccd4143..."
    },
    "security-review": {
      "source": "everything-claude-code",
      "sourceType": "local",
      "computedHash": "9814020fb226bff5..."
    }
  }
}

每个 skill 锁了三样东西:来源(source)、来源类型(sourceType)、内容哈希(computedHash)。

来源主要分三类,对应我用的三个上游生态:

  • superpowers(obra/superpowers)——工程纪律派:TDD、系统调试、写计划、验证后完成、git worktree。这套是我用得最重的,后面会展开。
  • everything-claude-code(ECC)——技术栈模式派:Spring Boot、JPA、API 设计、数据库迁移、Docker。按技术栈给最佳实践。
  • mattpocock/skills、caveman——单点工具派:诊断、原型、PRD 转换这类小而锋利的工具。

装的时候,管理器把每个 skill 的真实内容放到 .agents/skills/<name>/SKILL.md,再在 .claude/skills/<name> 建一个符号链接指过去。这样 Claude Code 启动时能按它自己的机制发现 skill,但内容的「真相源」始终是 .agents/ 那份,带哈希校验——谁要是手改了 skill 内容,下一次校验就报警。

装一个新的第三方 skill,流程就三步:在 lock 里登记来源、跑同步、链接挂上。跟你 npm install 一个包没本质区别。区别只在于——装 skill 装的是「行为约束」,不是「函数库」。

三、我在用的核心 Skills

项目 .claude/skills/ 下挂了一百来个,但真要讲「我在用的」,得砍掉大半。很多是装来备着、低频触发的。下面这套是我几乎天天会碰到的,我按一个需求从立项到上线的顺序来讲,这样你能看清每个 skill 卡在流程的哪个位置。

A. 动手之前——把方向对齐

这一组是最容易被跳过的,但它决定了后面百万行是不是白写。

brainstorming。这个 skill 的描述写得很硬:「在任何创造性工作之前必须使用」。它强制 AI 在动手前先跟你把意图、需求、设计聊透,而不是你扔一句「加个分销功能」它就闷头开写。项目做过一个代理商分润的需求,我先用它逼着 AI 把「分润是按单笔还是按周期结算」「退货怎么回滚分润」「多级代理怎么防环」全问清楚。这些问题不问清楚,代码写到一半必然返工。

to-prd / product-capability。聊完把结论沉淀成一份 PRD 或能力清单。product-capability 这个 skill 会逼着把「约束、不变量、接口、未决决策」在动手前就暴露出来。我特别看重它的「未决决策」——很多线上事故,根因是某个决策在编码时被默认掉了,而它本该在立项时就被拎出来拍板。

ubiquitous-language。DDD 统一语言。它会从对话里抽一份术语表,标出歧义、给出规范术语,落到 UBIQUITOUS_LANGUAGE.md。项目是社交电商,「订单」「佣金」「圈子」「代下单」这些词,产品和运营的理解跟代码里的领域模型经常对不上。这个 skill 就是治这个的——术语不统一,就是 bug 的温床。

grill-me。这个我最喜欢。它「像面试官一样拷问你」,把方案的每个决策分支逼到死角,直到你和 AI 达成一致。用它的感觉是:你以为想清楚了,它一问发现全是窟窿。

这一组的共同点是:让 AI 在写第一行代码之前,先帮你把方向收敛。投入产出比极高——花半小时对齐,省掉后面三天的返工。百万行代码的浪费,九成发生在方向没对齐就开干。

B. 写代码时——把纪律焊死

方向对了,进入编码。这一组是防止 AI 把代码写得「能跑但经不起推敲」。

test-driven-development(配合项目的 /tdd 命令)。红→绿→重构,失败按项目规范排查。项目后端是 Spring Boot,我把 TDD 设成硬要求,不是「建议」。原因很简单:AI 写的代码,你不写测试,你拿什么信它?测试不是验证 AI 写对了,是给 AI 的输出兜底——它写错了,测试先红,你才有机会在合并前发现。

executing-plans。它有个铁律我深以为然:「每一步结束都有一个验证门,门没过,不进下一步」。这治的是 AI 的一个通病——它特别爱「一口气把十步都做了,然后告诉你搞定了」。每步一个验证门,就是把进度切成可检查的小块。

ecc 的技术栈模式组:springboot-tdd、jpa-patterns、springboot-patterns、springboot-security、database-migrations。这几个按项目的后端栈定制,AI 写后端代码时会自动对齐项目模式——JPA 实体怎么设计关系、Flyway 迁移怎么做零停机、Spring Security 怎么鉴权。它的价值是统一风格:十个 agent 写出来的代码,读起来像一个人写的,而不是十种花式实现。

项目定制的两个:component-reference(组件库参考,前端实现前必须先查可复用组件)和 ui-spec-enforcer(UI 实现标准强制器)。这俩是 ECC 通用 skill 之外的「项目私有 skill」,把项目 v1.0 的组件库和 UI 规范固化进去。AI 写前端就不会再凭空造一个差不多的轮子。

C. 防 AI 自欺——验证与审查

这是百万行代码敢上线的底气所在。也是事故之后我补得最狠的一块。

verification-before-completion。描述一句话点透:「声称工作完成、修复或通过之前,必须运行验证命令并确认输出」。它治的是 AI 最致命的毛病:没跑就说跑通了。项目的 CLAUDE.md 里把这条上升成了硬约束,任何 agent 声称完成,必须贴出验证命令的真实输出,而不是「应该没问题」。

ai-regression-testing。这个 skill 特别有意思,它专门解决「同一个模型既写代码又审代码」的盲区——自己写的 bug,自己审的时候大概率也看不见。它的策略是沙箱模式 API 测试、自动化 bug 检查,专门抓 AI 的系统性盲点。

批量审查循环。这是我在项目命令里封装的,底层调 requesting-code-review。它的玩法是:并行派发约 21 个窄范围的审查 subagent,每个只盯一个维度(安全、事务边界、租户隔离、测试覆盖……),结果一到就派修复 subagent 去改,持续循环直到 🔴 🟡 🔵 三类问题全部清零才收口。这是「人审」根本覆盖不到的颗粒度——21 个视角并行扫一遍,远比一个人从头读到尾可靠。

receiving-code-review。容易被忽略但很重要:收到 review 意见时,不盲目同意、也不盲目反驳,而是「技术严谨地验证」。AI 特别爱在收到 review 时表演性同意——「您说得对,我改」,然后改出一堆新问题。这个 skill 让它在改之前先判断这条意见对不对。

这一组的底层逻辑是:不要相信 AI 的自我汇报。它的「我完成了」永远只是候选状态,跑了验证、过了审查、贴了输出,才算真的完成。没有证据链的 done,等于没 done。

D. 排障与诊断

systematic-debugging。任何 bug、测试失败、异常行为,在提出修复之前先用它。治的是 AI 的另一个通病:看到报错就猜一个原因,改一通,没好,再猜一个。systematic-debugging 强制走结构化诊断,而不是 shotgun debugging(霰弹枪式乱改)。

diagnose。更具体的诊断循环:复现 → 最小化 → 假设 → 插桩 → 修 → 回归测试。项目那次数据丢失,事后我让它走了一遍 diagnose:先在测试库复现「库有存储无」的时序,再最小化到能稳定触发的 case,才敢动修复方案。没复现就改,跟没验证就说完成,是一样危险的。

这两个 skill 的配合顺序不能反。systematic-debugging 先逼着把「现象是什么、我怀疑是什么、凭什么这么怀疑」摆清楚,假设站得住脚了再进 diagnose 跑复现循环。我见过直接跳进 diagnose 的坑——没想清楚假设就一遍遍改插桩代码,改了半天才发现方向一开始就错了。结构化诊断的好处不在更快,在它强迫你先把怀疑说出口:一句话讲出来,一半的臆想假设当场就漏气。

E. 安全与数据——项目的命门

事故之后,这一组成了硬约束。

security-review。涉及鉴权、用户输入、密钥、支付、敏感功能时强制启用,给一份安全清单。项目有支付、有佣金,这条线碰一次就触发一次。

git-guardrails。用 hook 拦截危险 git 命令——push -f、reset --hard、clean、branch -D,在它们执行前拦住。AI agent 在多模块仓库里一不留神就会跑破坏性 git 操作,这个 skill 把保险加上。

还有一类不是某个 skill,而是项目把教训直接写进了 CLAUDE.md 的硬约束:DELETE 三必查(租户隔离 + 业务键 + 范围)、定时任务必须有 dry-run 开关和删除上限、禁止 jdbcTemplate 做写操作。这些规则是从真实事故里长出来的,skill 负责执行,CLAUDE.md 负责定义。skill 是肌肉,CLAUDE.md 是骨头。

F. 多 agent 并行——百万行的吞吐策略

百万行代码不是一次会话写出来的,是无数次会话堆出来的。要提吞吐,就得并行。

dispatching-parallel-agents:遇到 2 个以上互不依赖、没有共享状态的任务,并行派出去。项目改一个功能经常同时动前端、后端、运营后台,这就是天然的并行机会。

subagent-driven-development:用 subagent 执行计划里的独立任务,主会话做协调。using-git-worktrees:保证每个并行任务有隔离的工作区,互不踩脚。handoff:把当前会话压成一份交接文档,让下一个 agent 接着干。

这一组解决的是:怎么让多个 AI agent 像一个团队一样协作,而不是互相覆盖。隔离、分派、交接,三件事。

这组里 worktree 的隔离最容易被低估。并行 agent 不隔离工作区,最典型的事故是两个 agent 都改了同一个文件,后提交的把前面的冲掉,AI 还自信地汇报「全部完成」。worktree 让每个 agent 在独立分支上干,合并时再人工把关,覆盖就不会发生。并行不是把活儿切开就完事,切开之后还得保证它们互不知道彼此的存在。

如果你也想给自己的项目挂 skill,但不知道该挂哪些,这张决策树是我用了一年多之后总结出来的——按「现在要干的事属于哪一类」对号入座:

四、装 skill 的 5 个反模式

讲完正确做法,有必要说说哪些做法不算正确——我在团队内外见过太多伪装成「我们在用 skill」的反模式,看起来装了一堆,实际工程纪律一点没变。

反模式一:装而不用,当收藏品。 一个项目里挂了 80 个 skill,但触发频率前 10 的占了 90% 调用量,其余 70 个一年没人碰一次。这是把 skill 当 GitHub star 在收藏——装上去心理上「我用了 AI 工程化」,实际纪律一点没变。判断标准:如果一个 skill 三个月没被触发过,要么删掉,要么它本来就不该挂。

反模式二:追新模型,不攒 skill。 团队每天讨论「Claude 4 出来要不要切」「GPT-5 出来谁先用」,但项目里连一个 verification-before-completion 都没装。模型每代都强,但你项目里「AI 一高兴就 DELETE FROM 不带 where」的风险,换什么模型都在。追模型是消费,攒 skill 是投资——前者享受新鲜感,后者沉淀资产。

反模式三:CLAUDE.md 和 skill 职责混淆。 把所有规则都塞进 CLAUDE.md 当 system prompt(上下文爆炸、规则漂移),或者反过来把该写在 CLAUDE.md 的硬约束散落到各个 skill(碎片化、难追溯)。正确分工:CLAUDE.md 写「绝对不能违反的底线」(DELETE 三必查、禁止裸 jdbcTemplate 写),skill 写「某种活儿该怎么干」(TDD 流程、debug 步骤)。前者是骨头,后者是肌肉。

反模式四:sub-agent 不注入协议。 主 agent 守了 PUA / verification 纪律,但 spawn 出去的 sub-agent 裸奔——主会话汇报「全部验证通过」,实际子 agent 一个测试都没跑。多 agent 协作,纪律必须穿透到每一层。我的做法是 dispatching-parallel-agents 强制在 prompt 里注入行为协议,子 agent 没注入就不允许派出去。

反模式五:按「看着不错」装,不按风险点装。 看到博客推荐一个 skill 就装一个,完全不考虑自己项目真正怕什么。社交电商项目最怕数据事故和租户越权,所以安全 / 验证 / 审查这组是重投入;设计探索类按需留。装 skill 的优先级,应该由你项目的历史事故决定,不是由博客推荐决定。

这五个反模式的共同点是「形式到了,实质没到」——看起来在用 skill,实际工程纪律一点没变。识别它们比记住正确流程更重要。

五、说点实话:有些 skill 我装了但很少用

为了不让你以为「装得越多越好」,得交代一下。

impeccable(前端设计精修)、benchmark(性能基线)、caveman(压缩)这些,装了但触发频率很低。有些是场景不对——项目前端有 ui-spec-enforcer 兜着,impeccable 的发挥空间就小;有些是还没排上用场。

装一个 skill 是有成本的:它占注意力、占上下文、还可能在你不希望的时候误触发。我的原则是:按项目的真实风险点装,不按「这个看着不错」装。项目最怕的是数据事故和 AI 自欺,所以验证、审查、安全、调试这几组是重投入;设计类、探索类就按需留着。

判断一个低频 skill 该不该删,我有个简单的办法:翻它最近三个月的触发记录,数不出来具体场景就先摘掉,放进候选列表观察一个月。摘掉之后如果某个需求做起来明显变卡,再加回来;如果风平浪静,说明它本来就是心理安慰。比起「装了再说」的囤积,这种「先摘后看」的反向筛选,更能逼出哪些 skill 真在干活。

别把 skill 当收藏品。它是工具箱里的扳手,不是陈列柜里的奖杯。

六、skill 上线前自检 checklist

把这套东西收敛成一份可拷贝的 checklist,你可以直接用来给自己的项目做体检:

  • 是否有 skills-lock.json(或等价物)锁定每个 skill 的来源和哈希?
  • 验证类 skill(verification-before-completion)是否被 CLAUDE.md 上升为硬约束?
  • 安全类 skill(security-review、git-guardrails)是否覆盖了 DELETE / 定时任务 / 危险 git 命令三类风险?
  • 多 agent 协作时,行为协议是否强制注入到 sub-agent?
  • 是否有「按真实风险点装」的清单(而不是「看着不错就装」)?
  • 三个月没触发的 skill,是否有定期清理机制?
  • CLAUDE.md(骨头)和 skill(肌肉)的职责是否清晰区分?
  • 团队是否共识过「追模型不如攒 skill」这个判断?

八条全部勾上,才算「skill 工程化」真正落地。少一条,就还在反模式的边缘。

七、skill 体系的演进路径:从散装脚本到肌肉记忆

如果你也想在团队里推这套,别一上来就铺全套——会把自己压垮。我们走过四个阶段,每个阶段解决一个具体痛点,后一个阶段以前一个为前提。这条路径可以照抄:

阶段一:散装脚本(0 → 1)。 这个阶段团队还没有 skill 概念,只有一些散落在 scripts/ 下的 shell 脚本(pre-commit hook、lint 脚本、build 脚本)。重点是先让最痛的那条规则机器化——通常是「禁止危险 git 操作」或「提交前必须跑测试」。一条规则机器化,就比口头约定强十倍。

阶段二:单条命令(1 → 命令族)。 把散装脚本收敛成几个有名字的命令(/tdd、/verify、/review),让 AI 能识别并主动调用。这个阶段重点是命名清晰、退出条件明确——命令名要短到 AI 一看就知道干嘛,退出条件要硬到「没过不算完」。我们在这个阶段吃过亏:有个 /check 命令没写退出条件,AI 永远回复「差不多了」,等于没检查。

阶段三:命令族 → harness + loop。 命令多了之后,需要一套 harness 串起来——plan → tdd → verify → review → push 形成闭环,前一个的输出是后一个的输入。这个阶段重点是让流程自循环:kd-review 这种「挖 → 修 → 再挖」的循环,能让 AI 自己收敛到质量达标,不需要人盯着每一步。

阶段四:loop 闭环 → skill 网覆盖。 流程跑顺之后,把每一步的「怎么干」沉淀成 skill,挂到 skills-lock 里统一管理。这个阶段的标志是新人 / 新 agent 进项目,不用读文档就能按规矩干活——skill 自动加载,纪律自动生效。我们到这个阶段才敢说「AI 在我们项目里敢放手干」。

这四个阶段不能跳。跳过阶段一直接铺 skill 网,会变成「形式到了实质没到」——skill 挂了一堆,但团队连「为什么要有这条规则」都没共识,出事时还是手动救火。每个阶段平均要踩一两个真实事故才会真正沉淀下来,急不得。

演进的标准:不是「我们装了多少 skill」,是「我们项目最怕 AI 在哪翻车,那个点有没有兜住」。前者是收藏家思维,后者是工程师思维。

八、底层逻辑

写到最后,我捋一下这套东西到底在干什么。

百万行代码能稳,不是因为模型一代比一代强。模型强,只会让「写错代码」变得更快。真正托底的,是把三件事变成 AI 逃不掉的流程:约束(动手前对齐方向、写代码守纪律)、可验证(声称完成前必须贴证据、过了审查才算数)、隔离(并行不互踩、危险操作有保险)。

Skills 是我把这三件事固化下来的载体。它解决的不是「AI 会不会」,是「AI 在你的项目里,敢不敢让它放手干」。纪律越密,你越敢放权;越敢放权,吞吐越高。这是个正循环。

回到开头那场事故。后来重写清理任务时,我让它走了 security-review,加了 dry-run 开关,改成「只删本业务刚创建的对象、带业务前缀白名单」。逻辑变了,但更重要的是——那把枪,这次上了保险。

如果你也在用 AI 大规模产出代码,我的建议就一句:别急着追更强的模型,先想清楚你的项目最怕 AI 在哪翻车,从那个点开始沉淀你的第一个 skill。

剩下的,都是水到渠成的事。急不得,也省不掉。
