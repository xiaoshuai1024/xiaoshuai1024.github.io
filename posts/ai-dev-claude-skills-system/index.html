把"用 AI"做成工程体系:28 个命令和一条自治评审回路

前三篇讲的都是「用 AI 做出了什么」。这篇回头讲「怎么用 AI」——不是技巧,是体系。

很多人对「AI 协同研发」的理解,还停在「每个人装个 Cursor,各写各的」。这种用法,4 个人撑死比以前快一倍。我们能做到 2 个月近百万行,是因为把 AI 协同沉淀成了一套可复用的工程体系:命令化的工作流、自治的评审回路、规范化的技能库。

关键不是 AI 多强,是这套体系让每个 Agent 开机即用。

这篇把这套体系拆开讲三块:命令化工作流(28 个命令怎么设计、为什么这么设计)、自治评审回路(它是体系的心脏,怎么兜住百万行的质量)、规范固化(怎么让每个 Agent 一进来就知道规矩)。最后聊聊这套体系的本质——它解决的核心问题是「怎么让 4 个人的经验,放大到指挥一群 Agent 还不失控」。这套体系不是一天建成的,是项目跑下来一边做一边沉淀的,这里讲的是沉淀后的终态。

一、不是「每个人会用 AI」就够

先说个反直觉的判断:「团队里每个人都会用 AI 工具」不等于「团队有 AI 协同能力」。

差别看两个场景。

没体系的用法:每个人各自调 AI,提示词五花八门,生成的代码风格不一,质量靠个人经验兜底,新人和 AI 都不知道项目规范,反复返工。

有体系的用法:PR 有标准命令、评审有自治回路、规范固化在仓库里、E2E 一条命令跑通,新人第一天装上工具就能产出符合规范的代码。

我们追求的是后者。这套体系跑起来,才是产能能持续放大的根,它让「4 个人 + AI」不只是 4 个独立单元,而是一支能协同的队伍。

这两者的差距,在项目规模小的时候看不出来——3 个文件的小项目,有体系没体系都能做。但项目一大(几百个文件、几十个模块),没体系的用法就会崩:代码风格分裂、规范执行不了、新 Agent 反复返工、质量全靠人肉兜底兜不住。体系的价值是随规模放大的:项目越大,体系的杠杆越大。所以我们不是「一开始就建体系」,是「项目大到一定程度,不建体系就跑不动了」才逼出来的。

还有一点要诚实:建体系是有成本的。28 个命令、评审回路、规范固化,这些东西不是免费的,前期投入的时间不少。但这个投入是一次性的——建完之后,第二个、第三个项目都能复用(命令和规范的思路是通用的,只是具体内容要按项目调)。所以体系的投入是「前重后轻」,熬过前期的沉淀期,后面就能享受复利。怕的是团队等不到复利就放弃,觉得「建体系太慢不如直接写代码」,结果永远停留在「每个人各写各的」的低效状态。

二、命令化工作流:28 个命令覆盖全流程

我们把高频操作固化成了 28 个自建命令(/command)。流程上大致是:/prd 出需求 → 编码 → /tdd 测试驱动 → /mp-e2e、/op-e2e 跑端到端 → /kd-review 自治评审 → /pr-fullstack 提 MR → /deploy-prod-be-ops 部署。

这 28 个命令不是一开始就有的,是项目跑下来一个个沉淀的。最早只有几个基础命令(提交、部署),后来发现某个环节反复出错或反复靠人肉,就固化成一个新命令。比如 /flyway-squash 是因为迁移脚本累积太多手动合并容易出错才做的;/kd-review 是因为人 review 跟不上 AI 的产出速度才做的。每个命令背后都有一个「为什么需要它」的故事,不是拍脑袋决定要 28 个。

命令覆盖的是「高频且标准化」的操作。低频的、一次性的操作不固化成命令(写了也用不上几次);高度个性化的操作也不固化(每次情况不同,命令兜不住)。28 个命令覆盖的是「几乎每天都会做、且流程固定」的操作——这正是工程化收益最高的部分。

为什么把操作固化成命令,有三个理由。

第一是一致性。「提 MR」这件事,以前每人一套 git 流程,现在 /pr-fullstack 一条命令,产出统一。AI 也用这条命令,不存在人和 AI 流程割裂。

第二是可审计。命令背后是脚本(/scripts/pr/publish-to-dev.sh),干了什么、什么顺序,白纸黑字。出了问题能复盘,不是「我当时怎么操作的来着」。

第三是降门槛。新人不用学全套流程,记住几个命令就能干活。知识从「装在脑子里」变成「存在命令里」。

这个「降门槛」的收益在 AI 时代被放大了。因为 AI 协同里,每天都有新的 Agent 加入(每次开 subagent 都是「新成员」),如果流程靠「装在脑子里」,你得给每个新 Agent 都讲一遍——这不可能。但流程固化成命令后,新 Agent 只要会「调用命令」就行,命令背后的流程、规范、约束都打包好了。所以命令化不只降低人的门槛,更降低了「高频 onboard 新 Agent」的门槛,这是 AI 协同能跑起来的前提。

举两个真实命令的设计。

/pr-fullstack 只提交主仓库并创建合并到 dev 的 MR。命令描述里有一句硬约束:只产生 MR,禁止直接推送 dev。直接推 dev 是事故源,必须走 MR 加评审。这种约束写进命令,比写进文档有效——人会忘,命令不会。

把约束写进命令有个隐性好处:约束会随命令传播。一个新人或新 Agent 学会 /pr-fullstack,它就自动继承了「禁止推 dev」这个约束,不用额外教育。命令是约束的载体,学命令等于学规矩,一步到位。如果约束写在文档里,学命令的人不一定会去读文档,约束就传播不到。

/flyway-squash 是上一篇讲的迁移合并,固化成命令。合并历史、重置历史表、删旧文件,高风险操作一条命令编排,人审关键节点。

高风险操作固化成命令有个原则:命令做执行,人做决策。/flyway-squash 跑的时候,合并哪些文件、怎么重置历史表,这些是确定的,命令自动做;但「现在该不该 squash」「squash 后验证什么」,这些需要判断,留给人审。命令把机械的执行包圆,把需要判断的节点暴露给人,这种分工让高风险操作既高效又可控。

这些命令的共同点是:把「需要经验才不出错」的操作,变成「按按钮就行」。这才是工程化。

三、自治多 Agent 评审回路:体系的心脏

评审这块,是整个 AI 协同体系最核心的设计,也是研发账本篇里说的「AI 省了执行,逼我们补兜底」的具体落地。

先说问题。AI 写得快,但快不等于对。幻觉、编造的 API、看似能跑实则埋雷的代码,如果靠人 review,4 个人根本 review 不过来百万行。所以必须用 AI 审 AI。

/kd-review 的核心思路是:把变更拆成约 20 个窄范围 bucket,全部并行审查,第一个结果回来就立刻派修复,审查和修复重叠执行,循环直到零红色问题收敛。

几个关键设计。

拆成窄范围 bucket,不是整体 review。一次性丢整个 diff 给 AI,它会漏。拆成「这个 Controller 的变更」「这段 SQL 迁移」「这套测试」等窄范围,每个 subagent 聚焦,准确率更高。

bucket 怎么切是个设计问题。切太粗(整个后端一桶),等于没切,AI 还是会漏;切太细(每个文件一桶),bucket 数量爆炸,调度开销大。我们实测的甜区是按「模块 + 变更类型」切——后端 Controller/Service 一桶、Mapper/SQL 一桶、前端组件一桶、迁移脚本一桶,一次大改动切约 20 个桶。这个粒度让每个 bucket 的变更量在 AI 能聚焦的范围内(30-50 文件),又不至于 bucket 太多调度不过来。

并行加流式。约 20 个 bucket 同时跑,哪个先回来就先处理,不等齐。这是压缩时间的关进——串行 20 轮会等到天荒地老。

问题分级收敛:红色必须清零,黄色尽量清,蓝色提示。不是无限 review 下去,是奔着「零红色」收敛,有明确退出条件。

分级收敛的设计有个考量:不同级别的问题,容忍度不同。红色(阻塞级,如资金用浮点、漏 tenant_id)是零容忍,不修完不算过;黄色(一般级,如命名不规范)是尽量修,修不完也能过但有记录;蓝色(建议级,如可以提取公共方法)只是提示,不强制。这个分级让评审回路「抓大放小」——把精力集中在会出事的问题上,不被鸡毛蒜皮拖住。如果所有问题一视同仁(都要修完才能过),回路会被黄色蓝色问题淹没,反而拖慢收敛。

退出条件(零红色)的设定也讲究。如果设成「零问题」(连黄色蓝色都清零),回路会无限循环——因为蓝色建议级问题总有改善空间,永远清不完。设成「零红色」是务实的:红色代表会出事的问题,清零了就安全了;黄色蓝色是优化空间,可以后续慢慢做。退出条件要可达成,不可达成的退出条件等于没有退出条件。

审查和修复重叠。不是「审完再修」,而是「审完一个,立刻派修那个,同时继续审别的」,流水线化。

这套机制跑起来,代码质量不再依赖「人愿意不愿意认真 review」,而是被自治回路兜住。这才是「4 个人能撑住百万行」的真正保障。

这套评审回路还有个被低估的价值:它是「持续的」。人 review 是间歇的——人要吃饭、要开会、要写代码,一天能 review 的 PR 有限;自治回路是持续的——只要在跑,就在 review,不占用人时间。这意味着同一个改动量,人 review 会成为瓶颈(4 个人 review 不过来百万行),自治回路不会(它能 7x24 跑)。所以自治回路不只提升了 review 质量,更消除了「review 是瓶颈」这个产能限制。

这套思路我之前在没有 CI 的日子里那篇系统讲过,这里侧重它在工作流里的位置。

四、规范固化:让每个 Agent 开机即用

命令和评审是流程,规范是地基。我们把项目规范全部写进了仓库,让任何人、任何 Agent 进来都开机即用。

这块我在把项目规范写进代码底层那篇详细讲过,这里点几个真实的硬规范。

AGENTS.md / CLAUDE.md 双约束。项目根有两份约束文件,合计 550 行。最前面就钉了两条硬约束:信息与代码必须真实,禁止推测和假信息——不允许凭空推测 API 参数、编造报错、用占位逻辑冒充实现,违反者驳回;微信接入代码必须经过文档验证——涉及微信(支付/登录/分账)的代码,必须先查官方文档确认 API 存在,对照已有集成风格,不确定的标注「需查阅最新文档」。

为什么这两条放最前。因为 AI 最容易在这两件事上翻车:编造信息和过时 API。把它提到硬约束的位置,比埋在文档深处有效,这是踩坑后的优先级排序。

规范的优先级排序是个工程判断。550 行规范,不可能都同等重要,必须排优先级——最致命的(编造信息、资金安全)放最前、最常犯的(命名、分层)放中间、最细节的(注释格式)放后面。Agent 加载规范时,前面的优先级更高,即使上下文有限只读到了前半部分,也抓住了最重要的约束。如果不排序,550 行平铺,Agent 抓不住重点,规范的执行效果会打折扣。

E2E 命令分场景。/mp-e2e(小程序端到端)和 /op-e2e(运营后台端到端)分开——两个端测试策略不同,混在一起跑会互相干扰。这种细分是实战出来的。

五、技能库:站在社区肩膀上

最后说技能库。我得诚实:.claude/skills/ 里有 74 个技能,但大部分是社区或第三方技能库(像 p7/p9/caveman/brainstorming/test-driven-development 这些都是成熟的社区技能),不是我们从头写的。

我们做的事不是「造 74 个轮子」,是「筛选 + 整合 + 配合自建命令」。这一点必须讲清楚,否则就是不诚实。

技能库的管理本身也有讲究。74 个技能不能一股脑全加载,那会撑爆 Agent 的上下文。我们的做法是按需加载——Agent 启动时只加载总入口和启动清单,具体技能在「用到时」才检索加载。比如 Agent 要写测试时,才加载 test-driven-development 技能;要画图时,才加载绘图技能。按需加载让 74 个技能不会成为负担,而是「随用随取的工具箱」。

技能的筛选标准是「这个技能解决的是不是通用问题」。通用问题(怎么写 TDD、怎么做 code review)用社区成熟的;项目专属问题(怎么走我们的 MR 流程)自建。这个筛选举措让技能库既丰富(社区贡献的通用能力)又聚焦(自建的项目专属能力),不会重复造轮子也不会缺失关键能力。

真正自建、体现工程化能力的是三块。

28 个自建命令(前面讲过)——这是项目专属的工作流,社区没有。

自治评审回路(/kd-review)——这套收敛逻辑是针对我们项目设计的。

配套脚本(scripts/pr/、scripts/contract-check.sh、E2E 编排)——命令背后的实现。

社区技能负责通用能力(比如 TDD、debugging、git worktree 这些方法论),自建命令负责项目专属流程。两层叠加,才是完整的 AI 协同体系。盲目追求「我们自建了多少技能」是虚荣指标,真正该问的是「这套体系让团队产出提升多少」。

这个诚实区分(自建 vs 社区)本身就是工程化的态度。工程化不是「什么都自己造」,是「分清哪些该自建、哪些该复用」。通用方法论(怎么写 TDD、怎么调试)社区已经做得很成熟,自己重写是浪费;项目专属流程(怎么提 MR、怎么评审、怎么部署)社区没有,必须自建。把精力投在「社区没有而项目需要」的地方,才是高效的做法。反之,什么都想自己造,既做不好通用部分(不如社区成熟),又分散了做项目专属的精力,两头不讨好。

判断哪些该自建,有个简单标准:这件事换个项目还适用吗。适用(比如 TDD 方法论)→ 用社区技能;不适用(比如我们项目的 MR 流程)→ 自建。按这个标准筛,自建的清单就聚焦了,不会陷入「什么都自建」的虚荣。

六、体系的本质:把经验外化成机器可执行的流程

回头看整套体系,它解决的核心问题是:怎么让 4 个人的经验,放大到指挥一群 Agent,还不失控。

经验如果只装在脑子里,4 个人就是 4 个独立单元,加 AI 也只是各快各的。但如果把经验外化成命令、评审回路、规范约束,就能驱动一群 Agent 统一产出。经验从「个人的」变成「团队的杠杆」。

这个「经验外化」的过程,本质上是把隐性知识变成显性资产。老兵脑子里的「该怎么做」是隐性的——他知道提 MR 要走什么流程、知道评审要盯什么、知道哪些操作危险,但这些知识没写出来前,只有他自己能用。外化成命令和规则后,这些知识变成了显性的、可被任何 Agent 调用的资产。一个老兵的经验,原来只能服务他自己;外化后能服务一群 Agent,杠杆就放大了。

经验外化还有个长期收益:它让经验「可传承」。老兵会离开、会遗忘,但固化成命令和规则的经验不会。新人来了用同样的命令、遵守同样的规则,等于继承了老兵的经验,不用「老带新」的口口相传。这在人员流动的团队里价值巨大——经验不随人走,留在体系里。

这也是为什么我反复讲老兵 + AI 的杠杆论:AI 是杠杆,但支点是这套体系,而体系来自经验。没有经验沉淀成的体系,给你最强的 AI,你也只是个更快的「手」。

展开讲这个「杠杆 + 支点」的比喻。AI 是杠杆——它放大你的执行能力,让你一个人干十个人的活。但杠杆要发挥作用,需要一个支点——这个支点就是体系(命令、评审、规范)。没有支点,杠杆再长也撬不动东西(你有 AI 但没有体系,AI 各写各的,产出无法统一);有了支点,杠杆才能把你的经验放大成一群 Agent 的统一产出(你的经验通过体系传给所有 Agent)。所以老兵的价值不只是「会用 AI」,是「能搭建让 AI 发挥作用的体系」——这个体系本身就是老兵经验的结晶。

换句话说,体系的厚度决定了 AI 杠杆能放多大。体系薄(规范少、命令少、评审弱),AI 杠杆放大的是混乱——快得更快,错得也更错;体系厚(规范全、命令齐、评审强),AI 杠杆放大的是秩序——快且不乱。所以投资体系不是额外开销,是放大 AI 收益的前提条件,没有这个前提,AI 越强反而越危险,因为放大混乱的速度也跟着变快,产出越多债务越重,最终拖垮整个项目。

到这里,实战系列主体讲完了:账本 → 分润 → 迁移 → 统计 → 体系(本篇)。

这套东西拆给你看,不是为了炫,是想说明一件事:「AI 时代研发」不是玄学,是可以被工程化的。而能把它工程化的人,正是有经验、能把判断外化成流程的老兵。这就是认知转变之后,老兵的红利所在。

最后给个判断:这套体系的可复制性。命令、评审回路、规范固化,这些不是某个项目的特权,是任何「AI 协同研发」的团队都能搭的。区别只在于「命令覆盖哪些流程」「评审盯什么规则」「规范固化多少条」——这些因项目而异,但「把 AI 协同做成体系」这个思路是通用的。所以这套打法能复制,前提是团队里有能拍板的老兵(定规则)和愿意投入前期沉淀的耐心(建体系)。老兵负责把经验外化成命令和规则,耐心负责熬过前期「投入大于产出」的沉淀期。两个前提满足了,任何一个团队都能搭出自己的 AI 协同体系,享受到「老兵 + AI」的产能杠杆。而搭不出来的团队,大概率不是缺工具,是缺老兵(没人定规则)或缺耐心(熬不过前期沉淀)。这两个前提,工具解决不了,只能靠团队自己补齐——而这正是老兵在新周期里最不可替代的价值:工具人人能装,经验体系不是。所以与其焦虑「AI 会不会替代我」,不如把精力投在「把经验沉淀成体系」上,这才是 AI 替代不了的那部分能力,也是老兵越往后越值钱的根本所在。
