最近一个项目 380 个页面,每个页面停留十秒就是一个多小时,还不算记录、复现、回归。第二次改完再走一遍。第三次。靠人走 UI 验收根本不现实。

我的做法是按场景分两条路(和测试纪律、契约闸是一套,前面讲「代码对不对」,这篇讲「长得对不对」):

  • 新功能:有高保真原型锚点,做像素级比对——保证 PRD 里的每个功能都被实现、样式和设计稿一致。
  • 老功能:没有原型可比对,做 vision 巡查 + 代码审计交叉——保证改动不破坏兼容性,顺便扫历史遗留问题。

这两条路用的工具不一样、判断标准不一样、能回答的问题也不一样。混在一起做,要么新功能漏检样式偏差,要么老功能因为没有原型而无从下手。

为什么 UI 走查最难自动化

功能测试有断言,性能测试有指标,UI 走查靠的是人眼——主观、慢、还容易漏。

380 个页面,每个页面停留十秒就是一个多小时,还不算记录、复现、回归。第二次改完再走一遍。第三次。没人扛得住。

丢给 vision 模型批量看图,听起来很美。实际跑下来,它会自信地报错,其中相当一部分是幻觉。直接采信,要么误报一堆不存在的 bug,要么漏掉真问题。

所以问题不是"要不要用 AI",是分场景用对工具。

人眼走查还有个被低估的成本:结论不可复现、不可传承。同一个页面,今天的小张说「可以」,明天的老王说「间距不对」,没有客观标准,验收质量完全取决于当时谁在审。更要命的是,人审过的页面出了问题,你没法回溯——「这个间距上周验收时到底量过没有?」没人答得上来。自动化验收的深层价值之一,就是把「当时谁在看、什么状态算过」固化成可复跑的脚本和可留存的报告,让验收从「一次性的人工判断」变成「可追溯的工程产物」。

这也是为什么我一直坚持验收结果要落成代码里的断言或扫描规则,而不是停留在一份 markdown 报告里。报告会丢、会过时,断言和规则每次跑 CI 都会执行,问题一冒头就拦下来。视觉验收只有沉淀成可执行的东西,才算真正嵌进了研发流程,否则就只是一次性的走过场。

视觉验收三原则

往下展开之前,先把这套方法的底层判断收敛成三条原则。后面所有的工具选择、流程设计,都是这三条的落地:

原则一:分场景用对工具。 有原型就用像素 diff(确定性高),没原型才用 vision(会幻觉)。混用等于把不确定性引入确定性场景,得不偿失。

原则二:真相源必须机器可读。 设计稿不能只活在 Figma 里(人能看,代码扫不到),必须沉淀成 design token——颜色、圆角、渐变都在 variables.css 里定义,代码能直接读。这一层是「正确答案」的物理载体,没有它,代码审计无从谈起。

原则三:vision 只提疑问,不下结论。 vision 的输出永远是「疑点列表」,不是「bug 列表」。每个疑点必须经代码审计交叉验证,才能升级为 bug。这条治的是 vision 的过度自信——它越自信,越要警惕。

这三条是后面所有流程的根。任何视觉验收方案,只要违反其中一条,迟早翻车。

值得强调的是,这三条原则的排列顺序也是优先级顺序。团队资源有限时,该先建哪一层?答案是从原则二(真相源)开始。没有 design token 这层机器可读的基线,vision 巡查(原则三)无处交叉验证,像素 diff(原则一)也没有稳定的颜色锚点。我们项目就是先花了两个版本把全站颜色、圆角、渐变收敛进 variables.css,统一切到 token 引用,之后才谈得上做 diff 和 vision 巡查。基线没建好就上自动化验收,等于在沙地上盖楼,跑出来的结果不可信,反而会让团队对自动化验收失去信心。

原则三(vision 只提疑问)是执行时的纪律红线。vision 越是「信誓旦旦」,越要克制直接采信的冲动。我给自己定的规矩:vision 报的每个疑点,要么有代码证据支持它,要么有代码证据否决它,不允许「没法验证但先信它」这种中间态。没有确定性证据的疑点,一律标记为「待人工确认」,不进 bug 清单。这条规矩挡住了所有追幽灵 bug 的时间浪费。

场景一:新功能 → 像素级比对

新功能的特殊之处是有高保真原型做锚点。设计稿/原型 HTML 就是"标准答案",实现页只要跟它对得上,功能就齐了、样式就对。这条路,像素 diff 是主角。

用 pixelmatch 做逐像素比对

纯算法,没有 AI。用 Playwright 起两个 context,一个渲染高保真原型 HTML,一个跑真实实现页,固定 375×812 视口(小程序 H5)截图,再用 pixelmatch 逐像素比对。

有两个消除噪音的技巧值得拎出来。

图片内容拦截。两张图里的商品图不一样会污染 diff,所以把所有图片请求拦截成灰色块,只比布局:

await page.route("**/*.{png,jpg,jpeg,webp,gif,svg}", (route) =>
  route.fulfill({ body: `<svg><rect fill="#E0E0E0"/></svg>` })
);

主题色断言。直接 poll CSS 变量,而不是从像素推断颜色:

await expect.poll(() =>
  page.evaluate(() =>
    getComputedStyle(document.documentElement)
      .getPropertyValue("--kd-color-primary").trim()
  )
).toBe("#FF3B47");

diff 结果按 strict/standard/loose 三档容差(5%/15%/30%)输出 PASS_A 到 FAIL_D 四级。

三档容差的用途要分清。strict(5%)是给「绝不该有差异」的关键页面用的,比如支付确认页、登录页,这种页面连一个像素的偏移都可能是按钮被遮挡的信号。loose(30%)是给「内容天然多变」的列表页、详情页兜底用的,这些页每个用户看到的内容都不一样,卡太严会天天误报。standard(15%)是默认档。把所有页面都套同一个容差,是新手最常见的错误——要么关键页漏报,要么列表页天天红。

这条路能回答什么

像素 diff 对新功能特别合适,因为它的判断标准很明确:实现就该跟原型一样。出现差异,要么是实现没跟上,要么是设计稿变了没同步——两种都是真问题,都该修。

这条路能回答两个问题:

  • 功能完整性:原型有的区块、入口、表单,实现里都有吗?(差异区域如果有缺失,大概率是漏实现)
  • 样式正确性:间距、对齐、颜色、字号,实现和设计稿一致吗?(像素差异定位到具体位置)

像素 diff 的硬伤:内容演进

但这条有个已知局限:纯像素 diff 在内容演进的页面上会失真。我那 8 对原型 vs 实现差异率 59%-77% 全 FAIL_D——不是实现坏了,是原型是早期稿、实现已加了 24%-53% 的新内容。逐像素比对会把"多了一块功能区"判成"大面积不一致"。

所以像素 diff 只能当线索,不能当结论。它告诉你"这两处不一样",但不告诉你"该不该一样"。碰到内容大幅演进的页面,还得人工或 vision 介入判定。

这里有个更细的判断:差异的「形状」比「面积」更有信息量。同样是 30% 差异率,如果差异点集中成一块、轮廓清晰,大概率是「多了一个区块」(内容演进,通常无害);如果差异像撒胡椒面一样均匀散布在全页,那才是样式真的崩了(间距、字号系统性跑偏)。我把这个启发也写进了 diff 报告——按差异的连通区域大小分类,而不是只报一个总差异率。只看总差异率,内容演进和样式崩坏会被混为一谈,报告对开发毫无指导意义。

另一个常见的坑是抗锯齿和字体渲染差异。同一个页面在不同机器上截图,字体边缘的抗锯齿算法不同,逐像素 diff 会报出一层均匀的「噪点」差异,差异率能到 5%-8%,但人眼看完全一样。这种情况要么调高容差档(从 strict 退到 standard),要么在 diff 前做一次轻微的高斯模糊预处理。这些细节不处理,pixelmatch 跑出来的结果会被噪音淹没,真正该看的差异反而被淹掉。

场景二:老功能 → vision 巡查 + 代码审计

老功能的难点是没有原型可比。你不能拿一个不存在的"标准答案"去 diff。这条路退而求其次:用 vision 发现疑点,用代码审计判真伪。

vision 巡查:全量截图 → 启发式筛选 → 5% 深审

380 张截图不可能全喂 vision(慢、贵、注意力分散)。我先按文件大小分布做启发式筛——异常小的可能是白屏/报错,异常大的可能是样式崩坏堆叠。筛完挑 18 张代表页(覆盖 C 煽、商家、IM、营销、企业各模块),约 5% 抽样,丢给多模态模型深审。

vision 看布局、配色、遮挡、一致性,确实能发现人眼容易漏的问题。但它会幻觉,而且幻觉得很自信。我实际踩到三类:

幻觉一:把红色角标认成客服悬浮按钮。 vision 报"客服悬浮按钮遮挡商品价格",信誓旦旦。代码核查发现,客服是通过 openShopCustomerService() JS 函数触发的,根本没有任何全局 fixed 悬浮按钮。被误识别的,是红色的购物车角标。

幻觉二:浅渐变认成白底。 vision 批评"列表页普遍纯白背景、无玻璃拟态"。实际 KdPageShell 默认应用了 --kd-bg-gradient 绿色渐变,卡片是 rgba(255,255,255,0.75) 玻璃拟态。问题出在渐变太浅(#e6eeea→#fafdfb),vision 模型直接判成白色。

幻觉三:凭空报紫蓝主色。 vision 说某大厅页"用紫/蓝色主色"。代码一查,全量用的 var(--kd-color-primary) 和 var(--kd-gradient-primary),完全合规。纯幻觉。

这三条如果直接写进验收报告,开发会去查一个不存在的 bug,浪费一整天。

这三类幻觉有个共同的结构值得警惕:vision 把「它没理解的东西」当成了「有问题的东西」。它看不出浅渐变,就报「背景纯白」;它分不清红色角标和客服按钮的语义,就报「悬浮按钮遮挡」。换句话说,vision 不是在描述它看到的,而是在描述它「以为」自己看到的——而且描述得很有底气。这就是为什么 vision 的输出必须降级成「疑点」,它的每一句判断都带着「它可能看错了」的前提。

还有一个量化的教训。18 张深审图里,vision 一共报了 407 个疑点,经代码审计交叉验证后,只有 4 个是真问题(就是后面要讲的 4 个硬编码颜色)。真阳性率不到 1%。如果按 vision 的报告去查,97% 的时间在追幽灵。但反过来看,vision 也确实捞出了那 4 个真问题——这 4 个如果不扫,可能要很久之后才被发现。所以 vision 不是没用,是必须配一个确定性极高的过滤器,否则它的高虚警率会让人彻底放弃用它。

代码审计:判真伪的真相源

这是对抗 vision 幻觉的兜底,也是老功能巡查的真相源。它不靠看图,靠直接读确定的代码事实。

第一手是 design token 基线。所有颜色、圆角、渐变都定义在 variables.css 里,通过 KdPageShell 壳组件默认应用。这一层保证"正确答案"是机器可读的,不是设计师脑子里。

第二手是 DOM computed-style 扫描。用 Playwright 跑真实页面,遍历每个元素的 getComputedStyle,转成 hex 后扫描有没有写死的旧颜色泄漏(比如 #00b341 而不是 var(--kd-color-primary)):

// 遍历 DOM, 收集所有颜色, 扫写死值泄漏
const colors = await page.evaluate(() => {
  const out = [];
  for (const el of document.querySelectorAll("*")) {
    const s = getComputedStyle(el);
    for (const k of ["color","backgroundColor","borderTopColor","fill","stroke"])
      out.push(s[k]);
  }
  return out;
});
// 扫描非 token 的写死绿

这套扫描一跑,真相就清楚了:423/427 页走 token,只有 4 个文件硬编码了别的颜色。其中 3 个是招聘模块硬编码了一整套蓝色当"模块品牌色",1 个是某 FAB 按钮用了蓝绿渐变。这 4 个是真问题,有可复现的代码证据。

代码审计还能澄清语义色这种容易误判的情况。全站对"需求"类内容统一用 indigo(#6366F1)做分类标记,和"供给用绿色"成对。vision 看到紫色容易报"主色不对",但代码审计确认这是 12 个文件一致使用的同一 token,是有意的分类编码,不是违规。

这条路能回答什么

老功能巡查能回答两个问题:

  • 兼容性:新改动有没有破坏老页面?(代码审计扫 token 泄漏、扫硬编码,改动点都能定位)
  • 历史遗留:老代码里藏了哪些不规范?(4 个硬编码颜色就是历史债,扫出来就能立项修)

它回答不了"这个老页面对不对"——因为没有原型当标准。但回答"这个老页面有没有变得更糟"足够了。

两条路的对照

维度新功能:像素比对老功能:vision + 代码审计
锚点高保真原型(标准答案)没有,靠 design token 兜底
主工具pixelmatch 像素 diffvision 看图 + DOM 扫描
判断标准实现就该跟原型一样vision 出疑点,代码判真伪
能回答功能全吗?样式对吗?兼容吗?有遗留问题吗?
vision 角色不用(有原型了)只提疑问,不下结论
局限内容演进会失真没有"绝对正确"参照

这两条路是互补不是互斥。一个版本里,新做的页面走像素比对,改动的老页面走 vision 巡查,两边一起跑,380 页才覆盖得过来。

一个容易忽略的前提是:新功能要能走像素 diff,得在设计阶段就约定好交付「可独立渲染的原型 HTML」。如果设计师只交付一张 Figma 静态图,开发还得手动把图还原成 HTML 才能当锚点,这个还原本身就是人工活,等于把走查的成本前置了。我们在流程上要求设计交付必须包含一份浏览器能直接打开的 HTML 原型,这样像素 diff 的「标准答案」是现成的,不用额外造。这条约定不立起来,像素 diff 这条路再好也跑不起来。

整个 pipeline 并行跑下来,一个版本的视觉验收从原来「人走一周」缩到「机器跑两小时 + 人审 30 分钟疑点」:

视觉验收的 5 个反模式

讲完正确做法,有必要说说哪些做法不算视觉验收——我见过太多伪装成「我们做了 UI 自动化」的反模式,看起来跑了流程,实际质量没保障。

反模式一:全量截图丢给 vision,直接采信。 把 380 张图全塞给多模态模型,让它「看看有没有问题」,vision 的输出直接进验收报告。结果是一堆幻觉误报 + 漏掉真问题。正确做法:vision 只在 5% 抽样深审时用,且每个疑点必须代码审计交叉验证。

反模式二:像素 diff 当结论。 pixelmatch 跑出 60% 差异,直接报「样式不合格」。实际是内容演进(实现加了新功能区),不是样式坏了。正确做法:像素 diff 是线索不是结论,差异区域必须人工或 vision 二次判定。

反模式三:没有真相源,vision 说了算。 项目没沉淀 design token,颜色散落在各处,vision 说「主色不对」开发只能去查,查了一圈发现根本没标准。正确做法:先把 design token 沉淀成机器可读的 variables.css,这是所有判断的物理基础。

反模式四:商业 SaaS 的黑盒 AI 直接采信。 用 Applitools 这类「Visual AI」商业方案,它的「智能消除误报」是黑盒,你不知道它消的是真误报还是真 bug。正确做法:代码审计这层必须自建,不能用商业 AI 替代,因为只有你能把判断和自己的 token 基线做交叉验证。

反模式五:没有 token 泄漏基线扫描。 只看「现在对不对」,不扫「历史遗留问题」。结果老代码里的硬编码颜色一直藏着,某天被人改了一行就炸。正确做法:DOM computed-style 扫描作为常规项,每个版本跑一遍,扫出来的硬编码立项修。

这五个反模式的共同点:把「跑了流程」等同于「保障了质量」。视觉验收的价值不在跑了多少张图,在每一个判断都有可追溯的真相源。

反模式之所以顽固,是因为它们在「看起来很努力」这件事上特别有说服力。全量截图喂 vision,跑了几个小时,生成一份几十页的报告,汇报的时候很有场面——但场面不等于质量。真正起作用的视觉验收,产出物往往是朴素的:一份 token 泄漏清单、几个待确认的差异区域、一串有代码佐证的疑点。这份朴素的东西才挡得住 bug。评估一套视觉验收有没有用,别看它跑了多少图、出了多厚的报告,看它最终挡下了多少本该上线后才暴露的问题。

方案类型适合场景局限
Chromatic商业 SaaSStorybook 组件库视觉回归按截图量收费;只管组件,不管整页路由
Percy商业 SaaS(BrowserStack)多浏览器整页视觉回归同样按量收费;仍是像素 diff,内容演进会失真
Applitools商业(Visual AI)大规模视觉回归,自称 AI 消除误报闭源 AI 黑盒,难判它"消的误报"对不对
Storybook + Playwright自研(组件级)已有 Storybook 的项目只覆盖组件 story,不覆盖真实业务页面
Figma → 实现 diff设计稿驱动有 Figma 设计稿、想自动对齐依赖 Figma API 和 token 同步,接入成本高

我的项目选自研,原因有三个:380 个真实业务路由不是 Storybook story 能覆盖的;商业 SaaS 按截图量收费、380 页全量跑成本不低;最重要的是,只有自研代码审计层才能对抗 vision 幻觉——商业方案的 AI 是黑盒,你没法把它的"判断"和你的 design token 基线做交叉验证。

如果你的项目组件库成熟、页面少、设计稿在 Figma,Chromatic 或 Percy 接入更快。但只要你想让 vision 介入整页走查,代码审计这层都得自己建,这是绕不开的。

选型上还有个务实的考量:先看团队最缺哪一层的确定性。组件级回归缺确定性,Chromatic 这种方案能补;整页走查 + vision 缺确定性,只能靠自建 token 审计补。我们项目两条都缺,但整页走查缺得更狠(380 个真实业务路由),所以重投入在自建代码审计上。判断投入方向,不是看哪个方案更时髦,是看哪一层的「不确定」最容易演变成线上事故,把钱砸在那里。

还有个隐性成本容易被忽略:维护。视觉验收不是搭一次就完事,页面在长、token 在加、组件在改,验收基线得跟着同步更新。商业 SaaS 这层维护它帮你做了(但收订阅费),自建得自己维护。我们每周花半天同步基线——新加的 token 进白名单、新路由加进截图清单、已确认无害的差异标进忽略列表。这个维护成本必须算进决策,光看「自建一次能省多少钱」是低估了自建的真实代价。

一个判断

很多人觉得 AI 视觉验收的瓶颈是"模型够不够强"。换个更准的模型,幻觉会少一点,但不会消失。

优势不是更好的 vision,是一个可确定的真相源。新功能有原型做真相源,所以像素 diff 够用;老功能没有原型,就得靠 design token 和 DOM 扫描当真相源。vision 再聪明,也只是个"提问题的实习生",终审永远得靠确定性的代码事实。

这跟规范固化是同一个道理:把判断从"人/AI 临场决定"挪到"机器可执行的规则"。视觉验收这一关,新功能的规则是"实现 vs 原型",老功能的规则是"vision 出疑点 + 代码判真伪"——vision 的幻觉从风险变成了噪音,过滤掉就行。
