鲁班低代码平台刚立项时，后端、接入层、对外站点、组件库五个仓各写各的，谁也说不清边界在哪——改一个接口，三个仓要跟着动。这篇记的就是怎么把这摊代码理成一套边界固定、契约稳定的全栈架构。

下面以「鲁班低代码平台」为例，给出一套可落地的企业级低代码平台全栈架构与实现路线。

项目归属说明：架构经验源自大厂时期主导的内部系统"鲁班"，代码实现以个人开源项目 OpenLuban（架构同源、代码全新重写，GitHub）为准。原公司系统源码不公开。

覆盖范围从组件库、设计器与运行时渲染（Render），到接入层 BFF，再到主后端 luban-backend（Java：Spring Boot + MyBatis + Redis + MySQL），最后到管理后台与对外站点的交付形态。

文章目标不是“把所有代码一次写完”，而是用清晰的边界把系统搭起来，并能在后续迭代中持续演进。

1. 系统全貌：多仓分工，边界固定

鲁班平台由多个独立仓库组成,用 meta 仓 + git submodule 的方式统一治理(强烈建议保持拆分)。核心是五个仓,加上架构文档仓和 AI 助手仓,各自职责如下:

  • luban-ui：Nx monorepo,内部三个子包——luban-base(原子级 Vue3 组件,不感知低代码)、luban-low-code(可配置/可拖拽层,负责 schema 解析、组件注册、按 schema 渲染 base)、luban-utils(纯逻辑工具)。物料按业务场景分类(form/general/layout/website/navigation/marketing/lead/data-display 等)。
  • luban-backend（主后端）：Java 实现(Spring Boot 3.2.5 + MyBatis + Redis + MySQL)。提供站点、页面、用户、系统设置等原子化接口;平台 API 权威文档在本仓库维护（luban-backend/docs/API.md,自题"权威定义")。后端按 publicside(C 端只读)/operatorside(运营写)/shared(共享领域)三侧分包。
  • luban-bff：接入层,Next.js(App Router)实现,既是 API 网关也承担短链跳转、AI 代理。对前端提供统一接口,对内通过 BACKEND_BASE_URL(默认 http://127.0.0.1:8080/backend)调用 luban-backend;负责认证上下文注入(Header)与必要的协议/聚合适配。
  • luban：低代码引擎(设计器 + 运行时),Vue 3.5 + Vite。负责设计态的拖拽编辑和 schema 生成。
  • luban-website：对外站点(Nuxt 3 SSR)。根据 URL 从 BFF 拉取已发布页面 schema,交给 luban-ui 的 Render 运行时渲染。
  • luban-architecture-design：架构文档仓,作为整个平台的 single source of truth。
  • luban-ai-assistant：AI 助手,自然语言生成/编辑低代码页面(后面单讲)。

一点演进说明:这套架构早期有过一个 luban-backend-go(Go 双后端)作为同契约参考,后来 Go 双后端的路线放弃了,以 Java 版为主。保留"主后端单一权威"比维护双实现省心得多——双实现的契约对齐成本,长期看比它带来的容灾收益高。

这套拆分的核心收益是三点：后端契约稳定、接入层可控、前端（管理后台/对外站点）都只关心“拿到 schema 并渲染”。

BFF 层:Next.js 不只是网关

luban-bff 用的是 Next.js(App Router),不只是个转发网关。它干三件事:鉴权、聚合并发、协议适配。

鉴权上,BFF 是 JWT 的签发方——用 AUTH_JWT_SECRET 签 7 天有效的 JWT,生产环境强制拒绝默认 secret(缺失或等于 'dev-secret-change-me-in-prod' 直接抛错)。BFF 解完 JWT,把用户身份注入成 X-User-ID/X-User-Role 两个 Header 透传给后端。后端不签 JWT、不解析 JWT,只信任 BFF 注入的 Header——这就是"BFF 负责认证,后端负责授权"的分工落地。

调用后端时,BFF 统一走 callBackend,带超时控制(BACKEND_TIMEOUT_MS 默认 15 秒,用 AbortSignal.timeout),后端 hang 住不会锁死 BFF。非 ok 响应统一抛 BackendHttpError,把后端的 code/message/details 透传给前端——错误信息不丢。

还有一个容易忽略的细节:BFF 还认服务间 token(X-Internal-Token + X-User-Id/X-User-Role),这是给 AI 服务回环用的——AI 助手调 BFF 不走用户 JWT,走服务间 token。双通道鉴权让"用户请求"和"服务间请求"各走各的路,互不干扰。

AI 助手:自然语言生成低代码页面

luban-ai-assistant 是这套架构里最后补上的一环,但很关键——它让低代码平台从"拖拽搭建"进化到"说话搭建"。技术栈是 Python(FastAPI) + LangGraph + LiteLLM,用 RAG 检索物料,自然语言生成符合 schema 的页面。

这里有个设计细节值得讲:AI 读的不是源码,是物料导出的 propsSchema。luban-ui 通过 scripts/export-materials.mjs 把每个物料的可配置属性导成机器可读的 materials.json,AI 仓持有副本,有校验命令 make check-materials-sync 保证两边不漂移。组件库不只给人用,也给 AI 用——AI 要知道"这个组件有哪些可配属性、每个属性什么类型",靠的就是这份机器可读的 schema。这把组件化复用和 AI 结合打通了:改了组件 props,导出新的 schema,AI 立刻知道新属性能配什么。

AI 助手还带了评测体系(LLM-as-Judge),从 schema 有效性、物料召回率、意图准确性几个维度给生成结果打分。生成低代码页面不是一次性的 demo,是能量化评估、能持续优化的工程。

2. 数据模型：站点 / 页面 / 用户 / 系统设置

后端以四类领域模型为主（详见 luban-backend/docs/API.md）：

  • Site（站点）：id / name / slug / baseUrl / status / seoJson / analyticsJson / createdAt / updatedAt。seoJson 和 analyticsJson 是后来加的——站点级的 SEO 配置和统计平台配置(GA4、百度统计、Facebook Pixel 的 ID 都存这儿)。
  • Page（页面）：id / siteId / name / path / status(draft|published) / schema(JSON) / seoJson / publishedAt / publishedBy / createdAt / updatedAt。publishedAt/publishedBy 是发布闭环加的,记录谁在什么时候发的,审计要用。
  • User（用户）：id / username / name / role(admin|user) / status(active|disabled) / password(bcrypt) / createdAt / updatedAt
  • SystemSettings（系统设置）：单行 JSON，承载系统级配置（站点名称、Logo、安全策略等）

数据库层面,这些表都用 InnoDB + utf8mb4,id 用 UUID(VARCHAR(36),便于分布式与跨系统整合),schema 用 MySQL 原生 JSON 类型。关键约束：

  • site.slug 全局唯一（用于对外站点寻址与运营管理）——UNIQUE KEY uk_sites_slug
  • 同一站点下 page.path 唯一（用于路由映射，如 /home）——UNIQUE KEY uk_site_path
  • page.status 决定能否被对外公开接口返回（只允许 published）

这些唯一约束不是只靠应用层校验,数据库层也建了唯一索引兜底——应用层校验可能漏(并发请求),数据库唯一约束绝不会漏。这也是前面 DDD 那篇讲的"领域不变性靠 DB 约束 + 应用校验双保险"。

3. 主后端：luban-backend（Spring Boot + MyBatis + Redis + MySQL）

3.1 为什么主后端选择 Java 版？

在企业级场景中，后端往往需要更成熟的三方面能力：

  • 规范化分层与扩展（Controller/Service/Mapper）
  • 统一错误模型与可观测性（日志、链路、异常处理）
  • 权限与审计能力的演进空间（RBAC、资源级权限）

因此将 luban-backend 定位为主后端，并把 API 文档固化为权威契约（luban-backend/docs/API.md），可以显著降低多端联调与跨团队协作成本。

3.2 MySQL：业务真相源（Source of Truth）

MySQL 存储核心业务数据（sites/pages/users/system_settings）。典型落地建议：

  • id：UUID（便于分布式与跨系统整合）
  • 时间：统一 ISO 8601 输出；数据库使用 DATETIME/TIMESTAMP 按团队习惯定
  • schema：JSON（MySQL JSON 类型或 TEXT 存储均可，视查询需求）

3.3 Redis：配置缓存与热点加速

Redis 在鲁班平台中非常适合做两件事:

  • 系统设置缓存（SystemSettings）：读多写少、强时效要求不高
  • 页面/站点热点缓存：对外站点高 QPS 时，按 siteSlug + path 做短 TTL 缓存

注意：缓存只是加速层，最终一致性以 MySQL 为准。

SystemSettings 的缓存 key 就是 settings:global,用 StringRedisTemplate 读写:命中直接返回,未命中查 DB 后写回缓存;更新时先写 DB 再同步更新缓存。这个"写后立即更新缓存"的策略,比"靠 TTL 过期"更新更及时——运营改了站点名,缓存立刻刷新,不用等 TTL。代价是写路径多一步缓存操作,但系统设置写频率极低,这点代价完全值得。

3.4 认证与授权：Header 上下文（由 BFF 注入）

平台采用“BFF 负责认证，后端负责授权”的分工:

  • BFF 完成登录态校验、JWT 验证、会话管理等
  • BFF 把用户上下文注入到后端请求 Header：
    • X-User-ID
    • X-User-Role（admin/user）
  • 后端只做明确角色字段的权限判断（例如 role == 'admin'），禁止用 username == 'admin' 之类的逻辑“猜权限”

后端的 AuthFilter 里有一条死规矩:判断 admin 只认 role == 'admin',绝不靠 username 猜。注释里写得明明白白,因为靠用户名猜权限是安全事故的温床——哪天有人注册了叫 admin 的账号,或者用户名规则变了,靠用户名的权限判断就崩了。认字段不认名字,这条原则看着废话,但它是踩过权限漏洞之后才立下的。

这种模式的优势是：后端保持原子化与领域内聚，接入策略（Web/APP/小程序、不同会话方案）都能在 BFF 层演进。后端不用管"这个请求是从 H5 来的还是从小程序来的",它只认 BFF 注入的两个 Header。接入渠道再怎么变,后端的鉴权逻辑一个字都不用改。

3.5 API 契约：以 luban-backend/docs/API.md 为准

平台后端 API 的权威文档在:

  • luban-backend/docs/API.md

这份文档开头就声明自己是"权威定义",由主后端仓库维护,BFF 和前端以它为对接规范。建议团队立下硬规则:

  • 先改文档，再改实现（避免“实现已变但文档没更新”的长期技术债）
  • BFF 与前端以文档为对接规范

“先改文档再改实现"这条看着像废话,但它是契约稳定的关键。反过来"先改实现、文档后补"的结局几乎都是文档永远不补——开发改完代码就提 MR 了,谁还记得回头补文档?三个月后文档和实现已经对不上,BFF 按文档对接,一调发现接口早变了。把"改文档"卡在"改实现"之前,等于把文档变成了开发流程的必经环节,而不是可做可不做的善后事。

这份 API 文档覆盖的接口远不止站点/页面/用户四类,还包括表单(Form)、留资(Lead)、营销活动(Campaign)、渠道(Channel)、AB 实验(Ab)、数据分析(Analytics)、模板(Template)、计费(Billing)等。一个低代码平台真正要做大,后端领域远比"搭页面"复杂——页面只是皮,留资归因、AB 实验、计费这些才是撑起 SaaS 商业模式的核心。这也是为什么前面强调"先跑通 MVP 闭环再扩”,而不是一上来就铺这么多模块。

公开接口那条值得单独说:GET /backend/public/sites/:slug/pages?path=:path,无鉴权,只返回 status=published 的页面。这条接口是整个对外站点的命脉——website 全靠它拿 schema 来渲染。它读的是 published_pages 快照表(不是 pages 草稿表),保证对外永远展示稳定的发布版本。

4. 接入层：luban-bff（对外统一接口、对内调用后端）

BFF 的职责通常包括:

  • 对管理后台提供 /api/*（需要登录与权限）
  • 对对外站点提供 /api/public/*（无需登录）
  • 统一错误格式、跨域策略、限流、熔断、重试（按需）
  • 把后端多接口聚合为前端易用的接口（但避免把领域逻辑搬到 BFF）

最后一条是个常见的坑——BFF 很容易越界,把本该在后端的领域逻辑(比如订单状态流转、价格计算)搬过来"图方便"。一旦搬了,BFF 就从接入层退化成第二个后端,领域逻辑散在两处,改一处忘另一处。BFF 该做的只是协议聚合和透传,不该承载业务规则。这条边界守不住,多端架构很快就乱。

4.1 对外站点的“公开页面接口”

对外站点必须能按 URL 获取已发布页面 schema。鲁班采用“后端公开接口 + BFF 转发”的方式：

  • 后端（无需鉴权）：GET /backend/public/sites/:slug/pages?path=:path
  • 行为：只返回 status=published 的页面（含 schema）；找不到返回 404

这样 luban-website 永远不用直连后端，也不需要 JWT。

5. 组件库与运行时：luban-ui（Designer + Render）

低代码平台的前端能力通常分三层:

  • 基础组件(luban-base:按钮、输入框、布局等,纯 props/events/slots,不感知低代码)
  • 低代码组件(luban-low-code:可被 schema 描述、可在运行时渲染,负责 schema 解析、组件注册、属性面板、按 schema 渲染 base)
  • 运行时渲染（Render）：把 PageSchema 转成真实组件树渲染

核心的运行时渲染组件叫 RuntimeRenderer,它递归地把 NodeSchema 渲染成组件树:<component :is="getComponent(root.type)"> 加 v-for child in root.children 递归。它要处理的事比"渲染"两个字复杂得多——条件渲染(isNodeVisible 求值)、循环渲染(loop.data 按元素多次渲染)、表单双向绑定(v-model 绑 formState[name])、事件绑定(resolveEvents 把事件名映射成动作表达式)、表达式插值({{}} 插值)、CMS 绑定。单是"按 schema 渲染"这五个字,背后是一套完整的迷你运行时。

页面级容器是 LubanPage,职责更高一层:管理 formState/formErrors、并发拉取数据源(loadDatasources)、收集 CMS 绑定(loadCmsBindings)再 provide 给 RuntimeRenderer。还有一个容易忽略的工程细节:RuntimeRenderer 带 ErrorBoundary,单个物料崩溃不会让整页空白,只会那个组件位置显示"渲染失败"。低代码平台物料多、第三方可扩展,一个物料挂了拖垮整页是不可接受的。

鲁班里,luban-website 与 luban（设计器）都通过 luban-ui 的 Render 能力渲染页面。对外站点只用运行时,不引入设计器。

在工程规范上,建议保持:

  • 统一设计语言（Material Design 风格、响应式）
  • 组件库产物必须生成 TS 类型声明
  • 新增/修改组件必须补齐单元测试与 e2e 测试（按仓库约束执行）

luban-ui 这块配套了 Storybook 文档和单测,每个基础组件都有 .spec.ts 和 .stories.ts。企业级复用的工程标配不是写个组件就完了,是组件 + 单测 + 文档三位一体。 这也是为什么物料能被放心地跨项目复用——有测试兜底,改一个组件不会悄悄搞挂依赖它的页面。

6. 对外站点：luban-website（Nuxt 3 SSR + Render）

对外站点是"把 schema 变成可访问网页"的交付面:

  • 输入：siteSlug + path
  • 输出：SSR/CSR 渲染后的 HTML 页面

基本流程:

  1. 浏览器访问某 URL
  2. Nuxt 调用 BFF public 接口获取页面数据（含 schema）
  3. 把 schema 传给 LubanPage 组件渲染

这里有个 SSR 的关键细节:website 用 useAsyncData + 顶层 await 拉取页面数据,刻意让 fetch 在 SSR 响应发送前完成。为什么?为了做"硬 404"——如果某个 path 没有已发布页面,要在 SSR 阶段就 throw createError({ statusCode: 404, fatal: true }),返回真正的 404 给爬虫。如果改成客户端再判断,爬虫拿到的会是一个 200 的空壳页再跳 404,SEO 上等于没有这个 404。SSR 不只是为快,是为 SEO 的语义正确。

LubanPage 在 SSR 阶段直接渲染(没有套 ClientOnly),保证爬虫能看到完整内容。website 还通过 provide('lubanFormSubmit') 把表单提交回调注入运行时,通过 collectionFetcher/datasourceFetcher 注入拉取器——运行时不直连后端,所有数据获取都由 host(website)注入。运行时保持纯净,不耦合任何后端细节,这是它能同时服务管理后台和对外站点的前提。

在企业实践中，SSR 的价值在于:

  • 更好的首屏与 SEO
  • 更稳定的多端渲染体验

7. 草稿与发布分离:published_pages 快照表

这套架构里有个容易被忽略但很关键的设计:对外公开接口读的不是 pages 表,是 published_pages 发布快照表。

pages 表存的是草稿(运营正在编辑的、随时会改的),published_pages 存的是发布快照(某次发布那一刻的固化版本)。运营点"发布",系统把 pages 的当前内容拷一份到 published_pages;对外站点永远只读 published_pages。这样设计的好处:运营编辑草稿不会影响线上页面。运营改到一半保存了草稿,线上展示的还是上次发布的稳定版本,直到运营点"发布"才更新。

这个分离还顺带解决了并发问题——多个运营同时编辑不同页面,各自改的是 pages 里的草稿,互不干扰;线上读的是 published_pages,也不受草稿改动影响。草稿和发布用两张表物理隔离,比在同一张表上用 status 字段区分更稳——后者一旦查询条件漏了 WHERE status='published',线上就可能读到半成品。

8. 工程化建议：测试、发布、演进

8.1 测试策略（建议最小闭环）

  • 后端：接口级测试（鉴权/权限/404/冲突码 409 等关键分支）+ ArchUnit 架构守护测试(分层依赖、聚合根隔离、命名规范都用测试锁死)
  • BFF：契约测试（确保转发/聚合不破坏后端语义）
  • UI：组件单测 + 关键交互 e2e（设计器拖拽、运行时渲染）
  • Website：最少做一条端到端链路（访问 URL -> 拉取 schema -> Render 出页面）

测试这块鲁班是有硬门禁的,不是"建议测一测"那么随意。后端用 JaCoCo 做覆盖率门禁(haltOnFailure=true,BUNDLE LINE ≥0.80),TS 引擎/BFF/website 覆盖率目标 85%、UI 组件库 90%。覆盖率不是 KPI,是入场券——达不到不让合。ArchUnit 那 8 个架构测试类尤其值得说:分层依赖、聚合根不能带 Spring 注解、publicside 不能依赖 operatorside(安全边界)、自定义异常必须继承 BusinessException……这些架构判断全用测试锁死,AI 改代码想越界,测试直接红。

E2E 还有一条硬纪律:禁止假绿。所有 E2E 真实执行,禁止 *.skip/条件跳过;需跳过须用户明确同意;禁止"没起依赖→全 skip→退出码 0"冒充通过。这条是踩过坑才立的——AI 最爱的作弊就是用环境变量让全红变全绿,得用规则堵死。

8.2 发布策略（推荐分层）

  • luban-backend：独立部署（容器化），MySQL/Redis 作为依赖
  • luban-bff：独立部署（容器化），通过环境变量指向后端
  • luban：静态资源部署（或容器化 Nginx）
  • luban-website：SSR 服务部署（Node 进程/容器），通过环境变量指向 BFF

8.3 演进路线（从 MVP 到企业级）

  • MVP：站点 + 页面 + 发布 + 对外访问闭环
  • 第 2 阶段：组件市场、权限细化（资源级）、审计日志
  • 第 3 阶段：多租户、灰度发布、CDN/缓存、页面渲染性能优化

9. 小结

一套可持续演进的低代码平台，关键不在“把所有功能都塞进一个仓库”，而在于三点：

  • 边界清晰：luban-backend 做领域与授权，BFF 做接入与认证，前端只管渲染与交互
  • 契约稳定：以 luban-backend/docs/API.md 作为权威 API 文档
  • 渲染复用：luban-ui 的 Render 运行时同时服务管理后台与对外站点

边界、契约、复用这三点,是低代码平台能长期演进的底座。边界清晰才能各自迭代不互相拖累——后端改领域模型不影响 BFF 转发,BFF 加聚合接口不动后端。契约稳定才能多端对接不乱——API 文档是 single source of truth,谁改了接口谁先改文档。渲染复用才能不重复造轮子——一套 Render 运行时服务设计器、管理后台、对外站点,改一处全受益。

如果准备从零落地企业级低代码平台，建议先把“站点-页面-发布-对外访问”闭环跑通，再逐步增强设计器能力与企业级治理能力。先跑通闭环的意义在于:它能立刻产生业务价值(运营能自己搭页面上线),而不是憋半年交付一个大而全但没人用的系统。鲁班也是这么长起来的——最早只有站点+页面+发布+对外访问,MVP 跑通后再逐步加组件市场、AB 实验、留资归因、AI 生成这些更重的模块。每一步都是"闭环 → 用起来 → 发现新需求 → 再扩",不是一次性画个大饼,平台才没有在中途失控。

深度展开:低代码平台的核心矛盾

低代码平台最核心的矛盾是灵活性和易用性的平衡。给用户越多配置项,平台越灵活,但越难用;给用户越少配置项,平台越好用,但越不灵活。

我们的解法是分级:

  • L1 配置型(业务运营用):只改文案、图片、颜色。零代码,所见即所得。
  • L2 编排型(实施顾问用):拖拽组件、配置流程、定义规则。低代码。
  • L3 扩展型(开发者用):写自定义组件、接外部 API、写复杂逻辑。Pro 代码。

三层共享一份 schema,但暴露的编辑能力不同。L1 用户看不到 L2 的流程编排,L2 用户看不到 L3 的代码扩展——每层只看到自己那层的复杂度。

这个分级的本质判断:不是所有用户都需要全部能力,给合适的人合适的能力,比给所有人全部能力更重要。给运营一套拖拽就够,他不需要看代码;给开发者代码扩展入口,他不需要被运营那套表单配置限制住。强行让所有人用同一套界面,结果就是运营嫌复杂、开发者嫌受限,两边都不满意。分级本质上是"复杂度分层暴露",每层只承担自己用户的认知负担。

落到代码里,这套分级不是靠权限开关硬切的,是靠 schema 本身的结构。L1 编辑的 schema 子集,L2 能看全但不能改 L3 的扩展点,L3 能改一切。同一份 schema,不同视角看到不同的可编辑范围——这是低代码平台"灵活性 vs 易用性"矛盾的解法:不是二选一,是分层各取所需。这也是为什么前面对外站点只读 published_pages——不同角色对同一份数据的读/写权限,在数据层就隔离开了。
