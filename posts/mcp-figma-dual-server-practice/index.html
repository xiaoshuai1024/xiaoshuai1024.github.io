MCP 接 Figma 实战:把工具接入从 M×N 标准化成 M+N

接个订单查询,写个 OrderQueryTool;接个设计平台的 API,又得写一套适配。工具越多,适配代码越堆。直到我接 Figma,踩了一通坑,才搞懂 MCP(Model Context Protocol)到底在解决什么——它不是又一个新协议让你学,是把"每个工具都重写一遍适配"这层标准化掉。

(上一篇我讲的是自研 Tool Calling 桥接,把工具定义和框架解耦;这篇往上走一层。)

这篇就用我把 Figma 接进 AI 工作流的真实过程,讲 MCP 怎么落地。不讲虚的,讲配额怎么省、双 server 怎么配合、怎么装。

一、MCP 到底解决什么:M×N 变 M+N

先说它解决的那个根本问题。

假设你有 3 个 AI 客户端(Claude Code、Cursor、Cline),要接 3 个外部工具(Figma、GitHub、数据库)。传统做法是每个客户端给每个工具写一套适配——3×3=9 套。客户端加一个、工具加一个,适配数就是乘法涨。这就是 M×N 问题。

MCP 做的事是中间插一个标准协议:客户端只跟协议说话,工具也只跟协议说话。于是 3 个客户端 + 3 个工具,只需要 3+3=6 个适配,每边各管各的。涨法从乘法变加法。

落到 Figma 这个例子上:我不需要为「Claude Code 怎么调 Figma」写适配,也不需要为「Cursor 怎么调 Figma」再写一遍。Figma 那边出一个 MCP server,任何支持 MCP 的客户端都能直接用。这就是它最实在的价值——工具方写一次,所有客户端通吃。

M×N 变 M+N 还有个容易被忽略的收益:工具方和客户端可以独立演进。M×N 模式下,Figma API 升级了,所有客户端的 Figma 适配都要跟着改——3 个客户端改 3 遍。M+N 模式下,Figma 升级了,只改 Figma MCP server 这一处,所有客户端自动受益。这个「解耦后各自演进」的红利,在工具多、客户端多的场景下特别值钱——每次 API 升级省下来的适配工作,是持续的复利。

反过来,M+N 也有代价:MCP server 成了单点。server 挂了,所有客户端都用不了这个工具。M×N 模式下,一个客户端的适配挂了,只影响那一个客户端。所以 MCP server 的稳定性要求比「客户端内部的适配」更高——它服务多个客户端,出问题影响面更大。这也是为什么选 MCP server 时要看维护活跃度:一个没人维护的 server,随时可能挂掉且没人修,所有依赖它的客户端一起遭殃。

二、协议三件套,知道这些就够用

MCP 的协议本身不复杂,记住三样:

消息格式:JSON-RPC 2.0。 客户端和 server 之间就是发 JSON 消息,请求带 method 和参数,响应带结果。没什么花活。

三类能力(原语):

  • Tools——模型可以调用的功能(比如 Figma 的「读取设计稿」「导出截图」)。这是用得最多的。
  • Resources——模型可以读取的数据(比如一份文档、一个配置)。
  • Prompts——预定义的提示模板(用得少,知道就行)。

外加几个反向能力(Sampling、Roots、Elicitation),新手不用管,用到再查。

传输方式(transport)两种:

  • stdio——本地进程,通过标准输入输出通信。适合跑在你机器上的工具。
  • Streamable HTTP——远程,一个 HTTP 端点搞定,支持流式。适合云端服务。

就这些。很多人被 MCP 吓住,是因为术语多,但骨架就这么点。后面全是往这个骨架上挂具体工具。

三、接 Figma:一个工具,为什么要两个 MCP server

这是整件事里我最想讲的一段,因为它最能说明 MCP 在真实项目里长什么样。

接 Figma 的时候,我没用「一个 server 搞定」,而是同时挂了两个:

  • figma-mcp-go:本地 server(stdio 传输),73 个读写工具,能绑定设计变量、管 Token、导出截图。
  • 官方 Figma MCP:云端 server(Streamable HTTP),核心是 Code to Canvas——能把一段 HTML 变成 Figma 里可编辑的图层。

为什么非要两个?因为各自都有硬伤,合起来才完整:

figma-mcp-go 不能导入 SVG 图标,创建渐变这种复杂样式也很费劲;而官方 MCP 的 Code to Canvas 恰好能把 HTML(里面塞 SVG、CSS 渐变、阴影)原样转成 Figma 图层。反过来,官方 MCP 没法做精细的变量绑定,figma-mcp-go 这块强。

所以分工是:官方 MCP 负责「生成高品质图层」,figma-mcp-go 负责「绑定 Token + 精细操控」。一个生成,一个打磨。

这给了我一个挺重要的认知:MCP 不是「一个工具一个 server」的对应关系。一个外部能力,可能要拆成几个 server 组合着用,各自补对方的短板。理解工具的能力边界,比理解协议本身更关键。

双 server 配合还有一个实操上的麻烦:两个 server 的状态不同步。figma-mcp-go(本地)连的是 Figma 桌面端的某个文件,官方 MCP(云端)通过 API 访问的是同一个文件,但两者的「当前文件」状态是各自维护的。在 figma-mcp-go 里切了文件,官方 MCP 不知道你切了,它还在访问旧文件的数据。所以每次切文件,两个 server 都要同步更新——这步忘了,就会出现「本地 server 绑定的 Token 是 A 文件的,官方 server 生成的图层跑到了 B 文件」的错乱。我们的习惯是切文件时做一个「双 server 重置」:本地 Run 一次插件桥,官方 server 重新确认 file key,两边对齐了再干活。

四、配额这个坑,差点让我白干一天

接完能用之后,第一个让我吃瘪的是配额。

官方 Figma MCP 免费版有个限制:读工具每月只能调 6 次。get_design_context、get_screenshot 这些都算。而写工具(generate_figma_design,也就是 Code to Canvas)不限次数。

我一开始没注意,边调边试,读一次看一眼结构、再读一次截图对照,几个组件没弄完,6 次就烧光了,后面大半天读工具全报额度用完。

教训很直接,后来我立了个规矩:

把读取集中批处理。 一次性把要看的组件、截图、Token 定义全捞回来,别「读一点改一点再读」。读操作是稀缺资源,得当成预算花。

能写就别读。 既然写工具不限次,设计生成就大胆用 generate_figma_design;需要确认效果时,合并成一次 get_screenshot,而不是每个小改动都截一次。

Token 两端对齐靠脚本,不靠肉眼。 用一个同步脚本把 Figma 里的变量导出成 CSS,跟项目里的 variables.css 比对,而不是一次次读 Figma 确认颜色对不对。

这套打法跑通后,6 次配额能撑住一整轮设计还原。MCP 的工具是好工具,但配额模型你得先摸清,不然会卡在半路干瞪眼。

配额这个坑还揭示了一个更通用的原则:用免费工具前,先摸清它的限额模型。不只是 MCP,Figma API、OpenAI API、地图 API,免费的都有限额,而且限额的维度各不相同——有的按次数、有的按流量、有的按并发。不提前摸清,用着用着就卡在半路。摸清之后,策略也类似:把稀缺的(读操作)集中批处理,把不限的(写操作)大胆用,用脚本替代肉眼确认。这个「摸限额 → 调策略」的套路,适用于所有有配额限制的外部服务。

还有一个关于配额的实战技巧:缓存读取结果。get_design_context 返回的设计稿结构,在本次工作期间不会变(你没改设计稿),所以第一次读回来后缓存到本地文件,后续需要再查时先读缓存,不调 API。这个简单的缓存,把读取次数从「每次需要都调」降到「整个工作流程调一两次」,配额压力直接消失。缓存对「数据不常变」的读取操作特别有效,是省配额的第一手段。

五、Code to Canvas:设计到代码的桥

官方 MCP 那个最香的能力叫 Code to Canvas,值得单独说,因为它是把「设计」和「代码」两头打通的关键。

它的玩法是反过来——不是从 Figma 读设计生成代码,而是把代码(HTML)变成 Figma 图层:

  1. 我(或 AI)按设计规范写一段 HTML,里面直接内联 SVG 图标、写 CSS 渐变和阴影。
  2. HTML 里注入一段官方的 capture.js 脚本。
  3. 起个本地 HTTP 服务托管这个页面(python3 -m http.server),注意必须用 HTTP,不能用 file://,否则捕获脚本跑不起来。
  4. 调 generate_figma_design,拿到一个 captureId,浏览器打开带这个 id 的链接。
  5. capture.js 捕获页面结构,几秒后 Figma 里就出现了可编辑的图层——矢量、文字、渐变都保留。

这一步的价值在于:AI 写 HTML 比让它直接操作 Figma 节点靠谱得多(HTML 是 AI 的母语),而 Code to Canvas 把这份 HTML 干净地翻译成设计稿。等于让 AI 走它最擅长的路去产出设计,再交给 figma-mcp-go 去绑 Token、精修。

为什么说「HTML 是 AI 的母语」?因为模型的训练语料里,HTML/CSS 的量远大于 Figma 的 API 文档。AI 写 HTML 时,它调用的是海量训练数据里的模式——布局怎么搭、样式怎么写、响应式怎么处理,这些它见过几百万遍,写出来的质量很稳。而让它直接调 Figma API 操作节点(创建一个 Rectangle、设置 fill、调 position),它要精确记住 API 的参数名和调用顺序,这些它在训练数据里见得少,出错率高。所以 Code to Canvas 的巧妙之处,是让 AI 用它最熟的语言(HTML)产出,再用工具翻译成设计稿——绕过了 AI 不擅长的部分,发挥它擅长的部分。

Code to Canvas 还有个限制要提:它不能 100% 还原 HTML 的效果。CSS 的某些高级特性(比如 backdrop-filter、复杂的 grid 布局),Figma 的图层模型表达不了,翻译时会丢失或简化。所以 Code to Canvas 产出的 Figma 图层,和 HTML 在浏览器里渲染的效果,偶尔会有差异。这要求使用时有个心理预期:它是「快速生成设计草稿」的工具,不是「像素级还原」的工具。草稿生成后,还是要在 Figma 里手动微调那些没还原好的细节。

两个 server 的分工,到这一步就闭环了:HTML(代码)→ 官方 MCP → Figma 图层(设计)→ figma-mcp-go → 绑定 Token + 组件化 → get_design_context → 还原成 Vue 代码。设计稿和代码两头都能走通,而且都走 MCP,不用人肉在两个世界之间搬。

六、怎么装:.claude/mcp.json + 一条命令

讲完道理,给能直接照做的安装步骤。

MCP server 的配置写在项目的 .claude/mcp.json 里。加一个 server 最方便的方式是用 claude mcp add:

# 本地 server(stdio):figma-mcp-go,73 个读写工具
claude mcp add -s project figma-mcp-go -- npx -y @vkhanhqui/figma-mcp-go@latest

# 云端 server(Streamable HTTP):官方 Figma MCP
claude mcp add --transport http figma-official https://mcp.figma.com/mcp

跑完之后,.claude/mcp.json 长这样:

{
  "mcpServers": {
    "figma-mcp-go": {
      "command": "npx",
      "args": ["-y", "@vkhanhqui/figma-mcp-go@latest"]
    },
    "figma-official": {
      "type": "http",
      "url": "https://mcp.figma.com/mcp"
    }
  }
}

本地那个(figma-mcp-go)还多一步:它需要 Figma 桌面端跑一个插件桥。流程是下插件包、在 Figma 桌面版里 Plugins → Development → Import plugin from manifest... 导入。每次操作不同文件,都要在新文件里 Run 一次这个插件,否则它还连着上一个文件。

云端那个(官方 MCP)首次用要走一遍 OAuth:Claude 会弹个浏览器链接,你在 Figma 授权页点允许,浏览器跳到一个 localhost 回调地址。如果报连接错误,把地址栏的完整 URL 复制出来贴给 Claude 就行——这是它拿 token 的方式。

改完 mcp.json 记得重启 Claude Code 会话才生效。装完用 get_metadata()(本地)或 whoami()(官方)各验证一下连通。

安装这步有个常见的坑:MCP server 的配置是项目级的,不是全局的。.claude/mcp.json 在项目根目录,意味着每个项目要单独配。如果你在多个项目里都要用 Figma MCP,每个项目都得加一遍。Claude Code 也支持全局配置(-s user),但全局配置的 server 在所有项目都加载,不用的项目也背着这些 server 的初始化开销。我们的做法是:通用的工具(Figma、GitHub)配全局,项目专用的工具配项目级。这个区分能避免「项目 A 加载了项目 B 才用的 server」的浪费。

七、什么时候用 MCP,什么时候还是自研桥

写到这里,自然会问:那我上一篇那个自研 Tool Calling 桥接,是不是多余了?MCP 能不能全包?

我的答案是分场景:

MCP 适合接「外部、标准化、复用度高」的工具。 像 Figma、GitHub、数据库、Slack 这些,工具方或社区已经出了成熟的 MCP server,你挂上就能用,别自己造。这类工具的特点是:协议相对固定、跨项目复用、你不掌握它的内部实现。

自研桥接适合「内部、业务强相关、需要深度约束」的工具。 像订单查询、分润这种业务方法,要穿透多租户上下文、要走确认门禁、要落审计日志——这些约束是你的业务独有的,MCP 不会替你管。这种就该自己写一层薄桥接,把业务约束焊死。

两者不冲突,是不同层:MCP 解决「外部工具怎么标准接入」,自研桥接解决「内部工具怎么安全调用」。一个管广度(接得多),一个管深度(管得严)。成熟的 AI 工作流,往往是 MCP + 自研桥并用。

八、收个尾

接 Figma 这件事,本来以为是个「装个插件就完」的小活,结果从双 server 分工、配额限制、Code to Canvas 工作流到安装配置,折腾出一套完整经验,值得好好记录下来,也给后来人省点踩坑的时间,少走些冤枉路。

回头看,MCP 真正的价值不在协议本身有多精巧——JSON-RPC 加几个原语,没什么神秘的。它的价值在于把工具接入这件事,从一个「每次重写」的体力活,变成一个「挂载即用」的标准动作。工具方写一次,所有客户端受益;客户端接一次,所有工具可用。

但 MCP 也不是银弹。它管的是「接得上」,管不了「接得对」——配额怎么分配、能力边界在哪、业务约束怎么补,这些还得你自己摸。Figma 这个例子里,真正费心力的不是协议,是搞清两个 server 各自能干什么、不能干什么,然后把它们的短板互补起来。这层「工具的组合与收口」的判断,是 MCP 标准化不了的,也是接工具这件事里真正需要人动脑的部分。

工具会越来越多,MCP 让接它们的成本趋近于零,这是它最大的贡献。但「知道该用哪个工具、怎么组合、在哪收口」——这层判断,是工具替代不了的,也是接完一堆工具之后最该沉淀下来的经验,值得认真对待。这大概也是我接完一堆工具之后,最确定的一件事。

深度展开:MCP 工具的安全边界

MCP 给 AI 工具能力,但工具=权限,必须配套安全边界。工具越多,AI 能干的事越多,能闯的祸也越大。我们踩过三个坑,每个都对应一条安全规矩。

第一个坑:工具返回了过多信息。一个「查数据库」的 MCP 工具,本来只是想查用户列表,结果返回了整张表的所有字段,包括密码哈希、token、身份证号。AI 把这些写进了回答——用户问「系统里有哪些用户」,AI 把密码哈希也列出来了。这还不是最糟的,最糟的是 AI 把这些敏感信息塞进了 prompt 的上下文,后续的对话里它可能随时引用。修复方式是工具返回前脱敏:只返回业务字段(用户名、注册时间),敏感字段(密码、token、身份证)一律不返回,在工具层面就拦掉,不依赖 AI 自己判断「这个该不该说」。

第二个坑:工具的副作用没限制。一个「发邮件」的 MCP 工具,AI 自己决定给谁发、发什么,结果给全公司群发了一封测试邮件。AI 没有恶意,它只是在执行「通知所有人」这个指令,但它不理解「全公司群发」的后果。修复方式是带副作用的工具(发邮件、发消息、写数据库、调支付)必须有确认机制:AI 调用前必须先问人「确认要执行吗」,把执行的参数(收件人、内容)展示出来,人点了确认才真正执行。这和自研桥接里写操作的确认门禁是同一个思路——副作用操作,人必须过一道手。

第三个坑:工具的输入没校验。AI 传了一个恶意 SQL 给「查数据库」工具,差点删表。AI 不是故意传恶意 SQL,是它把用户的自然语言理解错了——用户说「把测试数据清一下」,AI 生成了 DELETE FROM users,而不是删测试数据。修复方式是工具的输入必须过校验:查询工具只允许 SELECT,禁止写操作;如果确实需要写工具,写操作的参数要白名单校验(比如只允许删 test- 前缀的数据)。输入校验在工具层做,不依赖 AI 自觉——AI 传什么,工具按规则拦,传错了直接拒绝返回错误。

这三条安全规矩有个共同原则:安全约束在工具层做,不在 AI 层做。AI 的行为不可预测(它会幻觉、会误解指令),把安全寄托在「AI 自己会判断」上是危险的。工具层是确定性的代码,校验规则写死了就不会变,该拦的一定拦住。这和多租户隔离里「不靠人记得带 tenant_id,靠机制兜」是同一个工程哲学——不信任不确定性,只信任确定性的机制。

深度展开:MCP server 的选型与评估

接一个外部工具时,经常遇到多个 MCP server 可选的情况(比如 Figma 就有官方的、社区的、Go 版的、Node 版的好几个)。怎么选?我们总结了几条评估标准。

第一条:看维护活跃度。 MCP 生态还很新,很多 server 是个人项目,作者不维护了就成弃子。看 GitHub 的最近 commit 时间、issue 响应速度、star 数。一个半年没更新的 server,大概率跟不上 Figma API 的变化,接入后迟早出兼容问题。我们选 figma-mcp-go 就是看它 commit 还在持续,issue 有人回。

第二条:看工具粒度。 有的 server 把一个功能拆成几十个细工具(读文件、读节点、读样式、读变量各自一个工具),有的合成了几个粗工具。细粒度灵活但 AI 容易选错(几十个工具里挑哪个,模型容易蒙);粗粒度省心但不够灵活。我们的偏好是「中等粒度」——一个能力一个工具,既不太碎也不太粗,AI 选起来不容易蒙。

第三条:看是否需要额外依赖。 figma-mcp-go 需要在 Figma 桌面端跑一个插件桥,这意味着每次操作不同文件都要手动 Run 一次插件——这个额外步骤增加了使用摩擦。官方 MCP 走 OAuth,一次授权后自动续期,使用更顺。如果一个 server 的依赖太重(要装一堆运行时、要手动配桥),即使功能强也要考虑「值不值这个麻烦」。

第四条:实际跑一遍看效果。 选型不能只看 README,得真跑一遍。我们评估时,让两个 server 各跑一次「导出一个按钮组件的截图和 Token」,看哪个返回的数据更干净、耗时更短、是否报错。跑一遍比读十遍文档管用——很多 server 的 README 写得天花乱坠,实际跑起来各种坑。

深度展开:MCP 的调试与排错

MCP 接入后,调试是个痛点:工具调用失败了,错误信息经常不明确——是 MCP server 挂了,还是网络问题,还是参数传错了?

我们的排查路径分三步。第一步看 Claude Code 的 MCP 状态:用 claude mcp list 看哪些 server 在线、哪些掉线。掉线了大概率是 server 进程挂了或网络不通,重启 server 或检查网络。第二步看工具调用的返回值:Claude Code 会显示每次工具调用的入参和返回,从这里能看到是参数传错了(比如 Figma file key 写错了)还是 server 返回了错误。第三步直接跑 server 的原始命令:绕过 Claude Code,手动跑 npx @vkhanhqui/figma-mcp-go,看它自己的日志输出——MCP server 的日志在 Claude Code 里看不到(它只返回 JSON-RPC 结果),手动跑才能看到 stderr 里的详细报错。

还有一个调试技巧:给 MCP server 加日志。如果 server 是自己能改的(比如自研的),在关键步骤加日志,记录「收到了什么请求、做了什么、返回了什么」。这些日志在排查「为什么返回结果不对」时极有价值——你能看到 server 内部的处理过程,而不是只看最终结果猜原因。社区的开源 server 改不了,但自研的一定要加日志,不然出了问题两眼一抹黑。
