接完 Figma 才搞懂:MCP 不是协议科普,是工具接入的标准化

上一篇我讲了怎么自研一层桥接,把工具定义和 AI 框架解耦:业务同学写工具零框架负担,换框架一行不改。那篇解决的是「框架耦合」。

但写完我意识到一个更上层的麻烦:每接一个外部工具,我还是要写一遍适配。接个订单查询,写个 OrderQueryTool;接个设计平台的 API,又得写一套。工具越多,适配代码越堆。直到我接 Figma,踩了一通坑,才真正搞懂 MCP(Model Context Protocol)到底在解决什么——它不是又一个新协议让你学,是把这个「每个工具都重写一遍适配」的活儿,标准化掉。

这篇就用我把 Figma 接进 AI 工作流的真实过程,讲 MCP 怎么落地。不讲虚的,讲配额怎么省、双 server 怎么配合、怎么装。

一、MCP 到底解决什么:M×N 变 M+N

先说它解决的那个根本问题。

假设你有 3 个 AI 客户端(Claude Code、Cursor、Cline),要接 3 个外部工具(Figma、GitHub、数据库)。传统做法是每个客户端给每个工具写一套适配——3×3=9 套。客户端加一个、工具加一个,适配数就是乘法涨。这就是 M×N 问题。

MCP 做的事是中间插一个标准协议:客户端只跟协议说话,工具也只跟协议说话。于是 3 个客户端 + 3 个工具,只需要 3+3=6 个适配,每边各管各的。涨法从乘法变加法。

落到 Figma 这个例子上:我不需要为「Claude Code 怎么调 Figma」写适配,也不需要为「Cursor 怎么调 Figma」再写一遍。Figma 那边出一个 MCP server,任何支持 MCP 的客户端都能直接用。这就是它最实在的价值——工具方写一次,所有客户端通吃。

二、协议三件套,知道这些就够用

MCP 的协议本身不复杂,记住三样:

消息格式:JSON-RPC 2.0。 客户端和 server 之间就是发 JSON 消息,请求带 method 和参数,响应带结果。没什么花活。

三类能力(原语):

  • Tools——模型可以调用的功能(比如 Figma 的「读取设计稿」「导出截图」)。这是用得最多的。
  • Resources——模型可以读取的数据(比如一份文档、一个配置)。
  • Prompts——预定义的提示模板(用得少,知道就行)。

外加几个反向能力(Sampling、Roots、Elicitation),新手不用管,用到再查。

传输方式(transport)两种:

  • stdio——本地进程,通过标准输入输出通信。适合跑在你机器上的工具。
  • Streamable HTTP——远程,一个 HTTP 端点搞定,支持流式。适合云端服务。

就这些。很多人被 MCP 吓住,是因为术语多,但骨架就这么点。后面全是往这个骨架上挂具体工具。

三、接 Figma:一个工具,为什么要两个 MCP server

这是整件事里我最想讲的一段,因为它最能说明 MCP 在真实项目里长什么样。

接 Figma 的时候,我没用「一个 server 搞定」,而是同时挂了两个:

  • figma-mcp-go:本地 server(stdio 传输),73 个读写工具,能绑定设计变量、管 Token、导出截图。
  • 官方 Figma MCP:云端 server(Streamable HTTP),核心是 Code to Canvas——能把一段 HTML 变成 Figma 里可编辑的图层。

为什么非要两个?因为各自都有硬伤,合起来才完整:

figma-mcp-go 不能导入 SVG 图标,创建渐变这种复杂样式也很费劲;而官方 MCP 的 Code to Canvas 恰好能把 HTML(里面塞 SVG、CSS 渐变、阴影)原样转成 Figma 图层。反过来,官方 MCP 没法做精细的变量绑定,figma-mcp-go 这块强。

所以分工是:官方 MCP 负责「生成高品质图层」,figma-mcp-go 负责「绑定 Token + 精细操控」。一个生成,一个打磨。

这给了我一个挺重要的认知:MCP 不是「一个工具一个 server」的对应关系。一个外部能力,可能要拆成几个 server 组合着用,各自补对方的短板。理解工具的能力边界,比理解协议本身更关键。

四、配额这个坑,差点让我白干一天

接完能用之后,第一个让我吃瘪的是配额。

官方 Figma MCP 免费版有个限制:读工具每月只能调 6 次。get_design_context、get_screenshot 这些都算。而写工具(generate_figma_design,也就是 Code to Canvas)不限次数。

我一开始没注意,边调边试,读一次看一眼结构、再读一次截图对照,几个组件没弄完,6 次就烧光了,后面大半天读工具全报额度用完。

教训很直接,后来我立了个规矩:

把读取集中批处理。 一次性把要看的组件、截图、Token 定义全捞回来,别「读一点改一点再读」。读操作是稀缺资源,得当成预算花。

能写就别读。 既然写工具不限次,设计生成就大胆用 generate_figma_design;需要确认效果时,合并成一次 get_screenshot,而不是每个小改动都截一次。

Token 两端对齐靠脚本,不靠肉眼。 用一个同步脚本把 Figma 里的变量导出成 CSS,跟项目里的 variables.css 比对,而不是一次次读 Figma 确认颜色对不对。

这套打法跑通后,6 次配额能撑住一整轮设计还原。MCP 的工具是好工具,但配额模型你得先摸清,不然会卡在半路干瞪眼。

五、Code to Canvas:设计到代码的桥

官方 MCP 那个最香的能力叫 Code to Canvas,值得单独说,因为它是把「设计」和「代码」两头打通的关键。

它的玩法是反过来——不是从 Figma 读设计生成代码,而是把代码(HTML)变成 Figma 图层:

  1. 我(或 AI)按设计规范写一段 HTML,里面直接内联 SVG 图标、写 CSS 渐变和阴影。
  2. HTML 里注入一段官方的 capture.js 脚本。
  3. 起个本地 HTTP 服务托管这个页面(python3 -m http.server),注意必须用 HTTP,不能用 file://,否则捕获脚本跑不起来。
  4. 调 generate_figma_design,拿到一个 captureId,浏览器打开带这个 id 的链接。
  5. capture.js 捕获页面结构,几秒后 Figma 里就出现了可编辑的图层——矢量、文字、渐变都保留。

这一步的价值在于:AI 写 HTML 比让它直接操作 Figma 节点靠谱得多(HTML 是 AI 的母语),而 Code to Canvas 把这份 HTML 干净地翻译成设计稿。等于让 AI 走它最擅长的路去产出设计,再交给 figma-mcp-go 去绑 Token、精修。

两个 server 的分工,到这一步就闭环了:HTML(代码)→ 官方 MCP → Figma 图层(设计)→ figma-mcp-go → 绑定 Token + 组件化 → get_design_context → 还原成 Vue 代码。设计稿和代码两头都能走通,而且都走 MCP,不用人肉在两个世界之间搬。

六、怎么装:.claude/mcp.json + 一条命令

讲完道理,给能直接照做的安装步骤。

MCP server 的配置写在项目的 .claude/mcp.json 里。加一个 server 最方便的方式是用 claude mcp add:

# 本地 server(stdio):figma-mcp-go,73 个读写工具
claude mcp add -s project figma-mcp-go -- npx -y @vkhanhqui/figma-mcp-go@latest

# 云端 server(Streamable HTTP):官方 Figma MCP
claude mcp add --transport http figma-official https://mcp.figma.com/mcp

跑完之后,.claude/mcp.json 长这样:

{
  "mcpServers": {
    "figma-mcp-go": {
      "command": "npx",
      "args": ["-y", "@vkhanhqui/figma-mcp-go@latest"]
    },
    "figma-official": {
      "type": "http",
      "url": "https://mcp.figma.com/mcp"
    }
  }
}

本地那个(figma-mcp-go)还多一步:它需要 Figma 桌面端跑一个插件桥。流程是下插件包、在 Figma 桌面版里 Plugins → Development → Import plugin from manifest... 导入。每次操作不同文件,都要在新文件里 Run 一次这个插件,否则它还连着上一个文件。

云端那个(官方 MCP)首次用要走一遍 OAuth:Claude 会弹个浏览器链接,你在 Figma 授权页点允许,浏览器跳到一个 localhost 回调地址。如果报连接错误,把地址栏的完整 URL 复制出来贴给 Claude 就行——这是它拿 token 的方式。

改完 mcp.json 记得重启 Claude Code 会话才生效。装完用 get_metadata()(本地)或 whoami()(官方)各验证一下连通。

七、什么时候用 MCP,什么时候还是自研桥

写到这里,自然会问:那我上一篇那个自研 Tool Calling 桥接,是不是多余了?MCP 能不能全包?

我的答案是分场景:

MCP 适合接「外部、标准化、复用度高」的工具。 像 Figma、GitHub、数据库、Slack 这些,工具方或社区已经出了成熟的 MCP server,你挂上就能用,别自己造。这类工具的特点是:协议相对固定、跨项目复用、你不掌握它的内部实现。

自研桥接适合「内部、业务强相关、需要深度约束」的工具。 像订单查询、分润这种业务方法,要穿透多租户上下文、要走确认门禁、要落审计日志——这些约束是你的业务独有的,MCP 不会替你管。这种就该自己写一层薄桥接,把业务约束焊死。

两者不冲突,是不同层:MCP 解决「外部工具怎么标准接入」,自研桥接解决「内部工具怎么安全调用」。一个管广度(接得多),一个管深度(管得严)。成熟的 AI 工作流,往往是 MCP + 自研桥并用。

八、收个尾

接 Figma 这件事,本来以为是个「装个插件」的小活,结果从双 server 分工、配额限制、Code to Canvas 工作流到安装配置,折腾出一套完整经验。

回头看,MCP 真正的价值不在协议本身有多精巧——JSON-RPC 加几个原语,没什么神秘的。它的价值在于把工具接入这件事,从一个「每次重写」的体力活,变成一个「挂载即用」的标准动作。工具方写一次,所有客户端受益;客户端接一次,所有工具可用。

但 MCP 也不是银弹。它管的是「接得上」,管不了「接得对」——配额怎么分配、能力边界在哪、业务约束怎么补,这些还得你自己摸。Figma 这个例子里,真正费心力的不是协议,是搞清两个 server 各自能干什么、不能干什么,然后把它们的短板互补起来。

工具会越来越多,MCP 让接它们的成本趋近于零。但「知道该用哪个工具、怎么组合、在哪收口」——这层判断,是工具替代不了的。这大概也是我接完一堆工具之后,最确定的一件事。
