这篇属于「前端 → 全栈 → AI 开发」这条线里的 AI 开发阶段。前面那些文章讲的是用 AI 写代码,这篇换个方向:用 AI 给产品做能力。具体是用 Spring AI 1.0.6 从 0 到 1 搭一个多租户 AI 助手,做成能上线、能治理、有安全边界的生产系统,而不只是跑通一个 demo。

一、要解决的核心问题

运营后台要一个 AI 助手,运营同学能用自然语言查数据、改配置:“查一下今天的订单”、“把这几个订单打上「已处理」标签”。

难点不在"能不能对话",在多租户。我们的平台服务很多租户,每个租户:

  • 用自己的大模型(A 租户用 DeepSeek、B 租户用通义千问);
  • 用自己的 API Key(各自的账单和额度);
  • 有自己的知识库(检索各自的运营文档);
  • 有自己的限流和预算。

这意味着不能像 demo 那样写死一个全局 ChatModel,所有东西都得按租户动态来。多租户和单租户的差别基本就在这一句,这篇也围绕它展开。

二、整体架构

先看全貌。一次"运营问一句话、AI 流式回答并可能调工具"的链路:

关键设计决策,后面逐个展开:

  1. 多供应商统一走 OpenAI 协议,一套代码吃三家模型。
  2. 多租户隔离落在模型实例这一层,每个租户一个独立 ChatModel。
  3. RAG 靠 metadata filter,一个索引管所有租户。
  4. Agent 工具定义和 Spring AI 解耦,中间自研一层桥接。
  5. SSE 流式,前后端各有一套处理。

三、多供应商:三家模型,一套代码

我对接了 DeepSeek、智谱 GLM、通义千问三家,没有为任何一家写独立 SDK。

原因是这三家都兼容 OpenAI 协议。所以我只用 Spring AI 的 OpenAiChatModel,靠 baseUrl + apiKey + model 三个参数区分:

// SpringAiConfig.java —— 三家供应商的默认值
private static final Map<String, ProviderDefaults> DEFAULTS = Map.of(
    "deepseek", new ProviderDefaults("https://api.deepseek.com", "deepseek-v4-flash"),
    "glm",      new ProviderDefaults("https://open.bigmodel.cn/api/paas", "glm-4-flash"),
    "qwen",     new ProviderDefaults("https://dashscope.aliyuncs.com/compatible-mode/v1", "qwen-plus")
);

构建模型就一段:

OpenAiApi api = OpenAiApi.builder().apiKey(apiKey).baseUrl(baseUrl).build();
OpenAiChatOptions options = OpenAiChatOptions.builder()
    .model(model).temperature(temperature).maxTokens(maxTokens).build();
ChatModel chatModel = OpenAiChatModel.builder().openAiApi(api).defaultOptions(options).build();

这里有个反直觉的取舍:我没有用 Spring AI 的 starter 自动装配(spring-ai-starter-model-openai),而是手动构建。原因是 starter 要求在 application.yml 里配一个全局 api-key,但我们的 key 是按租户动态来的,全局 key 这个前提就不成立。手动构建换来多租户的灵活度。

方案优点缺点
每家模型独立 SDK能用各家特色功能三套代码、维护成本高、加一家写一套
统一 OpenAI 协议(我用)一套代码,加供应商加一行默认值用不了各家非标准能力
Spring AI starter 自动装配零配置只能一个全局 key,不满足多租户

所以构建 AI 应用时,别为每家模型单独写一套适配。国内外主流模型大多兼容 OpenAI 协议,统一走它,加一个供应商就是加一行默认值。

四、多租户:动态 ChatModel 工厂(核心)

多租户这块是整个系统最花心思的地方。

4.1 配置存哪:tenant_ai_config 表

每个租户的模型配置存一张表:

CREATE TABLE tenant_ai_config (
  tenant_id            VARCHAR PRIMARY KEY,
  provider             VARCHAR(32) DEFAULT 'deepseek',
  base_url             VARCHAR,
  model                VARCHAR(64) DEFAULT 'deepseek-v4-flash',
  api_key_encrypted    VARCHAR(512),   -- AES 加密
  temperature          DECIMAL(3,2) DEFAULT 0.7,
  max_tokens           INT DEFAULT 4096,
  enabled              TINYINT DEFAULT 0,
  rate_limit_rpm       INT DEFAULT 30,
  monthly_token_limit  BIGINT DEFAULT 10000000,
  ...
);

读取分两个视图——这是安全设计的关键:

  • 给前端(getConfig):API Key 自动脱敏,返回 wx12****cd 这种,永远不返回明文。
  • 给工厂(getRawConfig + getDecryptedApiKey):返回解密后的明文,用于构建模型。

写入时加密落库;更新时识别脱敏占位符(前端把 **** 原样传回)跳过覆盖,避免把脱敏值当真值存进去。

4.2 动态 ChatModel 工厂

真正核心的是这个工厂。它内部用 ConcurrentHashMap<String, ChatModel> 按 tenantId 缓存模型实例:

public class TenantChatModelFactory {
    private final ConcurrentHashMap<String, ChatModel> cache = new ConcurrentHashMap<>();

    public ChatModel getOrCreate(String tenantId) {
        return cache.computeIfAbsent(tenantId, this::createForTenant);
    }

    public void evict(String tenantId) {
        cache.remove(tenantId);
    }

    private ChatModel createForTenant(String tenantId) {
        TenantAiConfig config = tenantAiConfigService.getRawConfig(tenantId);
        ProviderDefaults defaults = DEFAULTS.getOrDefault(config.getProvider(), DEFAULTS.get("deepseek"));
        String baseUrl = StringUtils.defaultIfBlank(config.getBaseUrl(), defaults.baseUrl());
        String model   = StringUtils.defaultIfBlank(config.getModel(), defaults.model());
        String apiKey  = tenantAiConfigService.getDecryptedApiKey(tenantId);
        // ... 用 baseUrl/apiKey/model 构建 OpenAiChatModel(见上一节)
        return chatModel;
    }
}

两个设计点:

  • 懒加载加缓存(computeIfAbsent),第一次用到某租户才建模型,建完缓存复用。
  • evict 失效。租户在后台改了模型或 Key,清掉这个租户的缓存,下次请求重建,配置变更实时生效,不用重启。

租户隔离是落在模型实例这一层的。每个租户有自己的 ChatModel 实例,带自己的 Key 和参数,不是靠在 prompt 里提醒模型“你是哪个租户”。隔离是硬的。

方案隔离强度复杂度
全局一个模型 + prompt 里带租户弱(模型可能串)低
一租户一独立部署强极高
动态 ChatModel 工厂 + 缓存(我用)强(实例级)中

五、RAG:知识库管线

助手要能基于租户的私有运营文档回答,靠 RAG。一句话讲清:问大模型之前,先从租户文档里检索出相关片段,塞进 prompt。

拆开四步:切片 → 向量化 → 存储 → 检索。

5.1 切片:TokenTextSplitter

private static final TokenTextSplitter SPLITTER =
    new TokenTextSplitter(500, 50, 50, 100, true);  // chunk 500 token, overlap 50

overlap 这块要注意,相邻块留了 50 token 重叠,避免一句话被从中间切断、检索时上下文丢失。500 token 是针对运营文档(段落式说明)调出来的。

5.2 存储 + 检索:Elasticsearch 向量库 + metadata filter

入库时给每个 chunk 注入 tenant_id 元数据:

Document doc = new Document(content, Map.of(
    "tenant_id", tenantId, "doc_id", docId,
    "title", title, "tags", tags, "source", source));
List<Document> chunks = SPLITTER.apply(List.of(doc));
vectorStore.add(chunks);  // Spring AI 自动算 embedding 写 ES

检索时,用 filterExpression 限定本租户:

vectorStore.similaritySearch(SearchRequest.builder()
    .query(query).topK(5)
    .similarityThreshold(0.7)                              // 相似度低于 0.7 不返回
    .filterExpression("tenant_id == '" + tenantId + "'")   // 多租户隔离
    .build());

多租户隔离靠 metadata filter。所有租户共用一个 ai_knowledge 索引,靠元数据过滤分开,不用给每个租户建索引。

方案隔离管理/存储成本
一租户一独立索引物理隔离高(索引爆炸)
共用索引 + metadata filter(我用)逻辑隔离(够强)低

similarityThreshold(0.7) 是个质量闸,相似度低的检索结果不塞进 prompt,免得带噪声进去。

六、Agent:自研 Tool 桥接到 Spring AI

助手要"能干活"而不只是"能聊天",靠 Tool Calling:给模型一组工具,模型自己判断该不该调、调哪个、传什么参数。

6.1 关键决定:工具定义不依赖 Spring AI

我没有直接用 Spring AI 的 @Tool 注解,而是自研了一层。理由很直接:工具是业务资产,框架会换。每个工具都写死 Spring AI 的类型,换框架时就得全改一遍。

@AiTool(name = "query_orders", description = "查询订单列表",
        requiresConfirmation = false, module = "trade")
public class OrderQueryTool implements ToolHandler {
    @Override
    public ToolResult execute(Map<String, Object> args, ToolContext ctx) {
        // ctx.tenantId() 做租户隔离,调业务 service
        return ToolResult.success(data);
    }
}

业务同学写的工具类里,完全看不到 Spring AI 的任何类型。

这层的类关系:

业务工具(如 OrderQueryTool)只依赖自研的 ToolHandler 接口和 @AiTool 注解;桥接层负责把它适配成 Spring AI 的 FunctionToolCallback。换 AI 框架时,只要改桥接层,业务工具一行不动。

6.2 桥接层:适配到 FunctionToolCallback

一个 AiToolBridgeConfig 把两边接起来,用 Spring AI 的 FunctionToolCallback(不是 @Tool,更适合程序化注册):

FunctionToolCallback.builder(meta.name(), (args, springCtx) -> {
        ToolContext toolCtx = new ToolContext(
            (String) springCtx.getContext().get("tenantId"),
            (String) springCtx.getContext().get("userId"),
            ...);
        return handler.execute(args, toolCtx);
    })
    .description(meta.description())
    .inputType(Map.class)   // Spring AI 反射生成 JSON Schema
    .build();

6.3 让模型自动多轮调度

OpenAiChatOptions options = OpenAiChatOptions.builder()
    .toolCallbacks(toolCallbacks)
    .internalToolExecutionEnabled(true)   // Spring AI 自动跑工具调用循环
    .toolContext(Map.of("tenantId", tenantId, "userId", userId))
    .build();

internalToolExecutionEnabled(true) 是关键——模型说要调 query_orders → 框架自动调 → 结果喂回模型 → 模型继续生成。业务代码不写这个循环。toolContext 把租户上下文透传进每次工具调用,工具里 ctx.tenantId() 直接用——多租户隔离穿透到工具层。

我注册了 16 个工具,分两类:

类别工具是否需确认
只读查询query_orders / get_order_detail / list_products / get_shop_info / search_users / query_reviews / query_stats / list_wechat_templates否
写操作create_product / update_product / update_shop / update_order_remark / batch_remark_orders / batch_tag_orders / approve_proxy_order / publish_miniapp是

写操作配 requiresConfirmation = true,走确认门禁(下一节)。每次工具调用还落一条审计日志(tool_name / args / result / 耗时),Agent 干了什么全可追溯。

七、SSE 流式:前后端各一套

流式输出我没省,直接按标配做了,用 SSE(Server-Sent Events)。

7.1 后端:WebFlux Flux 跑在 servlet 栈上

@PostMapping(value = "/chat", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<Map<String, Object>>> chat(@RequestBody ChatRequestDTO req) {
    return chatService.orchestrate(tenantId, userId, convId, req.content(), null)
        .map(delta -> ServerSentEvent.<Map<String,Object>>builder()
            .event("delta").data(Map.of("type", "content", "delta", delta)).build())
        .concatWith(Flux.just(ServerSentEvent.<Map<String,Object>>builder()
            .event("delta").data(Map.of("type", "done", "conversationId", convId)).build()));
}

项目是 servlet 栈(Spring MVC),但引入 spring-boot-starter-webflux 后,MVC 接口也能返回 Flux<ServerSentEvent>——reactive 返回类型在 servlet 上也支持,不用整个切到 WebFlux。

7.2 前端:为什么不用 EventSource

这里踩过一个坑:EventSource 不支持 POST。我们的 chat 接口是 POST(带 body),所以前端不能用 EventSource,得用 fetch() + ReadableStream 自己解析:

// useAssistantSSE.ts —— 不用 EventSource,用 fetch + ReadableStream
const resp = await fetch('/api/platform/ai/chat', {
  method: 'POST', body: JSON.stringify({ conversationId, content, stream: true })
});
const reader = resp.body!.getReader();
const decoder = new TextDecoder();
let buffer = '';
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  buffer += decoder.decode(value, { stream: true });
  // 按 \n 切行,解析 data: 前缀的 JSON
  buffer = consumeSseLines(buffer, chunk => {
    if (chunk.type === 'content') appendDelta(chunk.delta);
    if (chunk.type === 'done') finish();
  });
}

前端用 Pinia 存消息流,每个 delta 原地更新那条"正在生成"的消息,实现打字机效果。

八、安全护栏:Prompt + 代码双层

AI 接进生产,安全和能力得分开看。我做了两层:

  • Prompt 层,系统提示词里写死资金类的硬禁令(不能查、不能改退款、支付、提现、分佣、结算),模型被告知这些事它不做。
  • 代码层,写操作工具标 requiresConfirmation = true,配一套确认门禁:Redis token 加参数 SHA-256 哈希,5 分钟 TTL,消费即删,防重放也防误触。模型要执行写操作,先返回一个待确认令牌给前端,运营确认后才真正执行。

加上 Redis 滑动窗口限流(超限 429)和全量工具调用审计日志,Agent 的每一次动作都可追溯、可限流、可拦。

九、几条沉淀下来的判断

做完这套,有几点我比较确定:

  1. 模型对接走 OpenAI 协议就够,别为每家写 SDK,加供应商就是加一行配置。
  2. 多租户隔离落在模型实例这一层才靠谱,动态工厂加缓存加 evict 失效。
  3. 工具定义和 AI 框架解耦,自研一层注解,以后换框架不用动业务工具。
  4. RAG 的多租户用 metadata filter,一个索引就够。
  5. 流式直接做,SSE 那个 EventSource 不支持 POST 的坑提前知道能省不少事。
  6. 安全护栏放 prompt 和代码两层,出了事能追溯,比能力多更重要。

调通一个 demo 和做成生产系统之间,差的就是多租户、治理、安全这些。这篇是总纲,RAG、Agent、多租户配置几个模块我拆成了单独的深度篇,链接在下面。
