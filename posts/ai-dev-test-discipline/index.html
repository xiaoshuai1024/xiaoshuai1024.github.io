测试金字塔谁画都顺手:底下 UT、中间 IT、顶上 E2E,贴在 PPT 上特别好看。可真到了项目里,这套三角形十有八九会缩水:

  • UT 漏边界,只测 happy path;
  • IT 跑得慢,有人嫌烦干脆 @Ignore;
  • E2E 全绿但一上线就翻车——因为只点了「正常路径」。

我以前也画过那种金字塔,画完贴墙上,没几个人看。金字塔不画在墙上,写进纪律里——而且这条纪律不仅人要守,Agent 也要守。

下面是我们怎么把「三层测试」从口号落成可执行规矩的。

三层,各兜一摊,谁也替不了谁

先把分工说清楚,省得有人想「用 UT 多写点、E2E 少写点」来偷懒:

层级命名约定兜什么跑多快典型反模式
UT*Test纯逻辑、边界、异常分支毫秒级只测 happy path,边界一个不测
IT*IT真实数据库 / 中间件集成秒级mock 掉一切,等于没集成
E2E*E2E真实用户路径,端到端分钟级全绿但只点了正常路径

这三层不能互相替代,理由很具体:

  • UT 全绿不代表集成没问题——你 mock 掉的那个表结构,可能压根和数据库对不上。我见过一个 UserMapperTest 全绿,但实际表里根本没有 deleted_at 字段,上线一查全表 NPE。
  • E2E 全绿不代表每个分支都对——你只点了「正常登录 + 正常下单」,边界条件(库存为 0、余额不足、并发抢同一商品)一个没覆盖。

不是「三层选一层」,是三层都得在,各管一段。任何「我们 UT 写得很全,E2E 就少点」的说法,都是在为偷懒找借口。

把「真金字塔」和「反金字塔」画出来对比,为什么底宽顶窄才对、倒过来为什么崩,一目了然:

反金字塔(E2E 肥大)是大多数团队的真实状态。UT 少写、IT 被 mock 掉、所有用例堆进 E2E,跑一次 40 分钟,团队嫌慢就降频率——从每次提交跑变成每天跑,再变成每周跑,最后 E2E 形同虚设,名义上有测试,实际没人看结果。金字塔倒过来不是「测试多了」,是「测试错位了」——快的该多的反而少,慢的该少的反而多,整套测试又慢又没用。

纪律一:E2E fail-fast,第一个失败就停

E2E 最贵的不是写,是跑。一个 E2E 套件几十分钟不稀奇。

如果第一个用例就挂了,后面几十个还在吭哧吭哧跑,跑完一堆红——纯浪费时间,还污染日志。我们直接定死 fail-fast:

// playwright.config.ts
export default defineConfig({
  // 第一个失败立刻停,不让后面的白跑
  maxFailures: 1,
  retries: 0,           // E2E 不重试,失败就是失败,重试会掩盖 flaky
  // ...
})
<!-- Maven surefire:IT 层也 fail-fast -->
<plugin>
  <artifactId>maven-surefire-plugin</artifactId>
  <configuration>
    <!-- 第一个失败就停,不再跑后面的 IT -->
    <skipAfterFailureCount>1</skipAfterFailureCount>
  </configuration>
</plugin>

理由很直接:E2E 是端到端的,前面的失败大概率会传染后面。比如登录挂了,后面所有用例都白跑(没登录态);首屏数据没加载,后面所有交互都失败。停在第一个,省下来的时间够你去看一眼到底哪断了,而不是等半小时后看一屏红。

这条对 Agent 尤其重要。Agent 不会自己判断「这个失败要不要继续」,所以必须用配置把它钉死,不让它有发挥空间。一旦给它「可以继续」的余地,它就会为了凑「跑完了」的指标,把一堆红跑完然后汇报「测试已执行」——执行了但没过,等于没测。

fail-fast 的代价:套件没跑完,不知道后面有没有别的 bug

这是 fail-fast 最被质疑的点。我的回应:E2E 的目的是「抓关键路径的崩溃」,不是「穷尽所有 bug」。如果第一个就挂了,说明关键路径已经崩了,后面的 bug 根本不重要——先把关键路径修好再说。

后面的 bug 由 UT / IT 来抓,它们便宜、快、可以全量跑。E2E 跑全量是奢侈,fail-fast 才是工程化。

fail-fast 还有个容易被忽略的配套措施:失败后要能快速复现。E2E 挂在第一个用例,如果只告诉你「失败了」,排查还是很慢——得手动跑一遍才知道为啥挂。我们的做法是 E2E 失败时自动截图 + 录视频 + dump 请求日志,失败报告里直接附带这些,让人不用重跑就能看到「页面长什么样、哪个请求 500 了」。没有这些附件,fail-fast 省下的时间又耗在「复现」上了,等于白省。fail-fast 配合「失败可复现」,才真正省时间。

还有一个关于 fail-fast 的误区:它不适用于 UT。UT 跑得快(毫秒级),全量跑完也就几秒,没必要 fail-fast——全跑完反而能看到「这次改动一共挂了几个 UT」,信息更全。fail-fast 是为「跑得慢、失败会传染」的 E2E 和 IT 设计的,不是所有层都该套。不分层地一律 fail-fast,UT 本来能一次看到 5 个失败,变成只看到第 1 个,反而拖慢修复。纪律要分场景,不能一刀切。

纪律二:禁止跳过,禁止占位

这条是我跟团队(跟 Agent)吵过最多的一条。

什么算「跳过」?@Ignore、skip、it.skip、条件性跳过(if 环境变量就 skip)——统统不许,明令禁止。连 Agent 自己都不许自动加 skip 来「先把测试跑绿」。

// ❌ 禁止:为了让构建通过而 skip
@Ignore("暂时跳过,环境有问题")
@Test
void testConcurrentOrder() { ... }

// ❌ 禁止:条件性跳过,是逃避
@Test
void testPay() {
    assumeTrue(System.getenv("PAY_ENV") != null);  // 没配环境变量就跳过
    // ...
}

更狠的是连占位测试也禁。就是那种 expect(true).toBe(true)、一个空壳断言、为了凑覆盖率凑门禁的「假测试」:

// ❌ 禁止:占位测试,凑覆盖率的假绿
it('should work', () => {
  expect(true).toBe(true)
})

看着绿,其实啥也没测,纯债务。我们见过不少 AI 生成的测试就长这样——为了「让测试通过」而写测试,本末倒置了。AI 的目标函数是「让测试绿」,如果你不禁止占位,它就会用最省事的方式达成绿——写一堆 expect(true).toBe(true)。

跳过和占位是同一种债:假装绿了

跳过和占位的共同点是:让测试报告显示绿,但其实没测。这种「假绿」比「真红」危害大十倍——真红至少告诉你有问题,假绿让你以为没问题,直到上线翻车。

我的原则:与其留一个绿油油的空壳,不如让它红着——至少红是诚实的。一个诚实的红,比一百个虚伪的绿有价值。

怎么 enforce 这两条禁令

光写在 wiki 里没用,得有机器化检查。我们的做法:

# pre-commit hook:扫测试文件,发现 skip / 占位 直接 fail
#!/bin/bash
# 1. 扫 @Ignore / it.skip / xit / xdescribe
if grep -rE '(@Ignore|it\.skip\(|xit\(|xdescribe\()' src/test/ ; then
  echo "❌ 发现跳过的测试,禁止提交。删掉 skip 或真正修好它。"
  exit 1
fi
# 2. 扫占位断言(expect(true).toBe(true) 这类)
if grep -rE 'expect\(true\)\.(toBe|toEqual)\(true\)' src/test/ ; then
  echo "❌ 发现占位测试,禁止提交。写真正断言。"
  exit 1
fi
echo "✅ 无跳过、无占位"

挂在 pre-commit,谁想用 skip 蒙混,本地提交就被拦。Agent 想自动加 skip 也一样被拦——它加了 skip,commit 不进去,只能老老实实写真测试。

pre-commit hook 还有个实操细节:hook 脚本本身要进版本管理。我们早期把 hook 放在每个人本地的 .git/hooks/ 里,结果有人忘了装、有人装的是旧版,扫描规则不一致,有人能 commit 有人不能。后来把 hook 脚本放进仓库(比如 .githooks/pre-commit),配一个 make install-hooks 一键安装,保证所有人的扫描规则一致。规则不统一,等于没有规则——有人严有人松,严的人觉得不公平,松的人钻空子,最后规则就废了。

这套禁令对 Agent 的约束力比对人更强。人会碍于面子不好意思加 skip(「别人都写真测试,我加 skip 不好看」),Agent 没有面子概念,它只看目标函数——如果你不禁止 skip,它会毫不犹豫地用 skip 来达成「测试通过」。所以对 Agent 来说,禁令不是「建议」,是「物理约束」:hook 拦住了,它想做也做不到。把纪律变成物理约束,比写在规范文档里有用一百倍。

纪律三:按页面类型规定最小用例矩阵

「测试要够」是废话,「够」是几个?我们干脆按页面类型写死最小用例数:

页面类型最少用例数必测场景为什么
列表页2 条正常数据 + 空态空列表那个边界最容易翻(分页、占位文案)
表单页2 条成功提交 + 校验失败只测成功路径的表单,等于没测
状态机N 条每个合法迁移 + 至少 1 个非法迁移非法迁移最值钱,往往能逮住越权
删除1 条确认删 + 删完真的消失软删了还在列表里,是最常见的"删了没删干净"
支付 / 金额3 条正常 + 余额不足 + 并发重复支付并发那个最贵,但最该测

这套矩阵的好处是:Agent 也能照着填。它不用判断「够不够」,照着矩阵凑就行;评审 Agent 也能照着查——少一条直接打回。

矩阵的维护有个要点:它是活的,不是写完就不动。每次线上出了 bug,复盘时都要问「这个 bug 的场景,矩阵里有没有覆盖」。没覆盖就加一条——矩阵就是这样从最初的 5 条慢慢长到十几条的。如果矩阵写完就挂墙上不更新,它和真实风险就会脱节:新业务场景没对应的用例类型,老类型里的必测场景过时了也没人删。矩阵要跟着业务走,定期拿线上 bug 回溯校准,它才一直有用。

还有个关于矩阵的边界判断:矩阵是下限不是上限。矩阵规定「列表页至少 2 条」,不是说列表页只能写 2 条——复杂的列表页(带筛选、带批量操作、带权限控制)写 5-6 条完全正常。矩阵防止的是「少写」,不是限制「多写」。有些团队把矩阵当上限用,「矩阵说 2 条,我就写 2 条,多了不写」,这又走偏了——矩阵是底线,不是天花板。

一个反面案例:为什么"非法迁移"必测

状态机的「非法迁移」用例,是我们吃过亏才加进矩阵的。比如订单状态机,合法迁移是 待支付 → 已支付 → 已发货,非法迁移是 待支付 → 已发货(跳过支付直接发货)。

某次一个 bug 让待支付订单能直接被标记为已发货——逻辑上用户没付钱就收到了货。人工评审没看出来(代码看着是正常的状态更新),UT 也没测(只测了合法迁移)。最后是 E2E 里加了「非法迁移必须失败」的用例才逮住。

从那以后,状态机的非法迁移用例进了矩阵,强制至少 1 条。这条用例的价值是:它测的不是"功能能不能用",而是"防线有没有被绕过"——这是安全相关的测试,漏了就是越权。

纪律四:测试数据契约

测试数据这事儿,不立规矩迟早互相踩。我们的契约就几条:

  • 租户软删除:别真把数据物理删了,下一个用例就没了。测试造的数据用 deleted_at 标记,跑完软删,不物理删。
  • 固定账号复用:别每次新建账号,环境会越来越脏。我们维护一套 test-tenant-A / test-tenant-B,所有测试用这两套,跑完清状态而不是清数据。
  • 谁建谁清:测试自己造的数据自己收,不污染别人。用 @BeforeEach 建,@AfterEach 清,禁止跨用例共享状态(那是 flaky 的源头)。
// ✅ 推荐:每个测试自建自清,不依赖执行顺序
@BeforeEach
void setUp() {
    // 用唯一前缀,避免和其他测试撞数据
    testOrderId = "test-" + UUID.randomUUID();
    createTestOrder(testOrderId);
}

@AfterEach
void tearDown() {
    // 软删自己造的数据,不物理删
    softDeleteOrder(testOrderId);
}

@Test
void testOrderFlow() {
    // 测试逻辑,只依赖 setUp 造的数据
}

这条看起来琐碎,但在多租户项目里是命根子。测试数据乱跑,轻则用例间互相干扰、偶发红(最难排查的 flaky),重则真把别的租户数据碰了(测试环境污染生产统计)。

测试数据的「自建自清」还有个容易翻车的场景:并发测试。多个测试并行跑时,如果都用同一个租户、同一批数据,@BeforeEach 建的数据可能撞在一起——测试 A 建了 test-order-001,测试 B 也想建 test-order-001,主键冲突直接挂。我们的解法是数据 ID 带随机后缀(test-order- + UUID),保证每个测试的数据互不干扰,并行跑也不会撞。这个细节不处理,测试就只能串行跑,速度直接掉一个量级;加了随机后缀,并行跑稳了,CI 时间砍一半。

还有个关于「清」的取舍:软删 vs 物理删。我们选软删(deleted_at 标记)而不是物理删,原因是物理删有时会触发级联删除或外键约束,把别的测试依赖的数据也删了。软删只标记不动数据,副作用最小。代价是测试数据会慢慢积累(软删的还在库里),所以才有第三层防线(前缀清理)定时扫 test- 前缀的软删数据做物理清理。日常跑用软删保证安全,定期清理防止膨胀,两者配合。

测试数据隔离的三层防线

光靠"约定"不够,我们还有三层机器化防线:

  1. 测试库独立:IT / E2E 跑的是独立的 test database,绝不连生产。
  2. 租户隔离:所有测试数据挂在 test-tenant-* 名下,生产租户 tenant_id 永远不出现测试数据。
  3. 数据带前缀:测试造的所有数据带 test- 前缀,定时任务扫 test- 前缀数据清理,防止长期积累污染。

三层叠加,即使某一层漏了,另外两层兜底。测试数据的安全,靠防线,不靠自觉。

这三层防线画出来,「每层挡什么风险」就清楚了:

三层防线的设计逻辑和前面讲的多租户隔离一样——不指望单层完美,靠多层叠加把漏的概率压低。第一层(独立测试库)是最基本的物理隔离,挡住「测试直接连生产」这种灾难性误操作;第二层(租户隔离)挡住「测试数据混进生产租户」,即使在共享环境里也能区分;第三层(前缀清理)挡住「测试数据长期堆积」,防止环境慢慢腐化。三层各有盲区,叠在一起才完整。

第三层(前缀清理)有个实操细节值得强调:清理脚本要幂等且只认前缀。我们早期犯过一个错——清理脚本写了 DELETE FROM orders WHERE created_at < NOW() - INTERVAL 7 DAY,本意是删 7 天前的测试数据,结果把一条 8 天前造但还在排查中的真实测试数据也删了。后来改成严格只删 test- 前缀的数据,绝不按时间删——时间判断会误伤,前缀判断不会。清理脚本的容错性,和测试数据本身的命名规范是绑定的,命名规范不守,清理脚本就不敢跑。

实战:一次多租户越权怎么被网住

讲个真事,这套纪律怎么真金白银地救了我们。

某次一个业务模块的查询接口,JOIN 表的时候漏了 tenant_id 过滤——典型的多租户越权,租户 A 能查到租户 B 的订单数据。这玩意儿人工评审很容易漏(SQL 看着没毛病,JOIN 语法正确),编译更管不着(类型完全对得上)。

最后是这么被网住的:

  • UT 没抓到——UT 层 mock 了 Mapper,只测了 Service 逻辑,SQL 根本没跑到。
  • IT 抓到了——IT 用 testcontainers 起真实 MySQL,跑了两条用例:租户 A 查询,断言结果里没有租户 B 的订单 ID。一跑就红,fail-fast 一停,直接定位到那个 JOIN。
// 这条 IT 用例逮住了越权
@Test
void testTenantIsolation() {
    // 造数据:租户 A 和租户 B 各一个订单
    Long orderA = createOrder("tenant-A", "订单 A");
    Long orderB = createOrder("tenant-B", "订单 B");
    
    // 用租户 A 的身份查询
    List<Order> results = orderService.listOrders("tenant-A");
    
    // 断言:A 只能看到自己的订单,看不到 B 的
    assertThat(results).extracting(Order::getId)
        .contains(orderA)
        .doesNotContain(orderB);  // 这一行红 → 越权暴露
}

这条用例的价值不是"测了查询功能",而是"测了隔离边界"。功能测试人人会写,边界测试才是质量保障的核心。后来这种「JOIN 漏 tenant_id」被进一步固化成 SQL 审查清单,跨层兜底——这是 事故即规范那篇的内容。

这个案例还有个值得提炼的点:越权测试不能只测「能查到自己的」,必须测「查不到别人的」。很多测试只断言「租户 A 查询能返回租户 A 的订单」(contains(orderA)),不断言「不返回租户 B 的订单」(doesNotContain(orderB))。前者只能证明「功能没坏」,后者才能证明「隔离没漏」。少了后半句断言,越权测试就是假的——越权的本质是「多返回了不该返回的」,你不断言「没有不该有的」,就永远抓不到越权。这条「正向 + 反向断言」的规矩,后来写进了我们的测试规范:涉及隔离的测试,必须同时断言该有的有、不该有的没有。

这个案例也说明 IT 层的不可替代性。UT mock 了 Mapper,SQL 没真跑,这种越权 UT 永远抓不到;E2E 能抓但太慢太贵,不可能每个 JOIN 都写 E2E。只有 IT——用真实数据库跑真实 SQL,成本低又能验隔离——是抓这类问题的最佳层。这也是为什么三层不能互相替代,每层都有自己的盲区,别的层补不了。

反模式:这些做法不算测试纪律

讲完正确做法,说几个伪装成「我们在做测试」的反模式:

反模式一:覆盖率达标就算过。 跑个 80% 覆盖率就放行,不管测的是不是真行为。expect(true).toBe(true) 也能撑覆盖率,但等于没测。覆盖率是必要不充分——不够肯定不行,够了也不一定够。

反模式二:E2E 全量跑,不分轻重。 把所有用例都塞进 E2E,一次跑 40 分钟,团队嫌慢就降低频率,最后 E2E 形同虚设。E2E 只测关键路径,边界和分支让 UT/IT 去,这是测试金字塔的本意。

反模式三:测试依赖执行顺序。 用例 A 造的数据,用例 B 依赖它。一旦执行顺序变了(并行跑、字母排序),全崩。每个测试必须独立,自建自清。

反模式四:flaky 测试"再跑一次就过了"。 遇到偶发红,不查根因,设置 retries: 3 蒙混。flaky 是债务,retries 是利滚利——迟早有一天 retries 也救不了,炸的就是生产。

这四个反模式的共同点:用「指标达标」替代「真测了」。测试纪律的价值不在指标,在每一个用例都诚实地验证了一个行为或边界。

这四个反模式里,flaky + retries(反模式四)是最需要警惕的,因为它会自我「治愈」到爆雷。团队遇到 flaky,第一反应往往是加 retries——跑三次,有一次过就算过。短期看 flaky 消失了,实际上是把它藏起来了。retries 把 flaky 从「显性债务」变成「隐性债务」:你不再看到它红,但它的问题还在,只是概率性地「碰巧过了」。随着系统演进,flaky 越积越多,retries 从 1 加到 3 再加到 5,直到某天 retries 也救不了——那一次往往就是上线后炸。我们立了条铁律:flaky 必须查根因,不准用 retries 掩盖。查不出根因的 flaky,宁可标红停在那儿逼着人解决,也不让它假装绿。诚实面对 flaky,是测试纪律最难守但最该守的一条。

还有一个反模式没列进去但值得提:测试只验证「不报错」,不验证「结果对」。比如测一个查询接口,只断言「接口返回 200」,不断言「返回的数据结构对、字段值对」。这种测试能跑通,但接口返回空数组它也绿,返回错的数据它也绿——它只验了「没崩」,没验「对不对」。AI 写测试特别容易犯这个毛病,因为「不报错」是最容易满足的断言。我们的规矩是:每个测试至少有一个针对业务结果的断言,不准只断言 HTTP 状态码。

测试纪律的本质:不信任自觉,只信任机器化的规则

测试金字塔不难画,难的是让它真的三层都在、且每一层都诚实。四条纪律叠起来(fail-fast + 禁跳过禁占位 + 最小用例矩阵 + 数据契约),测试才从「墙上的图」变成「网住 bug 的网」。

这套纪律对人有效,对 Agent 更有效——因为它把模糊的判断变成了可执行的规则。Agent 不会"自觉",但配置和扫描脚本能让它"无法不自觉"。maxFailures: 1 不让它继续跑红,pre-commit hook 不让它加 skip,最小用例矩阵不让它少写,数据契约不让它乱造数据。每条纪律都是一道机器化的闸,把 Agent 可能偷懒的路一条条堵死。

这就是工程化测试的本质:不信任人的自觉,也不信任 Agent 的自觉,只信任机器化的纪律。纪律立住了,人和 Agent 才能在同一个标准下产出,质量才不依赖「今天谁在写、心情好不好怎么样」,而是稳定地由规则托底,这才是工程化测试该有的样子。
