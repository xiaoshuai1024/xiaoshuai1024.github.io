【AI 研发实战】UT/IT/E2E 三层测试纪律:fail-fast、禁跳过、零占位

在「前端 → 全栈 → AI 开发」这条线上,这篇落在 AI 协同研发阶段——讲的是我用 Agent 干活时,测试纪律怎么落地。它和契约门禁、多 Agent 评审是一套:把模糊的「测试要写好」变成 Agent 也能照着执行的硬规则。

测试金字塔谁画都顺手:底下 UT、中间 IT、顶上 E2E,贴在 PPT 上特别好看。可真到了项目里,这套三角形十有八九会缩水。

  • UT 漏边界;
  • IT 跑得慢干脆禁用;
  • E2E 全绿但一上线就翻车。

我以前也画过那种金字塔,画完就贴墙上,没几个人看。

我的做法是:金字塔不画在墙上,写进纪律里。而且这条纪律不仅人要守,Agent 也要守。下面是我们怎么把「三层测试」从口号落成可执行规矩的。

三层,各兜一摊,谁也替不了谁

先把分工说清楚,省得有人想「用 UT 多写点、E2E 少写点」来偷懒:

  • UT(*Test)兜逻辑:纯逻辑、边界、异常分支,快、多、便宜。一个工具函数的入参组合,几毫秒一轮。
  • IT(*IT)兜集成:到真实数据库、真实中间件这一层,验「拼起来对不对」。慢一些,但能抓 UT 抓不到的连接、序列化、SQL 语义问题。
  • E2E 兜真实路径:从入口点进,模拟用户真实操作,验「整条链路通不通」。

这三层不能互相替代。UT 全绿不代表集成没问题——你 mock 掉的那个表结构可能压根对不上。E2E 全绿也不代表每个分支都对——你只点了「正常路径」。

所以关键判断是:不是「三层选一层」,是三层都得在,各管一段。

纪律一:E2E fail-fast,第一个失败就停

E2E 最贵的不是写,是跑。一个 E2E 套件几十分钟不稀奇。

如果第一个用例就挂了,后面几十个还在吭哧吭哧跑,跑完一堆红——纯浪费时间,还污染日志。

我们直接定死 fail-fast:maxFailures: 1。第一个挂,立刻停。理由很直接:E2E 是端到端的,前面的失败大概率会传染后面(比如登录挂了,后面所有用例都白跑)。

停在第一个,省下来的时间够你去看一眼到底哪断了,而不是等半小时后看一屏红。

这条对 Agent 尤其重要。Agent 不会自己判断「这个失败要不要继续」,所以必须用配置把它钉死,不让它有发挥空间。

纪律二:禁止跳过,禁止占位

这条是我跟团队(跟 Agent)吵过最多的一条。

什么算「跳过」?@Ignore、skip、it.skip、条件性跳过(if 环境变量就 skip)——统统不许,明令禁止。连 Agent 自己都不许自动加 skip 来「先把测试跑绿」。

更狠的是连占位测试也禁。就是那种 expect(true).toBe(true)、一个空壳断言、为了凑覆盖率凑门禁的「假测试」。看着绿,其实啥也没测,纯债务。

我们见过不少 AI 生成的测试就长这样——为了「让测试通过」而写测试,本末倒置了。

跳过和占位本质是同一种债:假装绿了。与其留一个绿油油的空壳,不如让它红着——至少红是诚实的。

纪律三:按页面类型规定最小用例矩阵

「测试要够」是废话,「够」是几个?我们干脆按页面类型写死最小用例数:

  • 列表页 2 条:正常数据一条 + 空态一条。别只测有数据的样子,空列表那个边界最容易翻。
  • 表单页 2 条:成功提交一条 + 校验失败一条。只测成功路径的表单,基本等于没测。
  • 状态机 N 条:每个合法迁移一条 + 至少 1 个非法迁移。非法迁移那个最值钱,往往能逮住越权。
  • 删除 1 条:确认删 + 删完真的消失(不是软删了还在列表里)。

这套矩阵的好处是:Agent 也能照着填。它不用判断「够不够」,照着矩阵凑就行;评审 Agent 也能照着查——少一条直接打回。

纪律四:测试数据契约

测试数据这事儿,不立规矩迟早互相踩。我们的契约就几条:

  • 租户软删除:别真把数据物理删了,下一个用例就没了。
  • 固定账号复用:别每次新建,环境会越来越脏。
  • 谁建谁清:测试自己造的数据自己收,不污染别人。

这条看起来琐碎,但在多租户项目里是命根子。测试数据乱跑,轻则用例间互相干扰、偶发红,重则真把别的租户数据碰了。

实战:一次多租户越权怎么被网住

讲个真事。某次一个业务模块的查询接口,JOIN 表的时候漏了 tenant_id 过滤——典型的多租户越权,A 用户能查到 B 用户的数据。

这玩意儿人工评审很容易漏(看着 SQL 没毛病),编译更管不着。

最后是怎么逮住的?就是靠上面这套纪律兜住的:IT 层用固定账号 A 跑、断言查不到 B 的数据,一跑就红。fail-fast 一停,定位到那个 JOIN。

后来这种「JOIN 漏 tenant_id」被进一步固化成 SQL 审查清单,那是另一篇的事了(事故即规范那篇会细聊)。

一句话总结

测试金字塔不难画,难的是让它真的三层都在、且每一层都诚实。

四条叠起来,测试才从「墙上的图」变成「网住 bug 的网」:

  • fail-fast 省时间;
  • 禁跳过、禁占位 防「假绿」;
  • 最小用例矩阵 让「够」有标准;
  • 数据契约 防互相踩。

这套纪律对人有效,对 Agent 更有效——因为它把模糊的判断变成了可执行的规则。
