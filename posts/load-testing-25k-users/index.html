2.5 万用户压测实战:压什么、怎么压、瓶颈怎么找

给一个多租户 SaaS 系统做 2.5 万用户压测，最容易跑偏的不是「起几万个线程」，而是压之前没想清楚压什么、压之后不知道哪儿慢——工具五分钟就能跑起来，值钱的是前面那套假设和后面那套对齐。

压测真正的难点不在"跑工具",而在"压什么、怎么压、压完怎么知道哪儿慢"。它检验的是一条从应用层、数据库到监控的完整链路能不能扛住真实流量。

先想清楚压什么,再谈怎么压

接到压测任务的第一反应通常是"全链路压一遍"。这是错的。

2.5 万用户不是个均匀的数字。真实业务里,有人登录、有人浏览、有人下单、有人支付,比例完全不一样。所以第一步是拆场景:

  • 核心链路:登录、浏览、下单、支付必须压,它们出问题用户最先骂。
  • 读多写少 vs 写重:列表页、详情页是读;下单、支付是写。两者瓶颈模型完全不同,分开压。
  • 峰值模拟:大促那种"0 点开抢"的瞬时尖峰,和平时的稳态是两个量级,分开建模。

这三件事想清楚,压测计划才算有了骨架。否则就是拿工具瞎打,跑完一堆数字,谁也说不清代表什么。

场景拆分实操:业务占比法

具体怎么拆?我们用的是业务占比法——拉过去 30 天的真实生产日志,统计各接口的实际调用比例,按比例还原压测脚本:

# 压测脚本里的业务占比(k6 / JMeter 都行)
scenarios:
  browse_list:        # 浏览列表(读多)
    weight: 45        # 占比 45%
    rate: 450
  view_detail:        # 看详情(读多)
    weight: 30
    rate: 300
  place_order:        # 下单(写)
    weight: 15
    rate: 150
  pay:                # 支付(写,关键链路)
    weight: 8
    rate: 80
  search:             # 搜索(读,但走 ES,重)
    weight: 2
    rate: 20

这个比例不是拍脑袋,是从生产日志里 group by api_path 统计出来的。如果压测脚本里把支付比例配成 50%,但生产里支付只占 8%,压出来的瓶颈点就是错的——支付链路过载是假象,真实生产里根本没这么高占比。

反面案例:我们第一次压测时,前端同学为了"覆盖全场景",把所有接口都配成均等比例。结果压出来的瓶颈是订单查询接口的连接池,但这个接口在生产里 QPS 极低,根本不是真痛点。改成业务占比后,真正的瓶颈(支付的分布式锁)才暴露出来。

业务占比法还有个进阶用法:区分时段占比。生产日志的 30 天平均占比,掩盖了不同时段的差异——白天浏览多、晚间下单多、大促峰值支付占比暴涨。如果只拿平均占比压,压出来的是「平均态」的瓶颈;但系统最可能出事的是「峰值态」。所以我们对大促这种特殊时段单独拉日志、单独算占比、单独压一轮。平均态压测保证日常稳定,峰值态压测保证大促不崩,两轮的结果不能混着用——平均态的瓶颈点在峰值态可能完全不出现,峰值态的瓶颈在平均态也看不出来。

读多写少 vs 写重:瓶颈模型完全不同

读多写少的链路(列表、详情),瓶颈通常在缓存和 DB 读。这类链路优化方向明确:加缓存、优化查询、读库扩容。它的特点是「线性退化」——压力越大延迟越高,但不会突然崩,缓存命中率掉到一定程度才明显变慢,有预警空间。

写重的链路(下单、支付),瓶颈往往在锁、事务、队列。这类链路的特点是「非线性塌方」——某个资源(比如行锁)一旦竞争激烈,延迟不是慢慢涨而是突然飙,几乎没有预警空间。而且写操作的瓶颈更难定位,因为锁等待不会直接体现在某个接口的 QPS 上,要靠链路追踪才能看到「卡在锁上」。所以写链路压测要特别关注锁竞争、事务超时、消息队列积压这些指标,光看 QPS 和延迟看不出问题。

梯度加压,别一上来就猛踩

压测最常见的坑:目标 2.5 万并发,直接从 0 拉到 2.5 万,然后看服务挂没挂。

挂了也没用——你不知道是哪儿挂的。我的做法是梯度加压:

  • 先 1 千,稳住看一遍;
  • 再 5 千,再看;
  • 再 1 万、2 万、2.5 万。

每升一档,人都盯着监控,看哪个指标先撑不住。

我们这次的实际加压梯度记录如下,每档至少稳态跑 10 分钟:

档位并发用户数持续时间关键观察结果
11,00010 分钟P99 < 200ms,连接池 < 50%✅ 全绿
25,00010 分钟P99 ≈ 280ms,连接池 65%✅ 健康
310,00015 分钟P99 ≈ 450ms,缓存命中率从 92% 掉到 85%⚠️ 开始有信号
415,00010 分钟P99 飙到 1.2s,支付接口报超时🔴 瓶颈暴露
525,0005 分钟服务雪崩,熔断全开❌ 拐点已过

最有价值的是第 3 档到第 4 档之间的过渡——P99 从 450ms 跳到 1.2s,这一档不是"慢慢变慢",是"突然塌方"。这种非线性塌方就是瓶颈点,也是压测要逼出来的核心信息。如果跳过中间档直接压 2.5 万,你只能看到"挂了",看不到"在哪个量级挂的"。

把这条梯度加压的曲线画出来,「拐点在哪、为什么不能跳档」就清楚了:

这张图最该看的是 10k 到 15k 之间那段陡升。线性变慢(5k→10k,280ms→450ms)是「负载增加的正常代价」,系统还在可控范围;非线性塌方(10k→15k,450ms→1.2s)是「某个资源被打破了临界点」,系统开始失控。压测的目标不是找到「系统能扛多少」,是找到「系统在哪个量级开始失控」——这个失控点就是容量规划的边界。生产流量永远不该超过这个边界,留出 30% 的余量,就是日常运营的安全水位。

接着是稳态压测:不是冲一下就撤,而是在某个档位保持一段时间(我们一般稳态跑十几分钟),看服务能不能稳定扛住,而不是靠一波脉冲蒙混过关。

为什么要稳态?因为有些问题只有"长跑"才暴露:连接池慢慢漏、缓存命中率缓慢下降、GC 时间随堆涨上来。冲一下看不到这些,得让子弹飞一会儿。

稳态压测还有个容易被忽略的价值:验证「恢复能力」而不只是「扛压能力」。稳态跑完后,我们会有意停掉压测,看系统多快恢复正常——连接池释放、GC 回收、缓存命中率回升。如果停了压测后系统迟迟恢复不过来(比如连接池一直高位不释放),说明有资源泄漏,这比压测中变慢更危险——压测中变慢你还能看到,泄漏是压测结束后才暴露的,生产上就是「高峰期过后服务一直慢,重启才好」。所以稳态压测的「停压后恢复」阶段,和「加压中」一样重要,不能压完就撤。

压测必须和可观测性对齐,否则压了个寂寞

这条是踩坑踩出来的规矩。

有一次压测,跑完 QPS 看着还行,P99 也在可接受范围,结果上线后还是出问题。回头复盘才发现:压测时监控没对齐。我们盯着的是"请求成功率",但某个数据库连接池在压测中其实已经打满,只是有重试兜着,成功率没塌。

问题的是压测指标和线上风险没对齐。所以从那以后我们立了规矩:压测期间 Prometheus 必须实时开着,Grafana 上把这几类指标摆在一起盯:

  • API 延迟(P50/P95/P99)、QPS、成功率;
  • 数据库连接池的使用率、等待数;
  • 缓存命中率、限流触发次数;
  • JVM 的 GC、堆、线程数。

压测不是"跑完看报告",是"边压边看哪儿先冒烟"。压和盯必须同一拨人、同一时刻,不然压完根本不知道慢在哪。

「压和盯同一拨人」这条规矩,是踩坑踩出来的。早期我们把压测和监控分开——运维跑压测,开发看监控,两边不在一个屋里。压测跑完,运维说「QPS 到了 2 万」,开发说「监控上 P99 正常」,两边都觉得没问题,上线后翻车。复盘才发现:压测时某个连接池其实打满了,但开发看的是「平均值」不是「峰值」,连接池的瞬时打满被平均掉了。从那以后,压测时压的人和盯的人必须坐在一起(或同一个语音频道),实时对齐「我现在压到哪一档」「监控上什么在冒烟」,信息延迟降到零。

还有个关于监控指标的实战经验:压测时要多看「资源指标」少看「业务指标」。业务指标(成功率、QPS)是结果,资源指标(连接池、CPU、缓存命中率)是原因。只盯业务指标,你看到的是「还行」还是「不行」,但不知道为什么;盯资源指标,你能在「业务指标还没塌」时就看到「连接池已经 90% 了,再压一档就要崩」。资源指标是预警,业务指标是结果,压测要盯预警而不是等结果。

瓶颈往往不在你想的地方

2.5 万用户压下来,我最确定的一条经验:瓶颈几乎从不在最初猜的地方。

我们一开始担心应用层、业务逻辑、某个慢查询。结果真压起来,最先冒烟的是:

  • 数据库连接池:配置太小,高并发下请求排队等连接,延迟直接飙;
  • 缓存:某个热点 key 没缓存住,全打到 DB;
  • 限流:阈值配低了,正常流量被自己限流误伤,看起来像"服务不行",其实是"自己挡自己"。

应用本身的业务代码,反而是最不容易先垮的那一层。

这个现象有个普遍规律:瓶颈总出现在「共享资源」上,不在「计算资源」上。应用代码跑在 CPU 上,CPU 不够就加机器,水平扩展相对容易;但数据库连接池、缓存、分布式锁,这些是多个请求共享的资源,加机器不一定能解决——连接池是每个实例独立的,加实例只是多了几个独立的池子,单池的瓶颈还在;分布式锁是跨实例的,加实例反而加剧锁竞争(更多请求抢同一把锁)。所以压测前,重点排查「哪些资源是被共享的」,这些才是最可能先垮的。计算资源(CPU、内存)反而好扩,不常是第一瓶颈。

还有一个容易被忽略的瓶颈类型:第三方依赖。短信网关、支付通道、地图 API,这些外部调用你控制不了它的性能,但它慢了会拖垮你的服务。我们压测时发现支付接口慢,一开始以为是自己的分布式锁,trace 一看,锁等待只占 200ms,剩下 1 秒卡在第三方支付通道的回调上。第三方通道在压测时也在被别的请求打(或者有限流),响应慢是正常的。这种瓶颈解法不是「优化自己」,是「给第三方调用加超时 + 降级」,超时就返回「处理中请稍后查」,不让第三方拖垮主链路。

瓶颈定位法:三层下钻

发现"慢了"只是第一步,定位到具体哪个组件慢才是压测的真正价值。我们用的是三层下钻:

第一层:看 P99 而不是平均值。 平均值会被正常请求拉低,掩盖少数慢请求。2.5 万并发时平均 200ms 看着没问题,但 P99 可能已经 1.5s——意味着每 100 个请求有 1 个用户在等 1.5 秒。生产体验由 P99 决定,不是平均值。

第二层:链路追踪拆耗时分布。 用 Skywalking / Jaeger 看慢请求的 trace,把总耗时拆成:网关 → 应用 → DB → 缓存 → 第三方调用。一次支付请求总耗时 1.2s,trace 显示应用层只用了 50ms,剩下的全卡在分布式锁等待上——瓶颈立刻定位。

第三层:监控指标交叉验证。 把 P99 飙升的时间点和资源监控对齐。我们那次 P99 在第 4 档突然飙升,Grafana 上同时刻 Redis 命中率从 92% 掉到 68%——锁竞争导致缓存击穿,连环效应一目了然。

这三层从粗到细,缺一不可。只看第一层,知道慢但不知道哪儿慢;只看第二层,知道单次链路但不知道全局影响;只看第三层,知道资源但关联不到业务。三层叠加,才能把瓶颈从"现象"挖到"根因"。

把三层下钻画出来,「每层回答什么问题、缺了会卡在哪」一目了然:

这张图右侧标注了每层的局限——这就是为什么要三层而不是一层。只看 P99,你知道「慢了」但不知道哪儿慢,定位停在第一步;加了链路追踪,你知道「卡在锁等待」但不知道「为什么锁突然变多」,定位停在第二步;加了监控交叉验证,才看到「锁变多是因为缓存击穿」,根因浮出。三层像三块拼图,少了任何一块,根因就拼不完整。

这也说明一件事:压测的价值不在"证明系统很强",而在"提前把脆弱点逼出来"。QPS 10 万+、可用性 99.99% 这些数字是结果,不是目的。

压测数据隔离,别污染生产

最后一个容易翻车的点:压测数据。

压测产生的写流量、脏数据,如果不隔离好,会污染生产库,影响真实业务的统计和审计。我们专门做了压测数据隔离:

  • 压测流量打的是一套独立的影子数据,不进生产表;
  • 也不进生产监控的业务大盘——否则压测的 QPS 会把真实大盘冲乱,告警乱飞。

这事看着是细节,但漏了就是事故。

数据隔离的实操有个讲究:压测流量要打标记,全链路透传。我们在请求头加一个 X-Pressure-Test: true 标记,这个标记从网关一路传到应用、到数据库。数据库层根据这个标记,把写操作路由到影子库(独立的测试数据库),不进生产库;监控层根据这个标记,把压测流量分流到独立的监控大盘,不冲乱生产告警。标记不透传到底,中间某一层漏了,压测数据就漏进生产——比如应用层识别了标记路由到影子库,但某个异步消息没带标记,消费者不知道这是压测数据,写进了生产库。全链路透传,意味着每一层都得识别这个标记,漏一层就漏数据。

还有个关于数据准备的实战经验:压测数据要「像真的」。压测前造一批测试数据(用户、商品、订单),这些数据要和生产数据的分布相似——不能全是 1 条订单的用户,要有下单 100 次的「重度用户」;不能全是小金额订单,要有大额订单。数据分布不像真的,压出来的瓶颈就不准——比如分页查询,10 条数据和 1000 条数据的性能完全不同,压测数据太简单,分页的瓶颈根本压不出来。我们造数据时,会参考生产数据的分布特征(订单数的长尾分布、金额的正态分布),尽量还原真实的数据形态。

结语

做完这次 2.5 万用户压测,我最核心的体会是:压测的本质不是"跑工具",而是用数据验证你的架构假设。

  • 你假设数据库连接池够用?压一下看看。
  • 你假设缓存能扛住热点?压一下看看。
  • 你假设限流配得合理?压一下看看。

压测就是把脑子里那些"我觉得没问题"挨个拽出来,用真实流量逼它们现原形。

所以下次有人跟你说"我们做个压测吧",别急着开工具。先问三个问题:压什么?怎么压?压完看什么? 这三个问题答清楚了,压测才算有灵魂。

这三个问题的顺序也有讲究:先压什么(场景拆分),再怎么压(梯度 + 稳态),最后看什么(三层下钻 + 监控对齐)。跳过第一个直接谈工具,压出来的数字是假的;跳过第三个直接谈结果,瓶颈定位停在表面。三个问题串成一条线,缺了任何一环,压测就退化成「跑工具看热闹」,白费功夫还可能严重误导决策。

压测前的检查清单

压测能不能压出真问题,关键不在工具多先进,而在这几件事有没有做到位——

业务场景拆分要看生产日志的真实接口占比,不是拍脑袋;读重、写重、峰值三类场景分别建模,而不是一锅炖;加压梯度进行(1k → 5k → 10k → 目标值),每档稳态至少 10 分钟,才能暴露"长跑才出现"的问题(连接泄漏、GC 抖动、缓存击穿)。监控(Prometheus + Grafana + 链路追踪)必须就位,关键指标(API 延迟、连接池、缓存命中率、GC)全在大屏上;压测数据隔离(影子库 / 独立 namespace),流量打独立监控大盘,不冲乱生产告警。回滚预案要就位,发现雪崩立刻降并发;压测后必须有"瓶颈-优化-复测"闭环,而不是压完报告交差。

这些不是清单条目,是压测能否得出可信结论的前提。少做任何一件,压出来的数字都是假的——而基于假数字做的容量决策,就是上线后事故的伏笔。

十条全部勾上,压测才算"带假设的工程实验"。少一条,就还在"跑工具看热闹"。

压测工具选型决策表

不同压测场景该用哪个工具?这是我们的实测对比:

工具适合场景优点局限
JMeter传统场景、GUI 编排、协议丰富生态成熟、插件多、支持各类协议GUI 重、分布式压测配置麻烦
k6云原生、代码化编排、CI 集成脚本即代码(JS)、轻量、易入 CI协议支持不如 JMeter 全
wrk2极致 QPS、单接口基准测试性能极高、延迟分布准确只能压单接口,无场景编排
LocustPython 团队、复杂业务流Python 脚本灵活、UI 实时单机性能弱,需多机分布
云服务(PTS / LoadRunner Cloud)不想维护压测基础设施全托管、报告漂亮贵、压测流量出口固定易被风控

中小团队 + 云原生架构,首选 k6(脚本即代码 + 易入 CI);传统企业 + 复杂协议,JMeter;要做极致基准测试,wrk2。别用 LoadRunner 这类商业重型工具除非老板特别要求——它的价值跟价格不匹配。

工具选型还有一个维度值得考虑:团队熟不熟。k6 脚本用 JS 写,前端团队能直接上手;JMeter 用 XML 配置,后端同学更熟;Locust 用 Python,数据团队可能更顺。压测脚本是要持续维护的(接口变了脚本得跟着改),选一个团队能维护得住的工具,比选「最强」的工具重要。一个功能强但没人会改的压测脚本,很快就会因为跟不上接口变化而废弃,等于白选。

还有一个选型的误区:别追求一个工具压所有场景。我们最后是 k6 做场景压测(业务流编排)、wrk2 做单接口基准(测某个接口的极限 QPS)、JMeter 做协议压测(比如 WebSocket、MQ),三个工具各管一摊。一个工具硬压所有场景,要么场景编排能力不够(k6 压不了 WebSocket),要么太重(JMeter 写业务流不如 k6 灵活)。组合使用,每个工具用它最擅长的部分,反而比「一个工具打天下」更高效。
