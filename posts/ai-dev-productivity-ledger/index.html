【AI 研发实战】4 个人 + AI,2 个月交付近百万行:一份真实的研发账本

xiaoshuai1024 included in AI编程

前面几篇一直在讲认知怎么转、转型往哪转。这篇不谈观点,只摆账本。

「老兵 + AI 杠杆」这个判断光讲道理没人信。我用我们刚交付的一个真实项目,把数据拆开:4 个人的团队,2 个月,做出了什么。

一、先看账本:4 个人干出了什么

这是我们这个多租户社交电商 SaaS 项目的实测数据,2 个月周期,不含重构掉的、不含被打回来的:

维度数据
周期约 2 个月
代码提交560+ 次,日均 14 次
后端2100+ Java 文件 / 30 万行,330+ Controller、540+ Service
数据库850 张表,380+ 个迁移脚本
前端 C 端530+ Vue 文件 / 19 万行,400 个页面、20+ 业务域
运营后台260+ Vue 文件 / 8.6 万行
测试后端 400+ 测试文件、前端 650+ 测试文件,合计近 19 万行测试代码
业务总量近百万行(业务 + 迁移 + 测试)

这个体量,过去要十几人的团队、半年起步。我们 4 个人,2 个月。

原因不是我们是超人,是因为我们身后站着一群 AI。

二、产能从哪来:4 个人在驱动 AI

先排除一个误会:这 4 个人没有通宵、没有 007。产能爆炸的真正原因,是每个环节都被 AI 撑开了。

我的日常主力是 Claude Code,配 Cursor 做编辑,OpenCode 做补充,再跑一批自治 Agent 做并行任务。4 个人指挥,一群 Agent 干活,这才是真实的画面。

具体省在哪,分四块拆。

第一,样板代码。 一个 CRUD 接口、一个列表页、一套表单校验,过去手敲半天。现在 Controller + Service + Mapper + 前端页面一套,AI 一次铺完,我只做审查和调参。330 个 Controller、540 个 Service,大头是这么来的。

第二,多端适配。 同一个业务,C 端小程序、运营后台、后端 API 三端都要。换以前这是三个人的活,现在 AI 把三端的样板同时生成,我用经验统一契约、卡边界。「全栈」这件事,AI 让它从「我能干」变成了「我干得快」。

第三,测试用例。 400 个后端测试、650 个前端测试、19 万行测试代码——纯人手根本铺不起来。AI 把测试生成从「成本」变成了「顺手」。这块怎么兜质量,我专门讲过。

第四,数据库迁移。 850 张表、380+ 个迁移脚本,DDL、索引、种子数据,AI 起草、我审。多租户的隔离字段漏一个就是事故,所以这块我立了硬门禁,不让人和 AI 越界。

三、但 AI 不是白嫖的

光说省力是骗人。AI 省了写的时间,但把成本挪到了「兜底」上。这部分恰恰是我整个 AI 系列真正值钱的东西。

第一块是返工成本。AI 写得快,但写得快不等于写得对。幻觉、编造的 API、看似能跑实则埋雷的代码,全得人拦下来。我们因此搭了多 Agent 评审回路,提交前自治评审过一遍。

第二块是上下文管理。项目越大,AI 越容易跑偏。我把 70+ 个自建技能、20+ 个命令、契约闸全沉淀进仓库,让每个 Agent 开机即用。详见把规范写进代码底层。

第三块是事故成本。AI 引入的 bug,典型的是「编译能过、逻辑是错的」。我们栽过一次「幽灵删除」,从那以后定了规则:每起事故必须固化成机器可执行的规则(事故即规范)。

一句话:AI 帮我们省了执行,但逼着我们把「质量体系」从可选变成了刚需。这套体系搭起来,才是产能真正能持续的根。

四、真正决定产能的,还是那 4 个人

讲到这里,可能会问:那是不是随便找 4 个人 + AI,就能干出百万行?

不能。这才是我想强调的核心。

同样是 AI,新人写出来的东西常常是「能跑但经不起推敲」——架构乱、边界糊、出了 bug 不知道往哪查。而我们这 4 个人,全是带过真实高并发、踩过生产事故的老兵。AI 写得越多,越需要有人一眼看出哪段迟早出事、哪个方案上线必挂。

这 2 个月能干出这个体量,不是 AI 多强,是我们这 4 个人手里的支点稳。架构怎么分层、质量怎么兜、边界划在哪,全是经验在拍板。

这也印证了我在认知篇的判断:老兵不但没被 AI 淘汰,反而成了最大受益者。

五、这不是「AI 替代人」

最后把话收一下,免得被读成「AI 万能,人可以躺了」。

这个项目的真相是:4 个老兵 + 一套 AI 协同工程体系,跑出了过去一支十几个人的队伍的产能。人没被替代,人的价值被放大了——从「写代码的执行者」,变成了「指挥 Agent 的决策者」。

这也是为什么我反复讲「认知转变」比「学工具」重要。工具谁都能装,但「知道让 AI 干什么、不让它干什么、出了事怎么兜」这套判断力,才是 2 个月干出百万行的真正底座。

后面我会把这个项目的关键模块一个个拆开讲——微信分润的冲正引擎怎么设计、多租户 850 张表的迁移怎么治理、15 个数据大屏背后的统计中台怎么搭。这篇先把总账立住:老兵 + AI,真能干出过去一个团队的活。
