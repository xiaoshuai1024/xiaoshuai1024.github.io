4 个人 2 个月近百万行:技术总监 + AI 带小团队的真实研发账本

前面几篇一直在讲认知怎么转、转型往哪转。这篇不谈观点,只摆账本。

为什么要专门摆账本。因为「技术总监 + AI 能放大小团队产能」这个判断,光讲道理没人信。有人会说「AI 写的代码能看吗」「是不是堆了一堆屎山」「是不是牺牲了质量换速度」。与其用观点反驳观点,不如把一个真实项目的数据摊开——交付了什么、花了多久、质量怎么兜的,让数据自己说话。

「技术总监 + AI 杠杆」这个判断光讲道理没人信。我用我们刚交付的一个真实项目,把数据拆开:4 个人的团队,2 个月,做出了什么。

先说清楚这个项目是什么。一个多租户社交电商 SaaS 平台,核心是 CPS 分销(达人带货分佣)、多租户隔离、微信支付分润、数据大屏运营。技术栈后端 Spring Boot + MyBatis、前端 Vue 3 + 小程序、数据库 MySQL、搜索 ES、缓存 Redis。典型的中型 SaaS 架构,麻雀虽小五脏俱全。这种项目,过去行业里通常是一个十人左右的团队、半年周期。

一、先看账本:4 个人干出了什么

这是我们这个多租户社交电商 SaaS 项目的实测数据,2 个月周期,不含重构掉的、不含被打回来的:

维度数据
周期约 2 个月
代码提交560+ 次,日均 14 次
后端2100+ Java 文件 / 30 万行,330+ Controller、540+ Service
数据库500+ 功能模块(633 张业务表),380+ 个迁移脚本
前端 C 端530+ Vue 文件 / 19 万行,400 个页面、20+ 业务域
运营后台260+ Vue 文件 / 8.6 万行
测试后端 400+ 测试文件、前端 650+ 测试文件,合计近 19 万行测试代码
业务总量近百万行(业务 + 迁移 + 测试)

这个体量,过去要十几人的团队、半年起步。我们 4 个人,2 个月。

原因不是我们是超人,是因为我们身后站着一群 AI。

解读一下这张账本的重点。后端 330+ Controller、540+ Service,看着吓人,但大头是 CRUD 样板——一个 Controller 平均也就几十行业务逻辑,真正复杂的逻辑(资金分润、状态机、多租户隔离)集中在少数几个核心 Service 里。测试代码 19 万行,接近业务代码的一半,这个比例在传统团队里几乎不可能——人手写测试写到这个量,成本会压垮项目。AI 让测试从「负担」变成「顺手」,这是质量能兜住的根。500+ 功能模块是多租户 SaaS 的正常体量(每个业务域十几张表,几十个域),但 2 个月建完,靠的是迁移治理和 AI 起草 DDL。

换句话说,这张账本里最「反常识」的不是代码量大,是「代码量大 + 测试覆盖也大」——产能和质量同时保住了,这才是技术总监 + AI 杠杆的证明。如果只有代码量大、测试稀疏,那不是产能高,是欠债快。

二、产能从哪来:4 个人在驱动 AI

先排除一个误会:这 4 个人没有通宵、没有 007。产能爆炸的真正原因,是每个环节都被 AI 撑开了。

先说团队构成。4 个人里我(京东 T7 出来的,带过大团队)算唯一一个有大厂和大型系统经验的,剩下 3 个下属:1 个原前端、1 个原后端、1 个设计师兼运营。两个研发都没有大厂背景,也没做过大型系统。不是什么梦之队,就是正常的初创小组合。变量在工作方式:每个人同时驱动多个 AI Agent,一个人干过去三个人的活,但干的是「指挥 + 审查」,不是「手敲」。

我的日常主力是 Claude Code,配 Cursor 做编辑,OpenCode 做补充,再跑一批自治 Agent 做并行任务。4 个人指挥,一群 Agent 干活,这才是真实的画面。

工具为什么这么排。Claude Code 当主力,是因为它的长上下文和 Agent 编排能力最强,适合做「读一大片代码、做一处改动、再跑测试」这种多步任务;Cursor 当编辑器,是因为它的实时补全和局部改写快,适合「边想边改」的探索阶段;OpenCode 做补充,处理一些特定的代码生成任务。自治 Agent(无监督的 subagent)专门跑评审、测试、迁移校验这些「可以并行、不需要人盯」的活。每个工具用在它最擅长的环节,不是逮着一个用到底。

再说一天的节奏。不是「从早到晚敲代码」,而是「分批派任务 + 集中审查」。上午集中派一批生成任务(AI 跑的时候我去审昨天未完的代码),下午集中审查上午的产出 + 派下一批。一天里,真正「手敲」的时间不到 20%,剩下 80% 是审查、调参、派新任务。这个节奏和传统开发完全不同——传统开发是人一直敲,现在是人一直判断,AI 一直敲。

代码审查的频率也变了。传统开发是「写完一个模块,review 一次」,节奏以天为单位。AI 协同后是「AI 每生成一批,就审一批」,节奏以小时甚至分钟为单位。审查变得更碎片化,但总审查量反而更大——因为 AI 产出快,要审的东西多了。这就是为什么前面说「返工成本」不可忽视:审得多,打回得也多,这是正常循环,不是效率低。

具体省在哪,分四块拆。

第一,样板代码。 一个 CRUD 接口、一个列表页、一套表单校验,过去手敲半天。现在 Controller + Service + Mapper + 前端页面一套,AI 一次铺完,我只做审查和调参。330 个 Controller、540 个 Service,大头是这么来的。

具体的工作流是这样的——一条命令铺完一整套 CRUD:

# 我们封装的命令:给一张表名,自动生成一整套后端 + 前端骨架
/gen-crud --table=order --tenant-aware --with-audit

这条命令背后是 Claude Code 跑了一串 subagent:

  1. 读 Flyway DDL,提取字段;
  2. 生成 Entity + Mapper + Service + Controller(带多租户字段);
  3. 生成前端列表页 + 表单页 + 详情页;
  4. 生成对应的 UT / IT 测试骨架;
  5. 把生成的代码挂到模块注册表里。

人只做两件事:审字段类型对不对、审业务逻辑特殊处。剩下的样板,AI 全包。一个完整 CRUD 模块,从半小时缩到 5 分钟,而且命名风格、目录结构、测试骨架全统一——十个模块写出来,看着像一个人写的。

这里有个反直觉的点:AI 生成的代码反而比人手写的更统一。人写十个 CRUD,十个人有十种命名习惯、十种目录摆法,后期维护成本高。AI 基于同一套 skill 生成,十个模块的代码结构高度一致,任何人接手都能立刻看懂。这是 AI 协同的一个隐性收益——不是写得快,还写得齐。代码一致性这种「非功能性质量」,靠人盯是盯不出来的,靠 AI 的模板化反而天然达成。

当然,前提是 skill 写得好。如果 skill 里的模板本身有问题(比如漏了多租户字段),AI 会把这个错误放大十次——十个模块全漏 tenant_id。所以 skill 的质量和 review 的把关,是这套流水线能不能信的命门。

第二,多端适配。 同一个业务,C 端小程序、运营后台、后端 API 三端都要。换以前这是三个人的活,现在 AI 把三端的样板同时生成,我用经验统一契约、卡边界。「全栈」这件事,AI 让它从「我能干」变成了「我干得快」。

多端适配的关键是契约先行:后端先定 OpenAPI schema,AI 根据同一份 schema 生成三端代码。

# openapi.yaml(契约源,三端都从这里生成)
paths:
  /api/orders:
    post:
      summary: 创建订单
      requestBody:
        content:
          application/json:
            schema: { $ref: '#/components/schemas/Order' }
      responses:
        '200':
          description: 创建成功
# 一份契约,生成三端代码
/gen-backend-from-openapi --schema=openapi.yaml   # Java Controller + DTO
/gen-frontend-from-openapi --schema=openapi.yaml  # Vue 调用层
/gen-admin-from-openapi --schema=openapi.yaml     # Element Plus 表单

契约变了,三端同步重新生成——不会有"后端改了前端忘改"的接缝 bug。这是契约闸要兜的事。

第三,测试用例。 400 个后端测试、650 个前端测试、19 万行测试代码——纯人手根本铺不起来。AI 把测试生成从「成本」变成了「顺手」。这块怎么兜质量,我专门讲过。

测试为什么能铺这么多。因为 AI 生成测试的逻辑很固定:读一个方法,生成 happy path(正常输入)、boundary case(边界值)、error case(异常输入)三组测试。这套模板 AI 一次跑完一个 Service 的所有 public 方法,几百个测试用例就出来了。人手写测试最痛苦的是「明明知道该测什么,但写起来繁琐」,AI 把这个繁琐吃掉了。

但测试铺得多 ≠ 测试有效。AI 生成的测试有个典型毛病:它倾向于测「方法能正常返回」(happy path 占比过高),而不是「方法在边界情况下行为正确」。所以我审测试时,重点看有没有覆盖「金额为 0」「并发竞争」「租户隔离」这些关键边界,光 happy path 绿的测试是假绿。19 万行测试代码里,有相当一部分是我补的边界用例,AI 只铺了底。

第四,数据库迁移。 500+ 功能模块(633 张业务表)、380+ 个迁移脚本,DDL、索引、种子数据,AI 起草、我审。多租户的隔离字段漏一个就是事故,所以这块我立了硬门禁,不让人和 AI 越界。

迁移这块 AI 特别好使,因为 DDL 是高度规则化的——建表、加列、加索引,语法固定、约束明确,AI 几乎不会写错。真正要人审的是「这张表该不该有 tenant_id」「这个索引是不是带 tenant_id 前缀」「这个字段类型跟其他表统一了没」。这些都是业务和架构判断,AI 看不出来。所以迁移的工作流是:AI 起草 DDL → 构建期脚本校验版本号唯一性 → 人审 tenant_id 和索引 → 合并。详细治理见500+ 功能模块的迁移治理。

产能账:算给你看

把上面四块折算成时间,直观感受「技术总监 + AI」的杠杆有多大:

工作类型纯人手估时AI 辅助实测杠杆倍数
一个 CRUD 模块(后端+前端+测试)4 小时20 分钟12x
多端适配(同业务三端)3 人 × 1 天1 人 × 1 小时24x
100 行测试用例(happy + 边界)1 小时10 分钟6x
一个表的 Flyway DDL + seed30 分钟5 分钟6x
架构决策 / 边界判断2 小时2 小时(不变)1x

最后一行是关键:架构决策、风险判断、边界划定的耗时,AI 完全没有压缩——这部分还是 2 小时,跟没 AI 时一样。AI 放大的是执行,不是决策;技术总监的价值在决策那一栏,杠杆为 1(不被替代),但执行被放大了 6-12 倍。

这就是「技术总监 + AI」的本质:有经验的人的判断力没变,但整个团队的执行被 AI 放大了一个数量级。普通研发 + AI 容易出事——执行得越快,埋雷越快;但有技术总监兜底 + AI 加速,就是快而不乱。

这张表还有一个隐藏信息值得点出来:杠杆倍数越高的是越「标准化」的活(CRUD 12x、多端适配 24x),杠杆倍数低的是越「个性化」的活(架构决策 1x)。这意味着 AI 协同最先吃掉的是标准化工作,越需要经验判断的越晚被影响。理解这一点,就知道该把自己的精力往哪投——少做机器能做的,多做机器做不了的。

三、但 AI 不是白嫖的

光说省力是骗人。AI 省了写的时间,但把成本挪到了「兜底」上。这部分是我整个 AI 系列真正值钱的东西。

第一块是返工成本。AI 写得快,但写得快不等于写得对。幻觉、编造的 API、看似能跑实则埋雷的代码,全得人拦下来。我们因此搭了多 Agent 评审回路,提交前自治评审过一遍。

返工成本具体多大。保守估算,我们跑掉的 560 次提交里,大概有 15%-20% 经历过「生成 → 评审打回 → 重新生成」至少一轮。也就是说,有近 100 次提交是 AI 第一版写错了、被评审或测试拦下、重来才过的。这部分如果没算进产能账,那账就是假的——AI 生成的速度很快,但「生成 → 否定 → 重来」的循环也要吃时间。真实的净产能,是毛产能减掉返工。

第二块是上下文管理。项目越大,AI 越容易跑偏。我把 70+ 个自建技能、20+ 个命令、契约闸全沉淀进仓库,让每个 Agent 开机即用。详见把规范写进代码底层。

上下文管理的成本是「隐性」的。它不体现在某一次提交上,而是体现在「如果不做,后面的返工会指数级增长」。早期我们没有把规范沉淀进仓库时,AI 经常写出不符合项目风格的代码——命名不一致、目录结构乱、测试用例不带租户上下文。每一处都要人工纠正,纠正一次还不算,下次它又犯,因为它不记得上次被纠正过。把规范固化成 skill 和命令后,AI 开机就用项目规范,这类低级返工才降下来。这笔投入是一次性的,收益是持续的。

第三块是事故成本。AI 引入的 bug,典型的是「编译能过、逻辑是错的」。我们栽过一次「幽灵删除」,从那以后定了规则:每起事故必须固化成机器可执行的规则(事故即规范)。

事故成本是最难量化的,因为它往往是「一次大」。幽灵删除那次,270 行代码静默蒸发,编译还绿着,如果不是测试门禁兜住,可能就上了生产。事后复盘花的时间、加规则的时间、补测试的时间,加起来远超 AI 省下的时间。所以事故成本的教训是:AI 协同必须有事故防线,而且防线要前置——不能等出事再补,要在日常流程里就把规则卡死。

一句话:AI 帮我们省了执行,但逼着我们把「质量体系」从可选变成了刚需。这套体系搭起来,才是产能真正能持续的根。

算一笔总账,这套「兜底成本」大概占多少。粗略估算,我们 2 个月里有大约 30% 的时间花在「不是写功能,而是搭质量体系」上——评审回路、契约闸、测试纪律、迁移治理、事故规则。这 30% 看起来是「额外开销」,但它是另外 70% 能高效产出的前提。没有这 30% 的兜底,AI 生成的代码会以「能跑但不可维护」的状态堆积,2 个月后项目就会因为技术债失控而停滞。所以兜底成本不是开销,是投资——投的是产能的可持续性。

这也是为什么我说,光看「4 个人 2 个月百万行」这个数字会误读。完整的故事是「4 个人用 30% 的时间搭质量体系,用 70% 的时间借助 AI 高效产出」。砍掉那 30%,剩下的 70% 就是空中楼阁。

四、真正决定产能的,还是那 4 个人

讲到这里,可能会问:那是不是随便找 4 个人 + AI,就能干出百万行?

不能。这才是我想强调的核心。

同样是 AI,新人写出来的东西常常是「能跑但经不起推敲」——架构乱、边界糊、出了 bug 不知道往哪查。而我们团队的配置是:一个有经验的技術总监(我,京东 T7 出来的)带三个普通研发 + AI。我负责架构决策、边界划定、风险判断,三个研发在 AI 辅助下高效执行。AI 写得越多,越需要有人一眼看出哪段迟早出事、哪个方案上线必挂。

举几个技术总监在 AI 协同里「值钱」的具体场景。

资金计算用什么类型。AI 默认会用 double 或 BigDecimal,因为它从训练数据里学到「金额用 BigDecimal」。但我知道,我们项目的铁律是「金额用分(long)+ 基点(int)」,绝不用浮点。AI 生成的资金代码第一版经常是 double amount,我一眼打回,改成 long amountFen。这个判断 AI 做不了,因为它不知道这个项目的资金规范。

并发场景该加什么锁。AI 看到「多人可能同时操作」,本能反应是加 synchronized。但我知道,synchronized 在分布式部署下没用,得用数据库的条件更新或乐观锁。AI 生成的并发代码经常是「单机思维」,我要改成「分布式思维」。这种改法不是改几行代码,是改整个并发模型。

状态机的边界怎么处理。AI 写状态机,主流程(正常路径)写得很好,但边界(异常路径、并发冲突、重试)经常漏。比如提现状态机,AI 会写 PENDING→APPROVED→SUCCESS 的正常流,但漏掉「两个人同时审核」「出款失败重试」「重试耗尽转人工」这些边界。我会补全这些分支,因为我踩过没补的坑。

这些场景的共同点是:AI 解决「怎么写」,技术总监解决「写什么、漏了什么、哪里会炸」。AI 越快,技术总监这道「防爆」关越值钱——因为快意味着出问题也快,没人兜底就是快速埋雷。

这 2 个月能干出这个体量,不是 AI 多强,是团队配置对:一个有经验的技术总监 + 三个普通研发 + AI 协同体系。架构怎么分层、质量怎么兜、边界划在哪,全是经验在拍板。

再深一层讲,技术总监在 AI 协同里有三个不可替代的作用。

第一,定边界。AI 不知道哪些地方不能碰——资金系统不能用浮点、状态变更必须落审计、并发要靠数据库约束。这些边界是有经验的人踩过坑立下来的,AI 不知道边界在哪,会自作主张越界。技术总监的工作就是把这些边界画清楚,变成 skill 和规则,让 AI 在边界内自由发挥。

第二,兜异常。AI 生成的代码,正常路径基本没问题,异常路径经常翻车。技术总监的价值在于「预判哪里会出异常」——并发竞争、部分失败、超时重试、脏数据。这些场景 AI 想不到(因为它没经历过生产事故),有经验的人想得到(因为经历过)。所以审代码,重点审异常分支,不审正常流程。

第三,做权衡。技术方案很少有绝对的对错,多是权衡。用乐观锁还是悲观锁、同步还是异步、强一致还是最终一致——这些选择取决于业务场景、性能要求、团队习惯。AI 给不出「这个场景该怎么选」的判断,因为它不懂业务上下文。技术总监能,因为在这个业务里泡过。

这三个作用——定边界、兜异常、做权衡——是 AI 替代不了的,也是「技术总监 + AI」这套打法里有经验的人不可替代的价值。AI 把执行放大了,但这三道关还是技术总监在守。

这也印证了我在认知篇的判断:有经验的技术总监不但没被 AI 淘汰,反而成了最大受益者。

五、这不是「AI 替代人」

最后把话收一下,免得被读成「AI 万能,人可以躺了」。

这个项目的真相是:1 个技术总监 + 3 个普通研发 + 一套 AI 协同工程体系,跑出了过去一支十几人团队的产能。人没被替代,人的价值被放大了——技术总监从「写代码的执行者」,变成了「指挥 Agent 的决策者」;普通研发从「需要多年经验才能胜任」,变成了「在 AI 辅助下能交付高质量代码的执行者」。

这也是为什么我反复讲「认知转变」比「学工具」重要。工具谁都能装,但「知道让 AI 干什么、不让它干什么、出了事怎么兜」这套判断力,才是 2 个月干出百万行的真正底座。

有个常被问的问题:这套打法能不能复制。能,但有前提。前提不是「装上 Claude Code」,而是「团队里有能拍板的技术总监」。如果团队里没有懂架构、懂资金、懂并发的人,这套 AI 协同体系跑不起来——AI 会飞速产出代码,但没人能判断这些代码对不对,最后交付的是一堆能跑但经不起推敲的代码。工具可以复制,经验不能,这是这套打法的复制门槛。

还有一个前提容易被忽略:工程体系的沉淀成本。70+ 个 skill、20+ 个命令、契约闸、评审回路、事故规则——这些东西不是项目第一天就有的,是我们在第一个项目里一边做一边沉淀的。如果你现在从零开始,前期的体系搭建会吃掉一部分「AI 省下的时间」,这是不可避免的投入。但一旦体系搭起来,第二个、第三个项目就能享受复利。所以这套打法的回报曲线是「前慢后快」,得熬过前期的沉淀期。

后面我会把这个项目的关键模块一个个拆开讲——微信分润的冲正引擎怎么设计、多租户 500+ 功能模块的迁移怎么治理、15 个数据大屏背后的统计中台怎么搭。这篇先把总账立住:技术总监 + AI,真能干出过去一个团队的活。
