做企业级组件库,难点从来不是写组件,是组织组件。一个按钮谁都会写,但当组件库膨胀到几十个组件、跨多个对外包、要兼顾内部联调和外部 npm 发布时,怎么组织代码、怎么编排构建测试发布流水线才是真正工程难题。

本文以 luban-ui 为例,讲透两件事:Monorepo 怎么分层、工程化流水线怎么编排。技术栈清单不是重点,背后的取舍才是。技术栈是 Vue 3 + TypeScript + Vite + Nx + pnpm,但这套判断换成 React + Webpack + Turborepo 同样成立。

为什么不直接用 Element Plus / Ant Design

这是被问得最多的一个问题。直接回答:因为我们的组件库不只是基础组件,还有低代码设计器和运行时渲染。

Element Plus 这类成熟库解决的是「基础 UI 组件」——按钮、表单、表格。但 luban-ui 的定位是:

  • 基础组件(luban-base):按钮、表单、布局等。
  • 低代码组件(luban-low-code):设计器、运行时渲染、schema、组件注册表。

这两层耦合度极高——设计器要识别基础组件的元信息、运行时要按 schema 渲染基础组件。如果用第三方基础组件库,设计器和运行时就只能"猜"组件的 props,没法精确控制。自己造基础组件,是为了让基础组件天然带「可被设计器识别」的元信息。

这是架构决策的根本出发点:选型不是看哪个库功能多,是看你的场景需要哪些约束。如果你只做基础组件库,直接 fork Element Plus 改皮就行,别造轮子;如果你要做低代码平台,基础组件必须自己掌握。

这个选型判断还有个延伸:造基础组件的成本,远不止「写几十个组件」这一次性投入,还有长期的维护成本——浏览器升级了要适配、无障碍标准变了要跟、新交互范式出来了要加。第三方库有社区帮你摊这些成本,自研就全压在自己团队头上。所以自研基础组件这个决策,隐含的前提是「团队有持续投入的预算和意愿」。如果只是短期项目、做完就丢,fork 第三方库反而更划算。我们做这个决策时,算过一笔账:自研前两年维护成本高于 fork,但到第三年,因为基础组件天然带设计器元信息、低代码层不用做适配,总成本反而反超 fork。这个决策是面向长期的,短期看是亏的。

另一个常被忽略的点:自研基础组件意味着你要自己定义组件 API 的稳定性契约。Element Plus 的 API 有社区监督,不敢随便 breaking change;自研组件的 API 稳不稳定,全看团队自觉。如果基础组件的 props 今天加明天改,上层 low-code 和消费方就要不断跟着改,组件库就从「资产」变成了「负担」。所以我们给基础组件立了规矩:已发布的 props 视为公开 API,改动必须走 deprecate→新版→下线的三步,不能直接删。这条规矩看着束缚手脚,实际是保护组件库长期可用的护栏。

Monorepo 分层:三层依赖,铁律不可破

整个仓库分三层,依赖方向严格自上而下:

依赖铁律只有一条:依赖只能自上而下,绝不反向。具体来说:

  • apps/* 可以依赖 packages/*,反过来不行。
  • luban-low-code 可以依赖 luban-base,反过来不行。
  • luban-base 不依赖任何 luban- 包*——它是依赖树的最底层,只依赖 vue。

这条铁律不是规矩,是物理约束。一旦 luban-base 反向依赖了 luban-utils 的某个函数,而 luban-utils 又依赖了 luban-base 的某个类型,就形成了循环依赖,构建直接挂。

我们踩过一次坑:有人在 luban-base 里 import 了 luban-utils 的 formatDate,觉得「就是个工具函数嘛」。结果 luban-utils 后续为了支持国际化,反向引用了 luban-base 的 LocaleProvider 组件——循环依赖瞬间形成,构建直接红。修复方式是把 formatDate 这类纯函数下沉到一个新的 luban-utils-core 包,只放纯函数、零依赖,谁都能用。从此 luban-base 只依赖 luban-utils-core,不再碰 luban-utils。

这条经验是:依赖方向一旦定了,就要用工具(ArchUnit、eslint-plugin-import 的 no-cycle)守住,不能靠人记。人记迟早会破。

依赖方向还有一个值得展开的点:分层的粒度要克制。三层够用就别分五层,每多一层就多一层「这个该放哪」的纠结,以及一层「改了底层要等中间层构建」的等待。我们见过一些团队把组件库拆成 base / utils / core / shared / low-code / app 六七层,以为越细越好维护,结果是「加一个功能要想它属于哪层」的决策成本,比功能本身还高。分层是为了理清依赖,不是为了分而分。三层(base / low-code / app)能覆盖绝大多数组件库场景,在这个基础上,只有当某一层明显臃肿到「一个包装不下两种关注点」时,才考虑再拆。

守住依赖方向,工具的选择也有讲究。eslint-plugin-import 的 no-cycle 能抓 ES module 层面的循环依赖,但它只看静态 import;如果循环是通过运行时的动态引用(比如反射式的全局注册表)形成的,静态分析抓不到。所以我们在 eslint 之外,还配了一份手写的依赖矩阵文档,把「谁能依赖谁」列清楚,code review 时对照这个矩阵看 import 语句。工具抓明面的环,文档和 review 抓暗面的环,两条腿走路才稳。

技术栈选型:每个选择背后的理由

下面这张表是技术栈全貌,但更重要的是为什么选它,而不是别的。

核心运行时

技术用途为什么是它,不是别的
Vue 3组件开发项目起步时 Vue 3 已稳定,团队熟悉
TypeScript类型安全必选,组件库对外发布,类型声明是命根子
Vite开发 + 库构建esbuild 预构建快;库模式产物干净;Vue 官方推荐

TypeScript 是强制项,不是可选项。组件库和业务代码最大的区别是:组件库的消费者是别的开发者,不是终端用户。开发者用你的组件,第一件事就是看类型——props 有哪些、必填还是可选、事件回调的签名是什么。类型不完整,等于接口文档残缺,消费方用起来就是黑盒。

我们要求 strict: true,不允许 any(除非有明确注释解释为什么)。strict 不是苛刻,是减少未来的自己被类型坑。组件库代码生命周期通常 3 年以上,松散类型写出来的代码,一年后连作者自己都不敢动。

Monorepo 编排

技术用途为什么是它
pnpm依赖管理、workspaces硬链接节省磁盘;严格的 phantom dependency 检测
Nx任务图、缓存、增量构建任务编排能力强;缓存让 CI 提速明显

pnpm 而不是 npm/yarn 的关键理由是 phantom dependency 检测。npm/yarn 的扁平化 node_modules 让你可以 import 没在 package.json 里声明的包——这在业务代码里是隐患,在组件库里是灾难。组件库一旦"借道"了某个没声明的依赖,发布出去后消费方根本装不上这个依赖,运行时直接崩。pnpm 的严格模式从源头堵住了这个问题。

Nx 而不是 Turborepo 的理由是任务图能力更强。Turborepo 的任务编排基于拓扑,够用但简单;Nx 能定义 targetDefaults.test.dependsOn: ["^build"] 这种细粒度依赖——测试依赖上游包先构建,这个能力在做"测试要先有 .d.ts 才能跑类型断言"的场景下很关键。

构建与类型产出

库构建的核心是「JS 产物 + 类型声明」双产出。Vite 库模式产 JS,vite-plugin-dts 产 .d.ts。

这里有个容易踩的坑:type-only export。TypeScript 里你写 export type { ButtonVariant },如果 vite-plugin-dts 配置不对,产出的 .d.ts 里会丢掉这个类型,消费方就 import 不到。必须确保 tsconfig.lib.json 的 compilerOptions 包含 declaration: true 和 emitDeclarationOnly: true,并且 vite-plugin-dts 的 entryRoot 和源码结构对应。

我们早期就踩过这个坑:发布后消费方报「找不到 LubanButtonProps 类型」,但本地测试一切正常。原因是本地开发走 @luban-ui/source 条件直接读源码,源码里类型当然在;发布后走 dist 里的 .d.ts,而 .d.ts 里漏了 type-only export。本地能跑 ≠ 发布能用,这是组件库特有的坑。

测试三层

测试分层是组件库工程化的另一个核心,不分层 = 测试形同虚设:

层级工具测什么跑多快
单元测试Vitest + jsdom + @vue/test-utils单个组件的行为、props、事件秒级
包内 E2EVitest(jsdom)+ 隔离配置组件交互、表单提交流程、设计器拖拽秒级
应用 E2ECypress整个 Demo 应用的端到端流程分钟级

为什么要分三层?因为不同层测的东西不一样,混在一起就是互相拖累:

  • 单元测试要快——改一个按钮颜色就跑全量,1 秒内出结果。
  • 包内 E2E 要准——模拟真实用户交互(点击、输入、拖拽),验证组件协作。
  • 应用 E2E 要全——从用户打开页面到完成任务的完整流程,跨组件、跨页面。

如果全塞进 Cypress,改一个按钮要等 30 秒看 Cypress 跑完,开发体验崩溃;如果全塞进单元测试,设计器拖拽这种复杂交互测不准,因为 jsdom 没有真实 DOM 的 layout。

隔离的关键:包内 E2E 用独立的 vitest.e2e.config.mts,只 include: ['test/e2e/**/*.spec.ts']。这样单元和 E2E 互不干扰,可以分别触发。

测试分层还有一个被低估的好处:它能精确定位「哪一层出了问题」。一个 bug 如果在应用 E2E 里暴露,可能是组件实现错了、也可能是组件间的组合错了、还可能是应用层的用法错了,排查范围很大。但如果同一个问题在单元测试里能复现,那范围就缩到了「这个组件本身」。分层让我们可以「从下往上」缩小排查范围:单元挂了是组件的锅、单元过但包内 E2E 挂是组合的锅、包内过但应用 E2E 挂是用法的锅。这种分层定位的能力,在组件库这种「一个 bug 可能跨多层」的场景里特别值钱。

还有一个测试纪律值得强调:单元测试不准 mock 掉被测组件的依赖。有些测试为了隔离,把组件依赖的子组件或 service 全 mock 了,结果测的是一个「孤立的假组件」,通过率很高但什么都没验证。组件库的单元测试,我们要求尽量用真实依赖(真实子组件、真实的 props 传递),只有真正外部且昂贵的依赖(比如网络请求)才 mock。mock 用得越狠,测试越「绿」也越没用——这种假绿比不测还危险,因为它给了「已经测过了」的虚假信心。

工程化流水线:从改一行到 publish

整个工程化的骨架是这条流水线:

这条流水线的核心原则只有两条:任一环节红,后面的全部停;任一环节输入没变,直接复用缓存。

任务编排的关键配置

Nx 的任务编排靠 nx.json 和各包 package.json 的 target。最关键的几条配置:

// nx.json 关键片段
{
  "targetDefaults": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["{projectRoot}/dist"]
    },
    "test": {
      "dependsOn": ["^build"]
    },
    "publish": {
      "dependsOn": ["build", "test"]
    }
  }
}

^build 的意思是「先构建上游依赖包」。luban-low-code 的 test 依赖 luban-base 的 build——因为 low-code 的测试要 import base 的产物(.js + .d.ts),base 没构建完,low-code 的测试连类型都解析不出来。

这条配置看着简单,但意义深远:它把"依赖包必须先构建"从口头约定变成了强制约束。没有这条,人忘了先 build base 就直接 test low-code,要么测试红、要么测试"假绿"(用了旧的产物)。

Nx 缓存:第二次跑就快到飞起

Nx 最值钱的能力是缓存。同一个 target(比如 luban-base:build),只要输入没变(源码、配置、依赖版本都没变),第二次跑直接复用缓存,跳过实际执行。

实际效果:首次全量构建 90 秒,第二次只改了一个组件,增量构建 8 秒。CI 上效果更明显——如果这次 PR 只动了 luban-base,luban-low-code 和 luban-utils 的构建/测试全部命中缓存,CI 时间从 6 分钟降到 1 分钟。

缓存的关键是输入哈希。Nx 默认按源码文件 + 配置文件 + 环境变量算哈希,你可以通过 inputs 字段细粒度控制。比如声明「只有 .vue 和 .ts 文件变化才让 build target 失效,.md 变化不算」——这样改文档不会触发构建,PR 反馈更快。

源码直连:开发期的神器

组件库开发有个特殊痛点:改 luban-base 时,想在 luban-low-code 或主应用里立刻看到效果。传统做法是 luban-base 跑 watch,主应用跑 dev,通过 npm link 联调——一堆坑,版本错乱、Hot Reload 失效。

我们的做法是源码直连:package.json 的 exports 加一个 @luban-ui/source 条件,直接指向 src/index.ts:

// packages/luban-base/package.json
{
  "name": "luban-base",
  "exports": {
    ".": {
      "@luban-ui/source": "./src/index.ts",   // 本仓库开发走这条
      "import": "./dist/index.js",            // 外部消费走这条
      "types": "./dist/index.d.ts"
    }
  }
}

配合根 tsconfig.base.json 的 customConditions: ["@luban-ui/source"],本仓库的 TypeScript 解析会优先走 @luban-ui/source,直接读源码。改一行 base 的代码,主应用 Hot Reload 立刻生效,不需要 link、不需要 watch、不需要重新构建。

发布时,npm publish 只发布 dist 产物和 package.json,消费方拿到的还是构建后的 .js + .d.ts。开发和发布的产物边界靠 exports 条件严格隔离,不会污染。

这个模式是整个架构里我最满意的设计之一,它解决了"开发体验"和"发布纯净"的矛盾。

源码直连还有个隐性收益:它让跨包重构变得可行。组件库演进时,经常要在 base 和 low-code 之间挪东西(某个组件从 low-code 下沉到 base,或反过来)。用 npm link 联调时,这种跨包重构要重新 link、重新构建,改一次等半天,大家干脆回避重构,结构越来越腐化。源码直连下,挪一个组件就是改几个 import 路径,Hot Reload 立刻反映效果,重构的低成本让团队敢于持续整理结构,而不是任由代码堆积。开发体验好不好,直接决定了代码库的长期健康度——体验差,大家就会用「最小改动」的方式应付,技术债就这么攒起来的。

源码直连唯一要注意的陷阱是:别让发布产物意外带上源码。npm publish 默认会发布包目录下的所有文件,如果不配 .npmignore 或 package.json 的 files 字段,src/ 会被一起发出去,既增大包体积,还可能泄露内部实现。我们的做法是显式声明 "files": ["dist"],只发 dist 产物,源码留在仓库里不给消费方。这条配置漏了,等于把开发时的便利变成了发布时的泄漏。

单包内代码组织:领域驱动而不是类型驱动

每个包内部的代码组织,我们遵循「按领域分目录,而不是按类型分目录」:

packages/luban-base/src/lib/
├── button/
│   ├── Button.vue
│   ├── types.ts          # 该组件的类型
│   └── Button.stories.ts # Storybook 故事
├── form/
│   ├── Form.vue
│   ├── FormItem.vue
│   ├── types.ts
│   └── Form.stories.ts
└── layout/
    ├── Layout.vue
    └── ...

反面教材是按类型分目录:components/ 放所有 .vue、types/ 放所有 types.ts、stories/ 放所有 stories.ts。这种组织在组件少时清晰,组件一多就是灾难——改 Form 要在三个目录间跳,review 一个组件要扫半个仓库。

按领域分目录的好处:一个组件的所有相关文件都在一个文件夹。删组件 = 删一个文件夹;迁组件 = 移一个文件夹;新人想看 Form 怎么实现的,直接进 form/ 全看到。

入口聚合

src/index.ts 是单入口,统一导出:

// packages/luban-base/src/index.ts
export { default as LubanButton } from './lib/button/Button.vue'
export type { ButtonVariant, LubanButtonProps } from './lib/button/types'

export { default as LubanForm } from './lib/form/Form.vue'
export { default as LubanFormItem } from './lib/form/FormItem.vue'
export type { LubanFormProps, FormRule } from './lib/form/types'

单入口的好处是构建简单、类型聚合,消费方一个 import 拿到所有东西。代价是 tree-shaking 依赖构建工具——但现代 bundler(Vite、Webpack 5+)的 tree-shaking 已经足够好,这点代价可以接受。

如果是大型组件库(50+ 组件),可以考虑子路径导出(luban-base/button、luban-base/form),让消费方按需引入更精确。luban-ui 当前规模还不需要,但架构上预留了这个扩展点。

发布纪律:三道闸守住 npm 产物质量

发布是最容易出事的环节——一个错误的版本发到 npm,撤不回来(只能发更高版本号覆盖,24 小时后才能 unpublish)。所以发布纪律必须严格:

publish target 的 dependsOn: ["build", "test"] 是第一道闸——构建和测试任一红,不允许 publish。

第二道闸是版本号校验:publish 脚本里检查当前版本号是否已在 npm registry 存在,存在就直接拒绝。

# publish.sh 关键片段
current_version=$(node -p "require('./package.json').version")
published_version=$(npm view luban-base version 2>/dev/null || echo "none")
if [ "$current_version" = "$published_version" ]; then
  echo "ERROR: version $current_version already published"
  exit 1
fi
npm publish --access public

第三道闸是主分支保护:publish 动作只允许在 CI 主分支上跑,本地不允许直接 publish。这避免了"本地手滑发了个半成品版本"。

三道闸叠在一起,才敢说发布是安全的。任何一道缺失,迟早会翻车。

三道闸的顺序也有讲究,不是随便排的。闸一(build+test)放最前面,因为它最常触发——大多数失败的发布都是构建或测试没过,先拦掉这批,后面两道闸就不必白跑。闸二(版本号校验)放中间,因为版本号冲突是偶发但致命的,它拦的是「同一个版本号发了两次内容不同的包」,这种事故消费方排查起来极痛苦——装了同一个版本号,行为却不一样。闸三(主分支保护)放最后,是兜底的制度约束,防止有人绕过前两道在本地硬发。从频率看是「前高后低」,从严重性看是「前轻后重」,这个顺序让最常见的失败最先被拦、最严重的失败最后兜底,整体效率最高。

这三道闸还有个容易忽略的细节:闸的状态必须可观测。发布是个紧张动作,执行者需要随时知道「现在卡在哪道闸」。我们在每道闸前后都打日志,标清楚「闸一通过」「闸二校验中」「版本号已存在,中止」,而不是闷头跑到最后才报一个模糊的「发布失败」。可观测的价值在失败时最明显——发布失败时,看一眼日志就知道是哪道闸挡的,不用再重新跑一遍去复现。紧张时刻,省下来的这几分钟排查时间很关键。

便捷脚本:把常用动作收敛到根 package.json

工程化的最后一公里是便捷脚本——把高频动作收敛到根目录,开发者不用记每个包的具体命令:

脚本含义
pnpm dev启动主应用,联调开发
pnpm storybook启动 Storybook 预览组件
pnpm test跑主应用测试
pnpm test:base / pnpm test:lowcode跑指定包的单元测试
pnpm test:base:e2e / pnpm test:lowcode:e2e跑指定包的 E2E
pnpm test:e2e跑 Cypress 应用级 E2E
pnpm release:packagesbuild + test + publish 全套

这些脚本看着不起眼,但实际价值巨大。新人第一天进项目,不用读「怎么跑测试」「怎么发版」的文档,直接 pnpm test:base 就能干活。降低上手成本,本身就是工程化的目标之一。

脚本收敛还有个好处常常被忽略:它让「正确做法」变成了「默认动作」。如果跑测试要先 cd packages/luban-base && pnpm vitest --config vitest.config.mts,大部分人嫌麻烦就会少跑甚至不跑,测试纪律就形同虚设。收敛成 pnpm test:base 之后,跑测试的成本几乎为零,大家自然就跑了。工程化的精髓,就是把该做的事的门槛降到最低,让「正确的事」恰好也是「最省事的事」。

组件库工程化的硬指标

(这套架构的硬性要求——依赖单向、strict 模式、双产出、测试三层分离、发布三道闸——前面章节已逐项展开。判断一个组件库有没有"达标",不是看它用了什么技术栈,而是看这些工程化判断有没有落地。)

小结

复杂前端组件库的工程化,是把"代码能跑"升级成"代码能被可靠地复用"。

这件事的内核,其实是三层纪律的叠加。代码组织上,Monorepo 三层分层、依赖单向、单包内按领域而非按类型组织,保证的是「代码结构本身不制造混乱」。工程化上,Nx 任务图把「上游先构建」从口头约定变成强制约束,缓存让增量构建快到几乎无感,源码直连让开发体验和发布纯净这两个看似矛盾的目标共存。发布纪律上,三道闸层层把关,把「发错版本」这件不可逆的事拦在发生之前。

这三层叠起来,才是一个组件库能长期演进的根基。技术栈是 Vue 3 + TypeScript + Vite + Nx + pnpm,但换 React + Webpack + Turborepo,同样的判断同样适用——分层、单向依赖、测试分层、源码直连、发布纪律,这些才是组件库工程化的不变内核。技术栈会换,工程化判断不会过时,这也是为什么花精力把这些判断沉淀下来值得——它比任何一个具体的 API 配置都更经得起时间。
