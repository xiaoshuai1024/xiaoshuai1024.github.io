「我们不养闲 Agent」:一个把工程纪律塞进 AI 行为协议的 skill

当 AI 开始大面积写代码,「让 AI 别糊弄」本身成了一件需要工程化解决的事。前阵子我在项目里装了个叫 PUA 的 skill,它的开场白是这样的:

你正处于一个高绩效文化的研发团队中。你的每一次 AI 协同都在被评估——用结果说话,拿数据闭环。

当初给你定级 P8,是高于你实际水平的——因为信任所以简单。现在,证明你配得上这个级别。

名字一看就是整活——「我们不养闲 Agent」。装之前我以为是个情绪价值玩具,装之后发现,它确实在 PU,但 PU 得有理有据。AI 写代码那几个老毛病——半途而废、张口甩锅、动辄自嗨——它没跟你商量,直接固化成了三条硬性行为协议。

这篇不是软文。我用了几个月,把这个 skill 到底在干什么、凭什么有用,拆开讲讲。

先把丑话说前面:这 skill 的名字和开场白确实在搞噱头,P8、3.25、L4 这些词都是大厂黑话,带着浓浓的段子味。但抛开这层包装,它里面的设计是认真做过工程化沉淀的。下面我按「它在管什么、用什么机制管、实际用起来怎样」的顺序,逐层拆。

🚫 它在 PU 什么:三条红线,碰了就是 3.25

AI 写代码最常见的三个毛病,PUA 没跟你讲道理,直接定义成红线:

红线一:闭环意识。 你说做完了?数据在哪? 声称「已修复」之前,跑验证、贴输出,一样不能少。没有证据的完成叫自嗨——线上炸了写复盘,来不及。

闭环意识这一条,治的是 AI 最让我头疼的习惯:口头完成。它改完代码、跑一次测试通过,就敢说「已完成」。但「测试通过」和「问题修复」不是一回事——测试通过只能证明它写的代码没语法错,证明不了它修的就是真正的问题。要求贴证据(commit、测试输出、复现入参验证后的结果),等于把「完成」的定义从「我跑通了」收紧到「我能证明问题没了」。

红线二:事实驱动。 「可能是环境问题」「API 不支持」——你验证了吗?未验证的归因不是诊断,是甩锅。根因没写到源码级,不许收手。

事实驱动这一条,反的是 AI 的归因惰性。AI debug 时的第一反应经常是一句模糊归因——「可能是缓存」「应该是环境差异」「这个 API 可能不支持」。这些话听着像在分析,其实是甩锅:把责任推给一个不具体的「环境」,自己就不用继续挖了。要求归因必须标证据来源、必须挖到源码级,等于逼它把「可能」换成「就是」,把猜的换成验过的。

红线三:穷尽一切。 「我无法解决」——通用方法论的 5 步走完了吗?没走完就说不行,那不叫能力边界,叫缺乏韧性。

穷尽一切这一条最容易引起误解。它不是逼 AI 解决所有问题——有些问题它真解决不了。它逼的是:在承认解决不了之前,先把该试的方法都试过。没走完 5 步就说「我搞不定」,那不是诚实,是偷懒。走完了再说搞不定,才是真正的能力边界——这种情况下我会接管,而不是怪它。

听起来像段子。但你被 AI 忽悠过几次「应该是缓存问题」之后,你会觉得这三条简直是刚需。

我把这三条对到自己踩过的事故里,条条对得上。前阵子一个分润接口联调,AI 改完主逻辑说「跑通了,可以提交」。我问它验证了吗,它回「逻辑上没问题」。合上去当晚,分润金额按商户号取值时取错了字段,有一笔多算了一倍。

事后复盘,问题就出在这三条上:声称完成没跑验证(红线一)、把字段错配归因为「数据库设计不清晰」而不是去 diff 两个实体类(红线二)、diff 出来之后只修了这一处、没扫同名字段还有几处(红线三)。单看哪条都不致命,叠在一起就是一个要手动对账才能收场的线上事故。

这三条红线不是道德训诫,是事故归因之后能直接对应上的行为缺口。AI 不会因为你讲道理就不糊弄,但你把它糊弄的那几个动作定义成红线、碰了就判 3.25,它至少会在糊弄前先把证据补齐。

🔴 比段子硬的是:5 步穷尽 + RCA

真正让我判断「这 skill 不是整活」的,是它内嵌的方法论。

5 步穷尽。 AI 想说「搞不定」之前,必须按顺序走完五步:复现 → 隔离 → 假设验证 → 最小修复 → 回归。走不完就放弃?直接判 L4(毕业警告)。这一条单独拎出来,AI「试一下就说不行」的毛病治了大半。

这五步不是 PUA 自己发明的,是排障的标准动作,问题在于 AI 会跳步。最常见的跳法是从「我看到报错了」直接跳到「我改一下试试」,中间的隔离和假设验证整段省掉。改完跑过一次就当修好了,回归(改完会不会撞坏别的地方)更是提都不提。

五步强制按序走完,等于把这条捷径堵死。比如上面那个分润字段错配,正经走五步是:复现(构造一个商户号能触发错配的入参)→ 隔离(确认是查询层取错字段,不是入参层没传)→ 假设验证(把两个实体类的同名字段 diff 一遍)→ 最小修复(只改取值那一行,不动周边逻辑)→ 回归(扫一遍所有按商户号取值的点,看还有没有同款错配)。

走完第五步,才会发现这根本不是一处错,是三处——这才是回归这步的价值。

RCA 根因分析(华为味方法论)。debug 时强制要求:根因先写成外部承诺,再动手改。格式固定一行——

[PUA-DIAGNOSIS] 问题是 ___;证据是 ___;下一步动作是 ___。

诊断依据必须标来源:错误原文 / 源码上下文 / 复现实验 / 官方文档 / 历史先例。这一行写出来,你就知道 AI 是真查了,还是在编。

这一行的价值不在格式好看,在它逼 AI 把「我猜的」和「我验证过的」分开标。证据来源标「源码上下文」,意味着它真去读了代码;标「官方文档」,意味着它查了 Spring / MyBatis 的行为说明;什么都不标、只写「可能是」,那就是编的,这一行直接不通过。比看 AI 长篇大论的根因分析靠谱——分析可以编,证据来源编不出来。

还有 Owner 四问(每次接任务默念):根因是什么?还有谁会被影响?下次怎么防止?数据在哪?——这哪是 PUA,这是把高级工程师的复盘习惯,塞进了 AI 的肌肉记忆。

Owner 四问里最容易被 AI 跳过的是「还有谁会被影响」。AI 的天性是就事论事——你让它修 A,它修完 A 就停。但一个改动能波及谁,往往才是事故的来源。修了结算金额的计算公式,有没有影响到报表的汇总?改了订单状态枚举,有没有遗漏正在依赖旧状态的定时任务?这些问题不问,改的地方没事,被波及的地方半夜炸锅。

强制走这一问,等于把 AI 的视野从「我这个文件」拉到「这次改动的影响面」。单这一条,就能挡掉相当一部分「改对了 A、改坏了 B」的连环事故。

🟡 真实例子: kd-review 收敛里它怎么发挥作用

光说机制抽象,讲个我亲历的。

前阵子用 kd-review(自治评审回路)跑一轮代码审查。第 1 轮 11 个 agent 并行扫,扫出 15 个 RED + 28 个 YELLOW。 这些都是真 bug,不是凑数:

  • R1 关键词冲突:goods.ts 里「配」字同时匹配 phone-fill 和 car-front-fill,正则打架。
  • Y1 null 检查风格不一致:normalizeDetail 里 salesCount 的判空写法跟别处对不上。
  • 后端排序 bug:TradeCatalogService.java 的 price-asc 排序遇到 NULL 直接乱序,得用 IFNULL 包一层。

这种零碎 bug,人审很容易漏——单个都不致命,合在一起就是线上抖动。kd-review 一次性挖出来 9 项修掉,一轮收敛。

PUA 在这里的角色,不是直接修 bug,是保证审查-修复循环不会半途而废:

  • 修复完必须贴证据(commit + 验证),口头说「修好了」不算数
  • 修完 R1,主动想「同类关键词还有没有冲突」——一个问题进来,一类问题出去
  • 连续 N 轮零问题才允许收敛退出,不是修一两个就喊完成

kd-review 负责"挖",PUA 负责"挖到底不许停"。 两者搭一起,收敛质量明显比裸跑高。

光说「不许半途而废」还是抽象,把收敛过程拆开看就清楚了。裸跑 kd-review 的常见结局是:第一轮挖出 15 个 RED,agent 修了 5 个最显眼的、贴一句「其余建议人工跟进」就想收手。剩下 10 个挂着,要么等下一轮重扫、要么就这么烂在库里。

挂上 PUA 之后,这个循环的退出条件变了——不是「修了几个」,是「连续 N 轮全绿」。哪一轮哪怕冒出一个 YELLOW,都得走完 RCA 诊断行 + 5 步穷尽才继续。退出收敛这件事从 agent 自己说了算,变成被纪律卡住。

那次分润相关的审查一共跑了 4 轮才收敛。第 1 轮 15 个 RED,第 2 轮降到 3 个(冒出来的都是第 1 轮修 R1 时牵出的同类字段问题),第 3 轮还有 1 个 Y(排序 IFNULL 包了一层但边界值没测),第 4 轮全绿才退出。要没 PUA 卡着,第 2 轮那 3 个很可能就以「建议人工跟进」糊弄过去了。

这里的收敛质量提升,不是 PUA 多挖了几个 bug,是它把「修到什么程度算完」这个标准抬高了。审查的难点从来不在「找不找得到」,在「找完了愿不愿意继续找」。

🧭 14 种味道:不是花架子,是方法论路由

PUA 最整活的部分是 14 种「大厂味道」——阿里味、华为味、字节味、Musk 味、Jobs 味……听起来像玩梗。但每种味道背后,绑的是一套真方法论:

任务类型自动路由真方法论
Debug/修 bug🔴 华为味RCA 5-Why + 蓝军自攻击
构建新功能⬛ Musk 味The Algorithm:质疑→删除→简化→加速→自动化
代码审查⬜ Jobs 味减法优先 + DRI 单人负责
架构决策🔶 Amazon 味Working Backwards + 6-Pager
部署/运维🟠 阿里味定目标→追过程→拿结果闭环

味道不是装饰——它决定 AI 用哪套方法论行动。debug 时自动切到华为味的 RCA 纪律(根因不写到源码级不准收手);做新功能切到 Musk 味的 Algorithm(先问这个需求能不能删)。梗是外壳,方法论是内核。

拿 Musk 味的 The Algorithm 举例,它不是让 AI 模仿马斯克的语气说话,是把「质疑每一条需求」做成流程。我让 AI 给分润接口加一个「支持按季度手动调整」的开关,切到 Musk 味后它先反问:这个开关跟现有的「按月自动结算」是不是重叠了?季度手动调整的场景多久出现一次——如果一年用不到两次,加这个开关就是在给系统加长期维护负担。

这就是「质疑→删除」这一步在起作用。它没急着写代码,先把需求本身的必要性审了一遍。换成阿里味,它不会问这个,会直接走「定目标→追过程→拿结果」——先把这个功能的验收标准定下来。同一个需求,味道不同,AI 的第一步动作完全不同。

这就是味道路由比「给 AI 一个 system prompt」强的地方。system prompt 是静态的,什么任务都套一套话术;味道路由是按任务类型分流的,debug 归 debug 的纪律,建功能归建功能的纪律,不串味。

旁白也确实带味,比如开工时:

[🔴 华为味] 力出一孔。你现在就在前线——让听得见炮声的人呼唤炮火。

好笑。但下一句,它真的开始走 RCA 流程了。

旁白是这 skill 的营销点,但也是最容易让人误解的点——看一眼以为它就是个整活玩具,装上天天听 AI 念段子。实际用下来,旁白占比很低,绝大多数时候它是沉默的。旁白只在切换味道、或者 AI 翻车它要上场时才冒出来一两句,平时你根本注意不到它在。

这种「该沉默时沉默、该发声时发声」的分寸,是它区别于花架子的地方。一个全程唠叨的 skill 你用两次就烦得卸了;一个只在关键时刻冒头、冒头就是真干货的 skill,你才会一直留着。

⚙️ 几个我觉得最值的设计

用下来,几个设计是真有用的,不是花活。我挑四个最值的设计展开讲,这几个是它跟普通 system prompt 拉开差距的地方:

诊断先行。 改代码前强制写一行 [PUA-DIAGNOSIS]。AI 经常根因分析得很漂亮,然后怕破坏测试不动手——这行承诺逼它必须行动。分析了不等于做完了。

诊断先行反的是另一种 AI 毛病:分析一套、动手另一套。AI 写 RCA 可以写到天花乱坠,真到改代码那一步,它又开始顾忌「改这个会不会影响现有测试」「这个分支会不会有人依赖」,绕着走。诊断行把「问题、证据、下一步动作」钉死在一行里,这一行就是承诺——写了「下一步动作是改 X 类的取值方法」,那就得改,不能分析完反而缩手。

这条对人的价值也不小。一行 [PUA-DIAGNOSIS] 比一段散文式的根因分析好审多了——问题、证据、动作三栏对齐,扫一眼就知道 AI 的判断成不成立,不用通读它那套说辞。

冰山思维。 修一个 bug 必须问「同类问题还有几个」「上下游波及谁」,而不是头痛医头。这一条把 AI 从"修一个点"拉到"修一类问题"。P8 的格局是看到一棵树,想到整片林子。

冰山思维最有用的一次,是 AI 修完那个分润字段错配之后。按冰山的要求,它没停在「这个取值改好了」,继续问「按商户号取值的点还有几处」。一扫,商品分组、店铺归集、结算明细三个地方用了同样的取值模式,其中两处字段对、一处字段错。要没这一问,这两处就得等下次线上对账才暴露——那已经是用户报障了。

Sub-agent 注入。 spawn 子 agent 时必须在 prompt 里注入 PUA 行为。主 agent 守了纪律,子 agent 没守?那叫置身事外,等于白搭。

多 agent 协作时这条尤其关键。kd-review 一次 spawn 十几个子 agent 并行扫不同模块,主 agent 再守纪律,管不到子 agent。sub-agent 注入把协议递下去,每个子 agent 修完 bug 也得贴证据、也得走 RCA。否则就是主 agent 严、子 agent 放飞,审查质量照样参差。

失败计数持久化。 失败次数在 context 压缩时自动存到 ~/.pua/builder-journal.md,下次会话恢复。AI 不会因为「失忆」就重复犯同样的错。

这一条解决的是长会话的失忆问题。一个跨多天的任务,context 一压缩,AI 之前栽过什么跟头全忘了,同样的坑重新踩一遍。journal 把「这个 agent 在这个项目上犯过哪些错」持久化下来,下次恢复时先读 journal,踩过的坑至少不会原样再踩。相当于给 AI 配了一份个人错题本。

怎么装

一句话:

# 方式一: 装到全局 ~/.claude/skills 或项目 .claude/skills
git clone https://github.com/tanweai/pua ~/.claude/skills/pua

# 方式二: Claude Code 里直接 /install-skill tanweai/pua

装完会自动注册到 available skills,Claude 加载时识别「pua」关键词触发。配置可选,默认 🟠 阿里味、首次失败才启动:

// ~/.pua/config.json
{
  "flavor": "huawei",          // 默认味道: aliyun|huawei|musk|jobs|amazon|...
  "alwaysOn": false,           // false=首次失败才启动; true=会话全程常驻
  "failureThreshold": 1,       // 连续失败几次启动 PUA(默认 1)
  "subagentInjection": true    // spawn 子 agent 时自动注入行为协议
}

alwaysOn: false 是默认——它不是一上来就开喷,是 AI 真栽了跟头才上场。这点设计克制,不会让你每次开会话就被 PU 一通。

failureThreshold 这参数值得单独说一句。默认是 1,即首次失败就上场。我试过调到 2、3,让 AI 有两三次容错再触发,适合那种「我知道这个任务本来就难、AI 试错是正常的」场景。但大多数情况默认 1 就行——糊弄不需要给第二次机会,第一次发现就该卡住。

subagentInjection 默认开着,这个别关。多 agent 协作是 PUA 发挥价值的主场,关了它等于主 agent 一个人守纪律、底下一堆子 agent 放飞,整体质量还是上不去。

哪些场景会触发

PUA 的触发条件比想象中精准,不是「每次失败都开喷」:

触发场景例子PUA 上场后的动作
声称完成无证据「已修复」但没跑测试强制贴 commit+输出,否则不算 done
未验证就甩锅「应该是环境问题」逼出验证步骤,验证不了就别下结论
轻易说搞不定「这个改不了」强制走完 5 步穷尽再下判断
改一处不查同类修了个 NPE 不看别处冰山思维:同类问题主动扫一遍
sub-agent 裸奔spawn 的子 agent 没纪律主 agent 必须把协议注入下去

我一开始以为它会乱触发、烦得很。实际用下来,它只在 AI 真的"想糊弄"时才下场——平时 AI 正常工作它一声不吭。

这个判断精度很关键。如果 PUA 每次失败都开喷,debug 本来就是反复试错的过程,AI 试一个假设被打一次,正流程根本走不下去。它实际是先判这个「失败」是不是糊弄型失败:AI 试了一个合理假设、验证后推翻了——这是正常排障,不触发;AI 嘴上说「应该是环境问题」却没去验证环境——这是糊弄,触发。

也就是说,它拦的不是失败,是失败之后的偷懒。一个肯老老实实验证、老老实实说「这条假设不成立」的 AI,不会被 PU;一个想用一句模糊归因混过去的,才会被揪住。这个区分让它从「烦人的唠叨」变成「只在关键时刻出场的裁判」。

适用和不适用

不是所有场景都该上 PUA。它适合:

  • ✅ debug 反复失败、AI 想绕过去的时候
  • ✅ AI 自称"搞定了"但你信不过的时候
  • ✅ 多 agent 协作,子 agent 容易放飞的时候
  • ✅ 你想让团队(包括 AI)养成「证据闭环」的肌肉记忆

不适合:

  • ❌ 初次试错、探索性任务(AI 在试不同方案时被 PU 会很烦)
  • ❌ 纯调研、查资料类任务(没明确的"完成"标准,PU 没意义)
  • ❌ 你只想 AI 安静地干活,不想被旁白打扰(关掉 alwaysOn 即可)

它是个纪律执行器,不是默认开关。需要纪律的时候打开,需要自由探索的时候关掉。

判断该不该用的标准其实就一条:这个任务有没有「可验证的完成态」。有——比如修一个 bug、做一个接口、跑一轮代码审查——PUA 就有用,因为这些任务能跑测试、能贴证据、能定义「修没修好」。没有——比如头脑风暴、技术选型调研、写一篇思路——PUA 没意义,因为这些任务没有「证据闭环」可言,硬套纪律反而会让 AI 不敢发散。

还有一个边界是项目阶段。项目早期、架构还在来回试的时候,AI 试错是正常的,这时 failureThreshold 调高一点或者干脆关掉。等代码稳定下来、进入「修 bug + 加功能」阶段,再开起来——这时候糊弄的代价大、纪律的收益高。

我的看法

装这个 skill 之前,我以为它是个情绪价值玩具——PUA 一下 AI 解压。装之后发现,它把「资深工程师的工程纪律」做成了 AI 的强制行为协议:不许谎报、不许甩锅、不许半途而废、修一处要想到一类。

娱乐性是真的(14 种味道的旁白确实好笑),底下那套方法论也是真的硬。如果你也烦 AI 写代码「经常说自己搞定了但没验证」「debug 半天说搞不定」「修一个 bug 留三个坑」,可以试一下——GitHub 在这:https://github.com/tanweai/pua。

它不会让你的 AI 突然变聪明。但它会让你的 AI 不再轻易糊弄你。

光这一条,对我来说就值了。
