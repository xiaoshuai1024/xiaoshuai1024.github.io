把"AI 保障"焊进系统骨架:让 AI 想跑偏都难

我们项目用 AI 协同开发两个月,产出近百万行代码,同时也产出了几百个被保障层拦住的隐患——跨租户查询、漏带权限校验、静默删行为代码、迁移版本撞车。这些隐患每一个溜进生产都是事故。保障层拦住的不是「AI 偶尔抽风」,是「AI 日常产出里必然包含的系统性风险」。

AI 让代码产得快,也让坏代码产得快。一个能稳定输出代码的系统,如果骨架本身兜不住,产能越高、负债累积越快。所以我现在反而比以前更花时间在那些"无聊"的基础设施上——多租户隔离、权限过滤器、契约扫描、迁移治理、可观测。不是因为它们新,因为它们旧、因为它们"枯燥",才成了 AI 时代最值钱的东西。架构师的活儿从"设计能跑的系统",变成了"设计能兜住 AI 的系统"。

先说我们这个项目的底座

我们做的是一个多租户 SaaS,典型的三段式结构:C 端小程序 + 运营后台 + Java 后端。规模不算夸张,但因为是多租户,出了事影响面大,所以对隔离和契约特别较真。

架构底座就这么几样,说出来都是老生常谈:

  • 多租户行级隔离,每张业务表都带 tenant_id,查询自动带过滤;
  • RBAC 权限过滤器链,权限校验不靠人记得加,而是在过滤器里强制走;
  • 契约驱动,前端调的 URL、后端的路由注解、SQL 引用的表、Flyway 的建表 DDL,两两对得上才算数;
  • Flyway 迁移治理,版本号唯一、有 seed 数据、不许手改线上表结构;
  • 可观测,Prometheus / Grafana / ELK 一套,出问题能追溯。

这里有个值得展开的设计取舍:多租户隔离我们没有用 MyBatis-Plus 的 TenantLineInnerInterceptor(自动给每条 SQL 拼 tenant_id)。不是不知道有这个,是有意不用。原因是它太「隐式」了——自动拼 WHERE 条件,开发者看不到,一旦某个复杂查询需要跨租户(比如平台超管看全量),要写特殊注解绕过,绕过逻辑又成了新的隐患。

我们选了「显式带」:每条业务 SQL 手动带 tenant_id 条件,配合 ArchUnit 规则强制检查 Repository 层 SQL 是否都带了租户过滤。显式的好处是每条 SQL 的租户边界都看得见,review 时一眼能确认;坏处是依赖人记得带——这就是为什么要配 ArchUnit 规则兜底,人忘了机器抓。

TenantFilter 的实现也值得展开。它是个 OncePerRequestFilter,从 X-Tenant-Id 请求头取租户,塞进 ThreadLocal(TenantContext),请求结束 finally 里 clear。关键是它按 path 前缀白名单:不是所有接口都要租户(比如登录、健康检查、静态资源),白名单内的直接放行。AI 加新接口时,如果忘了把需要租户的接口放进白名单外,过滤器会拦——这又是一道隐式保障。

没有任何一项是"AI 时代的新玩意儿"。但这些东西在 AI 时代的权重,是被拉高的,不是被降低的。

为什么"无聊"的基础设施变重要了

以前一个资深工程师手写代码,他脑子里有上下文:这条 SQL 多租户场景下要带 tenant_id、这个接口对外要校验权限、这次改表要写迁移。人会忘,但频率可控,靠 code review 兜得住。

现在不一样了。AI 生成代码的时候,它没有项目级的敬畏感。它会"合理地"省掉一个 JOIN 条件,因为单看那条查询,不带 tenant_id 也能跑通;它会"优化"掉一段它看不懂的并发守卫,因为编译器不报错。

问题不在 AI 笨。问题在于:单点视角的合理性,在系统视角上是危险的。

举几个 AI 实际会犯的错,都是我项目里真实遇到过的。

第一类,租户上下文丢失。AI 写一个定时清理任务,里面调了 TenantContext.getTenantId() 拿当前租户。单看这个方法调用,完全合理——Web 接口里到处这么用。但定时任务没有 HTTP 请求,没有 TenantFilter 设置上下文,getTenantId() 返回 null,清理逻辑要么全表扫(跨租户越权),要么直接 NPE。AI 不知道这个区别,因为它看到接口里这么写就照搬了。

第二类,SQL 漏带 tenant_id。AI 写一个查询,join 了三张表,主表带了 tenant_id 过滤,但它没意识到 join 的从表也要带——因为「主表过滤了从表自然就过滤了」在逻辑上成立,但在 SQL 执行计划里,优化器可能先扫从表再过滤主表,中间结果会跨租户。这条靠人 review 偶尔能抓住,但 AI 每次生成新查询都可能漏,靠 review 兜不住。

第三类,静默删除行为代码。就是前面说的 270 行事故。AI 删代码的判断标准是「这段代码有没有被引用、编译器报不报错」,而不是「这段代码守的是什么不变量」。并发守卫、降级逻辑、幂等检查这些「行为类」代码,没有直接引用,编译器不报错,AI 理所当然地认为是冗余,删了。

这三类错的共同点:从 AI 的单点视角看,每个决定都合理。它不是在捣乱,它在做局部最优的优化。但系统安全要求的是全局一致,局部最优凑起来可能正好打开一个越权口子。

所以能把"系统视角"焊死在骨架里的那些机制,就变成了保险绳。tenant_id 自动注入、权限走过滤器、契约两两校验、迁移版本唯一——这些是机器能 100% 执行的硬约束,不依赖任何人或 Agent 当下是否清醒。

这就是我定义的**“AI 保障”:当大量代码由 AI 生成,怎么保证质量与安全。答案不是事后多派几个 Agent 去盯,而是在系统骨架里就把保障建好**,让"想跑偏"这件事在机制层面就难发生。

有人会问:搭这么多保障层,前期投入不小,值得吗?我的算法是看「事故成本 × 发生频率」。一次跨租户越权,轻则数据泄露赔偿,重则监管处罚、客户流失;一次静默删代码溜进生产,排查 + 回滚 + 修复的人力成本远超搭建门禁的投入。AI 把代码产能拉高了一个数量级,也把事故频率拉高了——以前一个月手写可能引入一两个隐患,现在 AI 一天就能引入五六个。事故频率上去了,保障层的 ROI 就从「锦上添花」变成「不可省的基础设施」。

另一个角度:保障层一旦搭好,边际成本几乎为零。契约闸、ArchUnit、测试门禁都是写一次持续跑的,不会因为代码量增加而增加维护成本。但它们拦住的 bug 数量,是和代码量正相关的——代码越多,AI 写得越快,这些闸拦住的隐患越多,价值越高。这是典型的「前期投入大、长期收益指数级」的基础设施投资。

我们建进骨架的四样东西

具体来说,我们把四样东西长进了系统骨架。

一是评审回路

一次改动会扇出十几个只读评审 Agent,按模块分桶、worktree 隔离,只看不改;结果一回来就派修复 Agent 跟上,按严重度自动修。

这套回路为什么是「系统的一部分」而不是临时补丁?因为它解决的是 AI 生成代码的根本矛盾——写代码的 AI 和审代码的 AI 是同一个,盲区重叠。让写代码的 Agent 自己 review 自己的产出,等于让学生自己批改自己的卷子,系统性放过同类错误。

回路的设计是把这个矛盾拆开:写代码用一套 Agent,审代码用另一批完全独立的 Agent,每个只看一个窄领域(Controller 写法、SQL 安全、租户隔离、Flyway 规范……)。审查 Agent 的上下文里没有「这段代码是我写的」这个包袱,它只对着规则文件挑毛病。结果按严重度分级——🔴 阻断必须修、🟡 建议修、🟢 提示——回到主流程,派修复 Agent 处理 🔴 和 🟡,修完再跑一轮审查,直到连续两轮零 🔴 才放行。

这个「收敛循环」是关键。单轮审查会漏,但多轮收敛能逼近零——因为每一轮都从不同角度查,盲区在缩小。我们的实践是,大多数改动 2-3 轮就收敛,极少数复杂的 4-5 轮。这个我们后面会单开一篇细聊。

二是契约闸

pre-commit 阶段一个四层扫描器,跨层校验:

  • 前端 URL 对后端路由;
  • SQL 对 DDL;
  • 权限 key 对种子数据;
  • 迁移版本号唯一。

编译器管不到的跨层错误,它来管。曾经朴素正则没匹配上"类级 + 方法级注解拼出来的路由",漏检过,后来改成全文匹配才补上。详见契约闸那篇。

# pre-commit:四道契约闸任一红就拦下
python3 scripts/check_url_route.py      || exit 1  # URL ↔ 路由
python3 scripts/check_sql_ddl.py        || exit 1  # SQL ↔ DDL
python3 scripts/check_permission_key.py || exit 1  # 权限 key ↔ 种子
python3 scripts/check_flyway_uniqueness.py || exit 1  # 迁移版本号唯一

这四道闸怎么和 AI 保障扯上关系?因为 AI 最擅长的就是「改了一层忘了同步另一层」。它给后端加了个新接口 @PostMapping("/api/order/refund"),但忘了前端对应的调用还是旧路径 /api/order/cancel;它写了个新 SQL 查 order_item 表,但忘了这个表是新加的、Flyway 里还没有 DDL。这些错误单看每层都对,跨层就断了。契约闸的作用就是在提交那一刻,把跨层一致性整个过一遍——AI 可以忘,闸不会忘。

权限 key 这道闸尤其值得展开。我们的权限是 RBAC 模式,前端用 v-perm="'order:refund'" 控制按钮显示,后端用 @RequirePerm("order:refund") 校验接口,权限定义存在 Flyway seed 里。三处任何一处对不上,要么按钮显示了但接口 403,要么接口没校验但按钮藏了(越权风险)。AI 加一个退款功能,可能前端写了 v-perm、后端写了注解,但忘了在 seed 里插权限记录——这样按钮永远不显示,功能像坏了一样排查半天。权限 key 闸就是扫这三个地方,任一处缺失就拦下。

三是测试门禁

UT / IT / E2E 三层,提交前必须 verify 跑通。

这件事是被一次事故逼出来的(我下一篇会专门复盘):一次"搜索优化"改动静默删了 270 行并发安全代码,mvn compile 照样过。结论很硬:编译通过 ≠ 代码正确,只有测试跑过才算数。

<!-- pom.xml:verify 阶段强制跑 verify,不是 compile -->
<plugin>
  <artifactId>maven-failsafe-plugin</artifactId>
  <executions>
    <execution>
      <goals><goal>integration-test</goal><goal>verify</goal></goals>
    </execution>
  </executions>
</plugin>

提交门禁从 mvn compile 升级到 mvn verify,这一条改动拦住了所有"删了行为但语法照过"的 bug。

四是规范固化

项目顶层一份总规则文件,每个 Agent 启动都加载。再加三层机器可执行的约束:

  • 按文件类型自动触发的规则;
  • 90 多个按需加载的技能;
  • 运行时拦截危险 git 命令的 hook。

规范不是写在 wiki 里给人看的,是写成机器可执行的、给人和 Agent 共用的。

第五样:ArchUnit 架构守护(机器执行的架构纪律)

前面四样是流程层面的保障,这一样是代码层面的架构纪律,用 ArchUnit 把架构规则写成测试,违规就编译不过。它解决的就是 AI 最容易犯的「架构越界」错误。

举两个真实的规则。第一个,数据访问纪律:业务层(Service)禁止直接调 JdbcTemplate、禁止用 @Select 注解写 SQL。AI 写代码时,为了图快经常直接在 Service 里拼 SQL 查——它能跑,但它绕过了 Repository 层的所有封装(缓存、软删除过滤、租户隔离)。ArchUnit 的 ARCH-DATA-01 规则在测试阶段扫描所有 Service 类的方法调用,发现调了 JdbcTemplate 直接 fail。AI 可以图快,架构纪律不让它越界。

第二个更精彩,是异步上下文穿透的防线。规则叫 ARCH-SAFE-02:声明了 @Scheduled 的类,不得调用 TenantContext.getTenantId。为什么?因为定时任务跑在独立线程,没有 HTTP 请求经过 TenantFilter,ThreadLocal 里的租户上下文是空的。AI 写定时任务时,看到 Web 接口里 TenantContext.getTenantId() 到处用,照搬过来——单看合理,但运行时拿到的 null 会让清理逻辑要么越权全表扫,要么 NPE。

这条规则用 ArchUnit 怎么表达?遍历所有声明了 @Scheduled 方法的类,检查它们是否调用了 TenantContext.getTenantId,调了就 fail:

@ArchTest
static final ArchRule ARCH_SAFE_02定时任务不调TenantContext =
    classes().that(declaresScheduledMethods)
        .should().notCallMethod(TenantContext.class, "getTenantId")
        .as("@Scheduled 所在类不得调用 TenantContext.getTenantId")
        .because("定时任务无 HTTP 上下文,TenantContext 必为 null");

这条规则抓过真实的 bug。早期 AI 生成的一个「过期会话清理」任务,直接调 getTenantId() 想按租户清理,测试一跑就被 ArchUnit 拦住。如果没有这条规则,这个 bug 会潜伏到上线,在某个租户数据量上来后,清理任务要么扫了所有租户的会话(越权),要么什么都没清(null 匹配不到任何记录,功能失效)。

ArchUnit 还有个特性叫 freeze 基线:既有的违规可以冻结成基线不再报错,但新增违规立即失败。这适合存量项目渐进式治理——你不能一次性修完所有历史违规,但能保证新代码不再制造违规。对 AI 协同开发尤其重要:AI 不会知道你的架构规范,但 ArchUnit 会让它每次越界都当场被抓。

一个反例:没有保障会怎样

那次 AI 做了个看起来很合理的"搜索优化":它删掉了一段它判定为冗余的代码——270 行并发守卫。

mvn compile 绿油油地通过,因为没有语法错误,被删的是行为不是契约。要不是后面跑测试,这事能一路溜进主干。

270 行,一次改动,静默消失。 这就是"产能 + 没保障"的真实代价。如果当时骨架里就有"提交前必须 verify"这条硬门禁,删的那一刻就该被测试网住。

事后我们复盘了 AI 为什么会删这 270 行。那段代码是一个并发安全守卫——用 synchronized + 双重检查锁,保证缓存重建时只有一个线程执行。AI 判断它"冗余"的逻辑是:这段代码没有直接被任何地方调用(它是被 AOP 织入的),编译器不报错,删了测试也能过(因为并发场景单测覆盖不到)。

AI 的判断在它的视角内完全自洽。但它不知道的是,那 270 行守的是一个只在高压并发下才暴露的不变量——单测跑不出并发,集成测也很覆盖不到,只有线上流量打过来才会出问题。这种「行为类」代码,是 AI 最危险的盲区:它看不见行为,只看得见语法和引用。

这个事故直接催生了三件事:提交门禁从 compile 升到 verify(拦住行为删除)、增加并发场景的集成测试、以及在我们给 AI 的规则文件里加了一条「禁止删除带 synchronized/Lock 的代码块,除非有明确的测试覆盖」。事故 → 规则 → 机器执行,这个闭环是 AI 保障体系的运作方式。

所以保障不是给 AI 上的枷锁,是给整个系统上的保险绳。AI 产出越多,这根绳子越要结实。

架构师的新职责

在 AI 大量写代码的时代,架构师的核心职责悄悄挪了:从"设计能跑的系统"挪到"设计能兜住 AI 的系统"。

你要回答的问题变了。不再是"这个功能怎么实现",而是"这个系统,在 AI 拼命往里塞代码的时候,靠什么不塌"。

答案就在那些以前觉得"枯燥、该做但总排在后面"的东西里:多租户隔离、权限过滤器、契约闸、测试门禁、规范固化。把它们从 “nice to have” 提到"系统骨架"的位置——这就是 AI 时代的架构活儿。

别嫌它们土。越土的机制,AI 越绕不过去。

具体到设计这层保障,有三条原则。

保障要分层,不能只靠一道。任何单一保障都有盲区:契约闸抓跨层断裂,但抓不到行为删除;测试门禁抓行为变更,但抓不到架构越界;ArchUnit 抓架构越界,但抓不到跨层契约。五样东西叠起来,每一道兜住其它道的盲区。少一层,AI 就多一个逃逸口子。这不是过度设计——AI 生成代码的失败模式太多了,一道防线根本兜不住。

保障要机器执行,不能靠人记。AI 不看 wiki,人 review 会疲劳。只有写进 pre-commit hook、写进 ArchUnit 测试、写进 CI 门禁的东西,才会 100% 执行。判断一条保障有没有用,标准是「如果没有人盯着,它还会不会生效」——答「会」的才是真保障,答「不一定」的还是人治。

保障要可验证,不能是黑话。「这个模块要保证租户隔离」是一句废话,ArchUnit 的 DataAccessRulesTest 才是保障。前者靠人理解,后者机器执行。每一条架构纪律,都应该能翻译成一条可运行的测试或脚本。翻译不出来的,说明它还不够具体,继续拆。

保障层怎么和 AI 工作流衔接

保障层不是孤立的,它要嵌进 AI 写代码的日常流程里才有效。我们把它串在三个时机。

写之前:规则前置。每个 Agent 启动时,项目根目录的规则文件会被加载进它的上下文。这些规则不是泛泛的「请遵守编码规范」,而是具体的「新增迁移用秒级时间戳命名」「@Scheduled 不得调 TenantContext」「DELETE 必须带 tenant_id」。AI 还没动笔,就知道这个项目有哪些红线。这比事后 review 抓违规高效得多——预防的成本永远低于修复。

写之中:实时校验。AI 每改一批文件,IDE 层的 hook 会跑一次轻量检查(比如 lint、路由提取),有违规则当场提示。这一层是「即时反馈」,趁 AI 还在当前上下文里,立刻纠正,不用等到提交。

提交前:全量门禁。这就是前面说的契约闸 + 测试 + ArchUnit 全套。AI 觉得「改完了」,但门禁说了算。任何一道红,就打回去改。这一层是「最终防线」,不依赖 AI 自觉,完全靠机器。

三个时机对应三种反馈速度:规则前置是预防(零成本),实时校验是秒级纠正(低成本),提交门禁是分钟级拦截(成本最高但最可靠)。少了任何一个,保障都有缺口——只靠门禁,AI 会反复踩同一个坑浪费算力;只靠规则前置,AI 会「知道但没做到」。

还有一个容易忽视的衔接点:保障规则本身也要版本化、可演进。我们每出一次事故,都会转化成一条新规则(写进 ArchUnit 或加进规则文件),参考事故即规范。保障层不是一次性搭好的静态系统,是随着 AI 犯错而持续加固的活体系。AI 每翻一次车,骨架就多焊一根钢筋。半年下来,我们的 ArchUnit 规则从零条长到三十多条,每一条背后都是一次真实踩坑——这套东西不是设计出来的,是被事故喂出来的。有人觉得写架构测试花时间,但和「AI 引入一个架构违规导致线上事故」的代价比,这点时间不值一提。架构测试是买保险,事故是交保费——前者永远比后者便宜。这笔账算清楚了,你就知道该往哪里投入。说到底,AI 保障不是某个工具或某条规则,是一种「把人的判断焊进系统骨架」的工程习惯——每一条规则都是一次踩坑的固化,焊得越多,AI 能跑偏的空间就越小。这套骨架搭好之后,新人或新 Agent 进来,不用口头叮嘱「别犯这个错」,骨架自己就拦住了——这才是工程化保障比人盯人强的地方——它不依赖人的状态、记忆和责任心,7×24 在那儿守着。
