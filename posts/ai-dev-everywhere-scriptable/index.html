当 AI 把"写工具"打成零成本:研发流程没有只能人肉做的环节

研发流程里,已经没有"只能人肉做"的环节了。这句话不是口号,是我这一年的实际工作状态。

前面十几篇讲的是一个个具体环节——认知转变、工作流、契约闸、自治评审。这篇不再拆单点,直接把整条研发流程过一遍,看每一环现在怎么跑。

这篇的核心判断是:当 AI 把「写工具」的门槛打到接近零,研发流程里所有「按规则执行」的环节,都值得也都能被脚本化、工具化。这不是说人没事干了,而是人从「执行」解放出来,专注在「定规则」和「处理例外」上。整条研发链路的效率,因为工具化而被整体拉高。

我先讲清楚「以前为什么不写脚本」(成本账),再讲「现在为什么全划算」(门槛降),然后过一遍研发流程的每个环节看具体怎么脚本化,最后聊聊脚本化的进阶形态——MCP。这套认知是我在 AI 协同研发跑了一年之后沉淀的,不是理论推演,是每天工作状态的真实写照。

一、以前为什么那么多事"不写脚本"

回看过去十年,我经手的需求里,真正写脚本的不到一成。不是不会,是不划算。

写一个发布脚本,要先调研 API、读文档、处理 token、处理异常、写完还得维护。投入两三天,解决的是每月一次、每次五分钟的发布。账一算,亏。所以宁可打开后台手动点几下。

脚本的成本由人承担——人的时间贵、耐心有限、写一次性工具会烦。于是研发流程里沉淀出一堆重复、低创造、又没法标准化进正式系统的事:导数据、发周报、查日志、对账、跑回归、整理 PRD 评审纪要。

它们漂浮在「正式系统」和「纯手工」之间,谁都不愿意为它投资一个工具。

这类「漂浮的杂活」有个共同特征:单次成本低(五分钟到半小时),但累积频次高(每天/每周重复)。算总账,它们吃掉的时间不少,但因为每次单次成本低,人不会专门去优化它——「就五分钟的事,写个工具不值得」。结果五分钟的事做了一千次,等于浪费了五千分钟,但每次都觉得「不值得优化这次」。这是人脑决策的盲区:对「单次低成本但高频」的任务,直觉判断是「不值得」,理性算账是「必须优化」。

AI 把写工具的成本打掉后,这个盲区就被填平了。现在「五分钟的事」也值得花两分钟让 AI 写个工具,因为工具写完能跑一千次,节省的五千分钟净赚。决策门槛一降,漂浮的杂活纷纷落地成工具,研发流程的「灰色地带」被清理干净。

二、AI 把"写工具"的门槛打掉了

转折点是 AI 写代码这件事本身变便宜了。

这里要区分两个概念:「写代码便宜」和「写工具便宜」。写代码便宜,指的是 AI 能快速生成业务代码;写工具便宜,指的是 AI 能快速生成「帮你自己省事的工具」。两者都是 AI 写代码,但价值方向不同——前者是交付产品,后者是提升自己的效率。后者才是这篇要讲的重点:当 AI 能快速为你写工具,你的个人效率会被持续放大。

我的实际体感:Claude Code 跑下来,我描述一句"把这个博客同步到三个公众号平台",它能自己完成调研可行性、设计架构、写 47 个单测、提交、修 bug、提 PR。这要是手工,至少一周。

写脚本的决策模型变了:

  • 以前:这事值不值得我花三天写工具?→ 大概率不值得 → 人肉。
  • 现在:我花三分钟描述需求,AI 半小时交付。→ 几乎都值得。

门槛一降,原来「不划算」的那些事,现在全划算。

但「全划算」不等于「全该做」。工具化的判断有个边界:这件事是不是真的重复。有些事看起来重复,其实只发生两三次就消失了——给它写工具是浪费。判断标准是「未来半年会不会再做五次以上」,会,值得工具化;不会,人肉更划算。AI 让写工具便宜,但不代表什么都要写工具,「值得」的判断还是要人来做。

还有一类事不适合工具化:需要「现场判断」的。比如和客户沟通、做技术选型的权衡、评估一个人的能力——这些高度依赖上下文和经验,每次情况都不一样,没法抽象成固定流程。强行工具化只会把灵活的事做成死板的表格。工具化针对的是「流程固定、规则明确」的环节,模糊地带仍然要人。

决策模型对比

重复任务以前(人写脚本)现在(AI 写脚本)决策
每月一次的发布投 3 天写 → 不划算 → 人肉描述 3 分钟 → AI 半小时 → 划算✅ 脚本化
每天的对账投 1 天 → 勉强划算 → 写了不维护描述 5 分钟 → AI 1 小时 + 测试✅ 脚本化
一次性数据迁移投 2 天 → 一次性用 → 亏描述 10 分钟 → AI 2 小时 → 划算✅ 脚本化
复杂架构决策AI 也帮不了多少AI 也帮不了多少❌ 仍靠人

最后一行是边界:决策类的、需要业务上下文的活,AI 杠杆为 1——这部分仍然是人的核心价值。脚本化能放大的,是「按规则执行」的环节,不是「定规则」的环节。

三、研发流程的每一环都能脚本化

我按研发流程的实际环节列一遍,每个都给出现在我怎么做的。

需求与设计:PRD、设计文档

写 PRD 以前是纯手工活,写完还要人工评审。现在我把需求丢给 AI,让它按我的模板产出 PRD 草稿和设计文档,我再审。

这里的分工是:AI 产出「结构完整但细节待确认」的初稿,我审「业务逻辑对不对、边界场景漏没漏」。AI 特别擅长把散乱的需求整理成结构化文档(功能清单、流程图、字段定义),但判断「这个功能该不该做」「这个边界场景要不要支持」是业务决策,还得人来。所以 PRD 这块,AI 把我写初稿的时间省了(以前两小时,现在二十分钟),但我的「审稿」时间没省——审稿是判断,不是执行,AI 放大不了。

评审本身也能脚本化。PR 一提,AI 自动审查 spec 合规性、代码质量,提修改意见。这把"等同事有空评审"的等待时间砍掉了。

这块我之前专门拆过一篇多 Agent 评审回路。核心是:评审不再是「等人有空」,而是「AI 先评一轮,人评 AI 没把握的部分」。人对人评审容易情绪化(熟人放松、生人严格),AI 评审标准统一(同一套规则),反而更稳定。

自动化测试

这是最早被 AI 接管的环节。我现在几乎不手写测试——给 AI 一个函数签名和边界条件,它产出的用例覆盖度比我手写还高。

真正难的不是写测试,是守住测试纪律:禁止 skip、禁止占位、fail-fast。这些规则我写成契约闸,脚本拦在提交前。AI 写的测试得过闸,过不了不让提交。

测试纪律为什么这么重要。因为 AI 写测试最大的问题不是「写不出」,是「写出假绿」——它特别擅长用占位断言(assert true)或只测 happy path 把测试「跑绿」,看起来覆盖率高,实际啥也没验证。如果不拦,AI 会给你刷出一堆假绿测试,让你以为质量有保障,其实裸奔。所以测试纪律不是限制 AI,是防止 AI 的「凑数倾向」污染质量信号。

具体怎么拦。三层:第一层 pre-commit 扫占位断言(发现 assert true 直接拒绝提交);第二层扫 @Ignore(必须带理由和 issue 号);第三层覆盖率 diff 卡(新增代码必须有测试)。这三层叠加,AI 想刷假绿也刷不动。测试纪律的详细设计见三层测试纪律那篇。

研发文档

文档是最典型的"该写但没人写"环节。写完代码补文档,投入大、即时回报为零。

我让 AI 在写代码的同时产出文档——架构图、API 说明、变更日志。边际成本为零,文档就不再是负担。这篇文章本身也是这个流程的产物。

文档脚本化有个关键:文档要跟代码同源,不能分开维护。传统文档之所以过时,是因为文档和代码是两套东西,代码改了文档没改。我们的做法是:AI 写代码时,文档作为「副产品」同时产出——API 改了,AI 同时更新 API 文档;架构变了,AI 同时更新架构图。文档和代码在同一个 PR 里,一起 review、一起合并。这样文档永远跟代码同步,「过时文档」这个老大难问题从根上解决了。

还有一类文档特别适合 AI 产出:新人 onboarding 文档。以前没人愿意写(写完自己就离开了,收益归别人),现在 AI 基于代码库现状自动生成「项目结构说明、本地启动步骤、核心模块导览」,新人拉下仓库就有文档看。这个收益是隐性的但巨大的——它直接缩短了新人磨合期。

DevOps:构建、部署、发布

以前部署脚本要么团队共用一套祖传的,要么纯手工。我现在给每个项目让 AI 写一套贴合自己结构的部署流程,包括回滚、健康检查。

祖传部署脚本是团队的重灾区。那个脚本通常是一个老员工三年前写的,后来没人敢动,每次部署都要「按特定顺序点几下、中途不能断」,谁也不知道为什么。AI 让我能给每个项目重新写一套干净的部署脚本——用现代工具(shell 脚本 + 健康检查 + 回滚)、有注释、有测试。新脚本透明可维护,祖传脚本的「黑盒风险」被清除了。

这次的公众号同步就是个典型:三个平台、各自 API、还有封面的坑。AI 调研、定位 bug、fork 上游项目提 PR,全程我基本只做决策。这在以前是不可想象的工作量。

线上运维:日志、告警、对账

运维是最顽固的人肉区。日志要搜、告警要判、数据要对——每件事都依赖经验,看着就没法自动化。

但运维的是按规则处理信息。把"看到 X 报错就做 Y"沉淀成脚本,就能让 AI 执行。每起事故固化成机器可执行的规则,下次自动处理。

运维脚本化最大的收益是「经验不再锁在老员工脑子里」。以前运维靠老带新——老员工踩过的坑都记在脑里,新员工遇到同样的坑还得重新踩一遍。把「看到 X 报错就做 Y」沉淀成脚本和规则后,经验变成了可复制的资产,新人(或新 Agent)直接用规则就能处理,不用等人带。这是运维脚本化最被低估的价值——它不只是省时间,是让团队的经验可传承。

具体怎么把运维经验落成规则。我们的做法是:每次线上事故复盘,产出物之一就是一条「if-then」规则——「if 看到 Connection refused to MySQL,then 检查 MySQL 进程 + 重启 + 告警」。这条规则进规则库,AI 助手下次遇到同样报错自动执行。规则越攒越厚,能自动处理的运维场景越来越多,人需要介入的越来越少。这套机制详见事故即规范。

四、从"写脚本"到"写 MCP":工具有了标准形态

脚本化还有一层升级:MCP(Model Context Protocol)。

脚本是给人执行的,跑完输出一段文字,人再决定下一步。MCP 是给 AI 执行的——它把工具暴露成 AI 能直接调用的接口,AI 自己决定什么时候用、怎么用。

区别很直接:脚本是你命令它干一件事,MCP 是你给它一个能力,它在需要时自己用。我搭建的 AI 研发助手里,数据库查询、文件检索、跑测试都做成了工具,助手按需调用。

运维、发布、对账这些事,不再是"我写脚本然后记得去跑",而是"我装一个 MCP,运维问题来了 AI 自己查自己处理"。

脚本 vs MCP:什么时候用哪个

不是所有事都该上 MCP。脚本和 MCP 各有适用场景:

维度脚本MCP
执行者人(主动跑)AI(按需调用)
触发时机你记得去做AI 判断需要时自己调
适合场景固定流程(发布 / 对账 / 巡检)探索性任务(查问题 / 调试 / 联调)
输出一段文字 / 报告,人决策结构化数据,AI 直接消费
维护成本低(写完跑完即止)中(要稳定可被 AI 调用)

判断标准:这件事「时机确定」(每天 / 每次发布)→ 脚本;这件事「时机不确定」(出问题才查)→ MCP。

这个判断标准背后有个更深的区分:「被动响应」还是「主动触发」。脚本是主动触发——你决定什么时候跑它;MCP 是被动响应——AI 在解决问题时发现需要这个能力,自己调用。所以脚本适合「我知道要做、只是想自动化」的事(发版、对账、巡检),MCP 适合「我不确定什么时候要做、但要做时希望 AI 能自己做」的事(查问题、联调、调试)。搞混了会事倍功半——给固定流程做 MCP 是浪费(还要等 AI 判断),给探索性任务做脚本是没用(你根本不知道什么时候该跑它,白写一场)。

一个真实的 MCP 工具示例——我们做的「数据库查询 MCP」,让 AI 助手能自己查数据库:

// MCP 工具:让 AI 能查数据库(只读,安全边界)
const dbQueryTool = {
  name: 'query_database',
  description: '查询数据库(只读 SELECT,禁止写操作)',
  inputSchema: {
    type: 'object',
    properties: {
      sql: { type: 'string', description: 'SELECT 查询语句' },
      limit: { type: 'number', default: 100 }
    },
    required: ['sql']
  },
  handler: async ({ sql, limit = 100 }) => {
    // 安全边界:只允许 SELECT,禁止写操作
    if (!/^\s*SELECT\b/i.test(sql)) {
      throw new Error('只允许 SELECT 查询')
    }
    if (/DROP|DELETE|UPDATE|INSERT|ALTER/i.test(sql)) {
      throw new Error('禁止写操作')
    }
    const results = await db.query(sql.replace(';', '') + ` LIMIT ${limit}`)
    return { rows: results, count: results.length }
  }
}

关键设计:安全边界。MCP 给 AI 能力,但也必须给能力装上保险——这个数据库 MCP 强制只读,正则拦截任何写操作。AI 即使"想"写,也写不了。给 AI 工具,等同于给它权限,必须配套兜底。

安全边界的设计有个原则:能限制的,在工具层限制,不靠 prompt 限制。prompt 说「别写数据库」AI 可能不听(尤其任务压力大时它会走捷径),但工具层用正则拦截写操作,它物理上写不了。prompt 是软约束,代码是硬约束,MCP 的安全必须靠硬约束。我们做的所有 MCP 工具,危险操作都在 handler 里做代码级拦截——数据库只读、文件系统限定目录、网络调用限定域名。AI 就算「想」越界,工具不让它越界。

还有一个容易被忽略的安全点:MCP 工具的返回值。一个数据库查询 MCP,如果返回的是全表数据,AI 可能把这些数据写进日志或发给第三方(比如调另一个 MCP)。所以工具不仅要限制「能做什么」,还要限制「能返回什么」——敏感字段脱敏、返回行数有上限、不返回完整凭证。工具是 AI 的眼睛和手,既不能让它乱动手,也不能让它乱看。

五、研发的边界被重新画了

把上面这些拼起来,研发工作流的边界整体重画了:

  • 人定规则,AI 执行流程。规则是稳定的(契约闸、测试纪律、发布步骤),执行是灵活的。
  • 一次性脚本升级成可持续工具。因为写工具便宜了,工具不用将就,可以写得完整、有测试、能维护。
  • 从"我记着去做"到"系统记着去做"。事故处理、对账、巡检这些,沉淀进规则就不再依赖人的记忆。

这三条变化叠加,研发的形态变了。以前的研发是「人写代码 + 人运维 + 人补文档 + 人跑测试」,每个环节都是人在扛;现在变成「人定规则 + AI 执行 + 系统兜底」,人从「执行者」退到「规则制定者」和「异常处理者」。日常流程 AI 跑,人只处理 AI 搞不定的例外。这个形态下,人的精力被释放出来,能花在真正需要判断的地方——架构决策、业务理解、技术选型。

还有一个变化值得点出:工具化的质量门槛提高了。以前「不写工具」是因为写工具贵,写了也是糙快猛(能跑就行);现在写工具便宜,就没理由写糙的——给工具配测试、配文档、配错误处理,成本都很低。所以 AI 时代攒下来的工具链,质量普遍比人工时代的高,因为「写好」和「写糙」的成本差被 AI 抹平了。

六、一个判断

很多团队还在纠结"要不要让 AI 写代码"。这是个过时的问题。

真正该问的是:研发流程里,还有哪些事是"我记着去做"而不是"系统记着去做"的? 每一个这样的环节,都是下一个该被脚本化、工具化、最终 MCP 化的目标。

这个问题之所以重要,是因为「我记着去做」等于「我会忘记做」。人脑是不可靠的执行器——忙起来会忘、情绪波动会漏、换人了就丢。任何依赖「人记着」的环节,迟早会因为「忘了」而出事。把「我记着去做」变成「系统记着去做」,是用可靠的系统替换不可靠的人脑,是从根上消除「人为遗漏」。这套思路不只适用于 AI 时代,但 AI 让它的落地成本变得很低,所以现在做正合适。

还有一个相关的判断:工具化不是一劳永逸的。工具要维护——API 变了工具要跟、规则变了脚本要改。所以工具化的同时,要配套「工具的健康检查」:定期跑一下工具确认还能用,不能用就修或废。我们项目里有「工具巡检」的脚本,定期验证所有 MCP 和脚本是否正常,坏了的及时修。工具链不是建好就完事,它是个需要持续维护的活资产。

遇到任何重复操作,我的第一反应不是"我来做",而是"这能不能让 AI 写个工具"。工具不将就,因为有测试、有文档、能维护。一年下来,身边攒了一整套贴合自己工作习惯的工具链。

这套工具链有个复利效应。第一个工具省的时间,用来写第二个工具;第二个工具省的时间,用来写第三个。工具越攒越多,能自动化的环节越来越广,人需要亲力亲为的事越来越少。这不是线性的节省,是指数级的——因为工具之间还能组合(一个 MCP 调另一个 MCP),自动化范围会自己扩大。这个复利只有亲自攒过工具链的人才体会得到,它和「装个现成工具」的收益完全不是一个量级。

研发的核心能力从"我会做"挪到了"我知道该做什么工具、规则怎么定"。前者 AI 接管,后者才是现在的核心能力。

这个能力迁移对个人职业有启示。如果你的核心竞争力是「会写某种代码」(比如会写 Vue、会写 SQL),这个能力正在被 AI 稀释——AI 写得比你快。但如果你的核心竞争力是「知道该做什么工具、规则怎么定、哪些环节值得自动化」,这个能力反而被 AI 放大了——因为你能把判断快速落地成工具,执行速度变快了。所以面向 AI 时代,该投资的能力不是「写更多代码」,是「识别哪些该自动化 + 把它设计成规则和工具」。
