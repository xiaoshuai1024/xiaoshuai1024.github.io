京东大促缓存复盘:10W QPS 扛住了,11W 倒下了

讲一次大促前夕刚做完缓存优化、活动开闸后被现实教做人的事。整个故事一句话:我把缓存放 cache(缓存)这件事做得很扎实,却忘了还有另一类流量叫 attack(攻击),它压根不走缓存。

我那时在京东云做官网架构。大促前的几个月,我主导了一轮缓存改造——把吞吐量从 200 QPS 干到 2 万常态,大促峰值能扛到 10W。结果活动当天流量叠了一波 DDoS 冲到 11W,缓存架构被绕过,连接池打满,服务响应缓慢。最后靠 WAF 清洗和限流把系统拉回来。

这篇复盘整个过程,核心教训一句话:多级缓存解决的是流量效率问题,解决不了流量安全问题。

大促前夕:多级缓存怎么搭的

改造前的官网是单层 SSR,峰值 200 QPS 就顶不住。大促一开闸,服务直接拖地。

“200 QPS 就顶不住"听着夸张,但单层 SSR 的瓶颈很真实:每个请求都要服务端完整渲染 HTML,渲染过程要查数据、拼模板、跑逻辑,全是 CPU 活;而且 SSR 实例的本地资源(CPU、内存)是有限的,扛不住并发渲染。200 QPS 不是网络带宽的极限,是单实例渲染能力的极限。要扛更多,要么加实例(贵),要么减少回源(缓存)。

我搭了一条 6 层多级缓存链路,让流量逐层消化,尽量少回源:

每层各管一段:

  • ① CDN:边缘节点缓存静态资源,兜底静态页。绝大部分请求在这一层消化。
  • ② WAF:页面级缓存 + 流量清洗 + 限流 + 安全策略。当时这层只在大促专项开启,平时不常驻。
  • ③ Nginx:反向代理 + 健康检查,故障节点自动摘除。
  • ④ SSR 实例本地 LRU:渲染好的 HTML 缓在本地,后端全挂时降级兜底。
  • ⑤ Node BFF:接口聚合层缓存,减少后端回源。
  • ⑥ Redis:分布式缓存热点数据,带互斥锁防击穿。

兜底也做了:SSR 渲染的静态页 CI 构建后上传 OSS 并刷新 CDN,后端真全挂时降级展示 CDN 静态页。

这套兜底链路的设计思路是"最坏情况下也能展示点什么”。6 层缓存全挂、后端全挂、Redis 全挂——这种极端情况,网站不能直接白屏。CI 构建时把每个页面渲染成静态 HTML 上传 OSS,CDN 兜底返回这个静态版。用户拿到的至少是个能看的页面(虽然交互功能没了),而不是 500 错误页。降级不是"优雅地失败",是"最坏情况下保住最基本的可用"——这种底线思维,在稳定性工程里和"追求高性能"一样重要。

这套架构跑起来,效果很明显——优化前后的核心指标对比:

指标改造前(单层 SSR)改造后(6 层缓存)
常态吞吐200 QPS 顶死2 万 QPS 常态
大促峰值200 QPS 即拖地10W QPS 稳定
可用性大促必抖99.99%
P0 事故每逢大促清零
回源率接近 100%<5%

改造完那阵子,大促常态 2 万、峰值 10W 都稳,我当时心里挺踏实的。

这套缓存架构的设计有个核心原则:让流量尽量在离用户最近的地方消化。CDN 在边缘节点就把静态资源挡住,绝大部分请求根本到不了源站;到不了 CDN 的(动态请求)进 WAF,页面级缓存再挡一层;真到了 SSR,BFF 缓存、Redis 缓存层层兜底。每一层的设计目标都是"把流量拦在本层,别让它往下一层走"。这套思路对正常流量完美——正常流量的 key 是可预测的,缓存命中率自然高。但它有个我没设计到的盲区:不可缓存的流量,这套架构反而帮它一路放行。 这个盲区,就是后面 11W 翻车的根子。

活动开闸:10W 扛住了,11W 倒下了

大促当天,正常大促流量约 10W QPS 量级,6 层缓存工作正常。

先发现的是监控告警——P99 响应时间从 80ms 飙到 800ms,后端连接池占用率冲到 100%。我们用的是全链路监控,RequestId 串起来一看,卡顿全集中在「缓存未命中 → 回源」这条路径上,正常时段这条路占比不到 5%,此刻占了 40%+。

回头看完整时间线:

时刻事件关键指标
大促开闸流量 10W,系统稳P99 80ms, 连接池 30%
监控告警P99 飙升,连接池打满P99 800ms, 连接池 100%
流量见顶总流量冲到 11W(叠 DDoS)异常流量 ~1W
定性确认是攻击流量绕过缓存回源率 5% → 40%
启动应急WAF 清洗 + 限流—
恢复Socket 逐步释放P99 回落 <100ms

影响范围:官网主站访问卡顿约持续了一段时间,核心交易链路因后端响应缓慢出现排队,部分用户请求超时。降级页(CDN 静态页)兜住了纯浏览用户,但带交互的页面都受影响。

这里有个设计救了场:降级页。我们事先把 SSR 渲染的静态页 CI 构建后上传到 OSS,后端真全挂时降级展示 CDN 静态页。这次后端没全挂(只是响应慢),但纯浏览用户因为 CDN 兜底,体感没那么差;真正受影响的是带交互的页面(下单、查询),这些没法靠静态页降级。降级不是万能的,它只能兜"看"的,兜不了"做"的——但即便如此,它把影响面从"全员卡顿"收窄到"交互用户受影响",已经值得事先做了。

然后监测显示总流量冲到了 11W QPS——比正常大促多了 1W,但这 1W 是叠了一波 DDoS 上来的。

异常流量绕过了业务缓存,直接打到了后端。缓存是为"正常大促流量"设计的,没考虑攻击流量量级。表现很直接:后端连接池被打满,Socket 连接耗尽,服务响应缓慢。

那一刻我意识到一件事:我花了三个月优化的多级缓存,对攻击流量几乎不起任何作用。攻击请求都是构造的,缓存 key 里根本没有,每一层都"未命中"放行,最后全压到后端。缓存架构对正常流量是铜墙铁壁,对恶意流量却像筛子一样透。

恢复:WAF 清洗 + 限流

应急分三步。

WAF 识别异常流量特征做清洗。识别了三类特征:IP 频次(同一 IP 短时间高频请求)、UA(异常 User-Agent)、请求模式(攻击流量特有的请求路径规律)。命中规则的流量直接在 WAF 层拦截。

这三类特征是反复观察攻击流量后总结出来的。正常用户和攻击流量在这三个维度上表现完全不同:正常用户的 IP 请求频次有合理上限(再快也有个头)、UA 是真实浏览器、请求路径分散杂乱;攻击流量的 IP 频次异常高(脚本不怕累)、UA 经常是空的或明显非浏览器的、请求路径高度规律化(脚本按固定模式打)。WAF 清洗的本质,是用"正常流量的统计特征"反过来定义"异常"——不是识别攻击具体是谁,而是识别"不像正常人的"。

配合 WAF 做限流。超出系统承受能力的流量直接拒掉,保护后端连接池不被继续打满。这里的"承受能力"不是拍脑袋的数字,是压测出来的——我们知道后端连接池打满的临界点在哪,限流阈值就设在临界点之下,留出安全余量。没压测就设阈值,要么设太高(限不住,照样打满),要么设太低(正常流量也被误杀)。限流阈值必须基于压测数据,不是凭感觉。

等待恢复。已经耗尽的 Socket 连接逐步释放,服务慢慢恢复正常。

这三步有个执行顺序的讲究:先 WAF 清洗(把攻击流量从源头去掉),再限流(兜住清洗后仍超承受的部分),最后等 Socket 释放(被动恢复)。顺序不能反——如果先限流不清洗,等于给攻击流量也分配了限流配额,正常用户反而更难进来;先清洗再限流,清洗后剩下的主要是正常流量,限流配额留给正常用户,影响面最小。

整个过程没有重启、没有回滚、没有扩容——核心动作就是 WAF 清洗 + 限流,把流量降到系统承受上限以内。

这里有个判断值得讲:为什么当时没扩容?因为扩容解决的是"算力不够",而这次是"连接池被攻击流量打满"。扩容确实能加几个实例,但攻击流量也会跟着打到新实例上,新实例的连接池照样被打满——扩容是对抗正常流量洪峰的手段,对抗不了攻击。真正有效的,是在源头(WAF)把攻击流量识别出来清掉,而不是在后端被动加机器。这个判断让我们没把时间浪费在无效的扩容上,而是直奔 WAF 清洗。事后看,这个决策省了至少半小时。

为什么多级缓存没救场

事后复盘,根因想透了。

多级缓存解决的是流量效率问题。它的设计前提是:大部分请求命中缓存,少数未命中的回源,后端只承受一小部分流量。这套机制对"正常大促流量"完美生效。

但它解决不了流量安全问题。攻击流量的特征是:请求量大、命中率为零、目标是打满后端连接而不是获取数据。对这种流量,缓存的"未命中放行"逻辑反而帮了倒忙——每一层都判定"我这没缓存,放过去",攻击请求一路畅通打到后端。

这里有个反直觉的点:我们花三个月优化的多级缓存,在攻击面前等于没加。缓存层的"未命中就回源"这个设计,对正常流量是合理的(没缓存就去取嘛),但对攻击流量,它变成了"加速通道"——每一层都诚实地执行"我没缓存,放行",结果就是攻击请求被最快速地送到最脆弱的后端。缓存架构的设计假设是"流量是善意的",一旦流量变恶意,这套架构的每个优点都变成了漏洞。 这个教训让我后来在设计任何防御时,都会先问一句:“如果流量是恶意的,这套设计还成立吗?”

10W 和 11W 只差 1W,但性质完全不同。10W 是正常的业务流量,缓存命中率 95% 以上;那多出来的 1W 是攻击,命中率接近 0,全部回源。缓存架构的天花板,取决于流量的可缓存性,而不是 QPS 数字。

沉淀:WAF 清洗和限流从"专项"改成"常态化"

这次事故后,我们做了三件固化:

  • WAF 清洗规则常态化。从"大促专项开启"改成"7×24 常驻策略"。攻击不会挑日子来。
  • 安全策略持续优化。识别新型攻击模式,规则跟着演进。
  • 限流策略常态化。保护后端连接池不被打满,这是底线。

效果很直接:之后的大促再没出过类似问题,P0 级宕机事故完全清零。

这次之后我对"缓存防击穿"这件事的落地也更扎实了。后来在别的项目(多租户 SaaS)里,缓存击穿防护用的是 Spring Cache 的 @Cacheable(sync = true)——同一个 key 并发回源时,只允许一个线程执行方法体,其他线程阻塞等结果。这就是单飞(single-flight)互斥,代码上就一行注解,但底层是 Redis 加锁回源。热点配置(租户小程序配置、首页 Feed)全标了 sync=true,避免热点 key 过期瞬间把后端打穿。击穿防护要写在代码里,不能指望"这次没击穿"的运气。

TTL 这块我们也踩过雪崩的边——不是大批 key 同时过期,是早期所有缓存用一个默认 TTL,热点和冷数据一起 5 分钟过期,某个时段集中失效造成抖动。后来改成分级 TTL:首页商品流 30 秒(频繁改价,短 TTL)、配置类 2-5 分钟(读多写少,长 TTL)、默认 5 分钟兜底。用 TTL 错峰代替统一过期,是雪崩防护最朴实的解法。

限流:Redis Lua 滑动窗口

配合 WAF 清洗的另一道防线是应用层限流。京东那次是在 WAF 层做,后来在多租户项目里我把限流落得更细——用 Redis SortedSet 实现滑动窗口,核心是一个 Lua 原子脚本:

redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', ARGV[2])  -- 清窗口外
local count = redis.call('ZCARD', KEYS[1])                 -- 当前计数
redis.call('ZADD', KEYS[1], ARGV[3], ARGV[3])             -- 记本次
redis.call('EXPIRE', KEYS[1], 60)
return count + 1

这个限流是四维度 AND 判断:全局(防 LLM 账号被打爆,600 RPM)、租户(默认 30 RPM)、租户+角色、用户,任一超限直接返回 429。为什么用 Lua?因为"清窗口→计数→记录"这三步必须原子,不然并发下计数不准。 这种细节 AI 写限流时经常漏——它会给个"用 INCR 计数"的常规答案,但 INCR 没有窗口清理,做不出真正的滑动窗口。

还有个容易被忽略的防线是熔断。连接池被打满时,如果有熔断器(Resilience4j 之类),失败率超阈值就 OPEN,直接快速失败,而不是让请求排队耗尽 Socket。京东那次靠等 Socket 释放恢复,如果有熔断,恢复会更快。熔断的本质是"宁可快速失败一批,也不要让所有请求都卡死"——这个取舍 AI 给不出来,因为它的默认是"重试三次",而重试在连接池打满时只会雪上加霜。

三个容易混的缓存问题

顺带说一句,缓存相关的事故复盘,经常有人把三个概念混成一谈。它们根因不同、解法也不同,这次事故专门拎出来对过:

问题根因现象解法
缓存击穿单个热 key 过期瞬间,大量请求穿透到 DB一个热点失效,后端瞬间压力互斥锁 / 热点永不过期
缓存雪崩大批 key 同时过期,或缓存服务整体挂了后端被海量回源压垮过期时间加随机抖动 / 集群高可用
缓存穿透查根本不存在的数据,缓存永远 miss(攻击常用)每次请求都打到 DB布隆过滤器 / 缓存空值

这次 11W 事故严格说是缓存穿透的一种——攻击流量查的是构造的、缓存里根本没有的数据,所以每一层都 miss,穿透到底。前面 6 层缓存的设计本身没问题,但对穿透这类"不可缓存流量"是无效的,这也是我把 WAF 单独拎出来当成第二道防线的原因。

穿透防护在多租户项目里我们是有取舍的。一般教科书会给两个解法:布隆过滤器、缓存空值。但我们实际上两个都没上——原因是我们禁用了缓存 null(disableCachingNullValues),也没引布隆过滤器。为什么敢这么省?因为这是 SaaS 后端,所有查询都带合法的 tenant_id 校验,构造不存在 key 的成本高、收益低,穿透面小。穿透防护要看业务:对外暴露、能被构造任意 key 的场景(像京东官网),必须布隆+空值双保险;查询都带强校验的内网场景,可以不上。 这正好解释了为什么京东官网那次会被穿透——它对外暴露,可被构造任意路径,缓存又没做穿透防护。

深度展开:多级缓存的失效模型

多级缓存(L1 本地 + L2 Redis + DB)看起来很美,但它的复杂度全在失效上。这次京东大促 11W QPS 倒下,根因就是失效模型没设计好。

单层缓存的失效简单:key 过期或主动删。多级缓存的失效复杂:删了 L2,L1 还在;删了 L1,其他节点的 L1 还在。

我们的失效模型最终落成:L1 短 TTL(30s)自动失效 + L2 主动失效。L1 不主动失效(避免多节点广播的复杂性),靠短 TTL 自然过期;L2 走消息广播,改了就通知所有节点失效 L2。这样 L1 的不一致窗口最多 30 秒,业务可接受。

这个失效模型有个关键取舍:为什么不主动失效 L1? 因为 L1 是各节点本地的,主动失效要广播给所有节点,节点一多,广播本身的成本和复杂度就上来了(谁广播?广播失败怎么办?节点下线了怎么办?)。靠短 TTL 让 L1 自然过期,放弃了"立即一致",换来了实现简单。30 秒的不一致窗口,对官网这种内容展示场景完全可接受——一个商品价格 30 秒内更新到位,用户感知不到。失效策略的选择,本质是在"一致性精度"和"实现复杂度"之间找平衡点,不是越强越好,是够用就好。

后来在多租户项目里,失效策略走的是另一套:因为单实例,不涉及多节点广播,用的是写路径主动双删——改了数据后,先走 CacheManager 的 evict,再直接 redis.delete 兜底(防止 key 前缀不一致导致漏删)。AppCacheInvalidator 这个类就是专门干这个的,类注释写得很直白:“写路径统一调用,保证 Redis 与 DB 一致:失效优先于依赖 TTL。”多节点靠广播,单实例靠双删,方案随架构变——失效没有银弹,得看你的部署形态选。

还有个 key 设计的细节值得讲:缓存 key 要带版本号前缀。我们的租户缓存 key 是 v2:{tenantId},这个 v2 是修订版本号,响应形状变了就递增到 v3,旧缓存自然失效,不用挨个清。用 key 版本号做无侵入的缓存升级,比改一次结构全量清缓存优雅得多。

监控:这次是它先发现的

回头看整个事故,第一个发现问题的不是人,是监控。P99 从 80ms 飙到 800ms,这条曲线的异常是监控告警先报出来的,然后才是人去查。

这说明监控的价值不是事后查日志,而是第一时间发现异常。如果没监控,这次事故可能是用户大规模投诉"系统卡"之后才发现——那时候影响面已经很大了。监控把发现时间从"用户投诉级"提前到"指标异常级",这个时间差,就是事故影响面的差距。

监控配合 RequestId(见可观测性那篇),整个排查路径就清晰了:监控告警 P99 飙升 → 拿 RequestId 去日志里看卡在哪 → 发现卡顿集中在"缓存未命中→回源"路径 → 回源率从 5% 涨到 40% → 定性是攻击流量绕过缓存。这条链路,监控负责"发现问题",RequestId 负责"定位问题",两者缺一,这次事故的处置都会慢很多。

一个判断:稳定性是两条腿走路

很多团队做大促备战,重心全压在缓存架构、扩容、压测上。这些都没错,但它们解决的是流量效率问题——让系统能扛住更大的正常流量。

但大促出事,往往不是正常流量把系统压垮了,是叠加了攻击、爬虫、异常请求这些不可缓存流量,从缓存的盲区绕过去,直接打后端。

所以稳定性保障有两层:缓存架构扛效率,WAF 清洗和限流扛安全。这两层缺一个,大促都可能翻车。这也是我在稳定性监控体系和SSR + CDN 高并发方案里反复强调的:稳定性是分层兜底,谁也替代不了谁。

很多团队备战大促,把预算全砸在缓存和扩容上,觉得"缓存够厚、机器够多就稳了"。这次事故打脸的就是这种想法——缓存再厚、机器再多,防不住专门绕过缓存的攻击流量。稳定性是两条腿走路:一条扛正常流量(缓存+扩容),一条扛异常流量(WAF+限流+熔断)。 砍掉哪条腿,都会在某个时刻摔跤。这次是 11W 攻击摔的,下次可能是个爬虫、一个 bug 触发的重试风暴、或者一个突发的热点 key 击穿——形态不同,但都是"缓存拦不住的异常流量"。
