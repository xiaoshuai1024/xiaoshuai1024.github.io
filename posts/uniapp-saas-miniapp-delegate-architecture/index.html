一份代码怎么变成每个租户的小程序:代开 vs 按租户打包选型

做过 SaaS 多租户 Web 的人，第一反应都是「运行时识别租户就行」。但这套在微信小程序上直接死——manifest.json 里的 appid 在构建期就被烧死进产物，运行时改不了。这篇就讲这个硬约束逼出来的两种解法：按租户打包（构建期注入 AppId）和平台代开（一份模板 + ext_json 注入 tenantId）。

先把结论丢出来:

  • 按租户打包:构建期把租户的 AppId 烧进 manifest.json,一份代码跑 N 次构建,产 N 个小程序包。简单直接,但租户越多构建成本越线性涨。
  • 平台代开:基于微信开放平台第三方能力,一份代码模板给所有租户用,差异完全靠 ext_json 在 commit 时注入 tenantId。租户再多代码也只有一份,这是首选。

代开才是 SaaS 多租户小程序的正解,按租户打包是兜底。下文会展开为什么。

难点:微信小程序的 AppId 是构建期常量

做 SaaS 多租户 Web 的同学,第一反应通常是:“运行时识别租户不就行了?根据请求里的 tenantId 加载不同配置、走不同租户的数据。” 在浏览器里这套没问题,但在微信小程序里直接行不通。

原因藏在微信的约束里:manifest.json / project.config.json 里的 appid 字段,在构建期就被烧死进产物,运行时改不了。微信开发者工具、上传、提审、发布,整套流程都认这个 AppId——它对应的是微信开放平台里一个独立的小程序实体。

也就是说,“一份代码多个租户"在 Web 上成立,在小程序上必须在构建期(或发布期)就把租户的 AppId 注入进去。这是 SaaS 多租户小程序所有架构决策的前提,绕不开。

我们最初的设计也想"运行时识别”,做了两周发现走不通,才老老实实回到构建期注入这条路上。这个弯路本身值得单独说一节,因为很多人都在这里栽。

我们走过的弯路:运行时识别为什么死

最初的想法是用一个通用 AppId(开发者自己的)打底,运行时再根据用户登录的租户去走租户的逻辑。这个思路在"开发期"是成立的——本地用开发者 AppId 调试没问题。

但一上审核就崩了:

  • 微信审核认的是包里烧死的 AppId,你挂羊头卖狗肉,审核驳回。
  • 即使审核过了,用户搜不到这个小程序——它对应的微信实体是开发者自己的 AppId,不是租户的。
  • 业务上要求"每个租户有自己的品牌入口、自己的小程序码、自己的搜索结果",这就要求每个租户在微信侧是一个独立小程序实体。

所以最终只能回到"构建期注入 AppId"这条路。区别只是:为每个租户单独打一份包(模式一),还是用第三方平台代开,一份模板包走遍所有租户(模式二)。

模式一:按租户打包

第一种做法是每个租户打一个包。我用 Vite 插件在 buildStart 改写 manifest.json:

VITE_WX_APPID=wx<租户A的AppId> VITE_TENANT_ID=t_xxx pnpm build:mp-weixin:prod

插件定位 mp-weixin.appid,字符串级替换(要求 appid 以 wx 开头才换),uni 编译时烧进产物。tenantId 走 .env.prod 环境变量,运行时 import.meta.env 读取。

插件最核心的几行大概是:

// vite-plugin-inject-tenant.js
export default function injectTenantPlugin() {
  return {
    name: 'inject-tenant',
    buildStart() {
      const appid = process.env.VITE_WX_APPID
      const tenantId = process.env.VITE_TENANT_ID
      if (!appid || !/^wx[0-9a-f]{16}$/.test(appid)) {
        throw new Error(`非法 AppId: ${appid}`)
      }
      // 改写 manifest.json
      const manifestPath = 'src/manifest.json'
      const m = JSON.parse(fs.readFileSync(manifestPath, 'utf-8'))
      m['mp-weixin'].appid = appid
      fs.writeFileSync(manifestPath, JSON.stringify(m, null, 2))
      console.log(`[inject-tenant] appid=${appid} tenant=${tenantId}`)
    },
  }
}

注意那个 throw new Error——这不是炫技,是踩出来的。早期版本我们没校验 appid 格式,有一次环境变量没注入,跑出来一份 appid=undefined 的包,差点被运营拿去发体验版。从那以后,格式不合法直接让构建失败,这是构建脚本的底线:宁可不产物,也不能产错产物。

一份代码,跑 N 次构建,产 N 个租户的包。简单直接,但租户多了构建成本线性涨:

  • 30 个租户一次发布,意味着 30 次 pnpm build:mp-weixin:prod,每次约 90 秒,合计 45 分钟纯构建时间。
  • 一次发布动作要 45 分钟,意味着没法快速回滚——回滚也是再打一次包。
  • CI 资源消耗线性增长,租户一多,机器费用和等待时间都难受。

模式一的适用场景

不是所有场景都该上代开。按租户打包在这几种情况反而合适:

  • 租户数量少(< 10),且增长缓慢。
  • 租户之间代码差异较大(不只是配置,还有定制代码分支)。
  • 团队没接入微信开放平台第三方能力,不想搞 OpenAPI 编排。

我们早期 5 个租户就是用模式一起步的,简单可控,先跑通业务。直到租户数超过 20,构建时间成了痛点,才切到代开。

模式二:平台代开(第三方平台)

更好的是代开:用微信开放平台的第三方平台能力,一份代码模板给所有租户用,差异靠 ext_json 注入。

这里有个概念要先理清:第三方平台是微信开放平台里的一个角色,它和"小程序"“公众号"是并列的。你的 SaaS 公司注册一个第三方平台,然后让租户授权给你代管他的小程序——之后你就能以平台身份,代他提交代码、提审、发布。

关键在微信 /wxa/commit 的 ext_json 字段:

// 后端组装 ext_json(Java 伪码,关键结构)
Map<String, Object> extJson = Map.of(
    "extAppid", tenant.getWxAppid(),      // 租户的小程序 AppId
    "ext", Map.of(
        "tenantId", tenant.getId(),       // 业务侧租户标识
        "brand", tenant.getBrandName()
    ),
    "window", Map.of()                    // 可覆盖窗口配置
);
// 调微信 OpenAPI:POST /wxa/commit
wxOpenApi.commit(tenant.getWxAppid(), templateId, extJson);

小程序运行时(在租户的小程序里)用 wx.getExtConfigSync() 读出自己的 tenantId,再去后端拿自己的配置:

// 小程序端
const ext = wx.getExtConfigSync()
const tenantId = ext?.ext?.tenantId
if (!tenantId) {
  // 兜底:走默认租户或报错,绝不能"裸跑"
  throw new Error('无法识别租户,请联系平台')
}
wx.setStorageSync('tenantId', tenantId)

这和"按租户打包"的区别就在这:代码包是同一份,差异化完全靠 ext_json + 后端租户配置。租户再多,代码也只有一份。

一次真实的坑:ext_json 没注入,小程序裸跑

代开最隐蔽的坑是:ext_json 没注入,小程序也会照常启动,只是拿不到 tenantId。微信不会替你校验 ext 字段的业务正确性,它只校验格式。

有一次我们后端 commit 时 extJson 序列化错了,ext 字段变成空对象。小程序端 wx.getExtConfigSync() 返回 {},tenantId 是 undefined。结果就是用户打开小程序,看到的是一个"无主"状态:首屏白、接口全 403,但小程序本身没崩。

这种 bug 在测试环境根本测不出——测试租户永远是配好的。只有线上某个租户的 ext_json 真的没注入时才暴露。

修复之后,我们加了三层兜底:

  1. commit 后回查:commit 完立刻 getExtConfig 验证一次,确认微信侧的 ext_json 真的是我们注入的内容。
  2. 小程序端兜底:wx.getExtConfigSync() 拿不到 tenantId 时,不静默继续,直接报错给用户看(配文案"小程序异常,请联系平台”)。
  3. 后端启动校验:服务启动时拉一遍所有租户的小程序 ext 配置,比对数据库,发现漂移立刻告警。

这套兜底的逻辑是:不要假设外部系统(微信)按预期工作,假设它随时可能出岔子。

代开的完整生命周期

代开不是"传个包"就完,是一整条生命周期,后端编排微信 OpenAPI:

每一步都对应一个或多个微信 OpenAPI:

  • 授权:生成预授权 URL,租户扫码授权平台代管他的小程序。对应 createPreauthcode + 拼装授权 URL。
  • 代注册(可选):平台拿企业资料(企业名、统一社会信用代码、法人)替租户注册小程序。适合还没有小程序的租户。
  • commit:把代码模板提交到租户的小程序,ext_json 注入 tenantId。对应 /wxa/commit。
  • 提审:/wxa/submit_audit,进微信审核队列。
  • 发布:/wxa/release,审核通过后上线。
  • 回滚:按版本号查历史模板,重新 commit + 发布。
  • 体验版 / 二维码:设体验版、拉体验二维码,给租户做上线前验收。

每个写操作都有功能开关(FeatureGate)和权限校验,版本号自动递增(1.0.3 → 1.0.4)。

一个真实细节:curl 子进程调微信 OpenAPI

微信集成层后端是用 curl 子进程调微信 OpenAPI 的(不是 Java HTTP 客户端),临时响应落 /tmp 再读回。土,但稳、好调试。

这不是黑历史,是有意识的选择。原因有三:

  1. Java HTTP 客户端偶发 hang 死:微信 OpenAPI 偶尔会有慢响应,Java 的 HttpClient 在某些连接复用场景下会卡住,线程被吃光。换成 curl --max-time 30 子进程,强制的、不可绕过的超时。
  2. 临时响应落盘好排错:出问题时 cat /tmp/wx_commit_xxx.json 一眼看到原始响应,比扒应用日志快得多。
  3. 简单到不会出 bug:curl 是稳定的、单进程的、无状态的。Java HttpClient 那套连接池、SSL、超时配置,反而容易写错。

代价是性能差点(每次 fork 子进程),但微信 OpenAPI 一天也就调几百次,这点开销完全可以接受。“土但稳"在集成层是个优点,不是缺点。

curl 子进程方案还有个意外收获:它让集成层的故障隔离天然变好。Java HttpClient 是进程内的,一旦它卡住,连带把所在线程池甚至整个请求处理链路拖住,一个微信接口慢响应可能引发雪崩。curl 是独立子进程,最坏情况就是那个子进程超时被杀,主进程毫发无损,连带着它的内存占用也随子进程退出而释放。对于「调一个第三方接口」这种边界明确的活,进程隔离比连接池调优更省心。我们后来对接的其他第三方服务,凡是一次性请求响应的,基本都沿用了这套 curl 思路,没再回到 HttpClient。

凭证:三层读取视图

每个租户的微信凭证(AppSecret、商户号、API v3 密钥、证书……)是高敏感数据。同一张表,我做了三层读取视图:

具体规则:

  • C 端匿名:只返回布尔能力位(hasAppSecret、wechatJsapiPayEnabled),绝不返回 AppId / 商户号。前端只需要知道"这个租户开没开通支付”,不需要知道具体商户号。
  • 运营端:返回脱敏标识符(wx1234****abcd),不返回 secret 原文。运营认得出是哪个租户就行,密钥原文运营也不该看。
  • 内部服务:返回明文(仅限内网调用),AppSecret / 证书 / API v3 密钥,用于真正调微信 OpenAPI。

入库侧的三条铁律:

  1. 所有密钥类字段 AES 加密后存储,绝不裸存。
  2. 更新时传 null = 保留旧密文。这是为了运营在管理后台改别的字段时,不用每次都把 AppSecret 重传一遍——重传一遍就多一次明文暴露面。
  3. 脱敏值回传不入库。前端回传的 wx1234****abcd 只是给运营看的展示值,后端只信自己解密出来的原文,不信前端回传。

安全设计落到 SQL 视图层,不靠应用代码记。原因是:应用代码可以漏,数据库视图一旦定义好,谁调谁都被视图过滤一遍,不存在"我忘加脱敏了"的可能。把约束下沉到最接近数据的那层,是处理敏感数据的基本原则。

凭证还有个绕不开的麻烦是轮转。微信的 AppSecret、支付证书都有有效期,到期前必须换。我们的做法是给每个租户维护「当前凭证 + 备用凭证」两条记录,切换时先写备用槽、验证调通、再把当前指针指过去,旧凭证保留一个观察期才删。这套「双槽切换」的好处是轮转中途一旦新凭证有问题,回切是秒级的,不用重新走一遍生成流程。支付证书这种一旦掉线交易全停的凭证,尤其值得这么做——直接覆盖写,轮转失败就没有退路。

租户配置下发

小程序冷启动拉一份"安全子集"配置:模块开关(首页 / 社区 / 交易 / IM 哪些开)、首页楼层、Tab 显隐、落地页。

这里有个反模式要避坑:不要把首页做成一份大 JSON。我们最初就是把所有楼层、所有 Tab、所有开关塞进一个 homeConfig 大对象,结果:

  • 一份配置几 KB,租户一多,带宽和解析时间都吃紧。
  • 改一个楼层要重传整个大对象,没法做增量。
  • 配置加载失败,整个首页全挂(没有部分加载能力)。

后来范式化到 6 张子表:module_toggle、home_floor、tab_config、landing_page、navigation_map、tenant_brand。Service 组装后下发、带 60s 缓存,运营改完主动清缓存(走 Redis pub/sub 通知所有实例)。

不同租户冷启动进不同首页,靠 navigation.homeLanding 映射——比如租户 A 是商城型,首屏落 mall;租户 B 是社区型,首屏落 community。这个映射也是配置,不是代码分支,加新首页类型不用发版。

配置下发的安全边界

“安全子集"这个词是刻意的——不是所有配置都该下发给小程序端。判断标准很简单:

这个字段被前端拿到,即使被篡改,会不会造成数据泄露或越权?

会的话,就不下发,放后端校验。典型的反面例子:权限树。早期我们差点把整个租户的 RBAC 权限树下发给小程序,后来想明白——权限校验必须后端做,前端拿到权限树只是为了"亮起哪些菜单”,那就只下发"哪些菜单可见"的布尔结果,不下发权限树本身。

配置下发还有个容易忽略的点:版本与灰度。运营改了某个租户的首页楼层,如果直接覆盖下发,所有正在用小程序的用户下一秒就看到新配置,体验上很突兀,万一配置改错了就是线上事故。我们给下发接口加了版本号,小程序端用 If-None-Match 的思路带本地版本号上来,后端一致就返回 304 不重传;配置改动先在运营后台预览、再发布到单个租户灰度、确认没问题再推全量。这套机制让配置变更可控、可回滚,而不是改完即生效的赌博。

几条判断

最后把这套架构的判断点收敛一下:

  • 微信小程序的 AppId 是构建期常量,这是 SaaS 多租户小程序所有架构决策的前提,绕不开。所有方案都是在解决"如何在构建期 / 发布期注入正确的 AppId"。
  • 代开 > 按租户打包:一份模板 + ext_json,租户再多代码也只有一份。能用代开就别用按租户打包,除非租户少且代码差异大。
  • ext_json 注入是代开的命门:微信不校验你的业务字段,只校验格式。所以必须自己加"commit 后回查 + 小程序端兜底 + 后端启动校验"三层防御。
  • 凭证安全靠三层读取视图(公开布尔 / 脱敏 / 密文),别在应用层记。把约束下沉到 SQL 视图,谁调谁都被过滤。
  • 配置下发走"安全子集":范式化存储 + 缓存 + 主动失效;权限不下发,只下发"可见性结果"。

SaaS 小程序的架构,是在"微信平台约束"和"多租户灵活"之间找平衡。约束是死的(构建期 AppId),灵活是活的(ext_json + 配置下发),把活的灵活做到极致,把死的约束兜到最稳,这套架构就成立了。

决策清单:你应该选哪种模式

如果你也在做 SaaS 多租户小程序,这里给一份可拷贝的决策表。不需要看完上面所有论述,对号入座就行:

你的情况推荐模式理由
租户 < 10 且增长慢按租户打包代开的 OpenAPI 编排成本不划算,模式一更简单可控
租户 > 20 或会持续涨平台代开一份模板走天下,构建成本 O(1)
租户之间代码差异大(不只是配置)按租户打包代开要求所有租户共用一份模板,代码分支多就退化成模式一
租户都是统一模板,只是品牌 / 配置不同平台代开这是代开的最佳场景
团队没接入微信开放平台第三方资质按租户打包代开要求平台先有第三方资质,这是前置投入
需要快速回滚平台代开回滚 = 换版本号重新 commit,按租户打包要重新构建

一个常见的误判:很多团队一上来就想做代开,觉得"一份模板多酷"。但代开的前置成本不小——注册第三方平台、对接 OpenAPI、编排生命周期、做凭证管理——租户少的时候这套投入回不来本。先用模式一把业务跑通,等租户涨到痛点再切代开,这才是务实的选择。

从按租户打包迁移到代开

如果你已经在用按租户打包,迁到代开要分几步走,不能一刀切:

  1. 先在平台注册第三方资质,跑通代开的完整生命周期(授权、commit、提审、发布),用一个新租户做端到端验证。
  2. 存量租户保持按租户打包,新增租户走代开。两套并存一段时间,观察稳定性。
  3. 存量租户分批迁移:每次迁 20%,迁一批观察一周,确认无问题再继续。迁移动作是"租户授权平台 → commit 模板 → 提审 → 发布",和正常发布流程一致。
  4. 完全切到代开后,保留按租户打包的构建脚本作为应急方案——代开挂了(微信 OpenAPI 故障、平台资质过期)时,至少能退回模式一救场。

保留退路的迁移才是好迁移。我们当时迁移到一半,微信第三方平台 API 突然改了字段格式,代开链路短暂失效,靠保留的按租户打包脚本给最关键的几个大租户做了应急发布,扛过了那段窗口期。

两套模式并存期间,最大的管理成本是「哪个租户现在走哪套」。我们在运营后台给每个租户标了发布模式(打包 / 代开),发布动作前先校验模式与脚本匹配,避免出现「以为走代开、实际跑了一次打包」的乌龙。这种双轨状态越短越好,迁移一旦启动就盯紧进度,别让它长期挂在中间态——长期双轨,意味着两套都要维护、两套都会出问题,收益却只有一个,这笔账怎么算都亏,迁移就该尽快收尾。

关于审核:代开绕不开的坑

代开最容易被低估的坑是微信审核。模式一每个租户的小程序独立审核,模式二虽然代码是同一份模板,但每个租户的提交仍然要独立过审——微信不会因为"这是同一个平台的模板"就批量放过。

实际操作中要踩的坑:

  • 审核驳回一个,可能连锁影响一批:模板改了一处,所有租户下次提交都按新模板审,新模板如果某处不合规,所有租户的提审都会被驳回。
  • 审核期间无法回滚:已经在审核队列里的版本,不能撤回。如果想紧急回滚到旧版本,得等审核结果出来(通过或驳回)才能动。
  • 不同租户的审核速度不一样:有些租户的小程序历史记录干净,审核快;有些因为之前被驳回过,审核会被人工介入,慢很多。

应对策略:每次模板大改,先用一个小租户做"探针提交",确认审核通过后再批量推给其他租户。别一上来就给全部租户 commit + 提审,出了问题全军覆没。

探针提交还有个隐藏好处:它能提前暴露模板里那些「代码没问题但审核不认」的细节。微信审核除了查代码,还看类目资质、运营内容合规,这些在本地和测试环境完全验不出来。我们有一次新模板加了「拼团」功能,代码、接口、支付链路都自测通过,结果探针租户审核被驳回,理由是「该类目需补充增值电信业务许可证」。要不是先拿一个小租户探路,等批量提交后被全数驳回,几十个租户的发布窗口全得往后推。审核这道关,代码层面的自测覆盖不了,只能靠真实提交去探。
