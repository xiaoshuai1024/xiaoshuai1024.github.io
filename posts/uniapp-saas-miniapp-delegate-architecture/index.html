uni-app + SaaS 多租户小程序:从"按租户打包"到"代开"架构

这是「前端 → 全栈 → AI 开发」线上的 全栈阶段。我做的 SaaS 小程序平台,核心难题是"一份代码,怎么变成每个租户自己的小程序"。这篇讲两种模式:按租户打包,和平台代开。

难点:微信小程序的 AppId 是构建期常量

SaaS 多租户的小程序,理想是"一份代码运行时识别租户"。但微信不允许:manifest.json / project.config.json 里的 appid 在构建期烧死,运行时改不了。

所以"一份代码多个租户"在微信小程序上,得在构建期就把租户的 AppId 注入进去。

模式一:按租户打包

第一种做法是每个租户打一个包。我用 Vite 插件在 buildStart 改写 manifest.json:

VITE_WX_APPID=wx<租户A的AppId> VITE_TENANT_ID=t_xxx pnpm build:mp-weixin:prod

插件定位 mp-weixin.appid,字符串级替换(要求 appid 以 wx 开头才换),uni 编译时烧进产物。tenantId 走 .env.prod 环境变量,运行时 import.meta.env 读取。

一份代码,跑 N 次构建,产 N 个租户的包。简单直接,但租户多了构建成本线性涨。

模式二:平台代开(第三方平台)

更好的是代开:用微信开放平台的第三方平台能力,一份代码模板给所有租户用,差异靠 ext_json 注入。

关键在微信 /wxa/commit 的 ext_json 字段:

ext_json = { extAppid, ext: { tenantId }, window: {} }

一份模板代码 + ext_json 注入 tenantId,所有租户共用同一份包。小程序运行时用 wx.getExtConfigSync() 读出自己的 tenantId,再去后端拿自己的配置。

这和"按租户打包"的区别就在这:代码包是同一份,差异化完全靠 ext_json + 后端租户配置。租户再多,代码也只有一份。

代开的完整生命周期

代开不是"传个包"就完,是一整条生命周期,后端编排微信 OpenAPI:

  • 授权:生成预授权 URL,租户扫码授权平台代管他的小程序。
  • 代注册:平台拿企业资料(企业名、统一社会信用代码、法人)替租户注册小程序。
  • commit:把代码模板提交到租户的小程序(ext_json 注入 tenantId)。
  • 提审:/wxa/submit_audit,进微信审核。
  • 发布:/wxa/release,审核通过后上线。
  • 回滚:按版本号查历史模板,重新 commit + 发布。
  • 体验版/二维码:设体验版、拉体验二维码。

每个写操作都有功能开关(FeatureGate)和权限校验,版本号自动递增(1.0.3 → 1.0.4)。

一个真实细节:微信集成层后端是用 curl 子进程调微信 OpenAPI 的(不是 Java HTTP 客户端),临时响应落 /tmp 再读回。土,但稳、好调试。

凭证:三层读取视图

每个租户的微信凭证(AppSecret、商户号、API v3 密钥、证书……)是高敏感数据。同一张表,我做了三层读取视图:

  • C 端匿名:只返回布尔能力位(hasAppSecret、wechatJsapiPayEnabled),绝不返回 AppId/商户号。
  • 运营端:返回脱敏标识符(wx1234****abcd),不返回 secret 原文。
  • 入库:所有密钥类字段 AES 加密;更新时传 null 保留旧密文;脱敏值回传不入库。

安全设计落到 SQL 视图层,不靠应用代码记。

租户配置下发

小程序冷启动拉一份"安全子集"配置:模块开关(首页/社区/交易/IM 哪些开)、首页楼层、Tab 显隐、落地页。首页不是一份大 JSON,是范式化到 6 张子表,Service 组装后下发、带 60s 缓存,运营改完主动清缓存。

不同租户冷启动进不同首页,靠 navigation.homeLanding 映射。

几条判断

  • 微信小程序的 AppId 是构建期常量,这是 SaaS 多租户小程序所有架构决策的前提。
  • 代开 > 按租户打包:一份模板 + ext_json,租户再多代码只有一份。
  • 凭证安全靠三层读取视图(公开布尔/脱敏/密文),别在应用层记。
  • 配置下发走"安全子集",范式化存储 + 缓存 + 主动失效。

SaaS 小程序的架构,本质是在"微信平台约束"和"多租户灵活"之间找平衡。
