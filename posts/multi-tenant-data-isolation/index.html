多租户 SaaS 的数据隔离:从一行漏掉的 tenant_id 说起

多租户数据隔离是我做全栈架构时绕不开的一环——它是 SaaS 系统能不能交付的底线。这篇记录的就是这套隔离体系里最隐蔽的一个漏洞,以及我怎么把它从"靠人盯"改成"层层兜"。

先说一句结论:多租户的安全感,从来不来自"开发都很细心",而来自"假设迟早有人会漏,提前把网织好"。后面所有内容都是为这句话找证据。

一次越权,根因只有一个 JOIN

那次线上反馈很直接:某个客户说"我能看到别人的数据"。多租户系统听到这句话,基本可以判定为越权。顺着链路翻,查到一条排查 SQL,对着前端返回的字段一比,定位到一段 JOIN。

排查路径其实不复杂。先用前端能看到的那条"不该出现"的数据,反查它来自哪张表、哪个接口、哪条 SQL;再把这条 SQL 在测试库里重放,逐段拆 JOIN,看哪一段把别人的数据带了出来。定位到具体那段 SQL,大概花了不到半天——真正花时间的是后面想清楚"为什么会漏"。

根因是:某个 JOIN 关联两张业务表时,只在外层带了 tenant_id 条件,JOIN 的那张表没带。结果租户 A 的查询,把租户 B 的关联行也带了出来。

不是什么精妙的 0day,也不是鉴权被绕过。就是一段 JOIN ... ON a.id = b.aid,后面漏了 AND a.tenant_id = b.tenant_id。

修复很快,改一行加上去就完事。但这件事真正的风险在于:代码能跑、单测能过、功能自测也没问题——平时你压根不会想到去构造"跨租户数据"的用例,它只在真实多租户、真实数据并存的线上才会爆。

行级隔离的标配:为什么选它,代价是什么

先把我们的隔离方案交代清楚,后面才好聊为什么会漏。

业界多租户隔离大体三条路:独立库、独立 schema、行级隔离。我老老实实把它们的代价摆出来,你就能理解我们为什么选行级。

  • 独立库:每个租户一个物理数据库。隔离最干净,但运维成本爆炸——租户一多,实例数跟着涨,备份、监控、迁移都得一个租户一个租户搞;跨租户报表、平台级统计要跨库 join,基本写不下去。除非单租户数据量极大、合规要求极严(比如金融大客户),否则扛不住。
  • 独立 schema:一个库里每个租户一套 schema。比独立库省实例,但表结构变更要做 N 次 DDL,租户越多越痛;而且跨租户查询还是麻烦,得动态拼 schema 名。
  • 行级隔离:所有租户的数据混在一张表,靠 tenant_id 这列区分。运维最省心(一套表结构、一次 DDL),跨租户查询天然支持,扩展租户就是加一行配置。代价就一个:每一条 SQL、每一个 JOIN 都得记得带 tenant_id。

我们这套选的是行级隔离,业务表都带 tenant_id,所有查询、所有 JOIN 都必须带条件。取舍很清楚:用"每条 SQL 都要小心"换"运维和成本可控"。对中小客户为主、租户数量会涨的 SaaS,这笔账划算。

租户身份怎么来?请求头 X-Tenant-Id。请求一进来,过滤器链里有个 TenantFilter,负责把头里的租户标识解析出来,塞进一个请求级的上下文。后面业务代码取租户 id,都从这个上下文拿,不让业务自己再从 header 里抠——抠着抠着就有人抠错。

骨架大概长这样(Java 伪码,思路通用):

public class TenantFilter implements Filter {
    public void doFilter(HttpServletRequest req, HttpServletResponse resp, FilterChain chain) {
        String tenantId = req.getHeader("X-Tenant-Id");
        if (tenantId == null || tenantId.isEmpty()) {
            resp.setStatus(400); return;
        }
        try {
            TenantContext.set(tenantId);       // ThreadLocal / 协程上下文
            chain.doFilter(req, resp);
        } finally {
            TenantContext.clear();             // 必须清,线程池会复用
        }
    }
}
// 业务里:Long tid = TenantContext.get();  不许从 header 抠

为什么不让业务自己从 header 抠?三个理由。其一,过滤器是唯一入口,统一校验、统一打日志、统一兜异常,业务层各抠各的迟早有人漏校验。其二,后续要做横切(埋点、限流按租户),有上下文才能挂。其三,异步线程、定时任务里没有 HTTP 请求,业务直接读 header 根本拿不到,必须靠上下文透传。

思路很清楚:身份从入口注入,贯穿整个请求,业务只管用。问题在于,这套机制默认不会自动帮你把 tenant_id 拼到每条 SQL 上。你写一条原生 SQL 或者一个复杂 JOIN,它就是裸的,漏了就是漏了,框架不替你兜。MyBatis 拦截器可以自动拼,但只对走拦截器的语句有效,手写 SQL、动态拼的、跨库的,该漏还是漏。

“记得带 tenant_id” 为什么靠不住

刚上多租户那会儿,我们的"隔离保障"其实就是一句话:写 SQL 记得带 tenant_id。

现在回头看,这跟"记得关灯"是一个级别的要求——它依赖人,而人会忘。更糟的是,现在代码里很大一部分是 AI 写的,AI 也忘。

我主力用 Claude Code,配合 Cursor、OpenCode。这些工具写 JOIN 写得飞快,ON 条件写得规规矩矩,但 tenant_id 这种"业务约定"它们没有意识主动补,除非你在 prompt 里反复强调、或者在评审里盯着。

这事说穿了也好理解:模型训练语料里,JOIN 的标准写法就是 ON a.id = b.id,它没见过"还得带个 tenant_id"的多租户语境。你不主动在 prompt 里讲清楚"这是多租户系统,每条 SQL 必须带 tenant_id 条件",它就默认你想要的是"能跑通的标准 JOIN"。改一次、漏一次,评审一轮、补一次,下一轮换个 Agent 又重来。靠 prompt 记忆这件事,比靠人记还不稳。

那次越权之后我确认了一个判断:靠人记得,迟早要漏,只是第几个 JOIN 漏的问题。多租户的隔离不能建立在"大家都自觉"上,得建立在"假设有人不自觉,然后层层兜"上。

兜底:契约扫描 + SQL 审查清单 + 评审回路

后来我们做了三层兜底。

第一层,把规则固化成清单。 不是 wiki 上挂个"请注意"就完事,而是一份真正会用的 SQL 审查清单。核心条目我列几条:

  • 每个 JOIN 的 ON 子句,必须显式带 a.tenant_id = b.tenant_id(或等价的租户关联条件)。
  • 子查询、CTE、UNION 的每一段,都要能追溯到外层的 tenant_id 约束。
  • 走 MyBatis 拦截器的可以豁免手拼,但手写原生 SQL、动态拼条件的,逐条过。
  • 涉及删除、更新的语句,tenant_id 条件必须存在,而且必须是等值匹配(防止范围误删)。

评审时拿着清单过,不凭感觉。这一层成本最低,但要真用起来才有效——清单挂在那没人看,等于没有。

第二层,上契约扫描。 写个检查去扫代码里的 SQL/MyBatis 片段,看带 JOIN 的语句有没有都带 tenant_id 条件,扫到可疑的就标出来。它能拦掉一部分最典型的漏带,尤其那种"一眼看上去就少条件"的。

具体怎么扫,我们大致这样:解析 MyBatis 的 XML mapper 和注解 SQL,正则/AST 找出所有含 JOIN 关键字的语句;对每条 JOIN 语句,检查 ON 子句或 WHERE 子句里是否出现 tenant_id 的等值条件;原生 SQL(写在 @Select 或字符串里的)同样处理。扫到没带条件的,直接在 PR 里标红,要求要么补条件、要么在评审里说明为什么不用(极少数跨租户运维 SQL)。这个扫描不追求 100% 准,追求"覆盖最典型的漏带模式",它是个粗筛。

第三层,评审回路。 清单和扫描最终都要落到一个强制的人工(或 Agent)评审环节里,尤其是涉及跨表 JOIN 的改动。这块我们和 AI 评审是串起来的:契约扫描发现疑点,评审 Agent 重点盯这些疑点,人再复核。

回路大概是这样:扫描器把"可疑 JOIN 清单"作为输入喂给评审 Agent,Agent 不是泛泛地看 diff,而是带着"这 3 处 JOIN 没检测到 tenant_id,请逐条确认是漏带还是合理豁免"的具体问题去审;Agent 给出判断(漏带/豁免/存疑),人工再对"存疑"和"豁免"做最终拍板。这样人的注意力被精准地引到风险点上,而不是在一片正常改动里大海捞针。

三层叠起来,不是要保证一个都不漏,而是把"漏的概率"压到尽可能低。单层都不完美,但多层一起,想漏过去得同时骗过三个关卡。

租户配置:小程序 AppId 和微信 Secret

顺带说一个相关的坑:租户级别的配置项怎么管。

以微信小程序的 AppId 和 Secret 为例,一个租户一套。早期 AppId 写在构建产物里,打包时就定死了。这个做法成立的前提是:AppId 不在运行时拉取,它是构建期常量。AppId 本身不算特别敏感,且和租户强绑定、基本不动。

类似的"可以进构建产物"的还有:租户的静态资源 CDN 域名、租户绑定的 logo、一些几乎不变的展示配置。共同点是:不敏感、不随请求变、和租户一对一但基本静态。

但 Secret 不一样。Secret 必须按租户动态解析:哪个租户的请求进来,就用哪个租户的 Secret。这个必须走运行时的配置中心或加密存储,按 tenant_id 去取,绝不能写死。写死意味着要么所有租户共用一个(那多租户还有什么意义),要么每次发布都改代码。

必须走运行时的还有:租户的支付商户号、第三方 API key、数据库连接(如果走 schema 级隔离)、各种回调签名密钥。共同点:敏感、且必须按请求的租户动态取。

区分"什么能进构建产物、什么必须运行时按租户拉",这个边界要划清楚。划不清,要么泄漏,要么发布地狱。

安全感来自"假设会漏"

多租户数据隔离给我的最大判断就一句话:别相信"记得带 tenant_id"这件事。

行级隔离在工程上很简单——加一列、查的时候带上,讲起来谁都懂。难的是在长期的、多人(多 Agent)协作里,保证每一条 SQL、每一个 JOIN 都不漏。而一旦漏一个,就是一次跨租户越权,对 SaaS 来说这是非常伤信任的事故。

所以我们的最终策略是:

  • 架构上尽量让隔离"自动发生":上下文注入、统一取租户 id,能拦截器自动拼的就别让人手写。
  • 流程上假设它一定会有漏的,然后用 SQL 审查清单、契约扫描、评审回路一层一层兜。

下面这张图把"入口注入 + 三层兜底"串起来,你看一眼就明白为什么漏一个 tenant_id 要同时骗过好几关。

多租户的安全感,不来自"我们的开发都很细心",而来自"我们假设迟早有人会漏,然后提前把网织好"。

那次越权之后,我们再没出过跨租户数据的事故。不是因为大家突然变细心了,是因为漏了也有兜的。

你做多租户的时候,是靠"大家都自觉",还是也织了网?评论区聊聊,我想知道别人这套怎么兜的。
