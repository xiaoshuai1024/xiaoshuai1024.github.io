拿一份 JSON 渲个页面发个 demo，这类项目 GitHub 上一抓一把。真正卡人的是后面三件事：能编辑、能扩展、能演进。我主导的低代码平台「鲁班」核心就是一个自研的渲染引擎 + 设计器，这篇把它拆开——schema 怎么设计、渲染器怎么递归、设计器怎么拖拽编辑。

项目归属说明：渲染引擎的设计经验源自大厂时期主导的内部系统"鲁班"，本文代码实现以个人开源项目 OpenLuban（架构同源、代码全新重写，GitHub）为准。

schema:一棵树 + 一个表单值字典

低代码引擎的起点是 schema。我把它设计得很薄,有意压到最小:

interface NodeSchema {
  id: string
  type: string            // 组件类型,如 'LubanButton'
  props?: Record<string, unknown>
  children?: NodeSchema[]
}
interface PageSchema {
  root: NodeSchema
  formState?: Record<string, unknown>   // 表单值绑定容器
}

一个关键决定:表单控件的值不挂在每个节点上,而是统一放在顶层 formState 字典里,以控件的 props.name 为 key。运行时只要一个扁平字典就能驱动整页所有表单字段,提交、校验、回填都简单。

老实讲这步我纠结过。每个节点自己存自己的值,直觉上更「自然」,写 schema 的人也省心。但一落地就发现问题:提交时要遍历整棵树去收集,校验时要在树里找控件,后端回填要把值塞回各自的 props——三个环节都要重新爬树。换成顶层一个字典,提交是 JSON.stringify(formState),校验是对字典跑规则,回填是 Object.assign。一个数据结构,省掉三处脏活。

还有一个更隐蔽的取舍:schema 要薄,但版本治理要从第一天留口子。PageSchema 现在没显式的 version 字段,不是忘了,是早期的 demo 还不值得为它背兼容包袱。但 NodeSchema 的结构我刻意没塞多余东西,留出加 version / compatibility 的余地。一旦真有存量页面要演进,加个字段、走个 migrator 就能接住。第一天就把 schema 塞满,后面想改一处得动全身——这种坑我见太多了。

两种存值方式的代价,我列个表你直观感受下:

环节值挂节点上顶层 formState 字典
提交遍历整棵树收集JSON.stringify(formState)
校验树里找控件、回填路径对字典跑规则
回填把值塞回各自 propsObject.assign
嵌套表单容器要递归上传值name 带命名空间即可

右边整列都压成了一两行,这就是选 formState 的回报。

formState 这个设计还有个延伸价值:它是前后端契约的自然边界。表单提交时,前端给后端的就是 formState 这个扁平字典;后端回填时,塞回来的也是这个字典。前后端不用为了「表单数据怎么传」额外定义一套 DTO,schema 里的 name 就是字段名,formState 就是请求体。这在多端场景(比如同一套表单要在 PC 和小程序各跑一遍)下尤其省事——两端的渲染器不同,但提交和回填的数据结构完全一致,后端不用为多端写多套接口。

不过 formState 也有一个边界要划清:它只适合「扁平为主的表单」。如果表单嵌套很深(比如「订单里有多组商品,每组商品又有多规格」),用 name 带命名空间(如 order.items[0].sku.price)虽然能做,但校验和回填的复杂度会陡增,扁平字典的优势就没了。碰到这种深嵌套,我们的做法是混合:顶层用 formState 管简单字段,复杂嵌套部分用专门的子组件自己管状态,通过事件上抛。别教条地全塞 formState,它解决的是「扁平表单的提交/校验/回填」这个 80% 场景,剩下 20% 的复杂嵌套,该交给组件自己管就交给它。

渲染器:一个自递归组件

渲染器就一件事:吃一棵 NodeSchema 树,递归渲染。它是一个自己 import 自己的 Vue 组件,按节点类型分三条路走:

  • 表单控件(输入框、选择器等):用 v-model 双向绑定到 formState[name],blur 时跑校验。
  • 普通组件:从注册表取出 Vue 组件,v-bind 散开 props,有 children 就递归渲染填进默认 slot。
  • 未知组件:不报错,对其 children 递归透传(容错降级)。

三类节点的判定靠 registry + 一个 isFormField 标记。表单控件在注册 meta 时带上这个标记,渲染器拿到节点先看 type 在不在表单注册表里;在就走 v-model 那条路,不在就走普通组件那条路;两者都不是(注册表里压根没这个 type)就是未知。三个分支,一次 dispatch。

为什么未知组件要容错?因为物料是可扩展的,运行时拿到的 schema 里可能出现一个当前包没注册的 type(比如别人新写的组件、你本地没装)。这时候直接报错整页白屏,代价太大。我的做法是:渲染一个占位 + 把它的 children 透传下去接着渲染。视觉上掉一块,但页面还活着,用户至少能提交。

递归的骨架大致是这样(简化版,真实逻辑还掺了插槽和事件透传):

<template>
  <component
    :is="resolved"
    v-bind="node.props"
  >
    <template v-if="node.children?.length">
      <Renderer v-for="child in node.children" :key="child.id" :node="child" />
    </template>
  </component>
</template>

resolved 是从 registry 里取出来的组件;取不到就 fallback 到一个占位组件。注意 <Renderer> 在自己模板里又用了自己——这就是「自己 import 自己」的落点。Vue 对这种自引用是支持的,只要别把 name 写错。

整棵树的深度就是 schema 的嵌套深度。一个组件能渲染整页,靠的就是递归。深度边界我没做显式保护(没遇到恶意 schema),但理论上要卡一下——无限递归通常意味着 schema 被写坏了,不是业务需求。这种边界值不值得现在写,看你是否开放 schema 编辑入口;鲁班暂时只接内部,先欠着。

递归渲染器还有一个性能细节值得提:大列表的渲染优化。如果一个容器节点有 1000 个 children(比如一个长列表),递归渲染会一次性创建 1000 个组件实例,首屏卡顿明显。我们的做法是在渲染器里对「children 数量超过阈值(比如 50)」的容器,自动套一层虚拟列表组件,只渲染可视区域的那几十个。这个优化对 schema 写法是透明的——写 schema 的人不用关心「我这个列表要不要虚拟化」,渲染器根据 children 数量自己判断。这种「引擎兜底常见性能坑」的做法,让物料开发者不用人人都是性能专家。

递归还有一个隐藏的陷阱:key 的稳定性。递归渲染 children 时用 child.id 作 key,这个 id 必须在 schema 里是稳定且唯一的。如果 schema 生成时 id 用了数组下标(比如 item-0、item-1),排序后 id 全乱,Vue 的 diff 会把整个子树重新渲染,拖拽排序就变成「闪一下重画」。我们要求 id 用稳定的唯一值(比如 nanoid),这条约束写在物料注册的约定里,不遵守的物料排序会有视觉 bug——这种 bug 排查起来极痛苦,因为功能是对的、就是渲染闪一下,根因在 key 不稳定。

运行时 vs 编辑器:同一套组件

我做了两个渲染器:RuntimeRenderer(运行态)和 DesignRenderer(编辑态)。它们用的是同一组 Vue 组件实例——底层组件完全不知道自己被运行态还是编辑态渲染。

编辑态只是在外面多包了一层 wrapper。这层 wrapper 干三件运行态不需要的事:选中描边(点上去出蓝框,告诉用户当前选了谁)、空容器占位(一个空容器在画布上要有个看得见的块,否则用户不知道往哪拖)、drop 监听(接住拖进来的新节点或拖动重排)。底层组件一个字都不用改——它该 v-bind props 还 v-bind,该递归 children 还递归,wrapper 在它外面包了一圈而已。

这样做的好处是实打实的:任何组件只要注册进 registry + 注册 meta,就能同时进运行时和设计器,不用为编辑态单独写一份。我见过一些方案,运行时和编辑器各写一套组件——按钮写两遍、表单写两遍,改一个属性两边都得跟。鲁班没这个负担。

底层组件无感的关键,是它只跟「props + slot」打交道,从不关心外面是谁。运行态外层啥都没有,编辑态外层套了 wrapper,但传进去的 props 一模一样。这就像函数式组件的纯函数性质——同样的输入同样的输出,至于被谁调用,组件管不着。

这套「同一套组件」的设计,在组件升级时价值最大。假设按钮组件要加一个 loading 态,只改一处(底层按钮组件),运行态和编辑态同时获得这个能力。如果是两套组件,得改两遍,还容易改出不一致——运行态有 loading、编辑态没有,设计器里配置时看不到效果,到运行态才暴露。两套组件的方案,这种「两边不一致」的 bug 是常态,因为它依赖人去同步,而人同步必然有遗漏。共用一套,从根上消灭了这个 bug 类型。

不过「同一套组件」也有个边界要守:编辑态的 wrapper 不能往 props 里塞私货。wrapper 要选中描边,正确做法是在外层加一个 div 包蓝框;如果偷懒往组件 props 里塞一个 __designMode: true,组件内部就得判断这个字段——这就破坏了「组件不感知渲染模式」的铁律。wrapper 只能在组件外面加东西,不能改组件收到的 props。这条边界一旦破,「同一套组件」的承诺就失效了,慢慢地组件里全是 if (designMode) 的分支,和写两套没区别。

图里最中间那个浅蓝加粗的「共用 Vue 组件库」是整个设计的心脏——左边运行态、右边编辑态,最后都落到同一组组件实例上。区别只在编辑态多了一层 wrapper,以及属性面板由 meta schema 驱动。

设计器:两套拖拽机制并存

拖拽我用了两套,各管一摊:

  • HTML5 Drag API(面板 → 画布):从物料区拖一个组件类型出来,dataTransfer 带 type,画布或容器接住,触发「新增节点」。
  • SortableJS(画布内重排):绑在顶层 children 这一层,做兄弟节点拖动排序。

为什么不合并成一套?我试过。HTML5 Drag 在画布内排序体验差——拖动时的预览图丑、跨容器移动判定不准;SortableJS 做面板到画布又得自己造一套 dropzone,等于把原生 API 重写一遍。两边都不省心,不如各取所长:原生 API 干「新建」这种一次性动作,SortableJS 干「排序」这种连续交互。

两套机制的分工看这张表更清楚:

维度HTML5 Drag APISortableJS
用在哪面板 → 画布(新增)画布内(重排)
传什么dataTransfer 带 typeDOM 位置
触发动作新建节点改 children 顺序
合并代价排序体验差要自造 dropzone

面板分组(信息类、表单类)是引擎内置的静态配置;容器还有 acceptTypes 白名单——比如表单容器只接受 6 种表单控件,拖别的进去会被拒。白名单的意义不是限制用户,是挡脏数据:一个表单容器里混进个轮播图,schema 是合法的,但运行时就是个事故。这个约束放在编辑器入口,比放在运行时报错友好得多。

acceptTypes 还有个进阶用法值得提:它可以让物料自己声明「我能放什么」。一个「栅格布局」容器,物料开发者可以在 meta 里声明 acceptTypes: ['*'](什么都收);一个「步骤条」容器,可以声明 acceptTypes: ['StepPanel'](只收步骤面板)。这种「物料自治」的方式比引擎写死规则灵活得多——新加一种特殊容器,不用改引擎,在物料 meta 里声明就行。引擎只负责执行白名单校验,规则的制定权下放给物料。这种「引擎 dumb、物料 smart」的分工,贯穿了整个鲁班的设计。

拖拽还有一个容易翻车的场景:跨容器移动时的数据一致性。用户把一个节点从容器 A 拖到容器 B,如果「从 A 删除」和「加到 B」不是原子的,中途出错就会导致节点凭空消失(从 A 删了但没加到 B)。我们的事件设计是 node:move 带完整的 { fromParent, fromIndex, toParent, toIndex },宿主一次性处理这个完整信息,要么全做要么全不做。不拆成「先删后加」两个事件,就是为了避免这种中间态。拖拽看着是个交互细节,数据一致性做不好,用户拖一下丢个组件,这个设计器就没人敢用了。

属性面板:schema as form

属性面板完全由组件元数据(ComponentMeta.propSchema)驱动。常见的属性类型(string / number / boolean / select)只要在 meta 里声明,面板自动渲染对应编辑器。这是「schema as form」范式:加一个可编辑属性不用写 UI,声明一下就行。

举个具体的。给一个按钮加个「按钮风格」属性,在 meta 里这么声明:

propSchema: {
  variant: { type: 'select', label: '按钮风格', options: ['default', 'primary', 'danger'] },
  label:   { type: 'string', label: '按钮文案' }
}

面板上就自动出现一个下拉选风格、一个输入框填文案。改需求要加个 size?再声明一行 size: { type: 'select', ... },UI 自己长出来,组件代码不用动。这套范式把「加字段」的成本从「写一个表单 + 接事件 + 存值」压到「写一行声明」,扩展性就藏在这。

支持的类型目前就四种,够用了:

  • string:文本输入框。
  • number:数字输入,带 min/max 校验。
  • boolean:开关。
  • select:下拉,options 数组声明可选项。

要不要支持更花哨的(颜色选择器、图标选择器、富文本)?我倾向于不进核心。这些是业务定制,放进 propSchema 反而把引擎和具体业务绑死。更好的做法是留一个 custom 类型的口子,让业务自己注册渲染器——引擎只管调度,不管长什么样。这跟前面「引擎不持状态」是一个思路:引擎越 dumb,越能复用。

custom 类型这个口子还有个设计要点:它的渲染器注册要和物料注册解耦。物料注册的是「这个组件是什么」,custom 渲染器注册的是「这种属性怎么编辑」,两件事的生命周期不同——一个物料可能用内置的 select 编辑器就够了,另一个物料才需要 custom 的颜色选择器。如果 custom 渲染器跟物料绑死,换一个物料就得重写编辑器,扩展性就没了。我们的做法是 custom 渲染器单独注册,物料在 propSchema 里只声明 type: 'custom', renderer: 'colorPicker',引擎按 renderer 名去查 custom 注册表。物料和编辑器各注册各的,通过名字松耦合。

属性面板还有一个容易被忽略的体验细节:属性变更要实时反映到画布。用户在属性面板改了按钮文案,画布上的按钮应该立刻变,不用点「保存」。这要求属性面板和画布共用同一份 schema——属性面板改 schema,画布的渲染器收到新 schema 重新渲染。听起来理所当然,但如果属性面板自己维护一份本地状态、提交时才同步,用户改了半天看不到效果,会以为没改成功又改一遍。实时反映是属性面板的基本体验底线,实现上就是别让属性面板持有独立状态,它改的就是 schema 本身。

物料体系:两个注册表

组件有两个注册表:运行时 registry(Vue 组件实例)和设计时 meta(组件元数据)。第三方扩展就两个 API:registerComponent + registerComponentMeta。注册完,组件自动进运行时和设计器。

为什么是两个而不是一个?因为运行时要的是「怎么渲染」(Vue 组件),设计器要的是「怎么编辑、能放什么」(元数据)。两件事关注点完全不同,塞一个对象里会让两边都耦合。分开后,运行时打包可以不带 meta(瘦身),设计器可以注册一个占位组件先调试 meta(物料先行)。

元数据里带 category、label、propSchema、defaultProps、events、isContainer、acceptTypes,把一个组件「怎么渲染、怎么编辑、能放什么」一次声明清楚。

几条判断

整篇捋下来,真正撑起这套引擎的就这几条判断。我按重要性排了序:

  • schema 要薄,够用就行;版本治理要从第一天留口子。
  • 运行时和编辑器共用一套组件,靠 wrapper 区分,别写两份。
  • 属性面板用 schema 驱动,加字段不出 UI。
  • 引擎不持有 schema 状态,只渲染 + 发事件,变更交给宿主——这样引擎才通用。

最后那条是引擎能不能复用的命门。如果引擎自己 hold 住 schema、自己改、自己存,那它就跟宿主绑死了——换到另一个业务、另一个编辑器,得把这套状态搬过去。鲁班的引擎只做两件事:给我 schema,我给你渲染;用户在画布上动了什么,我发事件出去,怎么改 schema 是你的事。宿主是 Vue 的、React 的、甚至后端的,引擎都不关心。

举个事件流的例子。用户在设计器里拖了一个按钮进画布,引擎不会自己去改 schema 树,而是发一个 node:create 事件出去,带上 { parentId, type, index }。宿主接到事件,按自己的方式(可能是 immer 改 state,可能是发到后端再回填)更新 schema,再把新的 schema 喂回引擎。引擎从头到尾手里没有「可变的 schema」,它只有「当前 props 里那一份」。这种单向数据流,在 React 里见过的人一眼就懂,好处是引擎跟谁都能配。

把这条单向数据流画出来,「谁持状态、谁改状态、谁只渲染」就清楚了:

这张图里要特别注意的是右边那条「组件库不知自己被谁渲染」。这是引擎能 dumb 的前提——如果组件内部偷偷做了状态判断(比如 if (this.$designMode) 自己改行为),它就不再是纯组件了,引擎的「单向数据流」就被破坏,宿主换一套就跑不通。我们立了条铁律:组件只能跟 props 和 slot 打交道,绝不允许感知「我是被运行态还是编辑态渲染」。这条纪律不立,组件迟早会长出对环境的隐式依赖,引擎的可复用性就从这里漏掉。

「引擎不持状态」还有个连带的好处:引擎的可测试性极高。因为引擎是纯函数式的——给定 schema,产出确定的渲染;给定操作,产出确定的事件。测试时不用 mock 任何状态,传一个 schema 进去,断言产出的 DOM 或事件就行。如果引擎自己持有可变状态,测试要先构造状态、操作、再断言状态变了,复杂度立刻上来。dumb 引擎的测试也 dumb,这条看似不起眼,对长期维护帮助极大。

还欠什么账

诚实交代几个我没做、但知道该做的:

  • schema 校验:目前靠 TypeScript 类型 + 运行时容错兜底,没有正式的 schema 校验层。存量大了之后,这一层迟早要补。
  • 撤销/重做:宿主层的 state 如果用 immer + 历史栈能做,但引擎本身不掺和。这块目前依赖宿主实现。
  • 多端适配:现在只跑 Vue,React/小程序要不要进,取决于业务要不要扩张。引擎设计上没堵死,但也没现成实现。

这几条不是没意识到,是早期 demo 阶段不值得为它们投入。低代码引擎是个长期活,第一版能把「渲染 + 编辑 + 扩展」三件事跑通,后面才谈得上「演进」。

这几条欠账里,撤销/重做最容易被「先欠着」拖成大坑。早期 schema 简单,用户改几下就手动重来,没撤销也能忍;等页面复杂了、嵌套深了,用户改了一堆想退回去,发现没有撤销,只能逐个手动恢复——体验崩塌。撤销/重做最好从第一天就把数据结构留好(宿主 state 用 immer + 历史栈),哪怕第一版不实现 UI,数据结构先按「可追溯」的方式存。等 schema 复杂了再补撤销,往往要重构整个 state 管理,代价比一开始预留大得多。这是「欠账」里最该尽早打地基的一条。

低代码引擎的难点不在「能渲染」,在「能编辑、能扩展、能演进」——这三件事才是真正卡人的地方。
