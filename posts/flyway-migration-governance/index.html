某天早上我接到一个告警:某个服务启动不起来。日志翻到最后,就一行 Flyway 报错——迁移版本号对不上。

登上去一看,前一天两个人各自合了代码,各自加了一个 V12__xxx.sql。版本号撞车,Flyway 直接罢工,服务起不来。

排查过程:从"服务挂了"到"找到撞车"

这次事故虽然定位不难,但排查过程值得讲——因为典型的"启动失败"排查路径就是这样:

第一反应:看日志最后几行。 日志末尾就一行红字:Validate failed: Migration version mismatch for version "12",直接告诉你 Flyway 校验失败。这一步省了至少 10 分钟。

第二反应:grep V12 找撞车文件。 ls db/migration/ | grep V12 一跑,出来两个:V12__add_user_field.sql 和 V12__add_order_index.sql。撞车确认。

第三反应:看 git log 倒查谁合的。 git log --oneline -- db/migration/V12* 发现两个文件分别在两次 PR 里合入,review 时都没注意到对方用了同一个号——因为 review 时只看自己那个 PR 的 diff,看不到另一个 PR 已经合了同名号。

根因复盘(5 Whys):

  • Why1:为什么服务起不来?因为 Flyway 校验失败,版本号不唯一。
  • Why2:为什么版本号不唯一?两个人各自用了 V12,合并时没有自动检测。
  • Why3:为什么合并时没检测?因为 CI 只跑单测,没做迁移文件唯一性校验。
  • Why4:为什么没做唯一性校验?团队觉得"靠约定就行",没把它当工程问题。
  • Why5:为什么"靠约定"不可靠?因为多人协作时,约定无法强制——每个 PR 看自己的 diff,看不到全局,撞车必然发生。

根因清晰:不是谁粗心,是治理缺位——把本该机器化的协调问题丢给了人。

5 Whys 走到第五层,结论就清楚了:这不是"下次细心点"能解决的事故。两个人都在自己 PR 的视野里做事,谁都没错,但合起来就撞车。只要协调问题还压在人身上,撞车就是迟早的事,区别只是这周撞还是下周撞。 真正的修复不是叮嘱大家"小心版本号",而是把唯一性校验做成 CI 的硬门禁、把版本号换成不需要协调的时间戳——让机制替人把这道关守住。

事故的本质:不是粗心,是治理缺位

排查倒不复杂,改个号、重新合一下就好了。但事后我盯着那两个 V12 想了很久——这事的本质不是"谁粗心",而是治理缺位。

数据库迁移在多人协作下,如果不立规矩,迟早撞车。而且撞车的代价不是"改个号",是整个服务起不来。Flyway 在启动时校验 schema 一致性,版本号对不上直接抛异常,服务进程就停在那儿。生产环境要是赶上这个,影响面是全量的——不是某个接口慢,是整个服务不可用。

更要命的是这种事故的隐蔽性。本地开发时,你合并的是自己的分支,看不到别人的 V12;CI 如果只跑单测不校验迁移唯一性,撞车照样能过。等到部署那一刻才暴露,这时已经是生产事故了。所以版本号治理这种"平时看不见、出事就是大事"的东西,必须前置到本地构建阶段拦住,不能让它有机会走到部署。

下面是我们团队在 Flyway 上踩过坑后,最终固化下来的几条规矩。这些规矩看着零散,但它们针对的是同一类问题的不同侧面:怎么让数据库迁移在多人协作下不失控。版本号唯一是基础线,幂等 SQL 是 out-of-order 的前提,repair 自愈是脏数据的兜底,环境分流是 dev/prod 取舍的体现。每一条都是"出了事才补上"的,但合起来就是一套能扛住多人长期协作的迁移治理体系。

版本号:从号段分配到秒级时间戳

最早我们试过"号段分配"——张三负责 V100 到 V199,李四负责 V200 到 V299。想法挺好,各管一段。

但实际跑下来全是问题:

  • 号段会跨、会重;
  • 新人来了不知道找谁要号;
  • 合并的时候还是会有人偷偷用了别人的号段。

结论:号段分配是反工程化的,它把一个本该自动化的协调问题,丢给了人。 号段方案看着有序,实际上每一条"优点"都站不住:号段会跨(谁记着边界?),新人不知道找谁要号(谁负责分配?),合并时偷用别人号段(怎么强制?)。每个问题都得靠一条额外的人肉流程去补,补到最后这套"分配机制"比要解决的问题还复杂。

我们的做法:秒级时间戳

后来全换成秒级时间戳:V{YYYYMMDDHHmmss},比如 V20260611185900__create_merchant_frozen_ledger.sql。谁先建脚本,谁的时间戳就是谁。这是项目切到秒级时间戳格式后最早的一批文件之一,文件名就是建表那一刻的时间戳。

两个人同一天、同一分钟、同一秒撞上的概率,比号段撞车低太多了。弃用号段那一刻,工程上反而干净了——少一套要维护的"分配机制",代码和流程里的东西越少越省心。

时间戳方案还有个连带好处:文件名自带时间信息,排查时看一眼就知道这个迁移是哪天加的,配合 git log 能快速还原变更脉络。号段方案做不到——V2067 这个数字本身不携带任何语义,你得去查才知道它是什么时候、为什么加的。文件名长一点是唯一的代价,换来的是撞车概率归零 + 自带时间语义,这笔账很划算。

mvn validate 阶段校验唯一性

光靠人自觉是不够的。我们写了个 Python 脚本,挂在 mvn validate 阶段,扫一遍所有迁移文件,校验版本号是否唯一。不唯一直接 fail 构建。

这个脚本跑到现在,真实校验过的迁移有 233 个,版本号全唯一。不是团队运气好不撞车,是脚本一直在那儿守着——谁撞了在本地构建就过不去,根本到不了 CI。

这 233 个文件里三种命名方案是并存的:最早一批是顺序号(V1000__、V2067__),后来换成日期+序号(V202403831__),再后来才统一到秒级时间戳(V20260611185900__)。历史文件没改名,只是在 flyway_schema_history 表里 UPDATE version 迁过去。这套演进路径本身就是"号段会撞 → 日期会撞 → 秒级时间戳几乎不可能撞"的活证据。

唯一性校验脚本骨架

脚本不复杂,核心就是扫目录 + 集合判重。真实脚本叫 verify_flyway_migration_versions.py,挂在 mvn validate 阶段。它有几个写出来才知道的细节:

# backend/scripts/verify_flyway_migration_versions.py(简化骨架)
# docstring 写明:兼容 Python 2.7 和 3.x,因为有的环境 `python` 还指向 2.7
MIGRATION_DIR = os.path.join(BACKEND_ROOT, "src", "main", "resources", "db", "migration")

def migration_version(filename):
    if not filename.startswith("V") or "__" not in filename:
        return None  # 不以 V 开头、没有 __ 分隔的,不算版本迁移(跳过 R__ 可重复迁移)
    return filename[1:].split("__", 1)[0]

报错格式是给人看的,一眼定位撞车:

ERROR: duplicate Flyway migration version (multiple files, same version token):
  '12' -> ['V12__add_user_field.sql', 'V12__add_order_index.sql']

挂在 pom.xml 的 validate 阶段:

<plugin>
  <groupId>org.codehaus.mojo</groupId>
  <artifactId>exec-maven-plugin</artifactId>
  <executions>
    <execution>
      <id>verify-flyway-unique-versions</id>
      <phase>validate</phase>
      <goals><goal>exec</goal></goals>
      <configuration>
        <executable>${kd.flyway.version.check.python}</executable>
        <arguments>
          <argument>scripts/verify_flyway_migration_versions.py</argument>
        </arguments>
      </configuration>
    </execution>
  </executions>
</plugin>

这里有个写出来才踩得到的坑:可执行文件名特意没用 flyway. 前缀,而是 kd.flyway.version.check.python。原因是 Flyway Maven 插件会把 flyway.* 前缀的属性当成自己的配置项,用了就报 Unknown property。绕开前缀是最省事的解法。pom 里还为 Windows 单独写了个 profile——因为 Windows 商店版的 python 占位会触发 error 9009,得把 executable 指到真正的 python.exe。跨平台兜底本身就是一坨看不见的工程量。

为什么必须前置到 validate 阶段

版本号冲突这种事,一旦让它进到生产,服务启动直接失败,影响面是全量的。

所以这种东西必须前置到最早:validate 阶段,本地一跑就知道。

这种"不出事没人感谢、出事就是大事故“的基础设施,就得靠自动化兜,不能靠人。

squash 是个坑:只能在本地

Flyway 有个 squash,把一堆历史迁移压成一个新的基线。听起来很爽——迁移文件越积越多,看着烦。

但我们定的规矩是:squash 只能在本地开发库做,生产绝对禁止。

原因:这个本地 squash 脚本会 DROP DATABASE

我们自研的本地 squash 脚本 flyway-squash-local.sh,实现就是 DROP DATABASE 然后用新基线重建。脚本头写得明明白白,运行时还会打红色横幅:

⚠️  仅限本地开发库使用！会 DROP DATABASE 清空全部数据！

本地开发库你随便重建无所谓,生产库谁敢 DROP?

我在本地 squash 过几次,基线确实清爽了。但每次都想——这东西要是哪天有人手抖跑到生产上,那就不是"清爽”,是"清场“了。

所以这条规矩我们写得很死,新人入职第一件事就是讲清楚。

schema_history corruption 和 repair:从应急命令到开机自愈

Flyway 有张 flyway_schema_history 表,记录哪些迁移跑过了。我们踩过——这张表 corruption 了,状态对不上,服务又起不来。

schema_history 为什么会脏,常见的几种情况:一次迁移执行到一半被打断(进程被 kill、网络断了),表里留了一条 success=0 的半成品记录;号段迁移时某个 version 被反复 DELETE 再 INSERT,留下了重复的 DELETE 行;还有人误用了已经被迁出的旧 version 号,表里对不上。这些问题单独看都不复杂,但它们共同的特征是:本地开发库自愈能力强(大不了重建),生产库动不得。所以在生产上,我们不能指望"出事才想起来怎么修”,得把修复能力沉淀成开机就跑的自愈逻辑。

最开始我们对 repair 的认知就是"出事才跑的应急命令"。后来踩多了发现,光靠人记得去跑 repair 根本兜不住,于是把它做进了开机自愈:写了一个 FlywayMigrationStrategy Bean,在每次 migrate() 之前自动 repair()。

@Bean
public FlywayMigrationStrategy flywayMigrationStrategy(FlywayRepairProperties props, DataSource dataSource) {
    return flyway -> {
        JdbcTemplate jdbc = new JdbcTemplate(dataSource);
        if (props.isRepairBeforeMigrate()) {
            repairWithDuplicateDeleteFallback(jdbc, flyway);   // migrate 前先 repair
            removeDuplicateFlywayDeleteRows(jdbc);              // repair 可能再插重复 DELETE 行,收敛一次
        }
        migrateWithDuplicateDeleteFallback(jdbc, flyway, props);
    };
}

这里有几个数字能说明这事踩过多深的坑:MIGRATE_DEDUPE_MAX_ATTEMPTS = 6(migrate 最多重试 6 次),去重循环 for (int round = 0; round < 32; round++)——最多 32 轮。为什么要这么多轮?因为单次 repair 可能分批往 flyway_schema_history 写 DELETE 行,一批删不干净,得反复收敛。这种魔数背后全是真实事故的形状。

它到底在拦什么:两类报错原文

这个自愈逻辑判断两类异常,对应两种生产故障:

// 号段 gap:库上已经应用了更高 version,低号段脚本还没跑(并行开发常见)
return msg.contains("Detected resolved migration not applied to database")
    || msg.contains("resolved migration not applied");

// schema history 损坏:同一个 version 出现多条 DELETE 记录
return msg.contains("multiple delete entries") || msg.contains("Corrupted schema history");

对应日志里会出现的两行红字——Corrupted schema history: multiple delete entries for version 51 和 Detected resolved migration not applied to database。这两行背的是两种不同的锅:前者是表脏了,后者是号段对不上。自愈逻辑会按报错类型分流,不是一刀切 repair。

生产里的"降级告警"而不是"直接卡死"

号段 gap 这类,生产上我们做了个取舍:校验失败不直接拖垮服务启动,而是降级告警,让人工在维护窗口补跑。

if (props.isContinueStartupOnGapValidationFailure() && isPendingLowerVersionValidation(e)) {
    log.warn("Flyway validate: pending lower-version migrations (segment gap); "
           + "skipping migrate and continuing startup. Schedule out-of-order migrate in a maintenance window.");
    return;   // 生产不卡死,只告警,让人工补跑
}

这条是 application-prod.yml 里的 continue-startup-on-gap-validation-failure 开关控制的。有些校验失败该让服务挂,有些该让它带着告警继续跑——这个区分是事故喂出来的判断,不是一开始就有的。

repair 操作手册:不只会跑,还要会查

光有自愈代码不够,还得有手册,因为有些情况自愈兜不住,得人来。我们维护了一份 flyway-repair-schema-history-corruption.md,第一步永远是只读排查,先看清楚表里到底怎么脏的,而不是上来就 repair:

SELECT installed_rank, version, description, type, script, checksum, installed_on, success
FROM flyway_schema_history
WHERE version = '51'
ORDER BY installed_rank;

手册里还有一个真实案例备忘:某个迁移文件历史上误用 V59,被迁出后在表里留了一条 success=0 的失败记录,靠删掉那条失败行让 Flyway 重放(前提是脚本幂等)。光 repair 相关的生产脚本、工具、单测就拆成七八个文件——这本身就能说明"我们在这上头栽过多少次"。

迁移 SQL 怎么写:幂等是 out-of-order 的前提

前面讲了版本号治理,但迁移文件本身的写法也是治理的一部分。我们开了 out-of-order: true(允许乱序补跑),这就逼着迁移脚本必须幂等——同一个脚本跑两次结果得一样,否则乱序重跑就炸。

MySQL 没有 PostgreSQL 那种 ADD COLUMN IF NOT EXISTS,所以加列得用 INFORMATION_SCHEMA + PREPARE 套路自己守卫:

SET @preparedStatement = (SELECT IF(
  (SELECT COUNT(*) FROM INFORMATION_SCHEMA.COLUMNS
   WHERE TABLE_SCHEMA = @dbname AND TABLE_NAME = 'tenant_im_message'
     AND COLUMN_NAME = 'mentioned_user_ids') > 0,
  'SELECT 1',
  'ALTER TABLE tenant_im_message ADD COLUMN mentioned_user_ids JSON DEFAULT NULL ...'
));
PREPARE alterIfNotExists FROM @preparedStatement;
EXECUTE alterIfNotExists;
DEALLOCATE PREPARE alterIfNotExists;

加索引更重,用存储过程 + information_schema.STATISTICS 守卫。V20260613000000__add_concurrency_indexes.sql 那次并发优化,一口气给 6 张表加了覆盖索引,每个索引都这么包一层:

DROP PROCEDURE IF EXISTS sp_add_idx_mps_shop_listing;
DELIMITER ;;
CREATE PROCEDURE sp_add_idx_mps_shop_listing()
BEGIN
    DECLARE idx_count INT;
    SELECT COUNT(1) INTO idx_count FROM information_schema.STATISTICS
    WHERE TABLE_SCHEMA = DATABASE() AND TABLE_NAME = 'merch_product_spu'
      AND INDEX_NAME = 'idx_mps_shop_listing';
    IF idx_count = 0 THEN
        ALTER TABLE merch_product_spu
            ADD INDEX `idx_mps_shop_listing` (`shop_id`, `delete_time`, `shelf_status`, `library_role`);
    END IF;
END;;
DELIMITER ;
CALL sp_add_idx_mps_shop_listing();
DROP PROCEDURE IF EXISTS sp_add_idx_mps_shop_listing();

这套写法看着啰嗦,但它是 out-of-order 能安全开起来的前提。开了乱序补跑却不写幂等 SQL,等于把雷埋得更深。 文件头注释还把每个索引覆盖的查询模式写出来(WHERE shop_id = ? AND delete_time = 0 AND shelf_status = 'ON_SALE'),让后来的人一看就知道这索引是为哪条查询加的。

Flyway 配置:dev 关校验、prod 开校验

Flyway 的 validate-on-migrate 在不同环境我们配得不一样:

# application.yml(默认/dev)
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
    validate-on-migrate: false   # dev 关掉,本地号段并行开发不被卡
    out-of-order: true           # 允许乱序补跑

# application-prod.yml
spring:
  flyway:
    validate-on-migrate: ${SPRING_FLYWAY_VALIDATE_ON_MIGRATE:true}  # 生产开,由环境变量控制
    out-of-order: true

dev 关校验是因为本地开发时几个人各跑各的号段,严格校验会动不动就拦你;prod 开校验是因为生产绝不能带着不一致的 schema 跑起来。同一个开关,dev 和 prod 取舍相反,靠环境变量切,这比"统一开"或"统一关"都更贴近真实。

还有一种 dev 独有的纯内存启动方式(application-h2local.yml),直接 flyway.enabled: false,连 MySQL 都不连,用 H2 内存库跑测试。这种 profile 下 Flyway 整个关掉,适合 CI 里那些不需要真实数据库的快速单测。环境分得越细,每个环境才能配得越贴。

号段并行开发:gap 不是错,是常态

前面说号段分配反工程化,但有个现实矛盾:多人协作时,号段并行几乎不可避免——张三在 V20260612 那批迁移里干活,李四同时在 V20260613 那批,谁先合谁后合不一定。这就产生了一个 Flyway 默认会报错的情况:库上已经应用了更高 version,但低 version 的脚本还没跑(gap)。

默认 Flyway 遇到 gap 会拒绝启动,这在生产是个保护;但在并行开发里,这就是天天卡你。我们的解法是 prod 开 out-of-order: true(允许补跑低版本迁移)配合前面那个 continue-startup-on-gap-validation-failure 降级开关——不是把 gap 当错误,是把它当成"待补齐"的正常状态,让服务带着告警继续跑,等维护窗口人工补跑。

这个判断的分寸很关键:gap 在 dev 是常态、在 prod 是隐患。同一个现象,环境不同,处置完全不同。一刀切要么天天卡开发、要么生产裸奔,都不对。

迁移执行后必须重启

还有个我们立成规矩的点:迁移执行完之后,必须重启后端,而且只重启、不重跑迁移。

听起来像废话,但真有人会想——“迁移已经跑完了,我顺手再 flyway migrate 一次确认一下?“不行。

我特意把这条拎出来讲,是因为它违反直觉。直觉上,幂等操作跑两遍应该没坏处——但 Flyway 的 migrate 在运行中的服务上跑,风险不在"重复执行”(脚本幂等),在"中间状态”。服务正在用着这张表,你这边 migrate 给表加列,那边业务请求正在 INSERT,两个操作撞在同一个 schema 上,要么请求报错,要么迁移卡住。幂等保护的是"重复结果",保护不了"并发执行"。

原因:运行中的服务上做迁移,中间状态会乱

迁移是幂等的没错,但服务在跑的时候,你对它做迁移操作,中间状态很容易乱。

我们的做法是:

  • 迁移走专门的流程跑;
  • 跑完老老实实重启服务;
  • 别在跑着的服务上反复戳。

核心就是一句话:别图省事在运行中的服务上折腾数据库迁移。这句规矩背后是运维纪律:迁移和业务流量必须在时间上严格错开,迁移走维护窗口,服务重启完成后才放流量进来。

这条规矩写进 CI 流程之后,再也没有人能在运行的服务上乱跑 migrate 了——流程不给你这个入口。把"应该怎么做"从口头约定变成流程约束,是治理和约定的本质区别。

版本号撞车这件事,根子在协作不在粗心

回过头看最初那次 V12 撞车,我现在的看法和当时不一样了。当时觉得是两个人粗心,后来想通了:协作场景下的撞车,靠"大家细心点"是兜不住的。

每个人 review 自己的 PR 时,只能看到自己这个 PR 的 diff,看不到另一个 PR 已经合了同号。这不是谁不仔细,是 review 的视野天然是局部的。要让全局唯一性得到保证,只能靠机器——要么 CI 扫一遍全目录,要么用时间戳这种"天然全局唯一"的方案。

号段分配失败也是同一个道理:它把全局协调问题(谁用哪个号)拆给了人去管,但人没有全局视图。秒级时间戳赢在把协调问题消解掉了——时间戳本身就是全局唯一的,根本不需要协调。凡是能把"人去协调"变成"机制天然保证"的,工程上都更省心。 这条判断我们后来在别的地方反复用:能自动化的协调绝不交给人,能天然唯一的绝不靠分配。

写在最后:隐形工程的价值

数据库迁移治理是那种特别"隐形“的工程。不出事的时候,谁也不觉得它有价值——版本号唯一、命名规范、squash 有禁区、repair 自愈、环境分流,这些事平时毫无存在感。

可一旦出事,它就是大事:服务起不来、数据对不上、生产不敢动。而且这种事故有个讨厌的特征——它通常在你最不想它出事的时候出事,比如一次紧急部署、一次大促前的例行更新。越是关键节点,迁移问题的杀伤力越大。

迁移治理的水平,看的是团队愿不愿意下功夫

我现在的判断是,迁移治理的水平,看的是团队愿不愿意在这种"不出事没人感谢"的地方下功夫。

秒级时间戳、validate 阶段校验、squash 禁区、repair 自愈——每一条背后都是一次真实的事故或者差点的事故。

把这些固化下来,不是为了好看,是为了下次有人接手的时候,不用再把坑踩一遍。新人入职,看到 verify_flyway_migration_versions.py 那个校验脚本、看到 FlywayRepairConfiguration 那套自愈 Bean、看到 flyway-repair-schema-history-corruption.md 那份手册,不需要有人口头交代"千万别怎样”,代码和文档自己就把规矩讲清楚了。让代码替前辈把坑讲出来,这是工程治理比口头传承强的地方。

squash 真要整理,也得可追溯

前面说 squash 只能本地,但本地 squash 我们也设计了可追溯的整理方式,不是一删了之:旧文件挪到 migration/archived/ 目录(不在 Flyway 的加载路径里,但 git 里还在,checksum 能对账),只留一个 V1__squashed_baseline.sql 在加载路径里。

文档当时记的是"累计 111 个迁移文件",现在已经涨到 233 个了。整理一直没做,靠秒级时间戳 + 唯一性校验脚本就扛住了——这反过来印证了文章的主旨:治理比整理更重要。整理是清理历史,治理是让历史不再制造麻烦。前者是一次性的体力活,后者是长期生效的纪律。

命名规范决策表:为什么是秒级时间戳

迁移文件命名规范,我们试过三种方案,这是实测对比:

方案示例优点缺点判断
顺序号V12__add_user_field.sql简单直观多人协作必撞车❌ 弃用
号段分配张三 V100-V199,李四 V200-V299看似有序新人不知道找谁要号、合并时偷用别人号段❌ 弃用
秒级时间戳V20260611185900__add_user_field.sql撞车概率极低、自动生成无需协调文件名稍长✅ 采用

时间戳方案的唯一"缺点"是文件名长一点,但这是值得的代价——多打几个字符,换来撞车概率从"每周一次"降到"几乎为零"。

这套演进路径(顺序号 → 号段 → 时间戳)在我们仓库里是肉眼可见的:233 个迁移文件并存着三种命名方案,历史文件没改名,只在 flyway_schema_history 表里 UPDATE version 迁过去。每次新人来看迁移目录都会问一句"为什么文件名格式不统一",答案就是:这是踩了三个坑之后才收敛到现在的规范,改历史文件风险太大,留着当活教材。
