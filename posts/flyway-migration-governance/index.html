数据库迁移治理:Flyway 版本号、squash 和那些启动失败的事故

在我从前端走向全栈架构的阶段,数据库这一层是我必须打通的硬骨头。这篇讲的是其中最容易被忽视、却能让整个服务起不来的一环:Flyway 迁移治理。

某天早上我接到一个告警:某个服务启动不起来。日志翻到最后,就一行 Flyway 报错——迁移版本号对不上。

登上去一看,前一天两个人各自合了代码,各自加了一个 V12__xxx.sql。版本号撞车,Flyway 直接罢工,服务起不来。

事故的本质:不是粗心,是治理缺位

排查倒不复杂,改个号、重新合一下就好了。但事后我盯着那两个 V12 想了很久——这事的本质不是"谁粗心",而是治理缺位。

数据库迁移在多人协作下,如果不立规矩,迟早撞车。而且撞车的代价不是"改个号",是整个服务起不来。

下面是我们团队在 Flyway 上踩过坑后,最终固化下来的几条规矩。

版本号:从号段分配到秒级时间戳

最早我们试过"号段分配"——张三负责 V100 到 V199,李四负责 V200 到 V299。想法挺好,各管一段。

但实际跑下来全是问题:

  • 号段会跨、会重;
  • 新人来了不知道找谁要号;
  • 合并的时候还是会有人偷偷用了别人的号段。

结论:号段分配是反工程化的,它把一个本该自动化的协调问题,丢给了人。

我们的做法:秒级时间戳

后来全换成秒级时间戳:V{YYYYMMDDHHmmss},比如 V20260611185900__add_user_field.sql。谁先建脚本,谁的时间戳就是谁。

两个人同一天、同一分钟、同一秒撞上的概率,比号段撞车低太多了。弃用号段那一刻,工程上反而干净了——少一套要维护的"分配机制",代码和流程里的东西越少越省心。

mvn validate 阶段校验唯一性

光靠人自觉是不够的。我们写了个 Python 脚本,挂在 mvn validate 阶段,扫一遍所有迁移文件,校验版本号是否唯一。不唯一直接 fail 构建。

这个脚本跑到现在,真实校验过的迁移有 119 个,全唯一。不是团队运气好不撞车,是脚本一直在那儿守着——谁撞了在本地构建就过不去,根本到不了 CI。

为什么必须前置到 validate 阶段

版本号冲突这种事,一旦让它进到生产,服务启动直接失败,影响面是全量的。

所以这种东西必须前置到最早:validate 阶段,本地一跑就知道。

这种"不出事没人感谢、出事就是大事故“的基础设施,就得靠自动化兜,不能靠人。

squash 是个坑:只能在本地

Flyway 有个 squash,把一堆历史迁移压成一个新的基线。听起来很爽——迁移文件越积越多,看着烦。

但我们定的规矩是:squash 只能在本地开发库做,生产绝对禁止。

原因:squash 会 DROP DATABASE

squash 的实现是 DROP DATABASE 然后用新基线重建。本地开发库你随便重建无所谓,生产库谁敢 DROP?

我在本地 squash 过几次,基线确实清爽了。但每次都想——这东西要是哪天有人手抖跑到生产上,那就不是"清爽”,是"清场“了。

所以这条规矩我们写得很死,新人入职第一件事就是讲清楚。

schema_history corruption 和 repair

Flyway 有张 schema_history 表,记录哪些迁移跑过了。我们踩过一次——这张表 corruption 了,状态对不上,服务又起不来。

修复手段:repair,但要谨慎

这种时候就得用 repair。repair 是 Flyway 给的修复手法,能把 schema_history 里的脏状态清掉、重新对齐。

但 repair 也是有风险的——它不动业务数据,但会动这张表的记录,所以也得谨慎,不是乱跑的。

我们的一次实战

那次是 dev 环境先 corrupt,我们用 repair 把它救回来了,然后赶紧复盘为啥会 corrupt(具体原因不展开,跟一次中断的执行有关)。

这种事不常见,但一旦碰上,你不会修就很被动。所以 repair 的用法要进手册,而不是出事才去查。

迁移执行后必须重启

还有个我们立成规矩的点:迁移执行完之后,必须重启后端,而且只重启、不重跑迁移。

听起来像废话,但真有人会想——“迁移已经跑完了,我顺手再 flyway migrate 一次确认一下?“不行。

原因:运行中的服务上做迁移,中间状态会乱

迁移是幂等的没错,但服务在跑的时候,你对它做迁移操作,中间状态很容易乱。

我们的做法是:

  • 迁移走专门的流程跑;
  • 跑完老老实实重启服务;
  • 别在跑着的服务上反复戳。

核心就是一句话:别图省事在运行中的服务上折腾数据库迁移。

写在最后:隐形工程的价值

数据库迁移治理是那种特别”隐形“的工程。不出事的时候,谁也不觉得它有价值——版本号唯一、命名规范、squash 有禁区、repair 知道怎么用,这些事平时毫无存在感。

可一旦出事,它就是大事:服务起不来、数据对不上、生产不敢动。

迁移治理的水平,看的是团队愿不愿意下功夫

我现在的判断是,迁移治理的水平,看的是团队愿不愿意在这种"不出事没人感谢"的地方下功夫。

秒级时间戳、validate 阶段校验、squash 禁区、repair 手册——每一条背后都是一次真实的事故或者差点的事故。

把这些固化下来,不是为了好看,是为了下次有人接手的时候,不用再把坑踩一遍。
