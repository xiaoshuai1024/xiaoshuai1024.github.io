一个「下单」按钮点了没反应,可能是前端没绑事件、可能是后端端点 405、可能是小程序和 H5 的 DOM 结构根本不一样。人在这上面耗的时间,远多于写那几行 page.click()。

E2E 测试最贵的不是写脚本,是搞清楚用户旅程到底断在哪一端。 我的做法是把整个流程交给 Agent(和测试纪律、视觉验收是一套,前面讲「分层怎么守、样式怎么验」,这篇讲端到端这一层在 AI 时代怎么写):它拿着一份业务描述,自己分析业务逻辑、自己决定从哪一端进入、自己跨系统点到点地跑完整条旅程,跑挂了自己读日志定位卡点。整个过程的副产物——它点过的每一步——沉淀成 Playwright 脚本,下次回归直接复用。

我的做法是把整个流程交给 Agent:它拿着一份业务描述,自己分析业务逻辑、自己决定从哪一端进入、自己跨系统点到点地跑完整条旅程,跑挂了自己读日志定位卡点。整个过程的副产物——它点过的每一步——沉淀成 Playwright 脚本,下次回归直接复用。

下面拆开讲这套方法怎么落地,以及它和普通 E2E 到底差在哪。

为什么 E2E 这一层最该让 Agent 接手

先把范围说清楚。单元测试盯函数,集成测试盯模块边界,E2E 盯的是用户旅程——一个真实用户从进系统到完成目标的全过程。这件事的天花板不取决于代码能力,取决于对业务的理解深度:你得知道用户会怎么走、会卡在哪、前端后端小程序各自承担了什么角色。

这是 Agent 的强项。它不是来替你写 expect() 的,是来替你做业务逻辑分析的。

具体看普通 E2E 在多端全栈项目里的三个死结:

死结一:旅程横跨多个系统,人很难一眼看穿断点。 一个完整的「下单 → 支付 → 分账」旅程,要经过 H5 前端、运营后台、Spring Boot 后端、微信支付、定时任务分账。任何一环挂了,现象都是「页面上没反应」。人要定位,得翻前端日志、看后端日志、对账数据库、查定时任务执行记录——这是个体力活,但更是个信息整合活,是 Agent 最擅长的事。

死结二:多端的 DOM 结构根本不一样,脚本没法复用。 同一个「商品详情」页,在 H5 里是 <uni-input> 包着 <input>,在小程序里是裸 <input>,在运营后台是 Element Plus 的 el-input。普通 E2E 给每个端写一套定位策略,维护成本翻倍。Agent 不靠固定 selector,它读页面文本和语义来定位,一套业务理解能跨端复用。

死结三:测试脚本和业务理解是两张皮。 普通写法里,业务知识留在 PRD 和人的脑子里,测试脚本只是这些知识的一层「代码投影」。PRD 一改,人要先理解改动、再翻译成脚本改动,中间隔着一层翻译损耗。Agent 主导的做法,是让业务理解本身成为可执行的东西——它读 PRD、读代码、读数据库,直接产出测试动作,翻译这一层被压缩掉了。

这三个死结的共同点是:难点都在「搞清楚发生了什么」,不在「写代码」。 把「搞清楚」这件事交给 Agent,人只盯着「业务到底要什么」,效率就上来了。

两条铁律:先点页面,再沉淀脚本

往下展开之前,先收敛成两条贯穿全流程的铁律。后面所有的工具选择、流程设计,都是这两条落地:

铁律一:所有操作必须通过真实 UI 点击,禁止走捷径。 这是 E2E 的命根子,Agent 也不能豁免。它必须真的去点按钮、填表单、等页面渲染,断言 DOM 变化。禁止用 request.post() 直接调 API 代替页面表单、禁止直接改数据库状态代替点击审批、禁止 page.evaluate() 改应用状态。

判断标准只有一句话:如果一个手工测试人员没法通过点页面完成,Agent 也不应该这么做。

唯一的例外是前置数据准备——创建测试用户、铺测试商品——可以在 beforeAll 里走 API 或 SQL。但主流程必须一寸一寸点过去。理由很硬:绕过 UI 的每一步,都是在跳过「这步用户能不能完成」的验证,而 E2E 的全部价值就在这里。

铁律二:Agent 跑完的每一步,都要沉淀成可复用的 Playwright 脚本。 Agent 这次会点,不代表下次还要它现点。它点完一遍,把动作固化为 spec 文件,之后回归直接跑脚本,快、稳定、不消耗 Agent 算力。Agent 的角色从「执行者」退回「补全者」——只有脚本没覆盖的新功能、新边界,才轮到 Agent 重新下场。

这两条合起来就是整套方法的心智模型:Agent 是放大器,不是替代品。 它放大的是「一次理解、多端复用」的杠杆,但放大的是 E2E 本该有的纪律,不是绕过这些纪律的捷径。

三层执行阶梯:先 H5,再后台,最后小程序

Agent 跑 E2E 不是上来就开测,有严格的分层递进顺序:先在 H5 模式跑通,再走运营后台,最后才碰真实小程序。跳层是大忌。

为什么是这个顺序,理由很具体。

第一层:H5 Playwright。 这是过滤网。uni-app 的 H5 模式有标准 DOM、能开 headed 模式可视化跟踪、调试快、不依赖微信开发者工具。绝大多数页面功能在 H5 下都能操作。Agent 在这一层跑一遍,90% 的显性问题——前端没绑事件、路由 404、表单校验失效——会被直接揪出来。

让 Agent 先碰小程序 automator 是纯浪费。automator 启动慢、调试难、受微信开发者工具环境影响大,一个 H5 下三秒能定位的问题,在 automator 里可能耗半小时还分不清是测试代码错了还是环境抽风。

第二层:运营后台 E2E。 H5 验证通过后,Agent 切到运营后台。这一层验证的是管理和配置侧的旅程——比如「运营在后台开启积分配置 → C 端刷新 → 配置生效」这种跨端联动。运营后台用 Element Plus,DOM 结构和 H5 完全不同,定位策略得换,但业务逻辑是连续的,Agent 一套理解能贯穿。

第三层:微信小程序 automator。 只有前两层都过了,才上 automator。小程序和 H5 的差异 Agent 必须心里有数:小程序里 <uni-input> 就是裸 <input>,可以直接 fill();会话获取方式不同,得通过 wx.login() + 后端 code2session 换 token,不能像 H5 那样往 localStorage 里塞。这些都是 Agent 跨端时的硬知识,不是临场判断。

选择原则就一句话:H5 能验的绝不跑 automator,automator 是最后的手段,不是首选。 跳层等于把不确定性高的环境放到不确定性低的前面,只会拖慢整个测试。

一条完整的 Agent 测试旅程长什么样

把流程讲透了,看 Agent 实际跑一条「用户下单 → 商家分账」的旅程会做什么。这条旅程横跨 H5 前端、Spring Boot 后端、运营后台、微信支付 mock、定时任务——典型的全栈场景。

第一步:业务逻辑分析。 Agent 不直接点页面,它先读。读 PRD 里这条旅程的描述,读相关的前端页面代码、后端 Controller、数据库表结构,在脑子里(它的上下文里)建立一张「这条旅程会经过哪些系统、每个系统负责什么、数据怎么流转」的地图。

这一步是整套方法的命脉。普通 E2E 跳过这步,直接对着页面写 page.click('#submit'),所以它永远不知道 #submit 背后连着什么。Agent 建了地图,后面每一步点击它都知道自己在哪一层、这一层应该发生什么、断了应该往哪查。

第二步:端点冒烟测试。 进入 UI 点击之前,Agent 先验证后端路由真的通——这是血的教训换来的规则。静态代码审查(读 diff、看 @PostMapping 注解)无法发现路由运行时不可达。典型事故:后端写 @PostMapping("/orders/{orderNo}:hide"),语法合法、编译通过,但 Spring MVC 把 :hide 当矩阵变量解析,这个端点永远返回 405。

Agent 的做法是 curl 每个新端点,看 HTTP code:4xx(400/401/403)说明路由通了(只是缺认证或参数),算通过;404 或 405 是阻断,必须先修再往下。这一步廉价(每个端点不到 10 秒),但能抓住所有路由级 BUG,避免「UI 怎么点都没反应、查半天才发现端点根本不可达」的浪费。

第三步:UI 点到点。 路通了,Agent 才开始真的点。它从首页进入,用 ensureLoggedIn() 走真实 mock-login API 拿 token 注入 session(绝不用 SQL 伪造 token),然后按业务旅程一步步点:进商品页 → 选 SKU → 加购 → 下单 → 选地址 → 支付。

每一步它都断言三件事——UI 反馈(Toast 出来了、跳转发生了)、数据持久化(API 返回了、DB 里有记录了)、可逆性(能取消、能编辑、能删除)。这三件缺一就是无断言的操作,等于没测。Agent 不会犯「点了不断言」这种错,因为它的每一步动作都对应业务逻辑里的一个状态变更,它天然知道这一步该断言什么。

第四步:跨系统数据追踪。 这是 Agent 真正甩开普通 E2E 的地方。用户点完支付,Agent 不只看页面显示「支付成功」,它同时查后端订单表状态、查微信支付 mock 的回调记录、查分账任务的待执行队列。一个「支付成功」的旅程,Agent 在五个系统里同时验证数据闭环。

普通 E2E 做不到这一步——Playwright 脚本只看得见 DOM。Agent 能跨系统,是因为它会用 curl、会查数据库、会读日志,这些工具它样样都会。人以前要切五次窗口干的事,Agent 在一次旅程里并行干完。

第五步:沉淀脚本。 旅程跑通了,Agent 把这次的每一步动作固化为 trade-order-flow.spec.ts。下次回归,这条旅程直接跑脚本,Agent 不再下场。脚本里明确标注:这一步在等什么异步操作(比如 void toggleExpandSku() 没被 await,点击后得加 waitForTimeout)、这一步用了哪个 fixture、这一步为什么走了 UI 而非 API。这些标注是 Agent 留给未来回归的备忘录。

这五步跑完,一条横跨五系统的旅程被验证了一次,而且可复现地验证了无数次。Agent 的价值不是这次跑通了,是它跑通的方式能被复制。

Agent 怎么分析业务逻辑、找卡点

前面反复说「Agent 分析业务逻辑」,这一节具体讲它怎么分析,以及它怎么定位测试没通过的原因。这是整套方法区别于「AI 帮你写几个测试用例」的关键。

业务逻辑分析的输入是三份资料:PRD、代码、数据。 Agent 读 PRD 拿到「应该发生什么」,读代码拿到「实际怎么实现」,查数据库拿到「实际发生了什么」。三者对齐,功能正常;三者错位,就是 BUG。这套思路本来是高级 QA 干的活,Agent 把它常态化了。

举个具体的卡点定位。Agent 测试「积分抵扣下单」旅程,点完支付,UI 显示「支付成功」,但它查后端积分账户,发现积分没扣。普通 E2E 在这里就结束了——页面显示成功,脚本断言通过,收工。Agent 不会,它会继续往下挖:

  • 查订单表,points_deducted 字段是 0,确认是积分逻辑没触发;
  • 查后端日志,发现积分扣减接口被调用了,但抛了 INSUFFICIENT_POINTS 异常;
  • 查积分账户表,发现账户余额确实是 0——但这个用户刚才在另一条旅程里被扣过一次,测试数据没隔离;
  • 定位根因:不是业务 BUG,是测试数据污染——上一轮测试的积分扣减没回滚,这一轮继承了脏数据。

这个定位过程,Agent 沿着「现象 → 数据 → 日志 → 根因」的链路一路追下去,每一步都用真实证据推进,不靠猜。最后它给出的不是「这里有个 BUG」,而是「这是测试数据问题,建议在 afterEach 里重置积分账户」——带着建议,等人确认再改。

关键纪律:Agent 分析出问题,先给建议,等确认再动手。 它不自行修复业务逻辑 BUG。这是为了守住一条边界——Agent 是测试者,不是改业务代码的人。它发现的业务逻辑问题,记录为缺陷,人决定怎么改。但测试代码(spec 文件)不受这个限制,Agent 可以为新发现的功能补测试、为没覆盖的边界加用例。

这条纪律的是:覆盖完整性 > 规则遵守 > 代码风格。Agent 发现一个模块有功能没被测试覆盖,它会打破「不改已有逻辑」去补测试;发现现有测试缺断言,它会补断言;发现现有 spec 不适合扩展,它会新建 spec。它不会因为「不要动现有代码」而放过未覆盖的功能——那是测试者的失职。

测试脚本怎么沉淀、怎么复用

Agent 跑完的旅程要能持续复用,关键是脚本的组织和沉淀规则。这块规则不清,跑了几轮之后就是一堆谁也不敢动的「历史 spec」,Agent 的杠杆效应就归零了。

沉淀决策树。 不是 Agent 点过的每一步都该写成脚本。Agent 自己用一个决策树判断:

  • 这个操作是否频繁回归?是 → 已经有 E2E 覆盖吗?有 → 检查是否覆盖了这次发现的场景,覆盖了就更新断言,没覆盖就在现有 spec 追加;没有 → 是 UI 交互吗?是 → 创建 H5 Playwright E2E;否 → 创建单元测试。
  • 这个操作是否频繁回归?否 → 保持手工测试,标记 [MANUAL],每季度评审一次要不要自动化。

这个决策树保证脚本只沉淀值得沉淀的——P0 核心流程、高频回归路径、多组合参数场景。一次性验证、低频操作,留在 Agent 的手工模式里,不污染脚本库。

spec 文件的组织规则。 优先在现有 spec 里扩展,只有功能完全没覆盖时才新建。扩展时遵守现有 spec 的 fixture 使用模式、session 获取方式、数据创建方式——这些是项目级的约定,Agent 要先读再写,不能自己另起炉灶。

测试数据要自包含。每条 spec 在 test / beforeAll 里创建自己需要的全部数据,不依赖任何预存数据。用 INSERT IGNORE 避免重复运行的主键冲突,用 Date.now().toString(36) 生成唯一标识,在 afterAll 里清理。一个 spec 跑完,数据库回到跑前的状态——这是脚本能持续复用的前提,否则跑几轮数据就脏了。

死数据检测沉淀进 spec。 Agent 的强项之一是它顺手就能查死数据。脚本里固化几条检查:外键关联的父表记录是否都存在(孤立记录)、状态字段是否长时间卡在中间态(悬挂状态)、软删除记录是否还被 JOIN 查询命中(幽灵数据)。这些检查每次回归都跑,等于把「数据健康」纳入了 E2E 的职责。

这套沉淀规则的Agent 产出的脚本,要能脱离 Agent 独立运行。 脚本一旦沉淀,它的执行不再依赖 Agent 的临场判断,而是依赖代码里写死的断言和 fixture。Agent 的角色从「执行」退到「补全」——只有新功能、新边界、新发现的 BUG 需要它重新下场。这是杠杆效应的体现:一次 Agent 介入,长期回归收益。

和普通 E2E 到底差在哪

讲透了流程,来正面回答「Agent 主导的 E2E 和普通 E2E 有什么本质区别」。这不是「AI 帮你多写几个用例」的区别,是测试这件事的重心发生了转移。

用一张表把差异收敛清楚:

维度普通 E2EAgent 主导 E2E
重心写脚本读业务
跨系统能力只看 DOMDOM + API + DB + 日志,并行
多端复用每端一套定位策略一套业务理解跨端
失败定位脚本报错,人排查Agent 读日志追根因,给建议
业务知识载体留在 PRD 和人脑留在 Agent 的分析 + 沉淀的脚本
覆盖盲区人没想到的不测Agent 主动枚举状态和边界
维护成本PRD 一改,人翻译改脚本PRD 一改,Agent 重新理解再补
脚本沉淀人手写,质量参差Agent 跑通即沉淀,天然带断言

这张表里有几条值得单独拎出来讲。

普通 E2E 只看 DOM,这是它最大的盲区。 Playwright 的 page.click() 后面发生了什么,脚本不知道。它只知道「这个元素还在不在、文本变没变」。所以普通 E2E 经常出现一种诡异情况:页面显示「下单成功」,脚本断言通过,但后端订单根本没落库——因为前端有个 try/catch 把后端的异常吞了,UI 假装成功。Agent 不会被骗,它同时查后端,UI 和数据对不上它立刻报错。

普通 E2E 的覆盖盲区是人没想到的边界。 一个下拉框有三个选项,人写脚本大概率只测默认选项;一个状态机有五态,人大概率只测主路径的 A→B→C。Agent 不一样,它的纪律是「任何有状态的元素必须遍历所有状态」「任何可交互元素必须实际操作一遍」。这套穷尽原则本来是高级 QA 的内功,Agent 把它变成了默认行为。

普通 E2E 的业务知识是断的。 PRD 写一套、人脑记一套、脚本投影一套,三套之间永远有翻译损耗。Agent 主导的做法,业务知识的载体是Agent 的分析过程本身——它读 PRD、读代码、查数据,这个过程是可追溯的、可复现的。PRD 一改,Agent 重新读一遍,差异自然浮现,不需要人去做「PRD 改了哪几条、对应脚本改哪几行」的翻译。

但必须说清楚 Agent 主导 E2E 不擅长什么,免得被神化:

  • 它不擅长从零发明测试场景。 Agent 需要一份业务描述作为输入。没有 PRD、没有需求文档,它不知道该测什么。业务输入是人的责任。
  • 它的脚本沉淀质量依赖项目的可测试性。 如果项目本身前端没埋 data-e2e 属性、后端没有 mock-login、数据没法隔离,Agent 的脚本也会很脆。可测试性是架构属性,Agent 救不了架构债。
  • 它对小程序 automator 的稳定性依赖环境。 微信开发者工具的 CLI 经常抽风,Agent 在这一层偶尔会因为环境问题卡住,得人介入。这也是为什么把 automator 放在最后一层。

承认这些边界,不是贬低 Agent,是为了把它的能力用在刀刃上。Agent 主导的 E2E 不是万能的,但它在横跨多系统的全栈项目里,效率碾压普通 E2E——这恰好是现在大多数项目的真实形态。

最后说一个落地后会遇到的现实问题:Agent 产出的脚本怎么维护。Agent 写脚本快,但它写出来的东西不一定符合团队规范——命名混乱、重复代码、硬编码 selector 都可能出现。我的做法是给 Agent 配一份「E2E 编写规范」(放在 .agents/skills/e2e-testing/ 里),让它写之前先读。规范里钉死几条:所有 selector 用 data-e2e 不用 class、所有等待用 waitForResponse 不用 waitForTimeout、多端公用步骤抽成 helper。

即便有规范,Agent 产出的脚本还是要人过一遍。这一遍不是逐行 review(那样就失去了 Agent 的效率优势),而是抽审——挑两条核心链路读一遍,看业务逻辑对不对;再跑一遍 lint 看结构规范不规范。我发现 Agent 写的脚本,业务逻辑基本都对(它读过 PRD 和代码),问题集中在工程细节(等待方式、错误处理),而这些 lint 能抓。人审业务,机器审结构,两层配合,维护成本才可控。这套机制跑顺之后,Agent 产出的脚本能直接进仓库,不用人再大改,这才是它真正的效率价值所在。说到底,Agent 主导 E2E 不是让你少写脚本,是让你的精力从「写脚本」转移到「读业务」——前者是体力活,后者才是质量的关键。脚本谁都能写,但「这条用户旅程的哪些环节最可能静默失败」这种判断,只有懂业务的人做得出。Agent 把前者自动化了,你才腾得出手做后者。这才是 Agent 主导 E2E 这件事,对质量最实在的贡献。
