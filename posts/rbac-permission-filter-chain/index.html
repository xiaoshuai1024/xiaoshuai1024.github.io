权限校验散在每个接口里:我把它收成注解 + 过滤器链 + 契约闸

权限体系最怕的不是「没做」，而是「散着做」——这个接口开头一个 if (user.hasRole("admin"))，那个接口中间塞一段 if (!user.can("edit"))，还有几个接口压根没校验，靠「前端按钮不显示」硬撑。

把后端的权限校验从「散落在每个接口里的 if」收拢成一条声明 + 校验 + 对齐 + 兜底的链路，是用机制替代人记的工程判断。这篇讲怎么收。

「散着做」到底贵在哪

我接手过一个运营后台,权限校验是这种画风:这个接口开头一个 if (user.hasRole("admin")),那个接口中间塞一段 if (!user.can("edit")),还有几个接口压根没校验,靠「前端按钮不显示」硬撑。

老实讲,这种代码最致命的不是「能不能跑」,而是谁都不敢动。一段判断摆在接口里,三个月后你想删它,第一反应不是「这判断对不对」,而是「删了会不会越权」——因为没人知道这段判断是踩了哪个坑加的,删了出了事算谁的。于是它就赖在那儿,越来越多,越来越乱。

代价是具体的:

  • 谁也不敢删那段判断,因为谁也不知道删了会不会越权。 最后代码里堆满「祖传 if」,谁动谁背锅。
  • 前端按钮不显示,不等于后端没校验漏洞。 按钮显示是前端逻辑,接口鉴权是后端职责。黑客根本不经过你的前端,直接打接口。把安全寄托在「按钮藏起来了」,等于没做。
  • 几个接口漏校验,纯靠运气没出事。 接口多了之后,「哪个加了哪个没加」全靠人记。今天没出事,是因为没人想到去打那个接口;明天被人扫到,就是事故。
  • 权限和业务逻辑搅在一起,改一个判断得读懂整个方法。 这段判断嵌在业务代码中间,你要重构这个方法,就得先把权限逻辑摘出来——但摘出来的过程中,你怎么知道没摘错?

所以判断标准很简单:如果你问「这个接口的权限规则是什么」,得去翻业务代码才能回答,那就是散着做。 真正该有的状态是,看一眼注解就知道。

「散着做」还有一个被严重低估的成本:权限审查没法做。合规审计要求「列出所有需要 admin 权限的接口」,如果权限散在代码里,你得 grep 全项目,还可能 grep 不全(因为有些判断不是 hasRole("admin") 这种字面量,而是 user.getLevel() > 5 这种隐式的)。注解声明的好处在这里就体现了——@RequirePermission 是结构化的,grep 一下注解就能列出所有接口及其权限要求,审计报告几分钟就出。散着做的项目,这种审计能折腾一周还不敢保证全。

散着做还会让「权限测试」无从下手。测试要验「无权限的用户调这个接口应该 403」,但如果权限判断散在业务逻辑里,测试得把业务逻辑跑通才能验到权限——有些权限判断在业务流程的很后面(比如「提交订单时检查用户有没有下单权限」),测试得走到提交那一步才能触发。注解声明在接口入口就拦,测试直接调接口验 403,不用跑业务流程。权限测试的效率,差了一个数量级。

推倒重做时,核心思路就一句话——权限跟着接口走,不跟着人记。

注解声明,让权限长在接口上

定义一个注解,大概长这样:

@RequirePermission("ops:order:refund")
@PostMapping("/order/refund")
public Result refund(...) { ... }

注意那个权限 key:ops:order:refund。它不是随手写的字符串,而要在迁移脚本的权限种子数据里先注册过。这一条最关键,后面要出事就出在这。

key 的命名是有规范的,我们整个团队统一用「资源:动作」的三段式:{系统}:{资源}:{动作},比如 ops:order:refund(运营后台 / 订单 / 退款)、ops:order:export、fin:bill:audit。规范本身不复杂,但它解决三件事:

  • key 在种子数据里必须先注册。 没注册的 key 走到运行期要么报错要么放行,两种结果都不好。所以 key 是「先在种子表里声明,再在注解里引用」,顺序不能反。
  • 和接口绑死,新人不用通读业务。 新人看接口签名就知道「这接口吃什么权限」,不需要去翻业务逻辑、不需要去问老人。这点对团队扩张特别值钱。
  • 权限信息和接口定义不再分家。 改接口的人顺手就能看到权限要求,删接口的人知道顺手把种子 key 也清掉。

有了注解,接口要什么权限就是声明出来的:

  • 新人不用通读业务逻辑,也能知道「这个接口吃 ops:order:refund」。
  • 权限信息和接口定义绑在一起,不再散落在代码各处。
  • 改接口的人顺手就能看到权限要求,删接口的人知道该把种子 key 也清掉。
  • Code review 时,「这个接口有没有权限」从「翻业务代码」变成「看注解有没有」。

比满屏 if 清爽太多。注解本身的定义也就几行:

@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface RequirePermission {
    String value();          // 权限 key，如 ops:order:refund
    boolean required() default true;  // 是否强制，false 时走宽松校验
}

注解里那个 required() default true 有个用法值得说。大多数接口权限是强制的——没权限就是 403,没得商量。但少数接口是「有权限看全量,没权限看部分」,比如订单列表,有 ops:order:view_all 权限的看所有订单,没权限的只看自己的。这种接口标 @RequirePermission(value="ops:order:view_all", required=false)——required=false 意味着「没这个权限不拦,但业务层要知道用户没这个权限」,过滤器放行后在上下文里标记「该用户无 view_all 权限」,Service 层据此决定查全量还是查自己的。这个设计让一个接口能同时服务「有权限」和「无权限」两类用户,不用拆成两个接口。

注解还能标在类上(@Target 里有 TYPE),类级注解是「这个 Controller 下所有接口的默认权限」。如果某个方法需要不同权限,方法级注解覆盖类级。这个层级让「整个 Controller 都需要 admin 权限,只有某个查询接口放宽」这种常见场景写起来很简洁——类上标一次,方法上单独标例外。

过滤器链统一校验,顺序是命门

光有注解还不够,得有人去「读」它。这件事交给**过滤器链(filter chain)**统一做。请求进来,一路过:

认证 → 租户解析 → 权限校验

这个顺序必须刻死。问题的本质在于:权限校验这一步,要拿「当前用户属于哪个租户」去查权限表。如果它排在租户解析前面,租户上下文还没注入,拿到的永远是空,鉴权逻辑整个乱套。

顺序错乱的具体后果,都是事故级的:

  • 权限校验排到租户解析前面,租户拿不到,查权限表查不到该用户的记录——要么一律放行(越权),要么一律拒绝(全站不可用)。
  • 认证排到后面,你拿一个没校验过的 token 去解析租户,等于把黑客的输入当可信数据用,后面全白做。

所以我们把这套顺序白纸黑字写进文档,新加过滤器时必须按文档插位置,而不是凭感觉往链头一塞。顺序在过滤器链里是个隐式契约,不写下来迟早会乱。

// 过滤器顺序由 @Order 刻死,改顺序必须改文档
@Component
@Order(10)  // 1. 认证:校验 token,挂上当前用户
public class AuthFilter implements Filter { ... }

@Component
@Order(20)  // 2. 租户解析:注入 tenantId 到上下文
public class TenantContextFilter implements Filter { ... }

@Component
@Order(30)  // 3. 权限校验:读 @RequirePermission,查租户权限表
public class PermissionFilter implements Filter {
    // 伪代码:必须先从上下文拿 tenantId,再查权限
    Long tenantId = TenantContext.get();   // ② 注入的
    String key = method.getAnnotation(RequirePermission.class).value();
    if (!permissionService.has(tenantId, userId, key)) {
        throw new ForbiddenException(key + " not granted");
    }
}

这里的细节是:权限校验器自己不解析 token、不解析租户,它只消费前两步的结果。 职责单一,顺序才不会乱。

职责单一还有个连带好处:过滤器可测试。PermissionFilter 只依赖 TenantContext 和 userId,测试时直接 mock 这两个输入,不用起完整的 HTTP 请求链路。如果它自己解析 token,测试得构造完整的 token 和请求头,复杂度立刻上来。职责单一不只是架构好看,直接降低了测试成本——而测试成本低的组件,大家才愿意给它写测试,它的质量才有保障。

过滤器顺序还有一个容易被忽略的坑:Spring 的 @Order 数值是相对的,不是绝对的。两个过滤器标 @Order(20) 和 @Order(30),Spring 保证 20 在 30 前面,但中间如果有 @Order(25) 的过滤器,它就插在中间。所以加新过滤器时,要确认它的 @Order 落在期望的位置,而不是随手标个值。我们的做法是给 @Order 值留间隔(10、20、30、40,而不是 10、11、12、13),留出插入新过滤器的空间,不用改已有过滤器的值。

权限 key 和种子数据必须对齐

这是这类体系里最隐蔽的一类问题。

某次有人报:「这个资源我明明没给 create 权限,他怎么还能操作?」一查发现:代码里 @RequirePermission 写的是 create / edit / delete,但迁移脚本的权限种子数据里,这个资源只注册了一个 manage。

翻译成一句结论:接口要的权限 key,在权限表里压根不存在。

这种 bug 的危险在于它「静默地不安全」:

  • 查不到对应 key 时,有的实现是「查不到就放行」——理由是「没配就是不限制」,直接越权。
  • 它不崩、不报错,压测压不出来,QA 测不出来,只有真出事才知道。

「查不到就放行」这个默认行为,是权限体系里最该警惕的设计。它的逻辑是「没配权限 = 谁都能用」,听起来像是「宽松模式方便调试」,但在生产环境就是越权漏洞。正确的默认应该是「查不到就拒绝」——没配权限意味着这个 key 没被正式授权,不该放行。我们改过一次这个默认值,从「放行」改成「拒绝」,改完当天就发现了三个漏配的 key——之前它们一直静默放行,没人知道。这个改动把隐性漏洞变成了显性报错,虽然当天多了几个 403 要处理,但比线上越权强一万倍。

为什么压测测不出?压测打的是流量和性能,它假设逻辑是对的。逻辑错了,流量再大也只是一个错误结果重复一万遍,日志里不会冒出一个 warn。QA 测不出同理——除非测试用例专门覆盖「key 不对齐」这条边角,否则就是一路绿灯通过。

所以规矩是:权限 key 必须双向对齐。

  • 接口里 check 的 key,种子数据里必须有。
  • 种子数据里注册的 key,得有接口在用。
  • 谁多出来,谁就是问题。

种子 SQL 大概长这样,和注解里的 key 一一对应:

-- 种子数据:key 必须先注册,接口才能引用
INSERT INTO sys_permission(key, name, resource, action) VALUES
  ('ops:order:refund', '订单退款', 'order', 'refund'),
  ('ops:order:export', '订单导出', 'order', 'export'),
  ('ops:order:view',   '订单查看', 'order', 'view');

这层契约,靠契约闸兜

光靠「大家记得对齐」是不行的。人和 AI 都会忘——尤其是新加接口时,注解里随手写个 key,种子数据忘了补,review 时眼神一飘就过了。

我把这条对齐做进了契约闸:构建阶段扫一遍,接口上 @RequirePermission 声明的所有 key,去权限种子数据里逐个核对,缺一个就构建失败。

契约闸怎么落?大致三步:构建期扫一遍所有 Controller 的 @RequirePermission 注解,把 key 收集成集合 A;读迁移脚本里的权限种子 key,收集成集合 B;算 A - B,有差集就 fail,exit code 非 0,流水线挂掉。新加接口没补种子的那一刻,代码根本进不了主干。

伪代码,真没多少行:

# verify_permission_keys.py · 构建期执行(mvn validate 阶段)
keys_in_code = scan_require_permission_annotations("src/main/java")  # 集合 A
keys_in_seed = extract_keys_from_flyway_migrations("db/migration")    # 集合 B

missing = keys_in_code - keys_in_seed   # 接口要、种子没有 → 越权风险
orphan  = keys_in_seed - keys_in_code   # 种子有、接口没用 → 死权限

if missing:
    print(f"❌ 缺失权限 key:{missing}"); sys.exit(1)
if orphan:
    print(f"⚠️ 孤儿权限 key(可清理):{orphan}")

missing 是硬失败,orphan 是软告警——前者会越权,后者只是种子表里多了几条没人用的 key,清不清都行,但留着一眼就能看出来。

missing 和 orphan 区别对待是有讲究的。missing(接口要、种子没有)是安全隐患,必须硬失败拦住;orphan(种子有、接口没用)不是安全问题,只是技术债——某个接口被删了但种子 key 忘了清,留着的 key 不会造成越权,只会让权限表越来越臃肿。如果把 orphan 也设成硬失败,团队会因为「种子多了几条没用的 key」而构建失败,这种鸡毛蒜皮的阻断会让人烦,反而削弱契约闸的威信——大家会觉得「这闸动不动就拦,烦死了」,开始想办法绕过它。区分严重性,只拦真问题,放过小问题,契约闸才能长期被人接受、不被绕过。

这条兜底一上,「漏配 = 越权」的事故基本就绝迹了。日常做 AI 协同研发时(主力 Claude Code,其次 Cursor、OpenCode),契约闸同样是让 AI 不漏配权限的那道防线。AI 加接口比人快得多,注解里 key 写错了它自己不知道,但构建闸会挡住。

AI 加接口时的典型翻车场景:它看了 Controller 里有 @RequirePermission("ops:order:refund"),新加一个「订单退款审批」接口时,顺手也写了 @RequirePermission("ops:order:refund")——但审批应该是一个独立的权限 ops:order:refund_approve,复用 refund 意味着有退款权限的人都能审批,权限被放大了。这种「图省事复用 key」的毛病,AI 特别爱犯,因为它觉得「功能差不多,权限也差不多」。契约闸拦不住这种(因为 key 在种子里有),但 review 时要盯——接口的权限 key 应该精确反映这个接口的业务语义,不该为了省事复用相邻接口的 key。

安全这种事,不能指望人记,得指望机制兜。 人会犯错,注解会写错,种子会漏配,但构建闸不会放水。

最后说两句

回头看这套 RBAC 的演进,主线就一条:把权限从「散落在每个接口里的 if」,收拢成「声明 + 统一校验 + 种子对齐 + 契约兜底」的一条链。

一个权限体系成熟度怎么看?我的答案是:看它有多「不靠人记」。

如果一个新人改个接口,不需要懂历史、不需要问老人、不需要记住一堆隐式规矩,只要照着模板写注解、补种子,就能保证权限正确——那这套体系就算成了,可以放心交给团队和 AI 协同使用。如果还停留在「这个接口忘了加判断」的程度,那就还早。

权限这事,做得好的标志是**「没故事可讲」**。出故事了,基本就是越权了。

「没故事可讲」这个标准,反过来也是对权限体系的检验。定期问自己:「最近半年,权限出过几次事?」零次,说明体系在起作用;出过一两次,复盘时问「这套体系哪一层该兜住但没兜住」,补上那一层。权限体系不是搭完就一劳永逸,它是随着业务演进持续打补丁的过程——新业务模式带来新的权限维度(比如多店铺、多 realm),每次扩展都要回过头确认「现有机制能不能覆盖新维度」。体系活着,它才管用;搭完不管,它早晚漏。权限这块,持续维护的长期投入,远比一次性搭建更重要,这才是它能不能长期稳得住、不出故事的关键。

深度展开:过滤器链的真实实现

权限校验最容易出错的不是"逻辑写错了",而是"该加校验的地方忘了加"或"顺序排错了"。这里把我们真实的过滤器链拆开讲。

我们用 Spring 的 OncePerRequestFilter + @Order 刻死过滤器顺序。TenantShopAuthFilter 标了 @Order(4),排在 TenantFilter(@Order 更小,先执行)之后——它依赖 TenantContext 已经注入了租户 id,所以必须在租户过滤器之后跑。这个顺序是硬约束:@Order 的数值就是执行顺序,改顺序就是改 @Order 的值,改动会在代码里留下痕迹,review 时一眼能看到。

过滤器内部有个关键设计:它不自己解析 token,而是消费前序过滤器的结果。TenantShopAuthFilter 一进来先 TenantContext.getTenantId() 拿租户 id——这个 id 是 TenantFilter 注入的,不是它自己解析的。职责单一:认证过滤器管 token 解析,租户过滤器管租户注入,权限过滤器管权限校验,各管一段。如果权限过滤器自己又去解析 token,职责就串了,顺序一乱全乱。

权限加载的真实逻辑比注解声明复杂得多。我们有一个 TenantAuthzService.loadPermKeys 方法,它跑一条五表 JOIN 的 SQL:tenant_subject_membership(用户在某个 realm 的成员身份) → tenant_membership_role(成员和角色的关联) → tenant_realm_role(角色定义,带 realm 隔离) → tenant_realm_role_perm(角色和权限点的关联) → rbac_perm_catalog(权限点目录)。五个表 JOIN 完,拿到的是「这个用户在这个租户、这个 realm、这个 scope 下,拥有哪些权限点」。

这条 SQL 里有两个容易被忽略的细节。一是 realm 和 scope 两个维度:同一个租户下,「店铺」(SHOP realm)和「运营后台」(OPS realm)的权限是隔离的,一个用户在店铺有管理权限不代表在后台也有。scope 更细,同一个 realm 下还能按 scope(比如具体哪个店铺)再隔离。二是每个表都带 status = 1 过滤——禁用的角色、禁用的成员身份、禁用的权限点,都不参与计算。这条过滤不加,一个被禁用的角色还能授予权限,等于禁用没生效。

五表 JOIN 的链路画出来,「一个用户的最终权限是怎么算出来的」就清楚了:

这张图最能说明的一点是:权限不是存在某个表里的现成字段,是五个表 JOIN 出来的计算结果。这意味着权限的「最终状态」取决于五个表里每一行的 status——任何一个表的某行被禁用,权限就变了。这种设计的好处是灵活(禁用一个角色,所有持有该角色的用户权限立刻变),代价是排查时要顺着五张表追,不像「用户表里有个 role 字段」那么直观。这也是为什么我们强调权限问题的排查要从这条 JOIN 链路入手,而不是只看某一张表。

还有个实战中踩出来的设计:公开读接口的白名单。不是所有接口都要查 RBAC——店铺的动态列表、模板列表这些是公开可读的,游客(没登录)也能看。如果这些接口也走完整的 RBAC 校验,游客没 token 直接 401,页面白屏。所以 TenantShopAuthFilter 里有个 publicShopReadWithoutRbac 方法,对特定的 GET 路径放行,不查权限表。这个白名单的判定逻辑也讲究:只放 GET(读),不放 POST/PUT/DELETE(写);路径要精确匹配或带数字 ID 的前缀匹配,不做模糊包含(避免误放)。

这个白名单还有个更实际的考量:避免缺表 500。我们用 Flyway 管理迁移,新环境如果 RBAC 相关的表还没跑齐,公开接口也去查 RBAC 表会直接 500。公开读接口白名单绕过了 RBAC 查询,即使表没建好,公开接口也能正常返回——这对灰度部署和新租户初始化特别重要,不会因为 RBAC 表还没建好就整个店铺页面挂掉。

最后,和 TenantContext 一样,ShopSecurityContext 也是 ThreadLocal,finally 里必须 clear。线程池复用线程时,上一个请求的权限上下文不清,下一个请求拿到的是上一个用户的权限——A 用户的权限被用来校验 B 用户的请求,越权就这么发生了。这个 clear 和租户上下文的 clear 是同一类问题,同一个解法。

深度展开:权限校验的边界——什么不该交给过滤器链

过滤器链虽好,但不是所有权限校验都该塞进去。这里划几条边界。

业务级权限不适合放过滤器链。比如「用户只能删除自己创建的订单」,这种判断要看订单的 creator_id 是不是当前用户——这是数据级权限,要查数据才能判断,塞进过滤器链里会让过滤器变重(要查库),而且和业务逻辑耦合。这类校验放在 Service 层更合适,过滤器链只管「接口级权限」(这个用户能不能调这个接口),不管「数据级权限」(这个用户能不能操作这条数据)。两层分清,各管各的。

过滤器链里不要做复杂业务判断。过滤器应该是「快进快出」的——拿上下文、查权限、过或不过,不该在里头跑复杂业务逻辑。一旦过滤器变重(比如在里面调好几个 Service),每个请求都要承受这个开销,而且过滤器的职责就模糊了。我们的原则:过滤器只做「能不能进」,不做「进了干什么」。
