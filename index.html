AI 时代你还在古法学习吗?我用 Claude Code 搭了一套数据驱动的学习闭环

 included in AI 开发
我在 front-end → 全栈 → AI 开发这条路上走得很早,也很清楚一件事:技术人的成长从来不是线性的,而是被"市场需求 × 自我认知 × 学习方法"三个变量反复拉扯。多数人卡在第二个和第三个——不知道自己在市场上值多少钱,也不知道该怎么把短板补到及格线。 这篇文章讲的就是我怎么用 Claude Code 把一个私有的"学习系统"搭成一个可运行、可量化、可复盘的知识库。不是讲考试技巧,而是讲「在 AI 时代,一个技术人该怎么组织自己的学习」。整套东西都在本地跑,所有数据都是真实可验证的,所有流程都是一条命令触发的。 先看结论:古法学习的核心问题是没反馈回路;AI 驱动学习的核心优势是把反馈回路压到分钟级。 一、古法学习的三个致命缺陷 我过去十年的学习方式和大多数工程师一样:刷掘金 / 看官方文档 / 跟着教程写个 demo / 放到收藏夹里吃灰。这种"查资料式学习"有三个结构性缺陷,不是靠努力能弥补的。 先说一个我自己的账本。我在学 RAG 的时候,把 LlamaIndex 官方文档、几篇高赞博客、一个 GitHub 上的 demo 项目都过了一遍,笔记做了不少,本地还 clone 了 demo。那两周我觉得自己"掌握 RAG 了"。后来有次技术选型会,同事问我"RAG 里 chunk 大小和 overlap 在生产环境怎么定,你们踩过什么坑",我支吾了半天,只说出"按经验来"。回去翻收藏夹,文章还在,但具体数字全忘了,那个 demo 项目也早就没跑过。 这不是个例,这是古法学习的结构性失败。 第一个:学什么靠拍脑袋。 看到 RAG 火了就去学 RAG,看到 LangGraph 出新版就去翻文档,完全是被社区热点牵着走。结果半年下来,收藏夹里堆了几百篇文章,简历上写不出任何一项"市场需要、我精通"的技能。学到最后,自己觉得都会,一被追问就全露馅。 更要命的是,这种学法容易陷入"低难度重复"。人天然喜欢学自己已经会的东西——因为那种"我又看懂一篇"的成就感很爽。结果就是:精通的更精通,薄弱的永远薄弱。没有外部信号告诉你"你该学什么",你永远在自己的舒适区打转。 第二个:学了=会了是幻觉。 看文档时觉得自己懂了,写 demo 时照着抄一遍跑通了,就给自己打个勾。这种"理解"是经不起追问的——别人问"为什么这么设计"“生产环境出过什么问题"“和另一方案相比差在哪”,多半答不上来。看十遍文档不如被追问一次,但古法学习里根本没有"被追问"这个环节。 我自己有次被戳穿的经历:学 Vite 的时候,我照着官方文档配了一遍 plugin,跑通了,心里想"懂了”。后来有次技术讨论,对方问我"Vite 的 pre-bundle 阶段用的什么工具,为什么 dev 比 webpack 快这么多",我只答出"用了 esbuild",但说不清楚 esbuild 做了什么、为什么不直接裸跑 ESM、生产构建为什么又要切回 Rollup。回去看文档才发现,那些内容都在第 3 段,但我当时"看懂了"就跳过去了。

荒芜之地长出 CI/CD：没有流水线，三条命令跑通了 CI、CD 和 DevOps

 included in AI编程
跟同行聊 CI/CD，对方都默认你至少有个 GitLab Runner 或者 GitHub Actions。我接手的一个全栈项目，那段日子什么都没有：没有流水线，没有自动化测试，没有提交规范，连 PR 流程都是口头说一声「我推上去了你看下」。部署是本地 mvn package 出 jar，再 scp 拷到服务器，运营后台的 dist 目录手动传。 按教科书，这种状态就该立刻上 CI 平台。但我做了个反直觉的决定：先不碰平台，用三条 AI 命令把 CI、CD、DevOps 三阶段全跑通了。这三条命令是 /pr-all、/deploy（加 /prod-testing）、/prod-debug——挂在 Claude Code 上，每次发版一条龙走完。 更反直觉的是后面的事。一年后，我们补了一套基于阿里云云效的正规 CI/CD 系统。按常理，正规系统上线，土法命令行该退役了。但我一条都没退，因为这一年跑下来我发现：在「需要判断」的环节上，命令行比系统化流水线更强。云效流水线只会报红等人，命令行能让 AI 自己修完再合并；流水线进不去生产现场，命令行能连生产库调试；流水线没有「停下来问人」这个状态，命令行有。 这篇文章讲的就是这件事：怎么从一无所有搭起三阶段，以及为什么补了正规系统之后，土法命令行反而成了不可替代的那层。先说结论——它不是 CI 系统的替代品，是 CI 系统够不着的那层「判断力」的载体，是流水线进不去、又必须有人盯的那些环节的兜底。 一、荒芜之地：没有 CI 的研发长什么样 先把现状摆清楚，不美化。 那时候改完代码，流程是这样的：自己机器上 mvn clean package 编译，过就推到远端；没单测、没集成测，过不过全靠手点页面，点过就算过；评审是口头一句「帮我看看那个接口」，没人真看就合了；部署是 scp jar 包到生产服务器，运营后台的 dist 目录手动传；回滚靠记忆——出问题翻聊天记录找「上次能跑的那个 jar 在哪」，没有制品库，没有版本号。 举个真实的场景。有一次改了佣金结算的接口，编译过了、手点了几下没问题就推了。第二天运营来报障，说某个商家的佣金算出来是负数。排查发现是新代码里一个边界条件没处理，但这个 bug 在「手点几下」的阶段根本触发不到——它得某个商家当月刚好有一笔退款，金额恰好等于某笔佣金，才会撞上那个边界。这种 bug，没有系统化的测试覆盖，靠人手点是抓不到的。 更要命的是部署的不透明。jar 包 scp 上去，systemctl restart，进程起来了就觉得「上线成功」。但有一次上线后，新代码其实根本没生效——旧进程没杀干净，还占着端口，新进程绑端口失败静默退出，健康检查命中的是旧进程。我们盯着日志以为一切正常，直到用户开始报障才发现线上跑的还是上一版。 这套流程最伤的不是慢，是不透明。谁改了什么、推上去的是哪个版本、现在线上到底能不能用，全是黑盒。半夜发完版，第二天用户报障，你才知道某个接口昨晚就挂了。 有人会说，这阶段直接上 GitLab CI 不就行了？算过账不划算。我们当时是 3 个人、单机部署、流量可预期，养一套 Runner、写 pipeline YAML、处理证书和并发缓存，投入产出比很差。更要命的是，CI 平台解决的是「自动跑测试」，但我们当时连测试都没有——上了 CI 也不知道让它跑什么。更需要的是一套轻量、能立刻拦住「黑盒」的机制，而不是一套重型平台。

一片荒芜时先做哪个:初创团队工程化基建的四层优先级

 included in 工程化
我从前公司带过来一套完整的工程化体系——监控告警、自动化测试、CI 流水线、Code Review 流程,什么都有。跳到初创团队的第一天,打开电脑,什么都没有。 没有监控系统,线上出没出事全靠用户投诉。没有自动化测试,改一行代码心里没底。没有 CI,部署靠手动 mvn package 打个 jar 包再 scp 到服务器。没有敏捷流程,需求口头传达,排期拍脑袋。 更难受的是,团队里没人觉得这些是问题。业务在跑,功能在加,「先把功能做出来」是唯一的优先级。 这种「一片荒芜」的局面,是很多从大厂跳到初创的人都会遇到的情况。难的不是「工程化有哪些东西」——这个大厂都教过你了。难的是:这么多东西都要做,先做哪个?后做哪个?什么能缓一缓? 我自己花了半年,一点一点把这套基建搭起来。现在回头看,如果让我重来一次,我会用一套完全不同的顺序。 先说结论:优先级不按重要性排,按投入产出比 + 阻塞解除排 很多人排工程化优先级,默认按「重要性」排:测试最重要(质量是生命)、监控也重要、CI 也重要、敏捷也重要……最后发现什么都重要,什么都想做,半年过去什么都没落地。 我的做法是换一个维度:不看这东西「重不重要」,看它「能多快见效 + 不做它会卡住什么」。 这套顺序的逻辑是:先做投入产出比最高的,再做能放大前面效果的。 监控第一:投入最小(一周搭完),见效最快(立刻能看到线上有没有事)。不做它,你后面做的所有东西都是瞎的——你不知道测试有没有跑、CI 有没有生效、部署有没有出问题。 测试第二:投入中等(一个月搭框架),但它解除的阻塞最大——没有测试,你不敢重构、不敢升级、不敢让新人碰核心代码。 流水线第三:投入较大(两周配 CI/CD),但它把前两步放大——监控的数据靠流水线部署的探针采集,测试的价值靠流水线自动执行才不会变成摆设。 敏捷第四:投入最持续(一直在磨合),但它放大团队协作效率——前面三步是技术基建,敏捷是让团队用起来这些基建的流程保障。 这四块落在一张投入产出比矩阵上,位置是这样的: 下面四节,我按这个顺序,逐个讲每一层我做了什么、为什么这么排、踩过什么坑。 第一层:监控——不是因为重要,是因为一周就能看见效果 荒原里最怕什么 初创项目最可怕的不是「有 bug」,是「不知道有 bug」。 我刚去的时候,有一次用户反馈「页面打不开」。运维去服务器看了一眼,说「CPU 正常啊」。开发去看日志,日志只有 console.log,什么有用信息都没有。最后排查了三个小时,发现是某个接口超时——但没有任何监控告诉我「这个接口超时了」。 这种「靠用户投诉发现问题」的模式,在大厂是不可想象的。大厂的监控体系会主动告诉你「P99 延迟涨了 50ms」,甚至在你之前就告警了。初创团队没有这个,所以问题永远是滞后发现的——等用户受不了来投诉,问题已经发酵了几小时甚至几天。 先搭什么,后搭什么 监控也分层,不能一上来就搞全链路追踪。我的顺序是: 基础设施监控先上(2 天):服务器 CPU、内存、磁盘、网络。用 Prometheus + Node Exporter,或者更轻量的方案(初创团队可以直接用云厂商的监控)。这一步解决「服务器还活着吗」的问题。 应用存活 + 核心接口监控(3 天):健康检查接口、核心业务接口的响应时间和错误率。这一步解决「服务能用吗、好不好用」的问题。 RequestId 链路追踪(1 周):给每个请求打一个 ID,串起网关、服务、数据库的日志,一条 RequestId 贯穿整条后端调用链。这一步解决「出了问题怎么查」的问题。 详细怎么搭,我写过一篇可观测性:RequestId 与监控告警,这里不重复。 我要强调的是为什么这个顺序:从粗到细。先确保「能发现有问题」(基础设施监控),再确保「能定位到哪个服务」(应用监控),最后确保「能查到具体哪次请求」(链路追踪)。反过来做的话,你会在还没搞清楚「服务器活没活」的时候,就去折腾分布式追踪——那是本末倒置。 告警降噪:比加监控更重要的事 监控搭起来之后,最容易犯的错是告警太多。 我刚加完监控那一周,告警群一天响几百次。CPU 超过 80% 告警、某个接口慢了告警、磁盘用了 70% 告警……一周之后,所有人都把告警群设成免打扰。然后某天凌晨服务挂了,没有一个人看到告警。

你的 E2E 只是在点 UI:真实业务多角色闭环测试 SOP

 included in 测试
我审过很多项目的 E2E 测试,十有八九长一个样:一个用户登录,点几个按钮,断言页面出现某个文本,通过。这种测试我称之为「单用户点 UI」——它能抓住「页面崩了」「按钮不见了」这类前端问题,但抓不住最危险的一类 bug:跨角色、跨端、跨时间的业务闭环出错。 举个真实的例子。社交模块里,A 给 B 点赞,B 应该收到一条通知。单用户 E2E 永远测不出这个——因为它只有一个用户,A 点赞后没人去查「B 的通知列表有没有多一条」。这种 bug 会静默潜伏,直到用户投诉才暴露。(这篇和测试纪律、覆盖率诚实是一套:那两篇讲「测试要不要写够」,这篇讲「写出来的测试到底有没有抓住真 bug」。) 我定了一套写 E2E 的标准做法,核心是让多个真实角色互操作,把断言落在数据上,跨端用握手衔接。这套 SOP 在我的多租户 SaaS 项目里跑了大半年,抓出过十几条单用户测试根本碰不到的闭环 bug。这篇按标准 SOP 文档的格式写明白。 一、这套 SOP 要解决什么问题 先划清边界——它只解决一类特定问题。 痛点:传统 E2E 测试以「单用户点 UI」为主,抓不住以下三类 bug: 跨角色业务闭环出错(A 操作 → B 应感知,但 B 没感知) 静默后端失败(前端显示成功,后端数据没落库或落错) 跨端数据传递断裂(小程序下单 → 运营后台查不到订单) 这三类 bug 的共性是:UI 层看着全绿,业务事实是错的。单用户、纯 UI 断言的 E2E 对它们完全失明。 这套 SOP 的目标:用「多真实角色 + 数据落库断言 + 跨端握手」组合拳,让 E2E 能抓住业务闭环级的错误,而不只是页面渲染错误。 适用范围:有明确业务角色划分、且角色间存在状态流转的系统。典型如社交(用户互相关注/互动)、电商(买家下单→卖家发货→平台结算)、SaaS(租户管理员→普通成员→平台运营)。 不适用:纯展示型网站、无状态工具类应用。这类没有业务闭环可言,单用户点 UI 就够了,套这套是过度设计。 二、核心原则:三条铁律 整套 SOP 由三条原则支撑,每条都对应一类传统 E2E 的盲区。

备份 ≠ 能恢复:3-2-1 备份与每月强制演练的完整 SOP

 included in 架构
我见过太多团队对「备份」的理解停留在「mysqldump 定时跑一下」。这种理解会害死人。备份和恢复是两件事,而真正决定数据安全的,是恢复——更准确地说,是你能不能在灾难发生时,确定性地把数据救回来。 (这篇和多租户数据隔离、事故即规范是一组:那两篇讲「怎么不让数据出问题」,这篇讲「万一出了,怎么救得回来」。) 我主导的多租户 SaaS 项目有一次差点翻车:一个清理 OSS 孤儿图片的定时任务,因为差集逻辑写反,批量物理删了一批用户图片。那次之后我定了一条规矩:任何数据资产,都必须有一套可执行的恢复 SOP,而且每月真刀真枪演练一次。这套 SOP 我跑了快一年,这篇按标准运维文档的格式把它拆开。 一、这套 SOP 的目的与边界 目的:保证 MySQL 数据库、OSS 对象存储、秘钥配置这三类核心数据资产,在任何单一故障(磁盘损坏、误删、云厂商区域故障、勒索软件)下,RPO ≤ 5 分钟,RTO ≤ 1 小时。 边界——什么不纳入这套 SOP: Redis:缓存和会话,可从 MySQL 重建,自身有 RDB+AOF 持久化兜底 Elasticsearch:业务索引可从 MySQL reindex 重建;AI 向量库重建成本高但属非事务数据 把边界划清楚很重要。什么都想备份等于什么都没备份好——资源有限,精力有限,必须聚焦在「丢了真的会死」的数据上。MySQL(交易数据)、OSS(用户内容)、秘钥(服务能不能启动)这三类是命根子,其它都能从它们重建。 适用场景:单实例或主从架构的中型 SaaS(日均 API 百万级、OSS 日增 GB 级)。如果你已经是多活架构,这套方案是降级的;如果你只是个人项目,这套是过度设计。 二、核心原则:3-2-1 与「演练即门禁」 整套 SOP 立在两条原则上,缺一条就垮。 原则一:3-2-1 备份原则。 这是备份领域的经典铁律,含义是: 3 份副本(生产数据 + 2 份备份) 2 种不同介质(比如本地磁盘 + 对象存储) 1 份异地(不同可用区,最好不同云厂商) 3-2-1 的精神是消除单点:任何一种介质、任何一个机房、任何一家云厂商出问题,都至少还有一份能用的副本。我的项目里,MySQL dump 同时存「异地从库本地磁盘 + 七牛 OSS + 阿里云 OSS」三份,OSS 对象在「七牛主 bucket + 七牛备用 bucket + 阿里云冷备」三处。这就是 3-2-1 的落地。

我用 AI 编码助手做全栈项目两年,主力是 Claude Code,但同一个项目里 Cursor、OpenCode、Codex、Trae 都会切着用。每个 IDE 都要配 rules、commands、skills,第一年里我反复在做同一件事:改一份 rules,去五个目录同步五遍。 这篇讲的就是怎么把这块最底层地基——配置怎么不重复——解决掉(和命令体系是配套的:那篇讲「有哪些命令」,这篇讲「这些命令怎么在多个 IDE 之间不重复地维护」)。 为什么不能只守着一个 IDE?没有一个 AI 助手在所有任务上都最优。我的体感:Claude Code 做长链路规划(写方案、拆任务)最强,Cursor 改 bug 时定位相关文件最顺手,Codex 适合挂后台跑批量重构,Trae 在某些中文语境下理解更准。任务不同,用的工具就不同。 这种重复不只是烦。它的真正代价是配置会漂移:Cursor 里的命令修了 bug,Trae 里还是旧的;Codex 加了新 skill,Reasonix 不知道。漂移积累到一定程度,AI 的行为就不可预测了——同一个 /kd-review 命令,在 Cursor 里收敛,在 Codex 里死循环。更隐蔽的是,漂移往往是静默的:你以为五个 IDE 行为一致,其实早就不一致了,直到某天在某个 IDE 里踩了坑才发现。 我把这套重复彻底干掉,沉淀成一套 SOP:一个 .agents/ 目录作单一事实源,五个 IDE 用不同策略同步,锁文件保证可复现。这篇按标准 SOP 文档的格式写,你照着能在自己的项目里复现。 一、这套 SOP 要解决什么 先说清楚它只解决什么问题,避免你套用到不合适的场景。 痛点:一个项目同时用多个 AI 编码助手时,rules(项目规则)、commands(自定义命令)、skills(技能)这三类配置需要在每个 IDE 的约定目录里都存在。手动复制会导致: 配置漂移:改了 A 处忘了改 B 处,行为不一致 版本失控:没有锁文件,换台机器装出来的 skill 集合不一样 心智负担:每次新增命令,要记得「同步到哪些地方」 这套 SOP 的目标是:改一处,全 IDE 生效,且可复现安装。
