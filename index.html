AI编程之我见

 included in 随笔
这是「前端 → 全栈 → AI 开发」这条成长线上的序章。从这篇开始，我会从一名全栈开发的视角，记录 AI 是怎么一步步走进我的日常编码、最后把我写代码的方式整个改了的。 老实讲，这篇我不想再写那种「AI 真好用，大家都来拥抱」的稿子。过去一年我天天在用 Cursor、Copilot、Claude Code，最大的感受其实不是"快了多少"，而是这件事：写代码的门槛和重心，正在悄悄挪地方。 下面就把这个判断摊开讲——结合我的真实项目经历、踩过的坑、以及后来形成的几条应对规矩。这一篇是调子，后面整个 AI 研发实战系列（认知转变、岗位转型、工程化治理）都会从这儿接出去。 工具盘点：不是谁都能打，先认清楚各自的定位 先把市面上几款主力工具的边界说清楚，免得你拿着锤子到处找钉子。 Cursor：我目前主力。它的定位不是"聊天框里贴代码"，而是把模型直接焊进编辑器——你在哪一行问，它就读哪一行的上下文。适合中型到大型改动：重构一段逻辑、写一个完整接口、跨文件改命名。代价是它对项目结构和规范比较挑，项目乱它会跟着乱。 GitHub Copilot：更轻，定位是"行内补全 + IDE 联想"。写一个函数写到一半它帮你补完下一行，或者注释里写句话它生成实现。适合写样板、补常规逻辑，不适合让它从零搭一个模块。 Claude Code：命令行里的 agent，能自己跑工具、读文件、改文件、跑测试。适合那种"帮我把这事整体做完，做完自己验证一遍"的场景，比如批量重构、跑通一个端到端流程。它的长处是上下文长、能自主迭代，短处是你得给它明确的边界，不然它容易自己跑偏。 国内还有 CodeGeeX、通义千问代码版这些，网络环境下更顺，定位和 Copilot 接近，按需选。 一句话：没有"最好"的工具，只有"这件事更该丢给谁"。 我日常是 Cursor 做主线、Copilot 补行内、Claude Code 干重活，按场景切。 起点：OpenLuban 项目，从试试看到离不开 我真正开始重度用 AI 编程，是从个人开源项目 OpenLuban（我离职后把大厂鲁班的架构经验重新开源实现的一套低代码平台，全新代码，与原公司系统无关）开始的。一个人扛大部分开发，技术栈横跨好几个： 前端 Vue3 + Nuxt3 后端 SpringBoot 偶尔 Node.js 写个 BFF 层 有时还要补点 golang 脚本处理数据 多种语言来回切，最容易卡壳的不是"会不会"，而是不常用的语法记混了，得翻文档、查博客，一卡就是十几分钟。这种卡顿单看一次不大，但一天攒下来特别消耗状态。 抱着试试看的心态下了 Cursor。本来只想拿来补补语法、调个小 bug，结果用了不到半天，我就开了 Cursor Pro。 转折点很具体：我把需求说清楚，比如「用 SpringBoot 写一个站点列表查询接口，带分页和条件筛选，关联用户表」，它直接给出可运行的代码，连异常处理、参数校验都写好了。这件事让我意识到，AI 接管的不是某个语言点，而是那一整层"查语法、套样板、对模板"的执行活。 从那以后，这类活我基本不自己写了。 效果：提速是真的，但更值钱的是别的 先说能拿数字说话的。以前写一个完整后端接口——实体类、Mapper、Service、Controller，再到调通——至少半小时；用 Cursor 之后，10 分钟搞定，结构还规整，格式不用手调。前端组件也一样，「写一个 Vue3 表单组件，输入框 + 下拉框 + 校验 + 提交按钮禁用逻辑」，它生成完我调一下类名和样式就能复用。

AIAI编程随笔AI研发实战

【AI 研发实战】AI 时代研发认知转变:知识不再是护城河,老兵反而成了红利

 included in AI编程
这是我「前端 → 全栈 → AI 开发」这条成长线在 AI 开发阶段的一篇。序章讲 AI 怎么走进日常,转型篇讲我往哪转。这篇要讲一个更底层的问题:当 AI 变得全知全能,研发到底还靠什么吃饭。 一、知识,曾经是研发最硬的护城河 干过几年研发的人都清楚,以前这行最值钱的就是知识储备。 一段冷门配置怎么写、某个框架的坑在哪、一个诡异报错怎么解——这些没有捷径,全是时间和血泪堆出来的。趟过才记得住,趟过才有发言权。 我见过很多资深工程师,核心竞争力就是「我踩过的坑比你走过的路还多」。初级查一个 Nginx rewrite 要一下午,资深张口就来。信息差直接换算成工资差。 学习周期也长。一门新语言从入门到能写生产代码,少说半年;一个新框架,把生态摸熟又要几个月。知识壁垒高,掌握它的人才值钱。这套逻辑过去十年都成立。 二、AI 把这道墙推平了 现在不成立了。 我的日常主力是 Claude Code,配 Cursor 做编辑,OpenCode 做补充。一个最直观的感受:以前要翻一天文档的事,现在一句话。 不熟的语法,问一句直接给; 冷门框架的用法,它比翻官方文档还快; 诡异报错,贴过去它不光解释原因,还顺手把改法写了。 当人人都能「问出答案」,「我会你不会」这道护城河就塌了。 这不是说知识没用,而是「知识的储备量」不再是稀缺品——它从你脑子里的库存,变成了工具里的自来水,谁都能拧开。 三、值钱的东西,往上挪了一层 墙塌了,但人还得吃饭。价值没消失,只是挪了地方。 我的判断是:执行层的知识被削平之后,真正值钱的能力往上挪了一层。具体是三样。 架构能力。 系统怎么分层、边界划在哪、哪一步该做、哪一步该砍。这种取舍 AI 给不了答案,因为它取决于你的业务、团队和约束。这部分我专门拆过。 判断力。 一个需求该不该做、做到什么程度、哪条路是死的。这是决策的活,AI 只能给选项,拍板得你来。 经验嗅觉。 一眼看出这段代码迟早出事、这个方案上线必挂。这种东西 AI 学不会,它来自亲身趟过的坑,不在任何文档里。 这三样有个共同点:它们都建立在大量知识之上,但本身不是知识。知识是地基,AI 帮你把地基铺平了;可上面那栋楼怎么盖,还得你自己设计。 这也是为什么新人暂时还顶不上老兵。不是因为老兵懂的 API 多,而是老兵知道什么场景该用什么、什么坚决不该做。 四、认知得跟着转:从「储备知识」到「调用知识」 工具变了,认知不跟着转,人就卡在原地。 旧的认知是:知识装进脑子里,装得越多越值钱。所以拼命学、拼命记,生怕漏掉一个 API、一个新框架。 新的认知是:知识在工具里,随时能调。脑子该腾出来做判断和决策。 不是不用学了,是学的重点变了。以前学的是「记 API」,现在该学的是「懂原理、懂取舍」——原理决定你问得出对的问题,取舍决定你拍得了对的板。 一个现实的变化:我现在的「全知全能」,不是靠我背得多,是靠我会用工具。任何不熟的语言、不熟的领域,丢给 Claude Code 跑一遍,心里就有底了。这在过去是一个人根本做不到的事。 五、老兵,反而最受益 这就到了这篇我最想讲的一点。 过去研发圈总有个隐忧:年纪上来,体力下降、知识老化,性价比不如年轻人。这套焦虑的底层逻辑是,价值锚在「知识储备 + 编码体力」上,而这两样过去确实随时间往下走。 AI 换了坐标系。在新坐标系里,重新看这三条。

AI研发实战研发认知职业成长

【AI 研发实战】AI 时代,什么能力对研发才是真正重要的

 included in AI编程
上一篇讲知识不再是护城河。这篇接着说一个更实际的问题:知识被抹平了,什么才是新的护城河。 我的判断很直接:AI 时代真正重要的研发能力,不是"会写代码",是"知道怎么正确地做事"。 一、vibe coding 的热闹和它的代价 AI 编程门槛降到地板之后,vibe coding 成了一种流行玩法——不读代码、不懂架构,跟 Claude Code 或 Cursor 聊着天把应用"聊"出来。 我能理解这种爽感。但作为经常要接手、维护这类系统的人,必须说一句实话:这些产物里有一大批是玩具,却被当成工业级软件在用。 危险不在 demo 能跑,而在于"能跑"这件事制造了一种错觉。能跑的代码,不等于能上线;能上线的,不等于能扛住流量;能扛住流量的,不等于半年后还改得动。这三层差距,vibe coding 一个都没跨。 具体表现在四件事上,几乎每篇 vibe coding 产物都中招: 安全:AI 写的鉴权、SQL 拼接、密钥处理多半有洞。一个 tenant_id 漏校验,就是跨租户数据泄露。 可维护:没规范、没分层、命名混乱。三个月后作者自己都读不懂。 可扩展:逻辑全堆在一个文件,一次请求同步干十件事。用户量一上来就瘫。 可验证:没测试、没契约、没监控。改一行,不知道弄坏了哪里。 玩具和工业级软件的分界线,从来不是"能不能跑",而是出了事扛不扛得住、查不查得清、改不改得动。 二、AI 解决执行,解决不了"做对" 先表明立场:我不是反对 AI。恰恰相反,Claude Code 是我的主力工具,研发账本里 4 个人 2 个月干出近百万行,没有 AI 根本做不到。我比多数人都更依赖它,也更清楚它的边界。 这个边界就是:AI 负责执行——把你想清楚的东西快速写出来。但"想清楚"这件事,它替不了。 更麻烦的是,AI 不只是解决不了"做对",它还会主动做错。这些都是我在真实项目里反复见到的: 它会做错误的架构决策——让它设计一个模块,它图省事把所有东西揉在一起,因为它优化的是"这一刻能跑",不是"半年后好维护"。 它会写出屎山——能跑就行,重复代码、硬编码、魔法数字,它不认为这是问题。 它会跳过测试——不盯着,它就不写。写了也是占位,真出问题测不出来。 它会悄悄删功能——发布新版本时,你才发现登录被它重构没了。 最后这一条不是假设。我复盘过一次"幽灵删除"事故:AI 在重构里删了一段关键逻辑,编译过、测试没报错,上线才暴露。没有经验和门禁兜着,这种事就是定时炸弹。 一句话:AI 让"写出能跑的代码"变廉价了,却让"写出正确、安全、可维护的代码"更依赖人——因为只有人知道什么叫"正确"。 三、真正值钱的几种能力 知识被抹平之后,值钱的能力全在"执行"这一层之上。我拆成五块,挨个说。 架构能力是底座。 知道怎么建一个高可用、可扩展、可稳定维护的系统。这部分 AI 顶多打下手,核心判断必须人来: 系统怎么分层,边界划在哪,哪一步该做、哪一步该砍; 多租户怎么隔离,资金怎么防双花,高并发怎么扛——这些微信分润和Flyway 治理里都讲过; 一眼看出哪种方案上线必挂。 前瞻性来自经验。 AI 只看当下这一刻能不能跑。有经验的研发看得见半年后:这个设计现在没问题,用户量翻十倍会不会塌?

AI研发实战研发认知架构能力

【AI 研发实战】AI 时代研发转型之我见:从前后端到全栈,再到产品开发专家

 included in AI编程
这是我「前端 → 全栈 → AI 开发」这条成长线进入 AI 开发阶段的第二篇。它要回答的不是「用什么工具」,而是 AI 时代里,一个工程师的价值到底该锚在哪。 研发的真正瓶颈:不是技术,是分工 跟同行聊下来,大家的状态出奇地一致:越干越累,产出却没见涨。问题不在努力程度,而在分工切得太碎。 一个功能从想法到上线,链路是这样的:产品写文档,前端做页面,后端写接口,测试补用例,运维发版,中间还夹着一堆评审和对接。 链路一长,沟通成本就高得吓人,而每个人都只看得见自己那一段。后端不知道前端为什么这么调,前端不知道后端为什么这么返,测试不知道这块改动到底影响谁。 这套分工,在业务稳定、需求可预测的年代是高效的。但现在需求又快又碎,它就显得特别笨重。真正的瓶颈早就不是你会不会某个技术,而是你能不能把一条链路从头到尾串起来。 从前后端,到全栈,再到「产品开发专家」 我自己这几年的路径,可以划成三段。 第一段,纯后端。 Java 那一套,接口、库表、并发,管好自己这一摊。 第二段,碰前端,长成全栈。 从一个页面到一整套小程序,慢慢变成别人嘴里的「全栈」。全栈这词儿被用滥了,但落到我身上,它解决了一个真问题:一个人能把前后端打通,沟通成本直接砍掉一大半。很多以前要开三个会才能扯清楚的事,自己脑子里跑一遍就定了。 第三段,从全栈,到产品开发专家。 光打通「前端 + 后端」还不够。一个真正能闭环的工程师,得能从产品定义开始,一个人扛下整条链路:产品定义(解决什么问题、长什么样、怎么验证)、全栈开发、自动化测试、DevOps(怎么发版)、线上运营(看数据、迭代)。 我给这个角色起了个名,叫「产品开发专家」。它不是「什么都会一点」的万金油,而是对一条业务链路有掌控力的人。万金油靠广度堆,产品开发专家靠掌控力撑。 为什么是现在:AI 削平了那道坎 一个人要把产品、开发、测试、运维、运营全干了,过去基本不可能。要么得是天才,要么得熬十年。一个人精力有限,链路越长,某个环节就一定是你不熟的,卡在那儿就推不动。 AI 把这道坎削平了一大截。 我不熟的设计稿,工具能直接生成原型。我不熟的某段 DevOps 配置,丢给 Claude Code,它讲清楚还顺手写好。测试用例这种费时费力的事,AI 能铺一大半。 「一个人打通全链路」第一次从不可能变成了有可能。 我现在做的那个多租户全栈项目,从 C 端小程序到运营后台再到 Java 后端,基本是一个人加一堆 AI 协同在推。换十年前,这得一个小团队。 最大的转变,是认知 这点最容易被忽略:真正难补的不是技术,是认知。 很多人一听「转型」,第一反应是「我是不是该去学个新框架、学门新语言」。不是。技术是最好补的,AI 时代尤其如此——你不熟的东西,问一句就有答案。真正要转过来的,是你怎么定位自己的价值。 以前,工程师的价值很大一部分绑在「写代码」上:你写得快、写得对、写得漂亮。但现在,代码这件事 AI 越来越能干。 那价值往哪儿挪。我的判断是,从「执行」挪到「决策」:定义问题(这功能到底该不该做)、设计系统(架构怎么分层、边界在哪)、把控质量(怎么保证 AI 写出来的东西是对的)。 执行交给 AI,决策留给人。这听起来像甩锅给 AI,其实恰恰相反:决策比执行难多了,也更值钱。一个能把问题定义清楚的人,带着 AI 干活,产出顶过去半个团队;一个只会闷头执行的人,效率再高,也只是个更快的「手」。 这不是裁员叙事 最后把话讲清楚,免得被误读。 我讲的这套转型,不是「AI 要替代工程师」,也不是「以后只要产品开发专家,其他岗位裁掉」。我团队里照样有深耕某一端的专家,深度是替代不了的。 我想说的是:对个体工程师而言,认知升级的窗口现在打开了。你可以选择继续在某一端做到极致,也可以选择往全链路拓,变成能对一整条业务负责的人。后者这条路,AI 让它变得前所未有地可行。 转型的核心从来不是学某个技术,而是搞清楚:在这个新工具面前,你的价值到底锚在哪。 后面这个系列,我会把我们这套「AI 协同研发」的真实做法一篇篇拆开讲——工作流怎么搭、质量怎么兜、规范怎么固化、事故怎么变规则。这篇算个总纲,先把方向立住。

AI研发实战研发转型全栈

【AI 研发实战】4 个人 + AI,2 个月交付近百万行:一份真实的研发账本

 included in AI编程
前面几篇一直在讲认知怎么转、转型往哪转。这篇不谈观点,只摆账本。 「老兵 + AI 杠杆」这个判断光讲道理没人信。我用我们刚交付的一个真实项目,把数据拆开:4 个人的团队,2 个月,做出了什么。 一、先看账本:4 个人干出了什么 这是我们这个多租户社交电商 SaaS 项目的实测数据,2 个月周期,不含重构掉的、不含被打回来的: 维度 数据 周期 约 2 个月 代码提交 560+ 次,日均 14 次 后端 2100+ Java 文件 / 30 万行,330+ Controller、540+ Service 数据库 850 张表,380+ 个迁移脚本 前端 C 端 530+ Vue 文件 / 19 万行,400 个页面、20+ 业务域 运营后台 260+ Vue 文件 / 8.6 万行 测试 后端 400+ 测试文件、前端 650+ 测试文件,合计近 19 万行测试代码 业务总量 近百万行(业务 + 迁移 + 测试) 这个体量,过去要十几人的团队、半年起步。我们 4 个人,2 个月。

【AI 研发实战】微信分润从 0 到 1:CPS 分账、提现状态机与一次冲正降级

 included in AI编程
上一篇把总账立住了。这篇挑它最硬的一块——微信分润——拆给你看。 分润为什么难。一笔钱要拆成几份、分几次、走不同的通道、还得多租户隔离,搞不好就是资金事故。这是整个项目里我盯得最紧、门禁立得最硬的部分。 一、先看这笔钱怎么流 多租户电商里,一笔订单的钱不是直接进商家口袋。微信支付支持「分账」——付款时先冻结到特约商户,确认收货后再按比例拆给平台、达人、商家。 钱大致这样流动:用户支付时标记 profit_sharing=true,资金冻结到店铺特约商户池;确认收货后触发分账,平台佣金即时分润,达人佣金延迟分账(留在达人冻结池),商家货款暂时冻结;达人提现审核通过后,才真正把达人那一份分账到零钱;最后完结分账,解冻商家货款可提。 关键决策是达人佣金为什么延迟。确认收货那一刻,达人可能还没签约分账接收方、可能退款、可能违规。所以钱先标 UN_SPLIT 留在冻结池,等提现审核通过才真正分账。这就是后面提现状态机的来由。 二、分账比例:基点(bp)而不是百分比 钱的事,小数点算错就是事故。我们全程用基点(万分之一),绝不用浮点百分比。 TenantSplitConfigService 里分账比例是这样定义的: public record SplitConfig( String tenantId, @Min(0) @Max(5000) int platformCommissionRateBp, // 平台抽成 @Min(0) @Max(5000) int talentCommissionRateBp, // 达人佣金 @Min(0) @Max(5000) int otherSplitRateBp, // 其他分账 @Min(0) @Max(5000) int totalSplitLimitBp, // 总分账上限 boolean enabled, ...) { public int totalRateBp() { return platformCommissionRateBp + talentCommissionRateBp + otherSplitRateBp; } }几个硬约束。 比例上限 5000bp(50%),单方不能超过。 总比例守门 30%:totalRateBp() > 3000bp 时自动降级为即时分账,不走达人延迟分账,避免分账方拿太多、商家货款不够解冻。 配送费单独算:次日达有固定金额,配送中心和自提点按 bp 分,合计不超 10000bp。 为什么用基点。整数运算,不存在 0.
