项目规范写进代码底层:让新人和 Agent 开机即用

xiaoshuai1024 included in AI编程
我见过太多份「项目规范」,它们最后都死在同一个地方:某个 wiki 的角落里,落满灰尘,最后一次更新是半年前,内容早就和代码对不上了。 新人入职第一周没人看,三个月后也没人看,直到踩了坑,有人恍然大悟"哦这事儿其实文档里写过"——可那又怎样,文档不会自己跳出来拦你。 不能自己执行的规范,就不是规范,是建议。这篇回答一个具体问题:项目规范怎么写,才不会变成一堆没人看的废话——答案一句话,把它从「给人看的文档」改成「机器能执行的代码」。展开讲,就是我们在仓库里固化的四层规范:规则文件、技能库、运行时 hook、启动清单。 我见过太多份「项目规范」,它们最后都死在同一个地方:某个 wiki 的角落里,落满灰尘,最后一次更新是半年前,内容早就和代码对不上了。 新人入职第一周没人看,三个月后也没人看,直到踩了坑,有人恍然大悟"哦这事儿其实文档里写过"——可那又怎样,文档不会自己跳出来拦你。 不能自己执行的规范,就不是规范,是建议。建议这种东西,在人忙的时候、在 AI 自由发挥的时候,一律等于没有。 这个判断不是我一开始就有的,是被逼出来的。项目跑起来后,我开了大量并行 Agent,每个都是「零背景新人」。最初我用传统思路写规范文档,结果 Agent 全程不看,该怎么犯还是怎么犯。直到我把规范写成 hook、写成规则文件,Agent 才真的「守规矩」了。这个转折让我彻底想明白:规范必须可执行,否则在 AI 时代连人这关都过不了,更别说 Agent。 所以我们的做法是——把项目规范写进代码底层,让它机器可读、运行时可执行。这一篇就讲我们怎么做的,以及它怎么让新人和新 Agent 都做到"开机即用"。 新人最痛的不是技术,是"这事儿在我们这该怎么干" 先说我们到底在解决什么问题。 新人入职一个项目,真正卡住他的,通常不是某个框架不会用、某个语法没见过——那些查文档就行。 卡住他的是一堆项目专属的隐性规矩: 这个字段为什么要这么命名 那个 SQL 为什么必须带 tenant_id 提交前得先跑哪个脚本 危险操作有哪些不能碰 这些东西没写在任何教程里,只在老同事的脑子里。 更关键的是,这种隐性知识的传递成本极高。每来一个新人,老人就得口述一遍;老人忙了,新人就只能靠踩坑自学。 而且踩坑自学有个恶果:学到的东西是「碎片化」的。新人踩了 A 坑学到 A 规矩,踩了 B 坑学到 B 规矩,但他不知道还有 C、D、E 规矩存在——直到他也踩了 C、D、E 坑。这种「靠事故驱动学习」的方式,代价是真实的生产事故。规范固化后,新人不用踩坑就能知道所有规矩,因为规矩在代码里、在 hook 里、在技能里,系统性地摆在面前。 AGI 时代又多了个新麻烦——每开一个新的 agent,它也是一个"零背景新人",而且它不会主动来问你。 Agent 比新人更难管的一点是:新人踩坑会有痛感(被骂、被回滚、要背锅),他会长记性;Agent 没有痛感,这次踩了坑,下次新开的 Agent 还是会踩同一个坑,因为它是无状态的、不复用经验的。所以面向 Agent,「踩坑 → 长记性」这条路根本走不通,必须把记性固化成机器可读的规则,让每个新 Agent 启动就继承。 一旦你把"AI 当随时入职的新人"来 onboard,立刻就能看清:这事儿光靠文档不够,得靠机器能自动加载和执行的东西。 传统规范为什么不行 传统的规范有三个硬伤,改不掉的那种。

事故复盘的终点是规则不是文档:5 个真实案例的固化路径

xiaoshuai1024 included in AI编程
出事 → 群里炸锅 → 复盘会 → 写一份 wiki → 链接甩群里 → 三个月后没人记得 → 同样的坑再踩一次。我见过太多事故复盘,流程都长这一个样。 后来我想明白一件事:一份规范的价值,取决于它能不能被机器执行。多数团队的事故复盘产出物是「一份文档」,我的回答是「一条机器可执行的规则」。文档会过时、没人看,规则会一直生效——在 AI 协同研发的高频交付节奏下,这个差距被放大:交付越快,踩坑越快,如果每个坑只变成文档,挡不住下一次同样的坑。 我用 5 个真实事故案例,讲清楚「事故 → 规则」的转化路径:不同类型的事故固化成什么载体(hook/脚本/Agent规则)、怎么判断该用哪种、以及怎么让这套机制自我生长。 事故复盘最常见的死法 我见过太多事故复盘,流程都长一个样: 出事 → 群里炸锅 → 复盘会。 写一份 wiki → 链接甩群里。 三个月后没人记得,同样的坑再踩一次。 我自己也写过这种 wiki。后来我想明白一件事:一份规范的价值,取决于它能不能被机器执行。 挂在 wiki 上的教训,是"希望下次有人记得"。 固化成代码的教训,是"下次机器直接拦你"。 这两者的 ROI 差了一个数量级。 这个差距为什么是一个数量级。因为 wiki 依赖三个不确定环节:有人写(写的人得有时间)、有人看(看的人得有动力)、有人记(记的人得不出错)。三个环节任何一个断了,规范就失效。而代码规则只依赖一个环节:写一次(之后就自动跑)。一个是三连依赖,一个是一次投入,可靠性差一个数量级。而且 wiki 的「失效是静默的」——没人看也不会报错,你以为规范在生效其实没有;代码规则的「失效是显性的」——hook 没了、脚本坏了会立刻报错或构建红,你能马上发现。静默失效 vs 显性失效,这是 wiki 规范和代码规则的另一个本质差距。 所以这两年我们团队立了个规矩——事故即规范。每起线上事故或险情,复盘完不是写文档,是落成一条可执行规则。下面是我这几年攒下来的真实转化路径和几个例子。 规则的三种载体 先说方法论。我们把事故教训固化成三种东西,各有各的适用场景: git hook(运行时拦截):针对"绝对不能发生"的动作,比如推错分支、跑危险命令。在动作发生的那一刻直接拦下,不留任何侥幸。 脚本(提交前门禁):针对"跨层契约对不齐"这类编译器管不到的错误,放在 pre-commit,提交前就扫描。 agent 规则(评审时检查):针对需要"理解语义"才能判断的问题,写进 Agent 的规则文件,让评审回路在每次提交时自动查一遍。 判断标准很简单:这条规则能不能不靠人、自己跑起来。能,就固化;不能,就老老实实写文档,但别指望它生效。 这三种载体的本质区别是「拦截时机」。git hook 在「动作发生的瞬间」拦截,是最早的关口;pre-commit 脚本在「提交前」拦截,次早;Agent 评审规则在「评审时」拦截,最晚。关口越早,问题暴露越早,修复成本越低——一个危险 push 在 hook 拦下,零成本;同一个问题如果放到评审才发现,代码已经写完、可能还跑了测试,返工成本高得多。所以载体的选择原则是:能早拦的别晚拦。能用 hook 的不用脚本,能用脚本的不留给 Agent。

一行漏掉的 tenant_id 造成的越权,以及我织的三层兜底网

xiaoshuai1024 included in 架构
多租户数据隔离是 SaaS 系统能不能交付的底线，绕不开。这篇记录的是这套隔离体系里最隐蔽的一个漏洞，以及怎么把它从"靠人盯"改成"层层兜"。 先说一句结论:多租户的安全感,从来不来自"开发都很细心",而来自"假设迟早有人会漏,提前把网织好"。后面所有内容都是为这句话找证据。 一次越权,根因只有一个 JOIN 那次线上反馈很直接:某个客户说"我能看到别人的数据"。多租户系统听到这句话,基本可以判定为越权。顺着链路翻,查到一条排查 SQL,对着前端返回的字段一比,定位到一段 JOIN。 排查路径其实不复杂。先用前端能看到的那条"不该出现"的数据,反查它来自哪张表、哪个接口、哪条 SQL;再把这条 SQL 在测试库里重放,逐段拆 JOIN,看哪一段把别人的数据带了出来。定位到具体那段 SQL,大概花了不到半天——真正花时间的是后面想清楚"为什么会漏"。 根因是:某个 JOIN 关联两张业务表时,只在外层带了 tenant_id 条件,JOIN 的那张表没带。结果租户 A 的查询,把租户 B 的关联行也带了出来。 不是什么精妙的 0day,也不是鉴权被绕过。就是一段 JOIN ... ON a.id = b.aid,后面漏了 AND a.tenant_id = b.tenant_id。 修复很快,改一行加上去就完事。但这件事真正风险在于:代码能跑、单测能过、功能自测也没问题——平时你压根不会想到去构造"跨租户数据"的用例,它只在真实多租户、真实数据并存的线上才会爆。 这种 bug 还有个更阴险的特点:它不会报错。大多数 bug 会抛异常、会返回错误码、会让测试红,至少告诉你「这里不对」。但漏了 tenant_id 的 JOIN 不会,它正常返回数据,只是返回的数据里混进了别人的行——程序觉得自己成功了,用户看到了不该看的,系统毫无察觉。如果不是那个客户主动反馈,这个漏洞可能还要藏很久。「不报错的 bug」是最危险的,因为所有的自动化防线(测试、监控、告警)都建立在「出错会报出来」的前提上,前提没了,防线就瞎了。 这也是为什么多租户越权的事故,往往是「客户先发现」而不是「团队先发现」。团队的自测环境里租户少、数据简单,跨租户数据混在一起的概率低;线上的租户多、数据杂,混在一起的概率高,而且只有真实用户才会去点那些「不该看到的数据」。等客户反馈时,数据已经暴露了一段时间——这个时间窗口的长度,就是信任损失的量级。 行级隔离的标配:为什么选它,代价是什么 先把我们的隔离方案交代清楚,后面才好聊为什么会漏。 业界多租户隔离大体三条路:独立库、独立 schema、行级隔离。我老老实实把它们的代价摆出来,你就能理解我们为什么选行级。 独立库:每个租户一个物理数据库。隔离最干净,但运维成本爆炸——租户一多,实例数跟着涨,备份、监控、迁移都得一个租户一个租户搞;跨租户报表、平台级统计要跨库 join,基本写不下去。除非单租户数据量极大、合规要求极严(比如金融大客户),否则扛不住。 独立 schema:一个库里每个租户一套 schema。比独立库省实例,但表结构变更要做 N 次 DDL,租户越多越痛;而且跨租户查询还是麻烦,得动态拼 schema 名。 行级隔离:所有租户的数据混在一张表,靠 tenant_id 这列区分。运维最省心(一套表结构、一次 DDL),跨租户查询天然支持,扩展租户就是加一行配置。代价就一个:每一条 SQL、每一个 JOIN 都得记得带 tenant_id。 我们这套选的是行级隔离,业务表都带 tenant_id,所有查询、所有 JOIN 都必须带条件。取舍很清楚:用"每条 SQL 都要小心"换"运维和成本可控"。对中小客户为主、租户数量会涨的 SaaS,这笔账划算。

权限校验散在每个接口里:我把它收成注解 + 过滤器链 + 契约闸

xiaoshuai1024 included in 架构
权限体系最怕的不是「没做」，而是「散着做」——这个接口开头一个 if (user.hasRole("admin"))，那个接口中间塞一段 if (!user.can("edit"))，还有几个接口压根没校验，靠「前端按钮不显示」硬撑。 把后端的权限校验从「散落在每个接口里的 if」收拢成一条声明 + 校验 + 对齐 + 兜底的链路，是用机制替代人记的工程判断。这篇讲怎么收。 「散着做」到底贵在哪 我接手过一个运营后台,权限校验是这种画风:这个接口开头一个 if (user.hasRole("admin")),那个接口中间塞一段 if (!user.can("edit")),还有几个接口压根没校验,靠「前端按钮不显示」硬撑。 老实讲,这种代码最致命的不是「能不能跑」,而是谁都不敢动。一段判断摆在接口里,三个月后你想删它,第一反应不是「这判断对不对」,而是「删了会不会越权」——因为没人知道这段判断是踩了哪个坑加的,删了出了事算谁的。于是它就赖在那儿,越来越多,越来越乱。 代价是具体的: 谁也不敢删那段判断,因为谁也不知道删了会不会越权。 最后代码里堆满「祖传 if」,谁动谁背锅。 前端按钮不显示,不等于后端没校验漏洞。 按钮显示是前端逻辑,接口鉴权是后端职责。黑客根本不经过你的前端,直接打接口。把安全寄托在「按钮藏起来了」,等于没做。 几个接口漏校验,纯靠运气没出事。 接口多了之后,「哪个加了哪个没加」全靠人记。今天没出事,是因为没人想到去打那个接口;明天被人扫到,就是事故。 权限和业务逻辑搅在一起,改一个判断得读懂整个方法。 这段判断嵌在业务代码中间,你要重构这个方法,就得先把权限逻辑摘出来——但摘出来的过程中,你怎么知道没摘错? 所以判断标准很简单:如果你问「这个接口的权限规则是什么」,得去翻业务代码才能回答,那就是散着做。 真正该有的状态是,看一眼注解就知道。 「散着做」还有一个被严重低估的成本:权限审查没法做。合规审计要求「列出所有需要 admin 权限的接口」,如果权限散在代码里,你得 grep 全项目,还可能 grep 不全(因为有些判断不是 hasRole("admin") 这种字面量,而是 user.getLevel() > 5 这种隐式的)。注解声明的好处在这里就体现了——@RequirePermission 是结构化的,grep 一下注解就能列出所有接口及其权限要求,审计报告几分钟就出。散着做的项目,这种审计能折腾一周还不敢保证全。 散着做还会让「权限测试」无从下手。测试要验「无权限的用户调这个接口应该 403」,但如果权限判断散在业务逻辑里,测试得把业务逻辑跑通才能验到权限——有些权限判断在业务流程的很后面(比如「提交订单时检查用户有没有下单权限」),测试得走到提交那一步才能触发。注解声明在接口入口就拦,测试直接调接口验 403,不用跑业务流程。权限测试的效率,差了一个数量级。 推倒重做时,核心思路就一句话——权限跟着接口走,不跟着人记。 注解声明,让权限长在接口上 定义一个注解,大概长这样: @RequirePermission("ops:order:refund") @PostMapping("/order/refund") public Result refund(...) { ... }注意那个权限 key:ops:order:refund。它不是随手写的字符串,而要在迁移脚本的权限种子数据里先注册过。这一条最关键,后面要出事就出在这。 key 的命名是有规范的,我们整个团队统一用「资源:动作」的三段式:{系统}:{资源}:{动作},比如 ops:order:refund(运营后台 / 订单 / 退款)、ops:order:export、fin:bill:audit。规范本身不复杂,但它解决三件事: key 在种子数据里必须先注册。 没注册的 key 走到运行期要么报错要么放行,两种结果都不好。所以 key 是「先在种子表里声明,再在注解里引用」,顺序不能反。 和接口绑死,新人不用通读业务。 新人看接口签名就知道「这接口吃什么权限」,不需要去翻业务逻辑、不需要去问老人。这点对团队扩张特别值钱。 权限信息和接口定义不再分家。 改接口的人顺手就能看到权限要求,删接口的人知道顺手把种子 key 也清掉。 有了注解,接口要什么权限就是声明出来的:

Flyway 迁移踩坑复盘:版本号撞车让服务起不来

xiaoshuai1024 included in 后端
某天早上我接到一个告警:某个服务启动不起来。日志翻到最后,就一行 Flyway 报错——迁移版本号对不上。 登上去一看,前一天两个人各自合了代码,各自加了一个 V12__xxx.sql。版本号撞车,Flyway 直接罢工,服务起不来。 排查过程:从"服务挂了"到"找到撞车" 这次事故虽然定位不难,但排查过程值得讲——因为典型的"启动失败"排查路径就是这样: 第一反应:看日志最后几行。 日志末尾就一行红字:Validate failed: Migration version mismatch for version "12",直接告诉你 Flyway 校验失败。这一步省了至少 10 分钟。 第二反应:grep V12 找撞车文件。 ls db/migration/ | grep V12 一跑,出来两个:V12__add_user_field.sql 和 V12__add_order_index.sql。撞车确认。 第三反应:看 git log 倒查谁合的。 git log --oneline -- db/migration/V12* 发现两个文件分别在两次 PR 里合入,review 时都没注意到对方用了同一个号——因为 review 时只看自己那个 PR 的 diff,看不到另一个 PR 已经合了同名号。 根因复盘(5 Whys): Why1:为什么服务起不来?因为 Flyway 校验失败,版本号不唯一。 Why2:为什么版本号不唯一?两个人各自用了 V12,合并时没有自动检测。 Why3:为什么合并时没检测?因为 CI 只跑单测,没做迁移文件唯一性校验。 Why4:为什么没做唯一性校验?团队觉得"靠约定就行",没把它当工程问题。 Why5:为什么"靠约定"不可靠?因为多人协作时,约定无法强制——每个 PR 看自己的 diff,看不到全局,撞车必然发生。 根因清晰:不是谁粗心,是治理缺位——把本该机器化的协调问题丢给了人。 5 Whys 走到第五层,结论就清楚了:这不是"下次细心点"能解决的事故。两个人都在自己 PR 的视野里做事,谁都没错,但合起来就撞车。只要协调问题还压在人身上,撞车就是迟早的事,区别只是这周撞还是下周撞。 真正的修复不是叮嘱大家"小心版本号",而是把唯一性校验做成 CI 的硬门禁、把版本号换成不需要协调的时间戳——让机制替人把这道关守住。

线上报警却找不到那条日志:RequestId 串起可观测性链路

xiaoshuai1024 included in 架构
用户反馈"下单失败",登上去一查:前端 console 干净,network 显示 500,后端日志同时间段上千条 ERROR。哪一条是这一单的?不知道。堆再多工具也救不了这种痛——可观测性的关键不是工具多,是它们能不能串成一条能排查的链路。 线上出问题,最怕的不是错,是"不知道哪儿错" 做过稳定性的人大概都有过这种时刻:用户反馈"系统卡了"“报错了”,你登上去一看,前端没报错日志,后端日志一大堆,翻半天翻不到那一条。 最后只能去问"什么时候、什么操作、哪个页面",靠用户回忆拼现场。 这种痛感,堆再多工具也救不了。我对这件事的判断很明确:可观测性不是工具数量的问题,是这些东西能不能串成一条链路。 工具再多,各管各的、对不上号,排查时还是两眼一抹黑。你上三个 APM、两个日志平台,它们之间没有共同的 requestId 把同一次请求串起来,排查时还是得靠"猜"——猜这次请求在 A 工具里是哪条、在 B 平台里又对应哪条。工具数量解决不了对齐问题,串成链路才能。这篇文章,复盘我们团队从"堆工具"到"串链路"走了哪几步。 说个真实场景,你就能体会"串不起来"有多痛。某次用户反馈"下单失败",我们登上去一查:前端 console 干净、network 显示 500、后端日志里同时间段有上千条 ERROR。到底是哪一条对应这个用户?不知道。只能让用户复述操作时间,翻着时间戳一条条对,半小时才定位。后来铺了 requestId,同样的场景:用户报障时把错误提示里的 requestId 发过来,后端日志一搜,这一单从进入拦截器、走到下单 Service、在哪一步抛了 CONCURRENT_CONFLICT,完整轨迹一条线拉出来,3 分钟搞定。这就是"串成链路"和"堆工具"的效率差。 第一根线:全链路 RequestId 最便宜、收益最高的一件事,就是每个请求注入一个唯一的 RequestId,然后让它贯穿这个请求经过的所有日志。 听起来简单,但它解决的是排查里最要命的问题:对齐。 前端报了一个错,后端到底走没走到、走到哪一步挂了——以前靠猜,现在按 RequestId 一搜,前因后果全在一条时间线上。 我们团队的规矩是:排查线上问题第一件事,就是拿 RequestId 去后端日志里捞这条请求的完整轨迹。没有这个 id,前后端日志就是两座孤岛——前端日志记的是"页面 A 报了错",后端日志记的是"接口 B 抛了异常",两边都对,但你没法证明"页面 A 的错就是接口 B 的异常引起的"。中间可能隔着网关、缓存、异步任务,任何一个环节都可能是断点。RequestId 把这条链路缝起来,从用户点的那一刻到后端落库,一条 id 串到底。 RequestId 的生成也有讲究:前端生成优先,后端兜底。正常情况下前端 axios 拦截器给每个请求生成一个 id 放进 header,后端 Filter 取出来用——这样同一个 id 贯穿前后端。但如果请求没经过前端(比如第三方直接调后端、或 MQ 消费),header 里没 id,后端 Filter 兜底生成一个。两段逻辑都在那个 resolveRequestId 方法里,优先用外面传进来的,没有才自己造。这个顺序保证:同一次请求,前后端用同一个 id,绝不串号。
