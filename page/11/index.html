AI 编程的门槛和重心:从提速 40% 到不再自己写样板

xiaoshuai1024 included in 随笔
老实讲，这篇我不想再写那种「AI 真好用，大家都来拥抱」的稿子。过去一年我天天在用 Claude Code、Cursor、OpenCode、Copilot，最大的感受其实不是"快了多少"，而是这件事：写代码的门槛和重心，正在悄悄挪地方。 下面就把这个判断摊开讲——结合我的真实项目经历、踩过的坑、以及后来形成的几条应对规矩。这一篇是调子，后面整个 AI 研发实战系列（认知转变、岗位转型、工程化治理）都会从这儿接出去。 下面就把这个判断摊开讲——结合我的真实项目经历、踩过的坑、以及后来形成的几条应对规矩。这一篇是调子，后面整个 AI 研发实战系列（认知转变、岗位转型、工程化治理）都会从这儿接出去。 工具盘点：不是谁都能打，先认清楚各自的定位 先把市面上几款主力工具的边界说清楚，免得你拿着锤子到处找钉子。按真实使用频率排序，我的日常主力排序是 Claude Code > Cursor > OpenCode > Codex / Copilot，理由是各自的场景适配不同。 Claude Code（主力，使用频率最高）：命令行里的 agent，能自己跑工具、读文件、改文件、跑测试、自主迭代验证。适合「帮我把这事整体做完，做完自己验证一遍」的场景——批量重构、跑通端到端流程、从需求到可运行代码一气呵成。它的长处是上下文长、能自主迭代、敢端到端接活，短处是你得给它明确的边界，不然它容易自己跑偏。我现在但凡能端到端丢的活，首选它。 Cursor：把模型直接焊进编辑器——你在哪一行问，它就读哪一行的上下文。适合中型改动：重构一段逻辑、写一个完整接口、跨文件改命名。代价是它对项目结构和规范比较挑，项目乱它会跟着乱。 OpenCode：开源命令行场景，或者需要本地可控时切过去。定位接近 Claude Code，胜在可控性。 Codex / GitHub Copilot：更轻，定位是"行内补全 + IDE 联想"。写一个函数写到一半它帮你补完下一行，或者注释里写句话它生成实现。适合写样板、补常规逻辑，不适合让它从零搭一个模块。 国内还有 CodeGeeX、通义千问代码版这些，网络环境下更顺，定位和 Copilot 接近，按需选。 一句话：没有"最好"的工具，只有"这件事更该丢给谁"。 我日常是 Claude Code 干端到端的活、Cursor 做编辑器内的中型改动、Copilot 补行内补全，按场景切。 工具切换有个隐性成本要提:每个工具的上下文不共享。你在 Cursor 里喂的上下文,切到 Claude Code 得重新喂一遍;Claude Code 里跑通的逻辑,回到 Cursor 编辑时它不知道你刚才干了什么。所以工具不是越多越好,选两三个主力、用熟用透,比装一堆每个都半生不熟强。我固定 Claude Code + Cursor 两个主力,Copilot 做辅助,这个组合磨合够了,切换的摩擦最小。工具多了,光在工具间切来切去就耗掉不少精力。 起点：OpenLuban 项目，从试试看到离不开 我真正开始重度用 AI 编程，是从个人开源项目 OpenLuban（我离职后把大厂鲁班的架构经验重新开源实现的一套低代码平台，全新代码，与原公司系统无关）开始的。一个人扛大部分开发，技术栈横跨好几个： 前端 Vue3 + Nuxt3 后端 SpringBoot 偶尔 Node.

SSR 服务高可用实战:从 QPS 跃升到三层降级兜底

xiaoshuai1024 included in 架构
一个没做任何优化的 Nuxt3 SSR 服务，单实例 QPS 只有 200 左右——每个请求都要在 Node 里跑完整渲染，串行打 3 到 5 个后端接口，光等数据就 500ms 到 1 秒，CPU 大半时间在 idle 等 IO。业务流量一上来，高峰直接 502。这篇记的就是怎么把这样一台裸 SSR，从 QPS 200 一路拉到能扛高并发、且「永不宕机」：CDN 剥静态、接口缓存兜后端故障、三层降级保命、前端按需渲染。 一、服务架构基础：Nuxt.js SSR实时渲染服务架构 本次优化的核心载体是基于 Nuxt3 的 SSR 服务。整体架构以 “Node.js 实时渲染” 为核心。 它承接前端请求、完成服务端渲染与数据聚合,并联动 CDN、Nginx、BFF 层、后端服务,形成完整的请求链路。 核心架构链路如下: 用户请求 → CDN(静态内容缓存) → Nginx(负载均衡、探活、请求转发) → Nuxt.js SSR 服务(实时渲染、数据请求、兜底处理) → BFF 层(流量承接、数据聚合、接口缓存) → 后端服务(核心业务逻辑、数据持久化)。 核心职责上,Nuxt.js SSR 服务负责页面模板渲染、前端资源打包、服务端数据请求与整合,同时承接前端错误处理与兜底逻辑。它是连接用户与后端服务的关键枢纽。 二、核心问题：流量增长与SEO优化的双重挑战 业务规模扩大后,Nuxt.js SSR 服务面临两大核心痛点,严重影响用户体验与业务增长,是优化的出发点: 2.1 流量快速增长带来的性能与稳定性压力 业务推广与用户规模扩大,带来请求量激增。原有 SSR 服务存在明显瓶颈: 单实例 QPS 仅 200 左右,高峰时段出现请求排队、响应超时。 部分后端服务异常时,SSR 服务无缓存兜底,直接返回错误页面。 实例故障时无法快速切换,导致服务不可用,影响用户留存。 2.
