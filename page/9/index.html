全栈架构 × AI 研发实战

业务代码零框架依赖:自研 @AiTool 桥接 Spring AI Tool Calling

xiaoshuai1024 included in AI编程
AI 助手要"能干活"而不只是"能聊天",靠的就是 Tool Calling(工具调用)。这篇讲我怎么把工具定义和 Spring AI 解耦:业务同学写起来零框架负担,换框架一行不动,写操作有确认门禁,每次调用全审计。 一、要解决的核心问题 我们的多租户 AI 助手要服务运营同学。运营同学说的不是"帮我查订单"这种结构化指令,而是自然语言:“今天的订单怎么样"“把这几个待审的代下单批一下"“这几个订单统一备注成已沟通”。 对模型来说,这些话背后是不同的业务动作。难的不是对话,是怎么把自然语言安全地落到真实业务方法上。这中间有几个工程问题必须先回答: 谁决定调哪个工具? 不能写 if 用户问订单就查订单 的硬编码分支。自然语言的组合空间是无限的,if-else 永远写不完。 工具怎么定义? 业务同学写一个工具,要不要被 AI 框架的 API 绑架?今天用 Spring AI,明天换 LangChain4j,要不要把所有工具重写一遍? 多租户怎么穿透? A 租户的运营说话,工具只能查 A 租户的订单,这个隔离边界怎么从对话层一路传到业务 service? 写操作怎么兜底? 模型要是手一抖调错写操作(比如批量改错订单),怎么拦?Agent 直接改生产数据,谁来背锅? 这篇就是回答这四个问题。核心思路一句话:调度交给模型,工具实现交给业务,中间架一层自研桥接,把两边彻底解耦。 这四个问题里,第四个(写操作兜底)是最容易被新手忽略、却最先在真线上出事的。很多人做 Agent 工具调用,demo 阶段全用只读工具,觉得一切都很美好;一旦上写操作,模型一个理解偏差,批量改错一批订单,善后成本极高。所以这套设计从一开始就把「写操作和只读操作分开,写操作必须过确认门禁」当硬约束,而不是事后补丁。这个意识有没有,决定了你的 Agent 是 demo 还是生产系统。 二、整体设计 先看全貌。一次"运营问一句话、模型决定调工具、框架自动执行、结果回喂模型续生成"的完整链路: 几个关键设计决策,后面逐个展开: 不自研 Agent 框架,只桥接工具。工具调用循环交给 Spring AI,业务只实现工具本身。 工具定义和 Spring AI 解耦。自研 @AiTool + ToolHandler 一层,业务代码看不到任何框架类型。 租户上下文用 toolContext 透传,多租户隔离从对话层穿透到工具层。 写操作走确认门禁。Redis token + 参数哈希 + 消费即删,防重放防误触。 一次工具调用的时序 文字描述不如一张时序图清楚。运营问"今天订单怎么样”、模型决定调 query_orders、框架自动执行、结果回喂续生成的完整链路:

每个租户用自己的大模型:动态 ChatModel 工厂怎么落地

xiaoshuai1024 included in AI编程
多租户 AI 助手里最难的一块,不是 RAG、不是 Agent,是让每个租户用自己的大模型。这篇专门讲怎么用"动态 ChatModel 工厂"把这件事做扎实——从表设计、加密脱敏,到工厂的缓存与失效,再到为什么不用 Spring AI 的 starter 自动装配。 一、问题:为什么不能写死一个全局 ChatModel 随便找一个 Spring AI 的入门 demo,大概率是这样:在 application.yml 里配一个 spring.ai.openai.api-key,启动时由 starter 自动装配出一个全局 ChatModel bean,业务里 @Autowired 直接用。单租户、单 Key、单模型,这样没问题。 但我们的平台是多租户的。同一个 AI 助手,服务几十个租户,每个租户: 用自己的供应商(A 用 DeepSeek、B 用通义千问、C 用智谱 GLM); 用自己的 API Key(各自的账单、各自的额度、各自的限流); 用自己的模型参数(温度、max_tokens、具体模型型号都不一样); 在运营后台随时改这些配置,改完要立刻生效。 一个全局 ChatModel bean 在第一点上就垮了——它的 apiKey 在启动时定死,运行时拿不到租户上下文。这就是多租户 AI 和单租户 AI 最本质的区别:模型实例必须按租户动态创建、缓存、失效。 方案 能否多租户 配置生效 复杂度 全局单 ChatModel bean(starter 装配) 不能(只有一个 Key) 改配置要重启 低 每次请求都 new 一个模型 能 实时 极高(每次建连,性能不可接受) 动态 ChatModel 工厂 + 缓存 + 失效(本文) 能 配置变更 evict 后实时生效 中 结论很清楚:需要在两者之间取一个平衡点——按租户懒加载,建好后缓存复用,配置变了再失效重建。这就是下面整个设计的起点,也是多租户 AI 和单租户 AI 最根本的区别所在。

组件库工程化踩坑复盘:单包到 Monorepo 的三层演进

xiaoshuai1024 included in 工程实践
做企业级组件库,难点从来不是写组件,是组织组件。一个按钮谁都会写,但当组件库膨胀到几十个组件、跨多个对外包、要兼顾内部联调和外部 npm 发布时,怎么组织代码、怎么编排构建测试发布流水线才是真正工程难题。 本文以 luban-ui 为例,讲透两件事:Monorepo 怎么分层、工程化流水线怎么编排。技术栈清单不是重点,背后的取舍才是。技术栈是 Vue 3 + TypeScript + Vite + Nx + pnpm,但这套判断换成 React + Webpack + Turborepo 同样成立。 本文以 luban-ui 为例,讲透两件事:Monorepo 怎么分层、工程化流水线怎么编排。技术栈清单不是重点,背后的取舍才是。技术栈是 Vue 3 + TypeScript + Vite + Nx + pnpm,但这套判断换成 React + Webpack + Turborepo 同样成立。 为什么不直接用 Element Plus / Ant Design 这是被问得最多的一个问题。直接回答:因为我们的组件库不只是基础组件,还有低代码设计器和运行时渲染。 Element Plus 这类成熟库解决的是「基础 UI 组件」——按钮、表单、表格。但 luban-ui 的定位是: 基础组件(luban-base):按钮、表单、布局等。 低代码组件(luban-low-code):设计器、运行时渲染、schema、组件注册表。 这两层耦合度极高——设计器要识别基础组件的元信息、运行时要按 schema 渲染基础组件。如果用第三方基础组件库,设计器和运行时就只能"猜"组件的 props,没法精确控制。自己造基础组件,是为了让基础组件天然带「可被设计器识别」的元信息。 这是架构决策的根本出发点:选型不是看哪个库功能多,是看你的场景需要哪些约束。如果你只做基础组件库,直接 fork Element Plus 改皮就行,别造轮子;如果你要做低代码平台,基础组件必须自己掌握。 这个选型判断还有个延伸:造基础组件的成本,远不止「写几十个组件」这一次性投入,还有长期的维护成本——浏览器升级了要适配、无障碍标准变了要跟、新交互范式出来了要加。第三方库有社区帮你摊这些成本,自研就全压在自己团队头上。所以自研基础组件这个决策,隐含的前提是「团队有持续投入的预算和意愿」。如果只是短期项目、做完就丢,fork 第三方库反而更划算。我们做这个决策时,算过一笔账:自研前两年维护成本高于 fork,但到第三年,因为基础组件天然带设计器元信息、低代码层不用做适配,总成本反而反超 fork。这个决策是面向长期的,短期看是亏的。

搭一套企业级低代码平台:六库分工与全栈架构落地

xiaoshuai1024 included in 工程实践
鲁班低代码平台刚立项时，后端、接入层、对外站点、组件库五个仓各写各的，谁也说不清边界在哪——改一个接口，三个仓要跟着动。这篇记的就是怎么把这摊代码理成一套边界固定、契约稳定的全栈架构。 下面以「鲁班低代码平台」为例，给出一套可落地的企业级低代码平台全栈架构与实现路线。 项目归属说明：架构经验源自大厂时期主导的内部系统"鲁班"，代码实现以个人开源项目 OpenLuban（架构同源、代码全新重写，GitHub）为准。原公司系统源码不公开。 覆盖范围从组件库、设计器与运行时渲染（Render），到接入层 BFF，再到主后端 luban-backend（Java：Spring Boot + MyBatis + Redis + MySQL），最后到管理后台与对外站点的交付形态。 文章目标不是“把所有代码一次写完”，而是用清晰的边界把系统搭起来，并能在后续迭代中持续演进。 1. 系统全貌：多仓分工，边界固定 鲁班平台由多个独立仓库组成,用 meta 仓 + git submodule 的方式统一治理(强烈建议保持拆分)。核心是五个仓,加上架构文档仓和 AI 助手仓,各自职责如下: luban-ui：Nx monorepo,内部三个子包——luban-base(原子级 Vue3 组件,不感知低代码)、luban-low-code(可配置/可拖拽层,负责 schema 解析、组件注册、按 schema 渲染 base)、luban-utils(纯逻辑工具)。物料按业务场景分类(form/general/layout/website/navigation/marketing/lead/data-display 等)。 luban-backend（主后端）：Java 实现(Spring Boot 3.2.5 + MyBatis + Redis + MySQL)。提供站点、页面、用户、系统设置等原子化接口;平台 API 权威文档在本仓库维护（luban-backend/docs/API.md,自题"权威定义")。后端按 publicside(C 端只读)/operatorside(运营写)/shared(共享领域)三侧分包。 luban-bff：接入层,Next.js(App Router)实现,既是 API 网关也承担短链跳转、AI 代理。对前端提供统一接口,对内通过 BACKEND_BASE_URL(默认 http://127.0.0.1:8080/backend)调用 luban-backend;负责认证上下文注入(Header)与必要的协议/聚合适配。 luban：低代码引擎(设计器 + 运行时),Vue 3.5 + Vite。负责设计态的拖拽编辑和 schema 生成。 luban-website：对外站点(Nuxt 3 SSR)。根据 URL 从 BFF 拉取已发布页面 schema,交给 luban-ui 的 Render 运行时渲染。 luban-architecture-design：架构文档仓,作为整个平台的 single source of truth。 luban-ai-assistant：AI 助手,自然语言生成/编辑低代码页面(后面单讲)。 一点演进说明:这套架构早期有过一个 luban-backend-go(Go 双后端)作为同契约参考,后来 Go 双后端的路线放弃了,以 Java 版为主。保留"主后端单一权威"比维护双实现省心得多——双实现的契约对齐成本,长期看比它带来的容灾收益高。

前端到架构师:认知、技术、思维、视角的四层跃迁

xiaoshuai1024 included in 技术分享  个人成长
很多前端开发者在工作 3-5 年后，都会面临一个瓶颈：从单纯的业务开发，难以突破到架构设计层面。有人认为是技术不够深入，有人觉得是经验积累不足，实则核心是思维模式的跃迁。这篇记录的就是前端走向全栈、走向架构师时，真正发生转变的东西——不是技术本身,而是思维模式。 这个瓶颈为什么普遍存在。因为业务开发到架构设计之间,隔的不是「更多技术」,是「不同维度的思考」。业务开发的考核标准是「按时交付、bug 少」,这个标准奖励的是执行力和细致度;架构设计的考核标准是「系统健康、支撑业务」,这个标准奖励的是全局观和判断力。从一种奖励体系跳到另一种,不是多学几个框架能做到的,是思维模式要整体切换。所以很多前端开发者卡住,不是不努力,是努力的方向还在旧的奖励体系里(把代码写得更细),没有切换到新体系(把系统想得更全)。 前端架构师的自我修养，从来不是"技术堆砌"，而是认知升级、技术深耕、全栈视野与系统思维的综合提升。本文结合一线实践，聊聊从业务开发到架构设计的思维转变路径。 一、认知跃迁：从"完成需求"到"创造价值" 业务开发阶段，核心目标是**“按时完成需求、修复 bug”**，关注的是"怎么做"，聚焦于具体功能的实现，是"点"上的深耕。 而架构设计阶段，核心目标是**“用技术驱动业务、降低成本、提升效率”**，关注的是"做什么、为什么做"，聚焦于系统的全局与长远，是"面"上的把控。 这种认知的转变，是成为前端架构师的第一步，也是最关键的一步。其核心差异可通过以下对比清晰呈现： 举个简单的例子：业务开发接到"实现一个表单提交功能"的需求，会直接上手编写 Vue 组件、处理表单校验、对接后端接口。 而架构师会先思考一连串问题： 这个表单是否可复用？是否需要适配多端？ 数据流转是否规范？ 如何避免重复开发？ 如何支撑未来表单字段的扩展？ 前者是**“被动执行”，后者是“主动创造”**，这就是认知层面的核心差异。 这个差异怎么体现在日常工作中。一个信号是:你接到需求后,第一反应是「怎么实现」还是「该不该这么实现」。业务开发的第一反应是「怎么实现」——给个表单,想怎么写组件、怎么调接口;架构师的第一反应是「该不该这么实现」——这个表单是不是最优解、有没有更好的方案、会不会给未来留坑。前者是执行导向,后者是判断导向。当你开始频繁质疑需求本身的合理性、思考更优替代方案时,认知跃迁就已经在发生了。 另一个信号是:你关心的范围从「我的模块」扩大到「整个系统」。业务开发只对自己的负责的页面/组件负责,别人的模块出了问题「不归我管」;架构师关心的是「整个系统的健康」——前端的性能、后端的接口设计、数据库的表结构,任何一个环节出问题都会影响全局,架构师要全程跟进。这种「全局视野」不是多管闲事,是架构师的本职——架构师就是那个「对全局负责」的人。 认知跃迁还有一个容易忽略的维度:从「技术视角」到「业务视角」。业务开发想的是「这个功能用什么技术实现」,架构师想的是「这个功能对业务有什么价值、技术方案怎么支撑业务目标」。技术是手段,业务价值才是目的。能把技术决策和业务目标挂钩(比如「用 SSR 是为了首屏快,首屏快是为了转化率高」),才说明认知从「技术本位」转到了「价值本位」,这是架构师和高级开发的标志。 二、技术深耕：从"会用"到"吃透"，构建技术优势 业务开发阶段，我们对技术的要求是**“会用”**——能熟练使用 Vue、Nuxt.js 开发页面，能调用 Egg.js、Express 接口，能完成基础的性能优化即可。 但架构设计阶段，对技术的要求是**“吃透”**——不仅要知其然，更要知其所以然，能掌握技术的底层原理、适用场景与局限性，构建自己的技术优势。 结合前端核心技术栈，技术深耕的核心方向可分为三大模块，其知识体系架构如下： 比如对于 Vue3，业务开发可能只用到 setup、ref、reactive 等基础 API。 而架构师需要吃透响应式原理的实现、虚拟 DOM 的 diff 算法、组件的生命周期机制。只有这样，才能在架构设计时，根据业务场景选择最合适的技术方案。 这同时意味着能避免因技术选型不当导致的系统隐患，并快速定位、解决复杂的技术问题。 技术深耕不是"盲目追求新技术"，而是围绕核心技术栈，做深做透，形成自己的技术认知，让技术成为架构设计的底气。 技术深耕有个误区要避开:「广而不深」。有些开发者热衷于追新框架(今天 Vue、明天 React、后天 Solid),每个都「会用」但没有一个「吃透」。这种广度在业务开发阶段还行(能应对不同项目),但到架构设计阶段就成了短板——架构师需要在某个技术栈上有足够深度,才能判断「这个方案在这个技术栈下可不可行、有什么坑」。深度比广度值钱,因为深度赋予判断力,广度只赋予适应力。架构师靠判断力立足,不是靠适应力。 怎么判断自己「吃透」了一个技术。有个简单的自测:你能不能在不看文档的情况下,讲清楚它的核心原理和关键机制。比如 Vue3 的响应式,如果你能讲清楚 Proxy 怎么拦截、track 和 trigger 怎么收集和触发依赖、effect 怎么关联,那就算吃透了;如果只能讲「用 ref 和 reactive」,那还停在「会用」。吃透的标准不是「能背 API」,是「能解释原理、能预判边界、能定位疑难」。 还有一个深耕的维度容易被忽略:知道技术的「局限性」。会用一个技术,知道它能做什么;吃透一个技术,还要知道它不能做什么、在什么场景下会出问题。比如 Vue 的响应式对 Map/Set 的支持在 Vue2 时代是缺失的,对数组下标直接赋值不触发更新——这些局限性只有吃透了才知道,才能在架构设计时绕开。只知其能不知其不能,会在关键场景踩坑。 三、全栈思维：打破前后端壁垒，建立数据流全局视角 前端团队基建优先级:按投入产出比排序,不按重要程度

xiaoshuai1024 included in 技术分享  团队管理
当团队从单兵作战走向规模化，基建和影响力就成了绕不开的两件事——单纯依靠"单兵作战"的开发模式已难以为继。这篇记的就是我在前端工程师走向团队负责人那段路上的总结。 前端团队的核心竞争力，不仅在于业务需求的交付效率，更在于基建的完善度与技术影响力的辐射力。二者相辅相成，既是团队规模化发展的基石，也是技术价值落地的关键。 一、为什么要做前端团队基建与技术影响力建设 很多团队在发展初期，会陷入"重业务、轻基建"“重交付、轻沉淀"的误区，认为基建和影响力建设是"无用功”，不如聚焦业务需求更有价值。 但随着团队规模扩大（超过 10 人）、业务场景增多，这种误区会导致一系列问题： 开发效率低下、代码质量参差不齐； 新人上手缓慢、技术沉淀不足； 跨团队协作壁垒突出。 根据行业调研数据显示，完善的前端基建可使团队开发效率提升 30%-50%，代码缺陷率降低 40%，新人上手周期缩短 60%。 而有效的技术影响力建设，能提升团队凝聚力，降低人才流失率，同时增强团队在公司内部的话语权。 前端团队基建与技术影响力建设，是**“向内赋能团队，向外输出价值”**，其核心意义可通过以下架构图清晰呈现： 简单来说，基建是团队的"内功"，决定了团队的交付能力和抗风险能力；影响力是团队的"外功"，决定了团队的价值认可度和行业地位。二者缺一不可，共同支撑团队长远发展。 这层关系里有个容易被忽略的传导链:基建是影响力的前提,不是平级关系。一个团队基建稀烂,代码乱、bug 多、发版靠玄学,再怎么做影响力建设也是空中楼阁——对外分享时讲不出真东西,因为真东西(可复用的组件、可落地的方案)压根没有。基建先扎实,有了真材实料,影响力输出才有底气。反过来,影响力也能反哺基建——团队有了话语权,更容易争取到做基建的时间和资源。两者是「基建先行、影响力反哺」的循环,不是同时起步的两条线。 还有一点要讲清楚:基建不是一次性投入,是持续成本。很多团队对基建的预期是「花一个月搭好,以后就省事了」,这个预期是错的。脚手架要跟着技术栈升级(Webpack 5、Vite、ESM 这些变化),组件库要跟着设计规范迭代,监控要跟着业务调整告警阈值。基建搭完只是开始,维护它的人力是长期支出。算基建 ROI 时,如果只算「搭的时候省了多少」,不算「维护要花多少」,会严重高估收益、低估代价。诚实地把维护成本算进去,才能判断一个基建模块值不值得做。 二、如何落地前端团队基建与技术影响力建设 前端团队基建与技术影响力建设，并非一蹴而就。需要结合团队规模、业务场景，循序渐进地落地。 关键在于避免"大而全"的盲目建设，聚焦核心需求，逐步迭代优化。 （一）前端团队基建建设：筑牢团队发展基石 前端基建的核心是**“标准化、自动化、可复用”**，围绕开发全流程，搭建高效、稳定、可扩展的支撑体系。 它覆盖开发、测试、部署、运维全链路，具体可分为以下 6 个核心模块： 1. 通用组件库 基于 Vue3 + TypeScript 开发，覆盖基础组件（按钮、输入框、表格）、业务组件（表单、弹窗、列表），统一设计规范和交互逻辑。 支持按需引入，适配多端场景（H5、小程序、PC）。组件库需持续迭代，结合业务需求补充新组件，同时定期重构优化，保障性能和可扩展性，避免出现"组件冗余、维护困难"的问题。 组件库有个反直觉的陷阱:不是组件越多越好。团队早期热衷往组件库里堆组件,觉得「库越大越值钱」,结果一半的组件只用了一次,后续没人维护,出了 bug 没人修,反而成了负担。我们的规矩是:一个组件要进库,必须有至少两个业务场景要用,否则留在业务代码里。这个门槛挡住了「为复用而复用」的伪组件,让组件库里的每个组件都是真有复用价值的。组件库的价值不在数量,在平均复用率——100 个组件平均用 1.5 次不如 30 个组件平均用 5 次。 2. 工程化脚手架 基于 Node.js + Egg.js/Express 开发，集成项目初始化、代码校验、打包构建、部署发布等功能，统一项目目录结构、技术栈和依赖版本。 脚手架可根据业务场景，拆分不同模板（如 PC 端后台模板、H5 营销模板、小程序模板）。新人只需执行简单命令，即可快速搭建标准化项目，大幅缩短项目初始化时间。 3. 技术规范体系 制定覆盖全开发流程的规范： 代码规范：ESLint、Prettier 配置； 接口规范：请求方式、参数格式、返回格式； 提交规范：Commitizen 配置； 部署规范、命名规范等。 通过规范落地，减少代码冲突，提升代码可读性和可维护性，同时降低跨团队协作成本，避免因规范不统一导致的重复返工。
