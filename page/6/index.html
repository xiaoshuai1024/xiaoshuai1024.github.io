复杂中后台系统微前端（qiankun）接入实践

xiaoshuai1024 included in 技术分享  实践总结
这是我前端阶段一次典型的复杂系统接入实战——面对一堆技术栈各异、各自为战的老中后台系统，用 qiankun 把它们收拢成一套统一体验。它代表的是前端架构整合的能力:在不动业务、不推倒重来的前提下,让分散系统收敛。 在一线互联网大厂的业务发展过程中，中后台系统往往随着业务迭代逐步扩张，长期积累后会形成多个独立部署、技术栈各异的子系统。 这些系统各自独立维护，不仅导致用户需要频繁切换账号、体验割裂，还增加了开发维护成本和权限管控难度。基于此，我们选择 qiankun 微前端框架，实现多系统统一接入与管理，以下是完整实践过程与复盘。 一、系统基本情况：接入背景与核心诉求 我们所对接的中后台系统，是经过多年业务沉淀形成的复杂体系，核心痛点集中在**“分散管理、体验割裂、权限混乱”**，具体情况如下： 1. 权限管控诉求迫切 各子系统独立部署，用户权限需在每个系统单独配置，不仅增加运维成本，还存在权限不一致、管控不统一的安全隐患，亟需统一的权限入口和管控机制。 同时，需实现**“一次登录、多系统通行”**，减少用户切换成本，提升操作效率，保障系统数据安全。 2. 技术栈杂乱无章 各子系统建设周期不同，技术栈差异极大，涵盖多种前端框架，具体分布如下： 技术栈的不统一，导致跨系统协作困难、代码无法复用，且老旧 jQuery 项目维护成本高，难以迭代优化，无法满足现有业务发展需求。 基于以上问题，我们确定微前端接入的核心目标： 统一入口、统一权限、统一体验 兼容现有所有子系统 渐进式迭代优化，而非一次性推翻 二、基座开发与统一准备：筑牢接入基础 微前端接入的核心是**“基座 + 子应用”**的架构模式，其中基座作为统一入口，承担权限管控、子应用管理、数据通信等核心职责。 结合我们的技术栈（Vue3 + Node.js + Egg.js + SpringBoot），基座开发与统一准备主要分为 3 个核心环节，整体架构如下： 1. 基座技术选型与开发 基座采用 Vue3 + TypeScript 开发，基于 qiankun 框架的 registerMicroApps、start 等核心 API，实现子应用的注册、加载、卸载与通信，同时集成 Element Plus 实现统一的页面布局、导航栏、权限菜单。 基座核心职责是**“统一管控”**，不承担具体业务逻辑，重点保障稳定性和兼容性，适配不同技术栈的子应用接入。 2. BFF 层接口打平 基于 Egg.js 开发 BFF（Backend For Frontend）层，统一聚合各子系统的后端接口，解决不同子系统接口规范不统一的问题。 BFF 层负责请求转发、数据格式化、接口容错处理，让基座和子应用只需对接统一的 BFF 接口，无需关注各子系统后端的差异，降低接入复杂度。 3. 统一鉴权功能开发（与后端协同） 联合后端团队，基于 SpringBoot 开发统一鉴权服务，实现**“一次登录、多系统通行”**的单点登录（SSO）功能。

多租户 SaaS 的数据隔离:从一行漏掉的 tenant_id 说起

xiaoshuai1024 included in 架构
多租户数据隔离是我做全栈架构时绕不开的一环——它是 SaaS 系统能不能交付的底线。这篇记录的就是这套隔离体系里最隐蔽的一个漏洞,以及我怎么把它从"靠人盯"改成"层层兜"。 先说一句结论:多租户的安全感,从来不来自"开发都很细心",而来自"假设迟早有人会漏,提前把网织好"。后面所有内容都是为这句话找证据。 一次越权,根因只有一个 JOIN 那次线上反馈很直接:某个客户说"我能看到别人的数据"。多租户系统听到这句话,基本可以判定为越权。顺着链路翻,查到一条排查 SQL,对着前端返回的字段一比,定位到一段 JOIN。 排查路径其实不复杂。先用前端能看到的那条"不该出现"的数据,反查它来自哪张表、哪个接口、哪条 SQL;再把这条 SQL 在测试库里重放,逐段拆 JOIN,看哪一段把别人的数据带了出来。定位到具体那段 SQL,大概花了不到半天——真正花时间的是后面想清楚"为什么会漏"。 根因是:某个 JOIN 关联两张业务表时,只在外层带了 tenant_id 条件,JOIN 的那张表没带。结果租户 A 的查询,把租户 B 的关联行也带了出来。 不是什么精妙的 0day,也不是鉴权被绕过。就是一段 JOIN ... ON a.id = b.aid,后面漏了 AND a.tenant_id = b.tenant_id。 修复很快,改一行加上去就完事。但这件事真正的风险在于:代码能跑、单测能过、功能自测也没问题——平时你压根不会想到去构造"跨租户数据"的用例,它只在真实多租户、真实数据并存的线上才会爆。 行级隔离的标配:为什么选它,代价是什么 先把我们的隔离方案交代清楚,后面才好聊为什么会漏。 业界多租户隔离大体三条路:独立库、独立 schema、行级隔离。我老老实实把它们的代价摆出来,你就能理解我们为什么选行级。 独立库:每个租户一个物理数据库。隔离最干净,但运维成本爆炸——租户一多,实例数跟着涨,备份、监控、迁移都得一个租户一个租户搞;跨租户报表、平台级统计要跨库 join,基本写不下去。除非单租户数据量极大、合规要求极严(比如金融大客户),否则扛不住。 独立 schema:一个库里每个租户一套 schema。比独立库省实例,但表结构变更要做 N 次 DDL,租户越多越痛;而且跨租户查询还是麻烦,得动态拼 schema 名。 行级隔离:所有租户的数据混在一张表,靠 tenant_id 这列区分。运维最省心(一套表结构、一次 DDL),跨租户查询天然支持,扩展租户就是加一行配置。代价就一个:每一条 SQL、每一个 JOIN 都得记得带 tenant_id。 我们这套选的是行级隔离,业务表都带 tenant_id,所有查询、所有 JOIN 都必须带条件。取舍很清楚:用"每条 SQL 都要小心"换"运维和成本可控"。对中小客户为主、租户数量会涨的 SaaS,这笔账划算。 租户身份怎么来?请求头 X-Tenant-Id。请求一进来,过滤器链里有个 TenantFilter,负责把头里的租户标识解析出来,塞进一个请求级的上下文。后面业务代码取租户 id,都从这个上下文拿,不让业务自己再从 header 里抠——抠着抠着就有人抠错。

把权限校验从手写里解放出来:RBAC 注解 + 过滤器链的设计

xiaoshuai1024 included in 架构
这是我做全栈架构时的一类典型工作:把后端的权限校验,从「散落在每个接口里的 if」收拢成一条声明 + 校验 + 对齐 + 兜底的链路。它体现的是用机制替代人记的工程判断。 权限体系最怕的不是「没做」,而是「散着做」。 「散着做」到底贵在哪 我接手过一个运营后台,权限校验是这种画风:这个接口开头一个 if (user.hasRole("admin")),那个接口中间塞一段 if (!user.can("edit")),还有几个接口压根没校验,靠「前端按钮不显示」硬撑。 老实讲,这种代码最致命的不是「能不能跑」,而是谁都不敢动。一段判断摆在接口里,三个月后你想删它,第一反应不是「这判断对不对」,而是「删了会不会越权」——因为没人知道这段判断是踩了哪个坑加的,删了出了事算谁的。于是它就赖在那儿,越来越多,越来越乱。 代价是具体的: 谁也不敢删那段判断,因为谁也不知道删了会不会越权。 最后代码里堆满「祖传 if」,谁动谁背锅。 前端按钮不显示,不等于后端没校验漏洞。 按钮显示是前端逻辑,接口鉴权是后端职责。黑客根本不经过你的前端,直接打接口。把安全寄托在「按钮藏起来了」,等于没做。 几个接口漏校验,纯靠运气没出事。 接口多了之后,「哪个加了哪个没加」全靠人记。今天没出事,是因为没人想到去打那个接口;明天被人扫到,就是事故。 权限和业务逻辑搅在一起,改一个判断得读懂整个方法。 这段判断嵌在业务代码中间,你要重构这个方法,就得先把权限逻辑摘出来——但摘出来的过程中,你怎么知道没摘错? 所以判断标准很简单:如果你问「这个接口的权限规则是什么」,得去翻业务代码才能回答,那就是散着做。 真正该有的状态是,看一眼注解就知道。 推倒重做时,核心思路就一句话——权限跟着接口走,不跟着人记。 注解声明,让权限长在接口上 定义一个注解,大概长这样: @RequirePermission("ops:order:refund") @PostMapping("/order/refund") public Result refund(...) { ... }注意那个权限 key:ops:order:refund。它不是随手写的字符串,而要在迁移脚本的权限种子数据里先注册过。这一条最关键,后面要出事就出在这。 key 的命名是有规范的,我们整个团队统一用「资源:动作」的三段式:{系统}:{资源}:{动作},比如 ops:order:refund(运营后台 / 订单 / 退款)、ops:order:export、fin:bill:audit。规范本身不复杂,但它解决三件事: key 在种子数据里必须先注册。 没注册的 key 走到运行期要么报错要么放行,两种结果都不好。所以 key 是「先在种子表里声明,再在注解里引用」,顺序不能反。 和接口绑死,新人不用通读业务。 新人看接口签名就知道「这接口吃什么权限」,不需要去翻业务逻辑、不需要去问老人。这点对团队扩张特别值钱。 权限信息和接口定义不再分家。 改接口的人顺手就能看到权限要求,删接口的人知道顺手把种子 key 也清掉。 有了注解,接口要什么权限就是声明出来的: 新人不用通读业务逻辑,也能知道「这个接口吃 ops:order:refund」。 权限信息和接口定义绑在一起,不再散落在代码各处。 改接口的人顺手就能看到权限要求,删接口的人知道该把种子 key 也清掉。 Code review 时,「这个接口有没有权限」从「翻业务代码」变成「看注解有没有」。 比满屏 if 清爽太多。注解本身的定义也就几行:

数据库迁移治理:Flyway 版本号、squash 和那些启动失败的事故

xiaoshuai1024 included in 后端
在我从前端走向全栈架构的阶段,数据库这一层是我必须打通的硬骨头。这篇讲的是其中最容易被忽视、却能让整个服务起不来的一环:Flyway 迁移治理。 某天早上我接到一个告警:某个服务启动不起来。日志翻到最后,就一行 Flyway 报错——迁移版本号对不上。 登上去一看,前一天两个人各自合了代码,各自加了一个 V12__xxx.sql。版本号撞车,Flyway 直接罢工,服务起不来。 事故的本质:不是粗心,是治理缺位 排查倒不复杂,改个号、重新合一下就好了。但事后我盯着那两个 V12 想了很久——这事的本质不是"谁粗心",而是治理缺位。 数据库迁移在多人协作下,如果不立规矩,迟早撞车。而且撞车的代价不是"改个号",是整个服务起不来。 下面是我们团队在 Flyway 上踩过坑后,最终固化下来的几条规矩。 版本号:从号段分配到秒级时间戳 最早我们试过"号段分配"——张三负责 V100 到 V199,李四负责 V200 到 V299。想法挺好,各管一段。 但实际跑下来全是问题: 号段会跨、会重; 新人来了不知道找谁要号; 合并的时候还是会有人偷偷用了别人的号段。 结论:号段分配是反工程化的,它把一个本该自动化的协调问题,丢给了人。 我们的做法:秒级时间戳 后来全换成秒级时间戳:V{YYYYMMDDHHmmss},比如 V20260611185900__add_user_field.sql。谁先建脚本,谁的时间戳就是谁。 两个人同一天、同一分钟、同一秒撞上的概率,比号段撞车低太多了。弃用号段那一刻,工程上反而干净了——少一套要维护的"分配机制",代码和流程里的东西越少越省心。 mvn validate 阶段校验唯一性 光靠人自觉是不够的。我们写了个 Python 脚本,挂在 mvn validate 阶段,扫一遍所有迁移文件,校验版本号是否唯一。不唯一直接 fail 构建。 这个脚本跑到现在,真实校验过的迁移有 119 个,全唯一。不是团队运气好不撞车,是脚本一直在那儿守着——谁撞了在本地构建就过不去,根本到不了 CI。 为什么必须前置到 validate 阶段 版本号冲突这种事,一旦让它进到生产,服务启动直接失败,影响面是全量的。 所以这种东西必须前置到最早:validate 阶段,本地一跑就知道。 这种"不出事没人感谢、出事就是大事故“的基础设施,就得靠自动化兜,不能靠人。 squash 是个坑:只能在本地 Flyway 有个 squash,把一堆历史迁移压成一个新的基线。听起来很爽——迁移文件越积越多,看着烦。 但我们定的规矩是:squash 只能在本地开发库做,生产绝对禁止。 原因:squash 会 DROP DATABASE squash 的实现是 DROP DATABASE 然后用新基线重建。本地开发库你随便重建无所谓,生产库谁敢 DROP?

低代码渲染引擎与设计器:从 schema 到可编辑画布

xiaoshuai1024 included in 架构
这是「前端 → 全栈 → AI 开发」线上的 全栈阶段。我主导的低代码平台「鲁班」,核心是一个自研的渲染引擎 + 设计器。这篇把它拆开——schema 怎么设计、渲染器怎么递归、设计器怎么拖拽编辑。 项目归属说明：渲染引擎的设计经验源自大厂时期主导的内部系统"鲁班"，本文代码实现以个人开源项目 OpenLuban（架构同源、代码全新重写，GitHub）为准。 先把一句话结论撂这:低代码引擎的难点不在「能渲染」,在「能编辑、能扩展、能演进」。市面上拿 JSON 渲个页面、发个 demo 的项目一抓一把,真正卡人的是后面那三件事。下面所有取舍都围着它转。 schema:一棵树 + 一个表单值字典 低代码引擎的起点是 schema。我把它设计得很薄,有意压到最小: interface NodeSchema { id: string type: string // 组件类型,如 'LubanButton' props?: Record<string, unknown> children?: NodeSchema[] } interface PageSchema { root: NodeSchema formState?: Record<string, unknown> // 表单值绑定容器 }一个关键决定:表单控件的值不挂在每个节点上,而是统一放在顶层 formState 字典里,以控件的 props.name 为 key。运行时只要一个扁平字典就能驱动整页所有表单字段,提交、校验、回填都简单。 老实讲这步我纠结过。每个节点自己存自己的值,直觉上更「自然」,写 schema 的人也省心。但一落地就发现问题:提交时要遍历整棵树去收集,校验时要在树里找控件,后端回填要把值塞回各自的 props——三个环节都要重新爬树。换成顶层一个字典,提交是 JSON.stringify(formState),校验是对字典跑规则,回填是 Object.assign。一个数据结构,省掉三处脏活。 还有一个更隐蔽的取舍:schema 要薄,但版本治理要从第一天留口子。PageSchema 现在没显式的 version 字段,不是忘了,是早期的 demo 还不值得为它背兼容包袱。但 NodeSchema 的结构我刻意没塞多余东西,留出加 version / compatibility 的余地。一旦真有存量页面要演进,加个字段、走个 migrator 就能接住。第一天就把 schema 塞满,后面想改一处得动全身——这种坑我见太多了。
