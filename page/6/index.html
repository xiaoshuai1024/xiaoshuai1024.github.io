全栈架构 × AI 研发实战

从光标补全到流水线:一条跑通了的 AI 研发全链路

xiaoshuai1024 included in AI编程
大多数人用 AI 写代码,还停留在「光标后面补两行」,出 bug 了再问一句「这怎么报错了」。这跟当年用 IDE 自动补全区别不大,只是更聪明了点。 真正让玩法变了的,是把整条研发链路摊开,让 AI 在每一步都进场,把它当成流水线上的工位,而不是高级补全。下面这套,是我们团队跑了一年、产出近百万行代码之后沉淀下来的工作流,每一步都经过真实项目验证。后面讲评审、测试、契约的文章,都拿它当地图。 全链路总览 整条链路长这样: 确认需求 → HTML 原型 → 计划 → 多轮并行头脑风暴 → TDD 执行(红/绿/重构)→ 自治多 Agent 评审 → 提交/MR。 每一步都不是「问 AI 一句话」那么简单。每一步都对应一个明确的兜底目的:消歧、一致性、效率、质量、语义正确性、仓库门禁。如果某一步的兜底目的你能用别的方式达到,完全可以替换——但目的不能省,流程可以变。 第一步:需求之后,先出 HTML 原型 确认需求之后,先出 HTML 原型,而不是直接开写。 原因很直接:对着空气写需求,人和 AI 都容易跑偏。需求文档里的「支持商品管理」「首页可配置」这种描述,十个人能脑补出十种界面。先把界面用 HTML 画出来,需求和实现的歧义在原型阶段就消掉了大半。 具体做法是:需求一确认,丢给 AI 让它先出一版静态 HTML 原型。不用漂亮,关键是把关键页面、关键交互画出来。这个 HTML 后面会被丢掉,但它的价值不在产物本身,而在「强迫双方把模糊的需求变成具体形状」。 我们踩过一次坑:一个「会员权益」的需求,文档写得清清楚楚「会员可见专属价格」,直接让 AI 写代码,实现成了所有商品都展示会员价、非会员看不到价格——但实际需求是所有用户都看得到原价,只有会员额外看到优惠价。这种歧义,如果先出 HTML 原型,5 分钟就能在原型上发现。 原型还有个衍生用法:它后面会变成验收的参照物。功能做完,AI 把实现界面跟原型摆一起比对,缺了哪个按钮、哪个状态没体现,一目了然。比起读需求文档逐条核对,对着原型检查快得多也更直观——原型本来就是把需求具象化的产物,验收时复用它,等于让原型在整个链路里服务两次:一次消歧,一次验收。产出物复用,是流水线设计里很值得抠的细节。 原型阶段的产出不是代码,是共识。 这一步省下来的成本,是后面返工的好几倍。 第二步:计划阶段,确立唯一事实源 计划阶段是整个工作流里我最强调的一环。 做法是做两轮计划,产出一份任务图 JSON,并约定它是唯一事实源。两轮计划分别是: 粗计划:让 AI 把需求拆成大块(「前端」「后端」「数据迁移」三层),每块标个范围。 细计划:在大块内,把任务拆到半天到一天可完成的粒度,每个任务标明:输入是什么、输出是什么、依赖谁。 最终的产物是一份任务图 JSON,长这样: { "tasks": [ { "id": "T-001", "title": "订单详情接口", "layer": "backend", "input": ["订单 schema (T-000)"], "output": ["GET /api/orders/:id"], "depends": ["T-000"], "status": "todo" }, { "id": "T-002", "title": "订单详情页", "layer": "frontend", "input": ["GET /api/orders/:id (T-001)"], "output": ["订单详情.

AI研发实战工作流研发效能

一条 CI 都没有:自治多 Agent 评审回路怎么兜住代码质量

xiaoshuai1024 included in AI编程
跟同行聊起质量保障,大家第一反应都是「你们 CI 怎么配的?跑多少分钟?」。我的回答每次都让对方愣一下:我们的仓库,一条 CI 流水线都没有。 没有 GitHub Actions,没有 GitLab CI,没有 Jenkins。.yml 里一个 job 都找不到。 那质量靠谁?靠一套自治多 Agent 评审回路,我已经稳定跑了一年多。这篇把它拆开:它兜住了什么,又诚实地讲一句,它替代不了什么。 为什么一开始就没配 CI,不是嫌麻烦。这个项目是 4 个人 + AI、两个月冲百万行代码的节奏,代码改动密度极高,一天几十次提交是常态。传统 CI 跑一轮完整测试要十几分钟,等它绿了黄花菜都凉了——AI 协同研发的节奏根本等不起串行流水线。 其次,CI 抓的问题有限。流水线擅长抓「编译过不过、测试红不红、lint 报不报」,但抓不了「这段代码的设计对不对」「跨模块的契约对不对」「这坨改动有没有引入语义错误」。而 AI 写代码最容易犯的就是后者——编译能过、测试能绿,但逻辑是错的(270 行静默蒸发就是典型)。我需要一道能读语义的关,CI 给不了。 这两个动机合在一起,就催生了这套 Agent 评审回路:用并行的、能读代码的 Agent 替代 CI 的静态检查和人工评审,用双计数器收敛让它自治,用硬测试门禁兜住动态行为。它不是「省掉 CI」,是「换一种更适合 AI 协同节奏的质量保障形态」。 回路全貌:从「人盯」到「流程兜底」 先说整条回路长什么样,后面再展开关键设计: scope 分桶 → 并行只读评审 → 流式自动修复 → 双计数器收敛 → 硬测试门禁 → 放行提交。 整个评审回路是我们那条 AI 研发工作流 的最后一道关。前面 TDD 已经把红/绿/重构走完了,到这里要决定「这坨改动能不能进仓库」。 关键的判断是:既然能开并行 subagent,就尽量多开。这是团队写进硬约束的一句话。所以这套回路从设计第一天起,就是奔着「并行扇出」去的,不是排队串行。 scope 分桶:并行扇出的第一步 很多人把并行评审理解成「开 N 个 Agent 同时看一坨代码」,这是错的。N 个 Agent 看同一份 diff,要么互相重复(都说同一个问题),要么互相打架(一个说要加锁一个说不用)。并行的前提是scope 不重叠。

AI研发实战代码评审质量门禁

编译能过 ≠ 代码正确:一次 AI 静默删 270 行并发守卫的事故复盘"幽灵删除"事故复盘

xiaoshuai1024 included in AI编程
mvn compile 通过 ≠ 代码正确。 这条铁律挂在评审回路最显眼的位置,起因是一次 AI 改动静默删了 270 行并发守卫,编译照样绿。 过去那些年,我们把"编译绿了"默认等同于"代码对了"。这个默认在人工时代勉强成立,代码换成 AI 大规模产出后,代价会被成倍放大。 一句话讲完整件事:编译器只认语法契约,不校验行为;AI 删一段并发守卫时,编译照样绿、它还底气十足说"已完成"。所以门禁不能停在"能编译",得升级到"得过 verify";任何"我搞定了"必须自带证据(Proof-of-Work),焊进门禁,不靠自觉。 一次看起来无害的"搜索优化" 事情起因是一次很普通的提交,说明写的是"搜索优化"。diff 拉出来看也没问题:搜了什么、优化了什么,逻辑闭环,命名干净,像是 AI 认真做的一次重构。没人会为这种提交专门去翻一遍并发模块——毕竟提交说明里一个字都没提并发。 我们当时的评审回路已经跑起来了,流程走得很顺: Agent 评审 → 自动修复 → 收敛; 然后本地 mvn compile 一把过,绿灯; 提交,合进去。 直到后来有人回头去看那段并发相关的核心代码,才发现不对劲——整整 270 行,没了。不是改坏,是静默删除。那是一处并发安全的守卫逻辑,平时不出声,但在高并发路径上是兜命的。 整个事故链在事后看一目了然,但当时却是潜伏了好几天才爆: 排查过程:从「功能正常」到「定位到删代码」花了 4 天 事故是潜伏 4 天后,大促放量才爆的。复盘时回头去查排查过程,前三天全是弯路——这才是踩坑型该写的部分,因为弯路本身就是经验。我把每天的判断、走错的岔路、最后怎么转向的都记下来,这才是能被复用的东西。 弯路一(第 1 天):先怀疑是「重构改坏了功能」。 按功能回归测一遍——下单、查询、退款,全过。这是最自然的怀疑方向,因为事故现象是「下单异常」,自然往「下单逻辑」上靠。整整一天盯着订单 service 翻,没找到任何异常。这步走错的关键:把"现象发生在 X 模块"误判为"根因在 X 模块",但实际根因在并发守卫层。 弯路二(第 2 天):怀疑是数据库脏数据 / 缓存击穿。 让 DBA 翻 binlog,清 Redis 集群,压一遍灰度环境。压测时偶尔复现一次超卖,但很难稳定重现。这天最有价值的一个动作是压测时偶发了一次「同一订单扣减两次库存」,当时没重视,以为是测试数据污染,其实是关键线索。 弯路三(第 3 天):用 arthas 在线摸并发路径。 把所有可能并发的方法都加上监控,压测一段时间才偶发超卖。这天终于确认了:这不是「业务逻辑 bug」,是「并发安全 bug」——单线程跑永远不会复现,只有并发抢同一资源时才暴露。但具体哪段代码错了,还是定位不到。

AI研发实战事故复盘质量门禁

编译管不着的跨层 bug:用 4 道脚本闸守住 API/SQL/权限/迁移

xiaoshuai1024 included in AI编程
我最怕的 bug,从来不是某个函数写错,而是那种「编译器管不着、lint 管不着、只有跑到线上某条真实路径才炸」的跨层错误。 举个例子。前端调一个 /api/orders/detail,后端最近重构,路由注解改成了 /api/order/detail,少了个 s。结果就是:编译两边都绿,单测各自模块都过,然后某个周二下午运营说小程序白屏了。 这类 bug 的特点:每一层看着都对,错在层与层的「接缝」上。编译器只保证语法契约成立,保证不了「前端发的请求」真的能落到「后端注册的那个路由」上。 这件事我专门写过《编译能过 ≠ 代码正确》——一次 270 行并发守卫被静默删掉的事故,讲的就是同类的跨层盲区。所以我在提交前架了四道「契约闸」。 错在层与层的接缝上 先讲清楚「跨层契约」到底指什么——它有四个典型类别,每一类都曾让我们吃过亏: 跨层契约 事实源(权威方) 被校验对象 编译器能管吗 典型事故 URL ↔ 路由 后端 @RequestMapping 前端 axios 调用 ❌ 路由改名少个 s,前端 404 SQL ↔ DDL Flyway 建表 DDL Mapper 里的 SQL ❌ 表名拼错,运行时 NPE 权限 key ↔ 种子数据 迁移脚本的权限 seed 代码里的 @RequirePermission ❌ key 拼错,权限等于没配 迁移版本号唯一 文件命名约定 所有 V{N}__*.sql ❌ 两个 V12,Flyway 启动失败 这四样东西,编译器一个都不会管。它们都是「跨层的、声明式的、靠人对齐」的契约,而人和 AI(尤其是 AI)最不擅长对齐这种东西——AI 改后端时不知道前端在调什么,改前端时不知道后端路由叫什么。 AI 尤其不擅长跨层对齐,有个根本原因:AI 的上下文是局部的。它改后端 Controller 时,上下文里是 Controller 的代码,前端的 axios 调用不在它的视野里;它改前端时,后端的路由注解也不在视野里。两个文件同时打开都未必对得上,更别说 AI 一次只看一个文件。所以 AI 改了后端路由,它不会主动去同步前端——不是它懒,是它根本不知道前端在调这个路由。契约闸的价值在这里就体现了:AI 不需要知道,脚本替它查,查不过就拦下,逼它补全。这比在 prompt 里反复强调「改路由记得同步前端」可靠——prompt 是建议,AI 可能忘;脚本是强制,它忘不了。

AI研发实战契约驱动质量门禁

UT/IT/E2E 三层测试纪律:fail-fast、禁跳过、零占位

xiaoshuai1024 included in AI编程
测试金字塔谁画都顺手:底下 UT、中间 IT、顶上 E2E,贴在 PPT 上特别好看。可真到了项目里,这套三角形十有八九会缩水: UT 漏边界,只测 happy path; IT 跑得慢,有人嫌烦干脆 @Ignore; E2E 全绿但一上线就翻车——因为只点了「正常路径」。 我以前也画过那种金字塔,画完贴墙上,没几个人看。金字塔不画在墙上,写进纪律里——而且这条纪律不仅人要守,Agent 也要守。 下面是我们怎么把「三层测试」从口号落成可执行规矩的。 三层,各兜一摊,谁也替不了谁 先把分工说清楚,省得有人想「用 UT 多写点、E2E 少写点」来偷懒: 层级 命名约定 兜什么 跑多快 典型反模式 UT *Test 纯逻辑、边界、异常分支 毫秒级 只测 happy path,边界一个不测 IT *IT 真实数据库 / 中间件集成 秒级 mock 掉一切,等于没集成 E2E *E2E 真实用户路径,端到端 分钟级 全绿但只点了正常路径 这三层不能互相替代,理由很具体: UT 全绿不代表集成没问题——你 mock 掉的那个表结构,可能压根和数据库对不上。我见过一个 UserMapperTest 全绿,但实际表里根本没有 deleted_at 字段,上线一查全表 NPE。 E2E 全绿不代表每个分支都对——你只点了「正常登录 + 正常下单」,边界条件(库存为 0、余额不足、并发抢同一商品)一个没覆盖。 不是「三层选一层」,是三层都得在,各管一段。任何「我们 UT 写得很全,E2E 就少点」的说法,都是在为偷懒找借口。 把「真金字塔」和「反金字塔」画出来对比,为什么底宽顶窄才对、倒过来为什么崩,一目了然: 反金字塔(E2E 肥大)是大多数团队的真实状态。UT 少写、IT 被 mock 掉、所有用例堆进 E2E,跑一次 40 分钟,团队嫌慢就降频率——从每次提交跑变成每天跑,再变成每周跑,最后 E2E 形同虚设,名义上有测试,实际没人看结果。金字塔倒过来不是「测试多了」,是「测试错位了」——快的该多的反而少,慢的该少的反而多,整套测试又慢又没用。

墙上 95% 配置里 25%:覆盖率是体检指标不是 KPI

xiaoshuai1024 included in AI编程
我们项目上有个直接的对照。文档里白纸黑字写着后端行覆盖 95%、分支 85%,前端 85/75,运营后台 90/80;但翻构建文件里的门禁配置,实际卡的是后端 25%/20%、前端 80/60%、运营 70/60。 95% 和 25% 摆在一起,看着像双标,像 KPI 造假。它不是造假,它说明覆盖率是个滞后指标——把它当 KPI,数字迟早会变成表演。 这篇要讲清三件事:为什么墙上的 95% 和配置里的 25% 会并存,而且并存是诚实的;覆盖率这个指标到底能量什么、不能量什么;如果不用覆盖率当 KPI,质量该靠什么兜。 几乎每个项目的覆盖率目标都"虚高" 一个直接的观察:凡是设过覆盖率门禁的团队,墙上的目标几乎都比配置里实际卡的高。 目标往高了写,是政治正确; 实际往下调,是工程现实。 两者之间的缝,就是"说"和"做"的缝。我们项目也一样:规则文件里那串 95/85/85/75/90/80 写得漂亮,是大家心里的"理想态";但每天卡提交的门禁,是那串低得多的数字。谁也没撒谎,只是这俩数字量的本来就不是同一件事。 为什么目标总往高了写。因为定目标的人(通常是技术负责人)心里清楚「高覆盖率是好事」,写低了显得团队没追求。而且高目标写出来,对上级、对外都有面子——「我们团队覆盖率目标 95%」,听着就专业。但真到了落地,会发现 95% 在存量项目里根本卡不住(存量代码太多没测),硬卡只会卡死发版或逼人造假。于是配置里的门禁悄悄往下调,形成「墙上一套、配置一套」的双轨制。 这个双轨制不是中国特色,是工程普遍现象。国外的开源项目也一样,README 里写「aiming for 90% coverage」,CI 配置里卡的是 60%。区别只在于,有些团队诚实地承认这个差距(像我们这样把两个数字摆出来讲),有些团队假装差距不存在(然后被覆盖率原教旨主义者绑架)。诚实承认差距,反而能针对性地改进;假装没差距,只会让指标彻底失真。 95% 和 25% 都不是错:覆盖率衡量的是"被测",不是"正确" 这里有个基础认知,但容易被门禁数字带歪。 覆盖率只告诉你"这些行在测试里跑过了",不告诉你"这些行的行为是对的"。 一行代码被执行过,不代表它的语义被验证过。 这个区别是整个覆盖率讨论的基石,值得反复强调。「跑过」和「验证过」是两件事。一行 int result = calculate(a, b) 在测试里被执行了,覆盖率计 1 行;但如果测试只调用了 calculate 没断言 result 对不对,这行等于白跑——它被执行了,但行为没被验证。覆盖率工具不区分这两种情况,它只看「这行的字节码有没有被执行到」。所以覆盖率天然是个偏宽松的指标,它高估了「被验证」的程度。理解了这个偏差,就不会盲目相信高覆盖率。 所以一个项目行覆盖 95%,只能说明"测试碰到了大部分代码",不能说明"大部分代码被正确验证了"。反过来,25% 也不等于烂——如果关键路径都被测试网住,而那 25% 恰好覆盖在这些路径上,它可能比一个水分十足的 95% 更靠谱。 这个认知很多人嘴上同意,落到工程里就变形。一旦把覆盖率设成 KPI(跟绩效挂钩、跟发版挂钩),「覆盖率=质量」这个等式就被默认成立了,然后整个团队开始追求数字而不是质量。追数字最省力的方式不是「写更多有意义的测试」,而是「写更多能跑过的测试」——这两件事差别巨大。前者提升质量,后者只提升数字。而且后者还有隐蔽的危害:它制造了一种「已经测过了」的虚假安全感,让人放松对真实风险的警惕,比赤裸裸的低覆盖率更危险。 举个具体的。一个订单创建方法,有意义的测试要覆盖:正常下单、库存不足、并发下单冲突、优惠券叠加、跨租户隔离。这是 5 个测试,覆盖了关键语义。但如果只追求覆盖率,写一个「调用一次、断言返回不为 null」的测试,行覆盖就 100% 了——可并发冲突、库存不足这些真正会出事的地方,一个没测。前者覆盖率可能 20%(因为还有很多别的方法没测),后者覆盖率 100%(针对这一个方法),但前者比后者有价值得多。覆盖率数字骗不了工具,但能骗 KPI,这是它最危险的地方,也是最该警惕的地方,记住这一点比记住任何具体阈值都重要。
