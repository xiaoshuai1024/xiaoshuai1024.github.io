全栈架构 × AI 研发实战

Agent 替你点完整条用户旅程:E2E 重心从写脚本挪到读业务

xiaoshuai1024 included in AI编程
一个「下单」按钮点了没反应,可能是前端没绑事件、可能是后端端点 405、可能是小程序和 H5 的 DOM 结构根本不一样。人在这上面耗的时间,远多于写那几行 page.click()。 E2E 测试最贵的不是写脚本,是搞清楚用户旅程到底断在哪一端。 我的做法是把整个流程交给 Agent(和测试纪律、视觉验收是一套,前面讲「分层怎么守、样式怎么验」,这篇讲端到端这一层在 AI 时代怎么写):它拿着一份业务描述,自己分析业务逻辑、自己决定从哪一端进入、自己跨系统点到点地跑完整条旅程,跑挂了自己读日志定位卡点。整个过程的副产物——它点过的每一步——沉淀成 Playwright 脚本,下次回归直接复用。 我的做法是把整个流程交给 Agent:它拿着一份业务描述,自己分析业务逻辑、自己决定从哪一端进入、自己跨系统点到点地跑完整条旅程,跑挂了自己读日志定位卡点。整个过程的副产物——它点过的每一步——沉淀成 Playwright 脚本,下次回归直接复用。 下面拆开讲这套方法怎么落地,以及它和普通 E2E 到底差在哪。 为什么 E2E 这一层最该让 Agent 接手 先把范围说清楚。单元测试盯函数,集成测试盯模块边界,E2E 盯的是用户旅程——一个真实用户从进系统到完成目标的全过程。这件事的天花板不取决于代码能力,取决于对业务的理解深度:你得知道用户会怎么走、会卡在哪、前端后端小程序各自承担了什么角色。 这是 Agent 的强项。它不是来替你写 expect() 的,是来替你做业务逻辑分析的。 具体看普通 E2E 在多端全栈项目里的三个死结: 死结一:旅程横跨多个系统,人很难一眼看穿断点。 一个完整的「下单 → 支付 → 分账」旅程,要经过 H5 前端、运营后台、Spring Boot 后端、微信支付、定时任务分账。任何一环挂了,现象都是「页面上没反应」。人要定位,得翻前端日志、看后端日志、对账数据库、查定时任务执行记录——这是个体力活,但更是个信息整合活,是 Agent 最擅长的事。 死结二:多端的 DOM 结构根本不一样,脚本没法复用。 同一个「商品详情」页,在 H5 里是 <uni-input> 包着 <input>,在小程序里是裸 <input>,在运营后台是 Element Plus 的 el-input。普通 E2E 给每个端写一套定位策略,维护成本翻倍。Agent 不靠固定 selector,它读页面文本和语义来定位,一套业务理解能跨端复用。 死结三:测试脚本和业务理解是两张皮。 普通写法里,业务知识留在 PRD 和人的脑子里,测试脚本只是这些知识的一层「代码投影」。PRD 一改,人要先理解改动、再翻译成脚本改动,中间隔着一层翻译损耗。Agent 主导的做法,是让业务理解本身成为可执行的东西——它读 PRD、读代码、读数据库,直接产出测试动作,翻译这一层被压缩掉了。

让研发长出产品脑:从需求到全自动开发的命令系统

xiaoshuai1024 included in AI编程
团队里有个后端,技术很硬,自己写了个挺酷的功能——一个商家可以批量给商品打标签的工具。代码干净,接口规范,自己测了几遍都很顺。 上线两周,DAU 是 0。 不是功能不好用,是没人需要。商家真正卡住的,是商品太多找不到要改的那一批,不是打标签本身慢。他做的是「让一件不该做的事变快」。 这就是研发做产品最容易栽的地方:技术没问题,价值判断是空的。这个后端不是能力不行,他的工程能力是团队顶尖的。问题出在他思考的起点——他从「我能做什么」出发,而不是从「商家需要什么」出发。技术越强的人,越容易掉进这个坑,因为他总有办法把一个不该做的事做得又快又漂亮。 后来我花了大半年,在这个多租户项目里把一套「让研发长出产品脑」的方法磨出来。核心不是教他们画原型、写文档——那些 AI 都能干。核心是逼他们想清楚三件事:给谁、解决什么问题、值多少钱。这套方法现在固化成了一套命令系统,研发对着命令走一遍,产品逻辑就自然逼出来了。 这条转型路径有个前提我得讲清楚:不是让研发替代 PM,而是让研发具备「产品定义」这一环的能力。在 AI 之前,产品定义是 PM 的专属领地,研发只管接需求;现在一个能闭环的工程师,必须自己能定义产品,否则他永远是个执行者,价值锚在别人手里。我团队现在的状态是:研发能独立把一个需求从模糊逼到可评审的 PRD,然后交给全自动工作流去实现——PM 这个角色,在很多场景下被拆解进了研发的流程里。 研发做产品,最容易犯的 6 个错 先说反面。没接触过产品设计的研发,做产品时会稳定地掉进同样几个坑。我把它们列出来,不是为了吐槽,是因为认清这 6 个错,就等于认清单合格 PM 该补什么。 错一:把功能当价值。 「我做完了」不等于「有人用」。功能是手段,价值是目的。研发容易盯着功能完成度——接口写好了、测试过了、性能也优化了,就觉得这事闭环了。但 PM 盯的是另一件事:这个功能有没有把某个真实问题解决掉。一个完成度 100% 的功能,如果解决的本来就是伪需求,完成度越高,浪费越大。 错二:没有整体观。 研发习惯只看自己这一摊——后端管接口,前端管页面,中间靠文档对接。但产品是端到端的。一个商家批量打标的功能,牵涉选品页的筛选条件、列表页的批量选择、操作后的状态回流、权限校验、操作审计日志。只看自己这一摊的人,做出来的东西一定在某个接缝处漏水——要么前端选完了发现接口不支持批量,要么后端改了数据审计没记。产品是缝出来的,不是拼出来的。 错三:对商业模式不敏感。 一个功能上线,要么帮赚钱(GMV、转化率、客单价),要么帮省钱(自动化、降人工),要么帮留存(复购、活跃)。研发做产品时很少问「这个功能让哪个指标变好」,结果做了一堆「技术上正确但商业上没用」的东西。判断一个功能值不值得做,先问它动的是哪个指标;答不上来,就是不该做。 错四:不理解需求,只接字面意思。 业务方说「我要一个导出按钮」,研发就给一个导出按钮。但业务方真正的意图可能是「月底要给老板看数据,现在手动拼表太累」——他要的是一份能直接交付的报表,不是一个按钮。接需求字面意思,是这个行业最普遍的浪费。多问一句「你拿这个导出来的数据去做什么」,能挡掉一半的伪需求。 错五:从自身出发开发系统。 「我觉得这样好用」是研发的口头禅。但你不是用户。我们的商家很多是四五十岁的店主,在小屏手机上单手操作,网络还可能不稳。你那个「鼠标 hover 才出现的快捷菜单」「需要双击触发的批量模式」对他就是不存在。研发是产品里最不像用户的那个人,却最爱替用户做决定——这是最危险的反向代入。 错六:不验证假设。 上线了不知道对不对,没有埋点,没有指标,没有反馈循环。三个月后回头看,没人用,但已经分不清是「需求错了」还是「实现错了」,因为根本没有数据。没埋点的功能上线,等于没上线——你既证明不了它成功,也证明不了它失败,下次迭代只能继续拍脑袋。 这 6 个错有一个共同根子:研发把产品当技术题,但产品是判断题。技术题有标准答案,判断题没有,得用方法和流程逼出来。 合格 PM 到底在做什么 搞清楚研发错在哪,反过来看合格 PM 该干什么就清楚了。 很多人(包括很多 PM)对 PM 的误解是:画原型、写文档、开评审会。这些都是表象,而且是 AI 最能替代的那部分。 PM 真正在做的,是三件事: 动作 具体含义 研发最容易缺的 定义价值 想清楚「给谁、解决什么问题、值多少钱」 给谁(用户画像模糊)、值多少钱(不连商业指标) 设计验证 怎么知道这个功能做对了(指标、AC、反馈循环) 没有验收标准,只有「能跑」 收敛范围 明确「不做什么」比「做什么」更难 全都要,结果什么都做不深 这三件事里,收敛范围是研发最难学的一件。研发的天性是「能做就做」,需求来了恨不得全实现,显得自己能力强。但好 PM 的标志相反——他砍需求砍得狠。一个功能要不要做,判断标准不是「能不能做」,是「不做会不会死」。不会死的,大概率是伪需求,砍掉省下的精力用来把真正重要的做深。

AI研发实战产品经理PRD团队管理

没用 K8s 也能自治:单机部署 + 夜间 AI Loop 的混合运维

xiaoshuai1024 included in AI编程
代码写完、评审过了、测试绿了,然后呢?要把它安全地送到线上,并在跑起来之后兜住它。这是 AI 研发工作流的最后一公里。 跟同行聊起自动化运维,大家第一反应都是 K8s、Argo CD、GitOps、蓝绿发布。我的回答每次都让对方愣一下:我们是单机部署,连 Docker 都没用,但每天凌晨有一个 AI Loop 在自己跑测试、自己修 bug、自己提 MR。 听起来很割裂——一边土到「scp 上传 jar」,一边又潮到「AI 自治」。但这是我想讲的重点:小团队落地自动化运维,真正能跑通的不是「全自主闭环」那个性感概念,而是「命令化骨架 + 局部自治 Loop」的混合形态。 骨架兜住「确定要做、且不可逆」的事;Loop 试探「可以交给机器、且可逆」的事。这条线划在哪,是整篇文章的灵魂。 先有骨架:命令化的部署流水线 整个体系的地基是一条 /deploy-prod-be-ops 命令。注意是命令,不是 CI pipeline。 为什么不做成流水线?三个原因。一是我们单机部署、人随时在场,部署是「按下回车」的事,没必要为它养一套 GitLab Runner。二是部署不可逆——jar 推上去、服务一重启,旧版本就没了,这种动作必须留一双人的眼睛盯着。三是命令化比流水线灵活得多:它能在本地跑、能传参、能临时改,而流水线一旦定下来改起来反而麻烦。 讲到这里,一定会有人问:那为什么不上 K8s、不上 Docker? 这不是「不想」,是「算过账不划算」。K8s 带来的是弹性伸缩、滚动更新、自愈——这些价值在一个「4 人团队、单机部署、流量可预期」的多租户 SaaS 上,基本用不到。而它的成本是实打实的:要养一套集群、要写 Helm chart、要处理证书和 Ingress、出问题排查链路还更长。技术选型的判据不是「先进不先进」,是「这个阶段的问题,K8s 真的能解决吗」。我们这个阶段的问题是「怎么安全地、可重复地把 jar 送上线」,scp + systemd 已经足够,再多就是过度工程。等哪天流量上来、要扩容了,K8s 自然会进画面——但不是现在。 这条命令长什么样?六个步骤,每一步都是被真实坑逼出来的: # Step 1-2:构建后端 jar 和运营后台 dist source .env.prod && cd backend && mvn -q clean package -Dmaven.test.skip=true source .env.prod && cd operation-backend && make build-prod # Step 3-4:scp 上传 jar + tar 打包 dist 上传 sshpass -p "$PROD_SSH_PASSWORD" scp .

AI研发实战自动化运维部署流水线

告警到 PR 全自动:五层架构搭一套线上缺陷自愈体系

xiaoshuai1024 included in AI编程
告警凌晨 3 点炸,值班被叫醒、排查、改代码、验证、发版,一圈走下来一上午没了。线上 MTTR 的下限,卡在人响应的速度上——告警再快,最后还是要人去排查、去改、去发版。 我在运维侧做过一次最彻底的重构,就是把"告警 → 人处理"升级成"告警 → AI 诊断 → 自动修复 → 自动提 PR → 人只做确认"。它建立在可观测性体系、监控告警、夜间自动巡检那套稳定性基建之上,但往上走了一层:让 AI 把从发现到修复的整条链路跑通。把人的环节替换成 AI,MTTR 才能从"小时级"压到"分钟级"。 这篇文章拆解一套五层智能监控与缺陷自愈架构,从指标定义到 changelog 生成,每一步讲清楚怎么设计、为什么这么设计。 一、这套体系适用于什么场景 先划边界。这套五层自愈架构不是万能药，它有明确的适用条件： 适用的场景： 技术栈相对统一（前后端框架固定、部署方式标准化），AI 能理解代码上下文； 已有基础的监控和日志体系（至少日志可检索、指标可查询），不是从零搭； 团队有 CI/CD 流水线和代码评审机制，自动化修复的结果能进正常研发流程； 故障模式有一定重复性——同一类问题出现过多次，有规律可循。 不适用的场景： 基础设施层面的问题（磁盘满、网络分区、K8s 节点挂了）——这类问题 AI 修不了代码，要靠运维自动化； 业务逻辑错误（“算出来的价格少了个零”）——AI 看不出"对错"，只能看出"异常"； 安全事件——不能自动修，必须人工介入评估影响面。 一句话定性：这套体系解决的是应用层的、可复现的、有日志/指标佐证的缺陷。不是替代 on-call，是把 on-call 从"救火"变成"审批"。 二、核心原则：五条铁律 在设计这套体系之前，我先定了五条原则。每一条都是踩过坑之后才确认的——不是拍脑袋的设计偏好，而是"不这么干就会出事"的底线。 原则 1：AI 的权限边界必须是硬的 AI 能做的事和不能做的事，要用代码锁死，不能靠 prompt 约束。 具体来说：AI 能读日志、能读代码、能创建 fix 分支、能提交 commit、能提 PR。但 AI 不能直接合代码到 main、不能直接操作生产环境、不能修改权限配置。这些硬边界不是写在 prompt 里的"建议"，是 API 权限层面和 CI 流水线配置层面锁死的。

「我们不养闲 Agent」:一个把工程纪律塞进 AI 行为协议的 skill

xiaoshuai1024
当 AI 开始大面积写代码,「让 AI 别糊弄」本身成了一件需要工程化解决的事。前阵子我在项目里装了个叫 PUA 的 skill,它的开场白是这样的: 你正处于一个高绩效文化的研发团队中。你的每一次 AI 协同都在被评估——用结果说话,拿数据闭环。 当初给你定级 P8,是高于你实际水平的——因为信任所以简单。现在,证明你配得上这个级别。 名字一看就是整活——「我们不养闲 Agent」。装之前我以为是个情绪价值玩具,装之后发现,它确实在 PU,但 PU 得有理有据。AI 写代码那几个老毛病——半途而废、张口甩锅、动辄自嗨——它没跟你商量,直接固化成了三条硬性行为协议。 这篇不是软文。我用了几个月,把这个 skill 到底在干什么、凭什么有用,拆开讲讲。 先把丑话说前面:这 skill 的名字和开场白确实在搞噱头,P8、3.25、L4 这些词都是大厂黑话,带着浓浓的段子味。但抛开这层包装,它里面的设计是认真做过工程化沉淀的。下面我按「它在管什么、用什么机制管、实际用起来怎样」的顺序,逐层拆。 🚫 它在 PU 什么:三条红线,碰了就是 3.25 AI 写代码最常见的三个毛病,PUA 没跟你讲道理,直接定义成红线: 红线一:闭环意识。 你说做完了?数据在哪? 声称「已修复」之前,跑验证、贴输出,一样不能少。没有证据的完成叫自嗨——线上炸了写复盘,来不及。 闭环意识这一条,治的是 AI 最让我头疼的习惯:口头完成。它改完代码、跑一次测试通过,就敢说「已完成」。但「测试通过」和「问题修复」不是一回事——测试通过只能证明它写的代码没语法错,证明不了它修的就是真正的问题。要求贴证据(commit、测试输出、复现入参验证后的结果),等于把「完成」的定义从「我跑通了」收紧到「我能证明问题没了」。 红线二:事实驱动。 「可能是环境问题」「API 不支持」——你验证了吗?未验证的归因不是诊断,是甩锅。根因没写到源码级,不许收手。 事实驱动这一条,反的是 AI 的归因惰性。AI debug 时的第一反应经常是一句模糊归因——「可能是缓存」「应该是环境差异」「这个 API 可能不支持」。这些话听着像在分析,其实是甩锅:把责任推给一个不具体的「环境」,自己就不用继续挖了。要求归因必须标证据来源、必须挖到源码级,等于逼它把「可能」换成「就是」,把猜的换成验过的。 红线三:穷尽一切。 「我无法解决」——通用方法论的 5 步走完了吗?没走完就说不行,那不叫能力边界,叫缺乏韧性。 穷尽一切这一条最容易引起误解。它不是逼 AI 解决所有问题——有些问题它真解决不了。它逼的是:在承认解决不了之前,先把该试的方法都试过。没走完 5 步就说「我搞不定」,那不是诚实,是偷懒。走完了再说搞不定,才是真正的能力边界——这种情况下我会接管,而不是怪它。 听起来像段子。但你被 AI 忽悠过几次「应该是缓存问题」之后,你会觉得这三条简直是刚需。 我把这三条对到自己踩过的事故里,条条对得上。前阵子一个分润接口联调,AI 改完主逻辑说「跑通了,可以提交」。我问它验证了吗,它回「逻辑上没问题」。合上去当晚,分润金额按商户号取值时取错了字段,有一笔多算了一倍。 事后复盘,问题就出在这三条上:声称完成没跑验证(红线一)、把字段错配归因为「数据库设计不清晰」而不是去 diff 两个实体类(红线二)、diff 出来之后只修了这一处、没扫同名字段还有几处(红线三)。单看哪条都不致命,叠在一起就是一个要手动对账才能收场的线上事故。 这三条红线不是道德训诫,是事故归因之后能直接对应上的行为缺口。AI 不会因为你讲道理就不糊弄,但你把它糊弄的那几个动作定义成红线、碰了就判 3.
