【AI 研发实战】把"用 AI"做成工程体系:28 个命令和一条自治评审回路

xiaoshuai1024 included in AI编程
前三篇讲的都是「用 AI 做出了什么」。这篇回头讲「怎么用 AI」——不是技巧,是体系。 很多人对「AI 协同研发」的理解,还停在「每个人装个 Cursor,各写各的」。这种用法,4 个人撑死比以前快一倍。我们能做到 2 个月近百万行,是因为把 AI 协同沉淀成了一套可复用的工程体系:命令化的工作流、自治的评审回路、规范化的技能库。 关键不是 AI 多强,是这套体系让每个 Agent 开机即用。 一、不是「每个人会用 AI」就够 先说个反直觉的判断:「团队里每个人都会用 AI 工具」不等于「团队有 AI 协同能力」。 差别看两个场景。 没体系的用法:每个人各自调 AI,提示词五花八门,生成的代码风格不一,质量靠个人经验兜底,新人和 AI 都不知道项目规范,反复返工。 有体系的用法:PR 有标准命令、评审有自治回路、规范固化在仓库里、E2E 一条命令跑通,新人第一天装上工具就能产出符合规范的代码。 我们追求的是后者。这套体系跑起来,才是产能能持续放大的根,它让「4 个人 + AI」不只是 4 个独立单元,而是一支能协同的队伍。 二、命令化工作流:28 个命令覆盖全流程 我们把高频操作固化成了 28 个自建命令(/command)。流程上大致是:/prd 出需求 → 编码 → /tdd 测试驱动 → /mp-e2e、/op-e2e 跑端到端 → /kd-review 自治评审 → /pr-fullstack 提 MR → /deploy-prod-be-ops 部署。 为什么把操作固化成命令,有三个理由。 第一是一致性。「提 MR」这件事,以前每人一套 git 流程,现在 /pr-fullstack 一条命令,产出统一。AI 也用这条命令,不存在人和 AI 流程割裂。

Controller 直连了数据库、模块缠成死结:用 ArchUnit 把架构钉死

xiaoshuai1024 included in 架构设计
做过几年架构的人都遇到过这个场面:某天 code review,发现一个 Controller 直接 @Autowired 了 JdbcTemplate 在查数据库,绕过了 Service。问起来,答「这次比较急,先这么写」。 架构图贴在墙上,分层写得清清楚楚。可代码是活的,它会随每一次「图方便」悄悄偏移。等你发现时,Controller 已经散落在各处直连数据库、两个业务模块缠成了死结、循环依赖改都改不动。 靠人盯,盯不住的。 这篇就讲我怎么把架构边界写成测试,让机器来盯。 一、架构腐化的三种典型 不是有人故意搞破坏,都是「这一次图方便」累积出来的。我见得最多的三种: 跨层调用最常见。Controller 本来只该做路由和参数校验,但有人嫌 Service 套一层麻烦,直接把 JdbcTemplate 拉进来查库。一次两次没事,等这个模式扩散开,分层就成了摆设。 模块缠绕更要命。康豆有十几个业务模块——tenant、trade、circle、marketing…本来该各管各的。但今天 tenant 模块需要算一笔佣金,顺手 import 了 trade 的 Repository;明天 trade 又用了 tenant 的工具类。半年后想拆微服务,发现拆不动,因为没人记得清谁依赖谁。 循环依赖是缠绕的极端形态。A 依赖 B,B 又反过来依赖 A,编译能过(运行时可能爆),但删一个就崩一串。这种债最难还。 这三种,你写进规范文档没人看,写进周会强调没人记。唯一管用的,是让它编译不过、测试不过。 二、先解决一个现实问题:存量怎么办 引入架构测试,第一个拦路虎是存量。 康豆引入 ArchUnit 的时候,代码已经写了两个月了。一跑,Controller 直连数据层的违规 467 处,业务层用 SQL 注解的 84 处。要是让测试直接 fail,CI 当场全红,谁也推不动。 所以我们用了 ArchUnit 的 FreezingArchRule——冻结存量、只拦新增。 机制很简单:把规则包一层 Frozen.frozen(rule),首次运行时把所有存量违规记进一个基线文件(入库),之后只对新增违规报错。这样老代码不挡路,但谁再写新的越界代码,CI 立刻拦下来。 public final class Frozen { public static ArchRule frozen(ArchRule rule) { return FreezingArchRule.
