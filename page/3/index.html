技术驱动业务增长：前端架构如何赋能业务

xiaoshuai1024 included in 技术分享  架构设计
这是我走到全栈阶段的一个关键判断：当一个人开始要为整套 Vue + Node + Java 架构负责，问题就不再是“怎么把页面写好”，而是“技术怎么真正帮业务增长”。这篇整理的就是这一阶段的思考。 前端早就不是单纯的“页面展示层”了，它和用户、和业务的距离最近，技术上也足够成熟，具备帮业务增长的条件。 下面从数据流视角，结合一线实践，聊聊前端架构怎么真正帮上业务，而不是停留在“和业务脱节”。 一、现状：前端技术成熟，底子已经打牢 前端早已突破“切图、写页面”的初级阶段，形成了完整且成熟的技术体系。 主要体现在三个层面： 工程化工具完善：Webpack、Vite 等简化构建流程，降低开发复杂度。 AI 编程普及：AI 工具承担重复编码，让开发者聚焦核心业务逻辑。 数据流管理规范化：为业务数据流转提供可靠底座。 经过多年迭代，前端核心技术栈趋于稳定，能灵活适配各类业务场景，整体技术架构大致如下： 工程化和 AI 结合起来，前端开发效率上了一个台阶。 现在前端已经从“成本中心”向“价值中心”转型，具备了帮业务增长的技术实力。 二、前端架构最适合帮的业务场景 前端的几条核心优势——贴近用户、快速迭代、交互灵活——决定了它更适配前台型业务。 结合一线实践，以下几类场景最能体现前端架构的价值： 1. 营销类业务 这类业务的核心需求是快速上线、高频迭代，比如节日促销、新人福利、裂变拼团等活动。 前端架构通过组件化、低代码模式，能快速搭建营销页面，适配 H5、小程序等多渠道。同时通过前端埋点采集用户行为数据，为营销方案优化提供支撑，做到“快速试错、快速迭代”。 2. 小程序/公众号业务 小程序、公众号连接线上线下，对开发效率和用户体验要求很高。 前端通过多端复用技术，一套代码适配多终端，大幅降低开发与维护成本。借助规范化的数据流管理，登录、订单、推送等核心逻辑也能流畅运行。 3. APP 前端开发（含混合开发） APP 的竞争力在于用户体验和交互流畅度，这正是前端架构的核心发力点。 通过性能优化、组件复用，前端架构能持续提升 APP 的加载速度和交互体验，直接影响用户留存与转化。同时高效对接 SpringBoot 后端接口，支撑商品、购物车、结算等核心流程稳定运行。 4. 企业前台业务系统 包括 CRM、用户中心、订单管理前台等，直接面向企业员工或外部客户，对操作效率和易用性要求较高。 前端通过规范化的数据流设计、组件复用，优化操作流程，降低员工使用成本，间接推动业务增长。 5. 数据可视化业务 业务监控大屏、用户画像分析、经营报表等场景，需要把复杂的业务数据直观呈现出来。 前端通过 ECharts、d3、threejs 等工具，实现多维度可视化，帮助业务人员快速捕捉数据规律、发现业务问题。 三、前端帮业务的核心形式：低代码平台 在众多形式里，低代码平台是投入产出比最高、最贴合业务需求的一种。 它契合前端“快速迭代、高效复用”的优势，能快速打通技术与业务的壁垒，让业务需求快速跑起来。核心运行逻辑和系统关系如下： 低代码平台帮业务的几个原因 开发效率极致：可视化拖拽搭建页面，不用写大量重复代码，把传统前端开发的“天级”周期压缩到“小时级”甚至“分钟级”，快速响应业务的紧急需求。 投入产出比突出：业务人员经过简单培训即可参与开发，减少对专业前端人力的依赖；同时支持快速试错，营销、小程序等场景可快速上线测试、根据反馈调整优化。 场景适配灵活：平台内置标准化组件，覆盖绝大多数通用场景，同时支持自定义组件扩展，满足个性化业务需求。 数据流规范化：内置标准化数据流模板，对接后端接口更高效，避免数据混乱、冗余，为业务决策提供可靠支撑。 四、前端帮业务的其他形式 除了低代码平台，结合前端技术特性，还有几种方式可以帮到业务，和低代码平台形成互补。 1. 组件化架构复用 搭建企业级前端组件库，基于 Vue3 封装标准化组件，统一设计规范与交互逻辑。这些组件可跨项目、跨业务复用，减少重复开发，同时保障多产品的体验一致性。

【AI 研发实战】AI 时代的大型全栈项目架构设计:把"AI 保障"建进系统骨架

xiaoshuai1024 included in AI编程
这篇讲的是我走到 AI 开发阶段的一个核心判断:当大量代码交给 AI 生成,架构师的活儿从"设计能跑的系统",变成了"设计能兜住 AI 的系统"。它承接的是我全栈阶段沉淀的那套多租户、契约、迁移治理的底座——那些东西在 AI 时代不但没贬值,反而最值钱。 我的结论先放这:AI 让代码产得快,也让坏代码产得快。一个能稳定输出代码的系统,如果骨架本身兜不住,产能越高、负债累积越快。 所以我现在反而比以前更花时间在那些"无聊"的基础设施上——多租户隔离、权限过滤器、契约扫描、迁移治理、可观测。不是因为它们新,恰恰因为它们旧、因为它们"枯燥",才成了 AI 时代最值钱的东西。 先说我们这个项目的底座 我们做的是一个多租户 SaaS,典型的三段式结构:C 端小程序 + 运营后台 + Java 后端。规模不算夸张,但因为是多租户,出了事影响面大,所以对隔离和契约特别较真。 架构底座就这么几样,说出来都是老生常谈: 多租户行级隔离,每张业务表都带 tenant_id,查询自动带过滤; RBAC 权限过滤器链,权限校验不靠人记得加,而是在过滤器里强制走; 契约驱动,前端调的 URL、后端的路由注解、SQL 引用的表、Flyway 的建表 DDL,两两对得上才算数; Flyway 迁移治理,版本号唯一、有 seed 数据、不许手改线上表结构; 可观测,Prometheus / Grafana / ELK 一套,出问题能追溯。 没有任何一项是"AI 时代的新玩意儿"。但我的判断是:这些东西在 AI 时代的权重,是被拉高的,不是被降低的。 为什么"无聊"的基础设施变重要了 以前一个资深工程师手写代码,他脑子里有上下文:这条 SQL 多租户场景下要带 tenant_id、这个接口对外要校验权限、这次改表要写迁移。人会忘,但频率可控,靠 code review 兜得住。 现在不一样了。AI 生成代码的时候,它没有项目级的敬畏感。它会"合理地"省掉一个 JOIN 条件,因为单看那条查询,不带 tenant_id 也能跑通;它会"优化"掉一段它看不懂的并发守卫,因为编译器不报错。 问题不在 AI 笨。问题在于:单点视角的合理性,在系统视角上是危险的。 所以能把"系统视角"焊死在骨架里的那些机制,就变成了保险绳。tenant_id 自动注入、权限走过滤器、契约两两校验、迁移版本唯一——这些是机器能 100% 执行的硬约束,不依赖任何人或 Agent 当下是否清醒。

【AI 研发实战】我们的 AI 研发工作流:从原型到上线的全链路

xiaoshuai1024 included in AI编程
这是我 AI 开发阶段的主干实践。前端的工程化纪律、全栈的链路意识,在这一步被重新组织成一条可复用的 AI 研发流水线,后面讲评审、测试、契约的文章,都拿它当地图。 先说我的判断:大多数人用 AI 写代码,还停留在「光标后面补两行」,出 bug 了再问一句「这怎么报错了」。这跟当年用 IDE 自动补全区别不大,只是更聪明了点。 真正让玩法变了的,是把整条研发链路摊开,让 AI 在每一步都进场,把它当成流水线上的工位,而不是高级补全。 全链路总览 整条链路长这样: 确认需求 → HTML 原型 → 计划 → 多轮并行头脑风暴 → TDD 执行(红/绿/重构)→ 自治多 Agent 评审 → 提交/MR。 每一步都不是「问 AI 一句话」那么简单。下面逐个拆开。 第一步:需求之后,先出 HTML 原型 确认需求之后,先出 HTML 原型,而不是直接开写。 原因很直接:对着空气写需求,人和 AI 都容易跑偏。先把界面用 HTML 画出来,需求和实现的歧义在原型阶段就消掉了大半。 第二步:计划阶段,确立唯一事实源 计划阶段是整个工作流里我最强调的一环。 做法是做两轮计划,产出一份任务图 JSON,并约定它是唯一事实源。 这条规矩的本质是解决并行的一致性:每个 agent 各自理解任务、各干各的,很快撞车——你改的文件我也要改,我设计的接口跟你对接不上。任务图摆在中间,谁做什么、依赖谁,一目了然。 没有这个事实源,并行越多越乱。 第三步:多轮并行头脑风暴 然后是多轮并行头脑风暴。 我们有一条近乎硬约束的原则:能开并行 subagent 就尽量多开,团队甚至把「尽可能多地开并行 subagent 执行」写进了规矩里。 两种用法的本质区别在于: 串行问 AI,是把它当搜索引擎。 并行扇出去,是把它当一支团队——同一个问题让多个 agent 同时从不同角度去想,再汇总。 效率和质量都不是串行能比的。 第四步:TDD 执行,红 / 绿 / 重构 执行阶段走 TDD:红、绿、重构。

【AI 研发实战】没有 CI 的日子里:用自治多 Agent 评审回路兜住代码质量

xiaoshuai1024 included in AI编程
在「前端 → 全栈 → AI 开发」这条线上,这篇落在 AI 开发阶段,讲的是我把 AI 协同研发做成工程体系时,用来替代传统 CI 的那道质量关。 跟同行聊起质量保障,大家第一反应都是「你们 CI 怎么配的?跑多少分钟?」。我的回答每次都让对方愣一下:我们的仓库,一条 CI 流水线都没有。 没有 GitHub Actions,没有 GitLab CI,没有 Jenkins。.yml 里一个 job 都找不到。 那质量靠谁?靠一套自治多 Agent 评审回路,我已经稳定跑了一年多。这篇把它拆开:它兜住了什么,又诚实地讲一句,它替代不了什么。 回路全貌:从「人盯」到「流程兜底」 先说整条回路长什么样,后面再展开关键设计: scope 分桶 → 并行只读评审 → 流式自动修复 → 双计数器收敛 → 硬测试门禁 → 放行提交。 整个评审回路是我们那条 AI 研发工作流 的最后一道关。前面 TDD 已经把红/绿/重构走完了,到这里要决定「这坨改动能不能进仓库」。 关键的判断是:既然能开并行 subagent,就尽量多开。这是团队写进硬约束的一句话。所以这套回路从设计第一天起,就是奔着「并行扇出」去的,不是排队串行。 关键设计一:评审 Agent 只读,而且 worktree 隔离 这一条是被事故逼出来的硬规矩。 早期做法是让评审 Agent 直接在工作区里看代码、顺便「帮我改两行」。代价是:评审 Agent 自作主张地把一段它「觉得冗余」的代码删了,而那段是并发守卫。这件事我专门写过复盘 《编译能过 ≠ 代码正确》:270 行静默蒸发,mvn compile 照样绿。 从那以后铁了心:评审 Agent 一律只读。

AI研发实战代码评审质量门禁

【AI 研发实战】编译能过 ≠ 代码正确:一次"幽灵删除"事故复盘

xiaoshuai1024 included in AI编程
这条事故属于我「AI 开发」阶段的质量门禁系列:当 AI 开始大量写代码,「编译绿」这件事的含金量被彻底重估。复盘之后,我们团队立了一条铁律挂在评审回路最显眼的位置—— mvn compile 通过 ≠ 代码正确。 过去那些年,我们把"编译绿了"默认等同于"代码对了"。这个默认在人工时代勉强成立,在 AI 大量写代码的今天,代价会被成倍放大。 一句话讲完整件事:编译器只认语法契约,不校验行为;AI 删一段并发守卫时,编译照样绿、它还底气十足说"已完成"。所以门禁不能停在"能编译",得升级到"得过 verify";任何"我搞定了"必须自带证据(Proof-of-Work),焊进门禁,不靠自觉。 一次看起来无害的"搜索优化" 事情起因是一次很普通的提交,说明写的是"搜索优化"。diff 拉出来看也没问题:搜了什么、优化了什么,逻辑闭环,命名干净,像是 AI 认真做的一次重构。没人会为这种提交专门去翻一遍并发模块——毕竟提交说明里一个字都没提并发。 我们当时的评审回路已经跑起来了,流程走得很顺: Agent 评审 → 自动修复 → 收敛; 然后本地 mvn compile 一把过,绿灯; 提交,合进去。 直到后来有人回头去看那段并发相关的核心代码,才发现不对劲——整整 270 行,没了。不是改坏,是静默删除。那是一处并发安全的守卫逻辑,平时不出声,但在高并发路径上是兜命的。 那 270 行到底是什么 既然删掉的是"守卫",就得说清楚守卫长什么样,不然这事听着玄学。 并发安全守卫在我接触的 Java 服务里,通常落地成这么几类东西:入口的限流(令牌桶 / 漏桶,挡住突发流量)、关键资源的分布式锁(同一把 key 不能两个节点同时改)、写操作的幂等键(同一笔请求重复提交只生效一次)、还有租户 / 用户级隔离(查询不能串号)。270 行代码,基本就是这几类的组合:加锁顺序、超时降级、重试上限、隔离上下文传递。 这类代码有个共同点:它不参与"功能能不能跑通",只参与"高并发时会不会出事"。你在本地单线程点一遍接口,它一行都不会被执行;压测、大促、灰度放量那一刻,它才是兜命的那道闸。 我特地拆一下"删了也不立刻炸"的原因,这点是整个事故最难解释、也最容易被低估的地方: 触发是概率性的。分布式锁只在两个请求同时打到不同节点时才显形;幂等键只在重复提交时才暴露;限流只在 QPS 过阈值时才生效。本地手测一遍,触发概率约等于零。 失败是延迟的。数据写花、串号、超卖,往往要等到对账、或者用户投诉才被发现,事故现场已经凉了,日志可能还查不全。 影响是积累的。删一段守卫不会让 P99 变差,只会让"某一次高峰"直接塌方。日常监控全是绿的,出事那一刻就是大事故。 这三个特征叠加,就解释了为什么这类 bug 几乎不可能在"刚提交"时被抓到——它的潜伏期可能是几天、几周,甚至到大促当天才爆。等爆的时候,事故链早就被后续提交层层覆盖,溯源极其痛苦。 这就是为什么删掉它,编译器不报错、单测不红、本地联调也照跑、线上能稳跑好几天。它和功能的耦合是运行时的、概率性的,不是编译期的、确定性的。一句话,它是行为依赖,不是语法依赖——语法层面,删了它引用全部干净,等于没存在过。 为什么编译会"绿" 删掉 270 行并发守卫之后,mvn compile 照样通过,一点脾气没有。 要理解这点,得把编译器的职责边界画清楚。它真正管的就这几样:类型匹配(参数是 String 你别传 int)、方法签名(调用的方法在不在、可见性对不对)、符号引用(类、字段、import 指向的东西真的存在)、语法结构(括号、分号、泛型擦除)。这些是编译期就能确定的契约。

【AI 研发实战】提交前的四道契约闸:用脚本守住 API、SQL、权限、迁移边界

这是 AI 开发阶段的一篇:讲我怎么用一道提交前门禁,守住编译器和 lint 都管不着的「跨层接缝」。它属于我把 AI 协同研发做成工程体系的一部分,也就是契约门禁。 我最怕的 bug,从来不是某个函数写错,而是那种「编译器管不着、lint 管不着、只有跑到线上某条真实路径才炸」的跨层错误。 举个例子。前端调一个 /api/orders/detail,后端最近重构,路由注解改成了 /api/order/detail,少了个 s。结果就是:编译两边都绿,单测各自模块都过,然后某个周二下午运营说小程序白屏了。 错在层与层的接缝上 这类 bug 的共同点很明确:错不在某一层,错在层与层的「接缝」上。 编译器只能保证语法契约成立。 它保证不了行为契约。 更保证不了「前端发的请求」真的能落到「后端注册的那个路由」上。 这件事我专门写过一篇《编译能过 ≠ 代码正确》:一次 270 行并发守卫被静默删掉的事故,讲的就是同一个道理。 所以我在提交前架了四道「契约闸」。 四道闸分别守什么 说穿了就是一个 pre-commit 脚本里跑四层扫描,每一层守住一类跨层契约: 前端调用的 URL ↔ 后端路由注解。前端代码里所有请求路径,后端都得有对应的 @RequestMapping/路由注解接住;反过来,后端注册了但前端从没调过的路由也会标出来——通常是改名漏改,或接口已废弃。 后端 SQL 引用的表 ↔ Flyway 建表 DDL。Mapper 里 SELECT * FROM xxx 的那张表,Flyway 迁移脚本里必须真的建过。少建一张、拼错一个表名,当场拦下。 权限 key ↔ 迁移脚本里的权限种子数据。代码里 @RequirePermission("order:export") 用到的 key,在权限初始化的种子数据里得存在。否则权限配了等于没配,线上谁都能调。 Flyway 迁移版本号唯一。两个 V12__xxx.sql 抢同一个版本号,看着低级,但合并代码时真的会发生,而且 Flyway 起飞失败能把整个服务拖挂。 这四样东西,编译器一个都不会管。它们都是「跨层的、声明式的、靠人对齐」的契约,而人和 AI(尤其是 AI)恰恰最不擅长对齐这种东西。 为什么放在 pre-commit 一个字:早。 跨层 bug 越往后修越贵,这是铁律:
