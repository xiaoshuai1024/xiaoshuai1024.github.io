京东大促缓存复盘:10W QPS 扛住了,11W 倒下了

xiaoshuai1024 included in 架构
讲一次大促前夕刚做完缓存优化、活动开闸后被现实教做人的事。整个故事一句话:我把缓存放 cache(缓存)这件事做得很扎实,却忘了还有另一类流量叫 attack(攻击),它压根不走缓存。 我那时在京东云做官网架构。大促前的几个月,我主导了一轮缓存改造——把吞吐量从 200 QPS 干到 2 万常态,大促峰值能扛到 10W。结果活动当天流量叠了一波 DDoS 冲到 11W,缓存架构被绕过,连接池打满,服务响应缓慢。最后靠 WAF 清洗和限流把系统拉回来。 这篇复盘整个过程,核心教训一句话:多级缓存解决的是流量效率问题,解决不了流量安全问题。 大促前夕:多级缓存怎么搭的 改造前的官网是单层 SSR,峰值 200 QPS 就顶不住。大促一开闸,服务直接拖地。 “200 QPS 就顶不住"听着夸张,但单层 SSR 的瓶颈很真实:每个请求都要服务端完整渲染 HTML,渲染过程要查数据、拼模板、跑逻辑,全是 CPU 活;而且 SSR 实例的本地资源(CPU、内存)是有限的,扛不住并发渲染。200 QPS 不是网络带宽的极限,是单实例渲染能力的极限。要扛更多,要么加实例(贵),要么减少回源(缓存)。 我搭了一条 6 层多级缓存链路,让流量逐层消化,尽量少回源: 每层各管一段: ① CDN:边缘节点缓存静态资源,兜底静态页。绝大部分请求在这一层消化。 ② WAF:页面级缓存 + 流量清洗 + 限流 + 安全策略。当时这层只在大促专项开启,平时不常驻。 ③ Nginx:反向代理 + 健康检查,故障节点自动摘除。 ④ SSR 实例本地 LRU:渲染好的 HTML 缓在本地,后端全挂时降级兜底。 ⑤ Node BFF:接口聚合层缓存,减少后端回源。 ⑥ Redis:分布式缓存热点数据,带互斥锁防击穿。 兜底也做了:SSR 渲染的静态页 CI 构建后上传 OSS 并刷新 CDN,后端真全挂时降级展示 CDN 静态页。

3 人两月交付 512 功能:30 条 AI 命令锁死测试与评审纪律

xiaoshuai1024 included in AI编程
我主导的一个全栈项目(Java 后端 + Vue 小程序 + 运营后台,多模块多仓库),3 个人两个月交付了 512 个功能点。 这个数字不是堆人堆出来的,也不是靠加班。我把研发流程里每一步该守的纪律,都固化成了 AI 可执行的命令。最终沉淀了 31 条命令、3400 多行,挂在研发流程的每个环节上。 背后是两条工程主线,我先把它们说清楚,后面每一节都是它们的展开: harness 自治:人只定规则,把执行全权交给 AI。人不在审查的每一轮里,只看最终报告。harness(线束/约束)的意思是——命令不是"让 AI 做事",是"给 AI 套上纪律边界,让它在边界内自治"。 loop engineering:把审查、修复、验证都做成自动循环,直到收敛(零问题)才退出。能循环的就不让人介入,把"等同事有空"这类串行等待彻底消掉。 这是 「一切皆可脚本化」那篇的延续——那篇讲「为什么该脚本化」,这篇讲「具体沉淀了什么、怎么用命令的形态落地」。 一、命令不是脚本,是「流程契约」 先澄清一个概念。很多人把 AI 命令理解成「一段能跑的脚本」,这是低估了它。 我项目里这 31 条命令,是流程契约:每条命令规定了一个研发环节里「AI 该怎么做、必须守什么纪律、什么时候停」。脚本只负责执行,命令还要负责约束 AI 的行为边界。 举几个对比: 脚本:「跑一下测试」→ AI 跑了,失败也不知道怎么处理。 命令:/tdd → 锁定红绿重构、缺服务先探测再启动、跑通后必须汇报证据。AI 的行为被约束在 TDD 纪律内。 这个区别决定了命令体系能不能撑住 512 个功能的交付量。光会跑脚本,AI 跑偏了你拉不回来;有了契约,AI 在每个环节的行为都可预期。 再补一个对比,体会会更深。脚本版的「部署」是 ./deploy.sh,跑完就算,中间哪步挂了 AI 甩一句「脚本报错了」就停在那。命令版的 /deploy-prod-be-ops 把「构建产物 → 上传 → 重启 → 冒烟验证」串成一条带验证的链,任何一步失败都明确报出是哪一环,部署完还强制跑 /prod-testing 冒烟。区别在于:脚本只描述「做什么」,命令还描述「做完之后怎么确认做对了」。前者 AI 执行完就走人,后者 AI 执行完必须自证。512 个功能能交付,靠的就是这种「自证」嵌在每一步里。

380 页 UI 靠人走不过来:像素 diff 与 vision 巡查的视觉验收

xiaoshuai1024 included in AI编程
最近一个项目 380 个页面,每个页面停留十秒就是一个多小时,还不算记录、复现、回归。第二次改完再走一遍。第三次。靠人走 UI 验收根本不现实。 我的做法是按场景分两条路(和测试纪律、契约闸是一套,前面讲「代码对不对」,这篇讲「长得对不对」): 新功能:有高保真原型锚点,做像素级比对——保证 PRD 里的每个功能都被实现、样式和设计稿一致。 老功能:没有原型可比对,做 vision 巡查 + 代码审计交叉——保证改动不破坏兼容性,顺便扫历史遗留问题。 这两条路用的工具不一样、判断标准不一样、能回答的问题也不一样。混在一起做,要么新功能漏检样式偏差,要么老功能因为没有原型而无从下手。 为什么 UI 走查最难自动化 功能测试有断言,性能测试有指标,UI 走查靠的是人眼——主观、慢、还容易漏。 380 个页面,每个页面停留十秒就是一个多小时,还不算记录、复现、回归。第二次改完再走一遍。第三次。没人扛得住。 丢给 vision 模型批量看图,听起来很美。实际跑下来,它会自信地报错,其中相当一部分是幻觉。直接采信,要么误报一堆不存在的 bug,要么漏掉真问题。 所以问题不是"要不要用 AI",是分场景用对工具。 人眼走查还有个被低估的成本:结论不可复现、不可传承。同一个页面,今天的小张说「可以」,明天的老王说「间距不对」,没有客观标准,验收质量完全取决于当时谁在审。更要命的是,人审过的页面出了问题,你没法回溯——「这个间距上周验收时到底量过没有?」没人答得上来。自动化验收的深层价值之一,就是把「当时谁在看、什么状态算过」固化成可复跑的脚本和可留存的报告,让验收从「一次性的人工判断」变成「可追溯的工程产物」。 这也是为什么我一直坚持验收结果要落成代码里的断言或扫描规则,而不是停留在一份 markdown 报告里。报告会丢、会过时,断言和规则每次跑 CI 都会执行,问题一冒头就拦下来。视觉验收只有沉淀成可执行的东西,才算真正嵌进了研发流程,否则就只是一次性的走过场。 视觉验收三原则 往下展开之前,先把这套方法的底层判断收敛成三条原则。后面所有的工具选择、流程设计,都是这三条的落地: 原则一:分场景用对工具。 有原型就用像素 diff(确定性高),没原型才用 vision(会幻觉)。混用等于把不确定性引入确定性场景,得不偿失。 原则二:真相源必须机器可读。 设计稿不能只活在 Figma 里(人能看,代码扫不到),必须沉淀成 design token——颜色、圆角、渐变都在 variables.css 里定义,代码能直接读。这一层是「正确答案」的物理载体,没有它,代码审计无从谈起。 原则三:vision 只提疑问,不下结论。 vision 的输出永远是「疑点列表」,不是「bug 列表」。每个疑点必须经代码审计交叉验证,才能升级为 bug。这条治的是 vision 的过度自信——它越自信,越要警惕。 这三条是后面所有流程的根。任何视觉验收方案,只要违反其中一条,迟早翻车。 值得强调的是,这三条原则的排列顺序也是优先级顺序。团队资源有限时,该先建哪一层?答案是从原则二(真相源)开始。没有 design token 这层机器可读的基线,vision 巡查(原则三)无处交叉验证,像素 diff(原则一)也没有稳定的颜色锚点。我们项目就是先花了两个版本把全站颜色、圆角、渐变收敛进 variables.css,统一切到 token 引用,之后才谈得上做 diff 和 vision 巡查。基线没建好就上自动化验收,等于在沙地上盖楼,跑出来的结果不可信,反而会让团队对自动化验收失去信心。

收藏夹吃灰怎么治:用 RAG 给自己搭一个能提问的私有大脑

xiaoshuai1024 included in AI编程
我攒了十年的收藏夹、笔记、PDF,真到要用的时候一个都翻不出来。面试前临时背,学新东西时重新查,重复劳动。 资料够多,问题在它们是死的——只能人去翻,不能被提问、不能被检索、不能主动复述。 《RAG 知识库从 0 到 1》讲的是给业务系统搭知识库。这篇反过来,讲给自己搭一个学习大脑。 一、个人学习的三个真实痛点 先对号入座,看看是不是都中过: 收藏即遗忘。看到好文章存下来,“以后再看”,然后就没有以后了。收藏夹两千条,真记得的不到十条。 笔记即坟墓。记完的笔记再也打开。不是不想复习,是检索成本太高——记不清写在哪、关键词是什么。 学了等于没学。一个知识点当时懂了,过两个月忘干净。因为没有一个"定期把知识推回脑子里"的机制。 资料如果能被提问、被检索、能主动复述,学习闭环就成立了。这是个人 RAG 知识库要解决的事。 这三个痛点背后有一个共同的根因:知识的存储方式和大脑的使用方式不匹配。大脑不是靠「翻目录」找知识的,是靠「联想」——看到一个场景,联想到相关的知识。但收藏夹和笔记软件都是「目录式」的,要找知识得先记得「它存在哪」,这本身就需要记忆,而记忆正是最容易失效的。RAG 知识库把存储从「目录式」变成了「联想式」——你不用记得知识存在哪,你只需要用自然语言描述你的问题,RAG 自动把相关的片段「联想」出来。这个匹配方式,和大脑的工作方式一致,所以它比翻笔记有效得多。 还有一个被忽视的痛点:知识之间是孤岛。前端笔记里记了「虚拟 DOM 的 diff 算法」,架构笔记里记了「微前端的子应用隔离」,这两条知识其实是相关的(微前端隔离就用到了 diff 原理),但它们存在不同的笔记里,你从来不会把它们关联起来。RAG 知识库可以跨主题检索——你问「微前端隔离和虚拟 DOM 有什么关系」,它从两个库里各捞片段,帮你在孤岛之间搭桥。这种「跨域连接」是人脑最累的活,但恰好是 RAG 最擅长的。 二、核心思路:把资料变成"可对话的私有大脑" RAG 的原理在业务版讲过:问大模型之前,先从私有资料里检索相关片段,塞进 prompt,让它看着资料回答。 个人版的架构一模一样,只是「租户」换成了「学习主题」: 前端资料库:笔记、收藏、PDF、博客剪藏 切片与向量化:把长文切成块,算成向量存起来 按主题隔离:前端一个库、Java 一个库、架构一个库,互不串味 检索 + 对话:问问题时,先从对应主题库捞片段,再让 AI 看着回答 关键差异在用法。业务知识库只做问答,个人学习库要支持更多场景,靠的是改 prompt,不是改架构。 数据流向:从原始资料到回答 整套数据流一次说清,免得你照着搭时漏环节: 资料入库:把 PDF / Markdown / 网页剪藏扔进 incoming/ 目录。 切片:按段落 + 重叠窗口(overlap 200 字)切,避免关键上下文被切断。这一步的参数直接影响检索质量——切太大召回不准,切太小上下文丢失。 向量化:每块文本过 embedding 模型,生成向量(用 OpenAI text-embedding-3-small 或本地 bge-large-zh)。 入库:向量 + 原文一起存进 ES / pgvector,带上 topic 字段做 metadata filter。 检索:用户提问 → 同样向量化 → 在指定 topic 库里做 top-k(一般 k=5)相似度检索。 组装 prompt:把 5 个片段拼进 system prompt,加上"看着资料回答,没覆盖就说不知道"的约束。 生成:大模型出回答,标注出处片段编号。 《RAG 知识库从 0 到 1》有完整的 Spring AI 落地代码。真正难的不是写出来,是调参——chunk size、overlap、top-k、相似度阈值,这四个参数决定了回答质量,得拿自己的资料反复调。我踩过最大的坑是 chunk size 默认 1000 字太大,导致一个 chunk 里混了两个主题,检索时召回不准。改成 400 字 + overlap 100 字后,精度立刻提上来。

当 AI 把"写工具"打成零成本:研发流程没有只能人肉做的环节

xiaoshuai1024 included in AI编程
研发流程里,已经没有"只能人肉做"的环节了。这句话不是口号,是我这一年的实际工作状态。 前面十几篇讲的是一个个具体环节——认知转变、工作流、契约闸、自治评审。这篇不再拆单点,直接把整条研发流程过一遍,看每一环现在怎么跑。 这篇的核心判断是:当 AI 把「写工具」的门槛打到接近零,研发流程里所有「按规则执行」的环节,都值得也都能被脚本化、工具化。这不是说人没事干了,而是人从「执行」解放出来,专注在「定规则」和「处理例外」上。整条研发链路的效率,因为工具化而被整体拉高。 我先讲清楚「以前为什么不写脚本」(成本账),再讲「现在为什么全划算」(门槛降),然后过一遍研发流程的每个环节看具体怎么脚本化,最后聊聊脚本化的进阶形态——MCP。这套认知是我在 AI 协同研发跑了一年之后沉淀的,不是理论推演,是每天工作状态的真实写照。 一、以前为什么那么多事"不写脚本" 回看过去十年,我经手的需求里,真正写脚本的不到一成。不是不会,是不划算。 写一个发布脚本,要先调研 API、读文档、处理 token、处理异常、写完还得维护。投入两三天,解决的是每月一次、每次五分钟的发布。账一算,亏。所以宁可打开后台手动点几下。 脚本的成本由人承担——人的时间贵、耐心有限、写一次性工具会烦。于是研发流程里沉淀出一堆重复、低创造、又没法标准化进正式系统的事:导数据、发周报、查日志、对账、跑回归、整理 PRD 评审纪要。 它们漂浮在「正式系统」和「纯手工」之间,谁都不愿意为它投资一个工具。 这类「漂浮的杂活」有个共同特征:单次成本低(五分钟到半小时),但累积频次高(每天/每周重复)。算总账,它们吃掉的时间不少,但因为每次单次成本低,人不会专门去优化它——「就五分钟的事,写个工具不值得」。结果五分钟的事做了一千次,等于浪费了五千分钟,但每次都觉得「不值得优化这次」。这是人脑决策的盲区:对「单次低成本但高频」的任务,直觉判断是「不值得」,理性算账是「必须优化」。 AI 把写工具的成本打掉后,这个盲区就被填平了。现在「五分钟的事」也值得花两分钟让 AI 写个工具,因为工具写完能跑一千次,节省的五千分钟净赚。决策门槛一降,漂浮的杂活纷纷落地成工具,研发流程的「灰色地带」被清理干净。 二、AI 把"写工具"的门槛打掉了 转折点是 AI 写代码这件事本身变便宜了。 这里要区分两个概念:「写代码便宜」和「写工具便宜」。写代码便宜,指的是 AI 能快速生成业务代码;写工具便宜,指的是 AI 能快速生成「帮你自己省事的工具」。两者都是 AI 写代码,但价值方向不同——前者是交付产品,后者是提升自己的效率。后者才是这篇要讲的重点:当 AI 能快速为你写工具,你的个人效率会被持续放大。 我的实际体感:Claude Code 跑下来,我描述一句"把这个博客同步到三个公众号平台",它能自己完成调研可行性、设计架构、写 47 个单测、提交、修 bug、提 PR。这要是手工,至少一周。 写脚本的决策模型变了: 以前:这事值不值得我花三天写工具?→ 大概率不值得 → 人肉。 现在:我花三分钟描述需求,AI 半小时交付。→ 几乎都值得。 门槛一降,原来「不划算」的那些事,现在全划算。 但「全划算」不等于「全该做」。工具化的判断有个边界:这件事是不是真的重复。有些事看起来重复,其实只发生两三次就消失了——给它写工具是浪费。判断标准是「未来半年会不会再做五次以上」,会,值得工具化;不会,人肉更划算。AI 让写工具便宜,但不代表什么都要写工具,「值得」的判断还是要人来做。 还有一类事不适合工具化:需要「现场判断」的。比如和客户沟通、做技术选型的权衡、评估一个人的能力——这些高度依赖上下文和经验,每次情况都不一样,没法抽象成固定流程。强行工具化只会把灵活的事做成死板的表格。工具化针对的是「流程固定、规则明确」的环节,模糊地带仍然要人。 决策模型对比 重复任务 以前(人写脚本) 现在(AI 写脚本) 决策 每月一次的发布 投 3 天写 → 不划算 → 人肉 描述 3 分钟 → AI 半小时 → 划算 ✅ 脚本化 每天的对账 投 1 天 → 勉强划算 → 写了不维护 描述 5 分钟 → AI 1 小时 + 测试 ✅ 脚本化 一次性数据迁移 投 2 天 → 一次性用 → 亏 描述 10 分钟 → AI 2 小时 → 划算 ✅ 脚本化 复杂架构决策 AI 也帮不了多少 AI 也帮不了多少 ❌ 仍靠人 最后一行是边界:决策类的、需要业务上下文的活,AI 杠杆为 1——这部分仍然是人的核心价值。脚本化能放大的,是「按规则执行」的环节,不是「定规则」的环节。

Controller 直连数据库怎么治:用 ArchUnit 把架构钉成测试

xiaoshuai1024 included in 架构设计
做过几年架构的人都遇到过这个场面:某天 code review,发现一个 Controller 直接 @Autowired 了 JdbcTemplate 在查数据库,绕过了 Service。问起来,答「这次比较急,先这么写」。 架构图贴在墙上,分层写得清清楚楚。可代码是活的,它会随每一次「图方便」悄悄偏移。等你发现时,Controller 已经散落在各处直连数据库、两个业务模块缠成了死结、循环依赖改都改不动。 靠人盯,盯不住的。 这篇就讲我怎么把架构边界写成测试,让机器来盯。 一、架构腐化的三种典型 不是有人故意搞破坏,都是「这一次图方便」累积出来的。我见得最多的三种: 跨层调用最常见。Controller 本来只该做路由和参数校验,但有人嫌 Service 套一层麻烦,直接把 JdbcTemplate 拉进来查库。一次两次没事,等这个模式扩散开,分层就成了摆设。 模块缠绕更要命。我们项目有十几个业务模块——tenant、trade、circle、marketing…本来该各管各的。但今天 tenant 模块需要算一笔佣金,顺手 import 了 trade 的 Repository;明天 trade 又用了 tenant 的工具类。半年后想拆微服务,发现拆不动,因为没人记得清谁依赖谁。 循环依赖是缠绕的极端形态。A 依赖 B,B 又反过来依赖 A,编译能过(运行时可能爆),但删一个就崩一串。这种债最难还。 循环依赖还有个特别恶心的变体:间接循环。A 不直接依赖 B,而是 A 依赖 C,C 依赖 B,B 依赖 A。编译能过,但依赖图上 A→C→B→A 形成了环。这种循环肉眼很难发现(你要同时看三个文件才能拼出这个环),但 dependency-cruiser 的 no-circular 规则一扫就出来。这也是为什么架构守护要靠工具不能靠人——人能发现「A 直接依赖 B」的显式循环,但「A→C→B→A」的间接循环,人眼根本拼不出来,工具能。 这三种腐化还有一个共同特点:它们都是「温水煮青蛙」。单看每一次违规,都很小——「就这一次图方便,下次不这样了」。但「下次」永远不会来,下一次还是图方便。等你发现架构烂了,已经积了几百处违规,还债的工作量大到没人敢动。架构守护的价值,就是在「第一次图方便」时就拦住,不让温水开始煮。防微杜渐说起来老套,但在架构腐化这件事上,它是唯一有效的策略。 二、先解决一个现实问题:存量怎么办 引入架构测试,第一个拦路虎是存量。 我们项目引入 ArchUnit 的时候,代码已经写了两个月了。一跑,Controller 直连数据层的违规 467 处,业务层用 SQL 注解的 84 处。要是让测试直接 fail,CI 当场全红,谁也推不动。
