Agent / Tool Calling 桥接设计:把工具定义和 AI 框架解耦

xiaoshuai1024 included in AI编程
这是「前端 → 全栈 → AI 开发」线上的 AI 开发阶段。AI 助手要"能干活"而不只是"能聊天",靠的就是 Tool Calling(工具调用)。这篇讲我怎么把工具定义和 Spring AI 解耦:业务同学写起来零框架负担,换框架一行不动,写操作有确认门禁,每次调用全审计。 一、要解决的核心问题 我们的多租户 AI 助手要服务运营同学。运营同学说的不是"帮我查订单"这种结构化指令,而是自然语言:“今天的订单怎么样"“把这几个待审的代下单批一下"“这几个订单统一备注成已沟通”。 对模型来说,这些话背后是不同的业务动作。难的不是对话,是怎么把自然语言安全地落到真实业务方法上。这中间有几个工程问题必须先回答: 谁决定调哪个工具? 不能写 if 用户问订单就查订单 的硬编码分支。自然语言的组合空间是无限的,if-else 永远写不完。 工具怎么定义? 业务同学写一个工具,要不要被 AI 框架的 API 绑架?今天用 Spring AI,明天换 LangChain4j,要不要把所有工具重写一遍? 多租户怎么穿透? A 租户的运营说话,工具只能查 A 租户的订单,这个隔离边界怎么从对话层一路传到业务 service? 写操作怎么兜底? 模型要是手一抖调错写操作(比如批量改错订单),怎么拦?Agent 直接改生产数据,谁来背锅? 这篇就是回答这四个问题。核心思路一句话:调度交给模型,工具实现交给业务,中间架一层自研桥接,把两边彻底解耦。 二、整体设计 先看全貌。一次"运营问一句话、模型决定调工具、框架自动执行、结果回喂模型续生成"的完整链路: 几个关键设计决策,后面逐个展开: 不自研 Agent 框架,只桥接工具。工具调用循环交给 Spring AI,业务只实现工具本身。 工具定义和 Spring AI 解耦。自研 @AiTool + ToolHandler 一层,业务代码看不到任何框架类型。 租户上下文用 toolContext 透传,多租户隔离从对话层穿透到工具层。 写操作走确认门禁。Redis token + 参数哈希 + 消费即删,防重放防误触。 一次工具调用的时序 文字描述不如一张时序图清楚。运营问"今天订单怎么样”、模型决定调 query_orders、框架自动执行、结果回喂续生成的完整链路:

多租户 AI 配置:动态 ChatModel 工厂怎么设计

xiaoshuai1024 included in AI编程
这是「前端 → 全栈 → AI 开发」线上的 AI 开发阶段。多租户 AI 助手里最难的一块,不是 RAG、不是 Agent,是让每个租户用自己的大模型。这篇专门讲我怎么用"动态 ChatModel 工厂"把这件事做扎实——从表设计、加密脱敏,到工厂的缓存与失效,再到为什么不用 Spring AI 的 starter 自动装配。 一、问题:为什么不能写死一个全局 ChatModel 随便找一个 Spring AI 的入门 demo,大概率是这样:在 application.yml 里配一个 spring.ai.openai.api-key,启动时由 starter 自动装配出一个全局 ChatModel bean,业务里 @Autowired 直接用。单租户、单 Key、单模型,这样没问题。 但我们的平台是多租户的。同一个 AI 助手,服务几十个租户,每个租户: 用自己的供应商(A 用 DeepSeek、B 用通义千问、C 用智谱 GLM); 用自己的 API Key(各自的账单、各自的额度、各自的限流); 用自己的模型参数(温度、max_tokens、具体模型型号都不一样); 在运营后台随时改这些配置,改完要立刻生效。 一个全局 ChatModel bean 在第一点上就垮了——它的 apiKey 在启动时定死,运行时拿不到租户上下文。这就是多租户 AI 和单租户 AI 最本质的区别:模型实例必须按租户动态创建、缓存、失效。 方案 能否多租户 配置生效 复杂度 全局单 ChatModel bean(starter 装配) 不能(只有一个 Key) 改配置要重启 低 每次请求都 new 一个模型 能 实时 极高(每次建连,性能不可接受) 动态 ChatModel 工厂 + 缓存 + 失效(本文) 能 配置变更 evict 后实时生效 中 结论很清楚:需要在两者之间取一个平衡点——按租户懒加载,建好后缓存复用,配置变了再失效重建。这就是下面整个设计的起点。

DDD 领域模型在低代码后端服务中的应用

xiaoshuai1024 included in 架构  领域驱动设计  低代码
这篇文章属于我成长线上的**全栈阶段(后端建模)**节点:在前端架构之外,把后端服务的领域边界、契约、异常语义做扎实,是从前端走向全栈架构师的关键一跃。 低代码平台的后端不仅要支撑「站点、页面、用户、系统设置」等资源的 CRUD,还要在 API 契约统一、多端对接(管理后台 BFF、对外站点)的前提下,保持领域边界清晰、业务规则可演进。 本文从架构师视角,以 luban-backend 的实现为蓝本,说明 DDD 领域模型设计在低代码后端服务中的应用:战略层面的限界上下文划分,战术层面的实体、应用服务、领域异常及与 BFF 的协作方式。 项目归属说明：本文 DDD 建模经验源自大厂时期主导的内部系统"鲁班"，代码实现以个人开源项目 OpenLuban（架构同源、代码全新重写）为准。 一、为什么低代码后端需要领域模型 低代码后端的核心职责可以概括为四块: 内容模型:站点(Site)与页面(Page + PageSchema)的创建、发布、按路径访问。 身份与权限:用户(User)与认证(登录、/auth/me)、基于角色的访问控制(RequireUser / RequireAdmin)。 系统配置:系统设置(SystemSettings)的读写与缓存(如 Redis)。 对外能力:按站点 slug + 路径对外暴露已发布页面,无需鉴权(供官网等消费)。 若仅按「表结构 + Controller + Service」平铺,很容易出现三类问题: 业务规则散落在各处,修改一处忘记另一处。 错误语义不统一,前端/BFF 难以稳定处理。 扩展新资源或新上下文时边界模糊,影响可维护性。 引入 DDD 的领域模型思维(不必强求完整 DDD 框架),可以在保持实现轻量的前提下,获得清晰的边界、一致的业务语义和可演进的 API 契约。下面从战略与战术两方面结合 luban-backend 说明。 二、战略设计:限界上下文与能力边界 从业务能力出发,可将低代码后端划分为以下限界上下文(Bounded Context),与当前模块/包结构对应关系如下。 限界上下文 职责 主要实体与概念 对外暴露 站点与页面(Content) 站点与页面的生命周期、同站点下 path 唯一、schema 存储 Site(聚合根)、Page(实体,属 Site) 管理端 CRUD;公开接口按 slug+path 查已发布页 用户与认证(Identity & Auth) 用户管理、登录校验、当前用户解析 User(聚合根)、角色/状态 登录、/auth/me;Header 注入 X-User-ID / X-User-Role 系统设置(Settings) 全局配置的读写与缓存 SystemSettings(单例式聚合) GET/PUT /settings,仅 Admin 公开访问(Public) 按站点 slug + path 返回已发布页面,只读、无鉴权 复用 Site、Page,只读、过滤 status=published GET /public/sites/:slug/pages?

基于nuxtjs的SSR服务性能 + 稳定性优化 这一篇写在我从前端转向全栈架构师的阶段。当时承接的是一个高并发的 SSR 服务:既要做性能,也要做稳定性。它代表的是全栈阶段的高并发实战能力。 基于 Nuxt.js(本文以 Nuxt3 为主)的 SSR(服务端渲染) 服务,有两个核心定位: 它是解决 SPA 应用首屏加载慢、SEO 优化不足的核心方案。 它也是承接高并发流量、保障用户体验的关键载体。 本文结合生产环境实战,围绕 SSR 服务在流量快速增长场景下的性能瓶颈与稳定性隐患,给出可落地的优化方案。 核心目标只有两个:QPS 量级跃升,以及**“永不宕机”**。 一、服务架构基础：Nuxt.js SSR实时渲染服务架构 本次优化的核心载体是基于 Nuxt3 的 SSR 服务。整体架构以 “Node.js 实时渲染” 为核心。 它承接前端请求、完成服务端渲染与数据聚合,并联动 CDN、Nginx、BFF 层、后端服务,形成完整的请求链路。 核心架构链路如下: 用户请求 → CDN(静态内容缓存) → Nginx(负载均衡、探活、请求转发) → Nuxt.js SSR 服务(实时渲染、数据请求、兜底处理) → BFF 层(流量承接、数据聚合、接口缓存) → 后端服务(核心业务逻辑、数据持久化)。 核心职责上,Nuxt.js SSR 服务负责页面模板渲染、前端资源打包、服务端数据请求与整合,同时承接前端错误处理与兜底逻辑。它是连接用户与后端服务的关键枢纽。 二、核心问题：流量增长与SEO优化的双重挑战 业务规模扩大后,Nuxt.js SSR 服务面临两大核心痛点,严重影响用户体验与业务增长,是优化的出发点: 2.1 流量快速增长带来的性能与稳定性压力 业务推广与用户规模扩大,带来请求量激增。原有 SSR 服务存在明显瓶颈: 单实例 QPS 仅 200 左右,高峰时段出现请求排队、响应超时。 部分后端服务异常时,SSR 服务无缓存兜底,直接返回错误页面。 实例故障时无法快速切换,导致服务不可用,影响用户留存。 2.
