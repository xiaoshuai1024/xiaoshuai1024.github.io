500+ 功能模块 + 400 个迁移:多租户 SaaS 的 Flyway 治理实战

xiaoshuai1024 included in AI编程
上一篇讲分润,这篇讲它的地基——数据库迁移。 研发账本里提过这个项目有 500+ 功能模块(633 张业务表)。但表不是一次建好的,它们分散在近 400 个迁移脚本里,4 个人并行写,跨 2 个月。没有一套治理,这种规模必崩:版本号撞车、生产漏执行、多人改同一张表冲突、历史表脏数据。 这篇拆我们怎么用 Flyway 把它管住,全是真金白银的教训,包括一次生产 gap 事故的止血。 为什么选 Flyway 不选 Liquibase。两个都成熟,选 Flyway 是因为它的迁移用纯 SQL 写,而 Liquibase 用 XML/YAML/JSON 描述变更。对这个项目,纯 SQL 有两个优势:一是 AI 写迁移更顺——AI 对 SQL 语法的掌握远好过 Liquibase 的 DSL,生成质量更高;二是迁移文件可直接复制到数据库客户端手动执行,排查止血时不用再翻译。Liquibase 的抽象层在「跨数据库」场景有价值,但我们只用 MySQL,这层抽象是纯负担。工具选型要看场景,不是越强大越好。 还有一个背景:为什么迁移治理在这个项目这么吃重。因为这个项目是「4 人 + AI、2 个月冲百万行」的节奏,schema 变更密度极高——平均每天十几个迁移,4 个人加多个 AI worktree 并行写。这种密度下,如果没有自动化校验,撞号、漏执行、冲突是必然的,不是偶然。治理不是锦上添花,是能不能跑起来的前提。 一、规模先摆出来 db/migration/ 目录下,迁移脚本的真实分布: 维度 数据 迁移脚本总数 近 400 个(squash 之后) baseline 文件 1 个,4546 行,集中了 207 张表的完整 schema 其余增量迁移 约 190 个,含 86 个建表、98 个改表、36 个 seed 涉及 tenant_id 的迁移 93 个(多租户是核心隔离维度) 4 个人 + AI,2 个月产出这个量。这种密度光靠「大家注意点」是不可能的,必须靠机制。

微信分润从 0 到 1:CPS 分账、提现六态机与一次冲正降级

xiaoshuai1024 included in AI编程
上一篇把总账立住了。这篇挑它最硬的一块——微信分润——拆给你看。 分润为什么难。一笔钱要拆成几份、分几次、走不同的通道、还得多租户隔离,搞不好就是资金事故。这是整个项目里我盯得最紧、门禁立得最硬的部分。 一、先看这笔钱怎么流 多租户电商里,一笔订单的钱不是直接进商家口袋。微信支付支持「分账」——付款时先冻结到特约商户,确认收货后再按比例拆给平台、达人、商家。 有人会问,为什么这么绕,用户付钱直接到账不好吗。因为这是多角色分润的电商,一笔订单涉及平台抽佣、达人带货佣金、商家货款、配送费,各方的钱到账时点不一样(平台要即时确认收入、达人要审核后才结算、商家要等完结分账解冻)。如果用户付款直接进商家口袋,平台和达人的钱就得商家手动转,这在多租户 SaaS 里根本跑不通。微信分账这套机制就是为了多角色资金拆分设计的,绕是绕,但它把「钱怎么拆」变成了平台可控的状态流转。 钱大致这样流动:用户支付时标记 profit_sharing=true,资金冻结到店铺特约商户池;确认收货后触发分账,平台佣金即时分润,达人佣金延迟分账(留在达人冻结池),商家货款暂时冻结;达人提现审核通过后,才真正把达人那一份分账到零钱;最后完结分账,解冻商家货款可提。 关键决策是达人佣金为什么延迟。确认收货那一刻,达人可能还没签约分账接收方、可能退款、可能违规。所以钱先标 UN_SPLIT 留在冻结池,等提现审核通过才真正分账。这就是后面提现状态机的来由。 补充一点,这套系统里「提现」其实有三种人,出口不一样。 达人(TALENT)走微信分账:提现审核通过后,系统把这个人关联的 cps_order_attribution 里所有 UN_SPLIT 的佣金逐单调微信分账 API,分到他的 PERSONAL_OPENID。这是最绕的一种,因为它不是「转账」而是「分账」,要回挂到原始微信支付单。 配送点团长(DELIVERY_POINT_LEADER)和商家(MERCHANT)走出款渠道:不挂原始支付单,直接调微信商家转账到零钱或银行卡代付。团长和平台的资金关系相对干净,商家则是分账完结解冻货款后才有钱可提。 三种人看起来都是「提现」,但余额怎么算、钱从哪出、走哪条通道,完全不同。后面会展开。 再补一个容易忽略的环节:确认收货触发分账,不是「一次性把钱拆完」。它是一个编排过程(processTalentSplitForWithdrawal):先把这个达人关联的所有 UN_SPLIT 佣金行捞出来,逐单标 SPLITTING、逐单调微信分账 API、成功的标 SPLIT、最后还要尝试完结该订单的商家货款解冻。 这个逐单处理是有讲究的。一个达人一次提现可能关联几十笔订单的佣金,如果打包成一个请求,中间任意一笔失败整批回滚,前面的也白调了。逐单调的好处是:失败的那笔回退到 UN_SPLIT(代码里 catch 块专门做了回滚),成功的笔保留 SPLIT,下次重试只跑失败那几笔,不重复打款。代价是慢,但资金系统宁可慢也不能错。 还有个前置动作:分账前要先「确保达人签约了分账接收方」(ensureTalentReceiverAdded)。微信分账要求收款方是已绑定的接收方,没签约就分账会被拒。所以每一单分账前先检查、没签约先发起签约,再调分账。这一步漏了,提现就会整批失败,日志里全是「接收方不存在」。 二、分账比例:基点(bp)而不是百分比 钱的事,小数点算错就是事故。我们全程用基点(万分之一),绝不用浮点百分比。 TenantSplitConfigService 里分账比例是这样定义的: public record SplitConfig( String tenantId, @Min(0) @Max(5000) int platformCommissionRateBp, // 平台抽成 @Min(0) @Max(5000) int talentCommissionRateBp, // 达人佣金 @Min(0) @Max(5000) int otherSplitRateBp, // 其他分账 @Min(0) @Max(5000) int totalSplitLimitBp, // 总分账上限 boolean enabled, ...) { public int totalRateBp() { return platformCommissionRateBp + talentCommissionRateBp + otherSplitRateBp; } }几个硬约束。

AI 把"知识储备"推平后,老兵反而成了最大红利

xiaoshuai1024 included in AI编程
当 AI 变得全知全能，「我会你不会」这道优势就塌了。那研发到底还靠什么吃饭?这是序章留下的一个问题,转型篇讲我往哪转,这篇要讲一个更底层的判断。 一、知识,曾经是研发最硬的优势 干过几年研发的人都清楚,以前这行最值钱的就是知识储备。 一段冷门配置怎么写、某个框架的坑在哪、一个诡异报错怎么解——这些没有捷径,全是时间和血泪堆出来的。趟过才记得住,趟过才有发言权。 我见过很多资深工程师,核心竞争力就是「我踩过的坑比你走过的路还多」。初级查一个 Nginx rewrite 要一下午,资深张口就来。信息差直接换算成工资差。 学习周期也长。一门新语言从入门到能写生产代码,少说半年;一个新框架,把生态摸熟又要几个月。知识壁垒高,掌握它的人才值钱。这套逻辑过去十年都成立。 这套逻辑还有个隐性产物:「知识壁垒」制造了职业安全感。你会的别人不会,公司就得用你;你踩过的坑别人没踩,出事就得靠你。这种「不可替代性」是过去工程师最大的心理保障——「我会的东西不是随便能找人替代的」。所以大家拼命学、拼命囤知识,本质上是在囤「不可替代性」。AI 把这道墙推平,直接动摇了这个心理保障——如果 AI 也会,你就不再是「不可替代」的了。这也是为什么很多老兵对 AI 有本能的抗拒:不是 AI 不好用,是 AI 动摇了他们安全感的根基。 但安全感从「我会别人不会」挪到「我能判断别人(含 AI)做对没有」,其实更稳。前者的不可替代性建立在「信息差」上,信息差会被 AI 抹平;后者的不可替代性建立在「判断力」上,判断力不会被抹平。所以与其抗拒 AI,不如主动把安全感的基础从「信息差」挪到「判断力」——前者注定贬值,后者长期保值。 二、AI 把这道墙推平了 现在不成立了。 我的日常主力是 Claude Code,配 Cursor 做编辑,OpenCode 做补充。一个最直观的感受:以前要翻一天文档的事,现在一句话。 不熟的语法,问一句直接给; 冷门框架的用法,它比翻官方文档还快; 诡异报错,贴过去它不光解释原因,还顺手把改法写了。 当人人都能「问出答案」,「我会你不会」这道优势就塌了。 这不是说知识没用,而是「知识的储备量」不再是稀缺品——它从你脑子里的库存,变成了工具里的自来水,谁都能拧开。 这里要补一个重要的区分:「知识」和「理解」不是一回事。AI 把知识的获取推平了(语法、API、用法随取随有),但「理解」没被推平——「这个设计为什么这么选」「这个方案在什么场景下会崩」,这些需要理解原理和上下文,AI 给的是「通用答案」,你结合自己的场景理解后,才能变成「对的答案」。知识是素材,理解是把素材变成方案的能力。AI 提供了无限的素材,但「把素材变成对的方案」这步,还是人来做。所以 AI 推平的是「知识储备」,不是「理解能力」——后者反而因为素材充足,变得更值钱了:以前素材少,理解能力再强也巧妇难为无米之炊;现在素材无限,理解能力强的人能把无限素材组合出最优方案。 三、值钱的东西,往上挪了一层 墙塌了,但人还得吃饭。价值没消失,只是挪了地方。 执行层的知识被削平之后,真正值钱的能力往上挪了一层。具体是三样。 架构能力。 系统怎么分层、边界划在哪、哪一步该做、哪一步该砍。这种取舍 AI 给不了答案,因为它取决于你的业务、团队和约束。这部分我专门拆过。 判断力。 一个需求该不该做、做到什么程度、哪条路是死的。这是决策的活,AI 只能给选项,拍板得你来。 经验嗅觉。 一眼看出这段代码迟早出事、这个方案上线必挂。这种东西 AI 学不会,它来自亲身趟过的坑,不在任何文档里。 这三样有个共同点:它们都建立在大量知识之上,但本身不是知识。知识是地基,AI 帮你把地基铺平了;可上面那栋楼怎么盖,还得你自己设计。 这三样还有个共性值得点出:它们都是「慢能力」。知识是快能力——学一个 API,几分钟;记一个配置,一次就记住。但架构能力、判断力、经验嗅觉,都是慢能力——它们靠真实项目喂出来,没有捷径,不能靠「问 AI」速成。AI 时代,快能力被推平了(谁都能问),慢能力反而成了真正的壁垒——因为慢能力 AI 也帮不了你,它只能靠你在真实场景里一趟趟趟出来。这和「老兵最受益」是同一个逻辑的另一个面:老兵的慢能力攒得久,AI 推不平,所以反而最保值。 慢能力的积累还有个特点:必须亲身经历,不能二手获得。你可以问 AI「分布式锁有哪些坑」,AI 给你列十几个,但「哪个坑在你的业务里最可能踩到」「踩了之后是什么表现、怎么救」,这些只有亲身趟过才知道。老兵的优势不在「知道有这些坑」(AI 也能列),在「知道这些坑里哪个最可能出现在当前场景」——这种场景化的判断,只能靠亲身经历,不能靠二手信息。所以老兵的经验不是「知道坑的清单」,是「知道哪个坑在什么时候最危险」,后者比前者值钱得多。

把"AI 保障"焊进系统骨架:让 AI 想跑偏都难

xiaoshuai1024 included in AI编程
我们项目用 AI 协同开发两个月,产出近百万行代码,同时也产出了几百个被保障层拦住的隐患——跨租户查询、漏带权限校验、静默删行为代码、迁移版本撞车。这些隐患每一个溜进生产都是事故。保障层拦住的不是「AI 偶尔抽风」,是「AI 日常产出里必然包含的系统性风险」。 AI 让代码产得快,也让坏代码产得快。一个能稳定输出代码的系统,如果骨架本身兜不住,产能越高、负债累积越快。所以我现在反而比以前更花时间在那些"无聊"的基础设施上——多租户隔离、权限过滤器、契约扫描、迁移治理、可观测。不是因为它们新,因为它们旧、因为它们"枯燥",才成了 AI 时代最值钱的东西。架构师的活儿从"设计能跑的系统",变成了"设计能兜住 AI 的系统"。 先说我们这个项目的底座 我们做的是一个多租户 SaaS,典型的三段式结构:C 端小程序 + 运营后台 + Java 后端。规模不算夸张,但因为是多租户,出了事影响面大,所以对隔离和契约特别较真。 架构底座就这么几样,说出来都是老生常谈: 多租户行级隔离,每张业务表都带 tenant_id,查询自动带过滤; RBAC 权限过滤器链,权限校验不靠人记得加,而是在过滤器里强制走; 契约驱动,前端调的 URL、后端的路由注解、SQL 引用的表、Flyway 的建表 DDL,两两对得上才算数; Flyway 迁移治理,版本号唯一、有 seed 数据、不许手改线上表结构; 可观测,Prometheus / Grafana / ELK 一套,出问题能追溯。 这里有个值得展开的设计取舍:多租户隔离我们没有用 MyBatis-Plus 的 TenantLineInnerInterceptor(自动给每条 SQL 拼 tenant_id)。不是不知道有这个,是有意不用。原因是它太「隐式」了——自动拼 WHERE 条件,开发者看不到,一旦某个复杂查询需要跨租户(比如平台超管看全量),要写特殊注解绕过,绕过逻辑又成了新的隐患。 我们选了「显式带」:每条业务 SQL 手动带 tenant_id 条件,配合 ArchUnit 规则强制检查 Repository 层 SQL 是否都带了租户过滤。显式的好处是每条 SQL 的租户边界都看得见,review 时一眼能确认;坏处是依赖人记得带——这就是为什么要配 ArchUnit 规则兜底,人忘了机器抓。 TenantFilter 的实现也值得展开。它是个 OncePerRequestFilter,从 X-Tenant-Id 请求头取租户,塞进 ThreadLocal(TenantContext),请求结束 finally 里 clear。关键是它按 path 前缀白名单:不是所有接口都要租户(比如登录、健康检查、静态资源),白名单内的直接放行。AI 加新接口时,如果忘了把需要租户的接口放进白名单外,过滤器会拦——这又是一道隐式保障。
