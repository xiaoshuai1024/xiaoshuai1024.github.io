全栈架构 × AI 研发实战

【AI 研发实战】为什么端到端测试在 AI 编码时代被严重低估

xiaoshuai1024 included in AI编程
这篇是我「AI 开发」阶段质量体系里的一块,和之前几篇是一套: 那篇 三层测试纪律讲的是「单元测试、集成测试、端到端测试三层都得在、谁也别偷懒」; 覆盖率诚实讲的是「别拿覆盖率当 KPI」; 编译能过 ≠ 代码正确讲的是「门禁得跑到 verify」。 这篇专门把端到端测试拎出来当主角。因为我的判断是:它一直是金字塔尖上最不受待见的那层——慢、贵、脆,人人往后排。但在 AI 大量写代码的今天,它恰恰是杠杆最大、最该加码的那一层。 我的结论先放这:单元测试告诉你「零件没坏」,端到端测试告诉你「整车能不能开」。AI 让零件造得太快、装得太猛,这道整车下线检测被严重低估了。 先说为什么 AI 时代的测试压力是暴涨的 两件事同时发生了,叠在一起才可怕。 第一件,单位时间产出的代码量暴涨。以前一个工程师一天能写多少行有效逻辑,大家心里有数;现在主力工具是 Claude Code,辅以 Cursor、OpenCode、Codex,Agent 一个下午能产出的量,赶上过去一周。产能上去了,可审查的时间没涨,人 review 的带宽是死的。 第二件,团队在转向全栈。以前前后端分得很清,前端出 bug 多半在前端那层,后端出 bug 多半在后端那层,各管一段。现在全栈化了,一个人/一个小团队从页面一直写到数据库,中间跨了好几个域。没有哪一个人,对「从前端点击到落库」这条完整链路有完整心智。 这两件事单拎一个出来,都还能应付。叠加在一起,问题就变了味道:bug 不再平均地分布在「某个函数写错了」这种颗粒度上,而是整体上移,大量集中在「拼起来才出问题」的地方——服务和服务之间、域和域之间、接口承诺和实际行为之间。 而这个「拼起来」的层面,恰恰是单元测试够不着的那一层。 单元测试为什么兜不住 先说清楚,我不反对单元测试。它永远需要——快、便宜、能反复跑,抓纯逻辑和分支边界是它最擅长的事。我反对的是「有了单元测试就觉得稳了」这个错觉。 单元测试兜不住的,是几样东西: 环境差异。单元测试里你 mock 掉了下游服务和数据库,跑得飞快。但真实环境里那个下游服务返回的字段比你 mock 的多了一个,序列化行为不一样,单元测试看不见。 实现被替换。单元测试验的是「我假设它这样,我这样调」。可假设的那个实现,和线上真实的那个实现,常常不是同一个东西。mock 出来的接口永远会按时返回,真实的不会。 系统交互。一个请求要穿过网关、A 服务、消息队列、B 服务,最后写库。单元测试把这段切成一段段各自测,每段都对,拼起来挂了。挂的那个点,落在段与段的缝里,单元测试一个都摸不到。 真实调用链。单元测试不碰真实的网络、真实的中间件、真实的并发。而很多 bug 恰好就活在「真实调用链」里——超时、重试、幂等、串号。 说白了,单元测试验证的是「这个函数对不对」,可线上炸的从来不是「某个函数」,而是「这条链路」。 我自己就吃过这个亏。一个接口,单元测试写得满满当当,全绿。联调那天一跑,挂了——原因不在接口内部逻辑,而在它依赖的那个服务,线上版本和单元测试里 mock 的版本,字段语义对不上。全绿给了我一个「稳了」的幻觉,而这个幻觉差点带到生产。 这件事之后我就想明白了一个朴素的道理:单元测试是零件质检,端到端测试是整车下线。零件全合格,不等于整车能开出门。 端到端测试为什么是唯一能「跨」的 我把端到端测试的价值浓缩成一个字,就是「跨」。它能跨单元测试、集成测试怎么都够不到的边界: 跨系统:一个操作从 C 端入口出发,穿过网关、A 服务、消息队列、B 服务,最后落到数据库。单元测试只能测其中一段,端到端测试测的是这条整链。哪一段断了、哪一段字段对不上,一次跑穿就暴露。 跨域名:订单域、支付域、分润域,各自有各自的领域模型。一个业务流程要横穿好几个域。域和域交界处的字段映射、状态同步,是 bug 的高发地,而这片地只有端到端测试能踩到。 跨业务:一个完整的业务闭环——下单、支付、分润、对账——要跑通才算「这个功能真的能用」。单元测试测不到这种「业务闭环」,它测的是零件;端到端测试测的是业务跑没跑通。 落到数据库验证最终态:这条我最想强调。端到端测试不是「接口返回 200 就算过」,它跑完之后直接去查库,断言数据真的写对了——状态机迁移对不对、金额对不对、tenant_id 隔离对不对。接口返回和真实落库之间,隔着一个「数据到底对不对」的天堑,端到端测试是少数能跨过去的。 这四条,每一条都是单元测试、集成测试的天花板,端到端测试才够得到。

百万行代码全靠 AI 写:我把工程纪律沉成 100 个 Skills

xiaoshuai1024 included in AI编程
2026 年 6 月,线上数据丢了一批。 一个定时清理任务,逻辑是「扫整个对象存储,凡是数据库里没记录的,就当孤儿删掉」。听着合理,对吧?存储里有、库里没有的,可不就是垃圾。直到某天一个写入链路抖了一下——文件先落库、对象后上传,中间这几秒,定时任务扫过去了,「库里有、存储还没到」的对象被当成孤儿删了个干净。 这是我维护的一个多租户社交电商全栈项目。前端 uni-app(Vue 3)小程序、后端 Spring Boot 3 + MyBatis-Plus + MySQL + Redis + RabbitMQ、运营后台 Vue 3 + Element Plus。代码量早就过百万行,而其中绝大部分是 AI 写的。 事故复盘的时候我没怪模型。我怪的是:我给了 AI 一把没有保险的枪。它执行得又快又准,快到我没来得及看它要删什么。 百万行代码之后,我真正想明白一件事:瓶颈不在模型够不够聪明,在我怎么把工程纪律变成 AI 逃不掉的流程。我选的方式,是在项目里沉淀一套 Skills。 这篇就讲这套东西:我实际在用哪些 skill、每个在项目里管什么、以及它们是怎么装进来的。 一、先说清楚:Skill 不是教 AI 新知识 很多人对 Skill 有个误解,觉得它是「给模型补充训练数据里没有的知识」。不是。 模型的知识已经够多了。它知道 Spring Boot 怎么写、知道 Flyway 怎么迁移、知道 RBAC 怎么设计。它缺的不是知识,是约束——是你的项目里那套「必须先写测试」「DELETE 必须带 tenant_id」「声称完成前必须贴证据」的纪律。 Skill 干的就是这个:把「你应该怎么干活」写成一份可复用的指令,在合适的时机自动加载进会话。它不增加模型的能力上限,它抬高的是模型在你项目里的下限。 这是为什么我不去追「更强的模型」,而去攒「更密的 Skill 网」。模型每代都在变强,但你项目里那个「AI 一高兴就 DELETE FROM 不带 where」的风险,换什么模型都在。能焊死这个的只有流程,不是参数量。 二、这套 Skills 是怎么装进来的 先讲安装,因为这是最容易被问到的。我不是手写了几十个 skill——那不现实,也违背「复用」的初衷。项目里的 skill 来自一套类似 npm 的 skill 包管理器。

MCP 接 Figma 实战:把工具接入从 M×N 标准化成 M+N

xiaoshuai1024 included in AI编程
接个订单查询,写个 OrderQueryTool;接个设计平台的 API,又得写一套适配。工具越多,适配代码越堆。直到我接 Figma,踩了一通坑,才搞懂 MCP(Model Context Protocol)到底在解决什么——它不是又一个新协议让你学,是把"每个工具都重写一遍适配"这层标准化掉。 (上一篇我讲的是自研 Tool Calling 桥接,把工具定义和框架解耦;这篇往上走一层。) 这篇就用我把 Figma 接进 AI 工作流的真实过程,讲 MCP 怎么落地。不讲虚的,讲配额怎么省、双 server 怎么配合、怎么装。 一、MCP 到底解决什么:M×N 变 M+N 先说它解决的那个根本问题。 假设你有 3 个 AI 客户端(Claude Code、Cursor、Cline),要接 3 个外部工具(Figma、GitHub、数据库)。传统做法是每个客户端给每个工具写一套适配——3×3=9 套。客户端加一个、工具加一个,适配数就是乘法涨。这就是 M×N 问题。 MCP 做的事是中间插一个标准协议:客户端只跟协议说话,工具也只跟协议说话。于是 3 个客户端 + 3 个工具,只需要 3+3=6 个适配,每边各管各的。涨法从乘法变加法。 落到 Figma 这个例子上:我不需要为「Claude Code 怎么调 Figma」写适配,也不需要为「Cursor 怎么调 Figma」再写一遍。Figma 那边出一个 MCP server,任何支持 MCP 的客户端都能直接用。这就是它最实在的价值——工具方写一次,所有客户端通吃。 M×N 变 M+N 还有个容易被忽略的收益:工具方和客户端可以独立演进。M×N 模式下,Figma API 升级了,所有客户端的 Figma 适配都要跟着改——3 个客户端改 3 遍。M+N 模式下,Figma 升级了,只改 Figma MCP server 这一处,所有客户端自动受益。这个「解耦后各自演进」的红利,在工具多、客户端多的场景下特别值钱——每次 API 升级省下来的适配工作,是持续的复利。

AI 时代研发什么能力最值钱:架构、前瞻与指导 AI 做对

xiaoshuai1024 included in AI编程
上一篇讲知识不再是优势。这篇接着说一个更实际的问题:知识被抹平了,什么才是新的核心竞争力。 AI 时代真正重要的研发能力,不是"会写代码",是"知道怎么正确地做事"。 一、vibe coding 的热闹和它的代价 AI 编程门槛降到地板之后,vibe coding 成了一种流行玩法——不读代码、不懂架构,跟 Claude Code 或 Cursor 聊着天把应用"聊"出来。 我能理解这种爽感。但作为经常要接手、维护这类系统的人,必须说一句实话:这些产物里有一大批是玩具,却被当成工业级软件在用。 vibe coding 的爽感来自"快"——几个小时就能把想法变成能点的界面。但这种快掩盖了一个事实:demo 和生产软件之间,差着四道硬功夫(安全、可维护、可扩展、可验证),而这四道功夫,vibe coding 一道都没练。爽完之后的账,是后面接手维护的人(往往是别人,甚至是几个月后的你自己)在还。 危险不在 demo 能跑,而在于"能跑"这件事制造了一种错觉。能跑的代码,不等于能上线;能上线的,不等于能扛住流量;能扛住流量的,不等于半年后还改得动。这三层差距,vibe coding 一个都没跨。 具体表现在四件事上,几乎每篇 vibe coding 产物都中招: 安全:AI 写的鉴权、SQL 拼接、密钥处理多半有洞。一个 tenant_id 漏校验,就是跨租户数据泄露。 可维护:没规范、没分层、命名混乱。三个月后作者自己都读不懂。 可扩展:逻辑全堆在一个文件,一次请求同步干十件事。用户量一上来就瘫。 可验证:没测试、没契约、没监控。改一行,不知道弄坏了哪里。 这四件事,每一件都是"能跑"掩盖不了的。vibe coding 产物的典型路径是:跟 AI 聊出一个能跑的 demo → 看着像那么回事 → 直接上线 → 三个月后出事,要么被攻击(SQL 注入、越权)、要么扛不住(没缓存没限流)、要么改不动(屎山)、要么查不清(没监控没日志)。“能跑"是 demo 的标准,不是生产的标准,这两者之间的鸿沟,vibe coding 一个都没填。 我自己接手过这类系统,感受很直接:打开代码,逻辑全挤在几个大文件里,没有分层、没有契约;安全上,鉴权校验写在前端(后端裸奔)、SQL 用字符串拼接;可维护性上,改一个字段要全文搜索十几个地方。这种系统能跑,但它是隐患——不是会不会炸的问题,是什么时候炸。而把它从"能跑"改到"能上线”,花的力气比重写还大。 玩具和工业级软件的分界线,从来不是"能不能跑",而是出了事扛不扛得住、查不查得清、改不改得动。 二、AI 解决执行,解决不了"做对" 先表明立场:我不是反对 AI。相反,Claude Code 是我的主力工具,研发账本里 4 个人 2 个月干出近百万行,没有 AI 根本做不到。我比多数人都更依赖它,也更清楚它的边界。

把"用 AI"做成工程体系:28 个命令和一条自治评审回路

xiaoshuai1024 included in AI编程
前三篇讲的都是「用 AI 做出了什么」。这篇回头讲「怎么用 AI」——不是技巧,是体系。 很多人对「AI 协同研发」的理解,还停在「每个人装个 Cursor,各写各的」。这种用法,4 个人撑死比以前快一倍。我们能做到 2 个月近百万行,是因为把 AI 协同沉淀成了一套可复用的工程体系:命令化的工作流、自治的评审回路、规范化的技能库。 关键不是 AI 多强,是这套体系让每个 Agent 开机即用。 这篇把这套体系拆开讲三块:命令化工作流(28 个命令怎么设计、为什么这么设计)、自治评审回路(它是体系的心脏,怎么兜住百万行的质量)、规范固化(怎么让每个 Agent 一进来就知道规矩)。最后聊聊这套体系的本质——它解决的核心问题是「怎么让 4 个人的经验,放大到指挥一群 Agent 还不失控」。这套体系不是一天建成的,是项目跑下来一边做一边沉淀的,这里讲的是沉淀后的终态。 一、不是「每个人会用 AI」就够 先说个反直觉的判断:「团队里每个人都会用 AI 工具」不等于「团队有 AI 协同能力」。 差别看两个场景。 没体系的用法:每个人各自调 AI,提示词五花八门,生成的代码风格不一,质量靠个人经验兜底,新人和 AI 都不知道项目规范,反复返工。 有体系的用法:PR 有标准命令、评审有自治回路、规范固化在仓库里、E2E 一条命令跑通,新人第一天装上工具就能产出符合规范的代码。 我们追求的是后者。这套体系跑起来,才是产能能持续放大的根,它让「4 个人 + AI」不只是 4 个独立单元,而是一支能协同的队伍。 这两者的差距,在项目规模小的时候看不出来——3 个文件的小项目,有体系没体系都能做。但项目一大(几百个文件、几十个模块),没体系的用法就会崩:代码风格分裂、规范执行不了、新 Agent 反复返工、质量全靠人肉兜底兜不住。体系的价值是随规模放大的:项目越大,体系的杠杆越大。所以我们不是「一开始就建体系」,是「项目大到一定程度,不建体系就跑不动了」才逼出来的。 还有一点要诚实:建体系是有成本的。28 个命令、评审回路、规范固化,这些东西不是免费的,前期投入的时间不少。但这个投入是一次性的——建完之后,第二个、第三个项目都能复用(命令和规范的思路是通用的,只是具体内容要按项目调)。所以体系的投入是「前重后轻」,熬过前期的沉淀期,后面就能享受复利。怕的是团队等不到复利就放弃,觉得「建体系太慢不如直接写代码」,结果永远停留在「每个人各写各的」的低效状态。 二、命令化工作流:28 个命令覆盖全流程 我们把高频操作固化成了 28 个自建命令(/command)。流程上大致是:/prd 出需求 → 编码 → /tdd 测试驱动 → /mp-e2e、/op-e2e 跑端到端 → /kd-review 自治评审 → /pr-fullstack 提 MR → /deploy-prod-be-ops 部署。

15 个大屏不上 ClickHouse:多租户 SaaS 统计中台的务实选型

xiaoshuai1024 included in AI编程
上一篇讲数据库地基,这篇讲建在地基上的统计中台。 这个项目有十几个数据大屏:平台总览、交易分析、商家 KPI、连锁统计、运营统计、服务商看板……每个大屏几十个指标,全部要多租户隔离。听起来像该上 ClickHouse、上 Flink、上数仓那一套。 我们没有。用的是 MySQL 汇总表 + Redis 实时计数 + ES 搜索,三层数据源拼装。这篇讲清楚为什么这么选,以及落地时的几个关键判断。 先说清楚统计中台在这个项目里解决什么问题。多租户 SaaS 平台,每个租户(商家)要看自己的经营数据(GMV、订单、用户),平台运营要看全局数据(全平台 GMV、活跃租户、增长趋势),服务商要看自己服务的租户数据。这三类视角,加上时间维度(实时、今日、本周、本月),组合出几十个统计场景。如果每个场景都实时去查原始业务表,生产数据库会扛不住(一个大屏查询可能 GROUP BY 几十万行)。统计中台的存在,就是把这些查询压力从生产表转移到预聚合的汇总表,让「看数据」不打扰「做交易」。 统计中台的设计核心是「分层 + 隔离」。分层,是指数据按新鲜度分三层(实时/小时/T+1),各层用不同技术(Redis/汇总表/定时任务);隔离,是指多租户数据严格按 tenant_id 隔离,统计查询绝不扫生产表。这两个设计原则,贯穿整个统计中台的实现。理解了这两点,后面的技术选型和落地细节就好懂了。 一、为什么没上 ClickHouse 很多人一听「统计大屏」,第一反应是上 OLAP。我们也评估过,最后没上,理由很实在。 数据量没到那个量级。这是多租户 SaaS,不是全民级应用,日订单几十万量级,MySQL 加索引加汇总表完全扛得住。上 ClickHouse 是杀鸡用牛刀,运维成本反而上去了。 团队只有 4 个人。多引一个重型组件,就多一份运维负担。SaaS 创业期,运维复杂度是隐形成本。 实时性要求没那么极致。大屏数字延迟 15 分钟完全可以接受,它本来就是决策辅助,不是交易撮合。 不追求技术先进,追求「刚好够用 + 可控」。这个判断贯穿整个统计中台的设计。 这里要展开讲一下「不上 ClickHouse」的完整算账。ClickHouse 的优势是 OLAP 查询快(亿级数据秒级聚合),代价是运维复杂(分布式集群、数据均衡、版本升级)和学习曲线(它有自己一套 SQL 方言)。对一个日订单几十万的 SaaS,MySQL 加汇总表查询响应在百毫秒级,完全满足大屏需求;上 ClickHouse 查询能快到十毫秒,但用户感知不到这 90 毫秒的差距(大屏刷新本来就有延迟)。而运维成本是实打实的——多一个分布式组件,就多一套监控、备份、故障排查的负担,4 个人的团队根本扛不住。所以「不上 ClickHouse」不是技术落后,是 ROI 算账后的理性选择。 还有一个隐性成本:数据同步。ClickHouse 要从 MySQL 同步数据过去(用 Canal 或 Flink CDC),这套同步链路本身就是一个故障源——同步延迟、同步丢失、schema 变更不同步,每个都可能让大屏数据不准。而 MySQL 汇总表直接在同一个库里,没有跨系统同步问题,数据一致性天然有保障。少一个同步链路,就少一类「数据对不上」的事故。所以不上 ClickHouse 也少了一类故障面。
