【AI 研发实战】UT/IT/E2E 三层测试纪律:fail-fast、禁跳过、零占位

xiaoshuai1024 included in AI编程
在「前端 → 全栈 → AI 开发」这条线上,这篇落在 AI 协同研发阶段——讲的是我用 Agent 干活时,测试纪律怎么落地。它和契约门禁、多 Agent 评审是一套:把模糊的「测试要写好」变成 Agent 也能照着执行的硬规则。 测试金字塔谁画都顺手:底下 UT、中间 IT、顶上 E2E,贴在 PPT 上特别好看。可真到了项目里,这套三角形十有八九会缩水。 UT 漏边界; IT 跑得慢干脆禁用; E2E 全绿但一上线就翻车。 我以前也画过那种金字塔,画完就贴墙上,没几个人看。 我的做法是:金字塔不画在墙上,写进纪律里。而且这条纪律不仅人要守,Agent 也要守。下面是我们怎么把「三层测试」从口号落成可执行规矩的。 三层,各兜一摊,谁也替不了谁 先把分工说清楚,省得有人想「用 UT 多写点、E2E 少写点」来偷懒: UT(*Test)兜逻辑:纯逻辑、边界、异常分支,快、多、便宜。一个工具函数的入参组合,几毫秒一轮。 IT(*IT)兜集成:到真实数据库、真实中间件这一层,验「拼起来对不对」。慢一些,但能抓 UT 抓不到的连接、序列化、SQL 语义问题。 E2E 兜真实路径:从入口点进,模拟用户真实操作,验「整条链路通不通」。 这三层不能互相替代。UT 全绿不代表集成没问题——你 mock 掉的那个表结构可能压根对不上。E2E 全绿也不代表每个分支都对——你只点了「正常路径」。 所以关键判断是:不是「三层选一层」,是三层都得在,各管一段。 纪律一:E2E fail-fast,第一个失败就停 E2E 最贵的不是写,是跑。一个 E2E 套件几十分钟不稀奇。 如果第一个用例就挂了,后面几十个还在吭哧吭哧跑,跑完一堆红——纯浪费时间,还污染日志。 我们直接定死 fail-fast:maxFailures: 1。第一个挂,立刻停。理由很直接:E2E 是端到端的,前面的失败大概率会传染后面(比如登录挂了,后面所有用例都白跑)。 停在第一个,省下来的时间够你去看一眼到底哪断了,而不是等半小时后看一屏红。 这条对 Agent 尤其重要。Agent 不会自己判断「这个失败要不要继续」,所以必须用配置把它钉死,不让它有发挥空间。 纪律二:禁止跳过,禁止占位 这条是我跟团队(跟 Agent)吵过最多的一条。 什么算「跳过」?@Ignore、skip、it.skip、条件性跳过(if 环境变量就 skip)——统统不许,明令禁止。连 Agent 自己都不许自动加 skip 来「先把测试跑绿」。

【AI 研发实战】覆盖率门禁的"说"与"做":从墙上的 95% 到配置里的 25%

xiaoshuai1024 included in AI编程
这篇文章属于我「AI 开发」阶段质量体系的一环:在 Claude Code 主导写代码的流程里,我用覆盖率当体检指标,但从不让它当 KPI。 我们项目上有个直接的对照。文档里白纸黑字写着后端行覆盖 95%、分支 85%,前端 85/75,运营后台 90/80;但翻构建文件里的门禁配置,实际卡的是后端 25%/20%、前端 80/60%、运营 70/60。 95% 和 25% 摆在一起,看着像双标,像 KPI 造假。我的判断是:它不是造假,它恰恰说明覆盖率是个滞后指标——把它当 KPI,数字迟早会变成表演。 几乎每个项目的覆盖率目标都"虚高" 一个直接的观察:凡是设过覆盖率门禁的团队,墙上的目标几乎都比配置里实际卡的高。 目标往高了写,是政治正确; 实际往下调,是工程现实。 两者之间的缝,就是"说"和"做"的缝。我们项目也一样:规则文件里那串 95/85/85/75/90/80 写得漂亮,是大家心里的"理想态";但每天卡提交的门禁,是那串低得多的数字。谁也没撒谎,只是这俩数字量的本来就不是同一件事。 95% 和 25% 都不是错:覆盖率衡量的是"被测",不是"正确" 这里有个基础认知,但容易被门禁数字带歪。 覆盖率只告诉你"这些行在测试里跑过了",不告诉你"这些行的行为是对的"。 一行代码被执行过,不代表它的语义被验证过。 所以一个项目行覆盖 95%,只能说明"测试碰到了大部分代码",不能说明"大部分代码被正确验证了"。反过来,25% 也不等于烂——如果关键路径都被测试网住,而那 25% 恰好覆盖在这些路径上,它可能比一个水分十足的 95% 更靠谱。 理解了这一点,95% 和 25% 就能并存: 95% 是"我们想达到的理想覆盖"; 25% 是"我们当前在门禁里实际能卡住、卡了不会把人逼去造假的底线"。 覆盖率会撒谎的几种姿势 覆盖率最坑的地方,是它能被轻轻松松地"刷"上去。常见的姿势有这么几类: 占位断言:expect(true).toBe(true) 这种,行是跑过了,什么都没验证。这也是我们项目里明令禁止占位测试的原因(见三层测试纪律那篇)。 ignore 滥用:把不好测的类/方法直接 @Ignore,眼不见为净,分母变小,百分比自然好看。 测了分支没测语义:if/else 两支都走到,覆盖率拉满,但边界值、并发、异常路径一个没碰。 AI 写测试时这几样尤其常见。Agent 特别爱用占位断言把绿条先凑出来——对它来说"测试通过"就是目标函数。你不去盯,它能刷出一个漂亮的覆盖率,顺便埋一堆假绿。 一次"为发版下调门禁"的诚实复盘 最有意思的是后端那个 25%。它不是一开始就这个数,是某次为了发版,我把门禁从更高的值往下调了。 在覆盖率原教旨主义者看来这是"妥协"。我的复盘是:那次下调是工程判断,不是妥协。 当时的真实情况是存量代码覆盖率本身就低。门禁卡在高位会怎样? 要么发版卡死,团队交付停摆; 要么大家开始刷覆盖率造绿,污染指标。 两个都是更糟的结局。门禁存在的意义是"不让质量往下掉",不是为了一个数字好看。在一个本就不高的基数上硬卡高线,只会逼出一堆占位测试和 ignore,把覆盖率这个本来还有点用的指标彻底污染掉。

【AI 研发实战】把项目规范写进代码底层:让新人和 Agent 都"开机即用"

xiaoshuai1024 included in AI编程
这一篇属于我「AI 研发实战」系列里讲工程规范固化的部分。当我们把 AI 协同做成可复用的研发体系后,规范能不能自动执行,直接决定了人和 Agent 能不能开机即用。 我见过太多份「项目规范」,它们最后都死在同一个地方:某个 wiki 的角落里,落满灰尘,最后一次更新是半年前,内容早就和代码对不上了。 新人入职第一周没人看,三个月后也没人看,直到踩了坑,有人恍然大悟"哦这事儿其实文档里写过"——可那又怎样,文档不会自己跳出来拦你。 我的判断是:不能自己执行的规范,就不是规范,是建议。建议这种东西,在人忙的时候、在 AI 自由发挥的时候,一律等于没有。 所以我们的做法是——把项目规范写进代码底层,让它机器可读、运行时可执行。这一篇就讲我们怎么做的,以及它怎么让新人和新 Agent 都做到"开机即用"。 新人最痛的不是技术,是"这事儿在我们这该怎么干" 先说我们到底在解决什么问题。 新人入职一个项目,真正卡住他的,通常不是某个框架不会用、某个语法没见过——那些查文档就行。 卡住他的是一堆项目专属的隐性规矩: 这个字段为什么要这么命名 那个 SQL 为什么必须带 tenant_id 提交前得先跑哪个脚本 危险操作有哪些不能碰 这些东西没写在任何教程里,只在老同事的脑子里。 更关键的是,这种隐性知识的传递成本极高。每来一个新人,老人就得口述一遍;老人忙了,新人就只能靠踩坑自学。 AGI 时代又多了个新麻烦——每开一个新的 agent,它也是一个"零背景新人",而且它不会主动来问你。 一旦你把"AI 当随时入职的新人"来 onboard,立刻就能看清:这事儿光靠文档不够,得靠机器能自动加载和执行的东西。 传统规范为什么不行 传统的规范有三个硬伤,改不掉的那种。 第一,没人看。 wiki 的链接藏在 IM 群置顶里,大家点一次就忘。 第二,会过时。 代码三个月一迭代,文档没人同步。文档说"提交前跑 A 脚本",实际早就换成 B 了,照着做反而出错。 第三,不可执行。 就算文档全对,它也是一段给人看的文字,不会在你犯错那一刻拦下你。它只能事后当呈堂证供:“你看,文档里写了哦。“可钱已经亏了。 这三个问题的根因是同一个:规范和人之间,隔着一层需要人主动去读、主动去遵守的"自觉”。而自觉是最不可靠的工程依赖。 我们的做派:规则、技能、hook、清单 于是我们把规范拆成了四类东西,全部塞进项目仓库里,跟着代码一起版本管理。 一是规则文件 顶层有一份总规则,每个 agent 启动都加载,里面带一套启动检查清单——它开机第一件事就是照着清单把自己捋一遍。 下面还有一批按文件类型自动触发的规则: 标了 alwaysApply 的永远生效 标了 globs 的只在碰到对应文件时才激活 改到某个 SQL 文件,相关的多租户规则自动浮上来,不用人提醒。 二是「技能」库 我们攒了 90 多个按需加载的技能,每个都是一段"遇到这种活儿该怎么干"的可执行知识。

【AI 研发实战】事故即规范:把每起事故固化成机器可执行的规则

xiaoshuai1024 included in AI编程
这是我「AI 研发实战」系列里讲治理落地的一篇。背景是:当 AI 协同研发把交付速度拉满,人工盯防根本跟不上,真正的护城河就变成——能不能把每一起事故,固化成机器自动执行的规则。 事故复盘最常见的死法 我见过太多事故复盘,流程都长一个样: 出事 → 群里炸锅 → 复盘会。 写一份 wiki → 链接甩群里。 三个月后没人记得,同样的坑再踩一次。 我自己也写过这种 wiki。后来我想明白一件事:一份规范的价值,取决于它能不能被机器执行。 挂在 wiki 上的教训,本质是"希望下次有人记得"。 固化成代码的教训,是"下次机器直接拦你"。 这两者的 ROI 差了一个数量级。 所以这两年我们团队立了个规矩——事故即规范。每起线上事故或险情,复盘完不是写文档,是落成一条可执行规则。下面是我这几年攒下来的真实转化路径和几个例子。 规则的三种载体 先说方法论。我们把事故教训固化成三种东西,各有各的适用场景: git hook(运行时拦截):针对"绝对不能发生"的动作,比如推错分支、跑危险命令。在动作发生的那一刻直接拦下,不留任何侥幸。 脚本(提交前门禁):针对"跨层契约对不齐"这类编译器管不到的错误,放在 pre-commit,提交前就扫描。 agent 规则(评审时检查):针对需要"理解语义"才能判断的问题,写进 Agent 的规则文件,让评审回路在每次提交时自动查一遍。 判断标准很简单:这条规则能不能不靠人、自己跑起来。能,就固化;不能,就老老实实写文档,但别指望它生效。 案例 1:误推 dev → dev,加分支校验函数 有一次,有人(我就不说是不是我了)在本地切到 dev 分支拉了点东西,顺手把一个改动 commit 进 dev 再 push 上去——直接污染了主干分支。 好在是中午,被发现得早,回滚没费太大劲。 复盘的时候大家都说"以后注意别在 dev 上 commit"。这句话的保质期大概两周。于是我们在 git hook 里加了个分支校验函数:本地分支非 feature/、bugfix/、hotfix/ 开头的,一律拒绝 commit、拒绝 push。 dev、main 这些分支想推都推不动,hook 直接报错退出。从那以后这类事故再没发生过。规则一次写好,永久生效,比任何群公告都管用。 案例 2:并发守卫被"幽灵删除",加测试门禁 这个我在另一篇里详细写过。简单说:一次"搜索优化"提交,静默删掉了 270 行并发安全代码,而 mvn compile 照样通过——因为编译只校验语法契约,不校验行为。

这篇属于「前端 → 全栈 → AI 开发」这条线里的 AI 开发阶段。前面那些文章讲的是用 AI 写代码,这篇换个方向:用 AI 给产品做能力。具体是用 Spring AI 1.0.6 从 0 到 1 搭一个多租户 AI 助手,做成能上线、能治理、有安全边界的生产系统,而不只是跑通一个 demo。 一、要解决的核心问题 运营后台要一个 AI 助手,运营同学能用自然语言查数据、改配置:“查一下今天的订单”、“把这几个订单打上「已处理」标签”。 难点不在"能不能对话",在多租户。我们的平台服务很多租户,每个租户: 用自己的大模型(A 租户用 DeepSeek、B 租户用通义千问); 用自己的 API Key(各自的账单和额度); 有自己的知识库(检索各自的运营文档); 有自己的限流和预算。 这意味着不能像 demo 那样写死一个全局 ChatModel,所有东西都得按租户动态来。多租户和单租户的差别基本就在这一句,这篇也围绕它展开。 二、整体架构 先看全貌。一次"运营问一句话、AI 流式回答并可能调工具"的链路: 关键设计决策,后面逐个展开: 多供应商统一走 OpenAI 协议,一套代码吃三家模型。 多租户隔离落在模型实例这一层,每个租户一个独立 ChatModel。 RAG 靠 metadata filter,一个索引管所有租户。 Agent 工具定义和 Spring AI 解耦,中间自研一层桥接。 SSE 流式,前后端各有一套处理。 一次请求的时序 文字描述不如一张时序图清楚。运营问一句话、AI 流式回答并调工具的完整链路: 注意几个关键点:模型实例是按租户动态取的(工厂)、工具是模型自己决定调用并由 Spring AI 自动执行(internalToolExecutionEnabled)、全程流式返回。 三、多供应商:三家模型,一套代码 我对接了 DeepSeek、智谱 GLM、通义千问三家,没有为任何一家写独立 SDK。

这是「前端 → 全栈 → AI 开发」线上的 AI 开发阶段。上一篇《构建一个多租户 AI 助手》讲了整体架构,这篇把其中最被神话的一块——RAG 知识库单独拆开,从 0 到 1 讲它到底怎么落地。 RAG 在网上被讲得玄之又玄,什么"企业知识大脑"“智能问答中枢”。说穿了就是一句话:在问大模型之前,先从你的私有文档里检索出相关片段,塞进 prompt,让它"看着资料"回答。 它的价值不在魔法,在让大模型基于你的私有、可更新、可追溯的知识回答,而不是只靠训练时的旧记忆。 这篇不讲概念,讲工程:多租户怎么隔离、切片怎么配、向量库怎么选、入库失败的文档怎么重试、改了切片策略怎么重建。代码全部来自生产系统。 一、要解决的核心问题 运营后台的 AI 助手要能基于每个租户的私有运营文档回答问题——A 租户问"我们店铺的商品上下架流程",助手得去查 A 租户的运营手册,不能拿 B 租户的文档乱答。 需求拆成四个硬约束: 多租户隔离:A 绝对检索不到 B 的文档,这是底线,不是优化项。 文档可管理:运营能在后台上传、编辑、发布、删除文档,不是只能命令行灌库。 检索有质量闸:不相关的片段不能塞进 prompt,否则反而拉低回答质量。 可演进:切片策略、embedding 模型变了,得能批量重建,不能推倒重来。 RAG 的本质是四步:切片 → 向量化 → 存储 → 检索。但工程上远不止这四步,下面逐层展开。 二、整体设计 先看全貌。一篇文档从上传到被检索命中的完整链路: 几个关键设计决策,后面逐个展开: 双存储分工:MySQL 存可编辑的业务数据(标题/正文/标签/状态),ES 存向量做检索,两边各司其职。 多租户靠 metadata filter:所有租户共用一个 ai_knowledge 索引,检索时按 tenant_id 过滤。 质量闸在检索层:similarityThreshold(0.7) 卡掉不相关结果。 生命周期可重建:DRAFT/PUBLISHED 状态机 + es_indexed 标记 + 批量 rebuild。 一次入库的时序 文字不如图清楚。运营点"发布",一篇文档被切片入库的完整时序:
