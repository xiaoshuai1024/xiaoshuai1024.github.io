给一个 SaaS 做 2.5 万用户压测:我学到的几件事

xiaoshuai1024 included in 架构
这是我在全栈阶段做的一件事:给一个多租户 SaaS 系统做完整的 2.5 万用户压测。它检验的不是某个前端组件,而是一条从应用层、数据库到监控的完整链路能不能扛住真实流量。 很多人对压测有误解:找个工具,起几万个线程,对着接口猛打,看一眼 QPS 和 P99,完事。 我的判断是——压测真正的难点不在"跑工具",而在"压什么、怎么压、压完怎么知道哪儿慢"。工具五分钟就能跑起来,但前面那套假设、后面那套对齐,才是值钱的部分。 先想清楚压什么,再谈怎么压 接到压测任务的第一反应通常是"全链路压一遍"。这是错的。 2.5 万用户不是个均匀的数字。真实业务里,有人登录、有人浏览、有人下单、有人支付,比例完全不一样。所以第一步是拆场景: 核心链路:登录、浏览、下单、支付必须压,它们出问题用户最先骂。 读多写少 vs 写重:列表页、详情页是读;下单、支付是写。两者瓶颈模型完全不同,分开压。 峰值模拟:大促那种"0 点开抢"的瞬时尖峰,和平时的稳态是两个量级,分开建模。 这三件事想清楚,压测计划才算有了骨架。否则就是拿工具瞎打,跑完一堆数字,谁也说不清代表什么。 梯度加压,别一上来就猛踩 压测最常见的坑:目标 2.5 万并发,直接从 0 拉到 2.5 万,然后看服务挂没挂。 挂了也没用——你不知道是哪儿挂的。我的做法是梯度加压: 先 1 千,稳住看一遍; 再 5 千,再看; 再 1 万、2 万、2.5 万。 每升一档,人都盯着监控,看哪个指标先撑不住。 接着是稳态压测:不是冲一下就撤,而是在某个档位保持一段时间(我们一般稳态跑十几分钟),看服务能不能稳定扛住,而不是靠一波脉冲蒙混过关。 为什么要稳态?因为有些问题只有"长跑"才暴露:连接池慢慢漏、缓存命中率缓慢下降、GC 时间随堆涨上来。冲一下看不到这些,得让子弹飞一会儿。 压测必须和可观测性对齐,否则压了个寂寞 这条是踩坑踩出来的规矩。 有一次压测,跑完 QPS 看着还行,P99 也在可接受范围,结果上线后还是出问题。回头复盘才发现:压测时监控没对齐。我们盯着的是"请求成功率",但某个数据库连接池在压测中其实已经打满,只是有重试兜着,成功率没塌。 问题的本质是压测指标和线上风险没对齐。所以从那以后我们立了规矩:压测期间 Prometheus 必须实时开着,Grafana 上把这几类指标摆在一起盯: API 延迟(P50/P95/P99)、QPS、成功率; 数据库连接池的使用率、等待数; 缓存命中率、限流触发次数; JVM 的 GC、堆、线程数。 压测不是"跑完看报告",是"边压边看哪儿先冒烟"。压和盯必须同一拨人、同一时刻,不然压完根本不知道慢在哪。 瓶颈往往不在你想的地方 2.5 万用户压下来,我最确定的一条经验:瓶颈几乎从不在最初猜的地方。 我们一开始担心应用层、业务逻辑、某个慢查询。结果真压起来,最先冒烟的是: 数据库连接池:配置太小,高并发下请求排队等连接,延迟直接飙; 缓存:某个热点 key 没缓存住,全打到 DB; 限流:阈值配低了,正常流量被自己限流误伤,看起来像"服务不行",其实是"自己挡自己"。 应用本身的业务代码,反而是最不容易先垮的那一层。

接完 Figma 才搞懂:MCP 不是协议科普,是工具接入的标准化

xiaoshuai1024 included in AI编程
上一篇我讲了怎么自研一层桥接,把工具定义和 AI 框架解耦:业务同学写工具零框架负担,换框架一行不改。那篇解决的是「框架耦合」。 但写完我意识到一个更上层的麻烦:每接一个外部工具,我还是要写一遍适配。接个订单查询,写个 OrderQueryTool;接个设计平台的 API,又得写一套。工具越多,适配代码越堆。直到我接 Figma,踩了一通坑,才真正搞懂 MCP(Model Context Protocol)到底在解决什么——它不是又一个新协议让你学,是把这个「每个工具都重写一遍适配」的活儿,标准化掉。 这篇就用我把 Figma 接进 AI 工作流的真实过程,讲 MCP 怎么落地。不讲虚的,讲配额怎么省、双 server 怎么配合、怎么装。 一、MCP 到底解决什么:M×N 变 M+N 先说它解决的那个根本问题。 假设你有 3 个 AI 客户端(Claude Code、Cursor、Cline),要接 3 个外部工具(Figma、GitHub、数据库)。传统做法是每个客户端给每个工具写一套适配——3×3=9 套。客户端加一个、工具加一个,适配数就是乘法涨。这就是 M×N 问题。 MCP 做的事是中间插一个标准协议:客户端只跟协议说话,工具也只跟协议说话。于是 3 个客户端 + 3 个工具,只需要 3+3=6 个适配,每边各管各的。涨法从乘法变加法。 落到 Figma 这个例子上:我不需要为「Claude Code 怎么调 Figma」写适配,也不需要为「Cursor 怎么调 Figma」再写一遍。Figma 那边出一个 MCP server,任何支持 MCP 的客户端都能直接用。这就是它最实在的价值——工具方写一次,所有客户端通吃。 二、协议三件套,知道这些就够用 MCP 的协议本身不复杂,记住三样: 消息格式:JSON-RPC 2.0。 客户端和 server 之间就是发 JSON 消息,请求带 method 和参数,响应带结果。没什么花活。 三类能力(原语):

用 AI 产出百万行代码之后,我把这套 Skills 沉进了项目

xiaoshuai1024 included in AI编程
2026 年 6 月,线上数据丢了一批。 一个定时清理任务,逻辑是「扫整个对象存储,凡是数据库里没记录的,就当孤儿删掉」。听着合理,对吧?存储里有、库里没有的,可不就是垃圾。直到某天一个写入链路抖了一下——文件先落库、对象后上传,中间这几秒,定时任务扫过去了,「库里有、存储还没到」的对象被当成孤儿删了个干净。 这是我维护的一个多租户社交电商全栈项目。前端 uni-app(Vue 3)小程序、后端 Spring Boot 3 + MyBatis-Plus + MySQL + Redis + RabbitMQ、运营后台 Vue 3 + Element Plus。代码量早就过百万行,而其中绝大部分是 AI 写的。 事故复盘的时候我没怪模型。我怪的是:我给了 AI 一把没有保险的枪。它执行得又快又准,快到我没来得及看它要删什么。 百万行代码之后,我真正想明白一件事:瓶颈不在模型够不够聪明,在我怎么把工程纪律变成 AI 逃不掉的流程。我选的方式,是在项目里沉淀一套 Skills。 这篇就讲这套东西:我实际在用哪些 skill、每个在项目里管什么、以及它们是怎么装进来的。 一、先说清楚:Skill 不是教 AI 新知识 很多人对 Skill 有个误解,觉得它是「给模型补充训练数据里没有的知识」。不是。 模型的知识已经够多了。它知道 Spring Boot 怎么写、知道 Flyway 怎么迁移、知道 RBAC 怎么设计。它缺的不是知识,是约束——是你的项目里那套「必须先写测试」「DELETE 必须带 tenant_id」「声称完成前必须贴证据」的纪律。 Skill 干的就是这个:把「你应该怎么干活」写成一份可复用的指令,在合适的时机自动加载进会话。它不增加模型的能力上限,它抬高的是模型在你项目里的下限。 这是为什么我不去追「更强的模型」,而去攒「更密的 Skill 网」。模型每代都在变强,但你项目里那个「AI 一高兴就 DELETE FROM 不带 where」的风险,换什么模型都在。能焊死这个的只有流程,不是参数量。 二、这套 Skills 是怎么装进来的 先讲安装,因为这是最容易被问到的。我不是手写了几十个 skill——那不现实,也违背「复用」的初衷。项目里的 skill 来自一套类似 npm 的 skill 包管理器。
