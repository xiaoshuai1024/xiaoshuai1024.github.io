2.5 万用户压测实战:压什么、怎么压、瓶颈怎么找

xiaoshuai1024 included in 架构
给一个多租户 SaaS 系统做 2.5 万用户压测，最容易跑偏的不是「起几万个线程」，而是压之前没想清楚压什么、压之后不知道哪儿慢——工具五分钟就能跑起来，值钱的是前面那套假设和后面那套对齐。 压测真正的难点不在"跑工具",而在"压什么、怎么压、压完怎么知道哪儿慢"。它检验的是一条从应用层、数据库到监控的完整链路能不能扛住真实流量。 先想清楚压什么,再谈怎么压 接到压测任务的第一反应通常是"全链路压一遍"。这是错的。 2.5 万用户不是个均匀的数字。真实业务里,有人登录、有人浏览、有人下单、有人支付,比例完全不一样。所以第一步是拆场景: 核心链路:登录、浏览、下单、支付必须压,它们出问题用户最先骂。 读多写少 vs 写重:列表页、详情页是读;下单、支付是写。两者瓶颈模型完全不同,分开压。 峰值模拟:大促那种"0 点开抢"的瞬时尖峰,和平时的稳态是两个量级,分开建模。 这三件事想清楚,压测计划才算有了骨架。否则就是拿工具瞎打,跑完一堆数字,谁也说不清代表什么。 场景拆分实操:业务占比法 具体怎么拆?我们用的是业务占比法——拉过去 30 天的真实生产日志,统计各接口的实际调用比例,按比例还原压测脚本: # 压测脚本里的业务占比(k6 / JMeter 都行) scenarios: browse_list: # 浏览列表(读多) weight: 45 # 占比 45% rate: 450 view_detail: # 看详情(读多) weight: 30 rate: 300 place_order: # 下单(写) weight: 15 rate: 150 pay: # 支付(写,关键链路) weight: 8 rate: 80 search: # 搜索(读,但走 ES,重) weight: 2 rate: 20这个比例不是拍脑袋,是从生产日志里 group by api_path 统计出来的。如果压测脚本里把支付比例配成 50%,但生产里支付只占 8%,压出来的瓶颈点就是错的——支付链路过载是假象,真实生产里根本没这么高占比。

三家模型一套代码:多租户 AI 助手的动态 ChatModel 工厂

xiaoshuai1024 included in AI编程
前面那些文章讲的是用 AI 写代码,这篇换个方向:用 AI 给产品做能力。具体是用 Spring AI 1.0.6 从 0 到 1 搭一个多租户 AI 助手,做成能上线、能治理、有安全边界的生产系统,而不只是跑通一个 demo。 一、要解决的核心问题 运营后台要一个 AI 助手,运营同学能用自然语言查数据、改配置:“查一下今天的订单”、“把这几个订单打上「已处理」标签”。 难点不在"能不能对话",在多租户。我们的平台服务很多租户,每个租户: 用自己的大模型(A 租户用 DeepSeek、B 租户用通义千问); 用自己的 API Key(各自的账单和额度); 有自己的知识库(检索各自的运营文档); 有自己的限流和预算。 这意味着不能像 demo 那样写死一个全局 ChatModel,所有东西都得按租户动态来。多租户和单租户的差别基本就在这一句,这篇也围绕它展开。 多租户 AI 助手和单租户的差别,不止是「技术复杂度增加」,还有安全责任的质变。单租户系统里,所有数据是一个公司的,即使出了串租户的 bug,也是「自己看自己的数据」,影响可控。多租户系统里,一旦串租户,就是 A 公司看到了 B 公司的数据——这是商业机密泄露,可能引发法律纠纷,直接动摇客户对平台的信任。所以多租户 AI 助手的隔离,不是「优化项」是「底线」,做不好就不是「系统不好用」,而是「系统根本不能用」,性质完全不同。 还有一个容易被低估的难点:每个租户的 AI 使用习惯不同。有的租户把 AI 助手当「查询工具」用,主要是只读操作;有的租户让 AI 做大量写操作(批量改配置、批量打标签)。使用习惯不同,工具调用的频率和类型就不同,限流策略和审计重点也要跟着不同。我们的做法是给每个租户配独立的 rate_limit_rpm(每分钟请求限制),高频租户调高、低频调低,不是一刀切。这个「按租户差异化配置」的能力,是多租户 AI 助手比单租户复杂得多的地方——不只是隔离,还要个性化。 二、整体架构 先看全貌。一次"运营问一句话、AI 流式回答并可能调工具"的链路: 关键设计决策,后面逐个展开: 多供应商统一走 OpenAI 协议,一套代码吃三家模型。 多租户隔离落在模型实例这一层,每个租户一个独立 ChatModel。 RAG 靠 metadata filter,一个索引管所有租户。 Agent 工具定义和 Spring AI 解耦,中间自研一层桥接。 SSE 流式,前后端各有一套处理。 一次请求的时序 文字描述不如一张时序图清楚。运营问一句话、AI 流式回答并调工具的完整链路:

百万行 Vue2 升 Vue3:一个敢跑在真实项目的 codemod 怎么做

xiaoshuai1024 included in 前端
几百个文件、几十种破坏性变更,手改得改两周,还没人能 review 得过来——而且改的顺序会互相干扰,两个人动同一个文件,合并冲突能扯一下午。我做百万行 Vue2→Vue3 升级时,第一周就放弃了手改,写了个 babel codemod。这篇讲它怎么做到"敢在真实项目上跑"。 自动化升级的底气,从来不是"改得多快",是"改得多稳"。 一个敢往生产代码上跑的 codemod,是事务化 + 保守 + 安全网,缺一条都得崩。下面展开。 为什么不能手改 Vue2 到 Vue3 的破坏性变更远不止 new Vue → createApp。我盘点过,光"必动"的就一长串: 入口:new Vue().$mount() → createApp(App).mount() 全局 API:Vue.use / Vue.mixin / Vue.directive / Vue.filter 都挂在 new Vue 实例上,Vue3 要改成挂在 app 实例上(app.use(...)、app.directive(...)) Vue.prototype.xxx → app.config.globalProperties.xxx filters 整个被移除,模板里 {{ price | currency }} 全得改成方法或计算属性 v-model 的默认 prop 从 value/input 变成 modelValue/update:modelValue,组件库里几百个自定义 v-model 全得跟着改 $listeners 合并进了 $attrs,有 inheritAttrs: false 的组件行为会变 Vue.config.productionTip、Vue.config.ignoredElements 这些被删,留着运行时报错 <teleport>、<suspense> 是新东西,这次没动业务逻辑,只在脚手架里留好位置 一个中型项目几百个文件,上面这些散在各处。手改又慢又容易漏,而且——更关键的是——没人能 review 几百个人手改的 diff。改完根本不知道对没对。

低代码渲染引擎揭秘:schema 驱动到可编辑画布的取舍

xiaoshuai1024 included in 架构
拿一份 JSON 渲个页面发个 demo，这类项目 GitHub 上一抓一把。真正卡人的是后面三件事：能编辑、能扩展、能演进。我主导的低代码平台「鲁班」核心就是一个自研的渲染引擎 + 设计器，这篇把它拆开——schema 怎么设计、渲染器怎么递归、设计器怎么拖拽编辑。 项目归属说明：渲染引擎的设计经验源自大厂时期主导的内部系统"鲁班"，本文代码实现以个人开源项目 OpenLuban（架构同源、代码全新重写，GitHub）为准。 schema:一棵树 + 一个表单值字典 低代码引擎的起点是 schema。我把它设计得很薄,有意压到最小: interface NodeSchema { id: string type: string // 组件类型,如 'LubanButton' props?: Record<string, unknown> children?: NodeSchema[] } interface PageSchema { root: NodeSchema formState?: Record<string, unknown> // 表单值绑定容器 }一个关键决定:表单控件的值不挂在每个节点上,而是统一放在顶层 formState 字典里,以控件的 props.name 为 key。运行时只要一个扁平字典就能驱动整页所有表单字段,提交、校验、回填都简单。 老实讲这步我纠结过。每个节点自己存自己的值,直觉上更「自然」,写 schema 的人也省心。但一落地就发现问题:提交时要遍历整棵树去收集,校验时要在树里找控件,后端回填要把值塞回各自的 props——三个环节都要重新爬树。换成顶层一个字典,提交是 JSON.stringify(formState),校验是对字典跑规则,回填是 Object.assign。一个数据结构,省掉三处脏活。 还有一个更隐蔽的取舍:schema 要薄,但版本治理要从第一天留口子。PageSchema 现在没显式的 version 字段,不是忘了,是早期的 demo 还不值得为它背兼容包袱。但 NodeSchema 的结构我刻意没塞多余东西,留出加 version / compatibility 的余地。一旦真有存量页面要演进,加个字段、走个 migrator 就能接住。第一天就把 schema 塞满,后面想改一处得动全身——这种坑我见太多了。

一份代码怎么变成每个租户的小程序:代开 vs 按租户打包选型

xiaoshuai1024 included in 架构
做过 SaaS 多租户 Web 的人，第一反应都是「运行时识别租户就行」。但这套在微信小程序上直接死——manifest.json 里的 appid 在构建期就被烧死进产物，运行时改不了。这篇就讲这个硬约束逼出来的两种解法：按租户打包（构建期注入 AppId）和平台代开（一份模板 + ext_json 注入 tenantId）。 先把结论丢出来: 按租户打包:构建期把租户的 AppId 烧进 manifest.json,一份代码跑 N 次构建,产 N 个小程序包。简单直接,但租户越多构建成本越线性涨。 平台代开:基于微信开放平台第三方能力,一份代码模板给所有租户用,差异完全靠 ext_json 在 commit 时注入 tenantId。租户再多代码也只有一份,这是首选。 代开才是 SaaS 多租户小程序的正解,按租户打包是兜底。下文会展开为什么。 难点:微信小程序的 AppId 是构建期常量 做 SaaS 多租户 Web 的同学,第一反应通常是:“运行时识别租户不就行了?根据请求里的 tenantId 加载不同配置、走不同租户的数据。” 在浏览器里这套没问题,但在微信小程序里直接行不通。 原因藏在微信的约束里:manifest.json / project.config.json 里的 appid 字段,在构建期就被烧死进产物,运行时改不了。微信开发者工具、上传、提审、发布,整套流程都认这个 AppId——它对应的是微信开放平台里一个独立的小程序实体。 也就是说,“一份代码多个租户"在 Web 上成立,在小程序上必须在构建期(或发布期)就把租户的 AppId 注入进去。这是 SaaS 多租户小程序所有架构决策的前提,绕不开。 我们最初的设计也想"运行时识别”,做了两周发现走不通,才老老实实回到构建期注入这条路上。这个弯路本身值得单独说一节,因为很多人都在这里栽。 我们走过的弯路:运行时识别为什么死 最初的想法是用一个通用 AppId(开发者自己的)打底,运行时再根据用户登录的租户去走租户的逻辑。这个思路在"开发期"是成立的——本地用开发者 AppId 调试没问题。 但一上审核就崩了: 微信审核认的是包里烧死的 AppId,你挂羊头卖狗肉,审核驳回。 即使审核过了,用户搜不到这个小程序——它对应的微信实体是开发者自己的 AppId,不是租户的。 业务上要求"每个租户有自己的品牌入口、自己的小程序码、自己的搜索结果",这就要求每个租户在微信侧是一个独立小程序实体。 所以最终只能回到"构建期注入 AppId"这条路。区别只是:为每个租户单独打一份包(模式一),还是用第三方平台代开,一份模板包走遍所有租户(模式二)。

RAG 没那么神:用 Spring AI + ES 从 0 搭一个多租户知识库

xiaoshuai1024 included in AI编程
RAG 在网上被讲得玄之又玄,什么"企业知识大脑"“智能问答中枢”。说穿了就是一句话:在问大模型之前,先从你的私有文档里检索出相关片段,塞进 prompt,让它"看着资料"回答。 它的价值不在魔法,在让大模型基于你的私有、可更新、可追溯的知识回答,而不是只靠训练时的旧记忆。 上一篇《构建一个多租户 AI 助手》讲了整体架构,这篇把其中最被神话的一块——RAG 知识库单独拆开,从 0 到 1 讲它到底怎么落地。 这篇不讲概念,讲工程:多租户怎么隔离、切片怎么配、向量库怎么选、入库失败的文档怎么重试、改了切片策略怎么重建。代码全部来自生产系统。 一、要解决的核心问题 运营后台的 AI 助手要能基于每个租户的私有运营文档回答问题——A 租户问"我们店铺的商品上下架流程",助手得去查 A 租户的运营手册,不能拿 B 租户的文档乱答。 需求拆成四个硬约束: 多租户隔离:A 绝对检索不到 B 的文档,这是底线,不是优化项。 文档可管理:运营能在后台上传、编辑、发布、删除文档,不是只能命令行灌库。 检索有质量闸:不相关的片段不能塞进 prompt,否则反而拉低回答质量。 可演进:切片策略、embedding 模型变了,得能批量重建,不能推倒重来。 RAG 的是四步:切片 → 向量化 → 存储 → 检索。但工程上远不止这四步,下面逐层展开。 二、整体设计 先看全貌。一篇文档从上传到被检索命中的完整链路: 几个关键设计决策,后面逐个展开: 双存储分工:MySQL 存可编辑的业务数据(标题/正文/标签/状态),ES 存向量做检索,两边各司其职。 多租户靠 metadata filter:所有租户共用一个 ai_knowledge 索引,检索时按 tenant_id 过滤。 质量闸在检索层:similarityThreshold(0.7) 卡掉不相关结果。 生命周期可重建:DRAFT/PUBLISHED 状态机 + es_indexed 标记 + 批量 rebuild。 一次入库的时序 文字不如图清楚。运营点"发布",一篇文档被切片入库的完整时序: 注意 vectorStore.add(chunks) 这一行把 embedding 和写 ES 两件事打包了——embedding 由 starter 注入的 EmbeddingModel 完成,业务代码不碰向量细节。写成功后才回 MySQL 标 es_indexed=1,这个标记后面会反复用到。
